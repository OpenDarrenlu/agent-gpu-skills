本文介绍了什么

在训练场景下做MOE，可以结合FSDP和EP，顺着FSDP就把专家权重给迁移了，以实现卡间均衡。并且，基于这个机会，具体如何做卡间均衡。

介绍相关概念

为什么只有训练存在FSDP，而推理没有？

FSDP 解决的是训练内存问题。训练要存 parameters + gradients + optimizer states，而推理里后两者没有，所以它最大的收益少了很多。
FSDP 的代价是按层做 all-gather / unshard / reshard。训练时还能和前后层计算、反向传播重叠；推理没有 backward，尤其自回归 decode 很吃单 token latency，这个通信更容易直接落在关键路径上。
推理常见的大头瓶颈其实是 KV cache 和服务时延，不是 optimizer state。FSDP 只管参数分片，对 KV cache 帮助不大。
通信的优先级：MOE的dispatch优先级较高（在前向反向都是）
moe的dispatch需要优先于预取下一层MOE权重的通信，先做。（见下图的abc）
反向传播里moe的dispatch需要优先于下一层的梯度同步（见下图的de）

传统 FSDP 风格：某层梯度 ready -> autograd hook 触发 -> 立刻发起梯度同步
LAER 的做法：某层梯度 ready -> 先别急着同步 -> 延后到后续 expert backward 的计算窗口里再发通信




dX 和 dW

dX：对输入激活的梯度，记作 ∂L/∂x
dW：对参数的梯度，记作 ∂L/∂W

它们各自是干什么的

dX：用于把误差继续往更低层传，所以服务的是当前这次 backward 链条
dW：用于更新参数，所以服务的是本轮迭代末尾的 optimizer.step()

什么时候用

dX_L：第 L 层 backward 一算出来，更低层马上就要用
dW_L：不用立刻给下一层；只要在 optimizer.step() 前准备好即可
更新后的 W_new：通常在下一轮 forward里才生效

什么时候同步

dW：通常需要跨卡同步 原因：每张卡只算了自己那部分样本/token，对应的是局部梯度；要得到全局 batch 语义下的正确参数梯度，必须把各卡的 dW 聚合起来
dX：通常不做数据并行意义上的跨卡同步 原因：它主要是本地继续反传用，不是拿来做全局参数更新的

为什么 dX 有时也会通信

如果模型本身把激活/ token 分散到多卡上了，比如 TP / SP / MoE token dispatch
那么 backward 里的 dX 也可能伴随通信
但这类通信的目的，是把激活路径接回去，不是像 dW 那样做全局梯度聚合

为什么 dW 同步可以延后

因为下一层 backward 依赖的是 dX，不是 dW
所以某层的 dW 即使稍后再同步，只要在 optimizer.step() 前完成即可
这就是图 5 里能把 Sy 往后拖的原因
如何做专家权重的重排，以及token应该路由到哪个专家副本上

决定下一轮哪些 GPU 要临时恢复哪些完整 expert 副本来算

也就是求一个布局矩阵 A：

A[i, j] = 1：expert j 在这一轮会在 GPU i 上被恢复/放置
A[i, j] = 0：不会

同时再求一个路由矩阵 S：

S[i, j, k]：原本在 GPU i 上、属于 expert j 的 token，有多少会被送到 GPU k 上的那个 expert 副本去算

可以压成一句话：

先决定每个 expert 复制几份，再决定这些副本放到哪些 GPU，最后决定 token 发给哪个副本。

具体怎么做 它没直接解“全局最优”，而是拆成两个部分：

Token Dispatcher：快，负责 token 路由
Expert Layout Tuner：稍慢，负责 expert 重排

———

它用的不是“谁热就固定多给几份”这么简单，而是一个优先队列规则：

看每个 expert 的“平均负载”

也就是：

平均负载 = 当前 expert 的总 token 数 / 当前副本数

谁的平均负载最高，就给谁再加一个副本
然后更新这个 expert 的平均负载
继续挑当前平均负载最高的 expert
一直加到总副本预算用完

这个总预算大概就是所有 GPU 总共能容纳的完整 expert 数，也就是 N * C，其中：

N 是设备数
C 是每个 GPU 最多恢复多少个完整 expert

所以这一步的效果就是：

热门 expert 会被复制更多份
冷门 expert 复制更少
目标是把“每个副本平均承担的负载”拉平

———

第二步：决定这些副本放在哪些 GPU 这里它用的是一个 拓扑感知的贪心放置。

做法是：

先算每个 expert 的每个副本大概要处理多少 token 也就是 该 expert 负载 / 该 expert 副本数
把这些“副本任务”按负载从大到小排序
依次给这些副本选落点

每次放一个副本时，规则是：

先找“当前这个 expert 在哪些节点上的副本最少” 目的是让同一个 expert 的副本尽量分散到不同节点
再在这些候选节点里，选当前总负载最小的 GPU 目的是别把热点继续堆到已经很忙的设备上

所以这一步同时在做两件事：

让副本跨节点分散
让设备负载尽量均衡

———

第三步：token 怎么发给这些副本 布局定完，还要决定 token 发给哪个副本。

这里它没有做复杂的全局 token 级最优，而是一个很快的 lite routing：

如果某个 expert 在本节点内就有多个副本 - 那么发给这个 expert 的 token，优先在节点内副本之间均分
如果本节点内没有这个 expert 的副本 - 才在全局所有副本之间均分

这个策略的直觉很直接：

优先节点内，减少跨节点通信
不做重型全局协调，保证路由很快

———

第四步：不是只试一个方案，而是多试几个候选 论文还提到，只用“按负载加副本”未必最优，所以它会构造多个候选副本方案：

一个是按负载优先队列得到的方案
一个是所有 expert 尽量平均分副本的方案
再在这些基础上做一些随机扰动

然后：

对每个候选方案都跑一次“副本放置”
再估算整体时间代价
选最好的那个
