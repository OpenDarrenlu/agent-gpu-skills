前言
CPU-GPGPU 异构计算系统

GPGPU本身没有像Windows、Linux那样的通用操作系统，但它运行在一个由主机CPU操作系统和GPU驱动共同构建的、高度专门化的“软件环境”中。也就是说，GPGPU可以被认为是一颗协处理器，专门负责执行大规模数据并行的计算任务（内核），它依赖于主机操作系统 + 设备驱动 + 运行时库共同构成的一个专门化的软件栈来被管理和使用。目前的CPU-GPGPU异构平台可以大致分为三类。

经典的CPU-GPGPU异构计算平台。典型的CPU-GPGPU异构计算平台如下图所示，CPU的存储使用DDR，而GPGPU的存储往往使用更大带宽的GDDR或者HBM。CPU和GPGPU之间通过PCIE连接。CPU作为控制host，将程序的内核函数通过PCIE加载到GPGPU上进行计算。为了驱动内核函数的计算，所有需要的代码，配置和运行数据都需要从硬盘加载到主机端存储器（DDR），然后由一系列运行和驱动API将数据传送到GPGPU的设备端存储器中。当所有的配置、代码以及数据都准备完善后，GPGPU启动内核计算。当计算结果完成后，再传回主机端存储器中。

统一存储结构系统。统一存储的 CPU-GPGPU 共用主机端的存储器。采用硬件支持的统一寻址，CPU 和 GPGPU 都能直接访问主机端存储器，避免了 Host 和 device之间的数据传输过程。但是由于 GPGPU 需要更高的带宽，所以主机端的存储器往往会限制 GPGPU 的性能。另外，也可能由于存储器的争用导致访问延时的增加。

GPGPU 互连结构系统，通过如 NVlink 的互连协议实现多个 GPGPU 的并行工作。根据 NV 官网信息，NVLink 交换机能够在无阻塞计算结构中支持多达 576 个全连接GPU。

CUDA和OpenCL

CUDA，全称 Compute Unified Device Architecture（计算统一设备架构）, 是由NVIDIA公司推出的并行计算平台和编程模型。它允许开发者使用 C/C++、Python、Fortran 等语言，利用 NVIDIA 的GPU（图形处理器）进行通用目的计算（GPGPU）。CUDA 是一个专有的、封闭的生态系统，完全锁定在 NVIDIA 的GPU硬件上。但是他的“专一”也使得其软硬件配合更加完美，能达到更优的性能峰值，加上现在 NVIDIA的护城河已经建立，因此 CUDA基本是使用最多的GP/GPU编程框架。

OpenCL, 全称Open Computing Language（开放计算语言）, 是一个开放的、免版税的标准，用于跨平台的并行编程。它由Khronos Group（一个行业联盟，也维护着 OpenGL、Vulkan 等标准）管理。OpenCL的设计目标是硬件无关性。理论上，一段OpenCL代码可以在任何支持OpenCL的设备上运行，包括NVIDIA、AMD、Intel 的GPU, CPU, FPGA以及其他加速器（如DSP）。

编程模型

我们知道，GPGPU和CPU的一个区别在于，GPGPU擅长于使用大量小的计算单元完成并行度高的计算。这一节将说明一个高并行度的程序是如何分配到GPGPU的计算单元上的。

设备端代码常常由多个函数组成, 这些函数被称为内核函数收 (kernel)。当启动一个内核函数时，程序员需要显式指定一个庞大的线程层次结构，该结构中的所有线程都将并行地执行同一个内核函数的代码。例如, 在一个维度为 16×16 的矩阵乘法计算中,一种自然的方式就是1个线程计算1个结果矩阵的元素, 那么需要开启256个线程进行并行运算。

GPGPU 的线程分了三个层级：最小的执行单元是一个线程（Cuda称为Thread，OpenCL称为Work-Item，见表1，后文主要按照Cuda术语表达）。多个线程可以构成线程块（Block，在 NV 中还有一个称呼是CTA，cooperative thread array），线程块中的线程可以通过共享内存进行通信和同步。例如，程序员可以将上述的 256 个线程打包为一个Block。而一个Grid 网格中则含有多个Block。如图3所示，Grid 中的Block，以及Block 中的Thread可以有最大3维索引（也可以是1维或者2维，如图4）。在256个线程构成的Block在执行时，每个线程都运行相同的内核代码（指令），但会通过其唯一的线程索引来定位并计算结果矩阵中属于自己的那个元素，从而实现大规模并行计算和高性能。

这里需要注意，编程模型的Grid，Block的大小是由程序员指定，但是也需要根据硬件实现进行限制。比如，NVIDIA A100限制每个Block最大线程数为1024 个。另外需要注意kernel和Grid 的概念：一个GPGPU可以同时运行几个Kernel，一个kernel对应一个Grid，但是有数据依赖的kernel无法真正并发。

此外，为了解决传统模型中“Block之间无法高效通信和同步”的限制，NVIDIA在Hopper架构（例如H100 GPU）中引入的一个新编程模型层次——Cluster，位于Block和Grid之间，见5。它允许程序员将多个Thread Block组织成一个更大的可协作单元。Cluster内的不同Block可以通过新的Distributed Shared Memory进行通信，还可以通过__syncblocks()或cluster.sync()进行同步。

硬件架构

在编程模型中划分了三个层次（Grid，Block 和 Thread），本节希望弄清楚这些层次是如何映射到硬件执行的。

GPGPU 的核心部分包含了众多的可编程多处理器，NV 的称为流处理器（SM，Streaming Multiprocessor），AMD 称为计算单元（CU，Compute Unit）。而可编程对处理器中友包含了多个流处理器（NV 称为 SP，Streaming Processor，AMD 称为 PE，Processing Element)，支持整形、浮点、特殊函数和矩阵运算等多种不同类型计算。

在软件的编程模型里，当内核函数被启动时，GPGPU接受相应的线程网格，并以线程块为单位将不同的线程块分配给多个SM。GPGPU架构不同，每个SM能够接受的线程块数量也不同。比如，在NVIDIA的Volta架构中，每个 SM最多支持32个线程块。

线程块被分配到特定的SM后，同一个Block中的线程会被组织成以32个线程为单位的warp（硬件决定的参数，NV称为warp，OpenCL称为wavefront）。warp是SM调度和执行的基本单位，一个warp中的32个线程会在 SM 的多个SP上并行执行。

不同GPGPU架构下SM中SP的数量也不同。假设SP的数量是16，而warp大小为32，那么硬件需要2个时钟周期才能完成整个warp的指令执行。在同一个 warp中，32个线程运行相同的指令，但每个线程根据各自的 threadIdx 和 blockIdx 索引到各自的数据进行处理，从而实现 SIMT 计算模型。

值得注意的是，虽然GPGPU架构的不同导致各个架构参数都有所差异，但线程到硬件的分配过程对编程人员是完全透明的。这种方式使得CUDA和OpenCL程序可以在不同架构的GPGPU上具有良好的迁移能力。

GPGPU 指令流水线

图7显示了GPGPU-Sim模拟的SIMT Core微体系结构，SIMT核心模拟了一个高度多线程的流水线式 SIMD 处理器，大致相当于 NVIDIA 的SM或AMD的CU。SP（CUDA core）对应于SIMT核心中ALU流水线内的一个lane。

可以看到，每个 warp 按照流水方式执行指令的 fetch、decode、issue、execute 以及 writeback。这一过程和 CPU 类似，但区别在于从取指令开始，GPGPU 的流水线以warp 为粒度执行，且各个 warp 相互独立。同时 GPGPU 的指令调度器可以在任何已经就绪的 warp 中挑选一个并采用锁步（lockstep）方式执行。锁步执行使得所有的执行单元都执行同一条指令，从而简化控制逻辑，把硬件更多的留给执行单元。

Fetch 和 Decode

图7中的 I-Buffer 用于在从 I-Cache 中取出指令后对其进行缓存。I-Buffer 采用静态分区方式，以便在 SIMT 核心上运行的所有 warp 都有专用存储空间来存放指令。在当前模型中，每个 warp 拥有两个 I-Buffer 条目。每个 I-Buffer 条目包含有效位 (valid)、就绪位 (ready) 以及该 warp 的一条已解码指令 (inst)。有效位 v 表示该 I-Buffer 条目中存在尚未发射的已解码指令；而就绪位 r 则表示该 warp 的已解码指令已准备就绪，可发射至执行流水线。从概念上讲，就绪位是在调度与发射阶段根据记分牌逻辑和硬件资源可用性进行设置的。I-Buffer 初始状态为空，所有有效位和就绪位均处于未激活状态。

若 warp 的 I-Buffer 中不存在任何有效指令，则该 warp 具备指令获取资格。具备资格的 warp 按循环轮询顺序被调度访问 I-Cache。一旦被选中，系统会向 I-Cache 发送读请求，地址指向当前调度 warp 的下一条指令。默认情况下会连续获取两条指令。当warp 被调度进行指令获取后，其 I-Buffer 中的有效位将保持激活状态，直至该 warp 获取的所有指令均发射至执行流水线。

I-Cache 是只读的非阻塞组相连缓存，可模拟 FIFO 和 LRU 替换策略，并支持未命中时分配或填充时分配策略。对 I-Cache 的请求可能产生命中 (hit)、未命中 (miss) 或预留失败 (reservation fail) 三种结果。若缺失状态保持寄存器已满，或因缓存组中所有块均被先前待处理请求预留而无可用替换块时，则会导致预留失败。无论命中或未命中，循环轮询获取调度器都会继续处理下一个 warp。命中时，所获取的指令将被送往解码阶段；未命中时，I-Cache 将生成请求。当收到未命中响应时，相应块会被填入指令缓存，且该 warp 需要重新访问 I-Cache。在未命中请求待处理期间，该 I-Cache 不会访问指令缓存。

当 warp 的所有线程均已完成执行且不存在未完成的存储操作或对本地寄存器的待写入操作时，该 warp 视为执行完成，并不再被获取调度器考虑。当线程块内所有 warp均已完成且无待处理操作时，该线程块视为完成。当内核启动时分配的所有线程块均完成后，该内核视为执行完成。在解码阶段，新获取的指令被解码并存储至 I-Buffer 的对应条目中，等待发射。

Instruction Issue

第二轮循环仲裁器 (round robin) 从 I-Buffer 中选择一个 warp，将其指令发射至流水线的后续阶段。该循环仲裁器与用于调度指令缓存访问的循环仲裁器相互独立。Issue调度器可配置为每周期从同一 warp 发射多条指令。当前被检查 warp 中的每条有效指令（即已解码未发射）若满足以下条件即具备发射资格：（1）其所属 warp 未在 barrier等待;（2）其 I-Buffer 条目中包含有效指令（有效位已置位）;（3）通过记分牌检查;（4）指令流水线的操作数访问阶段未处于停滞状态。

内存指令（Load,Store 或者 barrier）被发射至内存流水线。对于其他指令，若可同时使用 SP 与 SFU 流水线，则始终优先选择 SP 流水线。然而，若检测到控制冒险，则清空该 warp 在 I-Buffer 中对应的所有指令。该 warp 的下一个程序计数器将更新为指向下一条指令（假设所有分支均未跳转）。

在发射阶段会执行 barrier 操作，同时更新 SIMT 堆栈并跟踪寄存器依赖关系。warp在发射阶段等待栅栏（例如”__syncthreads()”）。

SIMT Stack

为处理 SIMT 架构中的分支发散（branch divergence）执行问题，采用了基于每warp 的 SIMT 堆栈。由于 divergence 会降低此类架构的效率，可采用不同技术来缓解此影响。其中最简单的技术之一是后支配者堆栈重收敛机制，该技术通过在最早确保的重收敛点同步发散分支，以提升 SIMT 架构的效率。

SIMT 堆栈中的每个条目代表不同的发散层级。每当遇到发散分支时，新条目会被压入堆栈顶部；当 warp 抵达重收敛点时，栈顶条目会被弹出。每个条目存储着新分支的目标 PC、直接后支配者重收敛 PC 以及发散至该分支的线程活跃掩码。在本模型中，各 warp 的 SIMT 堆栈会随该 warp 的每次指令发射而更新。若无分支发散，目标 PC通常更新为下一条指令 PC；若出现分支发散，则新条目将被压入堆栈，其中包含新目标 PC、发散至该 PC 的线程活跃掩码及其直接重收敛点 PC。因此，当 SIMT 堆栈栈顶条目的下一 PC 与当前受检指令的 PC 不匹配时，即检测到控制冒险。

需要说明的是，业界已知 NVIDIA 与 AMD 实际通过特殊指令修改其发散堆栈内容。这些发散堆栈指令未在 PTX 中公开，但在实际硬件 SASS 指令集中可见（可通过decuda 或 NVIDIA 的cuobjdump 工具查看）。Scoreboard 记分牌算法用于检测 WAW 与 RAW 冲突。如前所述，warp 所要写入的寄存器会在发射阶段被预留。该记分牌算法以 warp 的 ID 为索引，将所需寄存器编号存储在与 warp 的 ID 对应的条目中。所预留的寄存器会在写回阶段被释放。如前所述，只有当记分牌指示不存在 WAW 或 RAW 冲突时，warp 的已解码指令才会被调度发射。记分牌通过追踪那些已被发射但尚未将结果写回寄存器文件的指令将要写入的寄存器，来检测 WAW 和 RAW。

Register Access and the Operand Collector

多家英伟达专利描述了一种称为” 操作数收集器” 的结构。该操作数收集器由一组缓冲器和仲裁逻辑构成，通过多组单端口 RAM 实现多端口寄存器文件的等效功能。这种整体设计有助于节约能耗和芯片面积，这对提升吞吐量至关重要。需要说明的是，AMD 也采用分库式寄存器文件，但其编译器需确保访问操作不会引发库冲突。

指令完成解码后，系统会分配一个称为收集器单元 (collector) 的硬件单元，用于缓冲该指令的源操作数。

收集器单元并非用于通过寄存器重命名来消除名称依赖，而是作为一种在时间上分散寄存器操作数访问的机制，以确保每个周期内对每个存储库的访问不超过一次。在图示架构中，四个收集器单元各自包含三个操作数条目。每个操作数条目包含四个字段：有效位、寄存器标识符、就绪位和操作数数据。每个操作数数据字段可容纳一个 128 字节的源操作数，该操作数由 32 个四字节元素组成（对应 warp 中每个标量线程的四字节数值）。此外，收集器单元还包含用于标识指令所属 warp 的标识符。仲裁器中为每个存储库设有读请求队列，用于暂存访问请求直至获得处理权限。

当从解码阶段接收到指令且存在可用收集器单元时，该单元将被分配给对应指令，同时设置操作数、warp 标识、寄存器标识符及有效位。此外，源操作数读请求会被加入仲裁器的队列。为简化设计，执行单元回写的数据始终优先于读请求处理。仲裁器会选择最多四个无冲突的访问请求发送至寄存器文件。为减少交叉开关和收集器单元的面积，设计确保每个收集器单元每周期仅接收一个操作数。

当每个操作数从寄存器文件中读出并存入对应收集器单元时，其” 就绪位” 将被置位。最终，当所有操作数准备就绪后，该指令将被发射至 SIMD 执行单元。

在本模型中，每个后端流水线（SP/SFU/MEM）都配有专属的收集器单元组，同时它们还共享一个通用收集器单元池。各流水线可用的单元数量以及通用单元池的容量均为可配置参数。

ALU

ALU 流水线实现了两种计算单元：1、SP 单元执行除超越函数之外的所有类型的ALU 指令。2、SFU 单元执行超越函数指令（正弦、余弦、对数等）。

两种单元均采用流水线设计和单指令多数据化架构。SP 单元通常每周期可执行一条 warp 指令，而 SFU 单元可能每隔数个周期才能执行新 warp 指令，具体取决于指令类型。例如，SFU 单元可每 4 周期执行一条正弦指令，或每 2 周期执行一条倒数指令。不同类型的指令还具有不同的执行延迟。

每个 SIMT 核心配备一个 SP 单元和一个 SFU 单元。每个单元均拥有来自操作数收集器的独立发射端口。两个单元共享连接至公共写回阶段的输出流水线寄存器。操作数收集器输出端设有结果总线分配器，以确保功能单元不会因共享写回阶段而停滞。每条指令在发射至任一单元前，都需要在结果总线中分配周期时隙。需要注意的是，内存流水线拥有独立的写回阶段，不受此结果总线分配器管理。

存储层级辨析

表2列出了 CUDA 和 OpenCL 对存储层次命名的差异，本节主要以 NV 为例说明各层次存储的作用。

存储种类
Register file：大容量的寄存器文件堆。
Global memory：位于 device memory，通过 32、64 或 128 字节的内存 transaction访问设备内存。这些内存事务必须自然对齐：内存事务只能读取或写入与其大小对齐的 32、64 或 128 字节的设备内存段（即，其第一个地址是这些 size 的倍数）。Global memory 的访问需要经过 L2 和 L1。
Local memory：位于 device memory。本地内存访问总是以与全局内存访问相同的方式缓存在 L2 中。可能放置在 local 内存中的自动变量包括：（1）无法确定是否使用常量索引的数组；（2）占用过多寄存器空间的大型结构体或数组；（3）当内核使用的寄存器超过可用数量时的任何变量（该现象称为寄存器溢出，registerspilling）。
Shared memory：shared memory 是 SM 层级的。共享内存比本地或全局内存具有更高的带宽和更低的延迟。为了实现高带宽，共享内存被分成大小相等的内存模块，称为 bank，可以同时访问。shared memory 用于线程间的通信和同步。
Constant memory：位于 device memory，有 constant cache。
Texture and surface memory：位于 device memory 并且缓存在 texture cache。

Cache

Global 和 local memory 经过 cache：L1 cache 的一个显著特点：L1 和 sharedmemory 共享（NV 中称为 share memory，OpenCL 中称为 local memory）共享一块存储区域。API 函数设置共享存储器的大小（如 cudaFuncSetAttributr()），则 L1大小为存储区域大小-共享空间大小。shared memory 为线程块中所有线程提供了一块公共的高速读写区域，以便线程间进行数据交互和协作。该存储区域由编程人员显式管理。

L2 Cache 由所有 SM 共享，用于缓存对 global memory 的访问，包括临时寄存器溢出。应用程序可以通过检查 l2CacheSize 设备属性来查询 L2 缓存的大小。Global memory 的访问 cache 在L2。

Constant memory 有自己的只读 constant cache，但是常量 cache 较小。texture memory 也有相应的只读 texture cache。
