0. 核心目标是什么？

既然要做LLM推理加速，首先要明确我们到底要“加速”什么？目前主流加速包括以下三种方向：

Latency (延迟): 即 用户发送一个请求，模型要多久能输出。Latency = 生成第一个 Token 的时间 (TTFT) + 生成每一个输出 Token 的时间(TPOT) *（要生成的 Token 数 - 1）
Throughput (吞吐): 即系统同时处理多个用户请求的能力，比如说系统中有两个用户的请求，每个请求每秒输出4个token，throught就是 2 * 4 = 8 token/s。但要注意，throughput 
≠
 latency。
Cost (代价)：主要指硬件成本和电力成本。


1. GPU 基本知识

关于GPU的硬核介绍有很多，如果要具体了解需要花很多时间。这里简单说一下我对GPU的理解：（摘抄自文章 图解大模型计算加速系列：FlashAttention V1，从硬件到计算逻辑）

GPU 是分层的（搬出flash attention神图）：
GPU分层结构，图源flash attention

简单理解下来就是，计算过程中我们首先要把数据从CPU搬到GPU上 (CPU DRAM -> GPU HBM)，但这还没完，真正计算的时候我们还要把数据从GPU的外部搬到内部 (GPU HBM -> GPU SRAM)。这里有一些很重要的结论：

(1) 由于SRAM的空间比较小，GPU在做运算的时候，如果矩阵大小超过SRAM，就要对矩阵进行切分；

(2) 由于SRAM的空间比较小，且每次运算都要把数据读进SRAM再写回HBM，因此实际的读写次数非常多，读写时间不可忽略；

(3) 由于计算和读写可以并行，所以一次运算的实际运行时间是 max(计算时间，读写时间) 。当计算时间更大时，称之为compute-bound；当读写时间更大时，称之为 memory-bound。

随着现在硬件的发展，算力增长非常迅速，以及linear attention等各种技术的出现在降低模型对算力的需求，计算时间将越来越不主导运行时间；相反，决定读写时间的带宽的增长反而较少，读写时间将越来越主导整个运行过程。因此，ai infra / mlsys 的一大主要目标就是降低读写次数 / 读写时间，这也是我认为speculative decoding很重要的原因：它是为数不多的能平衡compute-bound和memory-bound的方法之一。

这里并没有提到 "kernel" 这个概念，事实上我对此也不太清楚，只是简单的把kernel理解为启动读写和运算的函数/进程。既然是进程，就要有"准备进程" 和 "结束进程" 的额外开销，因此通过融合 kernel 来减少读写次数的方法，实际上的收益是 "使计算时间和读写时间更加overlap" + "减少kernel的额外开销"。我用一个例子来理解：

假设kernel 启动需要 1s, 结束需要1s，compute需要2s，读写需要1s，则

(1) 单轮运行的时间是 1s (启动) + max(2s, 1s) (计算和读写取较大值) + 1s (结束) = 4s，两轮运行需要8s；

(2) 假设我们将两轮的kernel融合，由于GPU的并行特性，compute保持 2s, 读写需要2s，则运行时间是 1s (启动) + max(2s, 2s) (计算和读写取较大值) + 1s = 4s;

因此kernel融合就可以得到一些额外收益。当然这个例子可能有很大问题，如有谬误请随时指出。




2. LLM的推理过程

了解了GPU的基础知识之后，我们结合的看一下LLM的推理过程，就可以从中推导出可以优化的点。

LLM的推理过程是一个 token-by-token 的过程。从用户输入一段文字开始，

首先通过一个分词器 (tokenizer) 将一段文字处理为 离散的数字id (one-hot化), 每个token id会对应一个独特的编码向量 (embedding)。这个embedding是LLM自带的；不妨假设输入的文字离散化之后的长度为 L , 每个embedding对应的维度是 d , 则这段文字对应的编码向量就是一个 L \times d 的矩阵 （省略batch size维度）
LLM的输入就是这个矩阵；我们把LLM看成一个黑盒函数，则输入 L \times d 的矩阵，输出 L \times d 的矩阵；
再通过一个 LLM Head 将这个矩阵转换至 L \times V ；其中 V 是词表大小。通过对词表这一维度做softmax操作，我们就得到了每个token预测next token的概率。在自回归解码中，我们使用最后一个token对应的概率分布来预测出next token. 此时，我们就得到了一个新的句子序列，长度为 L + 1 .
重复2-3，直至生成了停止词 (eos token) 或者达到了最大长度 max length。
LLM Inference草图

而这个过程其实可以分成两部分：prefill和decode。简单来说，prefill是一次性encode用户输入的过程；而decode则是在prefill之后，迭代式的生成及预测的过程。为什么要这么区分呢？

表象：(1) prefill阶段没有kv cache; (2) decode阶段有kv cache; 实际上：

(1) prefill 的输入是用户输入的完整文字，多个token；LLM是一次性的读取这些token完成运算，大部分情况下是compute-bound的；

(2) decode的输入是新预测的next token，只有一个；LLM 读取一个token然后运算，大部分情况下是memory-bound的。




通过以上分析我们就能得到一些优化技术的起源了：

prefill和decode的算力需求不一致 -> PD分离技术
decode阶段是memory-bound -> speculative decoding技术
