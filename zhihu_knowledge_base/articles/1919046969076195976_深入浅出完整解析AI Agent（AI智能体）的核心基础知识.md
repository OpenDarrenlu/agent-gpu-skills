2. 零基础深入浅出理解AI Agent的核心基础知识
2.1 AI Agent整体架构初识

AI Agent（人工智能智能体）是构建于LLMAIGC大模型之上，能够自主感知环境、自主思考、自主规划决策、调用工具并执行任务的计算机智能系统。其核心在于行为模式由“目标驱动”而非被动响应用户指令，它具备高度复杂的工作流设计，无需人类持续干预即可主动完成交互。作为人工智能技术的一种实体化形态，AI Agent 在自主决策与交互协作方面表现突出，不仅可独立运作，还能与其他智能体或人类进行有效协同。

AI Agent 的核心运行机制依赖于四大关键要素：任务规划、记忆存储、工具调用与行为执行。其中，LLM/AIGC大模型承担了“智能中枢”的角色，通过将复杂问题拆解为具有逻辑顺序的子任务，进而灵活调度大语言模型本身、检索增强生成技术以及文生图/文生视频等多种工具，最终实现整体问题的解决。

传统AI（如聊天机器人）仅能依次根据用户引导完成单轮问答，而AI Agent则可将复杂任务进行自主分解（如“订机票→查天气→生成行程表”），并且具备动态调整能力和长期记忆。

当前，一个主流AI Agent系统中通常包含：

感知（Perception）模块：获取外部环境信息，接收多模态输入（文本、图像、传感器数据），例如GPT-4o可端到端处理视觉与语音。
决策（Planning）模块：基于LLM/AIGC大模型进行思考、规划、决策，通过思维链（CoT）、思维树（ToT）等技术拆解复杂目标，并基于反馈持续优化策略。
行动（Action）模块：调用工具（API、数据库、搜索引擎、机器人肢体等）去完成任务。
记忆（Memory）模块：记忆模块又可以分为短期记忆和长期记忆。短期记忆依赖大模型的上下文窗口（如128K tokens）存储数据和知识，用于优化任务效果。长期记忆则通过向量数据库+RAG实现历史经验存储与检索。

下面Rocky举一个通俗易懂的AI Agent案例，让大家能够直观的了解AI Agent的感知-决策-执行闭环：

输入：用户目标（如“策划北京三日游”）。

推理：LLM分解子任务（订酒店→排行程→预算计算）。

执行：调用工具（携程API订房、高德地图路径规划）。

反馈：根据执行结果修正计划（如酒店满房时自动切换选项）。

同时，Rocky也认为我们应该学会运用「大模型的能力」，而「大模型的知识」可能就像人类一样，永远存在幻觉或者说缺陷。

在AIGC时代，不断有新的大模型出现，大厂不断推高大模型的参数规模、持续优化其性能与效率，本质上是为了让大模型「涌现」出更强大的“能力”，而不仅仅是为了往里面填充更多的“知识”。

从这个角度看，模型始终会存在幻觉——不管参数规模是从14B到32B，还是扩大到235B、671B，甚至突破1T，幻觉都不会消失。因为模型训练的本质，是对知识进行一种“有损压缩”。

而正是因为这样，大模型所提供的是“解决问题的能力”，而不是一个“可信的内部知识库”。所以在LLM/AIGC大模型基础上构建AI Agent系统，更是成为了AIGC时代发展的必经之路。我们需要给予大模型更多的支持和帮助，注入更充分的背景信息。

2.3 AI Agent的主流设计&工作模式

当前主流的AI Agent（Manus、Deep Research等）都是基于LLM大模型+ 一整套AIGC算法解决方案(Prompts工程、Function Call、MCP、AI工程策略、AI功能服务等)构建而成，同时未来还会持续扩展延伸其内涵。

基于上面的框架，接着再形成了5种主流的AI Agent设计模式：

反射模式（Reflection pattern）：这个模式的核心运作机制是构建自检-纠错迭代环，AI Agent会审查其工作以发现错误并迭代，直到生成最终输出结果。

通俗来讲，Reflection（反思） 就如同一位作家完成初稿后，进行的自我审阅与修改过程。他不仅检查错别字和语法，更会评估文章的逻辑结构、论证力度和文笔风格，然后针对性地进行重写和优化。

在AI Agent领域，Reflection被定义为一套让智能体对自身或他人的推理过程、行动计划或生成结果进行批判性审视与评估，从而发现错误、不足并实现自我改进的技术框架。其核心是引入一个独立的“反思循环”，将单一的前向生成过程，升级为一个“生成-评估-改进”的迭代闭环，其中的核心是构建一个双Agent系统：

“作家”Agent：负责代码的生成与编写。
“评委”Agent：负责对代码进行静态分析、逻辑审查和潜在漏洞检测，并生成具体的修改建议。

通过这两个角色的对话与协作，写作质量在多次迭代中得以显著提升。这一过程，便是Reflection的典型体现。

Reflection的本质是通过元认知（对认知的认知）来提升性能。它通过以下步骤实现：

步骤分离：将“执行”和“评估”两个任务分离开，由同一模型的不同实例或专精于批判性思维的不同模型来承担，以避免“当局者迷”的思维定势。
批判性审视：反思者会基于预设的标准（如正确性、效率、风格、安全性）对执行者的输出进行系统性检查。
形成反馈闭环：反思者产生的具体、可操作的反馈，被作为新的输入，引导执行者在下一轮迭代中进行改进。

Rocky接下来以代码的生成与优化为例，向大家详细讲解AI Agent中的机制与实现。

场景：生成一个数据处理脚本。
过程：
程序员Agent生成初始代码。
审查员Agent发现其未处理异常输入，且循环效率低下。
程序员Agent根据反馈，增加异常处理逻辑并优化算法。
循环直至审查员满意。

以下是一个高度通用的AI Agent Reflection代码框架，展示了Reflection机制的核心工作流程。

import openai

# 定义一个主执行Agent（例如程序员）
def actor_agent(task_description, feedback=None):
    system_prompt = "你是一名专业的程序员。请根据任务描述和反馈编写高质量的代码。"
    if feedback:
        user_content = f"任务: {task_description}\n之前的反馈: {feedback}\n请根据以上反馈改进你的代码。"
    else:
        user_content = f"任务: {task_description}\n请开始编写代码。"

    # 调用LLM生成代码
    response = openai.chat.completions.create(
        model="gpt-5.1",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_content}
        ]
    )
    return response.choices[0].message.content

# 定义一个反思Agent（例如审查员）
def reflection_agent(task_description, actor_output):
    system_prompt = "你是一名严格的代码审查员。你的目标是发现代码中的错误、低效之处和潜在风险，并提供具体的改进建议。"
    user_content = f"审查以下代码：\n任务描述: {task_description}\n生成的代码: {actor_output}"

    # 调用LLM进行反思
    response = openai.chat.completions.create(
        model="gpt-5.1",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_content}
        ]
    )
    return response.choices[0].message.content

# 主循环：Reflection Process
def reflection_loop(task, max_iterations=3):
    current_output = None
    feedback = None

    for i in range(max_iterations):
        print(f"--- 第 {i+1} 轮迭代 ---")
        # 1. 执行者生成输出
        current_output = actor_agent(task, feedback)
        print(f"【执行者输出】:\n{current_output}")

        # 2. 反思者进行评估
        feedback = reflection_agent(task, current_output)
        print(f"【反思者反馈】:\n{feedback}")

        # 3. 简单判断：如果反思者认为“无需修改”，则提前结束
        if "无需修改" in feedback or "代码良好" in feedback:
            print("反思完成，结果满意。")
            break
    else:
        print(f"已达到最大迭代次数 {max_iterations}。")

    return current_output

# 使用示例
if __name__ == "__main__":
    coding_task = "编写一个Python函数，计算斐波那契数列的第n项。"
    final_code = reflection_loop(coding_task)

# 代码框架解读：
# 角色分离：actor_agent 和 reflection_agent 是两个独立的函数，拥有不同的系统提示（System Prompt），模拟了不同的角色。
# 反馈循环：reflection_loop 函数实现了核心循环。在每一轮中，执行者根据任务和上一轮的反馈生成输出，然后反思者对该输出进行批判性评估，生成新的反馈。
# 迭代终止：循环设有最大迭代次数，防止无限循环。同时，可以基于反思者的反馈（例如，当反馈认为“代码已完美”）来提前终止，提升效率。

通过这种架构，AI Agent不再是“一次性”地生成答案，而是具备了自我质疑、自我改进的能力，这正是其迈向更高层次智能的关键一步。

工具使用模式（Tool use pattern）：AI Agent系统允许LLM/AIGC大模型通过使用外部工具获得更多信息，包括调用API、使用AI服务、查询矢量数据库、执行Python脚本等。这使得LLM/AIGC大模型不仅仅依赖于其内部知识，还可以获得互联网世界的庞大实时数据流来扩展知识边界。

Tool Use（工具调用） 的通俗比喻，确实如同一个人使用计算器解决复杂算术问题。然而，其技术本质是赋予LLM/AIGC大模型与环境交互并执行具体操作的能力。

这标志着LLM从一个封闭的、基于静态知识的生成式模型，向一个开放的、能够采取行动的问题解决者的根本性转变。模型不再仅仅“说出”它认为的答案，而是能够“动手”去获取或验证答案。

结构化调用：大模型不会随意地“说出”要调用的外部工具，而是会生成一个结构化的调用请求（例如，一个符合特定JSON/XML格式的请求）。这确保了后端的程序能够准确解析并执行。例如，对于计算请求，模型可能输出：{"tool": "calculator", "action": "evaluate", "input": "12345 * 67890"}。
结果整合与输出：模型接收到工具的返回结果（如上例中的 "838102050"）后，会将其作为新的事实依据，融入到最终的、自然语言的回答中。例如：“12345 乘以 67890 的结果是 83亿8千1百零2万零5百（83,810,20,500）。”

检索类工具：搜索引擎、知识库/向量数据库查询，用于获取实时、精准、私有的信息。
计算与数据处理类工具：计算器、代码执行环境（如Python）、API数据获取，用于完成精确计算和逻辑处理。
操作类工具：发送邮件、操作日历、控制智能家居，让模型能够影响物理世界。
专业领域工具：如法律案例检索、医学文献分析、金融数据分析等垂直领域的专业工具。

import openai
import json
import requests

# 1. 定义我们可供调用的工具列表
def get_current_temperature(city: str) -> str:
    """根据城市名获取当前温度。这是一个模拟函数。"""
    # 这里本应调用真实的天气API，例如OpenWeatherMap
    print(f"[系统日志] 正在查询{city}的天气...")
    # 模拟返回数据

    return f"{mock_data['city']}当前的温度是 {mock_data['temperature']} {mock_data['unit']}。"

# 将我们的函数描述成模型可以理解的形式
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_current_temperature",
            "description": "获取指定城市的当前温度",
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "城市名称，例如：北京，上海",
                    }
                },
                "required": ["city"],
            },
        },
    }
]

# 2. 与模型对话，并让它决定是否调用工具
def run_conversation(user_input):
    # 第一轮：将用户输入和工具描述发送给模型
    response = openai.chat.completions.create(
        model="gpt-5.1",
        messages=[{"role": "user", "content": user_input}],
        tools=tools,
        tool_choice="auto", # 让模型自动决定是否调用工具
    )
    
    message = response.choices[0].message
    print(f"【模型初始回复】: {message}")

    # 3. 检查模型是否想要调用工具
    if message.tool_calls:
        # 模型请求调用工具
        print("【模型决定调用工具】")
        # 可能存在多个工具调用，这里处理第一个
        tool_call = message.tool_calls[0]
        function_name = tool_call.function.name
        function_args = json.loads(tool_call.function.arguments)

        # 4. 在代码中执行被请求的函数
        if function_name == "get_current_temperature":
            city = function_args.get("city")
            function_response = get_current_temperature(city)
            print(f"【工具执行结果】: {function_response}")

        # 5. 将工具返回的结果作为新消息发送给模型，让它来总结
        second_response = openai.chat.completions.create(
            model="gpt-5.1",
            messages=[
                {"role": "user", "content": user_input},
                message, # 模型的第一次回复（包含工具调用请求）
                {
                    "role": "tool",
                    "tool_call_id": tool_call.id,
                    "content": function_response, # 工具的执行结果
                },
            ],
        )
        return second_response.choices[0].message.content
    else:
        # 模型没有调用工具，直接返回其回答
        return message.content

# 运行示例
if __name__ == "__main__":
    user_question = "请问北京现在多少度？"
    final_answer = run_conversation(user_question)
    print(f"\n【最终回答】: {final_answer}")

# 模型决策：我们将用户问题和工具列表一起发给模型。模型会推理出“北京的温度”需要调用天气工具，于是它返回了一个结构化的工具调用请求，而不是直接编造一个温度。
# 本地执行：我们的代码解析这个请求，并实际执行对应的本地函数（这里模拟了API调用）。
# 结果整合：我们将工具执行的结果（“北京当前的温度是22摄氏度”）作为新的上下文送回给模型。模型利用这个确凿的事实，组织成一句流畅的自然语言回复给用户。

通过这种方式，AI Agent不再是“万事通”，而是变成了一个“善于利用各种专业工具的团队协调者”，其能力和可靠性都得到了质的飞跃。

ReAct模式（Reason and Act）：ReAct模式结合了反射模式和工具使用模式，这使其成为当前AI Agent使用的最强大的模式之一。AI Agent既可以自我思考，自我纠错，还可以使用工具与世界交互。

规划模式（Planning pattern）：在这种模式下，AI Agent根据任务的复杂程度，设计任务计划流程，对任务进行细分，再对细分子任务动用ReAct模式进行处理。可以说这种模式是一种战略思维，可以更有效地解决战略级复杂任务。

这就像一位项目经理接到“举办一场产品发布会”的指令后，他不会立刻去搭建舞台，而是会先制定一个详尽的计划：市场调研 -> 确定主题 -> 邀请嘉宾 -> 撰写讲稿 -> 场地布置 -> 现场执行。这个制定计划的过程就是“规划”，而为每个步骤做出决策（例如选择哪位嘉宾、讲稿重点是什么）的思考过程就是“推理”。

在AI Agent中，Planning（规划） 指的是智能体为实现特定目标而制定一系列行动步骤的能力。Reasoning（推理） 则是支撑规划的基础，是智能体理解问题、进行逻辑推导、评估选项和预测结果的思维过程。其本质是让AI从“刺激-反应”的模式，升级到“目标-规划-行动”的高级认知模式。

任务分解：将模糊、复杂的用户指令（如“帮我做一份竞品分析”）转化为清晰、具体的子任务链（如“1. 确定竞争对手 -> 2. 爬取产品信息 -> 3. 对比功能特性 -> 4. 生成分析报告”）。
状态管理：在整个过程中，Agent需要维护一个“任务状态”，明确当前进度、已获得的信息以及下一步需要做什么。
工具编排：规划的核心作用之一就是智能地调用和组合工具。它决定在什么时间、以什么顺序、使用哪些工具，并将上一个工具的输出作为下一个工具的输入，形成高效的工作流。

上述讲到的流程主要是线性规划（Sequential Planning）是最简洁的一种。在实际中，还有更复杂的ReAct范式：

核心思想：将推理 和行动 交织在一起。
模式：Thought -> Act -> Observation -> ... -> Final Answer
示例：
Thought: 用户问珠穆朗玛峰的高度。我内部的知识可能不够准确，需要用一个搜索工具来查询。
Act: 调用 search_tool("珠穆朗玛峰 高度")
Observation: 工具返回结果：“珠穆朗玛峰高约8848.86米。”
Thought: 我得到了准确数据，可以回答用户了。
Final Answer: 珠穆朗玛峰的最新测量高度是8848.86米。

多智能体模式（Multi-agent pattern）：在这个模式中，AI Agent系统中包含多个子Agent，每个子Agent都分配有一个专用的角色和任务，同时每个子Agent还可以访问外部工具进行综合工作。最后，所有子Agent协同工作以提供最终结果，同时根据需要将细分任务委派给其他子Agent，形成一个复杂的“AI Agent协同社区”。

关于多智能体协作，ChatDev是一个绝佳的例子，它生动地展示了多个AI智能体如何像公司同事一样分工合作，共同完成复杂任务。

多智能体协作就是让多个具备自主决策能力的AI智能体通过沟通、协调与配合，去共同完成一个复杂任务。这背后的逻辑很简单：团结力量大。一个智能体可能只擅长某个特定领域，但当多个各怀绝技的智能体组成团队时，就能解决远超单个智能体能力范围的复杂问题。这就像组建一个项目团队，你会需要产品经理、工程师、设计师和测试人员协同工作。

要实现有效协作，智能体们需要一套运作机制。ChatDev中的角色专业化、记忆流和自我反思就是其中的关键部分。一套完整的协作流程通常包含以下几点：

角色专业化：每个智能体被赋予特定的角色和职责（如CEO、程序员、测试员），使其能够在专业领域内做出高质量决策。
记忆流：系统会保存对话的历史记录，使智能体能够记住之前的讨论和决策，从而在整体对话中保持上下文连贯。
任务规划与分解：一个“指挥”或“规划”智能体会将用户的复杂指令分解成一系列有序的子任务。
沟通与执行：智能体之间通过预定义的协议或通道进行对话，传递信息，并执行自己擅长的子任务。
结果汇总：最终，一个“编排器”或某个主导智能体会将各方的输出整合，形成完整的解决方案并交付给用户。

在实际应用中，多智能体系统演化出了多种协作模式，适用于不同的场景：

模式	运作方式	适用场景
工作流模式	智能体像流水线一样按固定顺序依次工作，前一个的输出是后一个的输入。	软件开发、多步骤推理管道。
路由模式	一个“路由器”智能体根据任务内容，将其动态分配给最合适的专业智能体处理。	客服系统、需要动态决策的场景。
层级模式	类似公司层级，顶层“经理”智能体负责规划和分发任务，并管理底层“员工”智能体。	复杂任务规划、需要全局协调的项目。
聚合模式	让多个智能体对同一问题给出方案，再由一个主智能体汇总提炼出最佳答案。	需要集思广益或进行质量评估的场景，如投票系统。
2.4 通俗易懂讲解AI Agent的System prompt、Assistant Prompt、User Prompt核心基础知识

在AI Agent主流框架设计中，定义了三种核心消息类型：System Prompt（System Message）、Assistant Prompt（Assistant Message）和User Prompt（User Message），三者功能明确区分：

User Prompt：代表用户的直接输入的问题。
Assistant Prompt：代表大模型生成的回复内容。
SystemPrompt：用于设定大模型的角色、基础指令（如身份界定、安全约束）等核心配置。

那么，System Prompt是如何在AI Agent中生效的呢？

在AI Agent中，System Prompt主要起到静默提示的作用，通常被置于用户输入之前，与Assistant Prompt和User Prompt组合输入到大模型中。

System Prompt与User Prompt的关键区别在于其位置与优先级：System Prompt固定设置在输入文本序列的开端。由于注意力机制的特性（序列首尾信息通常更受关注），该位置的内容更容易被模型识别和遵循。因此，一个完整的多轮对话提示词（Prompts）通常按以下模式拼接：

System Prompt -> User Prompt -> Assistant Prompt -> User Prompt ... -> Assistant Prompt

在此结构中，Assistant Prompt的主要作用是向大模型展示历史对话记录，并明确标注其中哪些内容源于用户的输入。经过这种结构模式数据预训练和微调的大模型能够理解：这些并非即时用户输入，而是对话历史。这有助于大模型更好地把握上下文信息，从而更准确地回应后续问题。

那么，有读者可能会问，为何不将System Prompt与User Prompt合并呢？一个重要考量在于安全性和可控性。通过在微调阶段区分消息类型，有助于防御提示词注入（Prompt Injection）等攻击手段。具体来说：

将核心角色定义和规则置于System Prompt中。
用户交互内容则放在User Prompt里。

上述这种分离机制能有效防范某些简单的提示词攻击或信息泄露风险。特别是在实际应用中，System Prompt对用户通常是不可见的。其定义的规则和角色经过充分训练，因而在模型中享有最高优先级。这显著提高了大模型遵循开发者意图的可能性，降低了因用户输入变化导致输出偏离预期的风险。

当然，仅依赖System Prompt并不能完全抵御攻击（例如，GPT-4 曾出现过System Prompt被诱导泄露的案例）。因此，对用户输入或模型输出进行二次校验，是更为稳妥的安全增强方案。

2.5 通俗易懂讲解AI Agent的上下文工程（Context Engineering）核心基础知识

从AIGC时代初期的“提示工程”概念到目前AI Agent中的“上下文工程”概念，其演变过程反映了AI Agent技术框架的持续发展与成熟。

传统的提示工程主要关注对单一、静态文本指令进行优化，力求在单次交互中获得理想输出。这种方法在处理简单任务时具有一定效果，但在面对需要多轮对话、外部知识整合或工具调用的复杂场景时，其局限性逐渐显现。

在此背景下，上下文工程应运而生，标志着AI Agent的研发重点从“优化单次提示”转向“构建动态信息系统”。它不再将提示视为孤立指令，而是将其纳入一个更广泛的框架中，该系统能够在LLM/AIGC大模型执行任务的每一步动态、智能地组合所有必要信息，并有效填充模型的上下文窗口。这一转变意味着LLM应用开发正逐渐从依赖经验的“手工艺”阶段，迈向注重系统化设计的“工程科学”阶段。

正如Andrej Karpathy所指出，可以将LLM/AIGC大模型理解为一种新型操作系统——LLM/AIGC大模型本身相当于“CPU”，而其上下文窗口则类似于“RAM”，承担着工作记忆的功能。与物理内存的容量限制类似，上下文窗口所能容纳的信息量也有限。因此，“上下文工程”的核心职责，正如操作系统对内存的调度管理，是在有限的上下文窗口内，科学、高效地组织信息，以支持模型完成当前任务。

Karpathy 对此作出了精辟总结：

“在每一个具备工业级规模的LLM应用中，上下文工程都是一门融合了艺术性与科学性的精细技艺，其关键在于为下一步任务精准配置上下文窗口中的信息内容。”

这一演进背后的根本动因在于，当前LLM/AIGC大模型的能力虽持续进步，但仍未达到无需工程辅助即可解决所有复杂问题的水平。因此，上下文工程在现阶段成为必要手段。未来，随着模型能力的进一步跃升，相应的工程范式也可能继续动态演变。

在这里，Rocky归纳一下上下文工程（Context Engineering）的核心概念，其主要指围绕大型语言模型（LLM）构建动态调度系统的能力，该系统的核心目标是在任务执行的适当时机，以结构化的方式为模型提供恰当的信息与工具支持，从而保障任务高效、可靠地完成。

其主要特征可概括为以下四个方面：

系统化构建，而非静态模板：上下文并非简单的文本模板，而是运行于LLM调用之前的软件系统所生成的动态输出，具有明确的程序化结构与调度逻辑。
高度动态，随任务实时生成：上下文内容根据具体任务需求实时构建。例如，处理会议安排时可能引入日历数据，处理客户查询时则可能整合邮件记录或数据库检索结果。
融合知识支持与工具调用：上下文不仅通过检索增强生成（RAG）等技术提供相关知识，还通过封装工具接口扩展模型的可操作能力，形成“信息+能力”的双重支撑。
注重效率与性能平衡：良好的上下文工程力求以尽可能少的Token占用，实现最优的模型输出质量。通过精准投放相关信息，避免因信息冗余或无关内容导致的模型性能衰减与成本上升。

对比总结：提示工程 vs. 上下文工程

特征维度	提示工程 (Prompt Engineering)	上下文工程 (Context Engineering)
主要目标	从模型中引导出特定响应	构建可靠、可扩展、有状态的完整应用系统
核心活动	手工优化静态文本指令	设计动态组合上下文的系统架构
隐喻比喻	如同念咒语或下达一次性指令	类似API函数调用、剧本编排或操作系统内存管理
覆盖范围	最终发送给模型的单条指令	完整处理流水线：指令、状态、检索数据、工具链与输出格式
交互性质	主要为单轮、无状态的问答	天然支持多轮、有状态且持续演进的交互
关键技能	创意写作 + 模型特性理解	系统设计 + 信息检索 + 数据工程 + 持续评估

【上下文工程的设计与管理】

设计原则：

明确性与精确性：指令必须无歧义、具体。避免模糊表述，明确具体操作步骤和条件。
相关性与简洁性：只提供完成任务所必需的信息，避免冗余内容干扰模型注意力并增加成本。
动态性与适应性：根据任务进度动态构建上下文，智能判断何时需要检索知识、调用工具或回顾历史。
结构化与可靠性：尽可能使用结构化格式传递数据和约束输出，节约Token并确保格式稳定。
模块化与关注点分离：将上下文视为独立”模块”，使管理、更新和调试更加清晰高效。

管理策略：

随着任务持续进行，上下文窗口易迅速膨胀，导致多种问题：上下文污染、干扰、混淆和冲突。为应对这些挑战，现有方法可归纳为四类策略：

写入策略：将繁多的上下文内容保存到数据库或文件中，按需写入和读取。
选择策略：借助检索工具，只将相关信息（记忆、工具、外部知识等）拉入上下文窗口，避免过载。
压缩策略：通过上下文摘要和修剪等技术，仅保留完成任务所需的最少token。
隔离策略：将上下文拆分给多个子智能体并行处理，各自拥有独立上下文窗口，探索问题不同方面。

3.1 深入浅出通俗易懂完整讲解Function Calling核心基础原理

Rocky认为Function Calling本质上是LLM/AIGC大模型能够调用外部工具和API的能力。我们在使用大模型的Function Calling能力时，首先需要定义相关的Functions，通过系统提示词告知大模型。当用户输入问题时，大模型首先需要判断是否使用Function，如果需要使用Function，再确定具体使用哪个Function。如果不需要使用Function，则直接生成回答。

如果需要调用Function，大模型会返回一个JSON或者XML格式信息，其中包括了需要调用的Function名称、参数名及对应的参数值等传参信息。

这时候整个Function Calling的流程可以分为：

LLM/AIGC大模型思考： 用户输入问题，AI Agent中的核心LLM/AIGC大模型先对问题进行分析思考。
Function Calling行动： 如果问到了股票涨跌问题，则需要LLM/AIGC大模型分析出需要调用的Function以及要传入的参数。
Function Calling响应：Function运行后返回输出结果，LLM/AIGC大模型再将答案整理成自然语言回复给用户。
流程的循环迭代：除了一次性将输出结果发送给用户，我们还可以设置循环迭代的Function Calling过程，让LLM/AIGC大模型带着回答和问题进行多次思考，多次调用Function，使得输出结果完善。但是多次迭代时，也可能会引入大模型的“幻觉”，从而使得回复内容有所偏差。

可以看到，Function Calling让LLM/AIGC大模型获得了调用外部Function的能力，这个Function可以是一个代码函数，也可以是一个API、数据库、搜索引擎等。

在AIGC时代飞速发展的当下，很多大公司都对Function Calling进行了封装，将这种调用模式集成在LLM/AIGC大模型API接口中，让用户更加方便的进行调用与AIGC应用的构建。

def send_messages(messages):

        messages=messages,

                "type": "object",

                        "type": "string",

                },

            },
        }
    },

print(f"Model>\t {message.content}")

可以看到，上述例子的执行流程如下：

用户输入：询问腾讯股票的涨跌幅情况。
模型第一次推理：返回Function：get_stock_change({stock_name: '腾讯股票'})。
Function调用：调用Function：get_stock_change({stock_name: '腾讯股票'})，并传给模型。
模型第二次整合推理：返回自然语言，"今天腾讯股票上涨了3.2%"。

3.3 LLM/AIGC大模型如何学习到Function Calling能力

通过上面几个小节的学习，我们已经知道了Function Calling的原理和用法。我们接下来更进一步思考，Function Calling能力不是LLM/AIGC大模型在预训练阶段原生具备的，我们该如何让LLM/AIGC大模型学习到Function Calling能力呢？

当前AI业界主流的方法是通过监督微调（Supervised Fine-tuning, SFT）训练来实现LLM/AIGC大模型对Function Calling能力的学习，而不是在预训练阶段从零开始专门训练。当前AI业界具备Function Calling能力的大模型包括DeepSeek系列、GPT-4o系列、Claude系列、Qwen系列、Gemini系列等。

同时在微调训练前，LLM/AIGC基础大模型需要先具备良好的指令遵循和代码/结构化数据生成能力。

Function Calling能力微调训练的核心思想：

获取识别意图（Intent Recognition）能力：理解用户的请求是否需要借助外部工具/函数来完成，而不是直接生成文本回答。
获取参数提取与格式化（Argument Extraction & Formatting）能力：如果需要调用函数，正确地从用户请求中抽取出所需的参数，并按照预先定义的格式（JSON、XML等）生成函数调用的指令。

接下来，我们再梳理一下Function Calling的微调全过程：

1. 训练数据集制作：可以说数据集整理是整个微调过程最重要的一步，因为我们需要构建一个包含Function Calling场景的指令微调数据集，让基础大模型能够充分的学习参数与格式化内容。每个数据样本通常包含以下内容：

数据集内容	详细解释
用户输入（Input/Query）	一个用户请求，可以是包含调用函数的内容，也可以是不包含调用函数的内容。比如：“查询今天腾讯股票的涨跌幅？”或者“给我写一首大气磅薄的诗词”。
可用外部函数/工具描述（Available Functions/Tools Description）	一个结构化的描述，告知大模型当前有哪些函数可用，每个函数的用途、所需参数及其类型和描述。这个描述本身通常就是文本，需要设计一种清晰的格式（JSON、XML等）。
期望的输出（Desired Output）	1. 如果需要调用函数：一个特定格式的字符串，通常是包含函数名和提取出的参数的JSON、XML对象。
2. 如果不需要调用函数：大模型直接生成文本回
答。例如：“好的，我写了一首大气磅薄的诗词：...”
数据集整体质量要求	1. 数据多样性：需要足够多、覆盖各种场景（需要/不需要调用Function、调用不同Function、Function参数变化、模糊表达等）的高质量数据。
2. 函数描述的清晰度：函数描述的质量直接影响模型能否正确理解和使用函数。
3. 负样本：需要包含足够多明确不需要调用Function的样本，防止模型“过度触发”Function调用。

Agent2Agent（A2A）协议关注的是如何让多个自主的AI Agent有效地一起工作，使得多个AI Agent/子Agent之间可以协同完成任务，而无需直接共享它们的内部记忆、思考或工具。

A2A协议的本质是定义了一套共享的“语言”和“行为准则”，使得不同来源、不同能力的AI Agent能够相互理解、相互协作，共同完成一个单一AI Agent无法完成的宏大目标：

通信语言：如FIPA ACL，定义了消息的格式、含义（如请求、提议、同意、拒绝）。
协商与协调：AI Agent通过对话来分配任务、解决冲突。例如，采用合同网协议：一个管理者智能体“招标”，其他智能体“投标”，最终管理者选择最合适的投标者。
角色与组织：预先定义AI Agent在组织中的角色（如CEO、CTO、算法专家、开发程序员、设计师等）及其职责和权限。
共享记忆与状态管理：维护一个共享的“黑板”或记忆流，记录团队当前的共识、决策和已完成的工作，确保所有成员上下文一致。

Model Context Protocol（MCP）协议：就像科技公司为所有员工统一采购和配置的办公软件套装和数据库访问权限。它确保每个员工都能安全、规范地使用计算器、访问公司内部wiki、操作CRM系统。其核心是智能体与工具/资源之间的纵向连接。

5.1 RAG核心基础知识初探

RAG（检索增强生成，Retrieval Augmented Generation） 已成为当前构建大语言模型（LLM）应用的主流方案。

其核心理念清晰：首先从自有垂直领域数据库中检索相关信息，随后将这些信息整合进提示模板，交由大模型进行润色和最终回答生成。

然而，将基础大模型直接应用于实际业务场景时，往往会面临通用模型能力不足的挑战，主要原因包括：

知识局限性： 主流大模型（如 DeepSeek、文心一言、通义千问等）的知识完全来源于其训练数据，而这些数据主要基于公开网络信息。因此，模型天然缺乏对实时更新、非公开或私有领域数据的认知。
幻觉问题： 所有深度学习模型，包括大模型，其底层原理均基于数学概率，输出本质上是数值运算的结果。这导致模型在缺乏特定领域知识或不擅长的任务场景下，容易出现看似合理实则错误的输出（即“一本正经地胡说八道”）。
数据安全性： 企业级应用对数据安全要求极高，尤其是大型企业，几乎无法接受将敏感私有数据上传至第三方平台进行训练或推理的风险。这使得完全依赖通用大模型自身能力的应用方案，必须在数据安全和应用效果之间艰难权衡。

RAG 正是解决上述关键痛点的有效途径。

核心定义：

RAG（Retrieval-Augmented Generation，检索增强生成） = 检索技术 + LLM 提示工程。
例如，当用户向 LLM 提出一个问题时，RAG 会先从多种数据源检索相关知识片段，然后将检索到的信息和原始问题一同整合进提示词（Prompt）中，最终由 LLM 生成答案。

基于 RAG 架构的产品形态多样，涵盖从整合网络搜索引擎的问答服务到利用私有数据的专业应用程序。

技术背景： 早在 2019 年，Faiss 库就实现了高效的向量嵌入搜索技术。如今，RAG 的蓬勃发展显著推动了向量搜索领域的进步。后续涌现的向量数据库初创公司（如 Chroma、Weaviate 和 Pinecone 等），其核心大多基于开源搜索索引引擎（如 Faiss、NMSLIB），并在其基础上增加了文本存储、元数据管理及其他功能组件。

RAG 优势：

提高准确性：通过引用外部知识库，减少幻觉现象。
实时更新：允许模型访问最新信息，解决知识局限性问题。
经济高效：与重新训练模型相比，降低了更新成本。

RAG 执行流程通常分为以下两个阶段：

1.生成索引阶段：这个阶段通常包括文档清理，以及分词也就是将它们拆分为更小的数据块，并把这些数据库转换成向量，最后存储在向量数据库的过程。

2. 检索阶段：使用用户提出的问题，在向量数据库中检索查询的过程。

5.5 RAG高级组件详解

【Embeddings重新排名和过滤】

在我们进行初步检索后，一般会具有一些局限性。比如有时向量检索基于语义相似度，检索到的段落可能与查询在主题上高度相关，但未必直接包含问题的答案。有时基于关键词的检索（如BM25）可能被文档中频繁出现的次要关键词干扰，返回并不匹配用户意图的结果。有时初步检索为了不遗漏任何可能相关的信息，通常会设置一个较大的top-K（例如，检索20个片段）。这保证了高召回率，但其中必然混杂着大量不精准或冗余的信息，直接塞给LLM会带来噪声和干扰。同时LLM大模型的上下文长度是有限的，我们无法将检索到的所有文档都喂给它。必须在有限的额度内，提供价值最高的信息。

所以我们在使用RAG获得初步检索结果后，通常需要进一步优化，具体方法包括过滤、重新排序或对结果进行某种形式的转换。比如LlamaIndex库中提供了丰富的后处理机制，能够依据多种条件——如相似度分数、关键词匹配、元数据属性等——对检索结果进行筛选。

此外，我们还可以通过Rerank模型来对初步检索的结果重新排列。

在RAG流程中，Rerank（重排序）模型是一个位于“初步检索”和“最终生成”之间的关键后处理模块。它的核心使命是对初步检索到的大量相关文档进行精细化的排序和筛选，将最相关、最高质量的上下文信息提供给大语言模型（LLM），以生成更准确、可靠的答案。我们可以将其理解为一个“智能过滤器”或“精排专家”。

Rerank模型的核心工作流程主要有以下几步：

1. 输入：接收来自初步检索器的“查询-文档对”列表。例如，一个用户查询 `Q` 和初步检索到的Top 20个文档 `[D1, D2, D3, ..., D20]`。

2. 精细化评分：Rerank模型会为每一个`(Q, Dn)`对进行更深层次的交互式计算和相关性评估。它不再仅仅依靠简单的语义嵌入相似度或词频统计，而是分析查询和文档之间更复杂的语义关系。

3. 重新排序：根据计算出的新相关性分数，对原始的文档列表进行降序重排。原本排名第15的文档，如果经模型判断为极度相关，可能会被提升到第1位。

4. 输出：输出一个经过重新排序且通常经过截断（Top-N）的高质量文档列表，将其作为最优质的上下文传递给LLM进行答案生成。

当前一个常见的模式是：“用便宜的向量检索器广撒网（高召回率），再用精准的Rerank模型精捕捞（高精度）”。

总的来说，这一环节处于将检索得到的上下文信息输入至LLM/AIGC大模型生成最终答案之前，是RAG的pipeline质量的关键步骤。

在AIGC时代的主流AI Agent系统中，Memory（记忆）机制的核心目标是为AI Agent提供统一的内存系统，让AI Agent能够存储、检索和利用在过去交互过程中获取的高价值信息，来解决AI Agent在长程交互、多任务执行、多Agent协作场景下的内存管理问题，从而实现跨用户、跨任务、跨Agent的记忆信息提取、复用与共享，最终提升AI Agent系统的上下文处理效率、任务执行准确性和长期学习能力。

我们知道目前AI Agent技术在飞速迭代和更新，不过在日新月异的Memory机制技术中，我们依旧能够提炼出跨周期的通识技术认知，那就是Memory机制能够分为短期记忆和长期记忆两大类：

AI Agent Memory = Long-Term Memory（LTM） + Short-Term Memory（STM）

没有Memory机制的AI Agent系统，就像金鱼一样，只拥有“7秒的记忆”，而拥有Memory机制的AI Agent则可以获得以下的能力：

记住用户偏好：例如，用户在历史对话中说喜欢科幻小说，下次推荐书籍时就可以优先考虑这个类型。
维持对话上下文连贯：例如，进行多轮对话时，能够理解一些名词的指代（如“他”、“她”、“它”、“那个”指的是什么）。
从历史对话中学习经验：例如，在交互中总结过去的成功或失败经验，优化未来的决策和行动。
持续积累先验知识：例如，将新获取的信息存入知识库，供AI Agent长期使用。
6.1 AI Agent Memory机制的生动初识

Rocky在本章节中通过一个生动的例子，向AI Agent的初学者们趣味讲解AI Agent Memory机制的概念。

我们也可以把AI Agent想象成一个刚刚毕业、超级聪明但患有严重“瞬间失忆症”的实习生。

这个实习生有个特点：

1. 聪明绝顶：我们问他任何问题，他都能基于已有的知识（训练数据）瞬间给出一个非常专业、高质量的回答。

2. 但瞬间失忆：只要我们关闭当前的聊天窗口，下一次打开一个新的窗口问他时，他完全不记得之前和我们聊过任何内容。他每一次都是“从零开始”地和我们对话。

上述这就是没有记忆机制的LLM/AIGC大模型。

它很强，但它的每次对话都是一个独立的“瞬间”，没有连续性，没有上下文关联。这显然无法完成复杂的、需要上下文的任务。

那么，记忆机制是什么呢？

就是给这个聪明的实习生配一个超级助理 + 一个无限空间的档案库。这个助理负责做三件事：

1. 做笔记（短期记忆）：在和我们的一次对话过程中，助理会把我们聊的所有内容都记在笔记本上。这样，当实习生说到后面忘了前面时，助理可以马上把笔记给他看，让他保持对话的连贯性。这个笔记本就是“短期记忆”或“上下文窗口”。但笔记本篇幅有限，记满了就得翻页，可能会忘掉最开始的内容。

2. 写纪要并归档（长期记忆）：一次漫长的对话结束后，助理不会把整整100页的聊天记录都塞进档案库，那样太乱了。相反，助理会提炼这次对话的“核心信息”和“关键结论”，比如：“客户张三，喜欢喝拿铁咖啡，对价格敏感，上次报价A方案未成功”。然后把这个精简的纪要存进档案库。这个提炼和归档的过程就是“长期记忆”的形成。

3. 适时提醒（记忆检索）：当我们再次和实习生聊天，提到“上次那个客户张三”时，助理会立刻心领神会，飞奔到档案库，把所有关于“张三”的纪要档案找出来，递给实习生。实习生看了档案后，就能好像从未失忆一样和我们无缝衔接地聊下去。这个“找档案”的动作就是“记忆检索”。

这个记忆机制（超级助理+档案库）解决了核心问题：让AI Agent拥有了连续性和个性，从而变得真正“智能”和“有用”。

其具体作用体现在四个方面：

1. 保持连续性，完成复杂任务

如果没有记忆机制：我们让AI Agent“帮我想个公司名”，它想了一个。你再说“要听起来科技感强一点”，它可能就懵了，因为它不记得上一句它自己说了什么。我们只能每次都把全部要求重复一遍。

如果有记忆机制：AI Agent会记得它刚才提议的名字，并能基于我们的反馈“科技感强一点”进行优化。理论上它可以陪我们反复讨论一整天，就像一个真正的合作伙伴。

2. 学习用户偏好，提供个性化服务

如果没有记忆机制：每次我们都说“我不吃香菜”，但它每次都会推荐有香菜的菜。因为它记不住。

如果有记忆机制：AI Agent会把我们“不吃香菜”这个信息写入长期记忆档案。下次我们让它推荐菜谱时，它会自动过滤所有含香菜的菜品，直接给我们个性化推荐。

3. 积累知识，自我演进

如果没有记忆机制：AI Agent无法从过去的错误中学习。它这次犯的错，下次还会再犯。

如果有记忆机制：AI Agent可以把“用户纠正了我的一个错误：XXX事件的年份其实是1991年，不是1992年”这个信息存档。下次再被问到相关问题时，它就会检索到这个纠正后的正确信息来回答，从而实现“成长”。

4. 构建身份和上下文

如果没有记忆机制：AI Agent无法扮演一个持续的角色。比如，它今天可以是我们严厉的编程老师，但明天它就不记得这个设定了。

如果有记忆机制：AI Agent可以把“用户希望我扮演一个严厉的编程老师，并且上次指出了他代码中的缩进错误”这个设定存档。下次互动时，它会首先加载这个“严厉老师”的角色设定和之前的教学进度，让体验非常真实和连贯。

组件	比喻	技术原理（简单版）
AIGC/LLM模型（如GPT）	聪明的失忆实习生	基于海量数据训练的大型神经网络，能力强大但无状态（Stateless）
短期记忆	助理的笔记本	上下文窗口（Context Window）。将本次对话的历史直接传递给模型。窗口大小有限（比如128K tokens），太长会遗忘开头或浪费算力
长期记忆	档案库里的纪要	向量数据库（Vector Database）。将历史对话的核心信息转换成数学向量（一串数字）并存储。需要时通过相似度搜索快速检索
记忆检索	助理跑去档案库找资料	检索增强生成（RAG）。根据用户当前的问题，去向量数据库里找到最相关的记忆，和问题一起送给模型，辅助它生成回答




6.2 AI Agent中Memory机制原理详解

当前在AI Agent中增加记忆机制有短期记忆、长期记忆以及工具记忆三种模式。

短期记忆 / 对话记忆(Short-term / Conversation Memory)主要用于维护当前对话/任务的上下文。通常有窗口限制（只记住最近N轮对话，有底层核心AIGC/LLM大模型上下文窗口决定）。
长期记忆(Long-term Memory)主要在较长时间跨度内存储和回忆重要信息。容量远大于短期记忆，需要通过检索来获取相关信息。长期记忆机制通常与向量数据库(Vector Database)结合，将信息转换为向量嵌入(Embeddings)后进行存储和相似性检索。
工具记忆 (Tool Memory)：记录AI Agent调用工具（如API、函数）的历史和结果。这对于需要多步执行复杂任务的Agent至关重要，它可以回顾之前的工具调用结果来决定下一步动作。

Memory机制的核心可以简化为两个基本操作：读取（Read） 和 写入（Write）。

写入（Write）操作在AI Agent完成一次动作（如回复用户、调用工具）后。将有价值的交互信息（如用户输入、AI输出、工具执行结果、自主总结的要点）持久化到存储中（内存、数据库、文件等）。
读取（Retrieve / Read）操作在AI Agent处理新的输入或开始新任务之前。根据当前的查询（Query）（如用户的新问题），从记忆存储中查找最相关的信息片段。对于长期记忆，通常使用向量相似性检索。将查询也转换为向量，然后从向量数据库中找出与之最相似的记忆向量。
6.3 AI Agent Memory机制通用代码案例解读

下面我们用一个深入浅出的代码示例来演示这个流程。我们将模拟一个具有短期对话记忆和长期知识记忆（使用向量数据库）的AI Agent：

import numpy as np
from typing import List, Dict, Any

# llm：一个大型语言模型的客户端。
# vector_db：一个向量数据库客户端，支持存储和查询向量。
# embedding_model：一个文本嵌入模型，用于将文本转换为向量。


# 伪代码：初始化阶段，创建模拟的嵌入模型、向量数据库和LLM大模型核心。
class EmbeddingModel:
    def generate_embedding(self, text: str) -> List[float]:
        """生成文本的向量嵌入"""
        # 实际应用中，这里会调用OpenAI、SentenceTransformers等的Embedding API
        return np.random.rand(512).tolist() # 返回一个随机向量模拟

class VectorDB:
    def __init__(self):
        self.memories = [] # 存储记忆条目
        self.embeddings = [] # 存储对应的向量

    def add_memory(self, text: str, embedding: List[float], metadata: Dict[str, Any]):
        """向长期记忆库中添加一条记忆"""
        self.memories.append({"text": text, "metadata": metadata})
        self.embeddings.append(embedding)

    def query_similar(self, query_embedding: List[float], top_k: int = 3) -> List[Dict]:
        """查询最相似的记忆"""
        # 实际应用中，这里会使用余弦相似度等计算
        # 这里简单返回前top_k条记忆作为模拟
        return self.memories[:top_k]

class DeepSeek-LLM:
    def generate(self, prompt: str) -> str:
        """LLM生成回复"""
        # 实际应用中，这里会调用GPT、Claude等模型
        return f"Based on the context: {prompt[-100:]}... I think the answer is 42." # 模拟回复

# 初始化组件
embedder = EmbeddingModel()
long_term_memory = VectorDB()
llm = DeepSeek-LLM()

# 短期记忆 - 简单用一个列表存储最近的对话记录
short_term_memory = []
SHORT_TERM_MEMORY_SIZE = 3 # 只保留最近3轮对话

# converse_with_agent 函数：这是核心函数，处理每一轮对话。 
def converse_with_agent(user_input: str):
    """
    与AI Agent进行一轮对话的核心函数
    """
    global short_term_memory

    # 1. 【MEMORY READ】检索记忆
    # 1.1 从长期记忆中检索相关记忆
    # long_term_memory.query_similar(...): 将用户的新输入转换为向量，并从长期记忆库中查找最相关的记忆。
    query_embedding = embedder.generate_embedding(user_input)
    relevant_long_term_memories = long_term_memory.query_similar(query_embedding, top_k=2)
    long_term_context = "\n".join([mem["text"] for mem in relevant_long_term_memories])

    # 1.2 从短期记忆中获取最近对话
    short_term_context = "\n".join(short_term_memory)

    # 2. 合成提示词：将检索到的长期记忆、短期记忆和用户新输入组合成一个完整的提示词，提供给LLM。这为LLM提供了丰富的上下文。
    full_prompt = f"""
    [系统指令：你是一个有帮助的AI助手。]

    [以下是来自你长期记忆的相关信息：]
    {long_term_context}

    [以下是最近的对话记录：]
    {short_term_context}

    [用户的最新消息：]
    {user_input}

    [请根据以上上下文进行回复：]
    """
    print(f"Debug - Full Prompt sent to LLM:\n{full_prompt}\n---")

    # 3. LLM生成回复
    ai_response = llm.generate(full_prompt)

    # 4. 【MEMORY WRITE】写入记忆
    # 4.1 将本轮交互加入短期记忆
    short_term_memory.append(f"User: {user_input}")
    short_term_memory.append(f"AI: {ai_response}")
    # 保持短期记忆的长度
    short_term_memory = short_term_memory[-SHORT_TERM_MEMORY_SIZE:]

    # 4.2 （可选）判断是否将重要信息写入长期记忆
    # 这里用一个简单的规则：如果用户提到了“记住”这个词，则写入。在实际应用中，这个“重要性判断”可以由另一个LLM调用、调用Tools工具等方式来完成，更加智能。
    if "记住" in user_input:
        # 生成要记忆的内容的嵌入向量
        memory_embedding = embedder.generate_embedding(ai_response)
        metadata = {"type": "user_preference", "source": user_input}
        long_term_memory.add_memory(ai_response, memory_embedding, metadata)
        print(f"Debug: Added to long-term memory: {ai_response}")

    return ai_response

# ------------------- 模拟对话 -------------------
if __name__ == "__main__":
    # 第一轮
    print("User: 你好，我叫Rocky Ding。")
    response1 = converse_with_agent("你好，我叫Rocky Ding。")
    print(f"AI: {response1}\n")

    # 第二轮 - 测试短期记忆
    print("User: 我最喜欢的水果是芒果。")
    response2 = converse_with_agent("我最喜欢的水果是芒果。")
    print(f"AI: {response2}\n")

    # 第三轮 - 测试短期记忆和指令写入长期记忆
    print("User: 请记住，我对芒果过敏。")
    response3 = converse_with_agent("请记住，我对芒果过敏。")
    print(f"AI: {response3}\n")

    # 第四轮 - 测试长期记忆的检索
    print("User: 我可以吃点什么水果？")
    response4 = converse_with_agent("我可以吃点什么水果？")
    print(f"AI: {response4}\n")

上述例子是Rocky构建的一个基础经典的AI Agent Memory机制例子，在实际的主流AI Agent框架中，在此基础上会有更多工程优化与策略优化，大家在学习好Rocky的例子后，可以进行更多的研究与探索。

总的来说，AI Agent的记忆机制，本质上是一套外挂系统。它通过记录、提炼、存储、检索历史信息，来弥补AI模型本身“没有记忆”的缺陷，从而让它能够像我们人类一样进行连续的、个性化的、不断进步的智能交互。

7. AI Agent中Skills的核心基础知识
7.1 什么是AI Agent的Skills技术？

Skills（技能）是AI Agent最核心的”动手能力”——它是将大语言模型（LLM）的文本推理能力，转化为可执行、可复用、可组合的具体行动单元的技术。

Rocky打个通俗的比方：

大模型是AI Agent的”大脑”，负责思考、理解和决策
Skills是AI Agent的”双手+工具箱”，负责把大脑的想法变成实际结果
没有Skills的AI只能”纸上谈兵”（给建议），有了Skills的AI才能”真刀真枪干活”（直接完成任务）

目前主流的AI Agent系统中，所有”能帮我们做事”的功能（写代码、跑命令、构建项目、生成PPT等），本质上都是它内置或扩展的Skills在发挥作用。

7.2 Skills技术的5大核心原理

【1. 技能的抽象与表示：把”事”打包成”能力模块”】

Skills技术的第一步，是将人类能做的具体工作，抽象成大模型能理解和调用的标准化能力单元。

比如我们以Claude Code为例，其核心技能抽象：

技能类别	具体能力	底层实现
本地操作	读取/编辑/创建文件、运行终端命令、Git提交	直接调用操作系统API和本地工具
编码能力	代码生成、调试、重构、Lint修复	大模型代码理解+文件操作技能组合
项目导航	分析项目结构、查找代码引用、解释模块功能	全项目上下文索引+语义搜索
外部连接	读取Google Drive、更新Jira、发Slack消息	MCP（模型上下文协议）统一接口

每个Skill的标准结构都包含3个核心部分：

意图描述：用自然语言说明这个技能能做什么（LLM/AIGC大模型靠它识别什么时候该用这个技能）
输入输出规范：明确需要什么参数、返回什么结果（比如”运行命令”技能需要输入命令字符串，返回输出和退出码）
执行逻辑：实际完成任务的代码或接口（比如调用os.system执行命令，调用Git API提交代码）

【2. 意图理解→任务分解：把自然语言变成技能执行计划】

这是大模型在Skills技术中最核心的作用：将用户的模糊自然语言请求，拆解为有序的、可执行的技能调用序列。

Claude Code的任务分解流程：

用户输入：”帮我给这个用户登录接口添加手机号验证码功能”
大模型理解意图：需要在现有登录接口基础上，新增验证码生成、验证、错误处理逻辑
分解为技能调用序列：
技能1：读取现有登录接口的代码文件
技能2：分析现有代码结构，找到需要修改的位置
技能3：生成验证码生成和验证的代码
技能4：修改登录接口，集成验证码逻辑
技能5：运行单元测试，验证功能是否正常
技能6：如果测试失败，分析错误并修复代码
技能7：提交代码变更，生成提交信息

关键特点：任务分解不是一次性的，而是动态迭代的——大模型会根据前一个技能的执行结果，调整后续的技能调用计划。

【3. 闭环执行与反馈：不是一次性调用，是持续迭代】

这是Skills技术和传统”工具调用”最本质的区别：传统工具调用是”大模型发指令→工具执行→返回结果”的单次交互，而Skills是“执行→反馈→调整→再执行”的闭环过程。

Claude Code的闭环执行示例（调试bug）：

用户粘贴错误信息：”登录接口返回500错误，日志显示NullReferenceException”
技能调用：读取登录接口代码→运行测试复现错误
执行结果：测试失败，错误指向第45行user.email为空
大模型分析：用户可能未传入email参数，需要添加空值检查
技能调用：修改代码添加空值检查→重新运行测试
执行结果：测试通过
技能调用：提交修复代码

【4. 技能组合：像搭积木一样完成复杂任务】

Claude Code严格遵循Unix哲学：”做一件事，并且做好它”。每个Skill都是单一职责的小模块，可以像管道一样组合起来，完成极其复杂的任务。

文档中的两个经典组合示例：

实时日志监控告警：
tail -f app.log | claude -p "如果看到任何异常，发Slack消息通知我"
技能组合：实时日志输出 → 日志异常分析 → Slack消息发送
CI自动化翻译：
claude -p "如果有新的文本字符串，翻译成法语并提交PR给@lang-fr-team审核"
技能组合：代码变更检测 → 文本提取 → 机器翻译 → PR创建 → 团队通知

可脚本化特性：你可以把Claude Code的技能调用写成Shell脚本或Python脚本，集成到你的CI/CD流水线中，实现完全自动化的开发流程。

【5. 技能扩展：通过MCP连接无限可能】

当内置技能不够用时，Claude Code通过MCP（模型上下文协议） 提供了统一的技能扩展机制，让我们可以连接任何外部工具和服务，甚至编写自己的自定义技能。

8. AI Agent中Harness Engineering的核心基础知识

AIGC时代技术迭代迅猛，在AI Agent领域，系统提示词工程侧已经从 prompt enginering -> context engineering， 进化到 harness engineering 了。

但是至今可能还有还有很多人陷在一个死循环里：AI Agent不好用，一定是核心大模型不够强。换GPT-4o不行就上Claude Opus，闭源不行就换开源70B参数大模型，总觉得只要核心LLM大模型脑子够好使，什么活都能干。

但现实狠狠打了所有人的脸：一模一样的GPT-4o、一模一样的Claude，有人能用它5个月零手写敲出100万行代码，有人连让它稳定跑2小时不翻车都做不到。

模型完全一样，结果天差地别。问题到底出在哪？

2026年AI Agent的竞争，可能早就不是单纯模型卷参数、卷聪明度的战争了。真正的胜负手，叫Harness Engineering——说白了，就是给我们的AI Agent搭一套专属「操作系统」。

先给所有人砸一个核心公式，记住这句话，我们对AI Agent的理解直接超过行业80%的人：

Agent = Model + Harness
8.1 Harness Engineering到底是什么？

总的来说，我们可以把一套AI Agent系统，完完全全想象成我们手里的电脑：

大模型 = CPU，只负责提供原始算力，是那个脑子好使的核心。
上下文窗口 = 内存，只能临时存点东西，一关程序就全没了。
AI Agent = 我们电脑里的APP，不管是写代码、做报表还是订机票，是真正要落地干活的。
那Windows、macOS是什么？就是Harness，这套AI Agent的操作系统。

就好比没有操作系统，我们手里哪怕有一块世界顶级的i9芯片，我们也不能直接对着硅片敲PPT、做表格。同样的道理，没有Harness，我们的模型哪怕再聪明，也只是个只能单次对话的聊天框。我们让它连续干8小时的复杂活，它中途忘了上下文怎么办？它写了一堆有bug的垃圾代码谁来拦？它犯了低级错误自己都意识不到怎么办？

这些问题，没有一个是「换一个更聪明的大模型」能解决的。

这就是为什么2026年，Harness成了AI Agent的胜负手：2024年大家还在卷谁的大模型做题更厉害，到了今天，顶级闭源模型之间的智商差距已经微乎其微，同一道题，GPT和Claude的得分差不了几个点。

但我们让它们连续一周、每天8小时落地干活，差距立刻就拉开了。这个差距，从来不在模型本身，而在模型外面那圈Harness。

OpenAI的Codex团队有个数据：他们用Codex做完整的产品，5个月写了100万行代码，零行手写。整个过程里他们发现，瓶颈早就不是「大模型能不能写代码」，而是「人类能不能赶得上审查代码的速度」。

大模型的产出速度，已经远超人类的把控能力了。这时候我们的重点是质量把控、流程约束、风险兜底、经验沉淀——而这些，全都是Harness要干的事。

8.2 Harness的三根核心支柱

总的来说，Harness的核心永远是三件事：

评估闭环——AI Agent不能自己给自己打分
架构约束——约束越多，反而越可靠
记忆治理——能持续进化的Agent，才真正有壁垒

【第一根支柱：评估闭环——Agent不能自己给自己打分】

这是Anthropic最核心的看家本领，也是所有Harness的底线。

我们举个最生活化的例子：我们让一个实习生写一份项目报告，写完我们让他自己评价写得好不好，他大概率会说“还行吧，该写的都写了”。我们永远不能让做事的人，自己给自己当裁判。

AI Agent也是一模一样的道理。

Anthropic把这套方法叫「评估驱动开发」，说白了就是AI Agent版的TDD（测试驱动开发）：先定义清楚什么叫“做得好”，先把验收标准、测试用例写死，再让AI Agent去干活，干完之后，由一个完全独立的评估器来打分。

这个评估器，不是随便看看内容就给分。它会真的去跑流程、测功能：比如AI Agent写了个预订机票的程序，它就真的用Playwright去点按钮、填表单、跑完整的预订流程，完全按照我们定的标准，一条一条卡。

这里有个特别有意思的案例，我看完直接笑了，又觉得细思极恐：

Anthropic的Opus 4.5在做航班预订测试的时候，直接发现了预订政策里的漏洞，找到了一个比人类给的标准答案更省钱、更合规的解法。结果呢？评估器直接判它「任务失败」。

为什么？因为人类写的评估标准里，只认那一个标准答案。AI Agent聪明到能跳出人类的认知找最优解，反而被死板的评估体系扣了分。

这个故事告诉我们两件事：

今天的大模型，已经聪明到超出绝大多数人的预期了；
90%的情况下，不是大模型不行，而是对应的Harness不行。

还有个数据更直接：CORE-Bench测试里，Opus 4.5初始得分只有42%，后来团队只是修复了评分bug、放宽了不合理的评估限制，什么都没改，模型还是那个模型，得分直接跳到了95%。

【第二根支柱：架构约束——约束越多，反而越可靠】

这是OpenAI Codex团队玩到极致的东西，也是最反直觉的一个点。

还是举个生活化的例子：我们跟新来的开发说“代码要分层，架构要规范，UI逻辑不能写进数据库层”，他点头说好，转头就给我们写出一锅乱炖。

靠嘴说、靠文档写规矩，根本没用。真正有用的，是用系统把规则焊死。

OpenAI的做法简单到粗暴，又有效到离谱：把所有的架构规则，全写进linter和CI流程里。只要他的代码违反了分层规则，系统直接给他打回，连人工review的机会都没有。

他们的代码分层是死的：Types → Config → Service → UI，每一层只能依赖上一层，绝对不能反向依赖。这个规则不是写在开发手册里让人自觉遵守的，是写在程序里，机械执行、零容错的。

更绝的是什么？这些检查规则的linter，本身也是Codex自己生成的。AI Agent自己给自己定规矩，然后自己严格遵守。

Martin Fowler看完OpenAI的实践之后，说了一句话，Rocky建议所有做AI Agent的人都刻在电脑上：

“增加信任和可靠性，需要约束解空间。这意味着放弃一些「生成任何东西」的灵活性。”

约束越多，反而越可靠。这句话听起来反直觉，但数据不会骗人：

LangChain做了一组对照实验，模型完全不换，只优化Harness的架构约束，Terminal Bench 2.0的通过率直接从52.8%跳到了66.5%；
Vercel更狠，直接把Agent的工具删掉了80%，结果任务步骤更少、执行速度更快、最终效果反而更好。

“工具越多越厉害”，这是很多人对AI Agent最大的误解。就像我们给一个实习生开了全公司的系统权限，他不仅干不好活，还大概率给我们捅出大篓子。真正高效的执行，永远是在明确的边界里，做精准的动作。

【第三根支柱：记忆治理——能持续进化的Agent，才真正有壁垒】

这根支柱，聊的人最少，但长期来看，它是决定AI Agent天花板的核心。

首先我们问一个问题：多个AI Agent共享一个知识库，AI Agent A往里面写了一条错误的经验，AI Agent B读到了，直接当成真理用了，会发生什么？

答案是：一个AI Agent的幻觉，会通过共享知识库，污染整个系统里的所有AI Agent。

这就像公司里，一个老员工把错误的工作方法写进了团队知识库，后面进来的新人全学了这套错的，整个团队的产出只会越来越歪。

那怎么解决？PrismerCloud的做法，给整个行业打了个样：他们做了一套「进化引擎」，Agent的每一次经验，先记录为「信号」，信号经过反复验证、确认有效之后，才能提炼为「基因」，只有「基因」，才能进入正式的知识库，被所有AI Agent调用。

简单说：没被验证过的，都只是道听途说；只有被实践证明有效的，才叫知识。

这里有个数据，直接把卷了好几年的prompt engineering按在地上摩擦：

3行基础prompt，加上一套完善的记忆治理系统，效果约等于200行精心打磨的专家prompt。而且前者会越跑越好、持续进化，后者写完的那一刻，就已经固定死了上限。

这意味着什么？意味着未来我们根本不需要熬几个通宵，去抠prompt里的每一个词、写几百行的指令。只要我们的记忆系统做得好，AI Agent会自己在干活的过程中，沉淀经验、修正错误、持续进化。

8.3 AI Agent掉的链子，本质都是Harness的漏洞

除了上述讲到的这三根支柱，还有一个绕不开的话题，叫「熵对抗」。

什么意思？Agent系统跑久了，一定会自然腐化：文档过期了、架构规则被绕过了、知识库里堆了一堆过时的垃圾信息。就像一个公司运转久了，流程会慢慢变形、制度会慢慢松懈、部门墙会越来越厚。

怎么对抗这种腐化？OpenAI的做法，给了所有人标准答案：他们定期跑一个专门的「重构Agent」，只干一件事——扫描整个系统里的文档不一致、架构违规、流程漏洞。

OpenAI有一句话，Rocky认为是整个Harness工程的核心心法：

“当AI Agent遇到困难时，我们把它当作信号：找出缺什么，然后反馈到代码库中，始终让Codex自己写修复。”

AI Agent出了问题，不要去修AI Agent，要去修Harness。

这次它忘了上下文，我们就给它补上下文管理的模块；这次它写了bug，我们就给它加自动校验的规则；这次它用错了工具，我们就给它收紧工具调用的约束。

每一次翻车，都是完善我们这套操作系统的机会。我们的Harness越完善，AI Agent的下限就越高，就越难翻车。

Anthropic的Claude Code + Agent SDK，是目前通用Harness的标杆，核心就是那套评估驱动的方法论，他们已经用这套体系做深度研究、视频创作、全流程开发；OpenAI的Codex，是架构约束的极致实践，百万行零手写代码的“神话”，全靠这套体系撑着。

Rich Sutton写过一篇经典的论文，叫《苦涩的教训》，核心意思是：长期来看，利用计算能力的通用方法，永远会打败人类精心设计的特定方法。

这个教训，在AI Agent领域，正在再一次应验。

Manus团队6个月里，把自己的Harness重构了5次，同样的模型，5种架构，每一次重写，效果都有质的飞跃；LangChain一年里，核心架构重新设计了3次；Vercel直接删掉了Agent 80%的工具，反而迎来了效果的暴涨。

他们都在做同一件事：Build to Delete，为删除而构建。

大家今天熬了好几个通宵写的「聪明prompt」、设计的精巧逻辑，可能明天模型一升级，就完全没用了。真正能沉淀下来、形成壁垒的，从来不是这些一次性的东西，而是我们的Harness，是这套能持续进化、能沉淀每一次成功与失败的操作系统。

Phil Schmid有一句话，Rocky觉得是2026年AI Agent领域最值得记住的一句话：

竞争优势不再是prompt，而是我们的Harness捕获的轨迹。每次AI Agent的成功和失败，都是训练下一代的数据。

我们的Harness跑得越久，积累的轨迹越多，我们的AI Agent就越强。这个壁垒，不是靠换一个最新的模型就能追上的。最后总结三句话，帮大家把整个逻辑彻底串起来：

Prompt Engineering，解决的是「说什么」，管的是单次交互的效果；
Context Engineering，解决的是「知道什么」，管的是给模型足够的信息支撑；
Harness Engineering，解决的是「怎么持续、稳定、大规模地干活」，管的是Agent的下限、上限和长期壁垒。
