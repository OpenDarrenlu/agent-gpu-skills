1. MatMul.py 中代码解读

经过阅读代码我认为最主要的是理解其中的 swizzle，MatMul.py 这个例子中演示了如何将矩阵乘法任务映射到 thor 上，其中最重要的思路便是将矩阵 C 切分成多个子 tile，将子 tile 的运算任务直接分配给一个 SM，然后由一个 SM 沿着 K 方向累加计算，最后完成一个 C tile 的计算将数据写回到 DDR 中(实际上就是 K stay)。

在将子 tile 的运算任务分配给不同的 SM 的过程，需要考虑 L2 Cache，为此 MatMul 使用了一种 block swizzle 的方式将计算任务分配给不同的 SM 中，如下图所示，在 MatMul.py 中并不会沿着 M 方向完全分割整个 C 矩阵，而是将只在 M 方向分割一部分随后就会沿着 N 进行分割。

这么分割的好处实际上是为了能够更好的利用 L2 Cache 对数据进行缓存，在使用 swizzle 后，当计算任务不断的进行实际上形成的效果如下图所示(这是一个效果示意图)。使用 swizzle 的 MatMul 实际上控制了矩阵 A 参与计算的规模，保证在不断的循环计算的过程中可以利用缓存在 L2 Cache 中的已经缓存的矩阵 A 数据，增加矩阵 A 的命中率。

2. 进一步实验发现问题

有了上面的基础后我就尝试着改变矩阵 M 和 N，并对 N 方向也进行 swizzle，最终绘制了下面两个实验的结果图，折线图的纵轴是相对于理想计算读取归一化后的读取数据量，对于矩阵计算来说最理想的情况应该是矩阵 A 和矩阵 B 都只读取一次，也就对应于纵轴的 1(也就是说假设 Cache 足够大，原本两个实验应该都是 1)。下面的实验显示了不管对 M 方向进行 swizzle，还是对 N 方向进行 swizzle，曲线的效果基本相同，这个基本符合预期。但是出现了两个比较奇怪的问题，一个是 tile 从 128 变成 256 之后出现了两个很明显的平台，第二个就是这两个实验从 1（矩阵 A，矩阵 B 只读一次）的拐点实际读取数据量不符合预期。本文没有研究清楚第一个问题，下面会着重讨论一下第二个问题。（为了能够成 tgen5 的指令切分 tie 的 k 不应太小）

设置tile为256x256之后，同步改变矩阵M和N的实验。
设置tile为128x128之后，同步改变矩阵M和N的实验

在对于矩阵计算进行分析后应该可以得出一个结论，就是在片上缓存刚好能够缓存矩阵 A 和矩阵 B 的数据时，实际上只需要读取一次数据即可。实际上可以更加优化，如果能够完整缓存 A 或 B 中的一个矩阵就可以实现读取一次数据，完成所有的计算任务，但是 Cache 目前还没有发现能够直接控制数据驻留的方法，已经实验了一些方法，还是没有办法精确控制 thor 的 Cache，因此将条件放宽到刚好能够缓存矩阵 A 和矩阵 B 的数据。但是在上面的实验中发生从 1.0 向上升高的拐点理论矩阵 A 和矩阵 B 的拐点发生在理论矩阵 A 和 B 读取量总共只有 10MB 的情况，和 L2 32MB 相差过多。

3. 对于拐点不符合预期的进一步实验

经过分析，认为可能是写回数据占用了 L2 的 Cache 内存，造成将矩阵 A 和矩阵 B 数据替换出去的问题，为此进行了进一步的实验，改变 K 同时测试从归一化的比例 1.0 出现拐点的位置在哪里。可以看到随着 K 的增大，出现拐点的理论读取量越来越接近 L2 Cache 的实际大小(32MB)。

改变K后的拐点出现位置以及理论读取量

总体来说实验时符合预期的因为随着 K 的增大，实际写入占用 L2 的比例相比于读数据也会减小，对于 L2 逐出读数据的影响也相对减小，因此 L2 的利用率可以更高，能够达到理论上限 32MB。

4. 结论和遗留问题

目前，可以得出的结论是：在 GPGPU 中进行矩阵计算，为了减少写回对于 Cache 的影响，尽量应该保持不切分 K。

由于还没有方法能够控制 L2 Cache 所以只能使用这种增加 K 的实验来验证写回对于 Cache 的影响，针对这些问题可以看看有没有办法控制 L2 Cache 的缓存策略。（在 GitHub 的工程中尝试对于 Cache 的缓存策略进行控制，但是目前还是没有看到有效。）
还有一个大的问题没有解决，就是控制切分 tile 的大小时为什么会出现，在 256*256 时会出现阶梯，而 128*128 就会很平滑。
除了 swizzle 这种任务切分模式是不是有更好的切分模式，能够增加数据的复用性。
