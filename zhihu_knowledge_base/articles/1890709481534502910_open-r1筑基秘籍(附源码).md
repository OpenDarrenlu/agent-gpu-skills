topic1: 基础模型选择

整个 open-r1 像小鼠走迷宫：

首先你得真有个迷宫（这不是废话，有的开源实现并不是迷宫）
起始状态。然后把小鼠放入迷宫
状态更新。小鼠得能动弹
奖励反馈。走出迷宫算满分、离出口越来越近也算分 whatever

回到 open-r1 模型选择上，对于 Qwen1.5B base 或 coldstart 模型，难以满足条件 2 和 3.

具体现象是，roll 了几个小时也出不来 reward > 0 的情况，面对 math 、code 问题，有大量自我重复。这个问题到 3B 规模缓解了很多。

我们从 Qwen3B coldstart 版开始训练，response length 远超 Instruct 版本（3k vs. 0.7k）。

coldstart 起步 ~3k response

这些从侧面证明了：

小模型想玩 RL，相对于全量 sft 需要更大算力，对没卡的人而言，无疑是雪上加霜
coldstart 能够加速启动
topic2: 奖励函数和更新

DAPO 不愿意折腾 rewards，直接把打分结果配成了 1/-1，并不适用 3B。

7B 或 32B 基础够强。二极管 reward 能让模型找其他合理的 response，而不是胡言乱语
DAPO 面对的 math 任务有标准解，很多现实任务并没有
头条卡多，可以这么浪

回到小鼠走迷宫的设想里，现在资源有限不能随意探索。比较合理的做法是：

rewards 质量上。代码实现更鲁棒一点，“写过程也算分”；每个函数返回空间在 [-0.5, 1] 之间，正负都要给
rewards 数量尽可能少。减少函数间的语义交叉
rewards weight 配比中，偏向结果和内容过程，而非形式

修改后的实现如下（源码在 https://github.com/tpoisonooo/open-r1/blob/main/src/open_r1/rewards.py，注意这些代码是实验性质，只能抄 func 源码，不能直接运行）：

# 下面的 func 已经改过，并不是原版
reward_funcs:
- cosine # 实际是 acc+len reward
- tag_count # 控制形式，防止 format hack
- reasoning_steps # 引导内容
- repetition_penalty  # debug 期间发现很多自我重复

reward_weights:
- 8
- 0.5
- 0.5
- 1

达到的效果是，1 个 epoch 就会收敛（相对于之前需要 1.5 个 epoch 左右）。

topic2.2 len penalty

前面说过 cold-start 起步就是 3k 长度，因此上来就简单粗暴地约束 len，会导致小鼠“难以动弹”。这是 len_penalty 的一些测试结果，可以看到 reward 曲线有剧烈抖动
