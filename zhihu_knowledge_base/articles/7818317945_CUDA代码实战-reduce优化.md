对于数组归约的并行计算问题，要从一个数组出发，最终得到一个数。假如数组元素个数是 2 的整数次方，可以将数组后半部分的各个元素与前半部分对应的数组元素相加，在GPU的并行计算中采用tree-based计算方法经过n轮迭代将数据做累加，最后得到sum。这就是所谓的折半归约(binary reduction) 法。

折半归约示意图a

具体的基准代码如下：

void __global__ reduce_global(real *d_x, real *d_y)
{
    const int tid = threadIdx.x;
    real *x = d_x + blockDim.x * blockIdx.x;

    for (int offset = blockDim.x >> 1; offset > 0; offset >>= 1)
    {
        if (tid < offset)
        {
            x[tid] += x[tid + offset];
        }
        __syncthreads();
    }

    if (tid == 0)
    {
        d_y[blockIdx.x] = x[0];
    }
}


这是最基础的 reduce算子的cuda实现，利用了gpu的全局内存而没有利用gpu的高速缓存，所以接下来我们将逐步介绍如何利用gpu的缓存结构以及其他基于gpu架构的优化手段来优化reduce算子的实现。

不过，上述代码我们仍然要注意的时候一个点就是__syncthreads(); 这条同步语句，用于在同一个线程块（Block）内的所有线程之间进行同步。这样同步是保证线程后序操作都能读到正确的数据，而不是因为线程执行快慢的原因导致错误数据的写入。

然后，让我们正式开始介绍reduce的优化手段

利用shared memory

第一招，也是最简单的恰恰也是所有优化手段的开始，那就是利用好GPU提供给我们的高速缓存结构

计算机内存架构

可以从上图清楚看到，GPU的SRAM是具有非常高的内存带宽以及非常小的存储空间的。所以我们需要好好利用到这个特性，对数据进行搬运从而提高运行速度。

所以，本节将使用 对整个线程块可见的共享内存来实现同样的规约操作。

使用shared memory进行reduce操作
    const int tid = threadIdx.x;
    const int bid = blockIdx.x;
    const int n = bid * blockDim.x + tid;
    __shared__ real s_y[256];
    s_y[tid] = (n < N) ? d_x[n] : 0.0;
    __syncthreads();


可以看到，我们定义一个带有__shared__修饰符号来定义s_y数组，然后将全局内存中的值搬运到线程在共享内存中的位置，这样后续我们对数据进行计算就能获得更高的内存带宽使用。值得注意的是，我们必须带有__syncthreadsared这条同步语句，效果还是一样确保后续操作中，线程块的线程能读到正确的数据。

解决bank confilct

在解决bank confilct之前，我们先了解一下什么bank，什么是bank confilct。

首先，为了实现高带宽，共享内存被分为大小相等的内存模块(32个)，称为存储体(bank)，可以同时访问。因此，由落入n 个不同存储体的n 个地址发出的任何存储器读或写请求都可以同时得到服务，从而产生比单个模块的带宽高n倍的总带宽。

bank示意图
bank 示意图

基于bank 的特性和shared memory的高带宽，可以实现并行数据读取。但是这个时候出现了一个叫做bank conflic的问题，这个问题描述为如果 同一个时间两个线程同时访问同一个bank里的两个数据，内存访问会从原来的并行读取退化成串行读取这会非常严重地浪费shared memory的高带宽。

bank冲突

就如上图，在左边的内存读取中，每一个线程读取一个bank里的数据，不会发生冲突，所有线程同时读取到数据并进行后续计算。在右边的内存读取中，也和左边一样，每一个线程读取一个bank里的数据虽然有点乱序，但是 不妨碍数据能同时被读取到。但是到了中间这个内存读取的过程中，对于bank 0 来说，线程0和线程16同时要读取bank0中的不同数据，这样就触发了2路bank conflict 导致并行读取退化成串行读取(其他 bank的读取同理）

无冲突的bank访问

如上图，中间和右边的内存读取操作，虽然有很多线程会去访问同一个bank但是他们访问的都是同一个数据，这样访问还是并行的，只是变成了随机选取一个线程访问数据然后由该线程广播给其他需要的线程，从而避开了bank conflict。

分析上图的reduce运算结合bank划分，我们以0号thread为例，0号线程需要去加载 shared memory 的0号和1号地址，然后写回0号地址。16号线程需要加载 shared memory 的32和33号地址并且写回32号地址。这样的话，0号线程 和16号线程需要访问数据处于同一个bank的不同数据，在bank0和bank1中都发生了2路bank confilt。在第二次迭代中，0号线程会加载0号和2号地址并写回0号地址，8号线程需要加载 shared memory 的32号和34号地址，并写回32号线程。16号线程会加载64号和68号地址，24号线程会加载96号和100号地址。然后0，32，64，96号地址都在一个bank中，所以这里产生了4路的 Bank Conflict 。以此类推，下一次迭代会产生8路的 Bank Conflict，使得整个 Kernel 一直受到 Bank Conflict 的影响。

传统的reduce计算会带来严重的bank confilct，所以我们修改一下循环迭代的顺序来解决bank confilct

修改迭代顺序

    {

        if (tid < offset)// 解决 reduce的warp divergence问题
        {
            s_y[tid] += s_y[tid + offset];

让我们来接着分析一下，为什么这种 迭代循环不会带来bank conflict。还是继续分析第一次迭代的0号线程，0号线程现在需要加载0号以及128号地址，并且写回0号地址。而1号线程需要加载1号和129号地址并写回1号地址。2号线程需要加载2号和130号地址并写回2号地址。我们可以发现第0个warp的线程在第一轮迭代中刚好加载shared memory的一行数据，不会产生 bank conflict。

完整的cuda核代码如下：

void __global__ reduce_shared(real *d_x, real *d_y)
{
    const int tid = threadIdx.x;
    const int bid = blockIdx.x;
    const int n = bid * blockDim.x + tid;
    __shared__ real s_y[256];//THREAD_PER_BLOCK = 128
    s_y[tid] = (n < N) ? d_x[n] : 0.0;
    __syncthreads();
//  从大到小去得到数据是为了避免bank-conflict ，这样从后往前的tid 就不会读到同一个bank的内容
    for (int offset = blockDim.x >> 1; offset > 0; offset >>= 1)
    {

        if (tid < offset)// 解决 reduce的warp divergence问题
        {
            s_y[tid] += s_y[tid + offset];
        }
        __syncthreads();
    }

    if (tid == 0)
    {
        d_y[bid] = s_y[0];// 每一个block的值的和reduce的和
    }
}

使用原子函数

可以看到，在上述的cuda核中，每一个线程的最后操作都是将0号线程的结果存储在全局内存中，然后由全局内存将得到的数组d_y传输会host端完成最后的运算。

//device端操作
    if (tid == 0)
    {
        d_y[bid] = s_y[0];// 每一个block的值的和reduce的和
    }

// host端操作
    CHECK(cudaMemcpy(h_y, d_y, ymem, cudaMemcpyDeviceToHost));

    real result = 0.0;
    for (int n = 0; n < grid_size; ++n)
    {
        result += h_y[n];
    }


这样非常非常非常非常明显是浪费资源和时间的操作，如果我们将余下计算 也放在gpu中计算，能显著减少计算时间(最明显的 就是直接去掉了通信时间)。我们可以直接在原核函数末尾利用原子函数来进行求和，代码如下

if (tid == 0) {
    d_y[0] += s_y[0];
}


很明显，这并不是正确的写法，因为tid==0代表着每一个 线程块的0号线程都有资格对全局内存中的d_y[0]进行读写操作。但是线程块的执行顺序是 不知道的，如果一个线程块的0号线程正在对d_y[0]进行操作但是还没有操作 结束，d_y[0]就被 其他线程块的0号线程抢走，但是 这样的话肯定会丢失掉一个线程块计算得到的数据 ，这是 非常不合理 的。所以我们应该 希望的是 一个线程的“读-写”操作不被其他线程干扰时，才能得到正确的结果。

所以我们使用原子函数：

    if (tid == 0)
    {
        atomicAdd(d_y, s_y[0]);
    }


原子函数 atomicAdd(address, val) 的第一个参数是待累加变量的地址 address，第二个 参数是累加的值 val。该函数的作用是将地址 address 中的旧值 old 读出，计算 old + val， 然后将计算的值存入地址 address。这些操作在一次原子事务(atomic transaction)中完成， 不会被别的线程中的原子操作所干扰。原子函数不能保证各个线程的执行具有特定的次序， 但是能够保证每个线程的操作一气呵成，不被其他线程干扰，所以能够保证得到正确的结果。

reduce计算图

基于上图和cuda 核函数 ，我们可以看到太多的thread被浪费了。对于第一轮迭代只有128个线程在工作，而第二轮迭代只有64个线程工作，第三轮迭代只有32和线程工作，以此类推，在每一轮迭代中都有大量的线程是空闲的。

    for (int offset = blockDim.x >> 1; offset > 0; offset >>= 1)

我们让每一轮迭代的空闲的线程也强行做一点工作，除了从global memory中取数之外再额外做一次加法。当前1个block有256个thread，只管256个数，现在要让这个256个thread管512个数。

void __global__ reduce_shared(real *d_x, real *d_y)
{
    const int tid = threadIdx.x;
    const int bid = blockIdx.x;
    const int n = bid * blockDim.x * 2 + tid;  // 每个线程负责了两个元素的加和
    __shared__ real s_y[256];//THREAD_PER_BLOCK = 128
    s_y[tid] = d_x[n]+d_x[n+blockDim.x];
    __syncthreads();
//  从大到小去得到数据是为了避免bank-conflict ，这样从后往前的tid 就不会读到同一个bank的内容
    for (int offset = blockDim.x >> 1; offset > 0; offset >>= 1)
    {

        if (tid < offset)// 解决 reduce的warp divergence问题
        {
            s_y[tid] += s_y[tid + offset];
        }
        __syncthreads();
    }

    if (tid == 0)
    {
        atomicAdd(d_y, s_y[0]);
    }
}

warp优化

需要说明的是，GPU需要分配两种资源，一个是存储资源，一个是计算资源。计算资源其实是根据thread数量来确定的，一个block中分配128个thread线程，32个线程为一组（即一个 warp），绑定在一个SIMD单元。所以256个线程可以简单地理解为分配了8组SIMD单元。

线程束(warp) 是 SM 中基本的执行单元。一个线程束由32个连续线程组成，这些线程按照单指令多线程(SIMT)方式执行（即所有线程执行相同指令，每个线程在私有数据上操作）。

基于上述概念，我们知道在最后我们不需要再对一个线程块中的所有线程进行同步了，转而选择一个更加廉价的线程束同步函数__syncwarp。

    for (int offset = blockDim.x >> 1; offset >= 32; offset >>= 1)
    {
        if (tid < offset)
        {
            s_y[tid] += s_y[tid + offset];
        }
        __syncthreads();
    }

    for (int offset = 16; offset > 0; offset >>= 1)
    {
        if (tid < offset)
        {
            s_y[tid] += s_y[tid + offset];
        }
        __syncwarp();
    }


当offset >= 32时，我们在每一次折半求和后使用线程块同步函 数__syncthreads; 当offset <= 16时，我们在每一次折半求和后使用束内同步函数__syncwarp。

另外还可以利用线程束洗牌函数进行归约计算，函数__shfl_down_sync的作用是将高线程号的数据平移到低线程号中去，这正是归约问题中需要的操作

    real y = s_y[tid];
    for (int offset = 16; offset > 0; offset >>= 1)
    {
        y += __shfl_down_sync(FULL_MASK, y, offset);
    }
    if (tid == 0)
    {
        atomicAdd(d_y, y);
    }


在这段代码中，我们将原来的s_y[tid]直接转移到寄存器中，寄存器一般来说比共享内存更高效，所以能用寄存器就当然用寄存器 了。__shfl_down_sync(FULL_MASK, y, offset);这是一个 CUDA 内置函数，用于在同一个 Warp 内的线程之间进行数据交换。将当前线程的y与offset偏移量后的线程的val进行交换。FULL_MASK是0xFFFFFFF，表示一个掩码，表示所有线程都参与交换。

使用这个内置函数，我们就没有再使用__syncwarp();因为洗牌函数能够自动处理同步与读-写竞争问题。
