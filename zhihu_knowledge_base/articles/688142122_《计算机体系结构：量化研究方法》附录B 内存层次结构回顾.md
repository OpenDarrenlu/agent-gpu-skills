Appendix B Review of Memory Hierarchy
B.1 介绍/Introduction

本附录是对内存层次结构的快速回顾，包括缓存和虚拟内存的基础知识、性能方程和简单的优化策略。

一些专用术语

缓存（cache）是地址从处理器离开后遇到的最高或第一级的内存层次结构的名称。 当处理器在缓存中找到请求的数据，称之为缓存命中（cache hit），反之未在缓存找到，称为缓存未命中（cache miss）。

一个固定尺寸的数据集合，包括请求的word，称之为块（block）或行（line），从主存（main memory）中取出并放入缓存中。

时间局部性告诉我们，我们可能在不久的将来再次需要这个word，因此将其放置在可以快速访问的缓存中是有用的。 由于空间局部性，block中的其他数据很快就会被需要的可能性很高。

缓存未命中所需要的时间取决于内存的延迟和带宽。 延迟决定了取回block中第一个word的时间；带宽决定了取回block剩余部分的时间。

缓存未命中由硬件处理，会导致顺序执行（in-order）的处理器暂停或停滞，直到数据可用。 对于乱序执行（out-of-order），使用该数据的指令仍必须等待，但在未命中期间，其他指令可以继续执行。

不是所有程序访问的对象都在主存中。

地址空间常常被拆分成固定大小的block，称为页（page）。

当处理器访问页中的对象不在缓存或主存储器中时，会发生页面错误（page fault），整个页面需要从磁盘移动到主存中。

由于页面错误需要很长时间，因此它们由软件处理，处理器不会停滞，在磁盘访问期间时，处理器通常会切换到其他任务。

缓存性能回顾

考虑处理器等待内存访问期间停滞的周期数，称之为内存停滞周期（Memory stall cycles ）。 性能是时钟周期时间与处理器周期和内存停滞周期之和的乘积

内存停滞周期取决于未命中的数量和每次未命中的开销，或者是未命中的代价：

IC（measure instruction count） 是测量的指令数量。 未命中率（miss rate）是缓存访问中导致未命中的比例（即未命中的访问次数除以总访问次数）。

进一步，由于读写未命中率和未命中开销通常是不同的。内存停滞周期时间可以通过根据每条指令的内存访问次数、读和写的未命中开销（以时钟周期为单位）以及读和写的未命中率来定义：

未命中率是缓存设计最重要的衡量标准之一，但是并不是唯一的衡量标准。

4个内存层次结构的问题

Q1: 可以将block放置在缓存哪个位置？

直接映射（direct mapped）缓存：每个block在缓存中只能出现在一个位置。映射方式为：
全相关联（fully associative）缓存：block可以放置在缓存中的任何位置
组相关联（set associative）缓存： block可以放置在缓存的一个受限位置的set中。一个set是缓存中的一组block。block映射到一个set上，可以在set内任何位置。通常通过bit选择来选择。

如果set中有n个block，那么这种缓存放置方式被称为n路组相联（n-way set associative）。这种结构提供了比直接映射更多的灵活性，因为它允许在set内部有多个block，同时又比全相关联缓存具有更低的硬件开销。

图B.2 举例缓存有8个block，而内存有32个block。从左到右显示了三种缓存类型。

在全相关联缓存中，来自较低层级的block 12可以放入缓存中的任何一个block。 对于直接映射缓存，block 12只能在缓存的第4个中（12除以8的余数）。 组相联缓存结合了前两种部分特点，允许block 12放在set 0中的任何位置（12除以4的余数）。这种每个set有2个block的方式，称为二路组相联（two-way set associative）

直接映射可以理解为是单路组相联，具有m个block的全相关联缓存可以称为“m路组相联”。相应地，也可以将直接映射视为具有m个set，而全相关联则视为具有一个set。

Q2: 怎么找到block是否在缓存中？

缓存在每个block的位置有一个地址标签（tag），提供block地址。每个可能包含所需信息的缓存block，都会检查tag是否与处理器提供的block地址匹配。所有可能的tag都是通过并行搜索。

必须有一种方法来确定cache block中不包含有效的信息。最常见的是在tag中添加一个有效位（valid bit），以指示是否包含有效的地址。

图B.3展示地址是怎么拆分的。首先地址被拆分为block address和block offset，block address进一步分为tag和index。block offset用于从选择block选择目标数据，index用于选择set，tag进行是否命中的比较。

图B.3 组相关联和直接映射缓存地址的3个部分

tag用于确认set中的block，index用于选择set。全相关联缓存没有index。

如果总缓存大小保持不变，增加关联性会增加每个set的block数，从而减少index尺寸并增加tag尺寸。

Q3: 缓存未命中时，应该替换哪个block?

当未命中发生时，缓存控制器必须选择一个block替换为所需的数据。

直接映射布局的一个好处是简化了硬件决策，只检查一个block是否命中和替换。

在全关联或组相关联中，有很多block可以选择。选择要替换的块主要有三种策略。

随机（random）：均匀分布分配，随机选择候选块。
最近最少使用（Least recently used, LRU）：为了降低丢弃即将需要的信息的几率，记录block的访问。依靠过去来预测未来，被替换的block是闲置时间最长的。
先进先出（FIFO）：通过确定最旧的block而不是最近最少使用的block。

图B.4 比较了LRU、random和FIFO方式，在几种不同缓存大小和关联下每1000条指令的未命中率

LRU和随机在大尺寸缓存中没什么差别，在尺寸小的时候LRU优于其他。在缓存较小的情况下，FIFO的性能通常优于随机。

Q4: 写会发生什么？

写策略通常区分缓存的设计，向缓存写入数据时，有两个基本选择：

write through：信息同时写入缓存中的block和更底层内存中的block。
write back：该信息只写入cache中的block。修改后的缓存block只有在替换时才写入主存。

为了降低替换时回写block的频率，通常使采用dirty bit。该状态位表示块是dirty(在缓存中被修改)还是clean(未修改)。如果该块是clean的，则不会在未命中时回写，因为在较低层次的内存中会找到与缓存相同的信息。

write back和write through各有优点。

write back：写操作以高速缓存的速度进行，一个block中的多次写操作只需要向底层内存写入一次。因为有些写操作不会进入内存，所以write back使用的内存带宽更少。
write through：缓存一直是clean的，读未命中不会导致向低层次内存进行写。低层级的内存一直有最新的数据，从而简化缓存一致性。

在write through过程中，当处理器必须等待写操作完成时，称为写停滞（write stall）。减少写停滞的一种常见优化是使用写buffer，它允许处理器在数据写入buffer后继续执行，从而将处理器执行与内存更新重叠。

因为在写时不需要该数据，在写未命中时有两个选项：

写分配（Write allocate）：在写未命中时分配block，随后执行先前的写命中操作。这种情况下，写未命中的行为类似于读未命中。
非写分配（No-write allocate）：写未命中不会影响缓存，只在较低层级的内存中修改blcok。

因此，在非写分配中，除非程序试图读取block，否则这些block不会出现在缓存中，而在写分配中，只被写入的block仍然会出现在缓存中。

PS： 假设有一个缓存，可以容纳4个block，分别为A/B/C/D，使用写分配策略。初始缓存是空的。考虑以下内存访问步骤：
读取block A（缓存未命中，从内存中获取block A并放入缓存）
写入block B（缓存未命中，从内存中获取block B并放入缓存，然后修改块B）
写入block A（缓存命中，直接在缓存中修改block A）
读取block B（缓存命中，从缓存中获取block B）

相反，如果使用非写分配策略，那么步骤2中的写未命中将不会影响缓存。block B将只在内存中修改，而不会进入缓存。因此，在步骤4中读取block B时，将发生缓存未命中，即使block B之前已经被写入过。

B.2 缓存性能/Cache Performance

因为指令计数是独立于硬件的，所以很容易用这个数字来评估处理器性能，在评估内存的性能时，往往关注未命中率，因为它也与硬件的速度无关。我们将看到，失效率与指令数一样具有误导性。

一个更好的内存层次性能度量是平均内存访问时间: 平均内存访问时间 = 命中时间 + 未命中率 × 未命中开销

Average memory access time = Hit time + Miss rate × Miss penalty

平均访问时间可以用绝对时间来计算，比如一次命中花费0.25 ~ 1.0 ns，或用处理器等待内存的时钟周期数来计算，比如等待150 ~ 200个时钟周期。

平均内存访问时间和处理器性能

有个明显的问题，缓存的平均访问时间是否可以预测处理器的性能？

存在其他导致停滞（stall）的原因，例如I/O设备使用内存时发生的竞争。 通常会假设所有内存停滞都是由于缓存未命中引起的，因为内存层次结构通常在其他导致停滞的原因中占主导地位。但在计算最终性能时，应该考虑所有内存停滞情况。
取决于处理器。如果是顺序执行处理器，那么基本上是肯定的。处理器在未命中期间会停滞，并且内存停滞时间与平均内存访问时间密切相关。

缓存命中的时钟周期是否应被视为CPU执行时钟周期的一部分，还是内存停滞时钟周期的一部分。最广泛接受的约定是将命中时钟周期包含在CPU执行时钟周期中。

例：用一个顺序指令的计算机作为例子。假设缓存未命中的开销是200个时钟周期，所有指令通常需要1个时钟周期(忽略内存停滞)。假设平均未命中率为2%，每条指令平均有1.5个内存访问，每1000条指令的平均缓存未命中数为30。当包含缓存的行为时，对性能的影响是什么样的？
答：
第一种。用每条指令的未命中数计算。
第二种。用缓存未命中率计算。
不管有没有缓存，时钟周期时间（clock cycle time）和指令计数（IC）都是相同的。因此，如果CPU时间增加7倍，对于一个“理想缓存”，CPI从1.00增加到7.00。如果没有任何内存层次结构，CPI将再次增加到1.0 + 200 × 1.5（301），是40倍多。

以上例子说明，缓存行为可以对处理器性能产生巨大影响。此外，缓存未命中对于低CPI和高速时钟的处理器有双重影响:

CPI越低，固定数量的缓存未命中时钟周期的相对影响就越高。
在计算CPI时，缓存未命中的开销是以处理器时钟周期为单位来度量的。因此，即使两台计算机的内存层次结构相同，时钟频率较高的处理器的每个时钟周期有更大的未命中数，因此CPI中内存部分占比更高。

因此，对于低CPI和高时钟速率的处理器来说，缓存的重要性更大。

B.3 6种基本的缓存优化/Six Basic Cache Optimizations

平均内存访问时间公式为我们提供了一个提高缓存性能的缓存优化框架:

平均内存访问时间 = 命中时间 + 未命中率 × 未命中开销

Average memory access time = Hit time + Miss rate × Miss penalty

因此，我们将六种缓存优化分为三类:

降低未命中率。更大的block尺寸、更大的缓存大小、更高的相关联性。
减少未命中开销。多级缓存，并读优先于写。
减少缓存命中的时间。索引缓存时避免地址转换。

为了更好地了解错误的原因，从模型开始，该模型将所有错误分为三个简单的类别：

强制性（Compulsory）：对某个block的首次访问不可能在缓存中，因此该块必须被加载到缓存中。这些也被称为冷启动未命中（cold-start misses）或首次引用未命中（first-reference misses）。
容量（Capacity）：如果缓存无法包含程序执行期间所需的所有block，则除了强制性未命中外，还会因为block被丢弃然后又被重新恢复而发生容量未命中。
冲突（Conflict）：如果block放置策略是组相联或直接映射的，那么除了强制性未命中和容量缺未命中之外，还会发生冲突未命中。因为如果太多block映射到同个set，block可能会被丢弃并在之后又被重新恢复。这些未命中也被称为碰撞未命中（collision misses）。
第一种优化：更大block尺寸降低未命中率

减少未命中率最简单的方法是增加block尺寸。更大的block尺寸会减少强制性未命中。同时，更大的block尺寸也会增加未命中开销。由于减少了缓存中的block数，因此如果缓存较小，更大的block尺寸可能会增加冲突未命中，甚至容量未命中。

Block大小的选择取决于较低层级内存的延迟和带宽。高延迟和高带宽鼓励使用大尺寸的block，因为缓存每次未命中时可以获得更多byte，而未命中开销的增加很小。相反，低延迟和低带宽鼓励使用较小尺寸的block。

第二种优化：更大缓存尺寸降低未命中率

减少容量未命中的明显方法是增加缓存的容量，其明显的缺点是更长的命中时间以及更高的成本和功耗。

第五种优化：优先考虑读未命中以减少未命中开销

在写完成前提供读。

对于write through缓存，最重要的提升是适当尺寸的写buffer。写buffer会使内存访问复杂化，因为需要保持读未命中时所需位置的更新值。

为了避免RAW冒险，最简单方法是让读未命中等待到写buffer为空。另一种方法是在读未命中时检查写buffer的内容，如果没有冲突且内存系统可用，则读未命中可以继续进行。

处理器中write back缓存的写开销可以降低。假设读未命中会替换一个脏dirty的内存block。我们可以将dirty block复制到buffer中，然后读取内存，然后再写入内存，而不是先将脏块写入内存。

第六种优化：在索引缓存时避免地址转换以减少命中时间

常见的高速准则，在缓存中使用虚拟地址，因为命中比未命中更加常见。 这种缓存被称为虚拟缓存（virtual cache），而传统缓存是采用物理地址的物理缓存（physical cache）。

区分索引缓存和比较地址2个任务很重要。

问题在用于缓存索引和tag比较的是虚拟地址还是物理地址。索引和tag的全虚拟寻址消除了缓存命中的地址转换时间。

一种兼顾虚拟和物理缓存优点的替代方案是，使用page offset的一部分(虚拟地址和物理地址相同的部分)来索引缓存。在使用索引读取缓存的同时，地址的虚拟部分被转换，并且tag使用物理地址进行匹配。允许立刻开始读缓存， 但也有限制，即直接映射缓存尺寸不能大于page。

B.4 虚拟内存/Virtual Memory

为每个进程分配一个完整的地址空间代价太大，特别是许多进程只使用其地址空间的一小部分。因此，一定有一种方法可以在许多进程之间共享较小数量的物理内存。

一种方法是虚拟内存，将物理内存划分为block，并将其分配给不同的进程。在这种方法中必须有一种保护方案，将block限制为只属于该进程的。大多数形式的虚拟内存还减少了启动程序的时间，因为在程序开始之前，并非所有代码和数据都需要放在物理内存中。

虚拟内存是为了减轻编程人员的负担，可以自动管理两级内存层次（主存，次级存储）。图B.19展示为了一个程序用4个page将虚拟内存映射到物理地址的示例。除了共享受保护的内存空间和自动管理内存层次外，虚拟内存还简化了加载程序以执行。这种机制被称为重定位(relocation)，它允许同一个程序在物理内存中的任何位置运行。

图B.19 左侧是连续虚拟地址空间。由A、B、C、D 4个page组成（3个block的实际在物理主存中，另1个位于磁盘上）。

程序在图B.19中可以在物理内存或磁盘中的任意位置，只需要改变映射关系。或者每次运行程序时，用软件更改程序中的所有地址。

用page或segment表示block，用页错误或地址错误表示未命中。使用虚拟内存时，处理器产生虚拟地址，这些虚拟地址由硬件和软件的组合转换为访问主存的物理地址。这个过程称为内存映射或地址转换。今天，由虚拟内存控制的两级内存层次是DRAM和磁盘。

Q1: 可以将block放置在主存哪个位置？

对于虚拟内存，未命中的开销涉及访问磁盘所以代价很大。如果要从更低的缺未命中率或更简单的放置算法中选择，操作系统设计者通常会选择更低的未命中率的方案，因为未命的代价过高。因此，操作系统会允许将block放置在主存中的任何位置，就像缓存的全相联。

Q2: 怎么找到block是否在主存中？

Page和segment两种方式通过page和segment的序号来索引。包含了物理地址的部分。

对于page的方式而言，offset只是和物理地址拼接在一起。如图B.23。
对于segment的方式而言，offset需要和segment的物理地址相加，获得最终的物理地址。

页表（page table）中包含了物理地址的信息。表的大小由虚拟页号索引，即虚拟地址空间中页的数量。 对于一个32位的虚拟地址，4 KB尺寸的page，每个页表项目（page table entry，PTE）为4 byte而言，页表的尺寸为（2^32/2^12）× 2^2 = 2^22 （4 MB）。

Q3: 虚拟内存未命中时，应该替换哪个block?

几乎所有操作系统都尝试替换最近最少使用（LRU）的block，因为如果从过去预测未来，该block可能不太需要。

Q4: 写会发生什么?

主存以下层级的内存访问时间过长，write through的方式不现实，一般策略都是write back。

快速地址转换技术

页表通常非常大，存储在主存中，有时自己进行分页。分页意味着每次内存访问在逻辑上需要花费至少2倍的时间，其中一次获得物理地址，一次是获得数据。

如第2章所述，使用局部性来避免额外的内存访问。通过将地址转换保存在一个特殊的高速缓存中，避免额外的地址访问。这种特殊的地址转换缓存称为TLB (translation look aside buffer)，也称为转换buffer(translation buffer)。

一个TLB的项就像缓存中的一样，tag保存虚拟地址，数据部分保存物理page号，保护字段，有效位，use bit 和dirty bit。

为改变页表项的物理page号或保护部分，操作系统必须确保旧的页表项不在TLB中。否则，系统将无法正常运行。 这边的dirty bit意味着对应的页是修改过的。

选择page的尺寸

更大尺寸的好处:

页表的大小与page的大小成反比；可以通过增大page的尺寸来节省内存(或用于内存映射的其他资源)。
更大的page允许更大的缓存，使得缓存命中更快。
在次级存储之间传输较大的page比传输较小的page更高效。
TLB项的数量是有限制的，因此较大的page意味着可以有效地映射更多的内存，从而减少TLB未命中次数。
