# 折扇模型：直观理解EP对云端推理服务性价比的意义

---

DeepSeek开源周极大地推动了国内AI Infra领域的技术进步。本文分享“折扇模型”，提供一个直观理解超大规模EP对推理性价比的方法，希望也能为国内AI推理Infra领域添砖加瓦。

过去半年，因工作需要，我持续对云端推理服务进行建模分析，逐渐意识到云端推理的设计空间与设计目标的复杂度远超训练阶段。

具体而言，云端推理的设计空间包含以下维度：

模型结构：最近半年新加入的MLA、MoBa、NSA、Lighting-Attention等算法都会显著影响推理性价比。
芯片：工作需要分析不同芯片的推理性能和成本，包括尚未问世的芯片。
并行度配置：PP (Parallel Pipeline)、TP (Tensor Parallel)、DP (Data Parallel)、EP (Expert Parallel)、SP (Sequence Parallel)，不同并行度配置可相互组合，形成多样的并行策略。若再考虑Prefill-Decode分离，设计空间将更加庞大。
推理场景：Batch Size、Sequence Length的不同，会显著影响推理性能和成本。

云端推理的首要设计目标包括：

用户交互速率Interactivity越高越好，即单个请求的TPOT (Time Per Output Token)越小越好。
每Token成本Cost per Token越低越好，可折算为TGS (Tokens per GPU per Second)。

去年阅读DeepSeek-V3报告时，发现Decoding阶段使用了EP320，这一设计对我的认知产生了巨大冲击。通过后续的模型分析，我逐渐理解了超大EP对云端推理性价比的重大意义。

本文介绍的"折扇模型"，是我在研究云端推理性价比时，为直观理解超大EP的意义而提出的一个模型。

图中数据不是最终结果，仅用作展示折扇模型的形状

在TGS-Interactivity空间中，折扇模型能直观展示超大EP对云端推理性价比的重大影响。

图中每个点代表一种并行度配置。由于TGS和Interactivity都是越高越好，因此存在一条帕累托前沿，前沿上的点代表了一组最优的并行度配置。

帕累托前沿（Pareto Frontier）是一个经济学和多目标优化中的重要概念，在折扇模型中具有特殊意义。具体而言：

它是TGS和Interactivity这两个指标的最优组合点的集合
前沿上的每个点都代表一种无法同时提高两个指标的并行度配置
从前沿上任意一点移动，要么TGS下降，要么Interactivity下降，无法两者兼得

AI Infra公司需要在帕累托前沿上，根据用户需求和成本预算，选择最合适的并行度配置。

非最优的配置，则像折扇的扇骨一样，从原点射向帕累托前沿。

每根扇骨上，TP * PP都是常数。准确地说，TGS-Interactivity的斜率，与TP * PP成反比。

EP与TP和PP的最大区别在于，它打破了TGS和Interactivity的Trade-off关系。增大EP，可以同时提高Interactivity和TGS。

云端推理需要先增大EP，打到帕累托前沿，然后根据用户需求和成本预算，调整DP、TP和PP，选择最合适的并行度配置。




下文将首先就推理系统设计目标形成共识，然后介绍折扇模型的推导过程。

大语言模型推理系统设计目标

目前云端推理系统的设计目标尚未统一，本文先给出相关定义。

在讨论自回归（仅解码器）模型中生成token的时间时，我们通常跟踪时间戳 $t_i$，其中：

$t_0 = 0$ 是推理开始时间。 $t_1$ 是第一个token生成完成的时间。 $t_2$ 是第二个token生成完成的时间，以此类推... $t_T$ 是第T个（最后一个）token生成完成的时间。

在基准测试实验中，我们使用以下大语言模型推理指标：

TTFT (Time To First Token)：

从发送推理请求到模型产生其第一个输出token所经过的时间。
$TTFT = t_1 - t_0$

TPOT (Time Per Output Token)：

生成第一个token后，生成每个后续token所需的平均时间。
$TPOT = \frac{t_T - t_1}{T - 1}$ - 定义用户交互速率Interactivity = 1 / TPOT，即单个请求的token生成速率，单位为token/s。
随着reasoning模型的流行，Interactivity越来越成为衡量推理服务性价比的重要指标。

端到端请求延迟 (E2E Latency)：

此指标表示从提交查询到接收完整响应所花费的时间。
E2E Latency = 排队时间 + $t_T - t_0$ - 包括CPU上调度、批处理和网络延迟。线上服务需要考虑用户等待时间，需要优化E2E Latency。但不是GPU侧的优化目标。
客户端测量TTFT时，难以去除​调度、批处理和网络延迟的影响。因此API服务应该以TPOT作为主要监测指标。

TGS（Tokens per GPU per Second），即Throughput per GPU：

衡量单GPU计算资源利用率的核心指标，反映分布式系统中单个GPU的token处理能力
与TPOT存在天然冲突：提升TGS需增大batch_size，而降低TPOT需减小batch_size
实际部署需在硬件成本（TGS↑）与用户体验（TPOT↓）间取得平衡
整个集群的Throughput = TGS * GPU数量。但TGS更重要，因为它能反映单个GPU的资源利用效率。

每Token成本 (Cost per Token)：

云端服务最终的计费单位是Token，所以每Token成本是云端推理服务最重要的指标。
每Token成本 = GPU机时成本 / TGS - 为了降低每Token成本，需要提高TGS。

综上所述，云端推理系统最重要的指标就是TGS和TPOT。TGS越大越好，TPOT越小越好。

TPOT等价于Interactivity。TGS和Interactivity都是越高越好。

下文将使用TGS和Interactivity作为优化目标，方便画出"折扇"和帕累托前沿。

去年研读这篇文章时，就注意到类似折扇的形状。但该文章所有坐标轴都没有数据，并且缺少token长度、batch size、模型结构等关键信息，因此无从推导折扇模型的由来。

直到最近，通过分析DeepSeek-V3/R1在Reasoning场景下的benchmark数据和性能建模，才逐渐理清折扇模型的推导过程。最关键地是要通过合理假设，简化问题。

具体而言，折扇模型基于以下假设：

忽略通信开销。
忽略Prefill开销（类似Reasoning场景下的超长输出）。
暂时忽略并行策略对Interactivity的影响，只关注如何从Interactivity推导出TGS。

最后一条最为关键。因为不同的并行策略配置会显著影响Interactivity，从而影响TGS。如果一直纠缠于这一点，就会陷入无穷无尽的细节中，从而错过了折扇模型的核心。

体系结构领域最重要的屋檐模型，也是在做出大量假设后得到的一个可视化工具。做出好的假设简化问题，是系统研究中最重要的能力。

TGS和Interactivity的关系

vLLM的benchmark中，会报告TPS (Tokens per Second)和TPOT (Time Per Output Token)。

如果Prefill非常短，可以近似计算：

$TPS = Concurrency / TPOT = Concurrency * Interactivity$

以下表格摘自E2E Cloud Inference Benchmarks，在H200 4卡上不同场景的性能测试数据。

Model	GPUs	Task Type	Input - Output	Request Rate	Requests/Sec	Throughput	Time to First Token (TTFT)	Time Per Output Token (TPOT)	Total Latency
llama3.1 (70B)	4	Simple Chat	[128:128]	24	15.3	3542 tokens/sec	90.59 ms	29.53 ms	3840.9 ms
llama3.1 (70B)	4	Summarization / RAG	[2024:128]	4	3.45	6139 tokens/sec	455.96 ms	46.11 ms	6311.93 ms
llama3.1 (70B)	4	Creative Writing	[200:2024]	8	2.35	2782 tokens/sec	69.44 ms	25.09 ms	50826.51 ms

可以清晰地看出，在Creative Writing场景下，由于输出长度2048远大于200，Throughput近似等于Request Rate / TPOT。

在使用Prefill-Decode分离的场景下，decode阶段的性价比分析，可以直接套用这个公式。

在分布式推理系统中，我们更关心TGS，而不是TPS，于是：

$TGS = TPS / #GPU = Concurrency / #GPU * Interactivity$

增加DP (Data Parallel，数据并行)

DP是最基础的并行策略，它将并发的请求，拆成DP路，每路大小为bs (batch size)。

整个集群中，模型的权重需要复制DP份。

$Concurrency = DP * bs$

$TGS = Concurrency / #GPU * Interactivity = bs * Interactivity$

一路DP可以固定大小的bs（通常跟kv-cache size和sequence length有关）。

增加DP不改变Interactivity，从而对TGS也不会有影响。

增加TP（Tensor Parallel，张量并行）

TP将每层的权重和KV-cache均匀分片到多个GPU上，线性扩展每个GPU的批处理规模。然而，TP涉及高通信成本，特别是在每层的自注意力计算和FFN中需要进行两次all-reduce操作。

矩阵乘法的分块计算（如$WX = [W_1|W_2][X_1;X_2]$）
每GPU持有部分参数，计算结果通过AllReduce通信聚合结果

下表中基于GPT-3的模型结构，标记了TP并行下各个算子按照Batched GEMM表述方式时的Layout。

特别需要注意：随着TP的增大，QKV_Proj和FFN_Up算子，计算访存比都在降低；其他算子计算访存比不变。

Decode阶段	B	M	N	K	计算量	读访存量	计算访存比
QKV_Proj	1	b	3h/TP	h	6bshh/TP	2(bsh+3hh/TP)	3bsh/(3h+bs*TP)
QxKT	b*n/TP	1	s	d	2bnssd/TP	2bn(sd+sd)/TP	s/2
QKTxV	b*n/TP	1	d	s	2bnsds/TP	2bn(ss+ds)/TP	sd/(s+d)

FFN_Down	1	b	h	m/TP	2bshm/TP	2(bsm+mh)/TP	bsh/(bs+h)

加入TP后Interactivity会增加，毕竟每个GPU上的计算量、访存量都变小了，导致TPOT降低。

然而，考虑到计算访存比的降低、通信开销等影响，Interactivity无法跟着TP线性增加。

$TGS = Concurrency / #GPU * Interactivity = bs / TP * Interactivity$

结合以上公式，TGS会随着TP增加而下降。

增加PP（Pipeline Parallel，流水线并行）

PP将模型按层分片，每个GPU负责一部分层。通过微批处理（micro-batching）保持所有GPU的忙碌状态。PP的通信效率高于TP，因为它只需要在层之间传递激活值，且通信操作仅限于点对点通信。

将Transformer的N个层分配到M个GPU
采用micro-batch流水线调度，掩藏"流水线气泡"开销

注意，推理流水并行和训练流水并行拆microbatch后的调度方式是不同的。

训练中，一个Batch的前向过程需要将N个microbatch的梯度计算出来，反向传播；等待所有microbatch的梯度计算出来后，统一更新参数。

推理中，只有前向过程，无需等待一个batch内所有的microbatch计算完成，就可以开始下一个batch的推理。因此，流水并行的调度上可以看到不同Batch的microbatch可以并行处理。




推理的流水并行中，不同Batch的microbatch需要并行处理

由于不同流水级之间的数据依赖关系，Decoding时，每个Token的下一级流水都需要等待上一级流水的结果，因此忽略通信开销时，流水并行不会改变TPOT。而考虑到通信开销，TPOT会稍有增加。

PP需要将bs再拆分成mbs (micro-batch size)，以充分利用PP级流水线。

$bs = PP * mbs$

每个mbs同一时间，只会利用TP个GPU。

$TGS = mbs / (TP) * Interactivity = bs / (TP * PP) * Interactivity$

忽略通信开销时，PP不改变Interactivity，因此TGS会随着PP增加而降低。

增加MoE-EP, Attention-DP（简称EP）后

EP将bs个请求，拆分到EP路GPU上；每路EP在decoding阶段处理的1/EP * #expert。

对于Attention，在EP路GPU之间，采用数据并行策略。

因为并发请求数受限于Attention的kv-cache容量，所以增加EP后，整个集群中可以处理的并发请求数变为EP * bs。

$TGS = EP * bs / (TP * PP * EP) * Interactivity = bs / (TP * PP) * Interactivity$

增加EP后，Interactivity会增加。

例如从EP8DP8到EP16DP4，每个GPU上需要处理的Expert数量减小了。

虽然单个GPU上要处理的Token数量不变，但每个Expert的计算访存比增加，从而Interactivity增加。

折扇模型与帕累托前沿

完整的分布式推理系统中，考虑TP PP DP EP四种并行策略：$#GPU = TP * PP * DP * EP$

Insight: 不同并行度配置下，TGS-Interactivity的斜率正比于1/(PP * TP)。与DP和EP无关。

DP不影响TGS和Interactivity。

增大TP，可以增大Interactivity，但由于算子计算访存比下降、通信开销等影响，Interactivity不会线性增加；同时TPS-Interactivity的关系中需要除TP，因此会导致TGS下降。

增大PP，几乎不影响Interactivity；但由于TPS-Interactivity的关系中需要除PP，因此会导致TGS下降。

增大EP，可以同时增加TGS和Interactivity，是最经济的并行策略。


当然，折扇模型中，忽略了通信开销、Prefill开销。实际的模型会更加复杂。因此我们面向云端推理开发了一个性能分析模型。之后会在进行数据脱敏后开源。

总结一下，折扇模型中，不同并行度配置下，Interactivity和TGS的斜率，与TP * PP成反比，与EP无关。EP可以同时增大Interactivity和TGS。因此云端推理需要在通信开销可控的范围内，将EP拉到最大，先摸到帕累托前沿；然后再根据客户需求，调整TP和PP，在TGS和Interactivity之间做trade-off。这就是DeepSeek-V3/R1报告中，Decoding阶段使用EP320的原因。
