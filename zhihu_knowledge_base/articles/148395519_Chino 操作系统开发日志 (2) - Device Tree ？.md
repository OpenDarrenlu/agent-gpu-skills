Chino 需要什么？

作为一个 IoT 系统，多半是运行在定制的 board 上，因此需要考虑以下几个问题：

设备和配置多种多样：不同的板子上有不同的外设、不同的引脚定义，系统必须具备可灵活配置设备树和参数的能力
总线一般不具备设备枚举功能：不像 PC 的 PCIe 总线可以通过 ACPI 枚举设备，一般嵌入式系统的 AHB、APB 以及 SPI、I2C 等等都需要 BSP 给系统提供设备树信息
有限的设备更改：一个作为产品的 board 基本不会在生命周期中发生设备修改的情况（除了 USB、SDIO 之类的自带枚举功能）；而作为开发板也基本不存在运行时修改设备的需求
不同的 board 存在设备和参数复用：不同的 board 可能有相同的 chip，他们可以共享 chip 自带的设备、引脚定义等等；不同的 board 也可能共享相同的板上设备
对于同一个 board，不同的 application 可能需要不同的设备集合
Device Tree 是什么？

Device Tree 在 linux 中被广泛用来描述设备树和配置信息。Device Tree 用文本形式进行编写，然后交由 dtc 编译成二进制的 fdt。系统在初始化中使用 libfdt 之类的库解析 fdt 还原设备树结构。

Device Tree 用树形结构描述整个系统中存在的设备，包括 cpu、内存、引脚及各类外设。它的详细介绍这里不再赘述，可以参考这个系列的文章

用 Device Tree 合适吗？

Device Tree 既然已经是一个久经考验的东西，说明它已经足够胜任描述设备树和配置信息的工作。但是它本身也存在一些问题：

写 dts 的时候缺乏类型检查，作为一种 DSL 没有主流语言那么好的工具支持，还要对照 binding 来写，不能在写的过程中发现和纠正错误
fdt 不存储类型信息，无法在运行时检查 property 的类型，使得错误不能尽早发现
没法定义宏、定义函数，降低了复用能力，甚至你想写 500kb 的 int 字面量也实现不了。。。

对于 IoT 场景，Device Tree 还有一些额外的不必要开销：

读取 fdt 需要额外引入 libfdt，增大了代码体积
本可在编译期完成的任务挪到了运行时，增加了时间和空间的开销：比如查找 dev 对应的 driver；应对多种配置可能而引入的本可在编译期优化掉的代码，如 pinmap（比如有一个 uart 设备有 A1 A2 和 B1 B2 两种 pin group，如果在编译期已知选择的哪组，那另一组的配置代码就可以被优化掉）
Chino 的选择是？

最终 chino 没有选择 Device Tree，也没有选择另一种 DSL，直接采用了 C# + Razor Template 实现设备树的描述和编译期解析。

所有的 chip、board、device 定义和配置直接用 C# 来写。选择 C# 的原因一个是我本人喜好，另一个是 C# 和其工具的易用性，再加上最近的 source generator 进一步提升了元编程能力。在 C# 里你可以简单写一个 prototype 模式生成 stm32f103 的各类型号，也可以用 angularsen/UnitsNet 实现 400.Kibibytes() 这种直观的容量写法。

设备树、引脚定义的和驱动配置的 parsing 则通过 Razor Template 来实现。C# 写的定义和配置作为 Model 传给 Razor Template，Render 出各种你想要的东西（比如 .cmake、.h、.cpp）。Razor 的语法也很简洁强大（比起一些 template DSL）。这里用到了一个库 toddams/RazorLight

比如我用这个模板

// This file is generated by chinoc
//
@using Chino

@{ int i = 0; }
@foreach (SimpleDeviceNode device in Model.FindCompatibleDevices(new Guid("09719CD9-29F1-48B6-9590-33BFD56C3506")))
{
@:const win32_fs_device_registration fs_dev@(i) = { { .drv = fs_drv }, "@device.Properties["Path"]" };
@:EXPORT_HARDWARE(fs_dev@(i++));
}


生成了下面的 .h

// This file is generated by chinoc
//

const win32_fs_device_registration fs_dev0 = { { .drv = fs_drv }, "share" };
EXPORT_HARDWARE(fs_dev0);


直接生成了驱动所需的数据结构，不需要像 fdt 一样运行时解析。
