背景

当前热门开源的推理框架，vllm和sglang各自实现了优化显存方案，比如paged attention和chunked prefill。一般来说，算子不能直接访问内存，显存不足推理会失败；显存的重要性自然不必多说。

实际上，大家更关注的是如何发挥显存利用率，在推理性能和吞吐两个维度间找到平衡，发挥硬件的利用率。

有cpu programming经验朋友可能都会想到一个词：虚拟内存管理，内存以page单位来分配和释放，真正使用时分配内存。那么gpu上可以这么做吗?

分页是可以的，这篇文章就是围绕按照一定粒度来分配显存的；但是显存和内存之间的数据移动是推理框架这层来实现的，相当于prefetch语义。

为什么不让linux kernel接管？

首先gpu显存管理是驱动范畴，每个厂商都有自己的实现。目前看gpu设备驱动是统一在pcie设备硬件抽象。

驱动无论是具体实现还是功能定义差异都很大。

执行流不同

在gpu上执行的任务一般统称算子。算子执行过程中访问的数据在cpu上需要搬运到gpu上。如果由驱动实现数据搬运任务，驱动是运行在cpu上，这相当于gpu执行依赖cpu，性能会受到很大影响。

在nvidia驱动生态里，有过uvm，ats之类特性，支持cpu和gpu之间按需搬运数据。因性能问题，这两个特性在真实场景下不常用。

chunked prefill意义

[我瞎说的]

decoder架构的模型以自回归方式来推理。根据prompts生成第一个token的阶段被称为prefill；然后依次生成后续token，一直到生成eos token或者达到限制结束推理，被称为decode阶段。为了避免对q的历史位置重复做k/v projection，引入kvcache把历史的k/v存储到显存中。随着q各个位置token依次被处理，kvcache使用量会变大。

简而言之，prefill阶段生产kvcache数据，decode阶段既生产kvcache数据，也消费kvcache数据。kvcache和模型权重占据显存绝大部分空间。

不幸的是，decode阶段何时完成？要分配多少kvcache空间？在执行完成前是不确定的。那么，prefill阶段激进地分配显存，对于decode阶段面临oom风险会不可控。如果用reservation方式，会降低batch size导致decode算力利用率下降。

其实chunked prefill最主要场景是long context长文本上下文处理。

开启chunked prefill第一次调度req时，调用get_new_batch_prefill生成一个新的batch。

代码中PrefillAdder表示prefill阶段调度req时管理信息：

ScheduleBatch -> ModelWorkerBatch -> ForwardBatch

一个batch中的不同req之间是无关的，调度过程就是持续把req放到batch里。调度何时结束呢？

AddReqResult.NO_TOKEN：预测decode阶段占用过多kvcache出现oom

class PrefillAdder:
    ...
    def add_one_req(
        self, req: Req, has_chunked_req: bool, enable_hierarchical_cache: bool = False
    ):
        total_tokens = req.extend_input_len + min(
            req.sampling_params.max_new_tokens, CLIP_MAX_NEW_TOKENS_ESTIMATION
        )
        input_tokens = req.extend_input_len
        prefix_len = len(req.prefix_indices)

        if total_tokens >= self.rem_total_tokens:
            return AddReqResult.NO_TOKEN

        if input_tokens > self.rem_input_tokens and len(self.can_run_list) != 0:
            return AddReqResult.OTHER

        with self._lock_node(req.last_node):
            if total_tokens > self.rem_total_tokens:
                return AddReqResult.NO_TOKEN
            ...
            if self.rem_chunk_tokens is None or input_tokens <= self.rem_chunk_tokens:
                # Non-chunked prefill
                self.can_run_list.append(req)
                self.tree_cache.inc_lock_ref(req.last_node)
                self._prefill_one_req(
                    prefix_len,
                    input_tokens,
                    min(
                        req.sampling_params.max_new_tokens,
                        CLIP_MAX_NEW_TOKENS_ESTIMATION,
                    ),
                )
            else:
                if self.rem_chunk_tokens == 0:
                    return AddReqResult.OTHER

                # Chunked prefill
                trunc_len = self.rem_chunk_tokens

                req.extend_input_len = trunc_len
                req.fill_ids = req.fill_ids[: len(req.prefix_indices) + trunc_len]

                self.can_run_list.append(req)
                self.new_chunked_req = req
                self.tree_cache.inc_lock_ref(req.last_node)
                self._prefill_one_req(prefix_len, trunc_len, 0)

        return self.budget_state()

    def _prefill_one_req(
        self, prefix_len: int, extend_input_len: int, max_new_tokens: int
    ):
        self.rem_total_token_offset += extend_input_len + max_new_tokens
        self.cur_rem_token_offset += extend_input_len
        self.rem_input_tokens -= extend_input_len
        if self.rem_chunk_tokens is not None:
            self.rem_chunk_tokens -= extend_input_len

        self.log_hit_tokens += prefix_len
        self.log_input_tokens += extend_input_len


    def budget_state(self):
        if self.rem_total_tokens <= 0 or self.cur_rem_tokens <= 0:
            return AddReqResult.NO_TOKEN

        if self.rem_input_tokens <= 0 or (
            self.rem_chunk_tokens is not None and self.rem_chunk_tokens <= 0
        ):
            return AddReqResult.OTHER

        return AddReqResult.CONTINUE

    @property
    def rem_total_tokens(self):
        return (
            self.token_to_kv_pool_allocator.available_size()
            + self.tree_cache.evictable_size()
            - self.rem_total_token_offset
        )

chunk size足够处理当前req：self.rem_chunk_tokens is None or input_tokens <= self.rem_chunk_tokens
当前req需要切分多个chunk：otherwise
req被切成多个chunk，其实是extend tokens被截断了，被truncated部分下次schedule再处理

chunked prefill影响是全局的，如果设置了chunked prefill size，每次schedule时所有请求的extend tokens总和不能超过chunk size。那么，全局只有一个被truncated的req且是最后一个，记在adder.new_chunked_req。

class Scheduler(SchedulerOutputProcessorMixin):
    """A scheduler that manages a tensor parallel GPU worker."""
    ...
    def get_next_batch_to_run(self) -> Optional[ScheduleBatch]:
        # Merge the prefill batch into the running batch
        if self.last_batch and self.last_batch.forward_mode.is_extend():
            if self.chunked_req:
                # Move the chunked request out of the batch so that we can merge
                # only finished requests to running_batch.
                self.last_batch.filter_batch(chunked_req_to_exclude=self.chunked_req)
                self.tree_cache.cache_unfinished_req(self.chunked_req)
                # chunked request keeps its rid but will get a new req_pool_idx
                self.req_to_token_pool.free(self.chunked_req.req_pool_idx)

        new_batch = self.get_new_batch_prefill()
        if new_batch is not None:
            # Run prefill first if possible
            ret = new_batch
        else:
            # Run decode
            if not self.running_batch.is_empty():
                self.running_batch = self.update_running_batch(self.running_batch)
                ret = self.running_batch if not self.running_batch.is_empty() else None
            else:
                ret = None

        return ret

被truncated的req从last_batch里清掉，后面scheduler会再次把req加入到某个batch中

self.last_batch.filter_batch(...)：从last_batch清理self.chunked_req
self.tree_cache.cache_unfinished_req(...)：已经处理的tokens产生的k/v保存在kvcache，上次处理的extend tokens的index缓存到RadixCache中；对于切分chunk的req，req数据结构始终是同一个，但是每次被scheduler调度执行分配到的req_pool_idx是不同的。req的prefix对应kv在kvcache位置是通过查Radix Cache获得的，extend部分kv是在ScheduleBatch::prepare_for_extend调用token_to_kv_pool_allocator分配的。然后调用triton kernel，把prefix和extend部分对应kvcache index写到新的req_pool_index位置的self.req_to_token_pool.req_to_token中。
self.req_to_token_pool.free(...)：如上所述，self.req_to_token_pool.req_to_token表仅在执行时维护，当前batch执行结束后，req_pool_index对应的元信息无意义了，下次调用会重新分配req_pool_index, 重新填写self.req_to_token_pool.req_to_token表格。

我们可以再回看一下get_new_batch_prefill。上次被truncated的req优先处理。为什么?

chunked prefill只是把extend部分切分chunk来执行，跟一般的prefill区别不大。prefill没有执行完不会执行decode，那么req也不会处理完成；已经处理过的tokens（query）占用的kvcache也不会释放。

显存分配：快速被处理
req e2e延迟：快速被处理

上面也分析过chunked prefill size是全局生效的限制，batch中所有req的extend tokens总和不能超过这个限制。而且，truncated的req比waiting_queue中的任何请求排序更靠前，优先被处理是合理的。
