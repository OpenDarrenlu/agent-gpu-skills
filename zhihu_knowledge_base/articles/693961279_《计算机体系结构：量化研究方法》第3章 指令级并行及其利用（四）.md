3.7 使用多发射和静态调度利用ILP/Exploiting ILP Using Multiple Issue and Static Scheduling

前面的小节用于消除数据和控制的停滞，实现了理想情况下CPI（clock per instruction）为1。如果为了实现CPI小于1，意味着我们每个周期需要发送大于1条指令。

多发射处理器主要有3种类型：

静态调度的超标量处理器。（顺序执行）
VLIW（超长指令）处理器。（发射将固定数量的指令，通过一条大的指令或者固定格式的指令包，同时明确指令间的并行。由编译器静态调度）
动态调度的超标量处理器。（乱序执行）

静态调度的超标量处理器每个时钟周期发射的指令数量是可变的，在概念上接近VLIW，这两种方法都依赖于编译器来为处理器调度代码。随着发射宽度的增加，静态调度的超标量的优势逐渐减弱，通常只发射两条指令。

图3.19总结了多发射的基本方法及其特点，并展示了使用每种方法的处理器。

图3.19展示了多发射处理器使用的五种主要方法以及区分它们的主要特点。
基本的VLIW方法

VLIW采用多个独立的功能单元，但是不是发射多个独立的指令到单元内，而是，将多个操作打包到一个很长的指令或者要求指令满足相同的约束。

VLIW的优势随着最大发射率的增加而增加，需要关注发射宽度更宽的处理器，但是开销的增长是限制更宽发射处理器的主要因素之一。

假设VLIW处理器的指令包含5个操作（1个整数或分支，2个浮点，2个内存访问）。该指令对于每个功能单元都有一组字段，每个单元可能占16到24比特，使得指令长度在80到120比特之间。

为了保持功能单元繁忙，代码序列中必须有足够的并行性来填充可用的操作位置，可以通过展开循环和调度单个更大循环主体的代码来发现。

如果展开生成直线性的代码（对于单个基础block的操作），可以采用局部调度（local scheduling）。
如果查找和利用并行性需要跨分支调度代码，则需使用更为复杂的全局调度（global scheduling）。全局调度算法不仅在结构上更复杂，而且还必须处理更加复杂的优化权衡。

还是基于之前的这个例子，假设VLIW处理器的指令包含5个操作（1个整数或分支，2个浮点，2个内存访问）。通过多次展开消除停滞。

图3.20展示了VLIW循环展开的示例，7个循环的主体并行执行。

图3.20 VLIW指令示例

对于原始的VLIW模型，存在技术和协调上的问题，使得效率变低。技术问题包括代码大小的增加和lockstep操作的限制。导致VLIW的代码大小大幅增加有2个原因：

在直线代码片段中生成足够的操作需要尽可能展开循环，导致代码增加。
当指令不满时，未使用的功能单元会导致指令编码的浪费。

为了应对代码增加，会使用巧妙的编码。（1）设计一个大的立即数字段提供给任意功能单元使用。（2）主存中压缩指令，在读到缓存或解码时进行展开。

3.8 使用动态调度、多发射和推测来利用ILP/Exploiting ILP Using Dynamic Scheduling, Multiple Issue, and Speculation

为简单起见，本节只考虑每个时钟周期发射两条指令的情况。假设需要扩展Tomasulo算法，以支持具有独立整数、加载/存储和浮点单元（包括浮点乘法和浮点加法）的多发射超标量流水结构，每个单元都可以在每个时钟周期开始一个操作。指令按照顺序发射到保留站。允许流水线在一个时钟周期内发射任意两条指令，使用硬件调度整数和浮点单元的操作。

如图3.21所示，基本组织结构与具有预测的单发射处理器类似，但是发射和完成逻辑必须增强，从而允许每个时钟周期处理多条指令。

图3.21 带有预测执行的多发射处理器基本组织结构。为了支持多发射，需要扩展几个数据通路：CDB、操作数总线以及指令发射逻辑（在图中未显示）

在动态调度处理器中每个时钟周期发射多条指令非常复杂，因为多个指令之间可能存在依赖关系。

有两种不同的方法用于每个时钟周期发射多条指令：

半个时钟周期完成处理，使得一个时钟周期内可以发射两条指令。这种方法不易拓展到采用更多发射的处理器。
构建处理两个或多个指令的必要逻辑，包括处理指令之间可能存在的依赖关系。

超标量处理器每个时钟周期发射四条或更多指令时可能会同时采用这两种方法：既采用流水线又扩展发射逻辑。但是不能简单地通过流水线解决这个问题，由于每个时钟周期都会发射新的指令，如果使指令发射需要多个时钟周期，必须能够分配保留站并更新流水线表格，以便下一个时钟周期发射的依赖指令可以使用更新的信息。

发射是动态调度超标量处理器中最基本的瓶颈之一，必须考虑每种可能的在同一个时钟周期内发射指令间的依赖关系。

在发射宽度为n的动态调度超标量处理器中更新发射逻辑和保留表的基本策略：

为可能在下一次发射的每条指令分配一个保留站和ROB。分配可以在知道指令类型之前完成，不考虑指令内容，按顺序为指令预分配n个ROB，并确保发射时有足够的保留站。通过限制给定类别的指令数量（例如，一个浮点指令，一个整数指令，一个加载指令，一个存储指令），可以预先分配必要的保留站。如果没有足够的保留站可用，按照原始程序顺序发射部分指令，剩余的指令在下一次发射。
分析所有要发射指令之间的依赖关系。
如果在一组发射指令中，有一条指令依赖于较早指令，则使用分配的ROB序号更新依赖指令的保留站。否则，使用现有保留站和ROB信息来更新发射指令的保留站。

3.9 指令传输和推测的高级技术/Advanced Techniques for Instruction Delivery and Speculation

在高性能的多发射流水线中，仅预测分支是不够的，需要能够传输高带宽指令流。在多发射处理器中，意味着每个时钟周期需要传输多条指令。

我们首先来看一下增加指令传递带宽的方法。然后，我们将转向在实现先进的预测技术时的一系列关键问题，包括使用寄存器重命名与重新排序缓冲区、预测的激进程度以及一种称为值预测的技术，该技术试图预测计算的结果，并进一步增强指令级并行性。

增加指令抓取带宽

一个多发射处理器要求每个时钟周期抓取的平均指令数量至少与平均吞吐量相同。获取这些指令需要足够宽的路径到指令缓存，但最困难的方面是处理分支。

在这个小节中，将探讨两种处理分支的方法，然后讨论现代处理器如何集成指令预测和预取功能。

分支目标buffer

为了减少流水线中的分支开销，必须知道尚未解码的指令是否是分支指令。如果是分支，如果知道下一个PC值是什么，可以将分支开销降为零。

存储分支后指令的预测地址的分支预测缓存，称为分支目标buffer或分支目标缓存，如图3.25所示。分支目标buffer预测下一条指令的地址，并在解码指令之前将其发送出去。

正在抓取指令的PC与存储在第一列中的指令地址进行匹配，这些地址表示已知分支的地址。如果PC与其中一个匹配，表示正在抓取的指令就是一条已经发生的分支，从第二列包含的分支PC的预测值立即进行抓取。因为只需要在分支目标buffer中存储预测执行的分支，所以如果没匹配上，表示预测不执行或没有预测执行的记录，而未执行的分支和正常流程一样抓取下一条顺序指令。

图3.25 一个分支目标buffer

如图3.26所示，显示了在简单的五级流水线中使用分支目buffer的步骤。如果在buffer中找到匹配的PC值，那么就不会有分支延迟；否则，至少会有两个时钟周期的开销。

图3.26 一个带有分支目标buffer的处理步骤

分支目标buffer的一个变体是存储一个或多个目标指令，而不是预测的目标地址，这个方法有两个潜在的优点。

允许分支目标buffer的访问时间比连续指令抓取的时间长，从而允许更大尺寸的分支目标buffer。
暂存实际的目标指令允许实现一种称为分支折叠（branch folding）的优化方法。分支折叠可以用于无延迟（0个时钟周期）的无条件分支或条件分支。
专用分支预测器：预测程序返回、间接跳转和循环分支

PS：中间省略一段。主要讲在程序运行过程中常常涉及到大量间接跳转。虽然可以使用分支目标buffer来预测过程返回，但如果从多个位置调用且单一个位置的调用时间不聚集，这种预测的准确性可能较低。可以采用一种的返回地址预测的buffer进行优化。

集成式指令抓取单元

为了满足多发射处理器的需求，设计者通过一个独立的自主单元实现集成式指令抓取单元，将指令发送给流水线的其余部分。鉴于多发射的复杂性，将指令抓取描述为一个简单的单个流水线阶段的不再有效。

集成式指令提取单元，集成了以下几个功能：

分支预测：分支预测器成为指令抓取单元的一部分，持续预测分支，以驱动指令抓取流水线。
指令预抓取：为了每个时钟周期传递多条指令，指令获取单元可能需要向前抓取。该单元会自主管理指令的预取，与分支预测集成在一起。
指令内存访问和暂存：在每个周期中获抓取多条指令时，会遇到各种复杂性，包括获取多条指令可能需要访问多个缓存行的困难。指令获取单元封装了这种复杂性，使用预取来尝试隐藏跨缓存块的成本。指令获取单元还提供缓冲，基本上充当按需单元，根据需要和数量向发布阶段提供指令。
