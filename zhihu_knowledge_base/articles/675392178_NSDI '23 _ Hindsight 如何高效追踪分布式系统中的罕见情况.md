分布式追踪，采样和罕见情况

想象你有成百上千个软件程序（如微服务 micro-services 等）部署在数以百千计的服务器上，一个用户请求（user request）就可能造访几十台机器。这种情况下，任何一台机器或一个网络连接出现了某些问题，或是在应用的某个部分有一些异常行为都有可能导致用户的请求失败或者延迟升高。在发现症状后，我们应该如何诊断，观测和理解我们的系统的行为和性能？

分布式追踪 (distributed tracing) 旨在解决这样一个问题。其原理可以简单地理解为在程序中插入日志收集语句 (instrumentation)，通过收集和分析这些取自不同进程的事件记录和系统状态等信息，构建出每个请求的整个足迹，来帮助工程师抽丝剥茧地理解软件和系统行为，找到问题根源。

然而追踪自有其代价，追踪指令往往在关键路径上（critical path，data plane），本身即会造成性能损耗。而随着规模的扩大，更大的问题也出现在对海量 trace 数据的处理，传输与储存上。想象对于一个每秒百万请求的服务而言，假设每个请求平均造访十个子服务，每个子请求即使是仅仅记录其开始和结束的时间戳，加上请求识别符等必须信息，每秒所产生的数据也在 GBytes 量级。更何况为了服务于多样的需求，甚至单个请求的端到端 trace 都可能达到MBytes 量级，这将使每秒数据产生量达到 TBytes 量级。这意味着仅仅是为了收集和处理 trace，我们也需要部署相当的计算，IO和网络资源。

应对这样的挑战，最常见的方法是对于所来的请求以一个较低的比例进行随机化的采样，仅对选中的请求进行追踪和记录。由于采样的决定往往发生在请求伊始，因此也叫 head sampling，我们称之为开头采样。采样率常常在万分之一以下，这样可以大幅降低观测所带来的开销，随机采样也一定程度上保证了观测结果统计上的代表性，可以满足大多数对于系统观测的需求。

然而这一种事实标准与一个重要的场景有根本性的矛盾，那就是对于罕见情况（edge-cases）的追踪。这里的罕见可以有很多种内涵，譬如说突发的硬件错误，网络故障，甚至是程序本身的 bug，造成高尾延迟的异常请求等等，但是根本特征是发生概率低。朴素的数学会告诉我们两个很小的概率相乘会得到一个小得多的概率。万分之一的采样率如果搭配上万分之一的发生概率，即使是每秒百万次的请求，也需要几分钟才能偶尔捕捉到零星几个异常情况，且绝大多数都以遗失。

想要采样方法对于罕见情况分析依然有效，需要较高的采样率，较高的罕见情况发生概率，以及一些运气。为了解决这一问题，社区中最近也出现了tail sampling（末尾采样）的方法，其想法是对于所有请求皆进行追踪收集 trace，但是在请求结束后对 trace 进行分析来决定要不要将其储存。这种方法事实上等价于使用一个更高的开头采样率再叠加上一个编码一些规则的过滤器。尽管其有效地优化了存储有效 trace 的开销，但生成，处理和传输海量请求的 trace 的开销依然使得对于罕见情况的追踪代价巨大，难以扩展因此不实用。

Hindsight 和追溯采样

在我们发表在NSDI ‘23的工作中，我们重新设计了对于罕见情况进行采样追踪的框架并将其命名为 Hindsight（后见之明），过程中我们挖掘出了几条核心洞见：

如果要想更好地捕捉罕见情况，理论上大量的 trace 数据生成是无可避免的，毕竟我们无法在请求执行前预知其是否会成为一个罕见病例，而往往是在请求执行过程中，甚至几乎结束时才能做出判断。然而和通常认知可能有些不同的一点是，生成大量数据本身不一定会非常昂贵，但是对于大量数据的处理才是。一些比较新的技术如 Intel PT可以做到每个核心每秒生成高达100-200MB的观测数据而仅仅带来5%-15%的运行时开销。当下流行的开源框架如 OpenTelemetry[1] 在关键路径/数据平面上耦合了许多功能，使得用户形成了一种收集数据越多，开销就越大的错觉。
单个请求的症状往往是本地即可观测到的。在末尾采样中，所有收集到的 trace 会被统一发送到一个单独的服务或者机器上来做聚合处理，通过用户所设立的规则来对逐一决定是否要留存某个请求的 trace 数据。用户常常设立的规则就包括有诸如尾延迟，error code（错误码）等。将一个请求的所有 trace 数据聚拢以后可以重建一个请求的全貌，然后再来做判断是一个很直接的想法。而事实上，对于前述提到的几个症状甚至是绝大多数症状，不需要收集到所有 trace，只是在运行时就能够很轻易的发现和做出采样的决策。我们认为检测与收集两个部分应当解耦（decouple），正确的做法应当是检测到症状来激活对于 trace 的收集，而不应当是反过来收集到 trace 以后再去检测症状。

我们在 hindsight 中引入了追溯采样（retroactive sampling）的概念，大概的工作流程是：

在程序运行过程中，持续不断地以低开销往内存环形缓冲区（ring buffer）中写入 trace 数据，但是不对齐进行处理。
在应用中嵌入可编程的触发器（trigger），一旦在本地生成的 trace 中检测到了症状就当即将改请求信息上报给协调服务器（coordinator），协调服务器之后会定位到所有该请求造访的机器让其将相关的 trace 上报到后端的收集器（collector）中。
一直没有被触发收集的 trace 数据在本地将会逐步过期而被覆盖（overwrite），从生成到被覆写的这个时间窗口我们称之为事件视界（event horizon），取决于本地缓存 trace 数据区域的内存大小。

得益于追溯采样的设计，我们在对于罕见情况进行追踪这一问题上取得了本质的进展。Hindsight 运行的开销仅相当于以1%比率进行的开头采样，却捕获了比其多近乎两个数量级的罕见请求 trace（1% v.s. 99.9%）。即使末尾采样在低请求率下依然运行良好，因为其尚未过度消耗计算资源，但是当我们将请求率持续提高以后很容易能看到末尾采样高开销的后果：大幅提升了用户请求的延迟，降低了吞吐量，又因为其事件视界的限制，所成功捕获到的罕见请求 trace 比例也大幅下降，相比之下 hindsight 表现出了很好的可扩展性和低几乎两个数量级的开销。在论文中我们也对于 hindsight 的各个组件进行了独立的性能测试。

一点幕后故事

22年初因为回国航班被熔断我滞留在德国，跟当时在马普所的导师 Jonathan 在酒吧聊到他这个工作，我很被这个问题设定吸引，于是加入帮忙完善实验和优化一点实现细节。尽管大家都爱讲不要做 incremental 的工作，但是即使是顶会的文章也主要以比较 incremental 的想法为主。Hindsight 一开始也有被审稿人质疑其是否仅是一种对于末尾采样的优化，但其实末尾采样出现在社区中比 hindsight 被设计出来的时间还要晚[2]。我觉得这恰恰体现了这项研究的价值：业界社区中对于问题的理解和迭代框架的开发往往是演进式的（evolutionary），在现有的框架下引入尾部采样作为一个 post-processing 的方法是一个很自然的选择，其后续碰上扩展性问题的解决方案也是设计更多的收集器来做负载均衡[3]。而 hindsight 在一开始就是往变革性的方向在努力，确立下基本的限制后（trace 数据必须要生成等），重新设计了完全不同的机制（追溯采样）。彼时我还没开始读PhD，很倾慕 Jonathan 的研究风格。现在我开始 PhD 已经一年了，终于把这篇一直卡了一半的博客写了，也作为对现在自己的一种勉励。

既然都已经写到这了就按照惯例打一点广告，位于德国小镇萨尔布吕肯的马普所软件系统研究所和计算机科学研究所在国内的名声仿佛沧海遗珠，但所里的老师们大多是领域里第一流的研究员，在今年还主办了领域内旗舰会议 SOSP[4]，常年招收 PhD 和研究型硕士生，大家在升学时也请多加以考虑。
