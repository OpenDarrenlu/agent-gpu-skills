文章开头先惯例回顾一下我们的定义的5层模型：

1.应用层：包括各种图形应用，如游戏、模拟器、设计工具等。这些应用使用下层的服务和功能，以创建各种各样的图形和交互体验。

2.跨平台层：这一层包括各种图形引擎和工具，如Unity、Unreal Engine、bgfx等。这些引擎和工具除了提供跨平台的能力之外，往往还会提供了一组高级的功能，如物理模拟、光照计算、动画系统等，使得开发者可以更容易地创建复杂的图形应用。

3.用户态图形驱动层(本章内容)：这一层包括操作系统和图形API（如OpenGL和DirectX）。操作系统负责管理硬件资源，如内存和CPU时间，以及提供基本的服务，如文件系统和网络。图形API则提供了一组函数和数据结构，使得开发者可以更容易地使用图形硬件的功能。

4.内核态驱动层：驱动层是硬件层和操作系统之间的接口。负责管理硬件设备，如初始化设备、处理设备错误、将操作系统的命令转换GPU件可以执行的命令等。

5.硬件层：这是计算机图形系统的最底层，包括图形处理器（GPU）、显示器、输入设备等硬件设备。GPU负责执行大量的并行计算，以快速生成图形。显示器则负责将生成的图形显示给用户。输入设备，如鼠标和键盘，允许用户与图形系统交互。

1 图形编程的前世今生

在探索用户态图形驱动之前，我们很有必要了解一下图形编程的发展历程，这样可以最直观地感受图形API所解决的问题。

1.1 历史上第一个图形设计系统

第一个计算机图形设计系统是 Evan Sutherland 于 1963 年在麻省理工学院作为他的博士论文开发的。它被称为Sketchpad，允许用户在计算机屏幕上绘制机械零件的草图。

https://web.archive.org/web/20130408133119/http://stinet.dtic.mil/cgi-bin/GetTRDoc?AD=AD404549&Location=U2&doc=GetTRDoc.pdf

Evan Sutherland也因此在1988年获得了图灵将，它开创了人机交互的先河，被认为是现代计算机辅助设计程序（CAD）的鼻祖。与此同时，计算机图形学从那时起就作为一种"计算机生成图像的事物"诞生了！

1.2 图形API的诞生(70年代)

在早期，如果你想创建计算机图形，你必须编写直接与图形硬件对话的程序。随着新硬件设计的诞生，软件必须完全重写才能与新硬件配合使用。

于是在70年代，出现了图形API的雏形，许多“图形库”被创建来解决这个问题。但大多数库都还是只能泡在某个操作系统上，无法轻松移植到新操作系统或新硬件。

到了80 年代，ISO（国际标准化组织）创建了一个名为PHIGS的标准计算机图形库。目的是解决跨所有计算平台创建计算机图形的问题。

20 世纪 80 年代末和整个 90 年代首屈一指的计算机图形公司是Silicon Graphics,Inc。他们是一家硬件公司，但他们的计算机会配备专有的计算机图形应用程序编程接口 (API)，这个API被称为 IRIS 图形语言 (IRIS GL)。1992 年 Silicon Graphics 正式发布了他们的图形库标准，OpenGL1.0就此诞生。 这意味着程序员可以更加轻易地编写快速、高效、跨平台的图形程序。

1.3 GPU的诞生(90年代)

OpenGL是一个图形API，它本身并不直接执行渲染，而是提供了一套接口，让开发者能够编写代码来控制图形硬件（如GPU）进行渲染。因此，OpenGL可以用于硬件加速（GPU）渲染，也可以用于软件渲染，这主要取决于具体的实现和硬件环境。

如果系统中有支持OpenGL的图形硬件（如GPU），并且安装了相应的驱动程序，那么OpenGL通常会利用这些硬件进行加速渲染。这种情况下，OpenGL的大部分工作（如几何变换、光照计算、纹理映射等）都会在GPU上执行，从而大大提高渲染性能。

如果系统中没有支持OpenGL的图形硬件，或者没有安装相应的驱动程序，那么OpenGL可能会退化为软件渲染模式。在这种情况下，OpenGL的所有工作都会在CPU上执行，渲染性能会比硬件加速模式低很多。

1981年：IBM的彩色图形适配器（CGA）是最早的图形卡之一，它为 IBM 的个人电脑提供了彩色图形。GPU的诞生主要是为了解决计算机图形处理的需求。在早期的计算机系统中，所有的计算任务，包括图形渲染，都是由中央处理单元（CPU）来完成的。然而，随着计算机图形的复杂性和需求的增加，CPU 在处理图形任务时开始显得力不从心。

1999年：NVIDIA的GeForce 256被广泛认为是第一款真正的 GPU。它不仅可以处理 3D 图形，还可以处理转换和光照等计算密集型任务。图形渲染，特别是3D图形渲染，需要大量的并行计算。例如，渲染一帧图像可能需要计算数百万个像素的颜色和亮度。这些计算任务之间没有依赖关系，可以同时进行。然而，CPU 是为处理串行计算任务设计的，它在处理并行计算任务时效率不高。因此，为了提高图形渲染的效率，研究人员开始开发专门的硬件来处理图形任务。这就是图形处理单元（GPU）的起源。GPU 是为处理并行计算任务设计的，它有数百或数千个处理核心，可以同时处理大量的计算任务。因此，GPU 在处理图形渲染等并行计算任务时，比 CPU 更加高效。

2 交错复杂的GPU驱动体系
2.1 GPU的驱动程序

GPU驱动，或者说广义的GPU驱动，是一个很庞大又互相交织耦合的若干工作在不同层次的模块的集合。关于GPU图形驱动，需要明确两点：

显卡驱动一般分为用户模式驱动程序（User Mode Driver，UMD）和内核模式驱动程序（Kernel Mode Driver，KMD）。
一般的驱动程序都是在内核态完成工作，但是GPU驱动的很大一部分工作都是工作在用户态的(划重点)。
2.2 按功能对显卡驱动的横向划分。

GPU的驱动，从横向上划分可以分为：

display 基本的多屏/屏幕分辨率/屏幕刷新率相关控制能力。
render 硬件渲染加速，D3D/openGL相关。
encode/decode 硬件视频编解码加速
general computing 通用计算，CUDA/openCL相关。

也就是说，你需要使用GPU的哪种能力，就选择其对应的驱动程序即可。

2.3 render驱动的纵向分层划分
linux图形渲染驱动

以Intel核显在linux上的开源图形驱动为例，render的驱动从纵向上划分可以分为：

i915/i810：跟芯片族(chip core)绑定的内核驱动 负责跟GPU硬件交互
drm：linux内核渲染驱动，dri直接渲染框架的核心部分
libdrm：drm的用户态接口，对drm的简单封装
mesa3D i965：openGL的开源实现 用户态驱动
bgfx/unity/ue4：在 D3D/openGL/Vulkun等图形api上再封装了一层，实现跨平台的渲染能力

用户态图形驱动就像是一个翻译器，开发者用他们熟悉的语言（API）来编写代码，然后API将这些代码翻译成硬件可以理解的语言（硬件指令）。这样，开发者就可以更容易地利用图形硬件的功能，而不需要关心底层的实现细节。

例如，一个开发者可能想要在屏幕上绘制一个三角形。如果没有图形API，他可能需要直接编写硬件指令，这需要对硬件的工作原理有深入的了解。而如果有了图形API，例如OpenGL或DirectX，他只需要调用API提供的函数，例如glDrawArrays或DrawPrimitive，API会处理剩下的工作，包括生成硬件指令、管理内存等。

图形API的本质实际上只是一个标准，它需要一个实体的库来实现和完成这些标准。

例如OpenGL是一种规范，而Mesa3D是这种规范的一个实现。这种关系类似于Java语言和Java虚拟机（JVM）的关系：Java语言定义了一种编程语言的语法和语义，而JVM是这种语言的一个实现。

又或者类比于C++98/C++11/C++ 14这些标准，3大编译器可能都会有不同的具体实现，但是作为开发者一般是不无需要关心的，作为程序员需要对标准和实现这种解耦的概念有比较清晰的认知。

4 用户态图形驱动架构(以OpenGL为例)
4.1 OpenGL的前世今生

OpenGL（Open Graphics Library）是一个用于渲染2D和3D图形的跨平台、跨语言的编程接口（API）。它最初由Silicon Graphics（SGI）在1992年开发，目的是创建一个硬件无关的图形接口。OpenGL发展历程反映了计算机图形学的发展历程。

随着图形硬件技术的发展，OpenGL也在不断演进。例如，OpenGL 2.0引入了可编程着色器，使得开发者可以更灵活地控制图形渲染过程。OpenGL 3.0和4.0则进一步增加了对新硬件特性的支持，如几何着色器、计算着色器等。

4.2 OpenGL的工作过程

OpenGL是由一个非盈利的技术联盟Khronos Group开发和维护的。Khronos Group的成员包括许多硬件制造商，如NVIDIA、AMD和Intel，以及一些软件公司，如Apple和Google。这些公司的工程师共同参与了OpenGL的开发。

当Khronos Group开发OpenGL时，他们不会直接编写GPU指令。相反，他们会定义一组函数和数据结构，这就是OpenGL API。然后，每个硬件制造商会为他们的GPU编写一个驱动程序，这个驱动程序会实现OpenGL API，并将API的调用转换为GPU指令。

比如当你调用glDrawArrays函数时，你的程序实际上是在调用GPU驱动程序提供的一个函数。GPU驱动程序的开源情况因厂商和产品而异。有些GPU驱动程序是开源的，而有些是闭源的：

NVIDIA：NVIDIA的官方驱动程序（称为NVIDIA CUDA Toolkit）是闭源的。但是，有一个名为Nouveau的社区驱动项目，它提供了一些NVIDIA GPU的开源驱动。然而，Nouveau驱动的性能和功能通常不如官方驱动。
AMD：AMD的GPU驱动程序有两个版本。一个是闭源的AMDGPU-PRO驱动，另一个是开源的AMDGPU驱动。AMDGPU驱动是基于Mesa 3D图形库的，它支持OpenGL和Vulkan API。AMDGPU驱动在许多Linux发行版中已经是默认的驱动程序。
Intel：Intel的GPU驱动程序通常是开源的。他们的开源驱动程序也是基于Mesa 3D图形库的，支持OpenGL和Vulkan API。在Linux系统上，Intel的开源驱动程序通常是默认的驱动程序。
4.3 OpenGL与Mesa3D
Mesa3D支持的标准API

OpenGL的本质实际上只是一个标准，它需要一个实体的库来实现和完成这些标准。

Mesa3D，是一个开源的图形渲染库，它实现了OpenGL的API。也就是说，Mesa3D提供了一组函数，这些函数的行为符合OpenGL规范的要求。因此，你可以使用Mesa3D来运行使用OpenGL API编写的程序。

简单来说，OpenGL是一种规范，而Mesa3D是这种规范的一个实现。这种关系类似于Java语言和Java虚拟机（JVM）的关系：Java语言定义了一种编程语言的语法和语义，而JVM是这种语言的一个实现。

又或者类比于C++98/C++11/C++ 14这些标准，3大编译器可能都会有不同的具体实现，但是作为开发者一般是不无需要关心的，作为程序员需要对标准和实现这种解耦的概念有比较清晰的认知。

5 Mesa3D源码分析

我们以支持openl4.4标准的mesa3D18.0版本为例，对mesa3D的源码进行分析。

5.1 mesa3D的前世今生

Mesa 3D是一个由许多贡献者共同开发的开源项目。项目的创始人是Brian Paul，他在1993年开始开发Mesa，最初的目标是实现一个完全软件渲染的OpenGL兼容库。

再重复一遍，OpenGL只是一个标准，你可以用任何图形硬件或者CPU纯软件的方式来实现它。

随着时间的推移，许多其他的开发者和组织也参与到了Mesa的开发中，包括但不限于Intel、AMD、NVIDIA、VMware等公司的员工，以及许多独立的开源贡献者。这900多位开发者共同开发和维护了Mesa，使其成为了现在支持多种硬件和图形API的强大图形库。

5.2 mesa3D架构分析

Mesa 3D 的源码架构可以分为以下几个主要部分：

API 实现层：这一层实现了 OpenGL、OpenGL ES 和 Vulkan 等图形 API 的功能。它将应用程序的图形 API 调用转换为 Mesa 3D 内部的数据结构和操作。
Mesa 核心：Mesa 核心是整个库的核心部分，它包含了实现图形渲染所需的各种算法和数据结构。这些算法和数据结构被封装成一组通用的接口，供不同的硬件驱动程序使用。
硬件驱动程序：Mesa 3D 支持多种不同的硬件平台，包括 NVIDIA、AMD 和 Intel 等。为了适应不同的硬件平台，Mesa 3D 提供了一组硬件驱动程序。这些驱动程序负责将 Mesa 核心的通用接口转换为特定硬件平台的操作。
Gallium3D：Gallium3D 是 Mesa 3D 的一个子项目，它提供了一个更简洁、更灵活的驱动程序架构。Gallium3D 的目标是简化驱动程序的开发，提高驱动程序的可重用性。许多现代硬件驱动程序都是基于 Gallium3D 架构开发的。
辅助库和工具：Mesa 3D 还包含了一些辅助库和工具，如 GLSL 编译器、SPIR-V 编译器和调试工具等。这些库和工具用于支持 Mesa 3D 的开发和调试。

再次重复一遍，mesa3D的核心逻辑实际上都是在GPU厂商开发的硬件驱动程序中。这些驱动程序负责将 Mesa 核心的通用接口转换为特定硬件平台的操作。

这些图形API的具体实现基本都是抽象工厂这种设计模式，mesa3D的API 实现层和Mesa 核心向上对接OpenGL的标准接口，然后根据运行时的情况选择和创建不同的渲染上下文和GPU硬件驱动。

在Mesa3D中，OpenGL的初始化通常在main/context.c文件的_mesa_initialize_context函数中进行。这个函数会初始化一个struct gl_context结构体，这个结构体包含了OpenGL的所有状态。

初始化过程通常包括以下几个步骤：

分配和初始化struct gl_context结构体。
初始化默认的OpenGL状态，例如颜色缓冲区的清除颜色，深度缓冲区的清除深度，视口大小等。
初始化各种表格，例如纹理对象表，缓冲区对象表等。
初始化各种驱动函数，这些函数是硬件特定的，用于执行实际的渲染操作。
初始化扩展功能，例如加载OpenGL扩展函数，设置扩展字符串等。

整个OpenGL都是围绕 gl_context这个核心结构体展开的：

/**
 * Mesa rendering context.
 *
 * This is the central context data structure for Mesa.  Almost all
 * OpenGL state is contained in this structure.
 * Think of this as a base class from which device drivers will derive
 * sub classes.
 */
struct gl_context
{
   /** State possibly shared with other contexts in the address space */
   struct gl_shared_state *Shared;

   /** \name API function pointer tables */
   /*@{*/
   gl_api API;

   /**
    * The current dispatch table for non-displaylist-saving execution, either
    * BeginEnd or OutsideBeginEnd
    */
   struct _glapi_table *Exec;
   /**
    * The normal dispatch table for non-displaylist-saving, non-begin/end
    */
   struct _glapi_table *OutsideBeginEnd;
   /** The dispatch table used between glNewList() and glEndList() */
   struct _glapi_table *Save;
   /**
    * The dispatch table used between glBegin() and glEnd() (outside of a
    * display list).  Only valid functions between those two are set, which is
    * mostly just the set in a GLvertexformat struct.
    */
   struct _glapi_table *BeginEnd;
   /**
    * Dispatch table for when a graphics reset has happened.
    */
   struct _glapi_table *ContextLost;
   /**
    * Dispatch table used to marshal API calls from the client program to a
    * separate server thread.  NULL if API calls are not being marshalled to
    * another thread.
    */
   struct _glapi_table *MarshalExec;
   /**
    * Dispatch table currently in use for fielding API calls from the client
    * program.  If API calls are being marshalled to another thread, this ==
    * MarshalExec.  Otherwise it == CurrentServerDispatch.
    */
   struct _glapi_table *CurrentClientDispatch;

   /**
    * Dispatch table currently in use for performing API calls.  == Save or
    * Exec.
    */

比如OpenGL里 glClear的调用，其实际路径是：

glFlush ----> _mesa_flush----> gl_context->Driver.Flush-->intel_glFlush

intel_glFlush就是Intel显卡的具体实现，mesa3D里还集成了若干个其它GPU厂商实现的XX_Flush方法，根据初始化时候的绑定信息来决定调用到哪一个。

继续展开intel_batchbuffer_flush()方法，实际是绑定到一个叫_intel_batchbuffer_flush_fence()的方法

int
_intel_batchbuffer_flush_fence(struct brw_context *brw,
                               int in_fence_fd, int *out_fence_fd,
                               const char *file, int line)
{
   int ret;

   if (USED_BATCH(brw->batch) == 0)
      return 0;

   /* Check that we didn't just wrap our batchbuffer at a bad time. */
   assert(!brw->batch.no_wrap);

   brw_finish_batch(brw);
   intel_upload_finish(brw);

   finish_growing_bos(&brw->batch.batch);
   finish_growing_bos(&brw->batch.state);

   if (brw->throttle_batch[0] == NULL) {
      brw->throttle_batch[0] = brw->batch.batch.bo;
      brw_bo_reference(brw->throttle_batch[0]);
   }

   if (unlikely(INTEL_DEBUG & (DEBUG_BATCH | DEBUG_SUBMIT))) {
      int bytes_for_commands = 4 * USED_BATCH(brw->batch);
      int bytes_for_state = brw->batch.state_used;
      fprintf(stderr, "%19s:%-3d: Batchbuffer flush with %5db (%0.1f%%) (pkt),"
              " %5db (%0.1f%%) (state), %4d BOs (%0.1fMb aperture),"
              " %4d batch relocs, %4d state relocs\n", file, line,
              bytes_for_commands, 100.0f * bytes_for_commands / BATCH_SZ,
              bytes_for_state, 100.0f * bytes_for_state / STATE_SZ,
              brw->batch.exec_count,
              (float) brw->batch.aperture_space / (1024 * 1024),
              brw->batch.batch_relocs.reloc_count,
              brw->batch.state_relocs.reloc_count);
   }

   ret = submit_batch(brw, in_fence_fd, out_fence_fd); //核心调用

   if (unlikely(INTEL_DEBUG & DEBUG_SYNC)) {
      fprintf(stderr, "waiting for idle\n");
      brw_bo_wait_rendering(brw->batch.batch.bo);
   }

   /* Start a new batch buffer. */
   brw_new_batch(brw);

   return ret;
}


这里继续调用了 submit_batch() 这个方法：

static int
submit_batch(struct brw_context *brw, int in_fence_fd, int *out_fence_fd)
{
   const struct gen_device_info *devinfo = &brw->screen->devinfo;
   __DRIscreen *dri_screen = brw->screen->driScrnPriv;
   struct intel_batchbuffer *batch = &brw->batch;
   int ret = 0;

   if (batch->use_shadow_copy) {
      void *bo_map = brw_bo_map(brw, batch->batch.bo, MAP_WRITE);
      memcpy(bo_map, batch->batch.map, 4 * USED_BATCH(*batch));

      bo_map = brw_bo_map(brw, batch->state.bo, MAP_WRITE);
      memcpy(bo_map, batch->state.map, batch->state_used);
   }

   brw_bo_unmap(batch->batch.bo);
   brw_bo_unmap(batch->state.bo);

   if (!brw->screen->no_hw) {
      /* The requirement for using I915_EXEC_NO_RELOC are:
       *
       *   The addresses written in the objects must match the corresponding
       *   reloc.gtt_offset which in turn must match the corresponding
       *   execobject.offset.
       *
       *   Any render targets written to in the batch must be flagged with
       *   EXEC_OBJECT_WRITE.
       *
       *   To avoid stalling, execobject.offset should match the current
       *   address of that object within the active context.
       */
      int flags = I915_EXEC_NO_RELOC;

      if (devinfo->gen >= 6 && batch->ring == BLT_RING) {
         flags |= I915_EXEC_BLT;
      } else {
         flags |= I915_EXEC_RENDER;
      }
      if (batch->needs_sol_reset)
         flags |= I915_EXEC_GEN7_SOL_RESET;

      uint32_t hw_ctx = batch->ring == RENDER_RING ? brw->hw_ctx : 0;

      /* Set statebuffer relocations */
      const unsigned state_index = batch->state.bo->index;
      if (state_index < batch->exec_count &&
          batch->exec_bos[state_index] == batch->state.bo) {
         struct drm_i915_gem_exec_object2 *entry =
            &batch->validation_list[state_index];
         assert(entry->handle == batch->state.bo->gem_handle);
         entry->relocation_count = batch->state_relocs.reloc_count;
         entry->relocs_ptr = (uintptr_t) batch->state_relocs.relocs;
      }

      /* Set batchbuffer relocations */
      struct drm_i915_gem_exec_object2 *entry = &batch->validation_list[0];
      assert(entry->handle == batch->batch.bo->gem_handle);
      entry->relocation_count = batch->batch_relocs.reloc_count;
      entry->relocs_ptr = (uintptr_t) batch->batch_relocs.relocs;

      if (batch->use_batch_first) {
         flags |= I915_EXEC_BATCH_FIRST | I915_EXEC_HANDLE_LUT;
      } else {
         /* Move the batch to the end of the validation list */
         struct drm_i915_gem_exec_object2 tmp;
         const unsigned index = batch->exec_count - 1;

         tmp = *entry;
         *entry = batch->validation_list[index];
         batch->validation_list[index] = tmp;
      }

      ret = execbuffer(dri_screen->fd, batch, hw_ctx, //核心调用
                       4 * USED_BATCH(*batch),
                       in_fence_fd, out_fence_fd, flags);

      throttle(brw);
   }

   if (unlikely(INTEL_DEBUG & DEBUG_BATCH))
      do_batch_dump(brw);

   if (brw->ctx.Const.ResetStrategy == GL_LOSE_CONTEXT_ON_RESET_ARB)
      brw_check_for_reset(brw);

   if (ret != 0) {
      fprintf(stderr, "i965: Failed to submit batchbuffer: %s\n",
              strerror(-ret));
      exit(1);
   }

   return ret;
}


在submit_batch()这个方法里又继续往下调用了execbuffer()这个方法：

static int
execbuffer(int fd,
           struct intel_batchbuffer *batch,
           uint32_t ctx_id,
           int used,
           int in_fence,
           int *out_fence,
           int flags)
{
   struct drm_i915_gem_execbuffer2 execbuf = {
      .buffers_ptr = (uintptr_t) batch->validation_list,
      .buffer_count = batch->exec_count,
      .batch_start_offset = 0,
      .batch_len = used,
      .flags = flags,
      .rsvd1 = ctx_id, /* rsvd1 is actually the context ID */
   };

   unsigned long cmd = DRM_IOCTL_I915_GEM_EXECBUFFER2;

   if (in_fence != -1) {
      execbuf.rsvd2 = in_fence;
      execbuf.flags |= I915_EXEC_FENCE_IN;
   }

   if (out_fence != NULL) {
      cmd = DRM_IOCTL_I915_GEM_EXECBUFFER2_WR;
      *out_fence = -1;
      execbuf.flags |= I915_EXEC_FENCE_OUT;
   }

   int ret = drmIoctl(fd, cmd, &execbuf);
   if (ret != 0)
      ret = -errno;

   for (int i = 0; i < batch->exec_count; i++) {
      struct brw_bo *bo = batch->exec_bos[i];

      bo->idle = false;
      bo->index = -1;

      /* Update brw_bo::gtt_offset */
      if (batch->validation_list[i].offset != bo->gtt_offset) {
         DBG("BO %d migrated: 0x%" PRIx64 " -> 0x%llx\n",
             bo->gem_handle, bo->gtt_offset,
             batch->validation_list[i].offset);
         bo->gtt_offset = batch->validation_list[i].offset;
      }
   }

   if (ret == 0 && out_fence != NULL)
      *out_fence = execbuf.rsvd2 >> 32;

   return ret;
}



到这里用户态驱动的流程就结束了，这里最核心的代码是 int ret = drmIoctl(fd, cmd, &execbuf) 这一行。

这里通过Linux的Ioctl机制调用到了内核态的i915图形驱动，注意这里DRM_IOCTL_I915_GEM_EXECBUFFER2 并不是一个GPU命令，它是一个 Direct Rendering Manager (DRM) IOCTL 命令，用于 Intel 的图形驱动。这个命令用于提交一个批处理缓冲区（batchbuffer）到 GPU 执行。

在图形渲染中，batchbuffer（也称为命令缓冲区或命令队列）是用于存储待执行的GPU命令的数据结构。这些命令可以包括各种渲染操作，如绘制三角形、纹理映射、混合颜色等。batchbuffer的主要目的是将这些命令集中在一起，然后一次性发送给GPU，以减少CPU和GPU之间的通信开销，提高渲染效率。

在Intel的Mesa驱动中，一个batchbuffer可能包含以下内容：

命令数据：这是实际的GPU命令，它们描述了GPU需要执行的操作。这些命令可能包括设置渲染状态、绘制几何图形、执行着色器等。
状态数据：这些数据描述了渲染的状态，例如颜色、深度、模板缓冲区的设置，以及各种渲染参数。
BO（Buffer Object）列表：这是一个指向GPU内存中的缓冲区对象的指针列表。这些缓冲区对象可能包含顶点数据、纹理数据、帧缓冲区等。
围栏（Fence）：这是一种同步机制，用于在CPU和GPU之间同步操作。例如，CPU可以设置一个fence，然后等待GPU完成这个fence，这样CPU就知道GPU已经完成了一部分工作。
其他元数据：例如，batchbuffer的大小、使用的命令数量、当前的渲染上下文等。

总结一下，glFlush的具体实现就是调用内核态提交GPU命令的接口，将batchbuffer全部交付给内核驱动，其完整调用路径是：

glFlush ----> _mesa_flush ----> gl_context->Driver.Flush ----> intel_glFlush ----> intel_batchbuffer_flush----> _intel_batchbuffer_flush_fence ----> submit_batch ----> execbuffer ---->drmIoctl ---->内核态驱动

这里10层函数调用我们还只展开了核心调用路径，如果要把所有细节展开就太复杂了，而且很多内容会涉及到具体硬件实现。我们只需要通过glFlush这个例子能感受到用户态图形驱动大致负责的工作即可。

当然了，glFlush其实比较特殊，是一个"提交"的API，下次再继续分析一些"绘制"的API。

#include <glad/glad.h>
#include <GLFW/glfw3.h>
using namespace std;

int main() {

    // 初始化GLFW
    if (!glfwInit()) {
        return -1;
    }

    // 创建窗口
    GLFWwindow* window = glfwCreateWindow(640, 480, "Hello Triangle", NULL, NULL);
    if (!window) {
        glfwTerminate();
        return -1;
    }

    // 使窗口的上下文当前
    glfwMakeContextCurrent(window);

    // 加载OpenGL函数指针
    if (!gladLoadGLLoader((GLADloadproc)glfwGetProcAddress)) {
        return -1;
    }

    // 渲染循环
    while (!glfwWindowShouldClose(window)) {
        // 清除颜色缓冲
        glClear(GL_COLOR_BUFFER_BIT);

        // 绘制三角形
        glBegin(GL_TRIANGLES);
        glColor3f(1.0f, 0.0f, 0.0f); // 设置颜色为红色
        glVertex2f(-0.5f, -0.5f); // 顶点1
        glColor3f(0.0f, 1.0f, 0.0f); // 设置颜色为绿色
        glVertex2f(0.5f, -0.5f); // 顶点2
        glColor3f(0.0f, 0.0f, 1.0f); // 设置颜色为蓝色
        glVertex2f(0.0f, 0.5f); // 顶点3
        glEnd();

        // 交换缓冲区和轮询IO事件
        glfwSwapBuffers(window);
        glfwPollEvents();
    }

    // 终止GLFW
    glfwTerminate();
    return 0;
}
