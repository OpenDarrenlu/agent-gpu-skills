对 Speculative Decoding 更高视角的总结与分析

在 2025 年，我觉得我做的最大的进步，就是开始有意识的去总结、审视、展望整个领域的发展。或许有不少朋友读过我5月份发的一篇知乎：Speculative Decoding: 总结、分析、展望 ，那么在2025年的年底，我已经有了进一步的思考和总结：

当我们希望所做的科研是真正有意义的时候，我们就需要思考其真正的使用场景。对于我所做的 Speculative Decoding (以下简称Spec) 来说，它的使用场景可以简单的划分为云侧 (大型服务器集群提供的推理服务，重视高并发) 和端侧 (个人PC提供的本地推理服务，重视时延)。从这两方面出发，所有的 Spec 工作都围绕这两个使用场景。

那么钉子找到了，锤子是什么？我们该用什么来分析 Spec？目前我认为最主要的角度就是 compute-bound 和 memory-bound。越memory-bound的场景，Spec越有效。

对于端侧：memory-bound 几乎毫无疑问的占主导。对此，我们就需要充分的榨干算力，因此领域里会有 tree attention 的出现，进一步用算力来换准确率。另外就是，端侧用户更prefer 简单、鲁棒性，因此会有 Retrieval Spec 的出现（不用额外部署一份draft model）。

Summary: 端侧 Spec 的进一步升级技术是 tree attention + retrieval spec

对于云侧：memory-bound 和 compute-bound 可谓是此消彼长。前有 MLA 降带宽加算力，后有MoE / linear attention 降算力增带宽。batch size大了，compute-bound 占主导；context length 大了，memory-bound 占主导。那么在如此复杂多变的场景下，Spec 的核心是什么？我认为是 draft token 的准确率, 这一点的需求远大于 draft model 的开销问题。因为在真正应用场景下，draft token 被拒绝的代价是几乎无法承受、且无法规避的；而 draft model的开销问题总可以通过 并行、流水线等方法去 overlap.

很多刚接触 Spec 的朋友经常问我，为什么 Spec 不适用于高并发？其实这是一个片面的结论。现阶段的 Spec 不适用于高并发，这个论点是对的。但未来的 Spec 不适用于高并发？不见得。试想，假如Spec的接受率能拉到99%，即使高吞吐的情况下，1%的overhead 也仍然会小于99%的draft token正确时带来的带宽收益。这个收益会随着 更稀疏的 MoE, Linear Attention, Long context 等技术的成熟而进一步放大。

因此，一个成熟的云侧 Spec 技术更在乎 draft model 的接受率。scaling draft model的趋势不可阻挡 (EAGLE-3就是一个最好的例子)。为了解决额外的draft model 开销而引入并行、流水线等手段也是工业场景的必需。额外提一嘴：我认为 Draft-Target Disaggregation (DT分离， 参考 nano-PEARL: DT分离与并行投机解码的框架级实现 ) 也是一个必经之路。

Summary: 云侧 Spec 的进一步升级技术是提高draft model的接受率 + Draft-Target 分离 + 并行/流水线技术

对于 draft model overhead 的进一步发展，我做出了 PEARL，第一篇 parallel speculative decoding. 核心思想是，把 draft tokens 进行切分并进行分段 verification. 如果某段 draft tokens 全对，则这部分的draft model的开销可以被完全 overlap (近乎于0)。因此 高并发下，只要 接受率足够高，就可以通过并行来拿到非常大的收益。为了证明这件事情，我最近 release 了基于 nano-vLLM 的 PEARL实现 nano-PEARL .

对于 Tree Attention 的进一步发展，我做出了 TALON，一个simple-yet-effective, 完全动态的 draft tree framework; 过两天会挂到 arxiv 上 (求关注! )

对于 Retrieval Spec 的进一步发展，我做出了 LogitSpec, 发现 LLM 推理过程中的last logit可以用来预测 next next token (这是一个 training-free的性质，和MTP那种不一样! 求求审稿人，别喷 novelty了) 这个性质是 universal 的，受 Zeyuan 的 Physics of LLMs 启发，我认为这个性质是现代LLM广泛的knowledge augmentation带来的。这个性质只需要使用 last logit, 对工程实现非常友好，本质上相当于多条sequence 构建的 draft tree, 因此并不需要像别的tree-based retrieval 方法那样需要特制的 attention kernel。过段时间我会 release 一个 基于nano-vLLM 的LogitSpec 实现来证明这件事情。
