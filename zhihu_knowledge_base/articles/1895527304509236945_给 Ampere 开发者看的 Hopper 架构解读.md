前言

本文主要参考 Dissecting the NVIDIA Hopper Architecture through Microbenchmarking and Multiple Level Analysis[1]

目前主流大厂用的云端 gpu 以 hopper-H800 / ada-RTX4090 为主，笔者之前一直是以 ampere-A100 为主要开发对象，此文是我自己了解 hopper 架构过程中梳理的笔记，也包含一些 ada 的信息，对 ampere 架构比较熟悉且希望转型 hopper / ada 架构的同学可能会有所帮助。

本篇文章主要偏理论，后续会结合 deepGemm / cutlass 等代码再出一篇实践篇。

动机


了解 gpu 架构的基本信息有助于我们分析一个实际问题的性能建模，以及知道我们还有多少优化空间。例如当考虑一个 gemm 问题时可以按照 tensor core 算力去计算其理论耗时，如果是用 cublas / cutlass 去完成同样的功能，通常可以把 tensor core 算力用到 80%～90%，而如果我们自己写的 kernel 只能达到 50%，显然是有一定优化空间的。另外一个例子就是 sage atten v1[2] 中提到，其 int8 实现相比 flash atten 获得了 2 倍多的耗时收益，这个说法其实有些不严谨，因为其实现基于 4090，这张卡有个特性：fp16 做累加器的 fp16 mma 指令带宽就是 fp32 的 2 倍。sage atten 除了在 QK 矩阵乘时使用 int8 mma 获得加速，P * V 的矩阵乘也需要换成 fp16 累加器也是 sage atten 的一大优化点，但是这个优化在 A100 / H800 上是无法拿到的。

此外，每一代 gpu 架构有其独有的特点，只有当我们了解更多细节才能发挥出 gpu 的极致性能，例如本文接下来介绍的，hopper 架构上也可以沿用之前架构的 mma 指令去完成 gemm，但是这个实现方法最高只能发挥出 60% 的理论算力，如果用 wgmma 指令，最高可以发挥出 95% 的算力。

（不过，这个是因为 cublas 没有对 mma 做细致优化，还是从硬件上就没法做呢？我思考下来其实倾向于是硬件上的限制，因为 mma 指令要求 tiled A/B/D 都存在寄存器上，会限制 occupancy，而 wgmma 可以把 A/B 都放在 shm 上，省下的 reg 可以用来做更大块的 gemm，从而让计算访存比更大，更好发挥出算力极限。）

新特性

Hopper 相比之前的架构有比较大的修改，其中在 LLM 领域里有用的主要是：

新的 tensor core（4 代）
支持 fp8 做 gemm
支持 wgmma 指令
新的数据传输单元：TMA
支持 global 到 shared mem 的异步拷贝
新增硬件分层 GPC (Graphics Processing Cluster)
同一个 GPC 中包含多个 SM
这些 SM 间的 shared mem 有专门的线路互联，即，可以完成 1 个 block 访问另一个 block 的 shm
为了支持 GPC 的分层，软件抽象上也多增加了一层：grid-cluster-block-thread


此外，Hopper 除了 LLM，也通常会被用于科学计算，因此在 SFU 单元上还支持了更高效的动态规划指令（DPX），本文对这个部分不做解读，感兴趣的同学可以查看原文。

值得一提的是，Ada 和 hopper 的差异还是挺大的，基本上是沿用了 Ampere 架构，然后把其中的 tensor core 的算力提升了，然后调整了一下各个执行器和各级存储的参数，所以其架构仍然是 sm8x 系列。
Ada 因为缺少了 TMA 单元，所以其 global 到 shared mem 的 copy 仍然需要借助 cp.async，而且无法使用 wgmma 指令，只能沿用 ampere 的 gemm 流程（这个部分可以参考 @reed 大神的 cute-gemm 教程[3]）。

Hopper 专属：TMA 单元

TMA 可以用来做两件事：

Global 和 shared mem 之间的数据移动
同一个 cluster 的 block 间的数据共享

总体来说用来做 1 的场景会更常用，使用 TMA 相比 cp.async，TMA 只需要一个 thread 即可发起，无需每个 thread 单独处理自己的读写地址（且可以自带 swizzle 操作），让每个 thread 的地址计算负担少了很多。
TMA 访问 global，会通过 L2 cache，即，L2 cache 命中率会影响 TMA 的访存效率。
TMA 的访存格式可以选择：non-tensor 访问（即，当作一个没有 layout 的连续数据行），1D 访问，2D 访问，3D 访问。通常来说要搬运的数据块越大，以及启动的 block 越多，能用上的带宽会越大。

non-tensor 访问，2D 访问，3D 访问在大块数据（eg. 16KB）的场景下，都能让带宽达到 1800 GB/s，基本上和不通过 TMA 访问的带宽对齐。
而 1D 访问这个 case，在设置数据大小的时候有限制，最多只能到 2KB，因此访问效率上没有优势，因此，当我们像访问 1D 数据的时候，我们可以直接用 non-tensor 的方式访问即可。

Figure 2. TMA 在不同 layout 以及 num blocks 下能达到的带宽。总体趋势为数据块越大，num blocks 越大，可达到的带宽越高。左二表示 1D 情况下数据块大小最高只能到 2KB，而只能达到较低带宽


此外，在设置 2D / 3D layout 的时候，应该优先排 x 轴的 size，x 轴 size 越大（即，数据的 col 维度越大），越容易达到最大带宽

Tensor Core 特性

编程接口

Tensor core 可以用来做 mma 计算（D = A * B + C），在 ampere 上有两种方式来调用：c 接口的 wmma 指令，以及 ptx 接口的 mma 指令。Ada 上也是沿用的这个方式，而 Hopper 上除了这两种方式外，新增了 wgmma 的指令，然而需要注意 wgmma 只能做（D = A * B + D），不再支持多出来的 C 矩阵了。
Hopper 和 Ada架构都支持 Fp8 的 mma 指令，这一点让模型量化更好做了，且能减少数据类型转换的开销，是很不错的特性。
此外，还有一直支持的结构化稀疏矩阵乘（mma.sp），在各个卡上还是都支持，但是基本上很少有场景能用起来，也不知道这个特性是真有用还是只是用来营销算力的手段。

table 5. Ampere / Ada / Hopper 架构支持的 mma 和 wgmma 数据类型与调用接口

另外，值得注意的是 Ada tensor core 是支持 int4 mma 的，且算力是 int8 的两倍，这一点和 ampere 是相同的。而 hopper 做 int4 mma 会把其编译为 IMAD 指令，实际上是用 cuda core 模拟 int4 计算，算力是非常弱的。这一点其实我挺疑惑的，不知道这只是编译层面上没做适配，还是第四代 tensor core 在硬件上就不支持 int4。

调用方法

要达到好的 tensor core 利用率，使用 ptx 接口调用 mma 指令是必要的。在 hopper 上新增的 wgmma 也是需要通过 ptx 调用的，指令对比如 fig，对比普通的 mma 指令，可以看到其一次做的 mma 块大小要大得多（m64nNk16，N 为 16 的倍数，通常设置为 128 / 256 这个量级），并且需要同一个 warp-group 中的 4 个 warps 来协同调用。


Figure 5. mma 指令与 wgmma 指令对比

以及从指令上也可以看到，wgmma 只接受 a，b，d 参数，其中 a 可以存在 reg 也可以在 shm，b 只能在 shm，d 则只能是 reg。这也就是为什么这个指令只能是 async 的，因为从 shm 中读数据的时延是不可控的。
这个指令相比每个 warp 独立做 mma 的好处是给 A & B 矩阵用的 reg 会少很多，原因是首先 B 从 shm 直接读，A 即使存在 reg，大概也是可以在 warp 间共享的（在 flash-atten v3 中有提到这个信息，但是目前我找到的 wgmma 代码中都是从 shm 读数据的，包括 deepgemm），从而提升了 occupancy。

分布式 shared mem

在之前的编程体系里，shm 是和 block 绑定的，而在 hopper 中，多了一个 cluster 层级，cluster 中的多个 block 可以共享 shm，当然硬件上 shm 还是只能绑定到 SM 上，那么一个 block 想要访问另一个 block 的 shm 就出现了两类情况：

两个 block 在同一个 sm 上，这时候不需要通过 sm2sm 网络互联，这个访问的带宽大概是 205 GB/s，大概是一个 block 访问自己 shm 带宽的 80%，时延上比走 L2 cache 少 20%，比走 global 则快 5 倍。
两个 block 在不同 sm 上，这时通过 sm2sm 网络互联，这种情况下需要具体分析带宽，总的来说是点对点的通讯带宽比广播通讯带宽高，而 cluster 中的 block 数量越多，带宽越弱。
Torch 实测带宽

torch 中想调用最高效的 gemm（尤其是 hopper），可以通过 nvidia 支持的 te (transformer Engine) 库来实现，单个 linear 层的计算带宽对比，基本与本文分析得到的结论一致，即，同数据类型 H800 > 4090 > A100，而数据类型的 bit 减半，则计算带宽增一倍。

Figure 6. te 库调用 linear 的实测带宽，同数据类型 H800 &gt; 4090 &gt; A100，而数据类型的 bit 减半，则计算带宽增一倍

总结

TMA 可以让访存需要参与的 thread 更少，需要做的坐标运算更少，使用时注意 x 维度优先。
用 hopper 时 wgmma 能发挥出其真正算力，只用 mma 只能发挥 60% 的算力，wgmma 可以从 shm 直接读 AB 数据。
Hopper 不支持 int4，而 ada 支持。
Ampere 不支持 fp8，而 ada 支持。
Ada fp16 做累加器计算带宽加一倍，hopper 和 ampere 都没有这个特性。
分布式 shm 慎用，尤其是涉及到不同 SM 间通讯。
