本章的向量体系结构是一个架构上的概念，对应于一般的标量一条指令针对一个元素进行处理（SISD，Single Instruction Single Data），向量体系结构是一条指令对一个向量进行处理（SIMD，Single Instruction Multi Data）。

向量处理器，多媒体SIMD指令扩展以及GPU都属于SIMD的变体，它们具有相同的SIMD的特性。但是也有各自的区别，本章主要记录向量处理器的特点，但是在这之前列出三种变体的区别。

向量处理器属于独立处理器架构，适用于高吞吐数据规整的科学计算，它的明显特点在于其计算数据的类型和长度是寄存器可配的。比如，如果向量寄存器位宽为512比特，可以通过寄存器配置指定计算是处理32 * k（k<=512/32）的数据还是处理64 * j（j<=512/64）的数据。这种动态配置的优势可以减少指令集的数量，利于软件开发使用，而只使用需要的寄存器可以减少上下文切换的代价。

多媒体SIMD指令属于CPU指令集的扩展，它使用于媒体处理和轻量级的并行计算。它和向量处理器最大的不同在于，它将数据类型融合到指令中进行，在指令中指定了计算的数据类型。

GPU属于大规模并行处理器架构，适用于图形处理和通用计算。另一方面，它属于线程级并行（TLP）和数据级并行（DLP）融合的架构。不同于向量处理器通过单条指令显式处理整个向量（如 VADD.VV v1, v2, v3 完成64个浮点加法），GPU则通过硬件自动管理数千个线程，线程组（Warp/Wavefront）以SIMT方式执行相同指令。

本章以RISC-V的RV64V向量扩展为例，说明向量处理器的架构特点。

以RISC-V 向量处理器的基本组件如下下图所示，其中包括了以下几个部分：


向量体系结构基本组件


向量寄存器(Vector register)：每一个向量寄存器保存一个向量。RV64V有32个2048bit长度的向量寄存器，每个寄存器保存的数据类型可以通过寄存器进行配置，比如，可以保存32个64位宽（double float）的元素或者256个8bit位宽（Int8）的元素。由于向量处理器能同时对多个数据元素进行计算，因此寄存器堆需要提高足够的端口以供应全部的向量功能单元。在针对不同向量寄存器执行的向量操作之间，这些端口运行高度重叠（即复用通道）。另外，对于如此大的寄存器堆，保证足够的带宽也至关重要，因此可以使用多个存储Bank来组成寄存器堆，以权衡访问延时和存储大小的矛盾。
向量功能单元：SIMD是一条指令对多个数据进行处理，实际上这种处理可以从两个维度考虑。假设一个向量包括32个元素，一个元素的计算需要4个周期，这种处理在时间维度上需要以流水线技术提高效率，通过流水线可以将原本32x4=128个周期减少为32+3=35个周期。从空间维度上，可以对多个元素都实现计算通道（lane)，假设有32个计算通道，那么4个周期就可以完成所有的计算。当然，考虑到面积和性能的折中，功能单元会结合多通道和流水线技术，并不会实现所有的计算通道。
向量load/store单元：向量处理器会将内存的load到向量寄存器堆，或者将寄存器堆的数store到内存中。由于向量处理器的数据较多，需要尽可能的提升存储器的带宽和吞吐率，因此向量load和store也是完成流水化的。
标量寄存器堆：标量寄存器可以将数据给到功能单元，完成标量和向量的计算。也可以计算load/store时存储器的地址。

从上面的组件可以大致看出向量处理器的基本架构，但是我们仍然关心一些细节问题：

计算通道和寄存器堆如何对应？

其中，a是标量，X/Y是向量，这就是SAXPY（单精度）或DAXPY（双精度）循环问题。假设向量X，Y有32个元素，X，Y的起始地址在x5和x6，则下面给出RV64G和RV64V的实现代码。

RV64G的实现代码：

      fld     f0, a          #载入标量a
      addi    x28,x5,#256       #要载入的最后一个地址
Loop:   fld     f1, 0(x5)      #载入X[i]
      fmul.d  f1, f1, f0       #a*X[i]
      fld     f2, 0(x6)      #载入Y[i]
      fadd.d  f2, f2, f1        #a*X[i]+Y[i]
      fsd     f2, 0(x6)      #存储到Y[i]
      addi    x5, x5, #8     #递增X的索引
      addi    x6, x6, #8     #递增Y的索引
      bne     x28, x5, Loop   #检查是否完成

RV64V的实现代码：

        vsetdcfg    4*FP64      #启用4个双精度浮点向量寄存器
        fld         f0, a       #载入标量a
        vld         v0, x5      #载入向量X
        vmul        v1, v0, f0  #向量标量乘
        vld         v2, x6      #载入向量Y
        vadd        v3, v1, v2  #向量-向量加
        vst         v3, x6      #存储向量和
        vdisable                #禁用向量寄存器

对比标量指令实现和向量指令实现，可以总结一下几点不同：

向量指令存在启用和禁用寄存器操作：向量指令在计算前启用需要的寄存器，在计算后禁用所有寄存器。这样如果在最后一条指令之后发生了上下文切换，将不需要保存额外的状态。（个人理解时，此时这些寄存器是无用的）
向量处理器大幅缩减动态指令带宽：在向量指令中，只需要实现8条代码即可完成所有计算，而标量实现需要2+8*32=258条指令。标量实现中，一次只计算一个元素，需要多余的指令为循环服务（如后三条），因此带来了无用的开销。
流水线互锁频率：在RV64G代码中，每一个fadd.d(指令6)必须等待fld(指令5)完成，每一次fsd(指令7）必须等待fadd.d(指令6）完成。而在向量指令中，只有等待第一个向量元素时会停顿，当某条指令的第一个数计算出来后，后续指令就可以开始进行计算，而不必等前面的指令全部完成，这种技术在向量处理器中称为链接(chainning)

在体系结构中还有旁路（bypass）的概念。旁路作用于流水线，将前面指令的结果bypass到后续指令进行计算，在指令级触发。它与链接很像，但是需要注意，链接是在功能单元间建立点对点直连总线，在元素级进行触发。

动态寄存器类型设定（Dynamic register typing）

动态寄存器类型设定（Dynamic Register Typing） 是向量处理器中一种运行时动态配置寄存器行为的核心技术。它允许程序在执行过程中按需调整寄存器的数据存储格式、聚合方式和掩码策略，从而实现对硬件资源的弹性管理。其核心是通过专用指令（如RISC-V的vsetvli）动态设置配置寄存器，无需重新编译即可适应不同计算需求。RV64V的一项创新是将数据类型和数据大小与每个向量寄存器关联起来，而不是在指令中提供这一信息。

动态寄存器设定有如下优点：

减少指令集数量：将数据类型和指令解耦，可以大幅减少指令集的数量，同时已有代码进行数据迁移时便于兼容。
动态寄存器设定让程序禁用没用到的向量寄存器：这样的优势在于可以将向量存储器分配给已启用的向量寄存器。比如，假设向量存储器有1024个字节，如果只有4个向量寄存器启用，且数据类型时64位浮点，那么处理器将1024字节的存储空间分配给4个向量寄存器，每个向量寄存器可以存放256字节，256/8=32个元素。这个值就是可以支持的最大向量元素（maxinum vector length, mvl)，它由处理器设定，不能软件改变。
便于上下文切换：向量体系结构的寄存器太大了，因此对于不被使用的向量寄存器可以设置为禁用，这样就不需要在上下文切换时去处理它们了。
隐式转换数据类型：当设定数据类型后，不同大小的操作数之间进行转换时不需要另外显式的增加转换指令。

下面介绍一些重要的动态寄存器。

向量长度寄存器（Vector-Length Registers）

核心问题：向量元素的适配

前面提到，对于向量寄存器，它能一次处理的元素个数是有一个最大值的(mvl)，对于能处理元素不等于最大值的情况，动态寄存器设定使用向量长度寄存器完成。

向量长度寄存器（Vector Length Register, VLR）是向量处理器的核心组件之一，用于动态控制单条向量指令处理的元素数量。它解决了硬件固定最大向量长度（Maximum Vector Length, MVL）与实际数据规模不匹配的问题，通过分段处理长向量实现灵活性。

考虑下面的例子：对于向量C=A+B，假设处理器能处理的最大向量长度为64，而实际的元素长度为100，那么需要进行如下操作：

步骤	描述	剩余
第一段	VLR = 64&A[0:63] + B[0:63] → C[0:63]	100-64=36
第二段	VLR = 36& A[64:99] + B[64:99] → C[64:99]	36-36=0

编译处理机制如下：

​    low = 0                 ; 起始索引
    remaining = N           ; 剩余元素数 (100)
    while remaining > 0:
        VLR = min(remaining, MVL)   ; 第一段VLR=64, 第二段VLR=36
        load A[low:low+VLR-1] → VA  ; 加载A段
        load B[low:low+VLR-1] → VB  ; 加载B段
        VADD VA, VB → VC            ; 向量加法
        store VC → C[low:low+VLR-1] ; 存结果
        low += VLR                  ; 更新索引
        remaining -= VLR            ; 更新剩余数​

也就是说，通过vl，可以设置需要的向量元素大小，且该设置项取实际剩余值和最大值的小值。

使用向量长度寄存器有以下优点：

透明扩展性：程序无需感知硬件MVL，相同代码在MVL=64或256的机器上均可运行。
减少指令开销：一条VADD指令替代了标量循环中的100条加法指令+100次循环控制指令。
避免冗余计算：若N=50（小于MVL），直接设VLR=50，仅计算有效元素（不操作寄存器后14个位置）
谓词寄存器（Predicate Registers）

核心问题：解决循环内部的条件（IF）情况。

考虑下面用C实现的代码：

​ for( i=0 ; i<64; i++)
        if( X[i] != 0)
            X[i] = X[i] - Y[i]


由于在这个循环体中存在条件判断，因此很难直接进行向量化。但是我们可以看到条件的成立与否决定了某个元素是否执行。因此我们可以根据条件只对某些元素执行操作。实现这一功能的常见扩展称为向量掩码控制（vector-mask control），它使用谓词寄存器控制。

谓词寄存器（Predicate Register），又称向量遮罩寄存器（Vector Mask Register），是向量处理器中实现条件并行计算的核心硬件组件。它通过位掩码控制向量指令对特定元素的操作权限，避免条件分支导致的性能损失。

执行步骤如下：

步骤	描述	
步骤一	生成条件掩码将比较结果存入谓词寄存器	VP = [1, 0, 0, 1, 0]，1表示执行，0表示不执行
步骤二	条件执行向量加法	根据VP的元素是否为1，对原操作向量的元素进行计算，可以得到C = [4, ?, ?, 8, ?] （? 表示未修改区域，保留原值）

谓词寄存器的优点:

消除分支开销：标量CPU处理条件需分支预测（错误率>10\%时性能骤降），而向量处理器无分支指令。
维持硬件并行度：所有元素并行比较生成掩码，加法器物理上仍全速运行（仅结果写入受控）。
减少冗余计算：对比无掩码的向量化（全量计算+选择性存储）：节省无效计算的功耗。
支持复杂条件组合：通过逻辑运算合并多个谓词（如 VP3 = VP1 AND VP2）

注意，第2点和第3点实际上是速度/功耗倾向下的两种方式。

Vtype配置寄存器

vtype寄存器是RISC-V向量扩展（RVV）的核心控制状态寄存器（CSR）之一，它包括向量长度乘数（LMUL，用于控制寄存器逻辑分组/分片，合并物理寄存器（如m4：v0-v3合并为V0）），元素位宽（SEW，用于设置单元素比特数 确定元素精度（e8/e16/e32/e64））等配置字段，可以通过配置该寄存器使得处理器适应各种场景。具体的细节可以通过RVV扩展指令集了解。

Memory Banks

存储一直是体系结构中需要关注的重点问题，尤其对于向量处理器而言，虽然为数据设置了更多的计算单元，但是计算单元是否会因为存储墙而空闲？存取数据往往面临多种情况，要存取的数据量较大或者多个数据在离散的地址，因此如何增大存储的带宽和优化离散访存的效率是向量处理器的存储关注重点。

一般情况下，访存单元的初始化延迟要高于算术单元，为了保持每个时钟存储可以供应上算术单元的需求，存储器系统必须能够提供或接受较多的数据。将访问对象分散在多个独立的存储体中，通常可以保证所需速率。这些单独的存储体运行多个独立访问。

使用多个独立存储体的原因如下：

许多向量处理器支持每个时钟周期执行多个访存操作，访问存储的周期通常比处理器周期高几倍。为了支持多个访存的同时进行，存储器系统需要多个存储体，还要能独立控制对这些存储体的寻址。
大多数向量处理器支持访存非连续的数据。该情况下需要独立的存储体（independent banks）而非交叉寻址（interleaving）。
大多数向量机支持多个处理器共享同一存储器系统，所以每个处理器会生成自己的独立寻址流。
步幅（stride）：处理多维数组

步幅（Stride）是向量处理器中控制内存访问模式的关键参数，定义为连续操作的元素在内存中的地址间隔（以元素大小为单位）。非单位步幅（Non-unit Stride）则打破连续访问限制，实现灵活的数据遍历。

向量处理器可以通过步幅寄存器（Stride Register）动态配置间隔值，由地址生成单元（AGU）自动计算目标地址：

单位步幅（Stride=1）：访问连续内存块（如数组元素A[0], A[1], A[2]...）。地址计算：BaseAddr + index × sizeof(element)
非单位步幅（Stride≠1）：以固定间隔跳跃访问（如矩阵的列、对角线）。地址计算：BaseAddr + index × stride × sizeof(element)

以一个例子说明：一个100×100矩阵M的每列求和

，如果内存是行优先进行布局，即元素排列为,（M[0][0], M[0][1], ..., M[0][99], M[1][0], M[1][1], ...），那么访问列元素，相邻列元素间隔100各元素，因此步幅为100。若排列按照列有限（M[0][0], M[1][0], ..., M[99][0], M[0][1], M[1][1], ...），那么列元素间隔为1，此时步幅为1。

向量处理器具有大型向量寄存器堆，通过非单位步幅对存储器进行访问，可以将不连续的数据放入连续的寄存器堆空间中，组成一个稠密的结构。同样，也可以将寄存器堆中相邻的数据放入不连续的地址空间中，如图

Gather-scatter：处理稀疏矩阵

稀疏矩阵很常见，因此需要一些特殊的技术使得稀疏矩阵的程序能在向量处理器中执行。在稀疏矩阵中，向量元素通常以某种压缩形式存储，然后被间接访问。比如下面的代码，它实现了数组A和数组K的求和，使用索引向量K和M来指定A和C中的非0元素：

​ for ( i=0 ; i < n; i++)
        A[k[i]] = A[K[i]] + C[M[i]]​

支持稀疏矩阵主要采用索引向量的Gather-Scatter（集中-分散）机制。该操作索引向量（Index Vector）实现元素级任意地址访问。

集中操作：用索引向量指定地址，一次性从内存非连续位置加载数据到向量寄存器。VDest[i] = MEM[BaseAddr + VIndex[i]]。
分散操作：将向量寄存器的值按索引向量指定的地址写入内存非连续位置。MEM[BaseAddr + VIndex[i]] = VSrc[i]。

Gather-Scatter

RV64V中实现这一扩展功能的指令为vldx（载入索引向量，Gather）和vstx（存储索引向量，Scatter）。假设x5，x6，x7，x28包含以上序列中向量的起始地址，则可以用向量指令对内层循环进行编码，如下：

​    vsetdcfg    4*FP64      #启用4个浮点向量寄存器
    vld         v0, x7      #载入K[]
    vldx        v1, x5, v0) #载入A[K[]]
    vld         v2, x28     #载入M[]
    vldx        v3, x6, v2) #载入C[M[]]
    vadd        v1, v1, v3
    vstx        v1, x5, v0  #存储A[K[]]
    vdisable                #禁用向量寄存器​
