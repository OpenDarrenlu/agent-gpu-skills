半年前，我在《对 cuTile 的感想》中写道，cuTile 是否能在新一轮 AI 革命中扛起大旗，”只能说他的发布提供了一种可能性，而这种可能性留待时间来验证”。GTC 2026 上 Stephen Jones 的这场 talk 展示了更多进展，尤其是 cuTile 在性能方面的提升。

从整场 talk 的内容来看，可以观察到 CUDA 编程正在经历一次范式演进——从线程级标量操作的 SIMT 范式，走向块级集合操作的 Tile 范式。而 cuTile 是这次演进中承上启下的关键组件。

1. cuTile C++ 来了：从 Python 到双语言

去年 cuTile 首次发布时，只有 Python 前端（ct.kernel）和 Tile IR。GTC 2026 的重要消息是 CUDA Tile C++ 即将在下一个 CUDA 版本中发布。

C++ 版本使用 __tile__ 关键字标注函数，语义上与 Python cuTile 对等。以矩阵乘为例：

template <int tn, int tk, int tm>
__tile__
void matmul(ct::tensor_span_like auto A,
            ct::tensor_span_like auto B,
            ct::tensor_span_like auto C) {
    auto acc = ct::zeros<ct::tile<float, shape<tn, tm>>>();
    auto aV = partition_view{A, shape<tn, tk>{}};
    auto bV = partition_view{B, shape<tk, tm>{}};
    for (auto k : ct::irange(0, aV.index_space(1))) {
        auto aTile = aV.load(ct::bid().x, k);
        auto bTile = bV.load(k, ct::bid().y);
        acc = ct::mma(aTile, bTile, acc);
    }
    ct::partition_view{C, P}.store(acc, ct::bid().x, ct::bid().y);
}

对应的 Python 版本语义完全一致：

@ct.kernel
def matmul(A: ct.Array, B: ct.Array, C: ct.Array,
           ts: ct.Constant[ct.Shape]):
    acc = ct.zeros(ts[0:1], A.dtype)
    (x, y) = (ct.bid(0), ct.bid(1))
    aV = ct.view.partition(A, (ts[0], ts[2]))
    bV = ct.view.partition(B, (ts[2], ts[1]))
    for k in range(pA.shape[1]):
        acc = ct.mma(pA[x, k], pB[k, y], acc)
    ct.store(C, (x, y), acc)

这种双语言对等是有意为之的设计。Python 前端适合快速原型和 JIT 编译场景（AI 框架集成）；C++ 前端则面向需要与已有 CUDA C++ 代码库集成的场景。两者共享同一套 Tile IR 和编译器后端。

此外，talk 还展示了一个非平凡的例子：用 Tile C++ 实现 Sparse Sliced ELLPACK（SELL）格式的 SpMV。这不是一个 dense 计算的 toy example，而是带有 load_masked、软件预取流水、非规则访存模式的真实内核。这说明 cuTile 的表达能力已经不局限于规则的 dense tensor 操作。下图是 SpMV 在 Blackwell B200 上相对于 cuSPARSE 的性能对比，大部分 workload 达到了 1.0x 到 3.0x 的加速：

2. FlashInfer on Tile IR 的性能

talk 中最值得关注的数据是 FlashInfer 在 Tile IR 上的性能表现。NVIDIA 用 cuTile 重写了 FlashInfer 性能内核库，并展示了与非 Tile 版本 FlashInfer 的直接对比。

下图展示了 Tile IR 实现对比 best non-Tile FlashInfer 的性能比值：

可以看到，绝大部分 workload 的 Tile IR 实现都达到了 non-Tile FlashInfer 的 0.8x-1.0x 水平，而且有相当数量的 workload 超过了 1.0x。特别是在 FusedMoE、block_sparse 和 ragged gemm 等场景上，Tile IR 版本甚至取得了 1.5x-3.5x 的显著加速。

以 FusedMoE 为例，这是一个典型的能体现 cuTile 这类 DSL 优势的场景。FusedMoE 的 mega kernel 融合了多个底层 kernel——包括 token routing、FC1、silu_and_mul、FC2 以及 reduced weighted sum 的计算。在此基础上，它还需要与多种量化方式组合，如 per-tensor scaling、DeepGEMM 风格的 block scaling、mxfp8/nvfp4 block scaling 等。这些组合极大地扩展了 kernel 的 fusion 变体和调优空间。

此外，实际部署中的 problem size 也会显著变化。当 DeepSeek R1、Llama4 等模型采用多节点/多卡并行时，MoE kernel 的实际输入规模可能使其变为 memory-bound，进一步增加了调优的复杂度。

这些因素对手工编写和调优每一种 kernel 变体提出了巨大的挑战——要覆盖所有组合并逼近理论峰值吞吐，工程成本极高。而借助 cuTile 这样的 DSL，开发者可以用同一份代码模板生成大量 fusion 变体，再通过 autotuning 和 Tile IR 编译器优化来发现和生成高性能 kernel。换言之，越是组合空间庞大、手工优化难以穷尽的场景，编译器驱动的方式越有可能胜出。

这组数据的意义在于：cuTile 已经不再只是一个”未来可期”的编程模型。在 FlashInfer 这样的实际高性能内核库上，Tile IR 的编译器优化在多数 workload 上已经能够接近甚至超越手工优化的 CUDA 实现。当然，部分 workload 仍存在差距，但整体趋势表明编译器驱动的优化正在快速缩小与手工调优之间的距离。半年前这还是一个悬而未决的问题，现在有了实质性的进展。

3. 性能可移植性：一次编写，跨架构调优

另一组重要数据是跨架构的 FMHA 性能。NVIDIA 展示了同一份 cuTile 源代码在五个架构（A100、H100、RTX、B10、B300）上的表现。

关键点在于：源代码在所有架构上完全相同。不同的只是自动调优后的 tile 尺寸。这直击了一个现实的痛点：传统手工优化的高性能 kernel 往往需要针对特定硬件特性深度适配，削弱了代码的代际可移植性。cuTile 通过将硬件适配下沉到编译器，用 autotuning 替代手工特化，恢复了”写一次，跑多代”的特性。

talk 中也标注了 “Results shown are without hand-tuning”。这意味着手工调优仍有空间，但自动调优的基线已经有竞争力。对于绝大多数开发者来说，这是一个实实在在的生产力提升。

4. 混合编程：SIMT + Tile + CuTe DSL

NVIDIA 没有试图用 cuTile 取代一切。talk 明确勾画了三种编程模型的共存关系：

CUDA SIMT：线程级标量操作，非结构化并行（排序、自定义逻辑）
CUDA Tile：块级集合操作，规则/dense 并行（线性代数、attention）
CuTe DSL (CUTLASS Python)：全面底层控制 tensor core、调度和数据放置

talk 展示了一组实际的 benchmark 数据：在 InferenceX DeepSeek R1 (prefill) 上（8x B200 180GB w/ NVLink, ISL=8k / OSL=1k），逐个将 kernel 替换为 CUDA Tile 实现，观察 tokens/sec 的变化：

可以看到，大部分 kernel 替换后保持了 1.0x 的性能（silu_and_mul, rms_norm, bmm, rope_quantize_fp8），而 w8a8_fp8_scale_matmul 达到了 1.17x 的加速。全部 kernel 切换到 CUDA Tile 后整体持平（1.00x），说明 Tile 编译器在端到端场景下已经具备与手写 kernel 竞争的能力。

更有意思的是 Thread/Tile 互操作的设计。从 Tile kernel 中调用线程级函数（如 cuFFTDx、CUB）”incurs almost no overhead”。talk 给出了一个完整的例子：在 cuTile kernel 中调用 cuFFTDx 做 FFT，然后计算 Parseval 定理。这种零成本互操作意味着开发者不需要在 Tile 和 SIMT 之间做非此即彼的选择。

5. 编译器栈的全貌：Tile IR 与 MLIR

talk 揭示了 CUDA 编译器栈的完整架构，这可能是对编译器从业者最有价值的信息。

整个栈分为三层：

语言层（Language Layer）：

C++: nvcc 编译 __global__(SIMT) 和 __tile_global__(Tile)
Python: cuTile, CuTe DSL

编译层（Compilation Layer）：

Tile IR Bytecode → 内部 MLIR Dialects → NVVM Dialect
PTX → NVVM IR / ToT LLVM IR (CUDA 13.2 编译器基于 LLVM 21)

执行层（Execution）：

SIMT、Tile、Tensor Core 三种执行路径

值得注意的是，talk 中明确标注了 “Tile IR backend for OpenAI Triton”。这意味着 Triton 可以编译到 Tile IR，利用 NVIDIA 的 MLIR 编译器栈生成优化代码。cuTile 的 Tile 级 SPMD 编程模型与 Triton 的方向是趋同的。

此外，talk 中还介绍了 CompileIQ——通过自动搜索数千种编译器优化配置，为每个 kernel 定制最优编译设置。talk 展示了 Meta 提供的两组 benchmark 数据，涵盖 Helion（Meta 基于 Triton 的 attention kernel 库）和 Flash Attention 等 workload，CompileIQ 带来了显著的性能提升。编译器自动优化正在成为性能工程的核心竞争力。

6. cuTile 路线图

talk 给出了清晰的 roadmap：

已发布（pre-13.3）：

Python cuTile 和 Tile IR
Linux & Windows，支持所有 Blackwell GPU
Ampere / Ada GPU 支持

即将推出（13.3）：

显著的性能优化
Hopper GPU 支持
CUDA Tile C++
FP4 支持（Blackwell）
更多 scatter/gather 和 partition/view 操作

未来（13.x）：

非 2 的幂次 tile 尺寸
Stencil 操作符
SIMT / Tile 同 kernel 互操作
Tile 原生通信调用（与 NVSHMEM 集成已有 demo）

关于通信，talk 展示了 cuTile + NVSHMEM 实现的多节点 3D FFT。nvshmem_tile.put() 和 signal_wait 被直接嵌入到 tile kernel 中，实现了计算与通信的融合（Fused Compute + Communication）。

回顾半年前的思考，我在《对 cuTile 的感想》中提到 cuTile 和 Tile IR 带来了一种可能性。GTC 2026 的数据让这种可能性变得更加具体：

性能已经 ready——SELL 和 FlashInfer on Tile IR 的数据表明，编译器优化的质量已经达到了与手工优化竞争的水平。这是半年来最重大的变化。
CUDA 生态需要在性能和可移植性之间找到新的平衡点——cuTile 的 “zero code change performance portability” 是对这个矛盾的回应。
双语言（C++ + Python）+ Triton 后端——Tile IR 正在成为一个多前端共享的编译器中间层，这对整个 GPU 编程生态的影响可能比 cuTile 语言本身更深远。

半年前还是可能性，现在有了实质性的数据支撑。cuTile 的未来仍有许多待验证的环节，但从 GTC 2026 展示的进展来看，这条路径正在变得越来越清晰。
