整体思路


考虑下面这种情况，在某层中，0 号专家被路由了 200 个 tokens，而其他专家平均被路由的 tokens 只有十几个，那么计算瓶颈会在 0 号专家这里。为了处理类似的情况，EPLB 会对 0 号专家进行冗余性的复制，比如复制出 3 份 0 号专家的副本（加上本来的就有 4 份副本），那么 200 个 tokens 就可以直接被均发给这 4 份副本，每份副本处理 50 个 tokens，从而降低了多卡中的计算瓶颈。在讨论具体细节前，为方便起见，我们先做如下规定：


在后面的篇幅中，我们将模型定义中的专家称为逻辑上的专家，实际部署中所有卡上的专家称为物理上的专家。比如 DeepSeek V3 671B 中每层逻辑上一共有 256 个路由专家和 1 个共享专家。在实际部署中所有卡上每层的专家总和会比 256 + 1 多。


由于层与层之间部署方案是独立的，为了方便讨论，下面只讨论每层的部署。
下文中的 weight 表示过往一段时间内的统计信息，在忽略层后可以认为 weight 是一个长度为专家个数的一维数组/张量， weight[i] 表示第 i 个专家在过往一段时间内处理的 tokens 数。




Decode

按 DeepSeek V3 技术报告中所述，decode 阶段使用了 40 个节点，共 320 张卡。据此将物理专家设置为一共 320 个，每张卡上 1 个物理专家。其中包含逻辑上的 256 个路由专家和 1 个共享专家，以及 63 个冗余专家。将共享专家当做一个（必选的）路由专家后一共有 257 个逻辑上的路由专家。令n = 257, n' = 320，根据历史路由统计weight调用 replicate_experts，选出物理上多出来的专家是哪些逻辑上的专家的冗余。




Prefill

按 DeepSeek V3 技术报告中所述，decode 阶段使用了 4 个节点，共 32 张卡。据此将物理专家设置为一共 320 个，每张卡上 10 个物理专家。其中共享专家在每张卡上都有完整的一份，所以还剩$32 * 9 = 288$个物理专家，一共部署 256 个路由专家和 32 个冗余专家。DeepSeek V3 671B 专家组数为 8（每组 32 个专家），可以被节点数 4 整除，此时 EPLB 采用分层负载均衡，先在专家组和节点层面做一次划分，再在每个节点内对分到该节点的若干专家组中的专家做冗余复制和划分。具体地讲，EPLB 先将历史路由统计weight 按组求和得到tokens_per_group。令n = 8, m = 4，根据tokens_per_group调用 balanced_packing，得到组的划分（每个节点上放哪两个专家组）。然后再对于节点内划分，一个节点内放置8*9=72个物理专家，放置2 * 32 = 64个逻辑专家，此时令n = 64, n' = 72，根据weight（要 gather 一下）调用 replicate_experts，选出多出来的 8 个冗余专家是复制的哪些逻辑上的路由专家。再令n = 72, m = 8，根据weight（要根据冗余复制转化一下）调用 balanced_packing，决定节点逻辑专家和冗余专家到底部署是怎么划分到 GPU 上的。

个人思考
在路由时，如果一张卡上有多个 tokens 要被路由给某个复制过的专家，需要决定每个副本路由多少 tokens。一个做法是均分当前卡上的 tokens 发到该专家的每个副本上，如果每次数据分布符合历史数据分布，该种做法可以达到上述做法所求出的负载情况。不过如果在 Moe Gating 后先做一次通信获取其他卡上 tokens 的路由情况，可以根据实际情况更均衡地决定每个 token 到底路由到该专家的哪个副本上。
对于 prefill 的分层部署，https://github.com/deepseek-ai/EPLB/tree/main#:~:text=Moreover%2C%20thanks%20to%20the%20group%2Dlimited%20expert%20routing%20used%20in%20DeepSeek%2DV3%2C%20we%20also%20attempt%20to%20place%20the%20experts%20of%20the%20same%20group%20to%20the%20same%20node%20to%20reduce%20inter%2Dnode%20data%20traffic%2C%20whenever%20possible.提到将相同组的专家放到一起可以减少跨机通信，但个人理解这种情况能减少跨机通信的前提是 token 要能合理地放到节点上（随机放节点的期望跨机通信量跟专家是否按组分到节点上没有关系）。
对于 prefill 的分层部署，在节点内，先调用 replicate_experts 再调用 balanced_packing 并不是最优方案，应该直接一起考虑。（不过前提是需要实现 1 中提到的 Moe Gating 后先路由通信。）
对n = 8, m = 4的 balanced_packing，直接暴力枚举就行，对于n = 72, m = 8的 balanced_packing，如果是离线部署推理任务的话或许可以使用求解器来计算最优方案。
