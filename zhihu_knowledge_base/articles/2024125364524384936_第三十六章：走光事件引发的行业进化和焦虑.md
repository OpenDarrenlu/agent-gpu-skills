# 第三十六章：走光事件引发的行业进化和焦虑

**作者**: 第欧根尼​知势榜科技互联网领域成长力榜答主

**原文链接**: https://zhuanlan.zhihu.com/p/2024125364524384936

---

2025年12月2日，JavaScript 运行时工具 Bun被Anthropic收购了。Bun 的极致性能被看中了，Anthropic希望将其作为底层的核心基础设施，用来驱动和加速 Claude Code、Claude Agent SDK 以及未来的各类 AI 编程产品。

这笔收购官宣时，正是Claude Code 的高光时刻。它刚创下了一个惊人纪录——面向公众发布仅 6 个月，年化经常性收入（ARR）就突破了 10 亿美元。产品的高速狂飙迫使 Anthropic 必须彻底夯实底层技术栈。这笔交易显然是经过深思熟虑的，从7月份开始，Bun 团队就一直在优先解决 Claude Code 开发者反馈的各种性能问题，这些问题里面，有很多其实来自于Anthropic内部。经过几个月的重度使用，技术团队终于确定了Bun项目的价值，整合它能大幅的优化 Claude Code 的响应时间和内存占用，使其能在开发者终端里极其流畅地运行。

于是，不差钱的公司在明确的技术需求的情况下，收购了一个开源团队，这种故事在硅谷可以说是司空见惯，几乎每天都在上演。但是谁也没想到，这会是一个造成整个行业巨大变动的导火线。

被收购后的Bun，依然保持着开源社区的运营，团队也依然在维护着GitHub上的开源项目。2026年3月11日，Bun 社区有人提交了一个 编号为Issue #28001的BUG，指出其在生产环境中依然会默认生成 Source Map 文件。这个 Bug 在随后的20天里并没有被修复。整个团队还忙碌在和Anthropic的整合工作中。

这个BUG被报告出来的20天后，准确的讲是2026年3月31日，Anthropic 向 npm 注册表发布了 Claude Code 的 2.1.88 版本。由于这个BUG的原因，main.js.map这个文件被默认生成了。这是一份约59.8 MB的包含Claude code完整前端工程源码的文件，如果随着版本发布则会导致严重的代码泄露问题。但是Anthropic内部并没有人发现这个问题。UTC时间4点23分，来自中国的安全研究员、前伯克利博士生 Chaofan Shou，通过日常的包监测发现了这个漏洞，并在 X上发布了包含直接下载链接的推文。消息瞬间在安全和 AI 圈引爆，该推文迅速获得了几千万次浏览。

接下来的两小时，开发者们疯狂下载源码并镜像到 GitHub。其中一个重写版本claw-code的仓库在不到 2 小时内飙升至 50,000 颗 Star，打破了前几天才被OpenClaw打破的 GitHub 历史最快涨星记录，并产生了超过 41,500 次 Fork。

UTC时间早上8点，反应过来的 Anthropic 紧急将该 npm 包从注册表中撤下。Anthropic 发布了修复后的 2.1.89 版本。官方发言人确认这是一次“人为打包错误”，并非安全漏洞被黑，也没有暴露任何客户数据或底层大模型权重。同时，Anthropic 挥起 DMCA数字千年版权法大棒，向 GitHub 提出了大规模侵权下架请求，超 8000 个相关仓库被强制删除。

但是，这已经来不及了。这份51.2万行、1906个文件的 TypeScript 源码已经被完全暴露在全世界的开发者面前，在他们面前，Claude Code几乎已经没有秘密可言了。这次意外的走光事件，强行拉平了之前社区的各种工具和方法论之间的参差不齐的差距，所有人站在了同一条起跑线上，准备重新起跑。但事实真的是这样乐观吗？抄作业真的能考的上清华北大吗？想要回答这个问题，我们必须回到2024年去详细分析。

让我们把时钟调到2024年，再来回顾一下当OpenHands和社区们推动整个Agent的管理范式进阶到S3阶段时，OpenAI和Anthropic都做了些什么吧。

2024 年 10 月，OpenAI发表了一篇名为《MLE-bench》的论文，在这篇论文里，OpenAI定义了一项评测，这个评测像SWE-bench一样，也是评估模型的能力的，不过这个能力不是来测试一个修代码的架构师，而是用来评测炼丹的数据科学家的。换句话说，OpenAI想用agent替代的，并不是普通的程序员，而是他们公司内部的程序员们。

OpenAI的这招是一步跳棋，冒险性极强。SWE-bench考验的是即时反馈，Agent 改完代码，跑一下 pytest，几秒钟内编译器就会用红字报错告诉它：“你写错了，第 8 行有语法错误。” Agent 可以迅速纠正。而MLE-bench对应的任务是当agent写好代码点下运行，可能要等几个小时才知道这个架构好不好用，在这几个小时内，模型除了完成逻辑工作之外，还需要管理好 GPU 资源，监控进程会不会内存溢出，其整体的工程难度又上了一个台阶。如果说之前的agent是模拟一个程序员的工作的话，那么OpenAI剑指的替代目标则是包含训练环节在内的整个MLE（Machine Learning Engineering）流程。

之所以OpenAI如此激进，和当时的市场竞争状况相关。在2024年中，Anthropic 的 Claude 3.5 Sonnet 正在SWE-bench榜单上疯狂屠榜。Claude 因为极其听话的指令遵循能力，成为了所有 Harness（如 OpenHands、Cursor）的默认底座。竞争的最好策略永远不是追赶跑在前面的对手，而是去重新定义终点的位置。比模型写代码更酷的事情是什么？当然是用模型训练模型了，OpenAI打算弯道超车，不在编程领域和Anthropic做过多的纠缠。

2024年的Anthropic则是显得不那么激进。OpenHands们掀起的浪潮，Claude是既得利益者，在那段时间里，几乎所有的程序员都认为Claude是编程能力和指令遵循程度最好的模型。但是它也深深的感受到Harness框架们处于前端的优势。

在2024年，当时模型只能输出文本，如果 Agent 在沙箱里报错了，Anthropic 官方是拿不到详细的“环境交互失败数据”的。这些数据都掌握在 OpenHands 等第三方 Harness 框架手里。这种前端的优势是有价值的，Anthropic必须要往前走，不能让Harness们站在它和用户中间。所以，2024 年中旬它极其高调地推出 Computer Use API，一种能让模型直接操控鼠标、键盘和屏幕GUI的能力。当 这个 API 开放时，全球无数开发者兴奋地用它去做网页抓取、填表、甚至打游戏。Agent 疯狂地点击了错误的按钮、陷入了死循环、误删了文件。Anthropic 借此兵不血刃地收集到了全球最大、最真实的“硅基智能体行为纠偏数据集“。
