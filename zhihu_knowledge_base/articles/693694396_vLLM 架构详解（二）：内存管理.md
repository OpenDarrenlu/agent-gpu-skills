CalebDu：vLLM 架构详解（一）

这一部分主要围绕vLLM核心的KV Cache内存管理展开。

1. 内存管理总览

LLM的内存管理的核心就是对KV Cache的管理。最朴素的KV Cache的管理方式是对每个Sentence 都开辟MaxSeqLen的KV Cache（MaxSeqLen通常为几K），在多batch下所需的KV Cache 大小为[Batch, SeqLen, n_head*head_size],但是每个Sentence 的Prompt+output token 长度变化很大，如果全部按照MaxSeqLen开辟KV Cache，其中大量的空间都浪费了，成为内部的碎片，影响推理可以并行的最大Sentence数，从而影响整体的吞吐。

vLLM的核心思想就是通过仿照OS 系统页表的管理提出了Page Attnetion，来更有效的管理KV Cache的存储。

如上图所示AB 2个request的Sentence各自维护一个Logical Block列表， 通过一个全局的Block Table来查找Logical Block对应的Physical Block，每个Block 能够存储Block_Size个Token，这样对于每个Sentence最大只可能浪费Block_Size-1个token，大幅减少了KV Cache的浪费，为更多的Sentence并行提供了可能，极大的提高了系统的吞吐。

2. 内存管理组件

vLLM中核心的内存管理组件为：

Scheduler：每次step模型执行前调度SentenceGroup，被成功调度的SentenceGroup根据需要被分配Physical Token Block，并提供索引哪些SentenceGroup 中的block需要被swap-in，swap-out，copy-on-write。
BlockManager：负责管理每个Sentence 对应id 的Physical Token Table，为Sentence拥有 Logical Token Block分配/释放(释放仅只把block加入free的列表，不释放真实的资源)对应的Physical Token Block。并Scheduler调度过程中把对应的block 从CPU swap-in或 从GPU swap-out或为新的Token append_slot等
Block Allocator：分配和释放block
Block：包含管理每个physical/logical block所需的必要信息，如block id，block size， ref count等，【不包含任何token的Cache数据】
Cache Engine：负责创建和管理真实的KV Cache Tensor。执行swap-in，swap-out，copy-on-write 真实的KV cache数据搬运。

Cache Engine 创建KV Cache Tensor [2, num_blocks, block_size * num_kv_heads * head_size], 其中 num\_blocks = \frac{total\_gpu\_mem\times gpu\_memory\_utilization-Model\_Size}{Block\_Size} ，以GPU 显存的 gpu\_memory\_utilization 的倍数减去模型权重的存储空间来创建KV Cache， 剩余的空间用于创建一些推理需要的input、output 以及中间变量。

每个Sentence拥有 \lceil \frac{seqlen}{block\_size} \rceil 个Logical Token Block（Logical Table），Block Manager通过Block Allocator管理num_block全部的block，并拥有全局的Physical Block Tables，存储每个seq 所属的Physical Block Table，用于查找Logical Block->KV Cache 中Physical Block的映射。

3. 内存管理的流程

本小结以_schedule_default 调度策略为例，介绍vLLM推理流程中如何进行内存管理。_schedule_default保证单次step 的SentenceGroup 都是prompt phase或decode phase。

【prompt phase】

prompt phase内存管理流程

prompt phase 的流程从用户添加一个新的request开始，LLM Engine根据request中的prompt string 创建一个新的Sentence，创建Sentence时构造 \lceil \frac{prompt\_len}{block\_size} \rceil logical token block 列表（Logical Table）。并由该Sentence 创建一个新的SentenceGroup，并加入Scheduler waiting 队列。在Step 最初Scheduler 将waiting 队列中的Sentence调度，被成功调度的SentenceGroup 根据其中每个Sentence Logical Table 的大小分配对应的physical token block列表（Physical Table），组成block table。prompt phase不涉及swap-in、swap-out、copy-on-write，把成功调度的SentenceGroupMetaData与KV Cache Tensor输入Model_Runner 调用模型的forward，在Attention Module中 通过自定义算子reshape_and_cache_kernel把kv-project计算得到的KV根据block table 每个token 对应的slot_mapping 存储到全局的KV Cache Tensor内。模型执行一次end2end 的推理之后得到SampleOutput，由于不同的Sample策略，一个parent Sentence 可能采样出多个output token，这些不同output token的child Sentence会fork parent Sentence的logical Table ，并把各自的output token加入每个child Sentence 各自的Logical Table并fork parents 的Physical Table，并将child Sentence 加入SentenceGroup供之后的 step Scheduler调度，直至满足结束条件。

【decode phase】

decode phase内存管理流程

decode phase的流程从新一次step 开始，Scheduler 会根据调度策略(见详解一)把running队尾的元素的block swap-out（Swap Mode），并将符合要求的swapped 队列中的SentenceGroup 的block swap-in，并对这一轮成功调度的所有Sentence append slot 用于存储这一轮decode 生成的新token。append slot会先校验physical table 长度是否与logical table一致，若不一致直接分配一个新的physical block加入physical table 并返回。若两表长度一致则校验last phyiscal block 的ref count，若ref count==1，说明该phyiscal block只被一个Sentence拥有，直接返回，否则说明多个Sentence 共享这个block，分配一个新的block把该block 加入copy-on-write，原始的block ref count--。将Scheduler 调度输出的结果传给Worker执行，Cache Engine 根据Scheduler 提供的swap-in 、swap-out、copy-on-write的block 索引 通过自定义算子swap_blocks、 copy_blocks进行实际的KV Cache数据搬运。之后的模型执行及采样后处理与 prompt phase基本一致。

vLLM的内存管理作为整个系统的核心比较复杂，涉及推理的全流程，感觉很难把整个链路讲的很清晰，大佬轻喷。

vllm 更新的_schedule_chunked_prefill 调度策略内存管理 和block manager v2 有一些新特性还没研究清楚，之后再更新。
