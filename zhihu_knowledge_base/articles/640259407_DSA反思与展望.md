1，数据流优化/图调度是挖掘DSA架构的钥匙。

flash attention是在gpu上搞一个比较大的融合算子，GPT2性能惊艳；此前早就了解写cuda大算子是字节AML优化模型的杀招。可见手撸大算子是把mem bound转化成compute bound有效的手段，应该百试不爽吧。在gpu上也就是小打小闹罢了，谁能把整个resnet50或者bert都驻留在gpu的shared mem里面不出来，做到记得发个顶会出来呀。

且先看看这里，我们就把resnet50，bert base驻留在片上了，在用LPDDR4的情况下，resnet50 fp16干到9000fps，比别人用HBM的还要好近一倍（友商应该还是在按照nVidia的思维做dsa编译），懂性价比的自己去算（sign，别问我为啥叹气）。健飞兄应该跟我一样，都是被带宽受限逼到这个路线上的，不做没办法啊，按照nv的模式，吞吐马上掉到90fps，差100倍！

但掌握这个技术后，带宽（dram + noc）的设计，就可以相对从容一点了，因为图调度技术可以弥补带宽一定程度的缺失。支持图调度背后的那个关键技术，叫自动算子融合，他要能做到，给你任意一个子图，输入输出的mem level指定，占用的mem大小指定，都要能产生一个性能比较好的融合算子出来。因为你没这个东西，手写融合算子，是难以支持那么灵活的图调度的。

2，关于抽象层次和系统构建哲学思考

我曾经举过这样一个例子：给小朋友买乐高，年纪小的小朋友可以玩的颗粒度大，年纪大的玩的颗粒度小。把这个应用到架构和软件栈上很贴切。例如，gpu是many core（几千），非常多的标量核搭建的。因此他是大朋友玩的，要用他搭建东西，比较难，时间周期比较长，投入要更大。dsa，最多是个几十核，粒度要粗得多，用它来搭建东西，就容易点，时间周期短一点。但表达能力上，gpu要更广一点。dsa搞定个ai/ml没啥问题。

整个系统的抽象，不是一个粒度，而是多种粒度，从底层向上，由细到粗。粒度可以是操作的粒度，也可以是数据的粒度，或者混杂着。

结合roofline model，就有点意思了：

1，如果保证每个抽象层都是 compute bound，自底向上组装起来，就可以接近性能极致。

2，如果某个抽象层自身无法做到 compute bound，则需要打破抽象，与其他层抽象联合优化，一起做到 compute bound。 ai 编译的一个趋势是在同一个上下文里面表达多层抽象（ MLIR，图算融合akg， unity/relax），背后的本质是方便做跨层联合优化（比如上面的图调度，就是打破了算子的抽象，跨层了）。

3，在越细粒度的抽象层，单个抽象内优化空间越小，把 memory bound 转换为 compute bound 就越容易（指令 ld/st->单核进出带宽->DDR 带宽->单机跨卡通信->跨机架通信），另外，增加片上缓存也可以缓解对带宽的压力。硬件在基于资源考虑与软件协同的情况下提供一个较好的基础，其他交给软件对程序进行优化，把 memory bound 转换成 compute bound。

4，增加片上缓存，可以缓解带宽压力，把数据驻留在片内也需要更大的片上缓存。

5，当对大多数模型都是 compute bound 的时候，进一步增大算力才有意义。




3，gpu vs dsa

有了riscv，dsa的构建变得简单了，可能会有越来越多的人意识到这条路的优越性。

因而dsa也变成了gp-dsa。

由于在软件的构建上速度的优势，以及基于rv的高效性，gp-dsa有了快速迭代和试错的机会。

可能我们要着重关注后发制人的架构。芯片架构的历史包袱很重，一旦有了，就很难演进或者拥抱新路线。而最先进的认知经验，杀伤力是很强的。

有人会想生态，不用管训练，llm时代下，推理就够大场景了。做个推理的生态，难度小很多。

等5年看，应该有dsa可以让人眼前一亮。dsa能有起来就算赢，NV gpu未来10年还是主力。
