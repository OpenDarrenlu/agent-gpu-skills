# 2025 infra 工作随笔

今年可以用一句话简单概括：从国产卡“跑路”，到在其他卡上“浑水摸鱼”。

去年年底到今年年初，一直尝试为所在的国产卡训练一个对应的算子大模型。从数据清洗、标注到训练，再到最终的部署推理和 open-webui 前端上线，可以说在国产卡上全流程都跑通了。模型表现尚可，LLM写的简单算子，交给有一两年经验的国产卡算子工程师稍作修改，基本就能使用。但效果也仅限于此，主要因为当时主要还是用 SFT，数据也反复清洗过很多遍，性能差不多触及天花板了。

比较有趣的是，当时不敢公开说在训练算子大模型，就对外称之为“刘备大模型”。还在躺平arch群里讨论用强化学习提升它的上限，没想到晨阳问了一句具体打算怎么做，只好私下告知实情，就这样和他熟络了起来。

但众所周知，几乎所有国产卡的软件生态相比开源生态都有几个月以上的差距，在强化学习这类前沿领域更是如此。虽然业界年初已普遍意识到强化学习的重要性，国产卡训练侧却仍集中在预训练和 SFT 阶段的基础设施搭建，RL 这块几乎没什么成熟的软件生态。直到后来，arch群里有群友分享了一篇 TinyZero 的知乎文章（只能说牢潘全责了），看后觉得很有意思，决定在国产卡上复现它！




于是，开始了漫长的踩坑之旅。Verl 1.0 的设计当时看来非常优雅和经典，它包含 ray、vllm、fsdp/megatron 等几个大模块，而每个模块在国产卡上几乎都会报七八个错误。不过，作为专业 QA，解决问题本就是本职工作。

最致命的问题是 vllm 在国产卡多卡推理时会随机出现乱码。只要Global Batch size 够大，第一轮 rollout 就会出错（一般在线服务不会设这么大的 batch size，所以 CI 很难测出来）。

对于这个报错，在国产卡上，我从驱动怀疑到通信库，从通信库怀疑到 torch，从 torch怀疑到 vllm，从 vllm 怀疑到 verl。最后，我疑神疑鬼。

于是，开启了一段漫长的精度对齐工作——不巧，本人正是有两年精度对齐经验的实习生。那就 print 大法启动！首先，在 forward 会经过的每一层添加了检查 inf 和 nan 的代码，发现 embedding 层就开始出错了。奇怪的是，对比和开源代码，似乎没发现什么不同，毕竟实验在 N 卡上跑得好好的。还去 vllm 群里请教了 ykc 大神，他提示可能是 nccl 问题，但我用的又不是 nccl（无法甩锅）。Ray 的日志也是一团糟，当时不知道要去 /tmp/ray 下按 pid 找日志，直接把 verl 的几十万行日志扔给通信库同事，结果被喷了一顿（磕头谢罪）。接着，开始怀疑 verl 框架本身，就在 rollout 前打印对比 N 卡上的结果，排查了几百个数据对之后，基本排除了 verl 的问题。

那就只能怀疑 vllm on 国产卡了。于是开始在框架层加打印，发现请求调度前后没有乱码，但在 forward 前出现了乱码。最终定位到一个异步操作，把它改为同步后，乱码消失了，当时真是喜极而泣。深入一看，发现是 torch 在国产卡上的那部分逻辑写错了，数据量大时会发生覆盖。查了下 commit 记录，发现是楼下同事写的——那一刻，真想去楼下约架！（开个玩笑）

然而，光是这个精度问题就花了两周时间，身心俱疲。训练好不容易跑起来了，训着训着却又崩了。当时百思不得其解，后来才知道很可能是训推不一致导致的。毕竟国产卡算子库大多是训练一套、推理一套，两者差异巨大。在国产卡上，只有一个人在折腾 RL，困难太多，进展缓慢，加上一些其他事情，渐渐也就没了继续做下去的动力。

正好晨阳在找人给 verl 接入 SGL，于是开始和 yitong、xinyuan 一起做 VLM 的 RL。没想到第一件事又是精度对齐——要对齐 SGL 和 vllm 在 RL 里的 eval 精度。反正还是不停地 print、print、print 找到精度出问题的地方，然后在 mick 老师的帮忙下解决了。之后就是跟着晨阳和一众伙伴，给 verl 不断添加新功能。随着 verl 越来越庞大，代码也逐渐成了一座“屎山”。

经历过从 0 到 1 的搭建过程，就不愿再在国产卡上继续啃精度对齐的“硬骨头”了——毕竟也只能望着社区的背影。尤其是一些新特性，国产卡根本跟不上。于是，决定从国产卡“跑路”，投身到 RL infra 领域去“浑水摸鱼”。

入职新公司半年，在推理这块，被紧急安排去训练一个 Eagle3。这方面确实一窍不通，只能每天不停地骚扰帅、超哥和企鹅。最终，在社区各位大哥的帮助下，跑通了 DPSK Eagle3 的全流程训练。后来，在大哥们的神秘训练技巧加持下，模型在特定领域的接受率甚至超过了 NextN。只能说大哥们太强了，完全是抱大腿飞。

这里要特别提一下 SpecForge，真的太强了。各家推理厂商如果想降本增效，真的该好好用用它。顺便宣传一下 spec-bundle ，一个 Eagle3 模型库，大家可以在 SGLang 里无痛体验飞一般的加速效果。







SGL & slime

RL 这边，主要是跟着社区一起做。一开始是抱着 zilin 哥和标哥的大腿，优化了 MoE 模型的权重更新；

之后和同事在 slime 上吭哧吭哧地把 Kimi Instruct 跑通了，也算体验了一把训练 1T 模型的快乐。

在低精度训练方面，跟着发哥、congkai 哥一起做了全流程的 FP8 RL训练。最近 RL 最火的话题是训推一致，发现 FP8 全流程训练居然能抑制 BF16 训练、FP8 推理带来的量化误差，算是个意外之喜。接着，又在 slime 上复现了 Kimi 团队的 Int4 weight-only rollout，发现 BF16-Int4 的训推差异竟与 BF16 训推处于同一量级——看来 Kimi 团队的知乎文章确实靠谱，值得深入研究。

后知后觉地看，无论是推理 infra 还是 RL infra，各种先进特性其实都在跟着头部闭源 infra 的路线走。比如 DPSK 在推理侧的 large EP、在 RL 侧的 rollout router replay，实在是给大家喂了一口好饭。这波属实得感恩 DPSK（不是）。

除此之外，还认识了很多老前辈和好玩的朋友。不正经的llllkkkk，一天两包烟的老渣，牢范和斌哥这对好homie，OD 大队长雨豪，diffusion 包工头 ruby，还有智慧和美丽的化身，申老师和董姐。nv那边的朋友也很nice，又会写代码又会炒股的，带带弟弟已经说累了。

差不多就写这些吧。不知道 infra 这条路还能做多久，但认识了社区里的大家，是今年最开心的事。
