三个库的总结

最近在研究在GPU上实现稀疏矩阵乘(稀疏矩阵-稠密矩阵乘，SpMM)。挑选了近几年具有代表性且开源的文章进行精读，并尝试复现实验、剖析代码。这几篇文章都同时实现了SpMM和SDDMM，此次只针对SpMM进行讨论，后续有机会再开一个SDDMM的专题。

本章主要介绍分别来自SC20，SC21，SC22的三篇文章。

Sparse GPU Kernels for Deep Learning(SC20)[1]------sputnik

该工作由Stanford和Google合作完成，至今是GPU上非结构化SpMM的SOTA方法。代码实现完整，优化度高，性能也是同平台下无敌。Sputnik在CUDA Core上实现，基于CSR(Compressed Sparse Row)压缩格式完成了非结构化稀疏矩阵乘，解决了cuSPARSE等稀疏矩阵库(当时的)针对高稀疏度的局限性，如图1。

图1

其重要工作可以分为四点：(1)1-D Tiling; (2)Subwarp Tiling; (3)Reverse offset memory alignment; (4)Row Swizzle。

1-D Tiling:将输出分解成一维块，并将独立的线程块(thread block)映射到每个块。

在真实网络中，稠密矩阵的列数（N维）可能变化很大。训练RNN时，N维通对应批次大小，常在16-128。在transformer中，N维对应批次大小和序列长度的乘积，常在256-2048。在部分网络中，N维最高可达到14400。如果不从N维划分，模型将难以具化成原子计算操作，编程难。

如图2所示，按照C矩阵分块划分thread block。由于非结构化稀疏矩阵的CSR格式按照行存储，每行的索引不同，每次计算只能取一行，所以thread block高为1，即1-D。并将thread block的宽设为定值BlockItemsX。

图2

1-D Tiling有三个好处：

1.可以很容易地将真实模型转化为tile，生成kernel。

2.对于M和K较小时，同样可启动更多的thread block，从而能够实现更高的占用率和更高的峰值吞吐量比例。

3.处理固定大小的thread block使我们能够在编译时展开循环并计算偏移量和常数等。

Subwarp Tiling:将一个warp的子集映射到输出的独立的1-D Tiling

GPU中还支持向量访存指令，将数据按照向量的型式进行加载/存储，更高效地利用了带宽和数据局部性的优势，增加了thread block同时加载的值的数量。

但也带来了两个问题：

1.如果在tile中不足规定个数，则无法完整执行一个kernel。

2.由于稀疏矩阵的非零数是任意的，压缩后要与访存长度匹配是很难的。

因此提出了两个解决方案：Subwarp Tiling和Reverse Offset Memory Alignment。

图3-2

如图3-1，允许将一个warp的子集映射到输出的独立的1D tiling。 这将访问宽度限制减少了一个使用的Subwarp数量的因子。如图3-2定义加载参数，将每个warp加载的行数设置为kBlockItemsY，用threadIdx.y来定义subwarp要加载的行号。这也使我们能够灵活地将thread扩展到输出矩阵的更多行，以解决密集和输出矩阵中列数较少的问题。但也带来了负载不平衡的问题，因为要同时加载多行，但是相邻行的非零数可能差距很大(CSR格式压缩后，只存储非零数)，因此又提出了Row Swizzle方法来解决负载不平衡问题。

Reverse Offset Memory Alignment

在加载行偏移量并计算行长度之后，每个thread block将其行偏移量递减到最近的向量宽度对齐地址，并更新它需要处理的非零点数。 为了保持正确性，thread会屏蔽在主循环的第一次迭代中累积结果之前从上一行加载的任何值。

图4
Row Swizzle

在GPU中是以warp为调度单位，单个SM核可以同时执行多个thread block，但是单个thread block中多个warp需要都执行完毕才能结束该thread block，因此warp/thread间工作负载如果不相似，将会造成warp/thread等待同步解决造成的开销。例如一个thread block同时处理两行数据，但是0第一行长度为1000，第二行长度为1，那么第二行就要等第一行处理完剩下的999个数据才能结束该thread block。因此，将相似的tile放到同一个warp中去执行，能够减少这种栅栏同步机制带来的开销。而Sputnik解决这个问题是很简单地用行长度来定义，按照每行nonzeros来对行重排序，如图5所示，原序列如左边所示，按照行长重排序后如右图所示。

Efficient Tensor Core-Based GPU Kernels for Structured Sparsity under Reduced Precision(SC21)[2]------vectorSparse

该工作是UCSB谢源老师团队完成的。个人认为这个工作是非常全面，且就有影响力的。他不仅从硬件调度上还从稀疏格式上进行分析，提出了几个SpMM操作高延迟的原因，并且提出了5个设计原则来提高SpMM Kernel性能。

该工作与Sputnik最大的不同在于，它在Tensor Core上实现结构化SpMM。其中Tensor Core在Volta之后的架构出现，以固定大小的矩阵为输入输出，高效实现矩阵乘加。在同精度下，Tensor Core能比CUDA Core加速4X，但是由于固定输入，使得它的编程粒度很粗，无法适应细粒度操作。但随着应用环境的不断变化，结构化SpMM逐渐也有了自己的市场。而由于它高度结构化的数据结构，似乎十分适合在具有高度并行性的Tensor Core上实现。

使用低精度后，高速内存可以存储更多的操作数，提高数据的复用率。但是与稠密GEMM相比，细粒度SpMM无法高效利用数据重用性。虽然可以引入结构化稀疏来提高数据重用，但现有库（如cuSPARSE）不能以较小的稀疏粒度提供实际的加速。当需要更大的块大小时，在保持模型精度方面给算法带来了更多的挑战。作者总结了三点高延迟原因：

1.“No Instruction”：无指令，每个子核有固定容量的指令缓存。

2.“Short Scoreboard”：IMAD（整数乘加）和IADD3(3输入整数加）占总执行指令的27.4%。从SASS代码中，这些整数指令计算全局和共享内存中的块的地址。

3.“Wait”：通常发生在warp从Shared加载数据时等待。

对于SPMM，(shared memory load requests)/(global load requests)=0.87

对于GEMM，(shared memory load requests)/(global load requests)=4.17

这意味着SpMM比GEMM数据重用性差。

作者设计了5个具有高效内存访问和计算模式的结构化稀疏Kernel的关键准则。

1. 减小程序大小避免指令溢出

2. 利用TLP，增大网格大小，隐藏延迟

3. 通过循环展开、在编译时计算偏移量和常数，以及将浮点运算与TCU合并到HMMA，减少固定延迟操作。

4. 将重用率低的数据存在寄存器中，而不是存在Shared Mem中。

5. 通过合并访存，提高带宽利用率

Column Vector Sparse Encoding

提出列向量稀疏编码。受CSR的启发，一个索引对应一个非零列向量且该压缩方式与块稀疏编码复用率相似但操作粒度更小。在SpMM操作中，若利用Block-Sparsity则每个B矩阵复用V次，而利用Vector-Sparsity则每个B矩阵也是复用V次。

数据重用与列数无关，因此块稀疏与列稀疏具有相同的数据重用性。而对于每列的向量长度V(2，4，8)，分别用half2，half4，float4表示和连续访存。压缩编码示意图如图7所示。

TCU-based 1-D Octet Tiling

由于硬件限制，CUDA Core和Tensor Core无法同时使用，提出了一种新颖的Warp Tile与Tensor Core之间的映射，即基于Tensor Core的一维八位拼接。该映射同时满足5个关键准则。

在介绍TCU-based 1-D Octet Tiling方法前，先介绍Tensor Core的计算过程，如图10所示。一个大小为MKN(8*4*8)的操作可以分为4个步骤，而这也是Tensor Core指令中粒度最小的操作mma.m8n8k4。在封装好的API，如wmma.m16n16k16或wmma.m8n32k16等都是由最小粒度的mma.m8n8k4组成的。

图10

首先，对于thread block分块，如图11(上)所示，每个warp计算规模为(V*Tile_K)*(Tile_K*Tile_N)=(8*32)*(32*64)的矩阵乘。但是特别的是，在图11(下)中，(4)表示SpMM中的RHS矩阵，但却被加载到Tensor Core中的A矩阵中，同样的(2)表示SpMM中的LHS矩阵但被加载到Tensor Core中的B矩阵中。这样，原来都为row-major存储的矩阵，RHS矩阵变为col-major，LHS矩阵变为row-major(Tensor Core中加载B矩阵按照col-major，因此LHS矩阵又变回row-major)。

因此原始SpMM中LHS矩阵和RHS矩阵就互换了位置，再回到图10，可以看到，当V=2和4时，可以省下step0-3中的step2和step3，因为他们的长度只有2或4，无法填满Tensor Core中的B矩阵，因此填充的0无需参与计算。

图11

其次，将一个warp分为8组thread，以保证高效的计算和内存访问。具体而言，一个warp分块被分为TileK/4步被执行，即每次迭代利用mma.m8n8k4指令执行分块（(V*4)*(4*64)）。为了符合准则4，将重用性低的RHS矩阵直接加载到reg中，为了符合准则5，将RHS矩阵分块中的64个fp16操作数映射到8个不同的thread中，且每个thread处理8个连续的fp16操作数，这样就能满足LDG.128的最大合并访存粒度，并且最终整个(4)生成4个128B合并的Global Memory访存事务。而重用次数多的LHS矩阵就会被加载到Shared Memory中在片上被反复存取来提高访存效率。

在存储mma.m8n8k4的操作数时，编译器倾向于利用寄存器重用技术来减少寄存器开销。但是这也会导致指令级并行(ILP)收到损失，因为在计算不同的分块（(V*4)*(4*64)）时会产生互相之间的数据依赖。因此，作者在加载数据完毕后，还加入了__threadfence_block()指令来阻止编译器重用寄存器。

Experiments

vectorSparse SpMM在V100上平均比基于Blocked-ELL SpMM加速1.71-7.19X。在稀疏Transformer模型端到端推理中，实现了1.41X的加速比和13.37X的内存开销减少。

Efficient Quantized Sparse Matrix Operations on Tensor Cores(SC22)[3]------Magicube

该工作是北邮的李士刚老师在ETH做博后时完成的，也有幸和李老师也交流过几句，该工作的代码基本是他一人完成的，工作量很大的，最终也是成功拿到了SC22 Best Paper Finalist。其实我也是从去年十一月份才开始接触GPU以及SpMM这个方向，这篇文章算是入坑看的第一篇文章，第一次精读一篇文章半个月，第一次扒CUDA源码。而上面两篇SC以及我看过的其他相关的文章，也是从这篇引文中找的。

Magicube中SpMM设计其实可以分为两个部分：1. 低精度SpMM Kernel的设计与优化；2. 混合精度的仿真。

减少内存占用和推理延迟的主要技术是矩阵运算的稀疏性和量化，低精度运算也开始成为算力缺口，因此在NVIDIA A100 GPU中的第三代Tensor Core开始支持Int4/Int8的IMMA操作。但是，由于对数据布局的严格要求以及缺乏对有效处理低精度整数的支持，从Tensor Core上的稀疏、低精度矩阵运算中获得实际的加速是非常具有挑战性的。Magicube是一种在Tensor Core上实现结构化SpMM和SDDMM的高性能稀疏矩阵库。

SR-BCRS Format

与vectorSparse类似，面向的是一维块稀疏(即2*1 / 4*1 / 8*1的向量)设计，所以基于原有的CSR压缩格式和BCSR压缩格式，该工作提出了基于Tensor Core编程友好的SR-BCRS(Stride Row-major Block Compressed Row Storage)(我觉得其实叫SR-BCSR也行),如图12所示。按照 stride 长度存储，如果每行中非零元素数量小于 stride，则补零，column indices 用*填充。而本文中，stride 等于 mma 操作中K(Int8 : stride = 16，Int4 : stride = 32)每个 warp 中的 thread 可以连续的将 A 矩阵的数据加载到 reg 中，并满足布局要求。SR-BCSR 支持的一维稠密块 V<=8(mma 中的 M)，如果 V=8，mma操作能百分百利用，如果 V=4，mma操作只能利用 50%。因此为了支持这种数据格式，需要2M个row pointer(每行2个pointer，一个指向起始位置，一个指向末尾)，相当于需要M行。

Thread blocks of SpMM：

图13(b)显示了在thread block级别的Magicube中SpMM的实现。每个thread block中有2个warp。每个thread block负责一 个大小为BSm * BSn的输出块。这里BSm = V(2或4或8)。可以将BSm设置为V的倍数，但是这无助于改善数据重用，因为每一行向量可能具有不同的列索引。在每个步骤中，每个thread block计算BSk维的部分结果。在这里，BSk等于SR-BCSR格式中的stride大小，也等于mma的K。总体而言，每个thread block都需要Nnz/BSk步(部分结果被累积)来获得最终结果。受益于SR-BCSR格式，通过stride连续加载数据直接满足mma的A矩阵的数据布局要求。A矩阵以合并的方式从全局内存(Global Memory)加载到共享内存(Shared Memory)，并且数据在warps之间共享。

在这里，对密集矩阵B使用row-major存储。但是，mma的RHS(Right Hand Side)矩阵(即B矩阵)需要column-major存储，因此布局不匹配。由于密集向量的列索引不是连续的，因此事先将B转换为列主将没有帮助。因此，为 B 的行主块提出了一种有效的在线转置策略，该策略包含三个主要步骤。

(1) 块中的thread协作地将B的行从Global Memory加载到Shared Memory上的填充Padding。BSn = 64 的示例如图14所示。每一行64个int8值被合并为一个64B内存事务，从而提高了Global Memory访问的效率。对于大 N，将 BSn = 128设置为合并到128B存储器事务以获得更高的效率。

图14

(2) 每个thread将依次从Shared Memory加载4个int32项到本地寄存器。warp0的每个thread访问的项目在图14 中标记。在GPU中Shared Memory被划分为32个bank，每个bank位宽为32bits。同一周期内可以访问不同bank，否则会导致bank冲突，降低性能。通过在每64个int32项目后填充8个int32项目，每个warp内没有bank冲突。

(3)每个thread以int8的粒度对本地寄存器进行转置，如图15所示。转置后，寄存器块的每一行都包含来自矩阵B的一列的4个连续的int8值，满足mma的数据布局要求。请注意，每个thread在转置寄存器块中有4行，分别被送到4个mma操作的RHS矩阵中。因此，对于一个thread block内的两个warp，每个步骤中有8个mma操作要执行，如图16所示。

Prefetch for matrix B

RHS矩阵B的数据块被加载到Shared Memory以用于转置。但是，由于矩阵A是稀疏的，因此在Shared Memory上没有用于矩阵B的数据重用。为了减轻矩阵B的内存流量成本，使用数据预取策略，该策略如图18所示。每个thread block都需要nnz/BSk步才能获得最终结果。此外，将数据从Global Memory加载到Shared Memory时，数据路径上有两个阶段: a)将数据从Global Memory加载到寄存器，b)将数据存储到Shared Memory(其实A100引入了异步访存，可以将数据直接从Global Memory加载到Shared Memory)。因此，将负载从Global Memory到Shared Memory分为两个阶段，并组成一个流水线。在for循环的第一次迭代中的第7-9行和第11行是流水线的启动，之后来自A和B的第一块被加载到共享存储器中。然后，for循环中的第11-16行形成流水线的主体。第12行将LHS密集向量和相应的列索引加载到Shared Memory中，以进行下一次迭代。使用列索引，RHS数据块被预取到第14行的寄存器中，该寄存器与第15行的当前步骤的mma计算重叠。因此，RHS块上的Global Memory访问的延迟是隐藏的。

Emulation for mixed precision

由于应用需求与算力之间的缺口，不同输入采用不同精度的数据也能成为节省开销的一种方式，而Magicube还能实现利用已有精度的计算，来模拟不同种精度之间混合相乘的操作。这个工作是具有开创性意义的，同时我认为也可以辐射到硬件设计中去，是否可以设计相似的硬件来支持这样的计算。

对于SpMM，矩阵A比矩阵B具有更高精度的情况，例如，A是int8，B是int4。由于A已经稀疏，因此A的更高精度可能会带来更高的模型精度。Magicube 还支持A和B都在int16中(Tensor Core并不支持Int16的计算)。这个想法是将一个高精度值分成几个低精度值，然后数学上模拟矩阵乘法，如图19所示。
