0x00 背景和现象

我们的结构是 qwen3 拼接 multi-encoder，让 LLM 理解 N 个模态信息。类似这样（自动忽略projector）：

现象比较奇特，多卡情况下，部分组合 <模型尺寸，zero级别> 总会卡死；数据量调小又正常。具体来说：

尺寸	zero	情况
1B	z0	正常
1B	z2	正常
8B	z0	正常
8B	z2	单 GPU 正常，多 GPU 卡死

进度条会在第一次 backward 之前不动，CPU 利用率 100%，GPU 利用率 100%

0x01 debug 了一下

因为 encoder 不参与训练，之前的实现是：

if not train_encoder:
  model.encoder.require_grad = False

但 deepspeed （0.18.0）切分的时候，不看 require_grad ，只看 model.named_parameters

两个 encoder 都是 transformer 结构，三个模型累计 1400 个 param 参与 deepspeed 切分。

deepspeed 单卡基本不生效
zero0 的时候，底层不是 stage_1_and_2.py （z0和z1z2实现是分开的）
zero2 多卡的时候，param 估计被切成小一万份。 CPU 要一直执行 mask_nan_or_inf_with_val_inplace ，里面会临时申请 torch.tensor

最终 4 卡训练时，2 个进程空耗 CPU；2 个进程在等 all_reduce 。现象就是 CPU 100%、GPU100%，进度条卡住，直到 nccl 超时崩溃。

定位了问题，把不训练的参数改成 buffer 应该可以修复。

0x02 总结
deepspeed 不看 require_grad，需要 frozen 的参数不能给 deepspeed engine
想看多少 param 被处理了，可以改 deepspeed engine 的 __init__。例如 https://github.com/deepspeedai/DeepSpeed/blob/v0.18.0/deepspeed/runtime/engine.py

至于训练数据调小不卡死，“我猜”是因为 mask_nan_xxx 的 callee 和数据正相关，数据少函数跑得快。 没超过 nccl timeout 时间（10min）
