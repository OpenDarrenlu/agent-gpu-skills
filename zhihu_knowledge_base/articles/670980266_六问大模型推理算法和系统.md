一问: 推理成本有多高？

Quora 上面有一个置顶的回答，我深深怀疑是 Poe 的广告

https://www.quora.com/How-much-does-a-single-search-cost-Google
Cost per single google search

这个问答指出 Google 每次搜索付出的成本在 0.5 美分左右。另有一个 Semianalysis 的分析称 LLM 的成本要远低于 <0.5 美分，否则 google 搜索无法盈利。虽然，以上的回答和评估未必靠谱，但是我们大致可以推测搜索引擎的每次查询成本是低于 0.5 美分的。

相比而言，大模型推理的成本目前还居高不下

爆料称微软 Copilot 产品人均每月亏损 20 美元
GPT4 每千 tokens 输入和输出的价格分别是 1 美分和 3 美分

与搜索引擎这种互联网时代的高转化率商业模型相比，将大规模模型运用于其他领域的成本更是令人望而却步。这种高昂的成本导致了大规模模型的商业应用推广策略更为保守。尽管大规模模型已经开始渗透到各个领域，但要实现其大规模商业和民用的应用，还有一段相当的距离要走。预计，为了真正实现大规模应用，成本还需要进一步降低1至2个数量级。

不知道大家对大规模模型推理的成本怎么看待？

二问: 模型和 KV Cache 有多占显存？

当前大语言模型的结构高度收敛，最为流行的是 GPT 和 Llama 类似的模型结构。之前有一篇文章分析了GPT-3 175B 的模型参数如何分布。

截图来自: https://aizi.substack.com/p/how-does-gpt-3-spend-its-175b-parameters

该文章指出了模型 FFN 和 Attention 参数占据了模型参数的绝大部分，基本上超过了 90%。其中 FFN 和 Attention 参数量比例接近 2:1。

除此之外，在大模型推理过程中 KV Cache 的显存管理和占用也是不可忽视的一部分。以 Llama1 结构 70b，batch=12 为例，对 KV cache 的 size 做了预估。根据预估，在 context 长度大于 5k 时 cache size 就超过了 80b(模型大小)。乍一看，这个用量太恐怖了。

Llama 70B KV Cache 大小预估

然而领域的快速进展，总是令人惊喜。不少研究成果都已经发现 Attention 开销太大，并做了模型结构方面的改进。不过这里先只讨论 KV Cache。

实际当中针对 GPT3 和 Llama1 一类的模型分析当中也很容易发现 KV Cache 具有很大的稀疏性。针对此类问题，当前主流的方案是采用 GQA/MQA。

采取这种方案 Llama2 70B 将 KV Heads 的个数从 128 个减少到 8 个，KV cache 的用量降低了 16x。实际当中通过对 KV Cache 进行量化可以进一步成比例地降低显存的占用(简易的 8bit 量化几乎精度无损)。这类方案极大地降低了 Attention 对显存的需求，或者说扭转了这部分的瓶颈也不为过。

考虑到超长上下文(>200k)和超长序列的生成的场景并不多见，尤其超长上下文本身是一个 Compute Bound 的任务；由此推导出采取 GQA 一类方案的大模型，相比于模型参数，大多数情形下推理的 KV Cache 的空间都很充裕。不知道大家同不同意？

三问: 系统的瓶颈在哪里？

自回归大语言模型在运行效率上面最大的缺陷是解码过程是串行和变长的，并行计算和内存带宽资源无法得到高效利用，进而也导致了内存的管理和回收问题。

截图来自:https://www.databricks.com/blog/llm-inference-performance-engineering-best-practices

在系统优化中，我们常提到的 Roofline 模型能够清晰地显示出性能瓶颈，并提供改进系统性能的可能路径。当 Batch 较小时，自回归模型推理通常受限于显存带宽。在不进行任何系统优化之前，使用 PyTorch/Transformers 等算法库运行的 decode 过程 MBU 很容易就超过 70%，但是 MFU 却很低，是典型的 Memory Bound 的系统优化问题。

针对此情形，工业界已经出现了不少的系统优化方案：

Contiguous Batching/Inflight Batching
PagedAttention
GPTQ/AWQ 等量化手段
投机采样

上面每种技术手段都可以大幅度地提升模型推理的速度、性能。在这些手段和优化的基础之上，我们再来回顾下，SOTA 大模型系统是否还是 Bound 在带宽上面？我想问题已经不是那么简单了。

Batching 和量化将模型推理瓶颈从 Memory Bound 逐渐转变为 Compute Bound
Tensor-Parallel 通信的 overhead 逐渐放大
A800 cublas tflops vs batch

上图显示了 A800 GPU 上面 cublas 矩阵乘法的 flops 随 batch 的变化，当 batch>128 时便基本上不成比例增长了，且存在明显的锯齿状。

截图来自:https://www.databricks.com/blog/llm-inference-performance-engineering-best-practices
上图引用自 Databricks 大模型推理的最佳实践，说明了 TP 策略没有达到线性的性能扩展
在没有 NvLink 的消费级显卡上面的 TP 并行若干测试，显示 >30% 时间会花在通信

实际上目前大多数推理芯片是不配置 NvLink 这种高带宽通信的。再结合美国芯片禁令对带宽的限制，当前 SOTA 推理系统在国内应用的一个重要瓶颈是通信 Bound。

四问: TP 并行还是 PP 并行？

下面的表格是对多种参数量模型部署所需的 GPU 最小卡数的预估(未考虑 activation 和其他显存占用)：

	H100(80g)	A100(80g)	RTX 4090(24g)	T4(24g)
140b(half)	4	4	not available	not available
70b(half)	2	2	8	not available
34b(half)	1	1	4	4
140b-W4A16	1	1	4	4
70b-W4A16	1	1	2	2
34b-W4A16	1	1	1	1

从上表可以得到的结论是大多数情况下，大模型推理部署是需要多卡并行的。在大模型推理场景中，目前 Tensor Parallel 相比 Pipeline Parallel 的应用相对更加普及。我推测主要有以下原因:

大模型推理相对耗时，Tensor Parallel 能够成比例降低时延
不合理的 Pipeline Parallel 策略会引入 bubble 加剧系统的瓶颈

除了上述的原因之外，个人觉得开源系统如 vLLM/TGI 还没有支持 Pipeline Parallel 也是一个重要原因。

然而 Pipeline Parallel 也具有明显的优点，那就是它只需要在 Pipeline 的 Stage 之间进行通信。而 Tensor Parallel 在每个 Transformer Layer 需要 2 次通信。Pipeline 策略对于显存小，没有高带宽通信(NvLink) 的系统来说很有意义。我相信拥有相似见解的同学不在少数，至少在 TGI 社区找到了一个 issuecomment:

除此之外，大模型推理的研究在这个方面也有进展。

SARATHI 研究中详细介绍了三种造成流水线泡沫的情况，分别是：

由于连续两个微批次中预填充令牌数量不同而导致的泡沫（PB1）。
当预填充和解码阶段的计算时间不同时，一个阶段紧随另一个阶段之后导致的泡沫（PB2）。
由于请求之间累积的上下文长度（KV缓存长度）不同，导致微批次之间解码计算时间差异而产生的泡沫（PB3）。

为了解决 Pipeline Parallel 相关问题，文章提出的 Chunked prefills & decode-maximal batching 的思路。实现了 PP 并行相比 TP-only 策略 1.48x 吞吐率的提升。得一提的是，在 vLLM 和 TensorRT-LLM 社区，我也发现了对这种策略的开发计划。

五问: 自回归解码算法就这样了吗？

前面提到了大模型推理的一些特点，比如解码并行度低，推理耗时长等问题。怎么提升自回归解码算法本身的并行度，这个领域也特别活跃，趋势也很明显:

Speculative Sampling
Medusa
Lookahead Decoding

这些方法普遍都能看到 2x 左右的加速效果。但是还是存在一些缺陷，限制其在各种场景中的采纳:

投机采样算法需要额外的小模型，会增加 SOTA 模型推理系统设计的复杂度
目前预测的 token 的接受率还较低，没有大规模验证
虽然在速度上面得到了提升，但是实际上浪费了更多计算，性价比可能存在劣势

但是，我仍然非常看好这个方向。2024 年，我对于自回归采样算法改进抱有很大的期待

预训练模型自身能够预测 Next-N Tokens
更高采样准确率和接受率

有没有人觉得投机采样+pipeline 靠谱的？毕竟像 SARATHI 这种系统中，batch 也是受限制的。如果加上投机采样，我相信 batch 还能往上撑一撑。抱着乐观主义的心态，系统优化本身就是一个 balance 和 trade-off 的哲学，我相信这里面的空间还不小 。

六问: 大模型推理框架鹿死谁手？

说句心里话，大模型的崛起给推理系统带来了很大的变化。但是离大模型的大规模和广泛应用估计还有一段时间。目前领域还处在快速变化和发展的阶段，实在难以下论断。但是，就服务器端我还是非常看好 vllm，原因也没啥就是拥有良好开放的社区。

当然 TensorRT-LLM 也是一个非常强劲的玩家，它的成功与否，决定性因素更多在于 NV 硬件能否继续保持垄断。不过，大模型的用户是否会像以往 CV 的用户一样为 TensorRT 的兼容性、灵活性、维护成本等继续买单也是一个值得考量的问题。

除此之外，国产芯片公司肯定会是这个领域的玩家。不过我的亲身体验是国产芯片的软件适配还很不成熟。某国产芯片硬件性能声称对标 A800/A100，但是软件上面差了一大截。对于国产芯片我想说的是，行业变化这么快别让软件拖了后腿，拥抱社区和开源才是正道。
