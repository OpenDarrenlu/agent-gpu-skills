更尴尬的是，在这个复杂项目里，uv 和 conda 并不完全兼容。

第一次运行：

假设 conda activate py3124 环境
直接 uv venv，ray node executable 指向的是 conda 的 python bin，而不是 venv 的 python bin。这会导致 import error

第二次运行：

假设 conda activate py3124 环境，然后 conda deactivate 回到 base
此时再 uv venv，ray node 运行会 python version mismatch。也就是说，deactivate 过的 conda env，在 uv 里依然生效

第三次运行：

我不用 conda 了，直接退

因为 uv 要从 github 下载 cpython3.12.11，刚好有墙访问不了.. uv 又是编译型的你改不了（后来找到个配置 UV_PYTHON_INSTALL_MIRROR）

没办法，conda env 升最新吧，跳过 mismatch。

第四次运行：

... os error 95..权限不足

这不是真的目录权限有问题。作为干崩服务器多次的老玩家，习惯性把系统盘和数据盘分开，做个 softlink，这样：

ls ~/
.cache ->  /data/user/.cache
.local

训练服务器挂载了不同类型的存储，导致代码和uv环境并不在同一块物理盘上。uv 要求 .local .cache 和源码需要在一起。

此外还有 wandb 不通，我已经发了 PR 支持。

0x02 data module 设计

官方的 data loader 设计，是最不像炸炉师的地方.. 对 user-defined data 支持度为 0，需要自己改源码。

data_type 和 data_path 耦合

    data_cls = data_config["dataset_name"]
    elif data_cls == "openmathinstruct2":
        data = hf_datasets.OpenMathInstruct2Dataset(
            split=data_config["split"],
            output_key=data_config["output_key"],
            prompt_file=data_config["prompt_file"],
        )
    elif data_cls == "openai_format":
        data = hf_datasets.OpenAIFormatDataset(
            data_config["train_data_path"],
            data_config["val_data_path"],
            data_config["chat_key"],
            data_config["system_key"],
            data_config["system_prompt"],
        )

假设训练数据是常见的 openai-format，名字必须叫 openai_format，而且只能有一个路径；一旦换了名字，config 不写 system_key 可能就崩溃。

现实是：训练数据来自很多地方，不可能只有一个文件固定叫 openai_format.json
