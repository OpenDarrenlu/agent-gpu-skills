本文是GPU渲染之路专栏的第二篇文章，应用层我们就不谈了，直接开始探索如何实现跨平台层的渲染引擎，尝试对不同设计的图形api进行抽象。在本章，我们将迈出自顶向下的第一步！也就是图形应用经过引擎层映射到某个具体的图形API的过程。

我们继续回顾一下，计算机图形系统大概分为：

(1)应用层：包括各种图形应用，如游戏、模拟器、设计工具等。这些应用使用下层的服务和功能，以创建各种各样的图形和交互体验。

(2)跨平台层：这一层包括各种图形引擎和工具，如Unity、Unreal Engine、bgfx、ANGLE等。这些引擎和工具除了提供跨平台的能力之外，往往还会提供了一组高级的功能，如物理模拟、光照计算、动画系统等，使得开发者可以更容易地创建复杂的图形应用。

(3)用户态图形驱动层：这一层包括操作系统和图形API（如OpenGL和DirectX）。操作系统负责管理硬件资源，如内存和CPU时间，以及提供基本的服务，如文件系统和网络。图形API则提供了一组函数和数据结构，使得开发者可以更容易地使用图形硬件的功能。

(4)内核态驱动层：驱动层是硬件层和操作系统之间的接口。负责管理硬件设备，如初始化设备、处理设备错误、将操作系统的命令转换GPU件可以执行的命令等。

(5)硬件层：这是计算机图形系统的最底层，包括图形处理器（GPU）、显示器、输入设备等硬件设备。GPU负责执行大量的并行计算，以快速生成图形。显示器则负责将生成的图形显示给用户。输入设备，如鼠标和键盘，允许用户与图形系统交互

1 图形应用的跨平台

对于大多数图形应用程序员来说，直接接触的往往是跨平台层而不是用户态的图形API，因为作为一个生产应用一般都需要具备跨平台的能力(安卓、IOS、Linux、Mac、Windows、Web、主机等等)。可参考如下信息：

如果 Vulkan 的优势是跨平台的话，那等于说没有优势。因为现在除非写玩具，每个 app 都要有自己对 graphics API 的抽象层。这个抽象层要是不能兼容主流的几个 API，就是设计失败。我们的 app 里有个三年前设计的 graphics 抽象层。去年花了半年就支持 Metal 了。

链接：https://www.zhihu.com/question/46536915/answer/129902493

一个图形应用需要具备跨平台的能力，至少需要解决以下4个问题：

(1)代码的跨平台

(2)图形API的适配

(3)平台API的适配

(4)资源的格式转换

举个例子，PC上的unity端游，如果要移植到浏览器Web端，那么对应的方案就是：

1.C++代码转换成js或者是webassembly的格式，内嵌在浏览器里运行。

2.OpenGL/Direct3D的API调用转换成WebGL的调用。

3.POSIX API 调用转换成对应的浏览器API调用。

4.图片、音视频、字体等资源的格式转换。

在游戏引擎架构里，会有一个"平台无关层"来实现这里所有的工作，这里就不展开说了，对游戏引擎感兴趣的读者可以自行阅读miloyip(叶劲峰)大佬翻译的《游戏引擎架构》。在跨平台层我们也可以细分两种情况：

(1)第一种就是游戏引擎以及设计工具，除了提供跨平台的能力，往往还会提供了一组高级的功能，如物理模拟、光照计算、动画系统等，这对创建复杂的图形应用来说相当重要(如上图，现代游戏引擎已经非常强大了)。

(2)第二种就是bgfx、OGRE这些跨平台图形库，主要是对图形API进行适配和抽象。

2 图形API的兼容和适配

本文就不对其它内容进行展开了，单纯只讨论图形API的兼容和适配。要做到这一点，大概需要完成以下工作：

1.抽象：提供一个通用的抽象层来简化底层图形API的使用。这个抽象层允许开发者使用一套统一的API来处理不同平台和图形API的差异。这样，开发者可以专注于实现渲染逻辑，而不需要关心底层API的具体实现细节。

2.集成渲染后端：集成主流的图形API。

3.调度：设计一个合适的选择策略，在不同的平台上将通用API转换为某个底层图形API的具体调用。

常见的跨平台渲染引擎如下：

引擎	封装的图形API	支持的平台
bgfx	Direct3D 9/11/12, OpenGL, OpenGL ES, Metal, Vulkan, WebGL	Windows, macOS, Linux, FreeBSD, iOS, Android, Raspberry Pi, HTML5
OGRE	Direct3D 9/11, OpenGL	Windows, macOS, Linux, iOS, Android, Windows Phone
Irrlicht	Direct3D 8/9, OpenGL, Software Renderer	Windows, macOS, Linux, Partial support for iOS and Android
Cinder	OpenGL, Partial Direct3D	Windows, macOS, Linux, iOS, Android
Panda3D	OpenGL, Direct3D	Windows, macOS, Linux, FreeBSD
Magnum	OpenGL, OpenGL ES, WebGL, Partial Vulkan	Windows, macOS, Linux, iOS, Android, Windows RT, HTML5
3 通用API如何调度映射到真实图形API

关于抽象这一块就不展开说了，作者也没有那么深厚的功力，希望大家能大致理解跨平台这层所负责的工作就行。到目前为止我们自顶向下的调用路径是：

应用层代码-->通用图形API-->调度策略-->真实图形API

调度策略的目的选择最合适的渲染后端，让应用程序在当前硬件和平台下能获得更优的新性能，这里我就直接研究bgfx的源码好了。

struct RendererType这个结构体枚举了目前bgfx支持了渲染后端，在新建Renderer时候会从这些枚举值里唯一确定一种。

	struct RendererType
	{
		/// Renderer types:
		enum Enum
		{
			Noop,         //!< No rendering.
			Direct3D9,    //!< Direct3D 9.0
			Direct3D11,   //!< Direct3D 11.0
			Direct3D12,   //!< Direct3D 12.0
			Gnm,          //!< GNM
			Metal,        //!< Metal
			Nvn,          //!< NVN
			OpenGLES,     //!< OpenGL ES 2.0+
			OpenGL,       //!< OpenGL 2.1+
			Vulkan,       //!< Vulkan

			Count
		};
	};


还有一个关键的结构体 struct RendererCreator，其定义图形API的创建函数、销毁函数、名称以及当前硬件和操作系统环境下是否被支持。在初始的时候就将渲染后端的这些信息注册到s_rendererCreator这个数组里。

	struct RendererCreator
	{
		RendererCreateFn  createFn;
		RendererDestroyFn destroyFn;
		const char* name;
		bool supported;
	};
	static RendererCreator s_rendererCreator[] =
	{
		{ noop::rendererCreate,  noop::rendererDestroy,  BGFX_RENDERER_NOOP_NAME,       true                              }, // Noop
		{ d3d9::rendererCreate,  d3d9::rendererDestroy,  BGFX_RENDERER_DIRECT3D9_NAME,  !!BGFX_CONFIG_RENDERER_DIRECT3D9  }, // Direct3D9
		{ d3d11::rendererCreate, d3d11::rendererDestroy, BGFX_RENDERER_DIRECT3D11_NAME, !!BGFX_CONFIG_RENDERER_DIRECT3D11 }, // Direct3D11
		{ d3d12::rendererCreate, d3d12::rendererDestroy, BGFX_RENDERER_DIRECT3D12_NAME, !!BGFX_CONFIG_RENDERER_DIRECT3D12 }, // Direct3D12
		{ gnm::rendererCreate,   gnm::rendererDestroy,   BGFX_RENDERER_GNM_NAME,        !!BGFX_CONFIG_RENDERER_GNM        }, // GNM
#if BX_PLATFORM_OSX || BX_PLATFORM_IOS
		{ mtl::rendererCreate,   mtl::rendererDestroy,   BGFX_RENDERER_METAL_NAME,      !!BGFX_CONFIG_RENDERER_METAL      }, // Metal
#else
		{ noop::rendererCreate,  noop::rendererDestroy,  BGFX_RENDERER_NOOP_NAME,       false                             }, // Noop
#endif // BX_PLATFORM_OSX || BX_PLATFORM_IOS
		{ nvn::rendererCreate,   nvn::rendererDestroy,   BGFX_RENDERER_NVN_NAME,        !!BGFX_CONFIG_RENDERER_NVN        }, // NVN
		{ gl::rendererCreate,    gl::rendererDestroy,    BGFX_RENDERER_OPENGL_NAME,     !!BGFX_CONFIG_RENDERER_OPENGLES   }, // OpenGLES
		{ gl::rendererCreate,    gl::rendererDestroy,    BGFX_RENDERER_OPENGL_NAME,     !!BGFX_CONFIG_RENDERER_OPENGL     }, // OpenGL
		{ vk::rendererCreate,    vk::rendererDestroy,    BGFX_RENDERER_VULKAN_NAME,     !!BGFX_CONFIG_RENDERER_VULKAN     }, // Vulkan

最后bgfx在rendererCreate()这个核心方法里实现了对渲染后端的选择，其方式是对每一个渲染API进行打分，记录在scores这个数组里，最后排序返回得分最高的渲染API。

RendererContextI* rendererCreate(const Init& _init)
	{
		int32_t scores[RendererType::Count];
		uint32_t numScores = 0;

		for (uint32_t ii = 0; ii < RendererType::Count; ++ii)
		{
			RendererType::Enum renderer = RendererType::Enum(ii);
			if (s_rendererCreator[ii].supported)
			{
				int32_t score = 0;
				if (_init.type == renderer)
				{
					score += 1000;
				}

				score += RendererType::Noop != renderer ? 1 : 0;

				if (BX_ENABLED(BX_PLATFORM_WINDOWS) )
				{
					if (windowsVersionIs(Condition::GreaterEqual, 0x0602) )
					{
						score += RendererType::Direct3D11 == renderer ? 20 : 0;
						score += RendererType::Direct3D12 == renderer ? 10 : 0;
					}
					else if (windowsVersionIs(Condition::GreaterEqual, 0x0601) )
					{
						score += RendererType::Direct3D11 == renderer ?   20 : 0;
						score += RendererType::Direct3D9  == renderer ?   10 : 0;
						score += RendererType::Direct3D12 == renderer ? -100 : 0;
					}
					else
					{
						score += RendererType::Direct3D12 == renderer ? -100 : 0;
					}
				}
				else if (BX_ENABLED(BX_PLATFORM_LINUX) )
				{
					score += RendererType::OpenGL   == renderer ? 20 : 0;
					score += RendererType::OpenGLES == renderer ? 10 : 0;
				}
				else if (BX_ENABLED(BX_PLATFORM_OSX) )
				{
					score += RendererType::Metal    == renderer ? 20 : 0;
					score += RendererType::OpenGL   == renderer ? 10 : 0;
				}
				else if (BX_ENABLED(BX_PLATFORM_IOS) )
				{
					score += RendererType::Metal    == renderer ? 20 : 0;
					score += RendererType::OpenGLES == renderer ? 10 : 0;
				}
				else if (BX_ENABLED(0
					 ||  BX_PLATFORM_ANDROID
					 ||  BX_PLATFORM_EMSCRIPTEN
					 ||  BX_PLATFORM_RPI
					 ) )
				{
					score += RendererType::OpenGLES == renderer ? 20 : 0;
				}
				else if (BX_ENABLED(BX_PLATFORM_PS4) )
				{
					score += RendererType::Gnm      == renderer ? 20 : 0;
				}
				else if (BX_ENABLED(0
					 ||  BX_PLATFORM_XBOXONE
					 ||  BX_PLATFORM_WINRT
					 ) )
				{
					score += RendererType::Direct3D12 == renderer ? 20 : 0;
					score += RendererType::Direct3D11 == renderer ? 10 : 0;
				}

				scores[numScores++] = (score<<8) | uint8_t(renderer);
			}
		}

		bx::quickSort(scores, numScores, sizeof(int32_t), compareDescending);

		RendererContextI* renderCtx = NULL;
		for (uint32_t ii = 0; ii < numScores; ++ii)
		{
			RendererType::Enum renderer = RendererType::Enum(scores[ii] & 0xff);
			renderCtx = s_rendererCreator[renderer].createFn(_init);
			if (NULL != renderCtx)
			{
				break;
			}

			s_rendererCreator[renderer].supported = false;
		}

		return renderCtx;
	}


代码虽然没有注释但是还是比较好读懂的，逻辑大致如下：

1.每个渲染API初始都是0分，先判断当前是否supported，是的话就+1000分。

2.然后分类讨论的不同操作系统的情况，首先是WINDOWS：

(1)如果windows版本=0x0602(WIN8)，Direct3D11+20分，Direct3D12+10分，其它不变，。

(2)如果windows版本=0x0601(WIN7)，Direct3D11+20分，Direct3D9+10分，然后给Direct3D12扣100分。

(3)其它windows版本，给Direct3D12扣100分，其余不变。

3.LINUX：

OpenGL+20分，OpenGLES+10分，其余不变。

4.OSX：

Metal+20分，OpenGL+10分，其余不变。

5.IOS：Metal+20分，OpenGLES+10分，其余不变。

6.安卓、H5、RPI

OpenGLES+20分，其余不变。

7.PS4：

Gnm+20分，其余不变。

8.XBOXONE、WINRT：

Direct3D12+20分，Direct3D11+10分，其余不变。

根据这个策略总结一下就是：

1.windows上最佳的选择永远是Direct3D，根据不用系统版本选择合适的版本。

2.linux用OpenGL，OSX和IOS都用Metal，XBOX可以用最新的Direct3D12、Direct3D11。

3.安卓和Web端用OpenGLES (WebGL就是OpenGLES)。

bgfx并没有对这部分逻辑进行解释，但大体看来是很符合直觉的，如果有想自己实现跨平台渲染引擎的同学可以自行研究其它引擎是如何实现的。

5 总结

这一章我们的应用层代码已经算是走到图形API这一层：

应用层代码-->通用图形API-->调度策略-->真实图形API

我们再来回顾一下跨平台层所做的工作是：

1.适配不同平台图形API，让你的图形应用能发布在各个平台而不再是个demo玩具。

2.提供图形相关的高级功能，如粒子系统、物理模拟、光照计算、动画系统等，使得开发者可以更容易地创建复杂的图形应用。
