背景

在学习蚂蚁贡献“Opt MHA chunked prefix: merge prefix and extend kv cache to run mha once (#10953)”，发现之前有一个mha fa3性能相关的pr #9551，通过观察extend * (extend + prefix)，分析mha和mla那种attention算法效率高。这个pr跟#5624作用类似，没有被接受。这两个pr告诉我们一个事情，mha和mla算法是可以转换的，具体内容可以参考DeepseekV2AttentionMLA的forward_normal和forward_absorb实现。

如何实现/优化MLA？选择MLA还是MHA？

对于这一话题，在过去的2025很多patch都对这个部分做了贡献。本文旨在记录相关patch，感谢各位大佬！

开启MLA时代

MLA支持始于#905, 这个patch打通了attention计算的common path，从scheduler, model (deepseek_v2), 到attention各个环节做了相应的支持。#2349在没有prefix场景prefill选择MHA进行计算。

直到#3553为止，只有tritton的backend支持MLA。

#3550，#3643这一组patchset实现中，修改直接在FlashInferAttnBackend实现的。

到这个patch为止，代码存在3种执行路径

enable_flashinfer_mla (true), disable_radix_cache (true): flashinfer算子不使用kvcache数据（菜鸟理解)
enable_flashinfer_mla (true), disable_radix_cache (false): flashinfer算子使用压缩版本kvcache（lora）数据（菜鸟理解)
enable_flashinfer_mla (false): 非deepseek v3/r1情况

非MLA的flashinfer backend

1) prefill:
   a) no prefix: MHA
      * BatchPrefillWithRaggedKVCacheWrapper
   b) has_prefix: MHA
      * extend: BatchPrefillWithRaggedKVCacheWrapper
      * prefix: BatchPrefillWithPagedKVCacheWrapper
2) decode: BatchDecodeWithPagedKVCacheWrapper

1) prefill: BatchMLAPagedAttentionWrapper

开启ragged prefill

forward_absorb在long context场景下可能会放大计算量，prefill阶段本身就是compute bound，放大计算量会对性能有直接影响。#3967进行prefill计算，没有prefix场景下利用ragged接口进行计算，避免不必要的计算量。

后续优化
MHA with chunked KV cache when prefilling on long sequences (#5113): 当sum(prefix_lens)超过threshold切换mha chunked kv方式，先计算extend的attention；然后分为bs批次计算prefix的attention，通过merge_state进行online softmax汇总，仅支持fa3
Fix FA3 DeepSeek prefill performance regression (#5624): 没有prefix，在prefill阶段选择mha
https://github.com/sgl-project/sglang/pull/5624

在comments解释mla与mha+k_b_proj计算量对比，但是fix很简单，只是在没有prefix场景切换成mha chunked kv

Support MHA with chunked prefix cache for flashinfer/flashmla (#8616): flash mla attention backend支持动态选择mha chunked kv
Opt MHA chunked prefix: merge prefix and extend kv cache to run mha once (#10953): 当满足sum(forward_batch.seq_lens_cpu) <= forward_batch.get_max_chunk_capacity，把mha chunked kv中extend和prefix合成一次attention计算，k包含了（prefix和extend），q（extend）
Support MHA with chunked prefix cache for DeepSeek chunked prefill (#5113)

chunked kv初衷是把整个bs中的prefix切分成若干个chunk，每个chunk可以处理的token数量是固定的，即：ForwardBatch::get_max_chunk_capacity；在chunk内部bs中每个seqs平分chunk_capacity，chunk保存的是一段token序列，即便某个seq没有token需要保存，这部分capacity也不会分配给其他seq。

一共需要多少chunk呢？有bs中最长的seqs决定。

如果当前batch中L（bs，seq）描述的工作量可以被一个chunk处理完，可以把extend和prefix的attention合并一次进行计算。

flashinfer mla backend

1) prefill:
   a) no prefix: MHA
      * BatchPrefillWithRaggedKVCacheWrapper
   b) has_prefix && prefix exceeds threshold: MHA
      * extend: BatchPrefillWithRaggedKVCacheWrapper
      * prefix: BatchPrefillWithRaggedKVCacheWrapper (prefix handled in chunked kv in deepseekv2 (prepare key) & FlashInferMhaChunkKVRunner (update kv_indptr))
   c) has prefix && prefix below threshold: MLA
      * BatchMLAPagedAttentionWrapper
2) decode: BatchMLAPagedAttentionWrapper
