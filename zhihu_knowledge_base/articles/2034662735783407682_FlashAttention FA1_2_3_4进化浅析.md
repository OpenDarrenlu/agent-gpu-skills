FA1

先问大家一个问题，实现FA里online softmax最大的阻碍是哪一个？

A. S=QK
B. rowsum
C. rowmax

我的答案是C，rowmax。

22年那会我恰巧也在做attention性能优化, 当时假设的attention是这么算的（这里我就忽略了很重要的东西rowmax M）：

Attention pseudo code
S=QK
P = exp(S)
L = sum(P, 1)
P = P/L
O = PV

这里先声明下，为了大家阅读顺畅我用了非常简化的式子，上面Q K V S P O不必多说，L就是softmax的分母也就是所谓的rowsum。

假设Q长度是m，K V的长度是n。每个cta会分到一个M上大小为(bM, d)的block （这里batch size 和 num heads我省略掉了）。结果O也是(bM, d)那么中间的S和P的计算是需要在n这个维度上累加的。为了方便适配smem的大小，K V也是会被分为(bN, d)的小block。假设smem的大小如下图 ￼ 那么这里性能瓶颈就是smem的大小。 如果算出来一整个S，他是挤不进SMEM里的，那么只能放gmem里，会变得超级慢，所以online softmax是一个自然的想法。下图就是大概的一个CTA里的tiles。

随便举的例子，不小心凑了个正好的长方形

当时想online softmax能有何难，就按照下面的思路实现了下：

My Attention pseudo code
loop n // bN
S=QK
P = exp(S)
L += sum(P, 1)
O = PV
O = O/L

做了两点改动：

让L变得累加，因为P每次是算的一个bN，下个循环就扔，L需要累加记录。
我不在循环中去P=P/L, 而是算完了最终在O上rescale一下，因为L只是一个(bM, 1)的vector，在N上处处相等，所以完全可以拿出循环。

上述的计算方式没有任何额外计算，完美实现了online softmax，可以说是最优解，直到数值错误给了我当头一棒。

浮点数的Exponential计算有个很重要的特性，在负数域接近0的位置数值最准确。 这就是为啥我们需要减去rowmax。 所以实际上的运算是这样的

Attention
S = QK
M = max(S, 1)
P = exp(S-M)
L = sum(P, 1)
P = P/L
O = PV

这个时候如果要实现online softmax可不是仅仅每个循环记录新的max M这么简单。一个重要的变化就是每个循环M的不一样导致了P的scale不一样，P scale的不同导致了，L和O在每个循环的scale都不同，由于L和O都需要累加，累加之前得rescale它们。 所以：

需要累计M
L累加前需要rescale
O累加前需要rescale

这个rescale是这样的：假设第一个循环的M是M0，第二个循环是M1。首先M1=max(M0, M1). 然后因为上个循环的 L0 = sum(exp(S0-M0), 1) = sum(exp(S0), 1) * exp(-M0). 以把scale变成exp(-M1)为目标的话，只需要乘以exp(M0-M1)即可。O也是同理。（注意：对于循环i, M_{i-1} 指的是之前所有循环的最大值。 L_{i-1} 也是之前的累加的rowsum）

FA2

S1=QK1
M1 = max(S1, 1)
M1 = max(M1, M0)
P1 = exp(S1-M1)
L1 = sum(P1, 1)
L1 += exp(M0-M1) x L0
O1 = P1V1
O1 += exp(M0-M1) x O0
O = O/L

为啥是FA2呢 因为FA1是一个劣化的版本

FA1
loop
S1=QK1
M1 = max(S1, 1)
M1 = max(M1, M0)
P1 = exp(S1-M1)
L1 = sum(P1, 1)
L1 += exp(M0-M1) x L0
P1 = P1/L1
O1 = P1V1
O1 += L0/L1 x O0

他没有考虑一步到位的 O = O/L, 所以每一步需要P=P/L让P成为一个临时的以当前L为基准的softmax。这增加了P1=P1/L1这一步的计算。（Note：P1/L1这一步操作把分子分母exp(-M)的scale给抵消掉了，所以后面的O的rescale是基于L的而不是M）。这体现了当时Tri Dao的手法还有些稚嫩，我觉得大家不需要记住FA1的这个版本。

反思一下自己，我之所以没有想到FA的算法一方面是浅尝辄止，另一方面是没有想到每个循环额外rescale两个操作居然依然还能节省时间。所以我感受的FA的最大贡献是展示了用额外计算来换访存时间的一个极其有效的案例。

LSE (log-sum-exp)

我们看到FA的计算除了返回O之外还要返回LSE，这是为啥？

这是给反向传播准备的 为了节省空间反向传播的时候需要重新计算S=QK, 但是P要不要从S重新算呢？之前说过了，这样需要再来一遍n上的循环累加，rescale什么的。其实我们之前算过一遍之后只需要记住一个 (m, 1)大小的LSE vector就可以把计算P的这个昂贵的reduction且多轮的过程转变成一个廉价的一轮element-wise的过程。

P=exp(S-LSE) = exp(S) / exp (log sum exp(S)) = exp(S) / sum exp(S) = P

最后一步就是softmax的定义

上面说的是怎么用LSE到反向传播中。还有个问题没解决就是怎么得到LSE

LSE = log sum exp(S) = log sum exp(S-M+M) = log sum exp(S-M) + M = log(L) + M

M在n上处处相等，当然是可以提出来的。用到这个式子的时候只需要放到循环外计算即可。

backward

既然说到反向传播了 咱就再提一下他的实现吧。本文的主旨依然还是浅析，所以backward不是本文的重点，并且FA4的backward还是很繁琐的。如果大家想看更深入的backward的实现可以评论区跟我说。

我们就说 计算 dSij = Pij(dPij - Di)这一步 原本Di这个东西是 Pi 和 dPi的点乘.

根据 dP = dO V

P_i \cdot dP_i = P_i \cdot \sum_j(dO_i \cdot V_j) = dO_i \cdot \sum_j(P_i \cdot V_j) = dO_i \cdot O_i

这个转换本质上就是把D的计算从P dP转换到O dO上了 这有两个好处

不依赖中间的计算结果P，dO和O都是backward的输入，可以在backward kernel之前就是预计算 D
Pi和dPi的点乘需要在n上累加，而Oi和dOi只需要在固定的d上累加即可

FA3

从FA3开始，是开始针对GPU架构了，FA3是hopper，FA4是blackwell

重要的instruction是: setmaxnreg, tma, wgmma. 主要是使用了warp specialization（WASP），我之前的文章有讲 [Kernel优化] Software Pipeline / Warp Specialization / Persistent 及其一些思考。总结来说,hopper带来的好处是：

tma减少了register的开销；
setmaxnreg使得WASP真正能够把producer的剩余register匀到consumer上，减小了consumer上的register压力；
wgmma加大了tile size和吞吐，CUDACore softmax成为瓶颈，使得CUDACore和TensorCore的overlap变得重要。

在WASP基础之上，为了增加CUDACore和TensorCore的overlap，又有两个优化：

Inter-warpgroup：一个是CUDACore和TensorCore之间的Pingpong。两个warpgroup的TensorCore如果同时执行是会有冲突的。所以两个wgmma互相竞争耽误，还不如pingpong起来：一个在wgmma的时候另一个去做softmax。用barrier来控制。如下图b
intra-warpgroup：在同一个warpgroup里不同的iteration之间也是可以overlap起来的。比如循环j先把 S_j=QK_j 和 P_j = Softmax(S_j) 给算了，在循环j+1里计算完 S_{j+1}=QK_{j+1} 之后开始 P_{j+1} = Softmax(S_{j+1}) 的时候同时把上一个循环没算的 O+=P_{j}V_{j} 给算了。实现了TensorCore和CUDACore的overlap。如下图c

FA4

可以直接看我写的文章：[深挖FA4] FA4的优化思想和代码解析

blackwell最明显的改动是曾经hopper的wgmma输出的结果必须在register里，所以mma consumer永远是register的主要压力来源。把mma warp拆分成mma 和 softmax/correction/epilogue等多个角色是毫无动力的，因为要有额外的通信把mma的结果给softmax/correction/epilogue，而且register压力直接加倍（mma warp不减，多的role还得加register），毫无收益。

现在tcgen05.mma的结果是存在tmem里的，这个一方面tcgen05直接可以让mma 只用一个warp，而且极少的占用register。同时softmax/correction/epilogue拿result完全不用考虑通信问题，tmem是cta内共享的。这也就造成了为什么FA4的分工可以如此的细化：

Warp(s)	角色	功能
14	Load	TMA 加载 Q, K, V 从 GMEM 到 SMEM
12	MMA	GEMM 操作 (Q×K 和 P×V)
0-3	Softmax0	Stage 0 的 softmax 计算
4-7	Softmax1	Stage 1 的 softmax 计算
8-11	Correction	重缩放累积的 O，写入 sO
13	Epilogue	从 SMEM 拷贝 O 到 GMEM
15	Empty	填充 warp (无工作)

fa4优化（摘自[深挖FA4] FA4的优化思想和代码解析）

softmax一行reduction只需要同一个thread。blackwell下的softmax因为是通过tmem来传输P，和hopper最大的不同就是，hopper用的register导致QK的结果每行会分散在4个thread里（https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#wgmma-64n16-d）。而tmem在load的时候就可以规定每行都在一个thread里，避免了warp内计算rowmax时shuffle sync的overhead。
rescale O被摘出了critical path：我们定义O1 += exp(M0-M1) x O0就是rescale或者叫correction。在hopper时，这个是跟着PV的mma的，因为需要用到PV的结果O，只有等他算完才能进入下一个循环。现在O在tmem里了，可以单独使用correction warp来做这件事，把rescale拿出critical path，直接进入下一个循环。
tmem上的multi stage：之前需要在smem上通过冗余空间来做multi stage，现在tmem也肩负起了mma/softmax/correction warp之间producer/consumer的通讯任务，为了能让它们overlap，也得有个multi stage的机制，这个是围绕着q_stage=2的情况。一个tmem上同时预留了两个tO和两个tS的空间（tS tP共享）使得High Q tile和Low Q tile可以交错进行
split p：1 是节省softmax warp register 压力。softmax warp输入S需要128个register输出P需要64个fp16x2的register。但是程序上我们只给了他192个register。所以把P存入tmem的过程分成3/4 和 1/4两个部分。2. 3/4的部分结束后直接发起PV mma，让softmax和mma还能一定程度上overlap。
FMA和MUFU的overlap：就是用polynomial approximation去模拟一部分的exponential function。一些用MUFU.EX2来跑，另一些用FMA来模拟。
跳过rescale O：上面2说了一个单独的correction warp在做rescale O。这个优化就是干脆不错了，只要M0和M1的差没大到一定程度就直接跳过。
总结和闲聊

整个的发展过成是：软件->调度->硬件。优化是越来越硬件导向，也越来越engineer了。

自从fa3开始wasp的崛起，以triton为首的tile language露出了个大破绽：用户无法手动给不同的warp指定工作。这本来不是大问题，可以自动帮用户加wasp嘛。但是想要把wasp自动化优雅的塞进triton的优化路线里需要好好设计，所以就硬是拖到tawa的出现之前triton都是用的普通SWP的attention实现。这个期间就出现了大量的dsl或者一些优化工作想要补上这个缺口。

从fa的进化来看，优化一直是走在这些dsl前面的。fa4很多深度定制的优化确实是high-level的dsl做不到的。但是另一方面，fa4整体上也是有些偏离学术论文，更像是一篇优化报告。全文着墨较多比较有创新点的ex2 emulation实际提升并不是很高。

总的来说，对于kernel的极致优化我还是比较推崇CuTeDSL的，但是以fa4为例感觉工程压力过大从而很难做出学术成果。目前看来可能也只有Agent GPU Kernel是一个好的出路了。目前Nvidia AVO已经占坑，不过还没开源，短时间内可能是个蓝海。
