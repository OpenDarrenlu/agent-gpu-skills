注：打满的严格定义需要在硬件上先做一遍microbenchmark。我们姑且假设 >80% DRAM bandwidth util 是我们期望的所谓 好 的perf。

如何在理论上打满dram bandwidth

根据 littles law ：loads in flight needed = throughput * latency

假设dram read latency 是1200 clock cycle, dram working at 2690MHz，那一个dram load latency 就是400ns。

所以要打满 3.35TB/s 的dram 需要整个 gpu 持续产生不少于大概1.5MB的dram loads in flight。

在unroll 8的情况下，我们的kernel 需要48个register，达到的 occupancy 是大概60%。(40 warps per sm)。

在hotness 足够的情况下，这8个load (ldg.e.128)是持续并发的。所以理论上我们能够产生 loads per warp * warp per sm * sm per GPU = 8 * 128B * 40 * 132 = 5.19 MB 的 loads in flight。

这5.19MB的load 跟实际的workload 大小无关。在embedding width 和hotness足够大的情况下，这5.19MB 是我们的kernel 的unrolled ldg.e.128 count和 occupancy决定的。

这些 loads in flight 有可能会被 L1 或者 L2 截获。那如果根据理论上的计算，即使cache hit 达到70%，我们也有足够的loads 能达到 DRAM，打满bandwidth。

反过来说，如果我们的cache hitrate 过高，5.19MB 的 load 被过滤了太多，产生不了1.5MB的dram loads in flight。那我们就没办法saturate dram。

所以我们需要观察有多少read 到达了dram 。关键指标是 (1 - L1 hitrate) * (1 - L2 hitrate)。理论上，根据我们上面的计算，这个指标不能低于30%。

首先几个趋势：

alpha = 0, 1.05, 1.15 的情况下， hotness 越多，dram util 越好，最终稳定在80% dram bw util以上。
这个符合我们的预期。我们的要求是GPU 能够产生持续的 8 个 并行的ldg.e.128。 那hotness 越多越符合我们的要求。
基本上大于dram loads pct > 30%的那些都能达到80% dram util 以上。
在 alpha = 1.2, 1.25 的情况下，即使有很多的hotness，我们也没办法达到80%的dram bw util。
当我们把 amount of dram reads out of all loads 画出来后，我们就看到，这些high alpha 的情况下我们到达dram 的数量基本都 < 30%， 甚至 < 25%。那肯定是打不满的。

所以这就说明了我们理论上的分析是基本正确的。
