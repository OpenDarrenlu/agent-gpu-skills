从SM80到SM90/100，mbarrier的SASS编译结果揭示了一个关键变化：指令不再是简单的共享内存直接访存，而是使用了新的SYNCS指令。第1节介绍了mbarrier指令族，第2小节介绍了SM80和SM90/100的编译SASS指令的对应关系，第3节介绍了一些微架构猜想，第4小节列出了测试程序的源程序。
1. PTX mbarrier 指令族：管理线程到达和异步事务的同步工具

整体来看，mbarrier 指令族可以分成几类：init/inval 负责对象生命周期，arrive/arrive_drop 负责线程到达，test_wait/try_wait/pending_count 负责等待和查询，expect_tx/complete_tx/cp.async.mbarrier.arrive 负责把异步事务也纳入同步过程。它的核心作用就是：把线程同步和异步内存操作同步放到同一个 barrier 机制里管理。

PTX 里和 mbarrier 直接相关的指令大概有 10 种：

mbarrier.init
mbarrier.inval
mbarrier.expect_tx
mbarrier.complete_tx
mbarrier.arrive
mbarrier.arrive_drop
mbarrier.test_wait
mbarrier.try_wait
mbarrier.pending_count
cp.async.mbarrier.arrive


mbarrier.init 和 mbarrier.inval 负责管理对象生命周期。前者把 shared memory 里的一块空间初始化成合法的 mbarrier 对象，后者表示这个对象之后不再作为 barrier 使用，可以释放出来做别的用途。

mbarrier.arrive 是最基础的到达操作。线程执行它之后，相当于告诉 barrier：“我这一份已经到了。”但它不会像传统 barrier 那样阻塞线程，所以线程可以继续执行后面的代码。mbarrier.arrive_drop 则多了一层含义：这次我到了，但后面的 phase 不用再等我了。这个在循环、pipeline，或者某些线程提前退出时比较有用。

mbarrier.test_wait 和 mbarrier.try_wait 用来检查当前 phase 是否完成。一个 phase 完成，不只是所有线程都 arrive 了，还要求这个 phase 里登记过的异步事务也都完成了。test_wait 更像是非阻塞查询，通常配合循环轮询；try_wait 则更像是让硬件帮你尝试等待一段时间。

mbarrier.expect_tx 和 mbarrier.complete_tx 用来处理异步事务计数。expect_tx 可以理解成提前告诉 barrier：“这一轮还要等一些异步 transaction。”而 complete_tx 则表示这些 transaction 中有一部分完成了。这样，mbarrier 就不只是线程同步点，也变成了异步操作完成状态的追踪器。

cp.async.mbarrier.arrive 是把 cp.async 和 mbarrier 接起来的指令。它会让 barrier 追踪当前线程之前发出的 cp.async 操作，等这些异步拷贝完成后，再自动对 barrier 做一次 arrive。这样程序就可以用同一个 mbarrier 同时管理线程到达和异步 copy 完成。

最后，mbarrier.pending_count 用来查询某次 arrive 之前还剩多少 pending arrival。它不是最常用的同步指令，更像是用来观察状态，或者构造特殊同步逻辑的辅助工具。

2. SASS 编译结果对比
__global__ void cp_async_mbarrier_kernel(const int* gmem, int* out) {
    __shared__ __align__(8) uint64_t bar;
    __shared__ int smem[256];

    int tid = threadIdx.x;

    if (tid == 0) {
        mbar_init(&bar, blockDim.x);
    }

    __syncthreads();

    uint32_t smem_addr = smem_ptr_u32(&smem[tid]);

    asm volatile(
        「cp.async.ca.shared.global [%0], [%1], 4;\n」
        :: 「r」(smem_addr), 「l」(&gmem[tid]) : 「memory」);

    // Tell mbarrier to track prior cp.async issued by this thread.
    asm volatile(
        「cp.async.mbarrier.arrive.shared::cta.b64 [%0];\n」
        :: 「r」(smem_ptr_u32(&bar)) : 「memory」);

    uint64_t state = mbar_arrive_state(&bar);
    mbar_wait_state(&bar, state);

    out[tid] = smem[tid] + 1;
}

cp.async
cp.async.mbarrier.arrive

mbarrier.arrive
mbarrier.test_wait

read shared memory


上面程序使用 SM80 进行编译（完整程序在附录中），到了 SASS 层，并不是每一条 PTX 都保留成同名指令，而是会被展开成更底层的一组指令。大致对应关系如下：

CUDA / PTX 层操作	SASS 中看到的指令	含义
mbarrier.init.shared::cta.b64 [bar], count	@!P0 STS.64 [RZ], R4	由 tid == 0 的线程初始化 shared memory 里的 mbarrier 对象。
__syncthreads()	BAR.SYNC 0x0	普通 CTA 级同步，保证所有线程都看到 mbarrier 初始化完成。
cp.async.ca.shared.global	LDGSTS.E [R7+0x8], [R2.64]	发起 global memory 到 shared memory 的异步拷贝。
cp.async.mbarrier.arrive.shared::cta.b64 [bar]	ATOMS.ADD.S32 [0x4], R0 + ARRIVES.LDGSTSBAR.64 [URZ]	把前面的 LDGSTS / cp.async 绑定到 mbarrier 上。
mbarrier.arrive.shared::cta.b64 state, [bar]	MEMBAR.ALL.CTA + ATOMS.ARRIVE.64 R2, [URZ]	当前线程自己到达 barrier，并返回 opaque state。
mbarrier.test_wait.shared::cta.b64 p, [bar], state	LDS R5, [0x4] + LOP3.LUT ... + @!P0 BRA .L_x_1	轮询 mbarrier 状态，判断当前 phase 是否完成。

上面程序使用 SM90（100 基本也是一样）进行编译，大致对应关系如下：




CUDA / PTX 层操作	SASS 中看到的指令	含义
mbarrier.init.shared::cta.b64 [bar], count	SYNCS.EXCH.64 URZ, [UR8], UR4	初始化 shared memory 中的 mbarrier 对象。这里 [UR8] 是 barrier 在 shared memory 中的地址，UR4 里是初始化后的 barrier 状态编码。
__syncthreads()	BAR.SYNC.DEFER_BLOCKING 0x0	普通 CTA 级同步，保证所有线程都看到 mbarrier 初始化完成。
cp.async.ca.shared.global	LDGSTS.E [R5], desc[UR6][R2.64]	发起 global memory 到 shared memory 的异步拷贝。LDGSTS 仍然可以理解成 LDG -> STS，也就是 global load 到 shared store。
cp.async.mbarrier.arrive.shared::cta.b64 [bar]	SYNCS.ARRIVE.TRANS64.RED.A0T1 RZ, [UR8], RZ + ARRIVES.LDGSTSBAR.64.TRANSCNT [UR8]	把前面的 LDGSTS / cp.async 绑定到 mbarrier 上。前一条 SYNCS.ARRIVE...RED... 更像是 transaction barrier 的计数/bookkeeping，后一条 ARRIVES.LDGSTSBAR.64.TRANSCNT 表示 LDGSTS 的完成事件会作用到这个 barrier 的 transaction count。
mbarrier.arrive.shared::cta.b64 state, [bar]	SYNCS.ARRIVE.TRANS64.A1T0 R2, [UR8], RZ	当前线程自己对 mbarrier 执行 arrive，并返回后续 wait/check 需要用到的 opaque state。这里 R2/R3 可以看成保存了 arrive 返回的 state。
mbarrier.test_wait.shared::cta.b64 p, [bar], state	SYNCS.PHASECHK.TRANS64 P0, [UR8], R3 + @!P0 BRA .L_x_0	检查当前 barrier phase 是否完成。如果 P0 为 false，就继续回到 .L_x_0 轮询。

通过这两代的 SASS 对比可以很清晰的看到在 SM90 和 SM100 上面 mbarrier 的指令并不是简单的，直接对于 shmem LOAD 和 STORE（在 SM80 的架构下还依然使用对于 shmem 的直接访存），而是使用了新的指令 SYNCS 这种新的 SASS 指令。

3. 微架构分析

通过上面的指令分析可以了解到，原始 SM80 应该没有做特殊的架构优化，而 SM90 和 SM100 应该做了一些相关的优化，最直接的应该是加了一层 cache 用于防止每次都从 sharemem 中 load 数据。知乎上 Nvidia Mbarrier 专利分析段博分析这个架构就可以很清晰的看到一个 cache 设计，更加详细的内容大家可以看上面的文章

mbarrier cache

由于 SM90 以后使用了新的指令，硬件可以很容易的分辨出哪些是需要进行 cache 的数据哪些是不需要的，而且这里的 cache 也不需要做的太难直接做成全相联的 cache 就可以了

// cp_async_mbarrier.cu
#include <cuda_runtime.h>
#include <stdint.h>

__device__ __forceinline__ uint32_t smem_ptr_u32(void* p) {
    return static_cast<uint32_t>(__cvta_generic_to_shared(p));
}

__device__ __forceinline__ void mbar_init(uint64_t* bar, int count) {
    uint32_t a = smem_ptr_u32(bar);
    asm volatile(
        「mbarrier.init.shared::cta.b64 [%0], %1;\n」
        :: 「r」(a), 「r」(count) : 「memory」);
}

__device__ __forceinline__ uint64_t mbar_arrive_state(uint64_t* bar) {
    uint32_t a = smem_ptr_u32(bar);
    uint64_t state;
    asm volatile(
        「mbarrier.arrive.shared::cta.b64 %0, [%1];\n」
        : 「=l」(state) : 「r」(a) : 「memory」);
    return state;
}

__device__ __forceinline__ void mbar_wait_state(uint64_t* bar, uint64_t state) {
    uint32_t a = smem_ptr_u32(bar);
    asm volatile(
        「{\n」
        「  .reg .pred p;\n」
        「L_wait_%=:\n」
        「  mbarrier.test_wait.shared::cta.b64 p, [%0], %1;\n」
        「  @!p bra.uni L_wait_%=;\n」
        「}\n」
        :: 「r」(a), 「l」(state) : 「memory」);
}

__global__ void cp_async_mbarrier_kernel(const int* gmem, int* out) {
    __shared__ __align__(8) uint64_t bar;
    __shared__ int smem[256];

    int tid = threadIdx.x;

    if (tid == 0) {
        mbar_init(&bar, blockDim.x);
    }

    __syncthreads();

    uint32_t smem_addr = smem_ptr_u32(&smem[tid]);

    asm volatile(
        「cp.async.ca.shared.global [%0], [%1], 4;\n」
        :: 「r」(smem_addr), 「l」(&gmem[tid]) : 「memory」);

    // Tell mbarrier to track prior cp.async issued by this thread.
    asm volatile(
        「cp.async.mbarrier.arrive.shared::cta.b64 [%0];\n」
        :: 「r」(smem_ptr_u32(&bar)) : 「memory」);

    uint64_t state = mbar_arrive_state(&bar);
    mbar_wait_state(&bar, state);

    out[tid] = smem[tid] + 1;
}
