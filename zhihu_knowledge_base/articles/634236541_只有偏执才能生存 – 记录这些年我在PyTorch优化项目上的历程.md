# 只有偏执才能生存 – 记录这些年我在PyTorch优化项目上的历程

**作者**: MingfeiPyTorch CPU Perf Maintainer

**原文链接**: https://zhuanlan.zhihu.com/p/634236541

---

年初的时候公司给了个奖 EoY (Employee Of Year)，得这个奖要写获奖感言的，而且要联系上Intel Values。我心里一点也不想写，回复说没时间写，无奈HR那边的小女生三番五次发邮件，不写不行。我看着Intel Values，翻来覆去好几遍，感觉我一条也不符合：

Customer First – 对客户说“soon”一般就是至少两周的意思，但“ASAP”是真的尽快的意思。
Fearless Innovation – 我的工作谈不上创新吧。
Results Driven – 目前的 Results 很不好。
One Intel – 我不是一个很好的团队合作者，而且一身反骨。

还有几条，不过我觉得我也都不符合。我不是严格意义上的“好员工”，我长期“不听管教”，“一意孤行”。有点像以前尼克斯队的斯普雷维尔，打裁判骂教练，各种坏事做尽，但斯普雷维尔是个好球员。

而且，就在这个当下，我怎么能说自己做得很好呢？

随着OpenAI 发布 ChatGPT，大语言模型这个概念似乎一夜之间传遍大江南北。各种同质化的模型于雨后春笋般地冒出来，Google 的 PaLM，Meta的 LLaMA，baidu的文心一言，等等等等，数不胜数。很多模型用南美的各种哺乳动物命名，像 Alpaca (羊驼)，Vicuna(骆马)等，一时间还有人担心神兽的名字不够用。不夸张地说，ChatGPT引爆了AI 的二次革命，不仅是因为人们第一次感觉到 AI 真的像“人”了，更是因为其显而易见的变现手段。

在这一切的背后，PyTorch 加 GPU 的软硬件组合几乎统治了所有大语言模型的开发工作。各大互联网巨头紧急下单，以至于一些硬件 up 主列出博人眼球的标题“英伟达A100又双叒叕买不到了！”上周五的一则新闻，“英伟达一夜之间长出两个英特尔，一个AMD”，让我简直内心无比纠结，那一天我都没怎么干活，对着手机屏幕发了很久的呆。

这则新闻几乎是英特尔在这波大模型风潮中的唯一存在感。

思绪逐渐运转起来，为什么会这样呢？我在问自己“为什么没准备好”？就像大话西游结尾紫霞仙子对至尊宝最后的那句话，“我猜对了故事的开局，但我没有猜对故事的结尾”。故事的结尾是那么让人痛心，更令人痛心的是当你设身处地在这个故事的中心，目睹了这一切的一切。

8年前，整个手机芯片行业颓势尽显，我随波逐流，决定跳槽。那时候我面了两家公司，一家是英特尔，另一家是英伟达。彼时英伟达还只是一家靠游戏显卡出彩的面向娱乐市场的专业公司，两家的 package 差不多，因为离家近一点，我选了英特尔。入职之后，做过短暂的 Competitive Analysis，之后团队决定转型，我开始负责 Torch 上的 CPU 性能优化工作，正式加入了AI行业（这里说的 Torch 现在一般称为 lua-torch，是PyTorch的前身）。Lua-torch 本身也挺好用的，那时候我们也有很多零零散散的客户，我们有一些 design win，也在超算竞标中 PK掉过 GPU的方案，我也第一次收获 OTB (one-time bonus)，当时都不知道英特尔还有这种奖金。当然它远远没有达到其继任者 PyTorch 所达到的高度。

2017年随着 PyTorch 的诞生，整个小组工作中心自然而然转移到了 PyTorch上面。接下来的两年是我最惨的日子，因为 PyTorch起初被当做是做 Research的框架，所以被认为不会被用在 Production 环境上，也就是没有“价值”。因为毕竟英特尔是个硬件公司，能大规模部署在 cloud 或者 data center 上才能产生利润。但直到 PyTorch 1.0 才提出“Research + Production”的概念，不过事实是 v0.4 版本之后 PyTorch 就越来越受欢迎，客户需求直线上升。那段时间我经常出去宣传，我陪着 sales 的人去忽悠客户，我给很多公司做过定制化的优化方案，我给美国的阿贡实验室免费打工（在 Xeon Phi集群上部署 PyTorch），我给 GCP 部署过 Intel-Optimized-PyTorch。我做这些都是为了说服管理者这是一个值得投资的项目，我还设计了一个口号“All in PyTorch”，但很可惜那时候我的老板选择了 MxNet。我进入了三不管境地，开组会我都是听歌的。更惨的是在 PyTorch 和 caffe2 合并之后，大牛Jia Yangqing 的一句“PyTorch 前端 + caffe2 后端” 几乎将我的职业生涯逼上绝路。因为我在优化 PyTorch的后端，如果这话真的应验了，那我做的东西就没用了。那段时间，我差不多和部门里所有PE都吵过架。其实我这个人脾气很好的，经常讲笑话，实在是被逼急了。但我还是要公正地说一句，就算吵架，大家也是对事不对人，人品都没问题的。那会，我知道了那个邪恶的字母组合 ZBB(Zero-Based Budget)，还有 LHF (Low Hanging Fruit)，LHF 本身是个褒义词，但我非常非常讨厌这个词，凭什么我是LHF，我做的工作一点都不简单。就是这段时间，我身边的战友一个又一个离我而去，因为当时完全看不到希望。我要了一个Back Fill的HC，招来了一个新人，现在大家叫他Bing神。我也在离职的边缘反复游走，那个时候我经常出去面试，拿了好几家互联网公司和硬件公司的offer，不过我始终不想放弃 PyTorch这个项目，我觉得我还有机会。简单的文字是无法表达当时我内心的失望和愤怒的。

面对无休止的争吵，我选择了闭嘴，讷言敏行，低头写代码。

之后我采取了对于个人职业生涯近乎自杀的策略，我把的首要任务定位在优化整个 PyTorch 上 CPU 的生态，而不是公司内部的一个又一个 KPI 。我并不热衷于那种 specialized 的优化方案，不想总去优化几个指标性质的模型，我觉得优化本身应该足够泛化。我不想把时间浪费在一遍又一遍集成 mkldnn，单纯的调包根本无法做到泛化。早期的我天真地认为CPU的优化就应该是我们来负责，我也自信地觉得只要性能够好，upstream就应该是水到渠成的，而现实总是那么残酷。我花了一年的时间认识到一个道理：PyTorch不是英特尔的项目，要想被社区接受就要按照社区的规则来。优化生态不是拆了重写，我说我只是在替Facebook完成他们应该完成的工作。现在我写的kernel长这个样子，换AMD或者MS的人来写，它也是长这个样子。我不再一天到晚地跑 ResNet50，而是对着PyTorch的源文件，一个算子一个算子地看，遇到性能有问题的地方我就做优化。这个工作量是非常巨大的，就像一座山一样，PyTorch本身的用法非常发散，也给优化工作提出了更高的要求，简单的一个BatchNorm要写十几个kernel。那时候我常引用愚公移山的故事，你不去写，那座山就永远在哪里。但这也有一个很显而易见的问题，我完不成KPI(当时并没有针对 Out of Box 优化的 KPI)。独特的个人成长经历让我拥有比常人强大得多的抗压能力，相对于当年高考失利对我造成的心理创伤，这点职场上小小的挫折算得了什么。我最喜欢的一句话就是英特尔创始人Andy Grove的那句“Only the paranoid survive!”现实中我也是这么做的。我相信我在 PyTorch 社区留下的每一个痕迹都会记录我做出的努力。我从社区收获的每一个肯定都是我内心的激励，这可能是个简单的一句“Good Job!” 夜深人静的时候，看着一行行log里时间在逐渐降低，内心会有一种平静的感觉，这是作为一名工程师的inner peace。

19年的时候，我换了一个老板，我本人非常尊敬这个人，很可惜后来他也离职了。这一年部门开始在这个项目上大规模投入，整个PyTorch项目迎来了转机，因为大家最终看到“PyTorch 前端 + caffe2 后端”是个美好的设想。我之前也在强调，PyTorch动态图的逻辑限定它的后端只能按照ATen现在的玩法继续写下去，哪怕是用caffe2的后端，也要服从这个规则，不可能打破这个规则。

几年前，我被promote成PyTorch CPU performance 的Maintainer。以前Facebook不怎么给别的公司 Maintainer，我是这个模块的第一个非FB的 Maintainer。用PyTorch 管理团队的话说，Maintainer 是基于 Individual Contribution，而不是公司行为。我当时这么做，也是有个私心的，就是给自己留条后路，方便被开除以后再就业。按我老婆的话说，也就英特尔这种外企能容忍我这种离经叛道的行为，放在她们公司早就被开除一百次了。

这两年公司开始重视社区贡献，我获得了来自团队的强力支持，我开始觉得不是一个人在战斗了。本质上我们有两种需求，一个是优化CPU的生态，另一个是在英特尔最新的硬件拿到最佳性能，这两方面有的时候是有冲突的，但都很重要。我们开始更多地深入参与 PyTorch 本身一些特征的设计和优化之中，主动关注社区提出的要求和意见。但我觉得目前的工作还有很多做得不到位的地方，还有很多地方需要发力。

目前，整个AI的生态都是被 GPU + CUDA 主导的，对 CPU 做 contribution的力量很小。作为一名底层工程师，我没什么资格对公司的政策评头论足。但这些年在AI领域摸爬滚打，一点一点看着差距被对手逐渐拉大，这是不争的事实。AI现在几乎和CUDA画等号，会有大量的公司和个人为GPU的软件生态助力，以PyTorch为核心，发展出了一个庞大的AI生态圈，包含 HuggingFace，TorchVision，DeepSpeed等等。英伟达十余年来长期对CUDA高强度在教育领域投入，学生都会写 CUDA，网上GPU优化相关资料比比皆是。而反观之，至今没有一个关于CPU性能优化的合格教程，本身CPU的SIMD写法就是反人类的，上手难度大于CUDA，在这种情况下，指望一般的社区contributor去优化CPU简直就是天方夜谭。我自己也会在知乎上写一些关于CPU优化的文章，试图简化一些专业概念，好让更多人容易上手，就像“快乐五子棋”可以先不考虑“五手两打”这种高级玩法。不过很可惜，目前看来收效甚微。

GPU + CUDA在生态圈的领先地位，决定英伟达可以几乎做到全面出击，而且还有左右护法(HuggingFace)，八大精钢(Megatron)，四值功曹与六丁六甲(DeepSpeed)，为其保驾护航，那么取得真经只不过是时间问题。而反观英特尔只能做到在重点领域以点带面。这也就是这波大模型需求爆炸之后，为什么CPU显得毫无还手之力，CPU的方案只有在单节点的GPT2上看起来还有点竞争力，无法拓展到大集群。

客观上英特尔的环境还是非常包容的，不然选择上应该不是给我发奖，而是把我开除。开会听歌也不影响公司给你发数额巨大的 OTB。但这个奖我最多高兴了两天，因为就像之前说的，“我们还有很多工作没有做好”。最后引用生活大爆炸里的一个桥段，Sheldon一行人获得Time Machine之后希望回到 “5 seconds from now on”那个时间，然后大家等了5秒钟，希望遇见来自未来的自己。如果我有这个机会，我也希望回到“5 seconds from now on”那个时间，我希望那个来自未来的自己对我说“Good job!”

补充说明一下：文中的 CPU 指的是服务器，优化中的 CPU 一般都是指服务器，还是很多人做这方面的。intel 有两个产品线，酷睿是消费级，Xeon是服务器。服务器一般的客户是 cloud，data center 还有 超算 等等，这个业务是非常赚钱的，AMD的竞品叫 Epyc。

我们一般不测酷睿的，但要保证正确性；对 Xeon 有很高的优化要求，一般的双路服务器有80-120个 core。




我们日常说的计算都是数据并发，@波尔德 看你回过一个“计算全部N个点的两两距离”的问题，如果用 PyTorch 写大概长下面这个样子，其实很简单了，几行就能处理掉(我写了很多 template utils 简化 kernel 搭建)。

写这个例子，是介绍一下平时我们都在干什么。另外这个逻辑和 transformer block 里面计算 attention 其实是一样的，不同的是 attention 优化是不会真的分配 N * N 的 output buffer，而是选择切分掉，因为这个太大了，非常吃带宽。

/*
 * [NOTES]: calculate the distance from a src tensor to a dst tensor
 *    src, dst: 2-d tensor shape of [N, d]
 *    out: 2-d tensor shape of [N, N]
 *  caller to `distance` should:
 *    a) allocate output buffer;
 *    b) call `contiguous()` on both `src` and `dst`
 *    c) handle data type dispatch
 */
template <typename T>
void distance(Tensor& out, const Tensor& src, const Tensor& dst) {
  auto src_data = src.data_ptr<T>();
  auto dst_data = dst.data_ptr<T>();
  auto out_data = out.data_ptr<T>();

  int64_t N = src.size(0);
  int64_t D = src.size(1);

  // Vec is a wrapper upon avx2 or avx512 vector
  using Vec = vec::Vectorized<T>;

  // parallel on the outer dim of `src` and vectorize the inner dim
  at::parallel_for(0, N, 0, [&](int64_t begin, int64_t end) {
    for (const auto i : c10::irange(begin, end)) {
      T* src_ptr = src_data + i * D;
      T* out_ptr = out_data + i * N;

      for (const auto j : c10::irange(N)) {
        T* dst_ptr = dst_data + j * D;
        T dist = sum = vec::map2_reduce_all<T>(
                [](Vec x, Vec y) { return (x - y) * (x - y); },
                [](Vec x, Vec y) { return x + y; },
                src_ptr,
                dst_ptr,
                D);
        out_ptr[j] = std::sqrt(dist);
      }
    }
  });
}


以上的是“快乐五子棋”的写法，还可以继续改进：

a) 对于 dst 的读取是有重复的，解决方案是对 src 的第一个 dim 做 blocking，就是一次计算kBlockSize 那么多个点的距离；

b) 有可能 D 特别大，然后导致a) 做完blocking 之后 dst 的一行 D 数据被踢出 L1 cache，这是需要对 D 做 blocking，当然这种情况应该不常见；

c) 如何处理 half 和 bfloat16，这两种 data type 都需要用 float32 做 accumulation，为了减少 data type conversion 的开销，还是需要 a)；

d) 原题主写的是 double 而且 D = 3，这就有个问题，D < VectorLength (4 for avx2 and 8 for avx512)，需要做 padding，map2_reduce_all 会做 padding，但逻辑不是最优的，需要 hack 掉 PyTorch 现有的逻辑，使用 masked load替换 (在 avx512 上只有一条指令)，现在是多条指令 movss + blend；

e) 另外这个问题有个隐藏 bug，就是 SIMD 是没有一条指令能完成的 horizontal reduce 的 （即把一个 vector reduce 成一个 scalar）,需要通过 permute 来做，不过好在这个问题我已经在 PyTorch 里面 fix 掉了。[这是个 size dependant 的问题，当 D 远大于 VectorLength 的时候不是主要矛盾，反之 D 越小这个地方开销越大；追求极限的情况应该把特定的 D = 1/2/3 等，用 template argument 在 compile time 处理掉]

f) 还有一些 PyTorch specific 的问题：比如 .data_ptr() 和 .size() 这种函数不能出现在 most inner loop。[这些都是 dispatched 函数，call stack 都很长，不能被 inline 掉，这个是动态图带来的问题]
