# NPU编译存储优化思路

---

最近在搭NPU编译链，总结归纳一下NPU编译存储优化思路。

NPU 的硬件结构对内存访问模式的要求比 CPU / GPU 严格得多。计算单元是固定形状的 MAC 阵列（典型如 16×16、128×128），片上 SRAM 是 scratchpad 而非 cache，因此没有硬件自动预取，没有自动替换，每一次数据搬运都需要编译器显式发出 DMA 指令，在 NPU 上必须由编译器在编译期一次性解决。

存储搬运的代价在两个维度上同时体现。

时间维度上，现代加速硬件的算力增长速度持续超过带宽增长速度。以 NVIDIA H100 为例，FP16 算力约 1000 TFLOPS，HBM3 带宽约 3 TB/s。换算下来，每从内存搬一个字节进芯片，硬件要在它身上做约 300 次浮点运算，计算单元才不会空转。这个比值（arithmetic intensity，AI）是判断算子瓶颈的基本指标。大多数深度学习算子的 AI 远低于 300，时间消耗主要花在等数据上。

能量维度上，数据移动的能耗远高于计算本身。在主流工艺下，从 DRAM 读一个 64-bit word 约 640 pJ，从片上 cache 读约 10 pJ，做一次 FP32 加法约 0.4 pJ。DRAM 访问的能耗是计算的 1000 倍以上。一个算子大部分能量花在哪里，由它的 AI 决定。

时间与能量共享同一个支配因素：每比特数据从距离计算单元越远的地方搬来，时间越长、能耗越高。NPU 编译器的全部存储优化工作，可以归结为在硬件给定的 memory hierarchy 上最小化数据移动的时间与能量乘积。

减少数据移动有四类基本手段：

Layout（排）：物理排布对齐硬件访存接口与 DMA burst 形状，让每次访问搬运的字节都是有用的。
Tiling（切）：把张量切到当前存储层级能装下，让数据靠近计算单元处理。
Movement（搬）：通过双缓冲、软件流水、多 DMA 引擎并发让搬运与计算并行。
Reuse / Fusion（用）：通过算子融合、循环重排、重计算让每个搬上来的字节做尽量多的事。

一、Layout

Layout 指张量在物理内存中的字节排列顺序。同一个逻辑张量可以有多种 layout，它们在数学上等价，在硬件访存效率上差距可达几十倍。

NPU 上 layout 选择的约束来自两个层面。第一是计算单元接口：固定形状的 MAC 阵列（典型如 16×16、128×128）每周期需要从 scratchpad 的特定偏移读取固定数量的值，layout 决定这些值在物理上是否连续。第二是 DMA burst 形状：从 DDR 搬入 scratchpad 的最高带宽形态是大块连续传输，layout 决定一次 DMA 能搬多大、需要发起多少次描述符。layout 不匹配，要么 MAC 阵列每周期只能拿到部分有效数据并空转，要么 DMA 退化成多次小传输并打不满带宽。

下面各举一例。

GEMM / Linear

形状 (M, K) × (K, N) = (M, N)，覆盖 Transformer 的 QKV 投影、FFN、Embedding、MoE 的专家层。

最朴素的 layout 选择是行主序还是列主序，对应 GEMM 的 NN / NT / TN / TT 四种组合。NPU 上常见的做法是把矩阵进一步预打包成块状格式。以 Ascend 的 FractalNZ 为例，权重矩阵被切成 16×16 的小块，块内按行存储，块间按预定义的二维顺序（zN 或 Nz）排列。这种排布让 Cube 单元（16×16 MAC 阵列）每周期从 L0B 读 16×16 个值时，刚好命中一个连续的物理块，地址生成器无需跨 stride 寻址。

权重打包在编译期完成，运行时模型文件里存的就是 FractalNZ 格式，DMA 直接灌进 scratchpad。激活值因为运行时才知道大小，需要在每次进 scratchpad 前由编译器插入的 reformat 算子做实时打包。

Conv

形状 (N, C, H, W)，覆盖视觉模型、Diffusion 的视觉前端、多模态模型的图像编码器。

考虑一个 16 路并行的 MAC 阵列，每周期需要消耗一个 (h, w) 位置上的 16 个通道值。数据是 NHWC 时，pixel(h, w) 的 16 个通道在物理上是 16 个连续元素，MAC 从一个基地址读连续段，一周期取齐。数据是 NCHW 时，这 16 个通道分布在 16 个相隔 H×W 个元素的地址上，要么发起 16 次串行读，吞吐降到 1/16；要么打散到 SRAM 的 16 个 bank，但规则的大 stride 通常会全部撞同一 bank，序列化无法避免。

Attention / KV Cache

形状 (B, H, S, D)，覆盖 Transformer 的注意力计算。

训练阶段 D 通常放在最内层，对 Q×K^T、softmax、A×V 这一连串 GEMM 与向量操作都友好。D 是 head dimension，head 内部计算沿 D 做 reduce，连续 layout 让向量化效率最高。

decode 时每步生成一个新 token，需要读取累积的全部 K、V 与新 token 的 Q 做 attention。S 维度随时间增长，预分配大块连续空间会浪费显存，按需扩展又会引发频繁 reallocation。vLLM 的 PagedAttention 把 (B, H, S, D) 沿 S 切成固定大小的页（典型 16 或 32 个 token 一页），每页内部仍是连续 layout，页之间通过 block table 索引。这是把 layout 设计与 tiling、显存分配耦合在一起的方案，本质仍是用 layout 去匹配硬件访存与显存分配的双重约束。

二、Tiling

Tiling 指把大张量沿一个或多个维度切成小块，每次只把一块加载到当前一级存储中处理。它的根本约束来自一个事实：完整张量装不下。

NPU 的存储层级通常包括 片外存储（GB 量级）、L1（MB 量级）、L0A / L0B / L0C（几十到几百 KB）、寄存器（KB 量级）。每往下一级，容量小一到三个数量级，带宽和访问粒度对应提升。一个 batch 16 的 ResNet-50 中间激活有几十 MB，一个 LLaMA-7B 单层 FFN 的中间结果在几百 MB 量级，没有任何一个能整块装进 L1，更不用说 L0。Tiling 不是优化项，是数据进入计算单元的必要条件。

Tile 尺寸由四个约束共同决定：

容量约束：tile 占用的字节数必须小于目标层级容量。GEMM 中三个 tile（A、B、C）同时存在于 scratchpad，约束是 Bm·Bk + Bk·Bn + Bm·Bn ≤ L1 容量。
硬件粒度约束：tile 在 MAC 阵列对应维度上的尺寸必须是阵列边长的整数倍。16×16 Cube 上，tile 的 M、N、K 三个维度都应是 16 的倍数。
DMA 效率约束：tile 太小会让 DMA 开销超过传输本身。每条 DMA 描述符有几十到上百 cycle 的发起延迟，tile 至少应让一次 DMA 搬运 KB 级数据才能摊平开销。
流水重叠约束：tile 还要与 double buffering 配合，L1 容量被切两半时实际可用容量减半，tile 尺寸进一步受限。

四个约束相互冲突：硬件粒度要求 tile 大、容量要求 tile 小、DMA 效率要求 tile 大、双缓冲要求 tile 小。NPU 编译器在这些约束下做整数搜索，找出一组合法且性能最优的 tile 尺寸。

同样的，下面分三类算子各举一例。

GEMM / Linear

(M, K) × (K, N) → (M, N)，取 M = 4096，K = 4096，N = 4096，FP16，总数据量 96 MB。L1 容量 1 MB，L0 容量 64 KB。

L1 层 tile 取 (Bm, Bn, Bk) = (256, 256, 256)，三个 tile 占用约 384 KB，留出 double buffer 空间。L0 层进一步切到 (16, 16, 16) 喂给 Cube。整个 GEMM 被分解为外层 (M/Bm) × (N/Bn) × (K/Bk) 次 L1 tile 循环，每次 L1 tile 内再做 (Bm/16) × (Bn/16) × (Bk/16) 次 Cube 调用。

K 是 reduction 维度，处理逻辑与并行维度不同。每个 K-tile 计算的 partial sum 必须累加。NPU 通常把累加器留在 L0C，K-tile 在 L0C 上原地累加，全部累加完成后才把结果写出 L1。这就是上一节提到的 Output Stationary 在 tiling 上的体现：tile 的最内层循环固定输出位置，沿 K 累加。

Conv

形状 (N, C, H, W)，3×3 卷积，输入 (1, 256, 56, 56)，权重 (256, 256, 3, 3)。

Conv 的 tile 维度比 GEMM 多。C 是 reduction 方向，H、W 是空间方向，K（输出通道）是另一个并行方向。常见的切分顺序是先在 H 方向切（保留 W 完整以利用 im2col 的连续访问），再在 K 方向切（输出通道间相互独立，切分不引入 reduction），C 方向尽量整段进 L1（reduction 沿 C 做，切 C 会增加 partial sum 的搬运）。

Conv tile 还涉及边界处理。3×3 卷积的输出 tile 需要输入 tile 在 H、W 各扩展一圈 padding，称为 halo region。tile 越小，halo 占整个 tile 的比例越大，重复加载的数据越多。一个 56×56 的输出在 H 方向切成 7 段，每段 8 行，halo 各 1 行，重复加载比例约 25%。这是 tile 尺寸下界的来源之一。

Attention

以 FlashAttention 为例。完整 attention 矩阵 S = Q×K^T 形状 (B, H, Sq, Sk)，Sq = Sk = 8192 时单 head 的 S 矩阵就是 128 MB，无法放入任何片上存储。

FlashAttention 沿 Sq 切成块 Bq（典型 64 或 128），沿 Sk 切成块 Bk（典型同等大小）。每个 (Bq, Bk) 子块独立计算 partial attention：先算 Q_block × K_block^T 得到一小块 score，做局部 softmax，乘以 V_block 得到 partial output。softmax 跨块需要重新 scale，FlashAttention 引入 online softmax 算法让多个 K-tile 的结果可以增量合并，无需先算完整的 S 再做 softmax。

这种 tiling 在 NPU 上落地的额外要求：Bq × D 与 Bk × D 必须同时放进 L1 并留出 double buffer 空间，softmax 的中间统计量（每行的 max 和 sum）需要单独的累加器空间。Ascend 等芯片为此提供了专门的 vector 单元路径，与 Cube 并行执行 softmax 与累加。

编译器视角

Tile 尺寸搜索在 NPU 编译器中通常作为独立的 pass，输入是已完成 layout 分配的计算图加硬件描述（各级容量、MAC 尺寸、DMA 参数），输出是每个算子在每个层级的具体 tile 形状。

实现方式上有两类。一类是基于 cost model 的解析求解：列出所有约束方程，目标函数取计算与带宽利用率的乘积，用整数规划或穷举搜索给出最优解。这类方法在规则算子（GEMM、Conv）上可行，因为约束可以闭式表达。另一类是基于 profile 的自动调优：候选 tile 尺寸通过实际硬件运行测量延迟，迭代逼近最优。TVM 的 auto-scheduler、Triton 的 autotune、华为 CANN 的 tiling search 都属于这一类。

GPU 上 tile 选错通常表现为性能下降，运行时仍可正常完成；NPU 上 tile 超出 L1 容量会直接编译失败，tile 不满足 MAC 整除约束会让 Cube 无法启动。tiling 与 layout 一样，是从优化项变为先决条件的典型例子。

三、Movement

NPU 上数据搬运是显式的：每条 DMA 指令由编译器发出，指明源、目的、形状、stride。计算单元（Cube、Vector）和 DMA 单元是物理上独立的硬件，原则上可以并行运行。但默认的执行顺序是串行的：DMA 搬入、计算、DMA 搬出、下一轮。如果不做显式重叠，计算单元在等数据时空转，DMA 在等计算结果时空转。

Movement 优化的目标是让这两者并发执行，把搬数据的延迟藏在计算后面。

双缓冲

最基础的技术是 double buffering，又称 ping-pong。把 L1 中放某个张量的区域切成两半（buffer A 和 buffer B），搬运和计算交替使用。

稳态运行时三个时刻的状态：

T0：DMA 把 tile 0 搬进 buffer A，计算单元空闲。 T1：DMA 把 tile 1 搬进 buffer B，计算单元处理 buffer A 中的 tile 0。 T2：DMA 把 tile 2 搬进 buffer A，计算单元处理 buffer B 中的 tile 1。

稳态时计算与 DMA 完全并行，总耗时为 max(计算时间, 搬运时间)，而非两者之和。如果计算时间不小于 DMA 时间，DMA 被完全掩盖；反之亦然。

代价是 L1 容量减半。这就是上一节 tile 尺寸"流水重叠约束"的来源：一个 tile 不能占满 L1，必须留出对偶副本的空间。

软件流水

一次计算往往涉及多步搬运。以 GEMM 为例，每轮外层 tile 循环包括四步：搬 A tile 入 L1、搬 B tile 入 L1、Cube 计算 C tile（在 L0C 累加）、累加完成后搬 C 出 L1。四步可以独立调度。

软件流水把多个 tile 循环的不同阶段交织在一起。最长的那个动作决定总吞吐。pipeline 深度从 2 级双缓冲扩展为 4 级以上，需要的 buffer 副本数量也相应增加。L1 被切成 4 份甚至 8 份，每份装一个 tile。

软件流水在 NPU 上不是隐式的。CPU 与 GPU 上编译器靠乱序执行单元和 score-board 自动重排指令，NPU 编译器必须显式生成搬运与计算的交错指令序列，并在每个依赖边界插入同步指令（semaphore、event）。错一个同步点就会数据竞争或死锁。

多 DMA 引擎并发

NPU 通常配备多条独立的 DMA 通路，每条通路对应不同的搬运方向或类型。Ascend 的 MTE（Memory Transfer Engine）按方向划分：MTE1（DDR ↔ L1）、MTE2（L1 ↔ L0）、MTE3（L0C ↔ L1）。三个引擎物理上独立，可以同时工作。

这一硬件结构让多级流水成为可能。一个 tile 在 L0 上参与 Cube 计算的同时，下一个 tile 从 L1 进入 L0（MTE2），再下一个 tile 从 DDR 进入 L1（MTE1），上一轮的输出从 L0C 写回 L1（MTE3）。整条流水线上每一段都有独立的搬运硬件，互不阻塞。

代价是编译器必须为每个引擎单独生成指令流，并保证跨引擎的依赖通过同步原语正确表达。Ascend 的 ISA 里 MTE 指令与 Cube 指令属于不同的 pipe，pipe 间通过 barrier 同步。

异步 DMA 与 prefetch

部分 NPU 提供 fire-and-forget 的异步 DMA 接口：发起搬运后立即返回，编译器可以把搬运指令尽量前置，与后续计算重叠。GPU 上的 cp.async（Ampere 起）和 TMA（Hopper 起）是同类机制。

Prefetch 是一种特殊的异步 DMA：预测下一轮需要的数据，提前发起搬运。在 GEMM 这类规则算子上预测准确；在动态形状或控制流复杂的算子上预测可能失败，搬来的数据用不上反而浪费带宽。

与 tiling 的耦合

Movement 不是独立的优化层，它的可行性完全由 tiling 决定。

双缓冲要求 L1 中每个张量留两份副本，tile 容量预算减半。
4 级软件流水要求四份副本，预算降到 1/4。
流水深度越深，硬件并发度越高，但 tile 尺寸越受限。tile 太小，DMA 启动开销超过传输本身，流水也无法掩盖。

存在一个 tile 尺寸的最优区间：太小则 DMA 启动开销主导，太大则流水深度不够、硬件并发用不满。这个区间通常在 KB 到几十 KB 之间，具体由 DMA 启动延迟、带宽、计算密度联立决定。

编译器视角

Movement 优化在 NPU 编译器中表现为指令调度（instruction scheduling）。输入是已完成 layout 与 tiling 的算子计算图，输出是带 pipe 分配和同步指令的指令序列。

调度算法通常基于 list scheduling 或 modulo scheduling。前者按依赖顺序逐条贪心调度，实现简单但不一定最优。后者面向循环结构，把循环展开成软件流水形式，求解最小启动间隔（initiation interval, II），在规则算子上能逼近硬件并发上限。

GPU 编译器靠硬件乱序执行掩盖大部分调度问题，软件流水在 GPU 上属于性能调优。NPU 的 pipe 之间没有硬件仲裁，软件流水是吞吐量的主要决定因素，编译器调度质量直接决定算子能跑出标称算力的 30% 还是 90%。

四、Reuse / Fusion

前几节讨论的都是怎么把数据搬得更快。Reuse 解决的是另一个方向：搬上来一次的数据能干多少事。

衡量这件事的指标是 arithmetic intensity（AI），定义为算子的总浮点运算次数除以从上一级存储读入的字节数。开篇提过 H100 的 roofline 拐点在 300 左右，AI 低于这个值的算子是带宽受限，AI 高于这个值才能让计算单元打满。NPU 上数字不同但结构相同。

提高 AI 有三类基本方法：算子融合让中间结果不落 DDR、循环重排让数据在最内层循环上反复使用、重计算用算力换内存。

算子融合

深度学习模型由许多小算子串联组成。一个典型的 LayerNorm 按数学定义展开包含七八个 element-wise 与 reduce 算子的串联（减均值、平方、求方差、加 epsilon、sqrt、除、乘 gamma、加 beta）。如果每个算子单独执行，每一步都要从 DDR 读入张量、计算、写回 DDR，张量被来回搬十次以上。

融合后这一串合并成一个 kernel：从 DDR 读一次张量，所有中间结果留在寄存器或 L1，最后写一次结果回 DDR。AI 提高一个数量级，DDR 流量降到原来的 1/N。

融合的边界由计算图分析决定。能融合的典型模式：

Element-wise 链：连续若干个 element-wise 算子（add、mul、relu、gelu）合并成一个 kernel，是融合中最简单的一类。
GEMM + epilogue：GEMM 输出后立即接 bias、activation、residual add，这些后置 element-wise 操作在 GEMM 的 L0C 累加器内就地完成，不需要先写出 L1 再读回来。
Reduction + element-wise：LayerNorm、Softmax 这类先 reduce 再 element-wise scale 的模式。
Producer-consumer：前一个算子的输出立刻被下一个算子消费，且形状一致。

不能融合的典型情况：算子之间有形状变化（reshape 跨越 reduction 维度）、有控制流分叉（输出被多个下游消费且时机不同）、跨设备边界（张量在多个 NPU 上切分）。

FlashAttention 是融合的标志性例子。原始 attention 由 Q×K^T、scale、softmax、× V 等步骤构成，朴素实现要先把 Q×K^T 的完整中间矩阵 S（前一节算过 8K×8K 时是 128 MB）写出再读回。FlashAttention 把整个序列融合在 tile 循环内：每个 (Bq, Bk) 子块算完 partial score 立刻做 softmax 局部化，乘 V 后累加进输出，全程不落 S 矩阵。AI 提升一个数量级以上，DDR 流量按 N/Bk 倍数下降。

循环重排

同一个 GEMM C[i,j] += A[i,k] * B[k,j]，三个循环 (i, j, k) 的不同顺序对应不同的复用模式。

最内层是 k：固定 (i, j) 沿 k 累加。C[i, j] 留在寄存器持续累加，A[i, k] 和 B[k, j] 流过。这是 Output Stationary，每个输出值复用次数为 K。
最内层是 j：固定 (i, k) 沿 j 流动。A[i, k] 留在寄存器，B[k, j] 和 C[i, j] 流过。每个 A 值复用次数为 N。
最内层是 i：固定 (k, j) 沿 i 流动。B[k, j] 留在寄存器，A[i, k] 和 C[i, j] 流过。每个 B 值复用次数为 M。

哪个值留在寄存器不动，决定哪一个操作数获得最高复用率。CPU 与 GPU 上这是软件循环顺序的选择；脉动阵列上同样的决策被焊进硬件，对应 Output Stationary、Weight Stationary、Input Stationary 三种 dataflow。同一道复用问题，软件解法是循环重排，硬件解法是 PE 互连结构。

实际系统在每一级存储层级上独立选择 stationarity。GPU GEMM 的典型布局：寄存器层 OS（输出 tile 在寄存器累加），shared memory 层 WS+IS（权重和输入都缓存），HBM 层无 stationary（流式访问）。NPU 类似：systolic array 内部 WS（权重灌进 PE 不动），L0C 层 OS（partial sum 累加），L1 与 DDR 流式喂数据。

重计算

重计算（recomputation，又称 gradient checkpointing）是反向操作：不保存中间结果，需要时重新算。

训练阶段反向传播需要前向的中间激活值计算梯度。一个标准 7B Transformer 在 seq 4096、batch 1 下全部激活合计在十 GB 量级，长序列或大 batch 场景下迅速突破单卡显存。完整保存意味着大量数据在 HBM 上反复读写。

重计算的做法是只保存每层入口处的激活，反向时从入口重新跑一遍前向得到中间值。代价是前向计算被做两次（一次正向、一次反向时重做），收益是激活内存降一到两个数量级。

重计算与提高 reuse 表面方向相反，实际是同一个权衡的另一极：当存储成本（搬数据 + 占容量）高于重新计算的算力成本时，丢掉再算的总成本更低。在 LLM 训练这种激活极大的场景，几乎所有现代框架（PyTorch 的 checkpoint、Megatron 的 selective recomputation）都默认开启重计算。

与 layout / tiling 的耦合

Reuse 优化与前两节的优化深度耦合。

融合要求被融合的算子共享 tile 划分。两个算子如果 tile 形状不一致，融合后需要在 kernel 内部插入额外的搬运或重组，可能反而降低 AI。编译器在做融合决策前必须先做 tile 一致性分析。

循环重排的最优顺序与 layout 直接相关。最内层循环遍历的维度应该是 layout 中最内层的维度，否则会产生 stride 访问。NCHW layout 上做 GEMM 时，K 最内的循环顺序产生跨通道连续访问；NHWC layout 上同样的循环顺序则产生跨像素 stride 访问，效率下降。layout 选定后，循环顺序的可行解集就被收窄了。

重计算与 tiling 的关系体现在反向时的 tile 重做：如果前向 tile 切成 (Bm, Bn)，反向重做时按相同 tile 划分，每个 tile 的中间激活独立重新生成、独立消费、独立丢弃，不会扩大显存峰值。

编译器视角

Reuse 优化在 NPU 编译器中由两类 pass 完成。

第一类是计算图层面的算子融合（graph fusion）。编译器在算子图上识别可融合模式（element-wise 链、GEMM + epilogue、reduction + scale），把多个算子合并成一个融合算子（fused op），由后端生成单一 kernel。TVM 的 FuseOps、MLIR 的 linalg fusion、华为 CANN 的 fusion engine 属于这一类。

第二类是循环层面的调度变换（loop scheduling）。在生成具体 kernel 时，编译器对算子的循环嵌套做重排（interchange）、合并（fuse）、分块（tile，与第二节的 tile 共享底层机制），生成 stationarity 最优的指令序列。Polyhedral compilation（多面体编译）是这一类的理论基础，TVM 的 schedule、Halide 的 schedule、MLIR 的 affine dialect 都建立在多面体模型上。

GPU 编译器把这两类 pass 都做，但在 GPU 上 reuse 优化更多是性能放大器，不做 fusion 跑得慢但跑得动。NPU 上 fusion 经常是跑通的前提，不做 fusion 的 kernel 可能因为中间张量超出 L1 容量直接编译失败，与上一节看到的 tiling、movement 是同样的从优化项变先决条件的现象。
