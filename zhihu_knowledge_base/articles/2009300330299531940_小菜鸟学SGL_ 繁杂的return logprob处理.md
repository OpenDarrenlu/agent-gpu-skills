Q1: chunked prefill生成logits逻辑有点怪啊

从event_loop的run_batch开始Scheduler从waiting_queue选择req来组装ScheduleBatch，batch会经过ModelWorkerBatch转变成最终执行的ForwardBatch。遗憾的是，ForwardBatch没有req信息，在模型执行过程中无法判断是当前chunk执行结束还是全部prompt执行结束。

以下从Scheduler选择好batch以后执行过程的主要代码逻辑。可以看出每次model_runner.forward执行结束都会生成logits，在TpModelWorker.forward_batch_generation对每次model_runner.forward结果都会进行sampling处理生成next token。

Q2: return logprob是啥？

为啥纠结这个？因为在看一个老的commit的时候(#5608)，突发奇想看了logits processor代码。发现下面的索引处理跟其他的list不一致。代码又看不懂，也没人可问。只能更新代码，看有木有人觉得"这是个bug[偷笑]"；结果，最新的commit逻辑跟这个是一样的。

（pr #5477)

那结论就很清楚了：是我理解的问题啦。切到最新代码看 (#19004), 发现有详细注释。

pruned_states, sample_indices: 生成logits和sampling
input_logprob_indices: 返回logprob
token_to_seq_idx: 返回logprob，如果设置了logprob元数据的chunk size
(pr #19004)

菜鸟的问题在于对"extend_len == extend_logprob_start_len"条件的处理。分析过程很繁琐，这里只写结论。

chunked prefill场景，extend_len == extend_logprob_start_len表示当前chunk不处理return logprob，下一个chunk处理时req.extend_input_lens（图中extend_len）变大自然可以处理return logprob逻辑。
非chunked prefill场景，从第1个生成token位置return logprob。有点怪吧
模型执行过程中，无法判断chunk结束是否意味着prefill结束；每次执行都要生成logits，pruned_states至少包含一个token (参见Q1）
有关return logprob的参数

return logprob是在确定性推理场景，可以返回生成token的logprob，用来对比当前代码实现对于一个确定的输入序列: 从输入的第一个token到当前token，预测下一个token的对数概率。

如果想获取prompts的logprob可以通过设置logprob_start_len来指定需要返回logprob的token开始位置，缺省是-1表示最后一个prompt token位置，也就是生成第一个token的对数概率。

还有其他的参数：可以只输出topk个token的logprob或者某些token的logprob

更重要的是：return logprob是可选参数，开启可能会引起oom。
