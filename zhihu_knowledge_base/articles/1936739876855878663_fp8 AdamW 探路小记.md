1.2 指令集

或许给 int 套个参数，换个 affine 公式还能救？

对于关系 1。我们知道，古早的量化公式是 fp = scale * quant + zp

16~18 年左右的工程实践消除了 zp ，毕竟类型转换在指令层面确实开销大，没法提速。

所以在均匀表达的 int 基础上，再套一些额外参数，历史已经验证过走不通。

1.3 实际意义
综上，只要指令集层面搞不定 \text{float} : \mathbb{Z} \rightarrow \mathbb{R} 开销，总觉得 int 量化要凉...
所有尝试在训练优化中以 int 为核的方案，高情商的说法是“仍有研究价值和潜在探索空间”；低情商是？..
fp 的指数部分要先 pow2再减常数、乘以均匀的小数部分，结果是天然不均匀的，更符合模型对数值特性的诉求
二、fp8 AdamW

既然不看 bnb-like 的方案，Optimizer 加速还剩两条路：

新方法，已经试过 Muon，不是 GPU-poor 该碰的 你有没有把AdamW换成Muon?为什么？
改进现有的成熟 AdamW

回顾 AdamW 公式，别的都还好，但有一步：

v_t = 0.99 * v_{t-1} + 0.01 * g^2

0.99 就是 beta2 的常见取值，这里直接代入
g 就是 gradient

这个加法的左值和右值，min-max 差太多。如果用 fp8 存 v_t ，大概率模型的 math 指标是会掉点的。

事已至此，先吃饭吧！办法总是有的。
