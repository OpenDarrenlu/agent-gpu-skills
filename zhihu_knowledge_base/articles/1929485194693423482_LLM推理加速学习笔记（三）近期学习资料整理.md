1. flash attention

之前尝试过仔细研究 flash attention，但每次都被繁琐的公式所劝退；最近花时间了解了一下之后，感慨 flash attention 真是神级算法，做 research 就该以这种级别的创新为目标。

2. page attention (vLLM 的核心技术)

关于这部分的介绍，其实也有很多写的很好的帖子，比如




但是对于一个操作系统学得不是很好的人来说仍然不够直观。这里，我想从 speculative decoding 的角度来解读 page attention 产生的动机。有如下几个问题：

Medusa, Eagle 等模型为什么要自己重写 modeling_llama.py 这个文件？
Medusa, Eagle 等模型为什么要自己重写 kvcache.py 这个文件？

对于刚入门 spec 的新手来说，可能会觉得重写以上两个文件的动机是为了实现 tree attention，但其实并不是这样。这两个文件主要是为了方便快速的管理 kv cache。

我们知道，speculative decoding 之中，一个很重要的部分是验证 draft tokens 之后的 kv cache 回退。但如果使用 transformers 官方给的 cache 回退，速度就会很慢，因为多层的 kv cache 需要使用 for 循环来进行回退。这个问题在应用 tree attention 时更为显著——回退的 indices 有时候都不是连续的。而 Medusa 通过重写 modeling_llama.py 和 kvcache.py 这两个文件，预分配了一块显存给 kv cache。此时，连续的显存管理可以显著提高 kv cache 的回退速度。（在做 speculative decoding 的小伙伴十分需要注意这个点，可能可以使你的最终效果达到 1.2 倍）

但与之对应的是另外一个问题，如果生成的长度超出了 kv cache 的预分配大小该怎么办？（有人在 Spec-Bench 里提了一个 issue，至今没人回答，但其实就是因为生成的长度超出了预分配的 kv cache 大小）

此时就对应了 page attention 出现的动机——我们既想享受静态预分配 kv cache 的速度提升，又想享受动态 kv cache 的显存利用率，怎么办呢？一个折中的方案就是，我们动态分配 block！block 内是连续的，block 之间是动态变化的。具体介绍则可以参考上面贴的链接.

Prefill 和 Decode 是不同特性的计算任务：prefill 是compute-bound，且通常情况下 prefill的时间都显著长于decode单个token的时间；
