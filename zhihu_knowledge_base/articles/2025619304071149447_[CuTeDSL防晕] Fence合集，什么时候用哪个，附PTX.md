---

在高性能计算和算子开发（如 CUTLASS/CuTe）中，理解内存一致性（Memory Consistency）是进阶的必经之路。随着 NVIDIA 架构从 Ampere 演进到 Hopper 乃至 Blackwell，Fence 的种类变得极其繁杂。

熟悉cpu fence的同学可能不太清楚GPU的fence为什么如此复杂，我这里大概总结下：

GPU的hierarchy使得fence需要声明scope，就是这个fence是在哪些thread里起作用，有可能是一整个CTA也有可能是一个cluster
async operation和独立的hardware unit比如tma，tcgen05，即使是在thread内的program order也无法保证thread内部的内存一致性了(或者官方说法，他们只建立了一个比program order更weak的顺序）。需要proxy或者特殊的机制来确保内存可见。
相比较于2，传统的sc，acquire，release其实用的不是太多，所以大家在cpu里的经验用不太到了。

关于2，大家耳熟能详的是Nvidia PTX提到的async proxy。如果熟练掌握了async proxy的fence使用方式，那这些async operation的内存可见问题就不是难题? 没有那么简单:

即使都是async operation他们之间的机制也是不完全一样的，比如cp.async和wgmma.mma_async就不太一样。
tma的tensormap又是一个特殊情况，因为他是在定制的hardware上操作，所以需要引入一个单独的tensormap proxy，而它和async proxy用法上又有区别：他通常是generic->tensormap单向的。原因应该是generic从来不会直接用tensormap来操作，只负责对tensormap单向更新，和smem(generaric和async都要读写)是不一样的。
tcgen05又是完全不同的另一个情况，他没有一个单独的tcgen05 proxy，而是创建了一个单独的机制。理由也很简单,generic和async proxy都是在smem上竞争，而tcgen05只需要在新开辟的赛道tmem上竞争，所有tmem相关的操作都是tcgen05旗下的，所以就单独开了个局。

希望上面这些能整体上给大家清晰的展示出来Nv搭积木的这个过程。多的不吐槽了，主要还是让大家在这个MLM乱象中保持清醒。

今天我们系统梳理一下 CuTeDSL/PTX 中的五类 Fence. 他们的机制是什么，什么时候用他们。

1. Thread Fence

代表指令： fence.acq_rel CuTe API: cute::arch::fence_acq_rel_cta / cluster / gpu / sys

这是最基础的内存fence。它的作用是确保在同一个 Scope（CTA、Cluster 或 GPU）内，所有在该 Fence 之前的内存读写操作，对于 Fence 之后的操作是可见的，反之亦然。

默认行为： 在 PTX 中通常默认是 fence.acq_rel。
现状：cutlassfence_view_shared 经常配合 synclog 使用，用于调试或简单的 smem 同步。翻遍tutorials我也没看到其它case了，我本人也没用过。CuTeDSL的TensorMapManager倒是有用到。
2. Op Fence

代表指令： fence.mbarrier_init.release.cluster CuTe API: cute::arch::mbarrier_init_fence

这是一种特化版的 Thread Fence。它不针对所有内存操作，而是专门为 mbarrier_init 设计的。 目前op fence只有这一种。

核心逻辑： 它只保证在当前 Cluster 范围内，该 Fence 之前的 mbarrier_init 操作已经cluster scope smem可见。
典型场景： 在 CuTeDSL 的 pipeline_init_arrive() 函数中，我们通常需要确保 Cluster 内所有线程都能看到初始化后的 mbarrier 对象，这时就会调用它来标示 Cluster 初始化到达。这应该尤其对2cta的mbarrier重要。

我看到有人说这个是用来让async proxy看到的。不是的, 没有fence.proxy的都没有这个功能。

3. Proxy Fence

代表指令： fence.proxy CuTe API: cute::arch::fence_proxy(proxykind, space)

核心概念：Proxy Kind
Generic: 普通的 Load/Store 指令。
Async/Async.shared: 通过 cp.async 触发的异步拷贝。
TensorMap: 特指 TMA (Tensor Memory Accelerator) 访问。
Alias: virtual同地址映射。
Async Proxy

随着异步拷贝（cp.async, cp.async.bulk, wgmma.mma_async）的引入，GPU 内部出现了Async Proxy（代理）。不同代理看到的内存视图可能是不一致的。 在这之前Program order是可以确保在smem上的内存是一致的，同一个thread里比如ld，st的先后顺序也代表了smem的可见性顺序。但是async op使得smem不再确保是对async proxy可见的。或者用官方的说法：“they are not part of the program order for that thread” （https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#program-order-async-operations） 这就相当于你有个分身，本体还是generic proxy，分身去做别的事了。本体摸了smem，又命令分身去拿smem做事，这个顺序是需要告诉分身的。不然没有顺序上的保证，async分身很有可能提前去拿smem。这就体现了causality order感知上的偏差：程序上看上去是先摸smem再命令async分身，但是因果顺序却没有跟分身讲清楚。

还有一个非常关键的知识点：Generic Proxy 通常能看到 Async Proxy 的结果，但反之不然

cp.async.bulkhttps://docs.nvidia.com/cuda/parallel-thread-execution/index.html#async-proxy
wgmma.mma_asynchttps://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-async-proxy

这也是合理的，毕竟async proxy是一个不可感知的过程，你得有个机制知道什么时候结束了，不然就像盗梦空间一样，永远不知道自己是不是已经醒了。 这也解释了为什么tma_store开始之前需要一个proxy fence，但是tma_load结束之后却不用。

// same as cute.arch.fence_view_async_shared()
cute.arch.fence_proxy("async.shared", space="cta")
cute.copy(...)
cute.arch.cp_async_bulk_commit_group()
cute.arch.cp_async_bulk_wait_group(stage-1, read=True)
//read=True: until read location, not until write to loc
cute.arch.cp_async_bulk_wait_group((0, read=True)
Tensormap Proxy

这个主要是支持经常更新tensormap的需求(如grouped gemm）。tensormap的这些更新和使用的instruction是写在本体,但是针对tensormap的因果顺序没有建立，我们需要说明一个正确的顺序：1.先replace了tensormap。 2.再去用它来tma load。这个顺序是需要fence.proxy.tensormap来建立的,不然2可能会执行在1前面。因为不存在generic本体读tensormap的情况，这个顺序的建立一般是单向的：从generic到tensormap（fence.proxy.tensormap::generic)，generic本体单方面的传达：我更新你先别动，和你现在去用吧，这种命令：

tensormap.replace.tile.global_address.global.b1024.b64   [gbl], new_addr;
fence.proxy.tensormap::generic.release.gpu;
cvta.global.u64  tmap, gbl;
fence.proxy.tensormap::generic.acquire.gpu [tmap], 128;
cp.async.bulk.tensor.1d.shared::cluster.global.tile  [addr0], [tmap, {tc0}], [mbar0];

CuTeDSL相关function

# PTX: fence.proxy.tensormap::generic.release.gpu;
cute.nvgpu.cpasync.fence_tma_desc_release()
# PTX: tensormap.cp_fenceproxy.global.shared::cta.tensormap::generic.release.gpu.sync.aligned [$0], [$1], 128;
cute.nvgpu.cpasync.cp_fence_tma_desc_release(tmem_desc_global_ptr, tmem_desc_smem_ptr)
# PTX: fence.proxy.tensormap::generic.acquire.gpu [tma_desc_ptr], 128; // generic global addr -> tensormap mem access ordering

4. WGMMA Fence

代表指令： wgmma.fence CuTe API: cute::nvgpu::warpgroup::fence()

在 Hopper 架构中，WGMMA（Warpgroup Matrix Multiply-Accumulate）是异步执行的。 之前提到过wgmma.mma_async和cp.async有些微不同，这个不同就是cp.async和generic前序代码是有顺序建立的，但是wgmma.mma_async并没有。 这样子不光smem有问题，连reg的可见性都不靠谱了。 基于此才有了wgmma.fence这个专门的指令。和async proxy没关系哈 (https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#program-order-async-operations)

隐式同步： 实际上在不同的 MMA 指令之间，硬件会提供一定的隐式同步。wgmma之间相当于建立了一个pipeline了，所以这个fence在这种情况下是不需要的。（https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-instructions-wgmma-fence）
cute.nvgpu.warpgroup.fence()
cute.gemm(...)
cute.nvgpu.warpgroup.commit_group()
cute.nvgpu.warpgroup.wait_group(2)
5. TCGen05 Fence

在最新的 PTX 规范中，针对 TCGen05（Tensor Core Generation 05） 引入了更独立的内存一致性模型。（https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#tcgen05-memory-consistency-model） 在DeepGemm这个项目中是用的比较多的。

同步与异步指令

前缀都是有tcgen05.

Synchronous: alloc, dealloc, relinquish_alloc_permit, fence::\*, wait::\*, commit
Asynchronous: mma, cp, shift, ld, st

这里我们假设只考虑tcgen05序列的指令。 同步指令之间是有顺序建立的，异步指令之间没有。

显式流水线

异步指令之间的顺序是有特例的，官方把它们都列了出来：

mma+mma, cp+mma, shift+mma, shift+cp, mma+shift

给不熟悉的朋友说一下,tcgen05.cp是从smem到tmem的传输，tcgen05.shift是在tmem上不load到reg直接平移（conv有用）。 这个显示流水线的pattern一出现，就说明他们之间天然的有order，不需要fence

联系的来看hopper wgmma之间其实也有一个显示的流水线

隐式流水线

隐式流水线就是你等了一下，等的过程中隐式建立了一个流水线 等有两种，就是我之前文章介绍过的mbarrier和thread wait

mbarrier: mma+commit, cp+commit, shift+commit
wait: ld+wait::ld, st+wait::st
定制化流水线

前面都介绍完了才是重头戏，定制化流水线就是fence了。因为他和前两种比一个巨大的好处就是可以inter-thread

fence是有两个指令：

tcgen05.fence::before_thread_sync，相当于release
tcgen05.fence::after_thread_sync，相当于acquire (我不明白它为什么要避开这两个本名）

这里一共有5种模式

模式1

显示流水线
隐式流水线 wait版

不用加fence开箱即用,自带顺序

模式2 隐式流水线 mbarrier版

tcgen05.mma [d], ...
tcgen05.commit.mbarrier::arrive::one
mbarrier.try_wait.relaxed.cluster (loop until successful)
tcgen05.fence::after_thread_sync
tcgen05.ld [d], ...

你只动tmem还好，动了mbarrier，那不好意思，必须fence了。 例子中mma+commit已经带顺序了，后面的ld不能跑try_wait前面，所以after_thread_sync即可

模式3: multithread 流水线，但是在不同thread上

Thread 0:

tcgen05.cp
tcgen05.fence::before_thread_sync
mbarrier.arrive.relaxed.cluster

Thread 1:

mbarrier.try_wait.relaxed.cluster // loop till success
tcgen05.fence::after_thread_sync
tcgen05.mma

模式4: multithread 不构成流水线，在不同thread上。就该加commit加commit，该加wait加wait 下面的例子是上面这些模式的大融合：

Thread0:

tcgen05.mma
tcgen05.commit.mbarrier::arrive::one [bar1]
mbarrier.try_wait.relaxed.cluster [bar1] // loop
...
tcgen05.fence::after_thread_sync
...// completion is guaranteed
tcgen05.fence::before_thread_sync
mbarrier.arrive.relaxed.cluster   [bar2] // loop
...


Thread1:

mbarrier.try_wait.relaxed.cluster [bar2] // loop
...
tcgen05.fence::after_thread_sync
tcgen05.ld

模式5 ld register dependency 就是这个模式只针对tcgen05.ld，只针对register dependency，只针对intra-thread

举例来说

tcgen05.ld %r1, ...;
tcgen05.mma ..., %r1, ...;

这里应该机智的同学会问了，mma的operand不应该只能是smem或者tmem吗？为什么会有register？ 其实有可能你的smem_desc是存在tmem里的，也有可能block_scale…我其实想不大明白这个case有什么特别的用处，和fence也没什么关系，大家就记得这个情况下有order就行。
