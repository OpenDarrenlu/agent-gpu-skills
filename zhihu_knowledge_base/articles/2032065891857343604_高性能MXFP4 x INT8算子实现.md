之前讨论了MXFP4 x FP8的算子实现，但一些旧显卡（如A100）和一些国产卡不支持FP8计算，只支持INT8的计算。那么有没有办法用类似的trick实现高性能的MXFP4 x INT8算子呢。

答案是肯定的，但与FP8完全无损的不同的是，使用INT8实现这个trick通常会有一些精度损失。但这个损失通常是可以接受的。

FP4到INT8的反量化方法

先不考虑scales的问题，看下怎么把FP4快速反量化到INT8 。

由于浮点数有指数部分，且浮点数和整数对于负数的表示方法不一样（浮点数使用符号位，整数使用补码），所以没办法像FP4到FP8那样单纯通过简单的位操作来进行反量化。

但我们还有一个很好的工具：PRMT指令。我们可以通过PRMT查表来实现反量化。具体来说

使用2个int32（8个字节）分别存储8个非负FP4反量化后的INT8值
以FP4的低3位为索引，从8个字节中查表取得目标的INT8值。

先看下8个非负FP4取值的所有可能取值：

FP4取值	目标INT8值
0.0	0
0.5	1
1.0	2
1.5	3
2.0	4
3.0	6
4.0	8
6.0	12 (十六进制0xC)

由于FP4有0.5这种小数，所以我们将FP4反量化为对应的数值乘以2后的INT8值，只需要在epilogue阶段额外乘以0.5就好了。

在INT8上应用E8M0缩放

在INT是应用E8M0缩放的过程相当于对INT进行移位，比如乘以 
2
3
 相当于对一个INT左移3位。

我们反量化后得到最大数值是12，而INT8的最大数值是127。因此我们有额外的8倍动态范围可以利用，即允许的指数部分动态范围为3。

但根据上一篇文章统计的：

而如果是考虑有多少元素会受到影响的话（从权重层面考虑的话，实际上只有小于最大值1/64的元素会受影响）：

    for key, tensor in data.items():
        if tensor.dtype != torch.float8_e8m0fnu and not key.endswith("_proj_scales"):
            continue

        weight_scale = tensor.view(torch.uint8).cuda()
        weight = data[key.replace("_proj_scales", "_proj_blocks")].view(torch.int32).cuda()
        unpacked_weight = ops.unpack_weight(weight, 4)
        dequanted_weight = ops.dequant_weight(unpacked_weight, 2, 1, True)
        dequanted_weight *= weight_scale.view(torch.float8_e8m0fnu).float().unsqueeze(-1)

        # 0b1000是负0，需要将其改为0
        unpacked_weight[unpacked_weight == 8] = 0

        scale_max = weight_scale.max(-1)[0].unsqueeze(-1)
        scale_min = weight_scale.min(-1)[0].unsqueeze(-1)
        scale_range = scale_max - scale_min
        max_range = torch.tensor(3, dtype=torch.uint8, device=scale_range.device)
        scale_range = scale_range.minimum(max_range)
        scale_min_new = scale_max - scale_range
        delta_scale_offsets = weight_scale.maximum(scale_min_new) - weight_scale
        weight_scale = weight_scale.maximum(scale_min_new) - scale_min_new        
        scale_factor = 2 ** (scale_min_new.view(-1).float() - 127)

        ops.process_mxfp4_w4a8_weight(weight, delta_scale_offsets, inplace=True)

        unpacked_weight = ops.unpack_weight(weight, 4)
        dequanted_weight_new = ops.dequant_weight(unpacked_weight, 2, 1, True)
        dequanted_weight_new *= (weight_scale + scale_min_new).view(torch.float8_e8m0fnu).float().unsqueeze(-1)

        total_count += dequanted_weight_new.nelement()
        unmatched_count += (dequanted_weight != dequanted_weight_new).sum()

print(unmatched_count.item())

也就是说19b参数中只有万分之2的权重会受影响。而这些元素都是小于对应channel最大值的1/64的元素。因此对最终计算结果的影响完全可以忽略不计。

事实上，这个过程基本等价于把MXFP4反量化到BF16，再量化到INT8。不同点在于这里介绍的方法得到INT8最大值只有96，而不是127。（不过由于量化到最大值为127的INT8会有额外的舍入误差，所以这里介绍的方法的精度其实已经是最高的了。）

反量化过程
CUDA_INLINE uint2 dequant_single_for_mxfp4_fp8(const uint32_t qb, const uint32_t exp_offset) {
  // qb打包了8个FP4值，这8个值在同一个group里面（共享同一个scale）
  // 8个FP4值是按02134657的顺序打包的，反量化后会变回01234567的顺序
  // exp_offfset位根据上面描述得到的处理后的scale
  // 注意：这个处理中假设了不存在"负0" (-0.0)，否则会有部分数值错误。
  // 所以需要在权重预处理阶段把-0.0改为0.0

  // 左移相当于乘以 $2^{exp_offset}$
  uint32_t buffer1 = 0x03020100 << exp_offset;
  uint32_t buffer2 = 0x0C080604 << exp_offset;

  // res: 存储反量化结果。需要2个uint32_t来存储8个INT8
  uint32_t res[2];

  // prmt指令: 以FP4的低3位的所有可能取值为索引，反量化得到INT8
  // 由于符号位被忽略，所以得到的实际是绝对值
  uint32_t int8s[2] = {
    __byte_perm(buffer1, buffer2, qb),
    __byte_perm(buffer1, buffer2, qb >> 16)
  };

  PRAGMA_UNROLL
  for (uint32_t i = 0; i < 2; i++) {
    // prmt指令：把02134657变回01234567的顺序
    uint32_t val = __byte_perm(int8s[0], int8s[1], 0x6420 + 0x1111 * i);

    // flag为4个INT8类型的0/1，其中1表示对应的FP4为负数
    uint32_t flag = qb & (i == 0 ? 0x08080808 : 0x80808080)
    flag = flag >> (i == 0 ? 3 : 7);

    // val为绝对值，需要再区分正负数
    // 对于负数部分，取补码（先取反码再加1）
    res[i] = (val ^ (flag * 0xFF)) + flag;
  }

  // 注意uint2是两个uint32_t打包，不是指2bit的uint
  return *reinterpret_cast<uint2*>(res);
}
