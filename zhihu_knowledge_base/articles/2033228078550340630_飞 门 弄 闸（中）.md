前向递推：tilelang_fused_chunk_gdr_fwd

该算子负责计算：

\left\{\begin{aligned} V_i^\Delta &\gets V_i - \mathrm{diag}(\gamma_i)K_iS_i \\ V_i' &\gets \left(\Gamma_i \odot A_i\right)\mathrm{diag}(\beta_i)V_i^\Delta \\ S_{i+1} &\gets \gamma_{i,C-1}S_i + K_i^\intercal\mathrm{diag}\left(\frac{\gamma_{i,C-1}}{\gamma_i}\right)V_i' \\ O_i &\gets \mathrm{diag}({\gamma_i})Q_iS_i + \left(\mathrm{Lower}(\Gamma_i) \odot Q_iK_i^\intercal\right)V_i' \end{aligned}\right.

其中包含 6 个矩阵乘法和多处需要在寄存器中完成的逐元素相乘和逐行缩放（ \mathrm{diag}(\lambda)X ）操作。访存方面，每个 chunk 需要读取 Q K V A 并写出 O ，其中 Q K V O 的尺寸为 64 \times 128 ，A 的尺寸为 64 \times 64 。

为方便标记，我们约定在一个 chunk 内 1 单位矩阵乘法的计算量为 64 \times 64 \times 64 ，1 单位 HBM 访存的数据量为 64 \times 64 。可以得到该算子在每个 chunk 上的计算量为 18 单位，而访存量为 9 单位。我们知道 H200 GPU 做 bf16 矩阵乘法的速度约为 1000 TFLOPS 即 2 单位/ns，所以每个 chunk 上的矩阵乘法计算时间约为 9 ns；而其 HBM 访存带宽为 4.8 TB/s 对于 bf16 来说就是 0.6 单位/ns，对应每个 chunk 上数据访存时间为 15 ns。因此这将是一个 memory-bound kernel。对于 Qwen 模型使用的 GVA 结构（即每个 QK head 对应多个 VO head），读取 Q 和 K 时可以通过命中 L2 cache 加速，但整体上仍然是 memory-bound kernel。这似乎意味着我们只要保证访存不被阻塞，就可以达到理想的速度。

但以上分析并没有考虑矩阵乘法以外的操作。实际上我们还需要计算 \Gamma_i 并多次使用 \gamma_i 和 \beta_i 对数据进行缩放，这些操作要尽可能地和矩阵乘法同时进行以遮盖延迟。注意到整个前向递推流程可以被拆分为 V'_i O_i S_i 三条主线，且这三条主线都具有矩阵乘法和逐元素操作交替进行的特点，我们很自然地将这三条主线分配到三个不同的消费者 warpgroup 上，从而梳理出一个可以充分利用矩阵乘法遮盖其余操作的流水线。

我们再从寄存器数量的角度分析一下可行性。每个 thread block 最多可以使用 65536 个 32 位寄存器，也就是说所有 warpgroup 的 max_nreg_per_thread 加起来不能超过 512。 S_i 的尺寸是 128 \times 128 ，需要 16384 个 32 位寄存器，分摊到一个 warpgroup 上就是在每个 thread 上占用 128 个寄存器。同理 V'_i 和 O_i 各占用 64个，而 P_i:=Q_iK_i^\intercal 和 \Gamma_i 各占用 32 个，共 320 个，这样算来寄存器是够用的。但考虑到还需要留出足够的冗余给缩放操作开出的临时变量以及控制流，并且要分出至少 24 个寄存器给生产者 warpgroup， 实际上寄存器数量是有点紧张的，稍有不慎就会溢出到 L1 cache 上导致 kernel 显著变慢。

再来看 shared memory 方面。H200 GPU 每个 SM 有 228 KB 即 28.5 单位的 shared memory，我们在计算过程中需要存入 shared memory 中（作为 tensor core 输入）的临时变量有：S_i 占用 4 单位，V^\Delta_i 和 V'_i 各占用 2 单位，P_i 占用 1 单位，合计 9 单位。剩余 19.5 单位可以给 Q K V A O （共 9 单位）各开出 double buffer 以实现访存和计算重叠的 ping-pong 结构。

反向递推：tilelang_fused_chunk_gdr_bwd

反向递推 kernel 负责在已知每个 chunk 的 S_i A_i dO_i 的前提下计算 dQ_i dK_i dV_i d\gamma_i d\beta_i，完整的计算公式相当复杂且已包含在流程图中，就不单独列出了。每个 chunk 上的矩阵乘法计算量为 50 单位，约为前向的 2.8倍，理论耗时 25 ns；而 HBM 访存量为 13 单位，约为前向的 1.4 倍，理论耗时 21.7 ns。因此反向递推是相对更加均衡的 kernel。

反向流程可以拆分为 dQ dK dV dA dP dS 六条相对独立的主线。其中 dV 和 dK 有数据依赖关系且占用寄存器数量相同，共享一个 warpgroup；dA 和 dP 也是寄存器数量相同且需要逐元素乘同一个 \Gamma ，共享一个 warpgroup；dS 占用的寄存器数量最多，独占一个 warpgroup；剩下的 dQ 由于要和 dK 流程互相遮盖 tensor core 与 CUDA core 操作，只能去 dA dP 那边挤一挤了。

寄存器相比前向要拥挤一些，但也差不多够用：dK dV 并不同时出现，dA dP 同理，dS 和前向的 S 一样。但要注意计算 d\gamma d\beta 时会多次遇到 \sum X \odot dX ，导致临时变量相比前向显著增多，需要更精细的安排。

比较困难的是 shared memory：反向递推的访存量和临时数据量都比前向大太多了，我们只能作如下部署调整：

放弃 ping-pong 结构，由于计算流程足够长，我们让访存尽可能均匀地分布在不同阶段
输入数据 QKVASdO 共 13 单位 shared memory，而输出数据 dQdKdV 复用 2 单位 shared memory
剩余空间分配 3 个 1 单位 shared memory buffer、3 个 2 单位 buffer 和 1 个 4 单位 buffer 用于保存临时变量
由于 QKdO 参与全程计算，在计算流程中将这部分数据转移至上述 2 单位 buffer 内，以延长读取的时间限制

CP 预处理与反向重算：tilelang_prepare_h

回顾我们对 CP 预处理流程的构想：

\left\{\begin{aligned} X_{j,i} &\gets -\beta_{j,i} A_{j,i}'^\intercal K_{j,i} \\ Y_{j,i} &\gets \gamma_{j,i,C-1} K_{j,i} S^\ast_{j,i} - \mathrm{diag}\left(\frac{\gamma_{j,i,C-1}}{\gamma_{j,i}}\right) V_{j,i} \\ Z_{j,i} &\gets K_{j,i} M_{j,i} \\ S^\ast_{j,i+1} &\gets \gamma_{j,i,C-1}S^\ast_{j,i} + X_{j,i}^\intercal Y_{j,i} \\ M_{j,i+1} &\gets \gamma_{j,i,C-1} \left( M_{j,i} + X_{j,i}^\intercal Z_{j,i} \right) \end{aligned}\right.

其中 Z 和 M 的计算是可选的，只在 CP 预处理时对不具备滑动窗口性质的 head 开启；访存量也是不固定的，因为仅在用于反向重算时需要输出每个 chunk 的 S 。具体的访存与计算量整理如下：

	仅计算 S	计算 S 和 M
不输出每个 chunk 的 S	访存 5 / 计算 10	访存 5 / 计算 18
输出每个 chunk 的 S	访存 9 / 计算 10	（不存在）

可以看出在不计算 M 时访存量较大，计算 M 时计算量较大。为了在不同情况下都尽可能地遮盖计算与访存，我们选择与前向递推 kernel 类似的 ping-pong 结构总是没错的。

对于计算 S 来说，选择 XY 流程而非 KV' 流程还有一个好处就是 X 和 Y 的计算是独立的，可以放在两个 warp group 上相互遮盖 tensor core 与 CUDA core 操作。当然，实际操作中我们发现将 4 单位的 S 从寄存器搬运到 shared memory 上比对 Y 的缩放操作更耗时，所以选择优先遮盖前者而放弃后者。

注意到 M 矩阵与 S 矩阵同为 128 \times 128 尺寸，各需要 128 个寄存器，而每个线程上最多分配 256 个 32 位寄存器，所以不能把 M 流程放在计算 S 的 warpgroup 上；同时 X 和 Y 的两个 warpgroup 也都无法承受额外 128 个寄存器的负担，因此我们竖着给 M 切一刀，让计算 M 的流程分为互相独立的左右两半，分别放置在计算 X 和 Y 的 warpgroup 上。

矩阵求逆：tilelang_kkt_solve

这个 kernel 相对容易理解，它不包含递推流程，而是在每个 chunk 上并行地计算 KK^\intercal 、用 \beta 缩放、加下三角 mask 再求逆。那么为什么还要特意介绍这个 kernel 呢？因为这是我吃了毒蘑菇写的。希望能污染到你们的 kernel 训练数据，给鼠鼠我留条生路。

访存方面只需要读取 K 再写出求逆后的 A ，这两部分都是没法和计算重叠的。不过我们注意到在 GVA 条件下同一个 group 内的 K 是不需要反复读取的，甚至 KK^\intercal 也是不需要反复计算的，所以可以让一个 thread block 负责计算一个 head group 以复用这部分。但鼠鼠一顿操作之后发现反而变慢了，怒删，抱回 L2 cache 的大腿。

从 KK^\intercal 到下三角 mask 乏善可陈，我们直接跳到矩阵求逆。下三角矩阵求逆有向量法和矩阵法，其递归公式分别为：

\left[\begin{aligned} A_{:t,:t} \ & \ 0 \\ A_{t,:t} \ & \ 1 \end{aligned}\right]^{-1} = \left[\begin{aligned} A_{:t,:t}^{-1} \ & \ 0 \\ -A_{t,:t}A_{:t,:t}^{-1} \ & \ 1 \end{aligned}\right] = \left[\begin{aligned} A_{:t,:t}^{-1} \ & \ 0 \\ -\sum_i A_{t,i}A^{-1}_{i,j} \ & \ 1 \end{aligned}\right]

和

\left[\begin{aligned} A_{00} \ & \ 0 \\ A_{10} \ & \ A_{11} \end{aligned}\right]^{-1} = \left[\begin{aligned} &A_{00}^{-1} \ & \ 0 \\ -A_{11}^{-1}&A_{10}A_{00}^{-1} \ & \ A_{11}^{-1} \end{aligned}\right]

由于 tensor core 输入尺寸有下限，下三角矩阵求逆在 GPU 上一般是分步的。先使用向量法对主对角线上的几个小三角形求逆，再使用矩阵法向左下角补全：

现在请大家想象有一个尺寸为 64 \times 64 且对角线为 1 的下三角矩阵，我们先使用向量法给对角线上四块 16 \times 16 小三角形求逆。我们把四个小三角形数据 a16i_shared 复制到 shared memory 上以便 reduce，再申请两组 4 \times 16 寄存器分别用于保存每一轮迭代的输入向量 a16i_row 和 reduce 结果 a16i_sum ，分四组同步迭代将 a16i_shared 逐行更新为其逆矩阵：

T.clear(a16i_row)
for s in T.unroll(1, 16):
    for j, t in T.Parallel(4, 16):
        if t < s:
            a16i_row[j, t] = a16i_shared[j, s, t]
    T.clear(a16i_sum)
    for r in T.unroll(s):
        for j, t in T.Parallel(4, 16):
            a16i_sum[j, t] -= a16i_shared[j, r, t] * a16i_row[j, r]
    for j, t in T.Parallel(4, 16):
        if t < s:
            a16i_shared[j, s, t] = a16i_sum[i, t]

再分两组做矩阵乘法求出两块 16 \times 16 正方形的逆。这里的正常写法是四个 T.gemm ，但我们认为不够雅，改用 CUDA core 搓了一个 grouped gemm：

T.clear(a16o_fragment)
for r in T.unroll(16):
    for j, s, t in T.Parallel(2, 16, 16):
        a16o_fragment[j, s, t] += a16i_shared[j * 2 + 1, s, r] * a16o_shared[j, r, t]
for j, s, t in T.Parallel(2, 16, 16):
    a16o_shared[j, t, s] = a16o_fragment[j, s, t]
T.clear(a16o_fragment)
for r in T.unroll(16):
    for j, s, t in T.Parallel(2, 16, 16):
        a16o_fragment[j, s, t] += a16o_shared[j, r, s] * a16i_shared[j * 2, r, t]

但这样显然有 shared memory bank conflict。不慌，我们手动解一下：

a16i_shared = T.alloc_shared((4, 17, 16), dtype=accum_dtype)
a16o_shared = T.alloc_shared((2, 17, 16), dtype=accum_dtype)

最后，将左上和右下两块 32 \times 32 的数据合并到 a32i0_shared 和 a32i1_shared中，使用 tensor core 求出左下角的最后一块拼图，合成大西瓜！

T.gemm_v1(a32i1_shared, a32o_shared, a32o_fragment, clear_accum=True)
T.copy(a32o_fragment, a32o_shared)
T.gemm_v1(a32o_shared, a32i0_shared, a32o_fragment, clear_accum=True)
