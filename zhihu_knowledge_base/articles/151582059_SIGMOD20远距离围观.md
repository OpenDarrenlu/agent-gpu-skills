1.2 Machine Learning for Databases I（17junesigmodresearch3）

Active Learning for ML Enhanced Database Systems

很多用ML优化DB的工作都在用监督学习方法，而它的一大问题就是如果training data和实际deploy后的情况很不一致，那ML model效果就不大行了。本文希望让train好的ML model上线后也能快速adapting to the current workload。其实有很多方法都可以实现这个目的，本文关注的是active learning。

Active learning的思想是先用少量有标签样本训练一个粗糙的ML model。之后让这个model挑选出一些无标签样本（选出的待标注样本对模型最有用）丢给人工去label，然后model再从中学习并更新，依此循环。和标注一大堆数据直接喂给supervised learning model相比，达到同样的performance，active learning需要的有标签样本总数会少很多。

在ML for DB问题中，有标记样本也是很稀缺的，所以很多人会用bayesian optimization/RL等方法来避免一开始对大量训练数据的依赖。而本文提出了另一种思路，我们可以进行workload-centric data collection，让ML model针对当前workload的情况，从production database的stat中挑出一些unlabelled data，然后为这些data打上predicted label（用ML model预测出）和actual label（相当于新标注的准确标签）。这些新标注的additional data又会被拿来retrain ML model。依此类推。另外，为unlabelled data打标签的工作可以扔到DB replica上完成，这样就不会影响线上系统了。

这么看来很多ML上用过的策略都可以丢给DB圈接着玩了……个人猜测下一个会不会是meta learning/one-shot learning呢:)

DB4ML - An In-Memory Database Kernel with Machine Learning Support[视频]

很多用户的ML training data都是存在传统数据库上的，但传统数据库很少有支持ML的，用户不得不把数据挪来挪去。那不如让DB直接支持ML运算，不要被大数据抢了饭碗。

作者发现，ML运算和Transaction其实是很像的：ML中的parameter vector对应DB的Table、ML workers对应transaction（而且都需要parallel）、ML需要的parallel execution model也可以通过transaction实现，唯一剩下的就是iterative。而本文的contribution就是支持iterative transaction，然后为ML实现新的isolation levels。

1.3 Machine Learning for Databases II（18junesigmodresearch11）

Tim Kraska在这个section连中三篇，都是将AI应用于传统数据结构/算法的。

ALEX: An Updatable Adaptive Learned Index

之前有过大名鼎鼎的The Case of Learned Index Structures，但那时候只支持lookup on read-only data。本文的工作使得Learned Index也支持update和range query了。

Learning Multi-Dimensional Indexes

上面的工作是在single dimensional index上做的。但是有些SQL query涉及到了多个dimension（对RDBMS表的多个列进行操作），因此我们需要multi-dimensional index（例如B Tree VS R Tree）。本文的工作就是把它也ML化。

The Case for a Learned Sorting Algorithm.

Index做的差不多了，现在我们要让sort也能ML化。它的思路是从基数排序上进行改动，先用ML学习到approximation of the scaled empirical CDF for each record key，把它map到output array的相应位置，得到一个近似排好序的array，之后再用一些合适的传统排序算法得到最终结果。

2. New Hardware

新硬件也是个比较热门的话题。为此还专门安排了一个workshop。可惜没有录像……

从相关的paper来看，DB对新硬件的利用可以分为三类：

计算：

multi-core/multi-socket：在云环境中，单机就可以有很多的CPU socket以及CPU core。特别是在单个cpu的核心数开始挤牙膏的情况下，多CPU又开始回到我们的视线。[1]针对一篇关于事务并发控制的paper[VLDB’14，其中Table2总结了many-core场景下并发控制中存在的一些bottleneck]，在实际硬件上进行了re-evaluation。在re-evaluation的过程中发现了和原文simulator上结果的不一致，之后又对原始codebase进行了分析改进，在1568 cores的真实系统上实现了scale up。从中作者总结出：
1）most schemes are now memory-bound, emphasising the need to invest in latency hiding techniques such as interleaving with coroutines as an intriguing direction for future work on scalable concurrency control；
2）在做evaluation时，不妨考虑下fully utilize memory 的情况；
3）non-uniform communication(NUMA) is an important hardware characteristic, as it can amplify the impact of contention points in the concurrency control schemes on any large hardware (multi-socket and many-core).
GPU：之前一直有零零散散的用GPU加速DB的文章。[2]提出DB中的很多操作都是memory-bandwidth bound。虽然GPU访问显存很快，但很多previous work还是把GPU当作一个coprocessor：query processing在GPU上完成，结果最终还是要回传给CPU，因此CPU-GPU之间的PCIe总线带宽（比内存/显存都慢多了）会对GPU Accelerator的效果有很大影响。本文提出：1）显存越来越大了，完全可以放弃coprocessor的思路，直接store working set directly on the GPU，不要再扔到PCIe上传来传去了；2）提出了tile-based execution model，treating the thread block(Tile) as the basic execution unit with each thread block processing a tile of items（和CPU上的vector-based processing比较像，但GPU的parallelism肯定更高），把threads绑成一个tile还可以减少对global memory的访问；3）提出了block-wide functions（以tiles作为输入输出）来实现tile-based execution model。
评论区也有大神提出：把数据全塞进显存也比较困难，但bandwidth gap又不太可能消失。For hybrid systems, it would be interesting to understand how to map data and plan queries so that such back and forth data movement can be minimized. This may require some unconventional techniques。另外xiangyao还表示，GPU will be a big part of my research agenda :)
有意思的是，同一个session下还有篇关于GPU的paper[7]，用NVLink这种新的快速总线缓解了bandwidth gap。这样就算coprocessor模式也可以达到不错的性能，还降低了对显存容量的要求。所以说每种设计都是有trade off的吧，没有严格的好与坏……
FPGA：有些paper提出可以用FPGA实现transaction、data compression等任务。很多工作其实可以参考GPU for DB的思路……

存储：

NVM：[4][5]分析了Intel Optane（性能介于SSD和DRAM之间的persistent memory）对DB的作用。

通信：

RDMA：本次SIGMOD20上还真没有相关的paper……不过有一篇student abstract[6]上提到了。在网络速度越来越快的情况下，可以借助RDMA来实现decoupling computation and storage。本文实现了一个分布式的Message Brokers，使用RDMA完成节点（partition）间通信。另外，在过去一些其他会议上也有用RDMA处理transaction、优化KV-store之类的工作[FaRM NSDI’14, SOSP’15, SIGCOMM’14, VLDB’19]。




另外这个session里还有个很有意思的paper[3]。在all in AI的大潮下，很多工作都在试图用black-box的ML model对DB或者某部分组件进行动态调优。本文在autotuning the memory allocation on in-memory data analytic system的实验中却发现：

1） 基于人工经验规则的white-box方法（称为RelM）效果也很不错，而且调优的overhead最小。RelM的思路是对application进行profiling得出一些stat，然后根据人工总结出的经验规则给出调优建议，同时保证safety（比如不要OOM）。
2） 作者测试了常用的black-box方法Bayesian Optimization(BO)，发现这种方法有一定概率会陷入local minima，导致虽然付出了很多时间，效果却没有比default好多少(only 10%)。另外，当hardware configuration 改变时，BO之前积累的训练数据就不能用了。
3） 另外，作者还开发了Guided Bayesian Optimization(GBO)，意思是用一些经验规则指导BO的调优过程（用一个white-box的model对原始metric进行分析，生成一些additional metric，和原始metric一起丢给BO），这样做的意义是separating out the most suitable region of configuration space from the more expensive region。GBO比起BO一定程度上缓解了陷入局部最小值的问题，调优耗时和效果也比BO好一些。但additional metrics也导致搜索空间的维度更高了，增加了model probing的时间。
4） 另一个常用的black-box方法是基于强化学习的Deep Distributed Policy Gradient(DDPG)。强化学习近年来也越来越多的被用到类似的调优问题上。实验发现单机上DDPG其实是调优耗时最多的（RL确实比较难训练…），但它是基于reward来学习，而不是具体的metrics数据，因此可以灵活适应hardware configuration发生变化的情况[SIGMOD’19 CDBTune上有过总结]。

作者提出，像memory allocation这种问题背后的系统结构是非常复杂的。RelM的表现证明，即使人们开发了各种炫酷的black-box ML model，仍然无法完全取代老DBA们对系统的深入理解。让AI作为一个起辅助作用的角色可能更靠谱。tuning algorithms developed by Database Performance Data Scientists who combine an understanding of the underlying database platform with the ability to develop data-driven algorithms must not be overlooked。

3. Stream Processing

整体看下来，感觉大家关注的焦点集中在elasticity/ reconfigurable（如何在云上自动扩容）、state（改进kv store来适应state management/migration）、HTAP（其实感觉工业界对这个关注更多…）。

3.1 Beyond Analytics: The Evolution of Stream Processing Systems（17junesigmodtutorial4）

尽管streaming近年来在工业界越来越火，但其实这个概念在SIGMOD’92就被提出来了（continuous queries that they are issued once and henceforth run “continually” over the database）。Stream processing system经过多年的演变，现在比较常用的是dataflow模型。在dataflow模型中，计算任务被抽象成dataflow graph，系统根据dataflow graph生成query plan（本质也是一个graph），然后丢给cluster去执行（其实和tensorflow的计算图有点像）。与早期的streaming system不同的是，dataflow模型的研究热点在于：data parallelism, state management, exactly-once fault-tolerance, load management/elasticity等等。

state management：streaming system中的一些计算模块需要在处理数据时记录一些东西（状态，比如用到的ML model、一些变量、窗口内数据等），state就是用来做这个的。在streaming system中state一般以key-value的形式来存储。而内存肯定是放不下所有state的，需要往硬盘上丢一部分（out-of-core/scalability）。Scalable state architecture又可以分为embedded state（用于Flink，每个task各自的state和计算模块放一起，不需要remote call）、external state（所有的state单独放在一个remote persistent store里）、embedded compute（直接在存储层上实现stream processing）。另外一个热点问题是consistency，如何保证exactly once（commit epoch atomically）。
load management：这节主要关注如何让streaming system根据workload的变化实时动态调优。大部分之前已经详细看过了。常用的方法有：1）Load shedding（选择性的drop一些record。牺牲一些准确度来保证latency）。2）load-aware scheduling（根据每个operator的资源消耗情况，动态决定它们的运行顺序、资源分配等）。3）back-pressure（用buffer把upstream到来的超出处理能力的request先存起来，并对downstream限流）。4）Elastic（根据workload变动情况自动分配资源，例如不同operator的parallelism）。这其实也是一个非常适合all in的方向（build predictive performance models and take actions in real-time），参考Turbine[ICDE’20]。
Future and prospects：1）stream processing的特性使得它在ML model serving/preprocessing中也有用武之地，但人们期望它也能像tensorflow那样做iterative computation on streams（比如training）。2）对graph进行stream processing。对此需要对dataflow model进行改进。另外还有streaming graph中的partitioning、state management等问题。3）让stream processing支持transaction。4）elasticity。这个在云上会特别有用。5)更好的storage engine for streaming state management[HotStorage’20]。6）新硬件，比如GPU/FPGA/multi-core cpu/RDMA/NVM。

3.3 Stream Processing（18junesigmodresearch28）

Rhino: Efficient Management of Very Large Distributed State for Stream Processing Engines

Stream processing中一个很重要的话题就是state management，本质上就是用key-value storage engine来存储stream processing时产生的state。本文关注的问题是当需要on-the-fly reconfigure query execution plan时，如何管理大量（TB级）的state，包括降低state migration的网络开销、保证migrate过程中的exactly-once processing consistency。而本文的创新点在于proactive and incremental state migrations，意思是即使一个key的value数据量非常大时，也可以break down even a single key's value to migrate it into smaller parts, and send only the last delta update of the v，同时还需要保证delta checkpointing。

另外一个相似的工作是[Megaphone VLDB’19]，提出了拆分state migration task然后tunable migration size，本文和它进行了性能对比。




3.4 OLAP, Data Warehouses, and Key-Value Stores（18junesigmodresearch23）

Adaptive HTAP through Elastic Resource Scheduling

目前的一个热点问题是希望把大数据和数据库合二为一（虽然实际上很难hhh），实现在同一套系统内同时完成TP/AP两种任务，也就是HTAP。然而它们要依赖不同的存储格式：TP对应行存（RDBMS表中的一行对应一个record，一次操作一整行），AP对应列存（比如有一个AP任务要对一列求和），这二者的读写模式其实是比较矛盾的。对此目前有两种常用方案：1）在同一个storage system上同时完成TP/AP[可以参考这里的解读]；2）把TP/AP拆分成两个系统，request到来时可以从OLTP向OLAP转存数据，或者直接remote access。本文关注第二种方案（TiDB用的也是第二种方案）。

既然是两套单独的系统，就得保证它们之间的数据一致，本文定义了data freshness这个指标对其进行衡量。另外，不同类型的workload对data freshness的要求也是不同的（例如daily batched query对比real-time statistic）。既然TP和AP两套系统对资源（CPU/内存总量）的需求相当于是相互竞争的关系，不如设计一个scheduler，根据workload情况对两种engine的资源进行动态分配（elastically trading resources between engines），另外对于新来的request，data access的方式也是用scheduler动态决定的（sec4.2）。scheduler在保证data freshness的情况下尽可能达到fairness。这就是本文的思路。

4. Cloud/Distributed DB

数据库上云已经是大势所趋了，serverless又是云计算中的一个热点。SIGMOD20中很多文章讨论了如何让数据库更加适应serverless的环境。

4.1 Le Taureau: Deconstructing the Serverless Landscape & A Look Forward（18junesigmodtutorial7）

Serverless使得人们部署应用时不需要花时间在搭建VM、安装软件、配置环境等琐碎的事情上（类似云计算VS自己买电脑搭机房）。如今已经有了很多cloud function services（比如AWS Lambda）允许人们把一些规模较小、运行时间短的应用打包成function，丢到第三方提供的无状态（stateless）计算容器上。容器启动/关闭速度很快、扩容缩容也更灵活，因此可以按需计费，而不用一直按最大需求量开着烧钱。这种服务又被称为FaaS（Function-as-a-service）。另外serverless还有一种类型，叫做Backend-as-a-service。BaaS和FaaS的区别是BaaS是把一些特定的、更加复杂的应用做成service供用户直接用（例如AWS S3、Aurora、BigQuery），而Faas的应用可以由用户自己program。

物理机->虚拟机->container->serverless也是虚拟化不断发展的结果。

Serverless的特点包括：

1) unlimited elasticity，随时扩容缩容、调整parallelism。用户再也不用担心load balancing了
2) 按需付费
3) Stateless，意味着applications do not need to share or exchange state between functions。Function worker本身不能持久化存数据。

很多需要parallelism的应用都可以从中受益。作者列举了video processing、graph processing、ETL、ML等例子。

最后，作者也提出了serverless平台本身的future work：

1) Security issue。Serverless的本质还是虚拟化，很多攻击手段都可能导致function在处理的数据被泄露。另外，function worker与cloud storage通信的过程中也可能导致泄密。
2) 因为Serverless的本质还是虚拟化，higher resource sharing also leads to unpredictable performance，比如某个function的latency（特别是cold start latency）突然增大。
3) 新硬件。比如让function worker支持GPU/TPU




4.2 Serverless and Cloud Data Management（16junesigmodresearch2）

Starling: A Scalable Query Engine on Cloud Function Services [解读]

之前我们看过一些如何让DB更加elasticity的工作，但很多 workload的变化是很难预测的，会有很多峰值/低谷而不是一个平稳的直线。如果部署在传统VPC上，服务器就算能动态扩容/缩容，开销也比较大。而现在的一个趋势是，不如拆开传统的VPC，数据直接存到cloud storage上，计算工作交给更灵活的FaaS系统。本文研究的就是如何用FaaS实现data analytic system，具体做了以下工作：

1) 把计算任务丢给Function以实现elasticity。这里elasticity体现在可以按需计费（pay per query），并且可以人为控制performance/cost之间的trade-off（Tunable Performance）。
2) 可以直接从raw data格式上开始处理，不需要load into internal format这种操作。适应data lake的要求。
3) 为了满足快速扩容缩容的要求，单个function worker的处理能力不可能做的很大，而且一般是无状态（stateless）的，意味着需要人工shuffle/aggregate data。本文使用AWS S3来存储intermediate state，并且parallelizing read来降低latency。
4) 在parallelism大的时候，可能有些奇怪的worker处理时间过长，成为straggler，导致总的latency上升。基本思路就是用一个model预测可能的latency，如果超时了就重试。简单粗暴但是管用…

Learning a Partitioning Advisor for Cloud Databases [视频]

这是个典型的all in的工作。分库分表是个经典的应届生面试题，本文就是用RL来自动进行分库分表（sharding）。Sharding的策略是有很多种的，需要考虑DB schema的结构、workload类型、DBMS类型、每个node的配置等等。作者希望用RL model去衡量不同sharding策略的cost和收益，选出一个最好的分表方式。总体思想和[SIGMOD’19 CDBTune]非常像，都是先Offline Training用simulated data训练一个不太精确的model，然后再Online Training用线上的真实数据训练。

5. Video Data Management

其实video在计算机网络研究中已经越来越多了，近一两年的NSDI/SIGCOMM中也有一些paper。但数据库中对它的研究还不多。这是个很新也挺有意思的方向。毕业就可以去抖音上班了:)

SIGMOD20上有以下几个相关的工作，主要集中在如何对视频内容进行查询。在CV界很多研究者开发了各种猛如虎的objection detection算法，但如果在大量的原始高清video上直接跑一遍代价是很高的，特别是VR video有360°的视角。而DB界研究的就是如何设计像query planning等策略，尽量减少查询代价。比如先用low-quality或者是局部视角的小video预处理，然后减少跑objection detection的次数。还有一个方向是如何降低存储/传输的开销。
