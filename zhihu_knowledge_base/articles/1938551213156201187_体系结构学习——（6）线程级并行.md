什么是线程级并行

我们知道提升处理器性能有不同维度的方案。从指令上看，流水线可以增大同时执行的指令数量，这是指令级并行（ILP）；从数据上看，向量计算的数据量比标量计算更多，这是数据级并行；此外，SMT可以利用单处理器中因为冲突或者访存等的stall空闲，将处理器的“工作”塞得“满满当当”，这其实算是线程级的一种。而另外的多线程并行，则是同时使用多个处理器进行协同计算工作，它也和SIMD对应，属于MIMD（多指令多线程）。

这里的多处理器（multiprocessor）定义为由紧耦合处理器组成的计算机，这些处理器的协调和使用通常由单个操作系统控制。也就是说，这些处理器属于一个整体，它们需要通过共享地址空间来共享存储器，更深层次的目的是，进行协调同步和通信。当然，需要注意的是，共享地址空间并不意味着只有一个存储器。

本章讨论的内容主要指拥有4到256个核的MIMD多处理器。由于这些处理器有各自的Cache，还会共享存储空间，而不同的存储分布方式对应着不同的通信延迟（处理器访存不同位置的存储器需要不同的物理时间），因此可以按照存储器的组织方式分类多处理器。

对称多处理器（symmetric multiprocessor， SMP）

也称为对称共享存储器多处理器（symmetric shared-memory multiprocessor）、集中式共享存储器多处理器（centralized shared-memory multiprocessor）或者一致存储器访问（uuniform memory access，UMA）多处理器。这类多处理器的特点是处理器数量较少，一般不超过32个，因此可以共享一个集中式存储器并且平等地（访问时间基本一致）访问它。

上图展示了SMP的基本结构。大型多级缓存可以大大的降低存储器对存储带宽的需求，这一发现促进了集中式存储器多处理器的发展。采用对称共享存储器的计算机通常支持对共享数据和私有数据的缓存。私有数据供单个处理器使用，存放在私有缓存（private cache）中，而共享数据则由多个处理器使用，放在共享缓存（shared cache）中，基本是通过读写共享数据来实现处理器之间的通信。

还有一个概念需要注意，一些多处理器虽然只有一个主存储器，但是一些多核对最外层的缓存有不一致的访问，这种结构称为NUCA（nonuniform cache access），因此不是真正的SMP。IBM Power8设计了一套分布式的L3缓存不同的地址有不同的访问延迟。

分布式共享存储器（distributed shared memory，DSM）多处理器

当处理器核更多时，集中式处理器无法在不大幅延长访问延迟的情况下为大量处理器提供高带宽支持，此时存储器必须分散在处理器之间，如图。

将存储器分散在节点上，即增加了带宽，也降低了到局部存储器的延迟。DSM多处理器也称为NUMA（非一致存储器访问），处理器访问的访问时间取决于数据在存储器中的位置。DSM的主要缺点是让在处理器之间传送数据的过程变复杂了，而且需要软件付出更多的努力才能利用分布式存储器提升的存储带宽。

线程级并行的挑战

并行处理的挑战主要在两个方面：运行程序的并行度不足和远程通信延迟太高。应用程序并行度不足使得多处理器的得不到充分的利用，这需要在软件中采用并行性能更高的新算法解决，还要在软件系统中尽可能多地使用完整的处理器。远程延迟过高的影响可以由体系结构和程序员来降低。从硬件上，可以使用缓存共享数据等方式，从软件上，可以调整数据结构，使应用程序尽量访问局部存储器。

使用缓存可以降低处理器远程访问频率，但是需要注意保持存储器的一致性。后续将对这一问题进行讨论。

缓存一致性问题与基本实现方案
多处理器缓存一致性

多处理器中的每个处理器拥有自己的Cache，可以减少访问共享存储的频率，但这会引入新的问题。因为不同的处理器是通过各自的缓存来保存存储器视图的，所以针对同一存储地址，它们可能会看到不同的值。下图演示了这种情况。CPU-A和CPU-B表示多处理器中的两个不同的处理器，它们拥有自己的Cache（A和B），当它们同时对Memory中的同一地址S进行通信时，可能出现如下情况：

初始时，Memory中的地址S存放的数据为0，CPU-A和CPU-B从Memory中找到S并放入自己的Cache；
CPU-A计算对S写入了值1，并且同步到了Memory中，但是CPU-B访问S时Cache-B一直hit，因此一直使用S=0的值；
此时Memory，Cache中S的状态就如图所示，CPU-B拿到的S值已经是旧值。

上述问题就属于缓存一致性问题，多处理器要能正常工作，就需要确保存储器系统的一致性。那么问题来了，何为一致性？

通俗地说：如果每次读取某一数据项都会返回该数据项的最新写入值，就说这个存储器系统是一致的。更细致地，这个定义应该包含两个方面：第一个方面是一致性，它定义读操作能返回什么值。第二个方面是连贯性：它决定了一个写入值什么时候被读操作返回。

第一个方面的一致性需要满足以下条件：

处理器P对位置X的读操作跟在P对X的写操作之后，并且在P的写操作和读操作之间没有其它处理器对X执行写操作，次读操作总是返回P写入的值。——这定义了同一处理器读写的顺序；
如果一个处理器对位置X的读操作紧跟在另一个处理器对X的写操作之后，读写操作的间隔时间足够长（确保能写入），且两次访问间没有其它处理器对X执行写操作，那么该读操作将返回写入值。——定义一个处理器读另一处理器的顺序；
对同一位置执行的写操作是被串行化的。即，在所有处理器的视角，任意两个处理器对相同位置执行的两次写操作顺序相同。

一致性和连贯性是互补的：一致性确定了向同一存储地址的读写行为，而连贯性确定了对于其它存储地址的访问的读写行为。如果一个处理器写写入A位置，然后写入位置B，那么任何能够看到B中新值的处理器也必须看到A中的新值。

一致性基本实现方案

为多个处理器保持缓存一致性的协议称为\textbf{缓存一致性协议（cache coherence protocol）}。实现缓存一致性协议的关键在于跟踪数据快的所有共享状态。任何缓存块的状态都是使用与该块相关联的状态位来保持的。目前使用的协议有两类：

监听协议——是最早也是最直观的缓存一致性协议类型之一。其核心思想是利用共享广播介质（通常是总线，也可以是支持高效广播的互连网络），让所有缓存控制器持续监听（Snoop）总线上发生的所有与内存访问相关的事务（尤其是写操作）。缓存控制器根据监听到的信息和修改自身对应缓存的状态。
目录协议——引入一个集中式或分布式的目录 (Directory) 来精确跟踪哪些处理器缓存了哪些内存数据块的副本，以及这些副本的状态（主要是共享者列表）。一致性操作不再是广播，而是点对点通信，只通知真正相关的缓存。
监听一致性

监听协议有多种，协议不同，状态集也不同，如MSI, MESI, MOESI等，但其核心思想类似。这里以MESI为例说明。

监听一致性之MESI

在监听协议中，所有处理器的缓存控制器连接到同一个共享总线（或等效的广播网络）上。并且每个缓存控制器都持续地监听（Snoop）总线上发生的所有事务（Transaction）。当一个缓存控制器需要对某个内存块（Cache Line）执行可能影响一致性的操作时（主要是写操作），它必须首先在总线上发起一个特定的一致性事务。其他缓存控制器通过监听总线，捕捉到这些事务。根据事务类型、自身是否缓存了该数据块、以及自身维护的该数据块的状态，它们会采取相应的行动（如使本地副本失效、提供数据、更新状态等）来维护一致性。

在MESI中，每个缓存行在每个缓存中都维护一个状态位。包括：

Modified (M)： 该缓存行已被修改（与主存不同），并且是系统中唯一有效的副本。该缓存有责任在必要时将数据写回主存或提供给其他请求者。
Exclusive (E)： 该缓存行是干净的（与主存一致），并且是系统中唯一的副本。可以本地写入而无需通知其他缓存（写入后会变成M状态）。
Shared (S)： 该缓存行是干净的（与主存一致），并且系统中可能存在其他共享副本（只读）。
Invalid (I)： 该缓存行数据无效（不能使用）。这是初始状态或数据被其他核心修改后被作废的状态。
关键操作流程

如下，（以写操作为例 - 写无效策略，写无效策略指处理器在写入某一数据项之前，获取对该数据项的独占访问）：

1、发起写请求：当一个核心（Core A）想要写入一个它缓存的内存块时，该核心检查本地状态：

State	operation
M或E状态	可以直接写入本地缓存（写入后状态变为M），无需总线事务（因为是唯一拥有者）。
S或I状态	说明可能有其他缓存拥有该数据的有效副本（S或M状态）。那么需要向总线发出写事务，以获得写权限。

2、发起总线事务：Core A的缓存控制器在总线上广播一个“总线读独占”或“无效化”事务。这个事务包含要写入的内存地址，并要求：

步骤1	获取该数据块的独占访问权
步骤2	使所有其他缓存中该数据块的副本无效

3、嗅探与响应：所有其他缓存控制器监听到这个总线事务。检查自己是否缓存了该地址的数据块及其状态:

4、获取数据与完成写操作：如果总线事务期间没有缓存提供数据（或者协议要求从主存取），Core A 会从主存读取该数据块到它的缓存中（虽然它要写，但通常需要先整块读取，再写指定部分）。

步骤1	Core A 现在拥有了该数据块的独占权（状态变为E或M），并且所有其他缓存中的副本都已无效（I状态）。
步骤2	Core A 执行写入操作到它的本地缓存行（状态变为M）。

5、对于读操作：读操作通常也会在总线上发起“总线读共享”事务：

嗅探者如果拥有M状态的数据	需要提供数据（并可能写回主存）或阻止主存响应，并将状态降为S
嗅探者如果拥有S或E状态	通常不需要提供数据（主存提供），但知道又多了一个共享者（状态保持或修改为S）。

请求者收到数据后，如果只有一个拥有者，状态设为E；如果已有其他共享者，状态设为S。


监听协议的优缺点

监听协议的优点

概念简单直观。
实现相对直接（依赖于共享总线/广播）。
延迟较低（对于小型系统），因为一致性操作（如无效化）是通过广播即时完成的。

监听协议的缺点

可扩展性差：这是最主要的缺点。随着处理器核心数量的增加
总线成为瓶颈：所有的一致性事务（尤其是写操作）都需要广播，导致总线流量爆炸式增长。
依赖于广播介质： 需要总线或支持高效广播的互连网络。

监听一致性协议利用共享总线的广播特性，通过让所有缓存控制器“监听”总线上的一致性事务（主要是针对写操作的无效化或更新请求），并根据这些事务和自身缓存的状态采取行动（如使副本无效、更新状态、提供数据等），来维护多个私有缓存之间的数据一致性。它以状态机（如MESI）为基础来跟踪缓存行的所有权和有效性。

监听协议是实现简单、低延迟的缓存一致性解决方案，特别适合于处理器核心数量较少（如2-8个）的小型多核系统（如早期的SMP系统、现代多核CPU的L1/L2缓存一致性）。然而，其总线带宽和嗅探开销限制了它在大型多处理器系统中的可扩展性。 对于大规模系统，通常会采用基于目录的协议来避免广播开销。

目录式缓存一致性

分布式存储器可以用于提升存储带宽核互联带宽，适合于多核的多处理器系统。而监听协议虽然协议简单，但是对通信带宽的消耗也很大。因此需要使用目录协议。

基于目录的缓存一致性协议引入一个集中式或分布式的目录 (Directory) 来精确跟踪哪些处理器缓存了哪些内存数据块的副本，以及这些副本的状态（主要是共享者列表）。一致性操作不再是广播，而是点对点通信，只通知真正相关的缓存，显著减少了不必要的网络流量。目录协议为每个节点添加目录，以管理相应位置的内存，如下图。




在基于目录的缓存一致性协议中，需要同时存在两种状态维护机制：一是目录（通常位于内存控制器侧），用于维护全局状态和共享者信息，二是每个处理器核的私有缓存控制器维护其本地缓存行的状态（如 MESI 或其变种）。

目录式缓存一致性介绍

目录协议的关键组件

目录 (Directory)：通常与内存控制器集成在一起（每个内存控制器负责其管辖范围内内存块的目录项），或者实现为分布式的目录模块（用于DSM）。其目录项(Directory Entry)为每个内存块维护一个条目，包含：
缓存状态：每个核心的私有缓存中，每个缓存行也维护本地状态，与监听协议类似（如 M, E, S, I），用于管理本地访问权限和数据有效性。
互连网络 (Interconnect)：不再是单一的广播总线，而是高性能的点对点互连网络（如 Mesh, Torus, Ring, Crossbar）。支持高效的点对点消息传递。
目录协议的工作流程

以写操作为例 - 写无效策略，假设处理器核心 C1 想要写入内存地址 X 所在的数据块。

1、C1发起请求

2、共享者/拥有者处理消息

C1 完成写操作

1、核心 C1 收到目录发来的 ACK（可能包含数据）
2、如果数据随 ACK 到达，C1 将数据块加载到其缓存
3、C1 将本地缓存行的状态设置为 MODIFIED (M)
4、C1 现在可以安全地执行写入操作到其本地缓存副本
目录协议的优缺点

目录协议的优点

可扩展性：核心优势。通过点对点通信只通知相关节点，避免了全局广播风暴。通信量随系统规模增长相对平缓
降低无关开销：节点只处理与自己缓存数据相关的目录消息，极大减少了功耗和网络带宽浪费。
高效利用互连网络： 天然适合点对点、高带宽、低延迟的现代互连网络（Mesh, Torus, Hypercube）。
减少对总线/广播的依赖： 摆脱了共享总线瓶颈。

目录协议的缺点

实现复杂度高：目录结构、状态机、点对点消息协议的设计、实现和验证都非常复杂。故障处理也更困难；
目录存储开销：目录本身需要消耗额外的存储空间（SRAM）。全映射目录开销巨大；有限指针和稀疏目录是必要的工程优化；
潜在延迟增加：请求需要经过目录控制器（可能成为瓶颈），点对点消息可能需要经过多跳网络，增加了请求完成时间（尤其是与监听协议在单跳总线上的延迟相比）；
COARSE 状态开销：在有限指针目录中，当共享者过多时退化的处理（广播/组播）会带来额外开销，需优化使其发生概率尽可能低
