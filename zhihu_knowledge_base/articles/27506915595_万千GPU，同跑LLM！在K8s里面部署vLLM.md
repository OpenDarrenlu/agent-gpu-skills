# 万千GPU，同跑LLM！在K8s里面部署vLLM

**作者**: 云山乱TensorMesh联合创始人，vLLM+LMC 开发者

**原文链接**: https://zhuanlan.zhihu.com/p/27506915595

---

LLM这个领域可谓是千帆竞发。Deepseek最近开源了通信kernel还有存储系统，同时UChicago研究团队与vLLM紧密合作的新项目——vLLM Production Stack——也开始正式发布。vLLM Production Stack是由UChicago团队开发，在vLLM官方project下的的一个开源的大语言模型部署框架。核心目标是既要有要：既要解决production的痛点（性能，稳定性，与K8s原生适配），又要有世界前沿的研究。它提供了分布式KV cache共享、K8s原生支持（包括Helm chart部署以及基于CRD的路由）、LoRA管理，自动扩缩容和多replica负载均衡，填补了vLLM项目在分布式部署上的空白。

UChicago团队一开始推出的LMCache项目是以研究为核心，专注于解决KV cache的压缩，共享以及复用。但同时，LLM部署的真正难点于我来说是怎么去跨层级进行调度和统筹：路由层需要从存储层拿到信息来作出正确的路由决定，同时计算层需要和存储联合协作来保证计算和存储可以尽可能并行）并且需要支持用户各种各样的定制化部署问题，比如LoRA和多模型。好在随着vLLM Production Stack的开源，这些问题至少有了一个可以供大家实验的优秀土壤

我自己也有一些前缀路由有关的idea，正好可以借这个机会在真正的Kubernetes环境里面的写一写测一测。

以下内容翻译自vLLM production stack的官方博客。

vLLM Production Stack是由芝加哥大学团队和vLLM团队的合作孕育而出的项目。已有来自IBM、RedHat, Alluxio， Lambda、HuggingFace等公司的30多位贡献者。

vLLM Production Stack的核心价值
分布式KV缓存共享
基于UChicago的LMCache技术，实现了跨多个模型replica的KV缓存共享，显著减少内存冗余，提升资源利用率。
K8s原生支持与自动扩缩容
提供CRD-based路由、自动扩缩容和LoRA适配器管理，轻松集成到K8s环境，同时保持高性能。
多模型replica的负载均衡
通过LLM感知的路由和分布式缓存共享，解决多replica的负载均衡问题，适合高并发、低延迟的生产环境。
与vLLM核心项目的无缝兼容
通过LMCache KV Connector，vLLM Production Stack与vLLM核心项目保持高度兼容，确保用户始终使用最新技术。
对社区的意义

vLLM Production Stack填补了vLLM生态在分布式部署上的空白，为大规模LLM服务提供了一个官方参考实现。项目完全开源，社区活跃，已有来自IBM、Lambda、HuggingFace等公司的30多位贡献者。
