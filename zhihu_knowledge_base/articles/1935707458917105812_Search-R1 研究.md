abstract：相比于传统采用prompt的LLM，search r1通过RL+基于结果的奖励函数+多轮查询交互优化了LLM的推理过程。其成功超过传统RAG 24%(Qwen2.5-7B)和20%（Qwen2.5-3B）。

intro：

1.目前训练LLM调用搜索引擎往往是两种策略：1.RAG；2.训练其作为tool。现有的依赖prompt的策略是欠佳的，会存在两个问题：（1）pretrain没遇到的任务无法解决；（2）受限于需要标注的数据，同时也因为搜索引擎不可微，所以无法用在梯度下降策略中。

2.RL：RL的魅力在DeepSeek的report中，我们已经充分见证过。但是把RL和tool调用结合存在以下问题：1）RL框架还没有和搜索融合；2）涉及多次推理中搜索 ；3）奖励设计；

3.search-r1创新点：1）在推理过程中，根据采样的token能够调用搜索引擎；2）支持多轮调用搜索引擎，通过<search> 和</search>调用，得到的是<information></information>。注意，这些是在<think></think>中的，答案用<answer></answer>表示。3）设置简单的奖励函数，避免了复杂的奖励函数。

采用了PPO和GRPO两个方法，利用了它们的优势（最近还有一个GSPO，有时间解析一下）

Loss Masking for Retrieved Tokens：传统RL算法往往基于LLM generate的tokens计算损失，但是现在既有LLM generate，也有搜索所得的tokens。如果优化LLM generate的tokens能促进其和搜索引擎的交互；但是如果计算搜索所得的tokens，整个系统的行为就难以预测。所以对搜索所得的tokens进行mask。

所以有对PPO和GRPO对改进如下：

J_{PPO}(θ) = E_{x~D,y~π_old(·|x;R)} [\frac{1}{Σ_{i=1}^{|y|} I(y_i) }Σ_{i=1:I(y_i)=1}^{|y|} min(\frac{π_θ(y_i|x,y_{<i};R)}{π_old(y_i|x,y_{<i};R) A_i}, clip(\frac{π_θ(y_i|x,y_{<i};R)}{π_old(y_i|x,y_{<i};R)}, 1-ε, 1+ε) A_i)]

J_GRPO(θ) = E_{x\sim D,{y_i}_{i=1}^G~π_old(·|x;R)} [\frac{1}{G} Σ_{i=1}^G \frac{1}{Σ_{t=1}^{|y_i|} I(y_{i,t})} Σ_{t=1:I(y_{i,t})=1}^{|y_i|} min(\frac{π_θ(y_{i,t}|x,y_{i,<t};R)}{π_old(y_{i,t}|x,y_{i,<t};R) Â_{i,t}}, clip(\frac{π_θ(y_{i,t}|x,y_{i,<t};R)}{π_old(y_{i,t}|x,y_{i,<t};R)}, 1-ε, 1+ε) Â_{i,t}) - βD_KL[π_θ||π_ref]]

mask:如果是LLM生成的token就让I为1，否则为0；




（2）Generation with Multi-turn Search Engine Calling：

设计了<search>xxx</search>,会调用搜索引擎搜索xxx，然后返回信息<information></information>,知道达到最大长度/遇到了<answer></answer>

（3）Training Template：

prompt：

算法流程：

Require: Input query x, policy model πθ, search engine R, maximum action budget B.
Ensure: Final response y.
1: Initialize rollout sequence y ← ∅ 
2: Initialize action count b ← 0 初始化句子和动作次数计数
3: while b < B do 在budget范围内，不断rollout
4:   Initialize current action LLM rollout sequence yb ← ∅
5:   while True do
6:     Generate response token yt ∼ πθ(· | x, y + yb)基于输入 query x 和当前上下文 y + yb，由策略模型生成下一个 token yt
7:     Append yt to rollout sequence yb ← yb + yt 将新生成的 token yt 加入当前 rollout 子序列 yb 中。
8:     if yt in [</search>, </answer>, <eos>] then break 遇到搜索，答案生成或者自然推出，就停止
9:     end if
10:  end while
11:  y ← y + yb
12:  if <search> </search> detected in yb then 如果遇到search，就提取内容，用information包裹插回去
13:   Extract search query q ← Parse(yb, <search>, </search>)
14:   Retrieve search results d = R(q)
15:   Insert d into rollout y ← y + <information>d</information>
16:  else if <answer> </answer> detected in yb then 如果是answer，可以返回了
17:   return final generated response y
18:  else 如果都不是，eos退出但没有答案，就引导重新思考一遍
19:   Ask for rethink y ← y + “My action is not correct. Let me rethink.”
20:  end if
21:  Increment action count b ← b + 1 增加b
22: end while
23: return final generated response y

（4）Reward Modeling

采用了简单的rule奖励规则（应该就是math boxed{}和code 沙盒环境测测试案例），不采用格式奖励（学习过的模型没有出现这个问题）；不采用reward model（因为其对格式敏感，需要额外训练）

experiment：:

数据集：(1) General Question Answering: NQ (Kwiatkowski et al., 2019), TriviaQA (Joshi et al., 2017), and PopQA (Mallen et al., 2022). (2) Multi-Hop Question Answering: HotpotQA (Yang et al., 2018), 2WikiMultiHopQA (Ho et al., 2020), Musique (Trivedi et al., 2022b), and Bamboogle (Press et al., 2022).

基线：

(1) Inference without Retrieval: Direct inference and Chain-of-Thought (CoT) reasoning (Wei et al., 2022).

(2) Inference with Retrieval: Retrieval-Augmented Generation (RAG) (Lewis et al., 2020), IRCoT (Trivedi et al., 2022a), and Search-o1 (Li et al., 2025).

(3) FineTuning-Based Methods: Supervised fine-tuning (SFT) (Chung et al., 2024), RL-based finetuning without a search engine (R1) (Guo et al., 2025) and rejection sampling (Ahn et al., 2024) with a search engine.

模型：Qwen-2.5-3B (Base/Instruct) and Qwen-2.5-7B (Base/Instruct)

搜索源：2018 Wikipedia dump (Karpukhin et al., 2020)

搜索引擎：E5 (Wang et al., 2022)

限制所有搜索文章为3篇，保证公平

性能我们跳过，重点看分析：

（1）PPO和GRPO分析：

1.GRPO更加快 ；2.PPO更加稳定；3.最终reward相同

（2）base和instruct：

instruct模型更快，但最终效果相同；表明虽然后训练有用，但是RL能够弥补差距；

（3）生成长度和搜索有效性：

早期生成长度显著下降，但是奖励缓缓上升；

后期生成长度和奖励都显著上升；（频繁调用llm，受到搜索信息的影响）

有效搜索结果则是逐步上升。

（4）mask的影响：

消融mask会带来显著的性能下降，所以mask掉搜索内容，性能提升更好。
