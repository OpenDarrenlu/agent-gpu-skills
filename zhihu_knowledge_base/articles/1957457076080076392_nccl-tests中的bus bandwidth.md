# nccl-tests中的bus bandwidth

**作者**: 朱然拉航母的小朱

**原文链接**: https://zhuanlan.zhihu.com/p/1957457076080076392

---

近期刚好有一批新架构的芯片交付，每次新机器上架，花时间最多的就是网络调试了，需要服务器、网络和云原生几个团队的共同努力，目标把allreduce的带宽利用率压测到95%+。




每次涉及到新硬件交付，有几个概念很容易被混淆，nccl-tests里的algo bw和bus bw分别代表了什么含义？怎么才算达标？NV硬件白皮书里的带宽是双工还是单工，究竟能跑到多少？下面就一一回答这几个问题。




参考NCCL官方文档，algo bw的计算公式很简单，就是数据量除以耗时

algbw = S/t

algo bw可以衡量一个操作的相对快慢，比如进行了一些网络调优之后，我们可以观察到algo bw的上升。但是我们往往还关心通信的速度是否已经达到了上限，还有没有调优的空间，带宽利用率如何了？algo bw就无法很好回答这些问题了，因为不同的设备数量、不同的通信算法都会影响实际的send/recv的数据量，这时候我们更多参考bus bw这个指标。

对于allreduce来说，bus bw的计算公式如下

B = S/t * (2*(n-1)/n) = algbw * (2*(n-1)/n)

为什么系数是2(n-1)/n呢，大包通信一般是Ring AllReduce算法，被拆分为reducescatter和allgather两个步骤。不论对于reduce scatter还是allgather。每个rank都要进行n-1次数据传输，每次传输（send）的数据量为S/n。所以考虑算法的传输数据量就需要乘以系数 2(n-1)/n。注意这里只考虑了send，也就是bus bw只是单向带宽。

我们在H100芯片上执行nccl test AllReduce时，一般可以跑到450GB/sec的bus bw。但是NV在H100的白皮书上会标注 900GB/sec的NVLink带宽。这个也经常让人引起困惑，实际上文档还会在不起眼的角落注明这是双向带宽，也就是每个GPU可以全双工通信，同时Send/Recv，每个方向都可以有450GB/sec的带宽。

所以至此我们就解答了我们之前的所有困惑

我们做网络压测一般看nccl-tests的bus bw
bus bw结果是网络链路的物理带宽，是单向带宽。跟网卡以及GPU的specification对比的时候，要注意文档标称的是单向还是双向带宽。
通常我们会期待NVLink的带宽利用率可以达到95%以上，RDMA的网卡带宽利用率可以达到90%以上
