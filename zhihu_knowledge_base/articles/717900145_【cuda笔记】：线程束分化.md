问题描述

以如下简单的 cuda kernel 为例， 代码中有 if-else 分支，即线程束分化（一个warp内，不同的线程走if-esle 中不同的path）

#include "cuda_runtime.h"
#define THREAD_PER_BLOCK 256

__global__ void kernel(int* in, int size) {
    int tid = blockIdx.x * blockDim.x + threadIdx.x;
    if(in[tid] > 0){
        in[tid] = in[tid] * 5;
        in[tid] = in[tid] - 8;
    }
    else{
        in[tid] = in[tid] + 8;
    }
}

int main(){
    const int N=32*1024;
    int *d_a;
    cudaMalloc((void **)&d_a,N*sizeof(int));

    dim3 Grid( N/THREAD_PER_BLOCK,1);
    dim3 Block( THREAD_PER_BLOCK,1);

    kernel<<<Grid, Block>>>(d_a, N);
}


逻辑上，warp 中不同的线程走不同的 path，这不符合 SIMT 原则（Single Instruction Multiple Thread），即同一时刻warp内所有的线程执行相同的指令（在新的gpu架构上，每个线程有自己独立的程序计数器，不严格遵守该原则，不过这里还是在该原则上来理解）。

但硬件执行层面依然符合 SIMT 原则， GPU 按照如下的方式来执行： warp 会跑 if-else 两个路径（指令的 issue 以warp 为单位），在每个路径中，不符合 condition的线程会被 mask-out，即它们的执行结果不会写回寄存器。

谓词指令

实际 GPU 中没有生成控制流跳转指令（bra），将上面代码编译，然后使用 cuobjdump 得到该 kernel 对应的 SASS 指令，其中分支部分对应的 SASS 指令如下：

// 分支部分对应 cuda 代码
   if(in[tid] > 0){
        in[tid] = in[tid] * 5;
        in[tid] = in[tid] - 8;
    }
    else{
        in[tid] = in[tid] + 8;
    }

//生成的 SASS 指令如下:

        /*0070*/                   LDG.E R0, [R2.64] ;                                /* 将 in[tid] 加载到寄存器 R0 */
                                                                          
        /*0080*/                   ISETP.GT.AND P0, PT, R0, RZ, PT ;                  /* in[tid] > 0，判断保存在 P0 寄存器中 */
                                                                                
        /*0090*/               @P0 IMAD R5, R0.reuse, 0x5, RZ ;                       /* in[tid] = in[tid] * 5 */
                                                                                      
        /*00a0*/               @!P0 IADD3 R7, R0, 0x8, RZ ;                           /* in[tid] = in[tid] + 8 ，结果保存到 R7 寄存器*/
                                                                                     
        /*00b0*/               @P0 IADD3 R7, R5, -0x8, RZ ;                           /* in[tid] = in[tid] - 8 */
                                                                                     
        /*00c0*/               @P0 STG.E [R2.64], R5 ;                                /* 0x0000000502000986 */
                                                                                     
        /*00d0*/                   STG.E [R2.64], R7 ;                                /* 0x0000000702007986 */
                                                                                    
        /*00e0*/                   EXIT ;                                             /* 0x000000000000794d */
                                                                                      /* 0x000fea0003800000 */

当所有的线程执行到 P0 IMAD R5, R0.reuse,0x5, RZ 指令时， 每个线程会先判断自己的 谓词寄存器 P0，如果为真，就把 R0 寄存器 + 数值5， 然后把结果保存到 R5 寄存器。如果为假，该线程被 mask-out，最后的执行结果不写回寄存器。

这样的话就不需要使用跳转指令了。每个线程会判断谓词寄存器，谓词为 true 的线程才会正真执行，为 false 的会空转（导致资源的浪费）。

总结下谓词指令：

逻辑上的不执行：

当谓词为假时，线程在逻辑上不会执行指令。这意味着这些线程不会对寄存器或内存进行任何修改。

硬件上的执行：

尽管谓词为假的线程在逻辑上不会执行指令，但硬件上仍然会进行计算。这是因为 GPU 的 SIMD 架构要求所有线程在同一时刻执行相同的指令。
具体来说，硬件会进行指令的取指、解码和执行阶段，包括计算操作数和执行算术或逻辑运算。

结果不写回：

在执行阶段，谓词为假的线程的计算结果不会写回目标寄存器或内存。这些结果会被丢弃，不会影响寄存器状态或内存状态。

代码逻辑上每个 thread 只跑某个path， 但在硬件执行层面每个线程依然会执行每个path，只有谓词为真的线程才会正在执行并影响寄存器状态。没有复杂的分支、跳转指令，减少了分支指令的使用，优化了代码性能，提高了分支处理效率，这种谓词机制使得 GPU 在保持 SIMT 执行模型的同时，实现了条件实行的效果。

分支预测？

GPU 没有动态分支预测（这需要很大的硬件成本），但是在一些简单的场景下，从编译生成的 SASS 指令上看，编译器可能会优化分支，比如使用先跑一个分支，再跑一个谓词指令。（实际上两个 path 依然还是跑了，但可能正常的指令比谓词指令效率更高？）

如以下 kernel ：

#define THREAD_PER_BLOCK 32

__global__ void kernel(int* in, int size) {
    int tid = blockIdx.x * blockDim.x + threadIdx.x;
    if(threadIdx.x > 15){
        in[tid] = in[tid] * 5;
    }
    else{
        in[tid] = in[tid] + 8;
    }
}


int main(){
    const int N=32*1024;
    int *d_a;
    cudaMalloc((void **)&d_a,N*sizeof(int));

    dim3 Grid( N/THREAD_PER_BLOCK,1);
    dim3 Block( THREAD_PER_BLOCK,1);

    kernel<<<Grid, Block>>>(d_a, N);
}


THREAD_PER_BLOCK 故意被设置为32，从代码逻辑上看，两条 path 各有一半的线程真正执行。逻辑上会生成如下的 SASS 指令：

@!P0  IADD3 R5, R0, 0x8, RZ 
@P0   IMAD R5, R0, 0x5, RZ


但实际生成的 SASS 指令如下：

可以看出，所有线程都会执行 else 中对应的路径（+5），然后根据谓词寄存器 P0，一半的线程会真正执行 if 中的 path。类似代码会先静态预测下是跑的 else 路径，对于一些线程来说，这个预测不对，然后继续跑 if 路径。

这种优化的收益，应该也是：

IADD3 R5, R0,0x8, RZ

比

@!P0  IADD3 R5, R0,0x8, RZ

效率更高一点吧~
