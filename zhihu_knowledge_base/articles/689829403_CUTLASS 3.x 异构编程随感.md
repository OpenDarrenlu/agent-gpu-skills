借着 GTC 2024 老黄刚发布 Blackwell 架构 GPU 的档口，我也来随便聊聊近代 NvGPU 的异构化对 CUDA SIMT 编程的影响。

NvGPU 异构化最先打破的是 CUDA SIMT 编程范式。近来，我们不难发现社区许多高效的 CUDA 算子是通过 CUTLASS 3.x 实现的。在 NV 的知名的开源软件下搜寻，如 TensorRT-LLM, Megatron-LM, 都不难发现 CUTLASS 3.x 的影子。除此之外，大模型训练几乎必备的 Flash Attention(FMHA) 2.0 也采用了 CUTLASS 3.x 的特性。

无奈 Blackwell 架构的细节太少，今天这篇文章打算借着 Hopper + CUTLASS 3.x 随便聊聊，隔靴搔痒。但是比较确定的是 Blackwell 这个架构对 CUDA SIMT 的破坏肯定更大。

Prologue：序言

在 AGI 的浪潮之下，深度学习模型结构日益向 Transformer 等架构收敛，在语言模型、图像、语音、视频等多模态模型中取得了巨大的成功。GEMM, Convolution 和 FMHA 等算子，在这些模型中的计算占比几乎超过了 90%，扮演着至关重要的角色。

根据官网的定义，CUTLASS 是一个 CUDA C++ 实现高效矩阵乘相关算子的编程模板库。

CUTLASS is a collection of CUDA C++ template abstractions for implementing high-performance matrix-matrix multiplication (GEMM) and related computations at all levels and scales within CUDA.

首先，理解 GPU 访存层级和结构，对优化和实现高效 GEMM 算子来说很关键。

在使用了 TensorCore 的 kernel 实现中，映射到最内层对应的是 mma instruction。Ampere 架构以前，内层的 mma 指令调用的 WMMA 对 CUDA SIMT 编程模型的破坏还没有那么严重。

但是随着，Volta → Ampere → Hopper 几代架构的演变，TensorCore peak performance 越来越高， matrix shape 的颗粒度也越来越大。原先基于 CUDA SIMT 的访存指令带宽已经逐渐无法跟上 mma 指令的效率。除此之外，这么大的 mma 输入和输出都难以完整放在一个 warp 的寄存器中。

Instruction	GPU Architecture	Matrix DType	Max Matrix Shape(MxNxK)	Peak FP16 Dense Tensor Core (TFLOPS)
WMMA	NVIDIA Volta	FP16	8x8x4	125 TFlops/V100
WMMA	NVIDIA Ampere	FP16	16x8x16	312 TFlops/A100
WGMMA	NVIDIA Hopper	FP16	64x256x16	1000 TFlops/H100

部分表格数据引用来源:

https://docs.nvidia.com/cuda/pdf/Ampere_Tuning_Guide.pdf
https://docs.nvidia.com/cuda/parallel-thread-execution/index.html#asynchronous-warpgroup-level-matrix-multiply-accumulate-instructions

相比 Ampere 及之前的架构，Hopper 架构主要的新特性包括:

New thread block cluster feature enables programmatic control of locality at a granularity larger than a single thread block on a single SM.
Distributed shared memory allows direct SM-to-SM communications for loads, stores, and atomics across multiple SM shared memory blocks.
New asynchronous execution features include a new Tensor Memory Accelerator (TMA) unit that can transfer large blocks of data efficiently between global memory and shared memory.
4th generation TensorCores, On a per SM basis, the Tensor Cores deliver 2x the MMA (Matrix Multiply-Accumulate) computational rates of the A100 SM on equivalent data types, and 4x the rate of A100 using the new FP8 data type, compared to previous generation 16-bit floating point options.

可以说这是一组套件，意味着 Hopper 之后访存结构、访存指令、mma 指令携手一起有了很大的改变。其中，TMA 和 MMA(TensorScores) 均为 SIMT 的异构指令。

MainLoop：CUTLASS 3.x + Hopper 优化技巧

为了适配 Hopper 架构之后加速器的异构化，CUTLASS 3.x 进行了大范围的重构，引入了新的 GEMM 编程模型。其中最主要的模块包括 CuTe， TMA， Warp-Specialization，Software Pipeline 1, 2。那么借助，CUTLASS 3.x 在 Hopper 架构上实现高效 GEMM 算子的常用优化技巧有哪些？

使用 CuTe 进行 Tiling

CuTe Layout 可以说是 CUTLASS 项目中最烧脑的部分，也是目前最科学严谨的模块

Cute Layout Algebra can help users do things like partition layouts of data over layouts of threads.

CUTLASS 3.x 中，CuTe Layout Algebra 是实现 GEMM 数据如何在 thread block、warp、thread 上面进行分块、分片；是分片之间，分片内部数据如何排布(如何 SMEM 的 swizzle) 的基础模块。

使用 TMA 和 WGMMA 的 Hopper 架构指令

Hopper 架构上面，WGMMA 可以提供数倍于 A100 WMMA 的性能，同时要求的 matrix shape 也大了很多。为了适配这种性能，新的 CUDA 允许在一个 thread 中异步调用 TMA loads 到 share memory，同时其他 threads 通过 cuda::barrier 等待数据加载完毕。

实际上，在此之前 GPU CUDA 上面实现一个高效的算子，大部分的时间需要花费在如何精细地计算和安排数据加载和存储。TMA 的首要的目标是提供 Global memory 到 Shared memory 高效的多维数组搬运，同时还提供按照 swizzle 方式存储数据到 share memory 的能力。

软件流水和Warp-Specialization

我们已经介绍了 GPU 上面存在的两个异构硬件 TMA 和 TensorCore。Pipeline 的引入基本上宣告了 SIMT 编程的让位。

截图来自 CUTLASS: The diagram illustrates the efficient, pipelined mainloop body used in GEMMs

如上图，深绿和浅绿部分是 TMA 加载指令，深蓝和浅蓝部分是 Math 指令。图例中通过深色和浅色两级的数据流水使得访存延时得到充分的掩盖。

借助软件流水，不同的 warp group 在执行中起到了不同的作用；有部分充当生产者，有部分充当消费者。这种将 warp group 划分不同工作区的方式称作 Warp-Specialization。有趣的是，这个词并不是第一次出现，早在 2011 年就有一篇来自 NVIDIA 的 论文 CudaDMA 提到了这个名词：

https://research.nvidia.com/publication/2011-11_cudadma-optimizing-gpu-memory-bandwidth-warp-specialization.

Persistent Warp-Specialized Kernels

Persistent kernel 是 CUDA 当中用于避免 kernel launch 和初始化开销的常见技巧。在 CUTLASS 中，通过persistent kernel 和 Warp-Specialization 结合使一个 warp group 在kernel 结束前处理多个 output tiles。特别是在使用了 pipeline 的情况下，这种优化技巧可以显著减少额外 kernel launch 带来的开销。而且还可以暴露更多利用局部性的优化空间。

Epilogue：尾声

当然，熟悉 CUTLASS 3.x 的朋友们肯定知道还有许多细节和技巧。但这不是今天我想聊的主题。CUTLASS 是 NVIDIA 开源算子库里面的翘楚。更重要的是，它本身是一个模板库，其对 NV 硬件算子开发流程的抽象和逻辑，对照和反应了 NVIDIA 硬件发展的趋势。

结合 CUTLASS 3.x 和 Hopper 的示例，以及 CUTLASS 及 GPU 架构本身的演化，我们可以获得一些对未来发展的初步了解。通过窥探这些示例和演化过程，我们可以洞察到 GPU 领域的一些趋势和技术创新，并将能够更加精准地把握GPU编程的脉搏。
