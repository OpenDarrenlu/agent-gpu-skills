概述
为什么要做MoE预取：

在GPU的memory不够用的场景下，放不下所有的专家权重。比如边缘计算，或者3090等消费级显卡。（那么孪生的问题就是如何做卸载，预取来的权重要保留多久。这就可以认为GPU-HBM是高速缓存，CPU-DRAM是低速缓存，可以参考已有cache的工作）（所以从系统角度就存在trade-off：缓存的memory大小-整体的latency，当然这里面还包含了缓存什么内容，维持多久，通算重叠等一系列系统优化问题。从算法角度就存在，预测正确性问题，考虑更多信息，使用更好的方法就更准，但是可能预测就慢）

2. MoE预取的trade-off是什么：

预测开销-准确性




FineMoE（Eurosys26）

Taming Latency-Memory Trade-Off in MoE-Based LLM Serving via Fine-Grained Expert Offloading

已有方法是请求级做预测（coarsed grained）。比如MoE-infinity。本文是layer级做预测（fine-grained）。预测准确性如下图所示

上图是system design。我们收集历史信息，形成expert map。然后每次对此map进行搜索。以此决定如何预取和卸载。场景是prefill和decode（好像没有区分开来讨论啊，这俩的特点应该是不一样的）

这是design的细节。在最开始几层，我们直接利用embedding信息，去历史数据库里寻找。在后续层里我们基于预测概率信息，去历史数据库里寻找。

讨论：

本文自己承认的问题：没有用大模型来做分析，而是用公式做分析。没有针对prefill/decoding分别讨论。
没有讨论自己的trade-off啊。比如预测开销（时间开销，计算资源开销等）和准确性的关系。那要是开销允许无限大的话，我能不能引入几百B的大模型来做每一步的预测？准确性是不是爆表了。
实验是6个3090
Fate（WWW26）

Fast Edge Inference of Mixture-of-Experts Models via Cross-Layer Gate

最主要的创新就是用相邻层的hidden state差不多这一特性，直接用这一层的hidden来预测下一层的routing（注意routing本身也有权重的）===trade-off是：距离越近，越准确，但是留给调度的时间窗口越小。这里写的很烂，不应该强行选择上一层，应该作为可选参数去调。其实信息也可以收集更多，比如前面连续好几层？之类的

其次是，区分了decoding和prefill。prefill更看重所有token都最喜欢的专家是谁，毕竟token多。decoding更看重单个token马上要算哪些专家，毕竟就一个token。所以预测器结构稍有不同。（但是没有讲的很清楚。。。）

此外还考虑了专家缓存（多放一点专家，就算预测没那么准也留有余地）（这里也没做trade-off，应该是内存约束-预测准确性的权衡，这里居然是想着越多越好，然后随便选了一个3层，见下图）；通算重叠。

实测到底需要预取多少层效果才好

CommitMoE(AAAI26)

Efficient Fallback-Free MoE Inference with Offloading Under GPU Memory Constraints

有点绷不住啊。。拿4090跑MoE，是要打入PC市场吗，那只有加密需求内部部署才用得到吧，但是为什么不搞个小集群呢。感觉市场好小啊，不是很有趣啊。。。

commitmoe是有损的方法！意思是，如果我们预测错了再回去cpu重新搬运的开销太大了。如果说，预测的置信度比较低，那我们将错就错也行。举例：

我们先用预测器做预测，搬运-》真的开始计算了，结果我们错误的搬来了expert2（置信度0.2），事实上应该是expert1（置信度0.2）-》这俩置信度都低，那将错就错吧，凑合拿expert2算也行。

相比于b图，本文的c图消掉了因为预测错误而导致的重读取的开销
