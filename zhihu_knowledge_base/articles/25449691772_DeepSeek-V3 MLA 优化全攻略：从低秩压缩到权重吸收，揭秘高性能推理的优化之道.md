MLA的关键优化思路

回顾一下MLA，本质上是对MHA标准Attention计算的扩展，通过KV进行低秩压缩，减少推理过程的KV cache；通过旋转位置编码的解耦，也就是对Q和K进行拆分，一部分维度做压缩，一部分维度做Rope位置编码，实现权重吸收。具体理解有三方面的优化：

低秩分解和KV压缩：不再缓存解压缩的Key/Value状态，而仅缓存低秩Key-Value联合压缩以及Key的解耦RoPE部分。，也就是compressed_kv。
权重吸收：采用权重吸收技术，通过改变计算Query和Output矩阵乘法顺序，不仅节省了中间张量的内存消耗，还减少了浮点运算的次数。
Move Elision：分别计算RoPE部分和非RoPE部分的注意力分数，然后将它们相加。原始实现将Query/Key向量的两部分Concat起来，但在缓存压缩Key/Value状态时由于不必要的数据广播和内存往返而被证明效率较低，分别实现完矩阵乘再相加有着更高的效率。

低秩投影与KV压缩：高维到低维的革命 （CacheDecompressed, CD -> CacheCompressed, CC)
1. 低秩投影的目的和意义：

在存储空间和计算量上都大幅度降低，对于q_proj，降低了存储权重的显存和计算量；对于kv而言，实现KV cache的极大压缩（CacheCompressed, CC）。

1）首先，分析一下q_proj的低秩分解带来的优势：

存储空间角度：

输入 shape：[bs, seqlen, hidden_states]

不做低秩分解存储量：Wq = [hidden_size, num_heads * q_head_dim] = 7168 * 24576
分解后存储：q_a_proj：[hidden_size, q_lora_rank]；q_b_proj：[q_lora_rank, num_heads * q_head_dim]
分解后总存储量：1536 *（7168 + 24576）
比较：分解后 1536 *（7168 + 24576）< 分解前 7168 * 24576

计算量角度：

分解前总计算量：input @ Wq = 2 *bs * seqlen *hidden_size * num_heads * q_head_dim
分解后：
Part 1: input @ q_a_proj = 2 * bs * seqlen * hidden_size * q_lora_rank
Part 2: q_down @q_b_proj = 2 * bs * seqlen * q_lora_rank * numheads * qheaddim
分解后总计算量：2 * bs * seqlen * numheads * q_lora_rank * (hidden_size + numheads * qheaddim)
比较：约掉公共因子，分解后 1536 *（7168 + 24576）< 分解前 7168 * 24576

2）KV cache的压缩（CacheCompressed, CC)

然后，回到重点的KV cache压缩上面来，核心是说KV cache无压缩的版本 CD（CacheDecompressed）到压缩KV cache CC（CacheCompressed）的版本实现。

CD将 Latent KV Cache 解压缩后，就是MHA。CD直接缓存单头的 Latent KV Cache 以及 K 的 RoPE 部分，极大地发挥了 MLA 的优势，直接减少KV cache \frac{2\ hidden\_size\ +\ num\_heads \times qk\_rope\_head\_dim}{kv\_lora\_rank\ +\ qk\_rope\_head\_dim} 倍，有效提升batchsize，降低推理成本。

2. 压缩的实现方案：

代码参考Zhang Mingxing 老师的实现：

https://github.com/madsys-dev/deepseekv2-profile/blob/main/mla/impl/cache_compressed.py

def compress_kv(self, hidden_states_kv: torch.Tensor, kv_position_ids: torch.LongTensor) -> torch.Tensor:
    # return the RoPE'ed & compressed kv
    bsz, kv_seq_len, _ = hidden_states_kv.size()
    compressed_kv = self.kv_a_proj_with_mqa(hidden_states_kv) 
    compressed_kv, k_pe = torch.split(
        compressed_kv, [self.kv_lora_rank, self.qk_rope_head_dim], dim=-1
    )
    compressed_kv = self.kv_a_layernorm(compressed_kv)
    k_pe = k_pe.view(bsz, kv_seq_len, 1, self.qk_rope_head_dim).transpose(1, 2)
    cos, sin = self.rotary_emb(k_pe) 
    k_pe = apply_rotary_pos_emb(k_pe, cos, sin, kv_position_ids).view(bsz, kv_seq_len, self.qk_rope_head_dim)
    return torch.cat([compressed_kv, k_pe],dim=-1)

压缩的本质理解：

多个head抽取出一个公共的信息，所有的head共用一个compressed_KV，num_kv_heads变成了1，达到节省KV cache的目的
对于差异性的内容，在推理时进行计算，q部分也有多个head，可以把每个k head上的独特信息转移到对应的q head上，也就是Wq, Wk的吸收合并（下一个Part重点介绍）

权重吸收：显存与计算强度的双赢（CacheCompressed, CC -> Absorbed_CacheCompressed, A_CC)
1. 为什么要做权重吸收？

从显存看：与不做吸收的CD比，极大节省KV cache的存储

从计算看：和CC比，权重吸收A_CC从MHA的计算变成MQA的计算，有效提升了计算密度，这样在decode阶段，就可以从memory bound变成compute bound，计算强度正比于head数（128），充分发挥算力

2. 如何做权重吸收？

将 kv_b_proj 中属于 K 的部分权重（论文中对应W^{UK}）吸收进 q_nope（论文中对应q^c，Runtime阶段吸收）；将代码中 kv_b_proj 中属于 V 的部分权重（论文中对应W^{UV}）吸收进 attn_out。

1）W^{UK} 的吸收

吸收前：attn\_weights = (W^{UQ}c_{t}^{Q})^{T} * (W^{UK}c_{t}^{KV})，括号表示运算顺序，即先各自算2个括号内的，再做 * 计算；

吸收后：attn\_weights = ((W^{UQ}c_{t}^{Q})^{T} W^{UK})c_{t}^{KV}

也就是说，分成3个step，step1是q的down proj， (W^{UQ}c_{t}^{Q})^{T} 计算得到q_nope，step 2是q_nope被q_absorb吸收，也就是 (W^{UQ}c_{t}^{Q})^{T} W^{UK} ，step 3就是attention weights的计算，和压缩后的KV cache c_{t}^{KV} 相乘。

2）W^{UV} 的吸收

吸收前：对于W^{UV}，先计算 v= W^{UV}c_t^{KV}

输出结果 o = (attn\_weights)vW_o=attn\_weights(W^{UV}c_t^{KV})W_o

吸收后，W^{UK}的吸收过程类似，利用结合律改变计算顺序，那么有：

o = （attn\_weightsc_t^{KV})W^{UV}W_o

对应的代码实现分成3个step，step 1是 attn\_weightsc_t^{KV} ，step 2是 attn\_weightsc_t^{KV}W^{UV} ，step 3是 attn\_weightsc_t^{KV}W^{UV}W_o

参考：https://github.com/madsys-dev/deepseekv2-profile/blob/main/mla/impl/absorbed_cache_compressed.py

q = self.q_b_proj(self.q_a_layernorm(self.q_a_proj(hidden_states_q)))
q = q.view(bsz, q_len, self.num_heads, self.q_head_dim).transpose(1, 2)

# Q切分维rope和nope部分
q_nope, q_pe = torch.split(
    q, [self.qk_nope_head_dim, self.qk_rope_head_dim], dim=-1
)
# KC cache压缩后只需要存储compressed_kv和k_pe，compressed_kv shape为[bsz, kv_len, kv_lora_rank]

    compressed_kv, [self.kv_lora_rank, self.qk_rope_head_dim], dim=-1
)
k_pe = k_pe.view(bsz, kv_seq_len, 1, self.qk_rope_head_dim).transpose(1, 2)

# 从kv_b_proj中分离出Wuk和Wuv两份，分别在不同的地方吸收
kv_b_proj = self.kv_b_proj.weight.view(self.num_heads, -1, self.kv_lora_rank)

# q_absorb的shape为[num_heads, qk_nope_head_dim, kv_lora_rank] = [128, 128, 512]
# out_absorb shape:[num_heads, v_head_dim, kv_lora_rank] = [128, 128, 512]
q_absorb = kv_b_proj[:, :self.qk_nope_head_dim,:]
out_absorb = kv_b_proj[:, self.qk_nope_head_dim:, :]

cos, sin = self.rotary_emb(q_pe)
q_pe = apply_rotary_pos_emb(q_pe, cos, sin, q_position_ids)

qk_head_dim = self.kv_lora_rank + self.qk_rope_head_dim
query_states = k_pe.new_empty(bsz, self.num_heads, q_len, qk_head_dim)
# q_nope被q_absorb吸收
query_states[:, :, :, : self.kv_lora_rank] = torch.einsum('hdc,bhid->bhic', q_absorb, q_nope)
query_states[:, :, :, self.kv_lora_rank :] = q_pe

key_states = k_pe.new_empty(bsz, self.num_heads, kv_seq_len, qk_head_dim)
key_states[:, :, :, : self.kv_lora_rank] = compressed_kv.unsqueeze(1)
key_states[:, :, :, self.kv_lora_rank :] = k_pe

attn_weights = torch.matmul(query_states, key_states.transpose(2, 3)) * self.softmax_scale

if attn_weights.size() != (bsz, self.num_heads, q_len, kv_seq_len):
    raise ValueError(
        f"Attention weights should be of size {(bsz, self.num_heads, q_len, kv_seq_len)}, but is"
        f" {attn_weights.size()}"
    )

# upcast attention to fp32
attn_weights = nn.functional.softmax(
    attn_weights, dim=-1, dtype=torch.float32
).to(q_nope.dtype)
# step1: attn_weights @ compressed_kv
attn_output = torch.einsum('bhql,blc->bhqc', attn_weights, compressed_kv)
# step2: attn_weights @ compressed_kv @ Wuv
attn_output = torch.einsum('bhqc,hdc->bhqd', attn_output, out_absorb)

if attn_output.size() != (bsz, self.num_heads, q_len, self.v_head_dim):
    raise ValueError(
        f"`attn_output` should be of size {(bsz, self.num_heads, q_len, self.v_head_dim)}, but is"
        f" {attn_output.size()}"
    )

attn_output = attn_output.transpose(1, 2).contiguous()
attn_output = attn_output.reshape(bsz, q_len, self.num_heads * self.v_head_dim)
# step3: attn_weights @ compressed_kv @ Wuv @ Wo
attn_output = self.o_proj(attn_output)
3. 显存是否要提前吸收？

Part 1: W^{UK} 的吸收

Wuq权重不提前吸收与提前吸收对比

a. 不吸收场景：attention_weights的计算：attn\_weights = (W^{UQ}c_{t}^{Q})^{T} * (W^{UK}c_{t}^{KV})，括号表示运算顺序，即先各自算2个括号内的，再做 * 计算；

b. 权重吸收场景：attn\_weights = ((W^{UQ}c_{t}^{Q})^{T} W^{UK})c_{t}^{KV}，不考虑乘压缩后KV cache也就是 c_{t}^{KV} 的话，这里就是两个步骤，先计算 (W^{UQ}c_{t}^{Q})^{T} 的结果，然后再计算该中间结果与 W^{UK} 相乘的q_nope结果。

此时，runtime的计算量：2 * bsz * num_head * q_len * q_lora_rank * nope + 2 * bsz * num_head * q_len * nope * kv_lora_rank。

c. 提前吸收场景：什么是提前吸收呢？就是说可以事先把Q 和K up proj的权重乘结果计算出来，也就是 W^{UQ} W^{UK} ，这样对应了：attn\_weights = (c_{t}^{Q})^{T}(W^{UQ} W^{UK})c_{t}^{KV}

也就是说（不考虑乘压缩后KV cache也就是 c_{t}^{KV} ），第一步先把 W^{UQ} W^{UK} 的结果计算出来，第二步再和 c_{t}^{Q} 相乘得到q_nope的结果，runtime的计算就只有第二步了。

runtime计算量：2 * bsz * q_len * q_lora_rank * kv_lora_rank

二者比值 =不提前吸收/提前吸收= \frac{128*1536+128*512}{1536 * 512} =33.33%

所以提前吸收反而增加了计算量~

Part 2: W^{UV} 的吸收

对于W^{UV}，先计算 v= W^{UV}c_t^{KV}

a. 不吸收场景： o = (attn\_weights)vW_o=attn\_weightsW^{UV}c_t^{KV}W_o

b. 吸收场景： o = (attn\_weightsc_t^{KV})W^{UV}W_o

对应的代码实现分成3个step，第一步是 attn\_weightsc_t^{KV} ，第二步是 attn\_weightsc_t^{KV}W^{UV} ，第三步是 attn\_weightsc_t^{KV}W^{UV}W_o

runtime的计算量：2 * bsz * num_head * q_len * nope * kv_lora_rank + 2 * bsz * num_head * q_len * nope * hidden_size

c. 提前吸收场景： o = （attn\_weightsc_t^{KV})(W^{UV}W_o) ，这样第一步是 attn\_weightsc_t^{KV} ，第二步是 W^{UV}W_o ，第三步是 attn\_weightsc_t^{KV}(W^{UV}W_o)

runtime的计算量：2 * bsz * num_head * q_len * kv_lora_rank * hidden_size

二者比值=不提前吸收/提前吸收= \frac{128*512+128*7168}{512 * 7168} =26.79%

结论：

不应该做提前吸收，提前吸收会增加额外的算力需求，也就是说：相比用这些特别大的低秩矩阵做投影，明显不如按照低秩分解形式依次相乘来得划算。

5. 权重吸收带来的计算变化：MHA -> MQA

为什么变成了MQA？

把KV的linear拆分成了两部分，也就是W^{UK}和W^{UV}，一个和q的linear结果也就是q_nope相乘，一个和attn_weights@v 结果相乘；
compressed_kv shape为[bsz, kv_len, kv_lora_rank]，这里kv_head_num变成了1，相当于从不同的kv_head中抽取出了一份共有的信息，用这个共有的信息做Attention计算；而对于相异的信息，通过矩阵吸收都转移到了对应的q_head上
q_head_num > 1，kv_head_num=1，所以用MQA来计算

相应的，MQA decoding Kernel优化策略也很清晰了：

一个KV head被多个query head共享
为了减少KV cache的访存，可以通过一个triton block处理多个query head来优化，这样就不用重复读取相同的KV cache了，所以要把KV cache放在L2 cache里

拆分计算：分而治之，降低显存 （Absorbed_CacheCompressed, A_CC -> Absorbed_CacheCompressed_MoveElision, A_CC_ME）

从显存来看：减少显存开销，从计算量来看，A_CC和A_CC_ME一致

在A_CC的基础上，在计算attn_weights的时候，把nope和rope部分拆开算，然后再求和。这样做是为了避开无用的数据拷贝和广播

A_CC为了做数据拼接，是先初始化一个一个拼接好的空张量，再往里塞数据，这样就是2倍的显存开销。

query_states[:, :, :, self.kv_lora_rank :] = q_pe

key_states = k_pe.new_empty(bsz, self.num_heads, kv_seq_len, qk_head_dim)
key_states[:, :, :, : self.kv_lora_rank] = compressed_kv.unsqueeze(1)
key_states[:, :, :, self.kv_lora_rank :] = k_pe

主要问题：

compressed_kv被压缩成单头的形式了，不区分heads，shape为[bs, kv_len, kv_lora_rank + rope]

如果TP的话，KV cache无法在number head维度切分，一个TP group内的每个TP rank都共享相同的compressed_kv。

1. Baseline：

除了compressed_kv，其余部分都采用TP切分，这里需要注意的事如果Q的低秩投影q_q_proj也做切分的话，后续因为有layernorm，所以在归一化之前需要多一次Allreduce
如果想不引入这次通信，那么可以在不同卡上复制q_a_proj的权重，也就是vLLM 和SGLang TP现在的做法，这样就要重复存储compressed_kv了。

2.改进点1：改进重复存储的compressed_kv的问题：

方案1：通过类似KV cache memory pool的方式，只维护一份compress_kv，然后每个tp_rank都从上面去取，就能避免数据的重复存储了

方案2：对Attention部分引入DP，这样每张卡上只有部分数据，存储的KV cache会相应减小，也不会重复存储了。但是由于Attention还是需要全部的token，这里又对其他的linear操作做了TP，所以需要引入两次AlltoAll通信，详细可见分析：pika-jy：DeepSeek-V2 MLA KV Cache 真的省了吗？（2）

方案3：直接用DP MLA，不用TP，这是当前SGLang DP MLA的做法。

具体可以参考BBuf：SGLang DP MLA 特性解读

3.改进点2：改进DP引入的负载不均衡的问题

DP虽然解决了重复存储的问题，但是每张卡上的token 数量不同，还是会引入负载不均衡的问题，所以本质上是解决KV cache的均衡存储的问题。

方案1：Attention部分吧KV cache分块在不同的卡上存储

在显存角度：解决了DP场景下不同卡上不同batch token数差距大，负载不均衡的问题

通信角度：对某个sequence的query而言，计算完当前卡的KV cache QK计算结果之后，需要收集其他卡上的结果，如果分布在更多的卡上，通信开销也更大，为了避免这样的问题，需要事先reorder，尽可能一个sequence的token的kv cache在临近的数量少的卡上

相当于分布式版本的flash decoding，每张卡计算完当前的KV cache的结果，要拿到softmax_lse，从而不同的卡要做allreduce_max来更新各自的结果，最后做allgather收集完整的attention计算结果

方案2：以通信换显存

每张卡上维护KV latent的切片，通过allgather全部到每张卡上

这里allgather KV latent的过程可以和计算overlap执行

缺点：损失一些带宽做数据搬运，从HBM到片上
