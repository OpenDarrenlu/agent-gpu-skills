# toasty(asplos26): CPU网卡调度优化（软件）

**作者**: Arsmart​上海交通大学 计算机科学技术博士在读

**原文链接**: https://zhuanlan.zhihu.com/p/2013999465883775728

---

先说清楚：这篇论文研究的是 CPU 服务器上的网络 I/O，不是 GPU。它讨论的是：网卡收到的数据，写到LLC，缓冲区多大，缓存策略如何。

1. 基本拓扑
网络
  │
  ▼
┌──────────────┐
│   NIC / 网卡 │
└──────────────┘
        │  DMA
        ▼
┌───────────────────────────────┐
│             CPU               │
│   ┌───────────────────────┐   │
│   │ LLC / L3 Cache        │   │
│   │ (DDIO 可直接写这里)   │   │
│   └───────────────────────┘   │
└───────────────────────────────┘
        │
        ▼
┌──────────────┐
│ 主存 / DRAM  │
└──────────────┘

现代高性能收包框架会预先准备很多包缓冲区。网卡收到包后，直接 DMA 到这些缓冲区里，CPU 再去处理。Intel 的 DDIO 允许网卡把数据先写进 LLC，而不是直接写主存，这样 CPU 随后处理包时更容易命中缓存，速度更快。

2. 问题出在哪

DDIO 的前提是：这些刚收到的数据，CPU 很快就会处理，而且在这之前能留在 LLC 里。

但如果缓冲区工作集太大，数据虽然先写进了 LLC，却很快又被挤到主存；等 CPU 真要处理时，还是得再从主存读回来。这样一来，反而比原本直接写主存的 baseline 更差，因为多绕了一圈 LLC。

3. trade-off

论文发现：平稳流量下，小缓冲区往往更快；但流量一旦突然升高，就更容易丢包，因为网卡一时找不到足够空缓冲区去接新包，总结：

缓冲区小：缓存友好，但容易在突发流量下掉包
缓冲区大：更稳，但缓存效果变差
4. Toasty 做了什么

Toasty 的思路是：

平时只循环一小批热缓冲区；流量冲高时，再临时扩大活跃缓冲区。

它做了两件事。

第一，原来的做法更像 FIFO，刚刚处理完、仍然很热的缓冲区，不会马上再被拿来用。Toasty 改成更接近 LIFO 的方式，让最近刚回收的缓冲区更快再次被网卡使用。这里重点不是说 LIFO 理论最优，而是：原来的 FIFO 显然不够聪明，而软件上又不方便做太复杂的缓存策略，所以一个简单的 LIFO 就已经明显更好了。

第二，它不会去动态重建总缓冲区，而是动态控制当前真正放在接收路径上、随时可被网卡使用的缓冲区数量。它看的只是一些低开销信号：这一轮收了多少包、回收了多少缓冲区、现在还剩多少可用缓冲区。 于是它就能做到：

平稳时，活跃工作集尽量小，尽量循环热缓冲区
流量波动时，临时拉进更多缓冲区，避免掉包
流量回落后，再缩回去
5. 两个启发

第一，某些 GPU workload 可能不该让所有 SM 都工作。 如果一个 workload 明显是 memory-bound、L2-bound 的，比如图计算、稀疏访问、pointer chasing，那瓶颈可能不在算力，而在 L2 和内存系统。这时所有 SM 一起上，不一定更快；反而可能让 L2 更乱。适当留出一部分 SM，让 L2 工作在更高效的点，整体吞吐可能更好。

第二，可以考虑做 GPU 版 DDIO。 Toasty 能成立，一个前提是网卡可以把数据先打进 CPU 的 LLC。那 GPU 上也可以想：能不能让远端/NVLink/NIC 进来的数据，优先进入一个受控的 L2 接收区，而不是只能先写 HBM。这样，马上就要被消费的数据就不必先落 HBM、再被 SM 读回来。
