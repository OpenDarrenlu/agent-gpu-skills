最近在看 Canon 这篇论文。先说明一下：下面很多地方不一定是论文有问题，也可能是我自己没看明白，或者作者有更深的考虑。 这里只是把我读下来最困惑的点简单记一下。

论文大意

按我的理解，Canon 想做的是把两类架构的优点结合起来：

类 TPU / 脉动阵列：静态数据流下效率高，但对动态 / 不规则数据流不够灵活；
类 GPU / CGRA：更能支持复杂和动态的数据流，但控制、调度、数据搬运开销更大。

所以它的核心思路大概是：

静态部分尽量静态排布，动态部分交给调度器在运行时调度PE。

几个主要疑问
1. 动机表述我没有完全看顺

论文把“DSA 不擅长不规则数据”和“可编程架构处理不规则数据也低效”放在一起讲。
我自己读下来会觉得逻辑有点绕。对我来说，更自然的说法是：

一类擅长静态高效；
一类擅长动态灵活；
Canon 想在两者之间折中。

也许是我理解不到位，但我觉得这样更清楚。

2. time-lapsed SIMD 我没有完全看懂

论文说指令会延后几个周期传到后面的 PE。
我猜它想表达的是一种空间流水 / 错时流水，而不是简单“空等”，但我还是没完全弄明白：

是传给一整行 / 一整列的所有 PE，还是可以细粒度选择？
面对不规则数据时，不需要计算的 PE 是不是只能 NOP / 空转？
动态数据流怎么和这种固定传播节拍配合？
3. 两块本地内存的理由不够让我信服

论文把每个 PE 的本地存储分成 data memory （给静态数据）和 scratchpad（给动态数据）。
但我自己会自然想到：为什么不做成统一存储，再由软件 / 编译器动态划分静态区和动态区？

所以我没太看明白的是：

分成两块到底是体系结构上的必要性，还是更偏实现上的妥协？
面对动态/静态任务占主导的情况，统一存储可以软件划分给静态和动态的内存尺寸，是不是更灵活？
4. 静态与动态数据流如何协同，论文没有讲到我想看到的程度

这是我最关心的一点。
论文提出“静态编排 + 动态调度”，但我一直没看到特别扎实的分析来说明：

二者如何协同；
共享哪些资源；
会有什么冲突；
冲突怎么缓解；
当动态数据流很重时，会不会反过来拖累静态部分。

也就是说，我能理解它“想配合”，但没有完全看清楚“配合的代价”。

5. FSM 这个方向并不陌生，baseline 也让我有点疑问

轻量级状态机 / 本地控制器这种思路，在 spatial architecture、CGRA、数据流机里应该不是完全全新的。
所以我会想问：

有没有更接近的已有设计可以作为 baseline？
论文目前的 baseline 是否足够说明 Canon 的独特性？
6. 我觉得它其实有往 wafer-scale 方向延展的潜力


Canon 这种思路其实很适合继续往 wafer-scale chip / 超大规模空间阵列 去想，因为阵列一大，问题就会变成：

动态数据流到底该走哪个 PE；
不同 PE 之间怎么配合；
路由、负载均衡、局部拥塞、跨区域通信怎么处理；
静态映射和动态调度怎么在更大规模上协同。
