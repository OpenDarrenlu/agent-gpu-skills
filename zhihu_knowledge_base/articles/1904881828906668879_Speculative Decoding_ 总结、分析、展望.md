0. Background

一年多来，Spec这个领域涌现了上百篇文章，记得ICLR 2025的keyword前十就有Spec，以及ICLR 2025的best paper也颁给了一篇Spec文章。随着LLM能力提升的边际效应逐渐明显以及reasoning model的普及，大家对inference acceleration也投入了更多的关注。Speculative Decoding (Spec) 是一个应用于LLM inference的推理加速技术，主要利用decoding阶段的memory-bound特性，利用冗余的算力来解码额外的token，具体介绍这里不详细展开。感兴趣的可以了解一些Spec的博客，当然最推荐的还是直接阅读Google ICML 2023的这篇论文 Fast Inference from Transformers via Speculative Decoding.

个人认为这个领域到目前为止已经是半红海状态了，Eagle-3已经把纯文本加速的税收到一百年后了 (笑) ，但用于应用的spec仍然是一篇蓝海。ICLR 2025里面，7篇application spec中了6篇，spec神教无敌。

1. spec 分类

为了提供一个更清晰的视角，这里简单的对Spec进行一次分类。spec领域最主要的topic就是draft model了，这方面的终极目标是得到一个又快又好的draft model，当然这是不现实的，围绕efficiency和accuracy的trade off，大体上 spec工作可分为两类：

parametric draft model
non-parametric draft model

为什么是这么分类呢？因为它们本质上是不同级别的spec。基于无参数的draft model的spec是更底层的spec: 因为它彻底解决了auto-regressive token-by-token的困境；而有参数的draft model的spec是把这个困境从target model转移给了draft model。

2. spec 进展总结

基于这两个类别，我们可以提出以下问题，分别对应一年来这个领域的进展：

draft model如何选择？
training from scratch: 即自己从头训一个draft model，优势是可以复用target model的hidden states / kv cache，劣势是需要很大的训练代价，包括但不限于计算资源、模型调参等等。
existing model: 即使用一个现有的、更小规模的模型来作为draft model，例如使用qwen 0.6B模型作为draft model来给32B模型加速。优势是开箱即用，劣势是很多模型不存在这么一个更小规模的模型。




verification能否改进？
tree attention: 即对于每个位置保留不止一个draft token，通过自定义attention mask来并行验证。优势是可以增加接受率，劣势是大幅增加了draft token的数量，在bs>1时效果会显著下降
theoretical perspective: 即原始版本的speculative sampling虽然能保证无损，但较为严格，有一些token本可以接受。优势是可以plug-and-play的提高接受率，劣势是提高幅度比较低，且门槛太高了。
lossy verification: 即很多时候spec的verification不需要那么严格，以及很多时候draft token可能是高质量，但仍被target model拒绝的；这种情况下就可以研究新的verification机制来实现更高的加速比。优势是加速比更高，劣势是无法保证无损的性质。

3. analysis 1: draft model如何继续提升

首先讨论一下我心中的理想的draft model是什么样的。这里就简单说一下一些个人的分析与见解。通过以下结论、分析与思考，逐层递进。

draft model的预测能力与其参数量应当存在一个scaling law。

尽管eagle-3的draft model已经取得了一个很吓人的预测能力，在某些任务上能达到>7的MAT，但其预测能力仍然是有限的。简单在spec-bench上测了一下eagle-3链式的极限预测能力（即不使用tree-attention, 把depth拉的很大）如下图：

eagle-3 draft model MAT (Vicuna-13B) on Spec-Bench

可以发现，eagle-3的MAT仍然主要聚集于8以下，也就是说，即使我们把depth设置的很大，eagle-3的MAT也不会显著的高很多；但像我之前在PEARL中测的，deepseek 1.3B作为draft model，33B作为target model时，极限预测能力下，MAT能拉到30。当然这个对比很不公平，不同的模型，不同的benchmark。但这个结论目测是站得住脚的：即draft model的预测能力与其模型规模强相关。

基于这个结论，draft model多层化是一个值得研究的问题。（注意到有一篇研究多层draft model的工作KOALA，但好像还没有中稿，而且做的有待完善）另外，也可以考虑把draft model做的更宽，也可以有效的scaling draft model的参数量。目前阶段好像draft model的尺寸都是和target model单层的尺寸一致，我觉得完全可以考虑一个宽而浅的draft model。




长距衰减问题是draft model进一步发展的主要问题。

无论是从spec的原始论文，还是从eagle-3的MAT图，还是从直觉上来说，离当前token越远的token都是更难预测的，而draft model想取得更好的效果，长距衰减问题会越来越显著。就像评论区大佬说的，长距衰减也可以归纳为一个数学问题，即一个等比数列的收敛问题。对于解决这个问题，归根结底还是要提高draft model的接受率（等比数列的公比）。我个人觉得缓解长距衰减问题有两个方向：

复用的特征从feature层转移到kv cache层。
增加draft model的参数量。

对于1的解释：feature是kv cache聚合之后的结果，而聚合就会有信息损失。从公式来看：

o_{i}=softmax(q_i^T K)V \\ h_{i} = o_i^TW_o\\ 即 h_i 是对之前KV cache的聚合。直接使用feature可以理解为使用之前计算的kv cache的attention系数来作为当前预测的近似。在预测较近的token时，KV cache的系数变化不明显，使用feature来预测准确率较高；但在预测较远的token时，KV cache的系数变化就会更为显著，复用之前的kv cache并重新计算系数会更加准确。

对于2的解释：见上一个分析。我个人也尝试过训练一个4层的eagle，发现确实会比1层的准确率高很多。

多层draft layer准确率显著高于单层




draft model 的 cost 是不可忽略的。

接上一段分析，随着draft model的scaling，其运行时间也在显著变长。同样简单测了一下eagle-3的运行时间，在depth=10的情况下，eagle-3的draft model运行时间已经达到了target model运行时间的一半。假如我们可以把这部分draft model的运行时间砍掉，那我们就可以在eagle-3的基础上再取得1.5倍的加速——理想情况下会有9倍的加速比。对于减少draft model的运行时间的方案，目前主要有2种手段：

PEARL类并行手段，通过overlap draft model和target model的运行时间来实现加速。当然PEARL目前对eagle的支持很差，PEARL更适合的还是Llama-3 8B这样接受率很高，但运行时间更长的draft model；
ParallelSpec类并行手段，把auto-regressive head 训成可并行的head （类似Medusa）。对于这种方案，我不抱有很大的期待（序列建模能力更差）

另外在spec领域，adaptive draft length也是一个很重要的topic，即研究draft model在当前context下，生成多少个draft token最合适。这个topic本质上没有改变draft model的运行时间，而是提高了draft model在整个decoding过程中的占比。对于减少draft model运行时间的问题，我个人觉得理想的方案是寻找更新的draft model架构，以此引出下一个分析。




draft model的架构选择有待进一步研究。

当前阶段，auto-regressive的draft model架构的预测能力显著优于其他结构（MLP-based head, RNN-based head）。这一点毋庸置疑，EAGLE-3把加速比都刷到6+了。但是，Auto-Regressive Head仍然不是一个理想的draft model。我在问题“大语言模型的推理速度与问题的难度无关，是否意味着其实现方式可能不是最佳的？”下的回答 大语言模型的推理速度与问题的难度无关，是否意味着其实现方式可能不是最佳的？ 中提到，这种动态使用参数的方式并不够elegant，这里“动态”可以理解为两种情况：(a) 一个token需要使用target model的全量参数才可以解码出来；(b) 一个token只需要只用draft model的全量参数就可以解码出来。

但这种动态仍然只有2种情况，无法实现真正的adaptive. 为了实现真正的adaptive，我觉得有两种方式可以考虑：

MoE类似的架构；
early-exit类似的架构；

比如说，我们训了一个4层带有early-exit能力的eagle-layer，那这种动态就变成了使用{1,2,3,4}层draft model的参数。MoE同理，使用了top-{1,2,3,4}个expert的参数。




多层draft model的训练成本会显著提高。

其实这一点从eagle-3开始就已经有端倪了。eagle-3用了8倍的数据量和3倍的输入feature量，已经对应了一个很大的训练成本。如果再把参数量进行scaling，那这个训练成本就已经超出了大部分人承受的范围了。这里再提出一个可能的方案：从现有的模型去蒸馏一个可以输入target feature的draft model。

为什么呢？从eagle-3的motivation来看，scaling training data可以很大程度的提高draft model的准确率。那这个scaling的极限是什么？数学上难以表达，但直觉上是target model的pretrain data，而开发large-scale的LLM的过程中通常都会存在一些small-scale的LLM，例如Qwen3-0.6B之于32B。Qwen3-0.6B和32B本身使用的训练数据集就非常接近，且数据量很大。在此基础上将其蒸馏成一个多层的、可以复用feature / kvcache的draft model会比training from scratch效率高很多。
