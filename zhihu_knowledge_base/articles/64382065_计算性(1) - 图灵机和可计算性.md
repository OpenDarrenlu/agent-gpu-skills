引入：什么是计算

话说有两个计算机系的同学小A和小B。在看过期中试卷的答案之后，小A和小B不由得产生了一下对话

小A: 明明桶排序是一个O(n)的算法，那么为什么答案还用复杂度是O(nlgn)的快排呢？
小B: 因为桶排序的复杂度依赖于比较的数有多少位，题目又没给数据范围......所以还是快排稳。
小A: .......
小B: 而且算分考试的确很偏爱基于比较的排序。比如说堆排序啦归并排序啦......
小A: 诶？你这么一说，比较两个数谁大谁小也依赖于比较的数有多少位啊？
小B: 啊？我们默认比较两个数的大小是常数时间啊？
小A: 比较两个长度是100000位的数显然比比较两个只有1位的数来的慢啊......
小B: (๑ŐдŐ)b

这个问题的本质在于，比较大小这个操作，究竟是应该看成一个运行时间为1的一个“黑盒子”，一个基本的运算单位，还是应该看他的内部，考虑内部的算法实现需要多少时间。

所以，什么算做是1次计算呢？

那......写一个代码，并且强制性的规定执行完一个语句就算计算一次？

的确有人这么干过。但是这么干的最大问题在于，表面上的一行代码，实际上可能内涵相当丰富，计算次数远远不止一次。举个极端的例子，你写了一行python代码：

x = [i for i in range(100)]

你跟别人说我这个是执行一次，大概别人不咋信。

那.......用汇编语言？

不错的主意。但是问题是一句汇编语言可能也是对应着多个运算。比如说有一个小数运算指令，可以一算算4个数，再比如说8(%ebp)这种指令，是用%ebp的值+8作为地址来寻址的......还是不够理想。

所以怎么办呢？







图灵机

图灵用图灵机这个模型，第一次成功的定义了什么是计算。这里只是简单介绍一下图灵机。

图灵机的灵感来源于人做计算的方法。人做计算，需要一张纸，一个笔，需要一个知道怎么算的大脑。图灵借鉴了这个想法，设计了图灵机。

图灵机由硬件和软件组成。

硬件：

一条一端无限的纸带，对应着人做演算的时候用的草纸。纸带上有编号为0，1，2，3，......的无穷多个格子。
一个指针（这可能也是C语言有指针的原因？），对应着人在演算的时候用的笔。

软件：

可以在纸带上写的字符的集合。这个集合必须有限。对应着人在演算纸上写的字符。
一个有限状态自动机。对应人的大脑。这个有限状态自动机以指针所指的内容为输入，输出可能是如下两种情况：
修改纸带上被指针指向的格子的内容
向左或向右移动指针一个单位

还有很多细枝末节的内容。在此忽略。

诶？为啥用图灵机做标准的计算模型啊？

问的好。图灵机能做计算模型的原因主要是有两点：

成功的定义了什么是“一次计算”。“一次计算”就是在有限状态自动机里面进行一次状态转移。
目前所有物理上能实现的计算模型能计算的东西，图灵机都能计算。

听起来很不可思议吧。这么简单的模型，在计算性上丝毫不逊于各种高大上的模型。所以在定义啥叫“可计算”上，图灵机目前还没法被超越。







有关用图灵机的符号和约定

我们做一些对图灵机的简化处理。这种简化处理不损失计算性，但是方便讨论。

首先，其实图灵机使用的字符的集合只需要0和1这两个字符就够了。原因在于我们可以把任意的字符集，比如说所有的ascii字符，编码到一个有限长的01串上，比如所有ascii字符的ascii码。也就是说，用更大的字符集不会带来额外的计算性上的好处。

其次，我们约定：

图灵机的输入一定从纸带的开头开始写，而不能在开头空几个格再进行输入。
图灵机的输出也一定从纸带的开头开始写，而不能开头空几个格再进行输出。

可以证明这种约定（包括字符集限制在0和1上）不会限制图灵机的计算性。即一个问题，图灵机能解决，当且仅当在这些约定下可以解决。很直观，但是还挺难证明的。

下面定义几个符号:

图灵机用 \mathcal{A} 表示
图灵机的输入用 x 表示，其中 x \in \{0,1\}^* 。 \{0,1\}^* 表示所有有限长的01字符串。
图灵机的输入长度用 |x| 表示。
图灵机的输出用 \mathcal{A}(x)\in \{0,1\}^* 表示。仅当图灵机 \mathcal{A} 能停机时才有输出，否则认为 \mathcal{A}(x) 没有定义。
对于给定输入，图灵机的计算次数用 C(x;\mathcal{A}) 表示。之所以用分号是因为图灵机不随着 x 的变化而变化。
这样的图灵机能解决什么?

图灵机的作用是想定义啥叫做计算。但是计算的目的就是为了解决问题嘛。那么图灵机能够解决什么问题呢？

不失一般性的，我们假设图灵机满足上面说的几个约定。（这几个约定不会让本来能解决的问题变得不能解决）。

图灵机 \mathcal{A} 能做的，就是建立一个输入到输出的映射，即 x\rightarrow \mathcal{A}(x) 。鉴于输入和输出都是属于 \{0,1\}^* 的，所以图灵机的定义域和值域都是 \{0,1\}^* （定义域可能只是这个集合的一部分）。

那么，图灵机能解决的问题的类型就出来了。

图灵机只能计算一个从 \{0,1\}^* (的一部分)到 \{0,1\}^* 的一个映射

下面我们把从 \{0,1\}^* 的一部分映射到 \{0,1\}^* 的一个函数叫做一个字符串问题（这个名词是我自己造的，只是帮助大家理解）。

也就是说，图灵机只能计算字符串问题。

但是字符串问题能量很大。世界上好多问题都可以变成一个字符串问题，只要你能把问题的输入和输出编码成二进制。比如说，求神经网络的全局最优值，给数字们排个序，都可以变成一个字符串问题。所以在下面，我们不再区分字符串问题和问题这两个名词。

那么问题来了，是不是所有的问题都能用一个图灵机来解决呢？

悲剧：图灵机不能解决的问题远比图灵机能解决的问题多。

证明： 图灵机的数量等于构成图灵机的有限状态自动机的数量。有限状态自动机的数量是可数多的。而问题是一个从 \{0,1\}^* 到 \{0,1\}^* 的映射。这样的映射是不可数多的。（就算只考虑从 \{0,1\}^* 到 \{0,1\} 的映射的个数，就等于 \{0,1\}^* 的子集个数，一个可数集的子集个数是不可数多的）。而一个图灵机只能解决一个问题。所以只有可数多的问题能被图灵机解决，有不可数多的问题不能被图灵机解决。

下面，我们把能用图灵机解决的问题叫做可计算问题，不能用图灵机解决的问题叫不可计算问题。有关于计算性的核心主要集中在可计算问题。这个主要在以后讨论。下面简单介绍几个不可计算问题。

停机问题：一个（常见的）不可计算的问题

停机问题就是判断一个图灵机能不能正常的终止。

下面我们形式化的描述一下“停机问题是否可计算”，并且解决他。

给定一个图灵机 \mathcal{A} ，总可以把他编码成一个二进制串 bin(\mathcal{A}) （即用二进制编码编码一个有限状态自动机）。那么问题来了，是否存在一个图灵机 halt ，以任意一个图灵机的二进制编码 bin(\mathcal{A}) 和这个图灵机的任意一个输入 x 为输入，以图灵机 \mathcal{A} 以 x 为输入的时候能否最终停止运行为输出呢？也就是是否存在一个图灵机 halt 满足：

halt(x + bin(\mathcal{A})) = [\mathcal{A}以x为输入时停机]

（由于仅当图灵机终止时才有输出，上面这句话潜藏着 halt 在任意的形如 x+bin(\mathcal{A}) 的输入下必定终止）其中 [P] 当 P 为真时值为1，否则值为0。然后 + 表示字符串拼接。

直观上想一想，一个程序能否停机，只要运行一下，等足够长的时间，如果程序还运行，就算不能停机，否则就算能停机？

但是等多长时间算长呢？你觉得10年算长？那我构造一个程序，不断获取当前时间，如果发现已经过了十年，就停机。这样一个能停机的算法，如果以10年为界，就算是不能停机。

那如果理解一下代码，然后根据逻辑判断能否停机呢？

问题是我们人类的知识储备太有限了。比如一个试图证伪黎曼猜想的图灵机 \mathcal{R} ，那么 \mathcal{R} 能停机当且仅当黎曼猜想不成立。但是谁知道黎曼猜想成不成立......

所以，这么简单的问题，都没有一个图灵机可以解决它。也就是这个问题是不可计算的。下面给个证明。

证明：反证，假设存在 halt 。那么 halt 自己一定可以停机。构造另一个图灵机 halt\_ ，干的活用python表示如下

def halt_():
    if halt(bin(halt_)) == 1: # 如果halt_经halt判断可以停机
        loop_forever() # 就死循环
    else:
        return # 否则停机

那么问题来了。由于 halt 必然可以停机，那么用 halt 判断 halt\_ 能不能停机就该有个输出。如果输出为'1'，即 halt(bin(halt\_))=1 ，说明 halt\_ 能停机。但是此时，由于halt(bin(halt\_))=1，导致 halt\_ 里面的

    if halt(bin(halt_)) == 1: # 如果halt_经halt判断可以停机

得到满足，也就是 halt\_ 会执行死循环，也就是 halt\_ 不能停机，矛盾。类似的，当用 halt 判断 halt\_ 能不能停机输出为0也矛盾。这些个矛盾说明 halt 不存在。
