这篇文章还是标题党了，我也不知道，只能从技术角度分析，也记录一下我学习的过程。

1.有没有

DeepSeek V4 发布之后，国产芯片的好消息出了一长串。八家厂商 Day 0 适配、华为昇腾首发支持、技术报告里第一次把华为 NPU 和英伟达 GPU 写进同一份硬件验证清单。这个消息面伴随着西海岸那边的财报，这几天也带动了港股半导体大涨。

但每条新闻仔细看，都带着同一个限定词：推理。推理和训练是两件事，训练里面预训练和续训又是两件事。V4-Pro 的主预训练用了什么芯片，DeepSeek 没写。这与 2024 年 12 月 V3 技术报告中"2,048 张 H800、约 54 天、$5.576M"的细节描述完全不同（这也是去年闪击美股的关键）。昇腾这次同步开源的训练代码，对应的是 V4-Flash 的续训，不是从零开始的预训练。续训几百卡几天能跑，预训练要万卡集群连续跑几个月，对互联带宽、长稳运行、故障恢复的要求不在一个层级。

所以这三件事的难度是阶梯状的：推理最容易，续训中等，预训练最难。国产芯片现在的进度就卡在这个阶梯上——推理过了，续训刚到，预训练目前还未知。

技术报告 PDF中关于硬件的唯一明确陈述出现在 §3.1:

"We validated the fine-grained EP scheme on both NVIDIA GPUs and Huawei Ascend NPU platforms."

以及颇具争议的明示"昇腾"名字的位置,在 V4-Pro API 定价页的小字注脚中

:受限于高端算力,目前 Pro 的服务吞吐十分有限,预计下半年昇腾 950 超节点批量上市后,Pro 的价格会大幅下调。

现在唯一知道的是MoE routed expert 权重为 FP4,attention / embedding / LM head / 非 MoE dense = FP8; vllm激活 FP8;优化器状态 FP32;梯度 FP8 累加到 FP32 主权重。技术报告明确指出"当前硬件上 FP4×FP8 与 FP8×FP8 峰值吞吐相同,未来原生 FP4 硬件可再带来 1/3 效率增益"——这句话直接对应 Ascend 950 系的 MXFP4 2 PFLOPS 规格。FP4 恰好是华为昇腾 950PR 支持的精度，为后续的国产芯片训练提供了可能。

这部分我最近在搞datapath的工作，这方面微架构的实现确实很麻烦。。。FP4(E2M1 格式)只有 4 比特,总共只能表示 16 个不同的数值——±0.5、±1、±1.5、±2、±3、±4、±6 这一组。对比 FP8 的 256 个值,分辨率和动态范围双双崩盘,这是所有麻烦的源头。

FP8 时代用 per-tensor scaling(整个张量共享一个 FP32 scale)就够了,但 FP4 上这招直接失效——16 个值的颗粒度太粗,只要张量内数值分布稍不均匀,要么大量 underflow 到 0,要么 outlier 直接饱和。OCP 标准的 Microscaling(MX)格式是当前解法:每 32 元素一组共享一个 8-bit E8M0 指数 scale,把动态范围拉回 ~10^38(和 FP32 同级),但块内仍是 4-bit 分辨率——动态范围解决了,分辨率没解决。

更棘手的是训练里前向和反向的非对称性。一次反向涉及三种矩阵乘:fprop(激活×权重)、dgrad(梯度×权重^T)、wgrad(激活^T×梯度)。dgrad 的数值分布有严重的 outlier 长尾——梯度反向穿过各种放大缩小因子后,某些位置比平均值大几个数量级,FP4 在 dgrad 上跑不住,精度直接塌。Blackwell 的解法是非对称配置:fprop 用 MXFP4,dgrad 改用 MXFP6 或 MXFP8。这意味着硬件必须支持多种位宽混跑——Tensor Core 要能切换精度,datapath 按最差情况设计累加器位宽,scheduler 和 scale decode 逻辑都更复杂。如果一个芯片只做了 MXFP4 单一 4-bit 单元,要么训不收敛,要么 dgrad 退回 BF16 把红利吃掉一半。

对模型训练影响最致命的是累加器精度。矩阵乘是 K 次部分积累加进一个累加器,理论上 K 次浮点累加的相对误差正比于 √K × ε(ε 是累加器机器精度)。FP4 输入下每个部分积本身只有 大约8 位有效信息,等效引入约 2 倍累加噪声,对累加器尾数位宽的要求成倍上升。DeepSeek V3 论文里明确指出 H800 的 FP8 累加器只有 FP22 等效精度(13 位尾数 vs FP32 的 23 位),K=4096 时最大相对误差近 2%,V3 不得不每 128 个 WGMMA 把部分和搬到 CUDA Core 用 FP32 再累一次,牺牲 10-20% 吞吐换稳定。FP22 累加器跑 MXFP4 误差会比 FP8 严重几倍,而且 V3 那种”搬到向量核”的 trick 在 FP4 时代救不回来——必须硬件原生支持真 FP32 累加器,否则精度退化没法在软件层补救。个人觉得Blackwell 的 FP4 GEMM 累加器加宽到真 FP32 是 B200 训练能跑通的隐性关键。

而其他厂商披露相关信息比较少，AMD MI300X 已被独立实测确认为真 FP32 全精度累加,但国产芯片这方面几乎没披露。DeepSeek V3.1 引入 UE8M0 缩放因子并在公众号置顶评论中明示"为下一代国产芯片设计"，这是不是一个留下的口子？

2.为什么没有

也就是说，并没有严密的逻辑链说明是用昇腾训练的。不只是昇腾，其实这也是所有国产芯片的缺点。

老规矩，先看看性能差距：


因为llm训练是高度compute-bound，因此相比于inference，所需卡数目更多，往往需要千卡甚至万卡集群。那么就需要两件事： 第一是互联带宽。预训练不是单卡的事，万卡集群之间要频繁同步梯度，每一步都要 all-reduce 一次。华为 HCCS 单向 784 GB/s，大概是 NVLink 5 的四成。同样的训练任务，国产卡花在等数据上的时间比例更高，算力利用率上不去。这是预训练最大的硬件短板。

第二是长稳能力。万卡跑几个月，中间任何一张卡坏了、网络抖一下、某层数值溢出，整个任务可能就要回滚甚至作废。举个例子，Meta Llama 3 405B 训练 16,384 张 H100 跑 54 天,平均 3 小时一次组件故障,共 419 次,但有效训练时间维持在 90% 以上,checkpoint+恢复仅占 2.1%。这个数字是产业基准。这套东西要靠软硬件配合、运维、调度、故障恢复一起磨，磨的过程就是不停踩坑。英伟达这套体系是从一次次大规模训练里跑出来的，坑都填过一遍。国产这边没有这种经验积累。

第三是数值稳定性。训练用低精度（FP8、FP4（实验性））能省一半显存和算力，但低精度容易溢出，几百层网络累积下来信号可能放大几千倍，模型直接训崩。要稳住就得在算法和硬件上都做文章——硬件要保证累加精度足够，算法要做各种约束（V4 的 mHC 就是干这个的）。推理就宽松多了，推理用 BF16 甚至 INT8 都能跑，掉一点点精度无所谓。

最后是软件栈。CUDA 二十年积累的软件生态、profiling 工具、checkpoint 机制、故障恢复工具，是最重要的护城河。比如说tpu的软件团队比硬件团队都大，很多都是专攻训练部分。CANN 这两年追得很快，并且TileLang、TorchTitan-NPU、AutoFuse 也都在补，V4 的续训能跑通就是这套工具链阶段性成熟的体现。但预训练这种长周期、大规模、对长尾故障敏感的任务，最依赖的就是工具链成熟度。这一块比硬件本身更难补。

说到底是个鸡生蛋的问题。没人在国产卡上做过万亿级预训练，工程经验就攒不起来；没攒够经验，又没人敢拿自己的旗舰模型来当第一个吃螃蟹的。所以谁开第一枪、什么时候开，比硬件参数本身重要得多。

3.软硬件适配

不过技术报告中还有挺多有意思的点，感觉也是给国产芯片训练和推理进行软件优化。

1. mHC：

残差连接在数百层网络中累积放大信号。原始的 Hyper-Connections（把单条残差扩展为 4 条并行流，通过 4×4 矩阵 M 在层间混合）虽然增加了表达能力，但无约束的 M 在 27B 消融实验中产生了 3000 倍信号放大（论文数据）——这意味着 1.6T 训练几乎必然崩溃。

mHC（Manifold-Constrained Hyper-Connections，梁文锋本人 2026 年 1 月 arXiv 论文）的核心创新是强制 M 必须是双随机矩阵（每行每列和均为 1、所有元素非负），等价于把 M 约束在 Birkhoff 多面体上，几何性地保证"信号守恒"，同时强制谱范数 ≤ 1。每步训练通过 20 次 Sinkhorn-Knopp 迭代（交替行/列归一化）把约束误差压到 1e-5，把信号放大从 3000× 降到 1.6×——三个数量级的稳定性提升。

mHC 把信号放大压三个数量级，本质上是用算法稳定性补偿硬件层的容错差距，我感觉这也是给国产芯片训练留个口子。

2.2. Muon

业界标准的 AdamW 优化器为每个参数维护两份 FP32 状态（一阶矩 m、二阶矩 v），对 1.6T 参数模型意味着 12.8 TB 仅优化器状态，比模型权重本身都还大几倍。

Muon 不存梯度统计量，而是对梯度矩阵 G 本身做正交化（把所有奇异值拉到 1 附近，使每个正交方向的更新幅度均衡），用 Newton-Schulz 迭代多项式 aX + bX(X^TX) + cX(X^TX)²（系数 a=3.4445、b=−4.7750、c=2.0315 是 Keller Jordan 数值优化得出的最优三阶系数）5 步逼近正交。

V4 采用 Muon+AdamW 混合策略：

Muon 负责：hidden、attention 投影、MoE 专家等 2D 矩阵参数
AdamW 仍负责：embedding（查表无正交意义）、LM head（对输出分布敏感）、RMSNorm（标量参数）、mHC 静态混合矩阵（已有 Birkhoff 约束）

Muon 不维护 m、v，只保留单份 FP32 momentum，主干部分省下约 6.4 TB 显存。

Ascend 950后面应该只能从cxmt拿hbm了，存储容量和带宽都相对较小。单卡 HBM 144 GB，对标 H200 的 141 GB，但也落后最先进的B300的288GB。但即使用 ZeRO-1 把优化器状态分摊到 1024 卡，每卡仍要承受 12.5 GB 的 AdamW 状态——加上模型分片、激活、KV、通信缓冲后接近 HBM 上限。Muon 把单卡这一项压到 6.25 GB，直接为同卡数下的更长序列、更大 batch、更稀疏的 EP 切分腾出 HBM 余量。

3. CSA + HCA：

标准 Transformer 自注意力的 O(N²) 计算量与 O(N) KV Cache 显存，在 N=1M 时双双爆炸，这会导致单卡 HBM 装不下 KV、降低计算单元利用率。

这一次没用mla了，V4 用两条互补路径化解：

CSA（Compressed Sparse Attention）：先把每 m=4 个连续 token 的 KV 用 softmax-gated pooling 压缩成 1 个 entry，再用 FP4 精度的 Lightning Indexer 为每个 query 选 top-k 个最相关压缩块（V4-Pro k=1024、V4-Flash k=512），并保留最近 128 token 的滑动窗口不压缩。
HCA（Heavily Compressed Attention）：反向操作，用 m'=128 的激进压缩率把 1M 序列压成 ~7800 entries 后做完整稠密注意力。

两者在网络深度方向交替堆叠，有效复杂度降到 O(N·k/m + N·w) ≈ N×384——渐进线性，1M 上下文的 attention FLOPs 比稠密版本降低 2500 倍，KV Cache 占用降到 V3.2 的 10%。（论文数据）CSA/HCA 的压缩比把跨节点 KV 传输量压到 V3.2 的 1/10，直接补偿了带宽差距。同时 1M 上下文的完整 KV 能在 8×910C 单节点装下，避免跨节点通信主导延迟。
