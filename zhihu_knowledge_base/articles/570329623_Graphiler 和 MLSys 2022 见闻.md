经历疫情动荡的两年后，MLSys 终于又回到湾区线下举办，知乎上已经有一些论文解读[1]，我也就不重复车轱辘话，论文的 slides 和视频在官网上都可以公开获取，在这里索性就讲讲自己的工作和一些个人体会。

Graphiler: Optimizing Graph Neural Networks with Message Passing Data Flow Graph

Graphiler 是我去年在 AWS Shanghai DGL 团队实习时的成果，其主要的贡献可以概括为两点，1. 基于现有的易于编程的用户自定义函数 (User Defined Function, UDF) 接口构建编译栈而不是另起炉灶设计新的接口。2. 通过对计算图抽象进行补充提出消息传递数据流图 (Message Passing Data Flow Graph, MP-DFG) ，来为新型的计算任务开启更多的优化空间。

我首先从第二点开始讲起，通过在传统数据流图中加入数据驻留（Data Residency）和数据流动（Data Movement），使得我们能够在计算图的层级描述基于信息传递 （Message Passing）的图神经网络有别于经典深度神经网络的独特的计算模式，从而能够在编译期间开启一系列的优化 pass（在文中我们着重强调了广播重排，Broadcast Reordering 和广播融合，Broadcast Fusion）。具体实现上而言，得益于 TorchScript 提供的基础设施，最终的 prototype 比较轻量级。我们只需要对于原本计算图中的成员添加注释 (annotation)，进行类型推理来构建 MP-DFG 和通过 模式匹配 (Pattern Matching) 的方法来实现优化 Pass 和移植与实现少量的计算内核（kernel）。

这一想法和 MLSys 2021 的 distinguished paper 之一，《Data Movement Is All You Need: A Case Study on Optimizing Transformers》是有些相似的，在他们的工作中将 Transformer 训练中大量的数据移动融合进计算图来帮助做更好的计算图层级优化决策。而在 Graphiler 中我们将消息传递模型的语义引入计算图扩展了点和边的内涵，出发点都是通过添加注释来完善计算图的语义。

其实我感到花费更多精力也学到更多的地方在于第一点。System 研究里面完全新颖的想法不太多见，而在一些基本的方法论指导下达成一个不错的优化效果很多时候是相对容易的，往往困难的可能是分辨什么是重要的问题——因为设计往往意味着取舍，鱼和熊掌不可兼得。如果一开始就直接抱着做一个绝对高性能框架的目的，最后面对多样化的需求就容易积重难返。具体到 Graphiler 这个项目，在项目开始之初我就提出可以去重新设计一套接口来解决 UDF 和异构图的需求，因为现在的这套 UDF 接口给与用户的自由度也太高了，在我看来效率低下完全是不可避免。 @王敏捷 鼓励我去完善接口设计的同时，也反复地追问，这些设计为什么不能够在已有的框架下实现？什么是最根本的阻碍？往往我提出一些困难以后，最后我们总能发现其实是可以克服的。我起初很不适应这种 push back，认为是无用功，但也正是在这段时间里，通过和 AWS 的同事交流——其中有不少 MXNet 和 DGL 的核心开发者，我从他们过去的教训中才逐渐理解，到底上广大的用户才是开源社区软件的尺度，易用性并不是种可有可无的添头而却可能是决定生死的支柱。我借用之前有一次 PyTorch 团队分享社区成长心得时讲过的一句话来提供另一个视角，大意是当你能够获得十倍的用户以后，你也获得了十倍的开发者来帮助你完善优化这个框架[2]。我来斯坦福以后第一次跟 Christos 开会，在讲了一些自己感兴趣的技术以后，他给我的第一个建议就是，去遵循一种 Top-down 的逻辑，从重要的问题/需求出发构建解决方案，而不是反过来。这同我在 AWS 学到的东西是一致的，我还需要花一些年头去好好践行这一理念。

为什么我们依然需要线下会议？

在 Graphiler 报告以后，UWisc 的 Shivaram 带着他的学生们围过来，聊起 Graphiler 与他们之前发表在 OSDI 2021 上的 GNN 训练系统 Marius 集成的可能性。我很遗憾地讲因为疫情的缘故，我在去年年底回到 AWS 帮助完善工程开发集成进产品的计划被迫搁浅，团队找的一位暑期实习生也因为类似的原因没法回国，因此 Graphiler 始终停留在一个 prototype 的阶段。我想这可以部分回答这个问题：即使我们今天已经有了足够快的网络，足够好的软件帮助我们远程开会交流，但这种无时差的，不需要反复几轮邮件约定时间，面对面屏蔽其他干扰，全身心投入的交流，依然是更高效和无可取代的。更不必说真正有效的社交需要相对长时间，放松地互相认识和了解。尽管我们已经有了很多的线上交流活动，线下会议也依然是向同侪了解研究细节，寻求合作机会，向前辈求取经验的最低成本的机会——毕竟你只需要走上前打个招呼。

Why MLSys?

MLSys 作为一个新兴会议，在国内认可度还相对较低，尤其很多同学因为学校毕业要求的缘故会尽可能将工作投往类似 CCF 推荐列表收录的会议，而这些列表往往又是更新很慢的。如果从论文水平，影响力等标准评价，我认为其已经超过了许多 CCF B 类会议。如果回归到学术会议交流和社交的本质，那么 MLSys 可能是机器学习系统研究这个新兴领域最好的 community。尽管现在各大系统和机器学习的会议都接收二者相结合的工作，但是如果你真的走进现场同其他人接触，你会发现每个 community 都有他们的核心圈子，总是先入为主的从该领域的习惯评判交叉性的工作。譬如说系统领域内的人不了解机器学习的前沿而更欣赏硬核的底层技术栈，机器学习领域内的人很多时候不了解部署的困难甚至有时根本不在乎。如果我们也用一种 Top-down 的思路来考虑会议定位的话，在各种场景部署机器学习负载，协同演进系统和算法依然是当下和未来一段时间内业界的重要需求，那么 MLSys 就有其绝佳的前景。

当然当下的 MLSys 会议也存在其问题，譬如怎样促进会议中来自不同背景的人有效地互相交流，提升审稿质量等，都需要一些时间探索出合适的模式。相信在下一届组织者的努力下会更向前一步，尤其是 @陈天奇 将担任联席程序主席（program chair）。在会场时我因为是志愿者的缘故，有借口掂着脸上前问了他很多，他将 MLSys 同数据库相比，"同样是脱胎于系统研究，当下的机器学习系统研究在工业界的重要性和学界中的存在感依然是不成正比的，而这也是年轻一辈学者的机会"。
