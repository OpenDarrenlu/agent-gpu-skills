熟悉triton的同学写CuTeDSL的时候会有这样的问题吧。triton有@triton.autotune 为什么没有@cute.autotune?

其实是需要自己实现一个grid search的。官方文档有给一个例子Guidance for Auto-Tuning 。

但是我就是想要一个简单点的autotune，和triton一样用的舒服些。我这里就vibe了一个，和我之前的gemm专栏[CuTeDSL防晕]高性能GEMM 用的repo是一样的。https://github.com/deciding/cutez，(另外讲一下本来我是叫cutex的但是发现pypi上和别人重名了，于是改一下repo名字cutez。)大家可以通过pip install cutez 直接安装。使用方法如下：

首先

import cutez

第一步：定义所有candidate config

AUTOTUNE_CONFIGS = [
    cutez.Config(kwargs={"mma_tiler_mn": (256, 256), "cluster_shape_mn": (2, 1), "ab_stages": ab_stages,})
    for ab_stages in (6, 7, 8)
]


上面我们的ab_stages就可以是6，7，8. mma_tiler_mn 和 cluster_shape_mn 我给固定住了，大家也可以通过for循环加别的

第二步：修饰cute.jit host function

@cutez.autotune(configs=AUTOTUNE_CONFIGS, key=["m", "n", "k"],  cache_path='/workspace/dump/dense_gemm_7min.json')
@cute.jit
def host_function(
...

需要输入configs，以及key（cache key用来说明只要这些key相同我们就用同一套config），cache_path可选，这样可以保证cached best config可以被重复利用。

注意和triton不同，triton.autotune 虽然也是用来修饰triton.jit，但是triton.jit并不是用来修饰host_function。

第三步：cutez.compile

compiled_gemm = cutez.compile(
    ...
    verbose=True, # To show what is the best config
)

这一步必须使用cutez.compile而不是原生的cute.compile。autotune就是在这里被调用的。开启verbose可以看到autotune的过程，哪个config是最优的。最优的那个config会以json的形式存在你的cache_path里。

欢迎大家尝试并给出建议。目前我也把这个例子写到https://github.com/deciding/cutez/blob/main/modal/blackwell/dense_gemm_7min.py 里了。大家可以modal run 01_dense_gemm.py来玩耍。
