QServe 原文地址

abstract：

现有的INT4 量化方法在大batch的云服务场景下不能带来明显的性能收益，原因是现有的INT4的方法在运行时需要大量的dequant/partial sum 开销。为了解决这个问题，MIT的韩松组提出了QoQ （W4A8KV4 量化）工作，主要思想是1. 优化在cuda core 上低吞吐的dequant 计算，从而提出了progressive quantization 量化方法 2. 提出了 SmoothAttention 来缓解int4量化造成的accuracy 下降 3.在给予QoQ 方法构建的QServe系统中，通过 protective range /register-level parallelism / subtraction after multiplication / compute-aware weight reordering 等方法来优化计算性能和保证量化精度。 llama3-8B 模型上在a100 和l40s 上相比tensorrt-llm有1.2x、1.4x的性能提升。

motivation：

现有的LLM量化方法主要分类3大类： W4A16(per_group), W8A8(per_channel weight+per_token activation), W4A4(per_channel)

W4A4KV4：gemm 循环的固定开销大

如图2b， 现有的W4A4的方法atom、quarot表现出比W4A16更差的性能，其主要原因是 W4A4 的per_group 量化存在固有的困难，如图5， w4a16/w4a4 的方法需要在GEMM 的K reduce 循环内分别进行INT4toFP16数据类型转换/INT32toFP32 转换和patial sum accumulation。然而 Nvidia 的GPU cuda core FP32算力与 tensor core INT4算力相差了64倍，如下图，导致GEMM loop中 cuda core 的FP32 accumulation 成为计算瓶颈。此外Atom中需要同时存储FP32和INT32 的patial sum，需要更多的register file，这导致了SM 中active warp 的数目减少，SM的occupancy降低。因为NV GPU 通过大量的warp上下文切换来隐藏延迟，active warp减少导致更难隐藏延迟，加剧了计算的开销。

QoQ quantization
progressive group quantization

为了低比特量化达成更好的精度，提出了progressive group quantization方法，采用了2-level 的量化模式。fp16->int8->uint4。 对于fp16 的 W\in \mathbb{R}^{K\times N} ，先采用per channel symmetric 量化到INT8,其中channel-wise scale= s_{fp16}\in \mathbb{R}^{N\times 1}

\hat{W} = Qw_{s8}\cdot s_{fp16}\\ Qw_{s8} = (Qw_{u4)-z_{u4}})\cdot s_{s8}\\ 第二阶段采用per_group asymmetric 量化从int8->uint4,其中group scale s_{s8}\in \mathbb{N}^{N\times k/g} ， zero point z_{s8}\in \mathbb{N}^{N\times k/g}

在GEMM的计算中，只需要计算从uint4->int8的反量化即可。




Protective Quantization Range

从int8量化到uint4，再从uint4反量化到int8 这个过程并不能保证数据范围的正确表示。例如 group内的数据范围是[-113,120],per_group asymmetric 的 scale = \frac{120-(-113)} {(15 - 0)} = 16 、 Z\_point = \frac{0-(-113)}{16}=7 。int8 120 量化到uint4 \frac{x}{scale} + Z\_point = \frac{120}{16}+7=15 ，再从uint4 反量化到int8 (15-7)*16=128>2^7-1=127 ,溢出。

通过把int8表示范围限制到[-119, 119]之内，可以保护数据反量化溢出。如上图6所示，

SmoothAttention

如图7， K中的存在显著的outiler，这会导致量化阻碍。 受smooth quant的启发， 本文引入了smooth attention， 对Q K per_channel进行缩放，缓解outiler 的影响

Z = （Q\Lambda）\cdot(K\Lambda^{-1})^T , \Lambda=diag(\lambda), \lambda_{i} = max(|K_i|)^\alpha \\
由于KV cache 再KV project 后需要计算rope, rope中 head_size/2 的K V 需要做交换计算，如下

y[0:\frac{head\_size}{2}] = cos(freq)\cdot x[0:\frac{head\_size}{2}] -sin(freq)\cdot x[\frac{head\_size}{2}:] \\ y[\frac{head\_size}{2}:] = sin(freq)\cdot x[0:\frac{head\_size}{2}] +cos(freq)\cdot x[\frac{head\_size}{2}:] \\

需要修改 \lambda 的计算方式

\lambda_{i}=\lambda_{i+\frac{head\_size}{2}} = max(max(|K_i|), max(|K_{i+\frac{head\_size}{2}} |))^\alpha \\可以在预处理阶段就把 \Lambda fuse进Wkv的权重中， W_Q = \Lambda W_Q, W_K=\Lambda^{-1}W_K

General LLM quantization optimization

为了解决activation outiler 的问题，本文采用了一些通用的LLM 量化优化策略

如图8，Block Input Module Rotation， module input 对于的权重乘上一个旋转矩阵，activation中每个channel 的值为所有channel 的线性组合，更容易压制outiler 。
如图9，Block Output Module Smoothing， 对module output 对于的权重进行per channel smooth 操作。
如图10，Activation-Aware Channel Reordering, 把weight 中对salient weight 进行重拍，让salient weight 放在同一个group内进行group quantization。

QServe System

QoQ量化的目的是保持量化后的推理精度，构建QServe 系统的目的是提高整体的吞吐效率。 主要通过两部分来达成1. 减少GEMM 循环的开销 2. 保持decode phase。Attention kernel 是一个memory bound

QServe System Runtime

在QServe中以w4a8 的模式，GEMM层调用int8 的tensor code 计算 uin4（weight）、int8（activation）-> fp16(output)， decode Attention 层调用cuda core 计算fp16（input），所有block的输入输出都是fp16. 为了保证GEMM的activation 输入是INT8， 在gemm前的layernorm / acitvation 算子融合了量化算子，保证norm 和activation 函数输出的结果是int8，且在attention之后插入了一个独立的量化算子。

QServe 对KV cache 采用了per_head dynamic 量化在更低的比特精度的情况下保持整网的准确性。

W4A8 GEMM

QServe 中通过compute-aware weight reorder , subtraction after multiplication , register-level parallelism 三种方法来减少GEMM循环中的开销。

compute-aware weight reorder

如上图，调用 tensorcore 计算gemm 对每个线程排布有一定的限制，t0 线程需要拥有第一行 0-3 、16-19 的元素。直接计算地址的偏移是低效的，会导致2个性能问题，1. 因为地址偏移是调用cuda core 进行计算的，cuda core 的int32 指令吞吐是远低于tensorcore int8 指令吞吐，1:32，所以在GEMM 循环内频繁的计算地址偏移是不可忽视的性能开销。

a100 tensorcore

2. nvidia gpu 中对dram 的memory transaction 是以32Bytes 为单位，如上图a， t0的一次访问只有4B（4*int8），一次memory transaction只有1/8是有效的， 影响内存访问的效率。在相同数据类型的gemm情况下，可以直接调用ldmatrix 指令来高效的解决。 在w4a8的情况下，ldmatrix 实效了，会错误把本不该属于本线程的输入读取，造成mma 计算的错误。如上图b。

为了解决以上2点问题，本文提出了compute-aware weight reorder，采用32x32的tile策略，将属于同一个线程的数据重新排列，让这些数据保持内存上连续，每个线程所需的数据组成32B（32*int8）满足一个memory transaction，如图c， 对量化所需的scale/ zeros 同样进行重拍。

2. register-level parallelism

对于uint4->int8这一过程，QServe 把weight 的内存排布重新排序， 按照 从w0 w1 w2 ... w31 的顺序改为 w0 w16 w1 w17 .... w15 w31. 如上图， 只需要 一次 右移和两次 and 操作 就可以完成32个uint4->uint8 的转换。

3. subtraction after multiplication

gemm per group量化的过程可以用下面的公式表达，

\begin{aligned} \mathbf{O} & = \hat{\mathbf{X}} \hat{\mathbf{W}}=\left(\mathbf{Q}_{\mathbf{x}} \odot \mathbf{S}_{\mathbf{x}}\right)\left(\left(\mathbf{Q}_{\mathbf{w}}-\mathbf{Z}_{\mathbf{W}}\right) \odot \mathbf{S}_{\mathbf{w}}\right)\\ &=\left(\mathbf{Q}_{\mathbf{X}} \odot \mathbf{S}_{\mathbf{X}}\right)\left(\mathbf{Q}_{\mathbf{W}} \odot \mathbf{S}_{\mathbf{W}}-\mathbf{Z} \mathbf{S}_{\mathbf{W}}\right) \\ & =\left(\mathbf{Q}_{\mathbf{X}} \mathbf{Q}_{\mathbf{W}}\right) \odot\left(\mathbf{s}_{\mathbf{W}} \times \mathbf{s}_{\mathbf{X}}\right)-\left(\mathbf{Q}_{\mathbf{X}} \odot \mathbf{S}_{\mathbf{X}}\right) \mathbf{Z} \mathbf{S}_{\mathbf{W}} \end{aligned}\\ \left(\mathbf{Q}_{\mathbf{X}} \odot \mathbf{S}_{\mathbf{X}}\right) \mathbf{Z} \mathbf{S}_{\mathbf{W}} = \mathbf{X}\left(\mathbf{Z S}_{\mathbf{W}}\right)=\mathbf{t}_{\mathbf{X}} \times\left(\mathbf{z}_{\mathbf{W}} \odot \mathbf{s}_{\mathbf{W}}\right)\\仿照 TensorRT-LLM 中w8a8的方式，把 (Q_xQ_w)\odot (s_w\times s_x) 从gemm的循环中移动到epilogue 阶段计算, 同样的， zeros point 的减法操作\left(\mathbf{Q}_{\mathbf{X}} \odot \mathbf{S}_{\mathbf{X}}\right) \mathbf{Z} \mathbf{S}_{\mathbf{W}}同样可以从循环移动到epilogue计算。减少循环中的反量化开销。

在gemm per group 量化中， zero point 不再可以移动到epilogue进行计算，但是本文仍然采用subtraction after multiplication，因为能搞好的发挥register-level parallelism 【这部分不太理解，好像是】。

4. general optimization

比较常见的一些优化手段，async memory pingpong 数据搬运和计算冰箱， swizzle shared memory layout t避免bank conflict， 在小m场景下warp-level k 纬度并行计算 gemm 等方法

KV4 Attention in QServe

直接在tensorrt-llm上修改per_head dyanmic kv cache int4量化,在l40s上有1.7倍的性能提升，但是在a100上只有baseline 的0.83x。探究原因，是因为cuda core 的计算瓶颈， 2个int4->2个fp16 需要mask、 cvt，shift、mul、sub 操作，共5 alu， a100的 cuda core fp32计算密度仅为 9.8op/byte， 说明 已经到达compute bound。

为了解决这一问题，本文把fp32 的操作用fp16 代替， cuda core fp16指令吞吐是fp32 的2x， 通过一些trick 把dequant 的操作减少到2op/element， 把compute bound 转化到memory bound。 kv4成功在a100上达到kv8的1.5x。
