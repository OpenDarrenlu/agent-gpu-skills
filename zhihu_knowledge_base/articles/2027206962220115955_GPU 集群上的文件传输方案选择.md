# GPU 集群上的文件传输方案选择

**作者**: Arsmart​上海交通大学 计算机科学技术博士在读

**原文链接**: https://zhuanlan.zhihu.com/p/2027206962220115955

---

核心结论

三种传输需求，三种方案：

需求	方案	原因
首次传大文件	cp -a（多路并行）	速度与 tar 相当，但可续传、可观测进度
首次传大量小文件	tar cf - \| tar xf -（多路并行）	源端/目标端延迟重叠，减少每个文件的启动开销
非首次传输	rsync -a	只增量传差异，其他方案都做不到
第几次传输？
├── 首次（目标端无数据）
│   ├── 大文件为主 → cp -a（多路并行）
│   └── 大量小文件 → tar cf - | tar xf -（多路并行）
└── 非首次 → rsync -a（多路并行）

并行度对两种场景的影响完全不同：

大文件受限于带宽，并行度超过带宽上限后性能持平。小文件受限于元数据服务器（MDS）处理能力，并行度存在一个先升后降的最优点，需实测确定。

为什么这样选
大文件 vs 小文件：瓶颈完全不同

大文件（10GB）：元数据操作（open/close）只有 2 次，数据传输占 99.99% 时间 → 瓶颈是带宽。

小文件（1KB）：每个文件需要 stat/open/read/close 共 4 次网络往返，数据传输约 0ms → 瓶颈是每个文件的启动开销（元数据延迟）。CephFS 中元数据由 MDS 处理，数据由 OSD 处理 [1]，小文件场景下 MDS 成为主要瓶颈。

tar 管道为什么对小文件更快

cp 是一个进程交替跟源端和目标端对话，延迟叠加：

cp:  [源端]read → [目标端]write → [源端]read → [目标端]write → ...

tar cf - | tar xf - 是两个进程通过本地内存管道连接，各守一端，延迟重叠：

tar cf (只跟源端说话):       tar xf (只跟目标端说话):
  read file1                    |
  写入 pipe ───────────→   从 pipe 读出
  read file2                 create+write file1
  写入 pipe ───────────→   create+write file2

大文件时两者差不多（都打满带宽），所以选更可控的 cp。减少 per-file 开销的思路与 metadata-intensive workload 优化领域的研究方向一致 [6]。

并行度怎么选

大文件：性能随并行度先升后平（带宽打满就到顶了）。

小文件：性能随并行度先升后降（倒 U 形）[3]。上升阶段：多进程有助于提高 MDS 利用率 [2]。下降阶段：高度怀疑是目录索引同步开销激增 [5]，与该类争抢现象的特征一致。

最优并行数因环境而异（取决于 MDS 线程数、网络延迟、目录结构），不同平台下 I/O 行为差异很大，参数选择应以实测为准 [4]。我们的环境中粗测了两个点：12 路正常，86 路严重退化，仅供参考。容器内无法直接观测 MDS 指标（/proc/fs/ceph/ 不可见），只能通过目标端文件增长速率间接判断。

硬件背景




	CephFS	NFS /data	localssd
介质	HDD	NVMe SSD	SATA SSD
连接	网络	网络	直连，无网络
理论峰值带宽	1250 MB/s (网卡上限)	1250 MB/s (网卡上限)	550 MB/s (SATA 上限)
随机读延迟	~10 ms	~100 μs	~50 μs
跨节点可见	是	是	否

CephFS 和 NFS 共享同一张网卡。localssd 最快但换机器就没了。

优化过程

同一批文件（conda 环境，13 万+ 小文件，~25GB），CephFS → NFS 的四次尝试：

阶段	方案	吞吐	每步解决了什么
1	单线程 tar -cf file.tar	~50 KB/s	—
2	20+ 路并行 cp -a	~30 MB/s	并行化，吞吐大幅提升（×600）
3	12 路并行 tar cf - \| tar xf -	318 MB/s	tar 管道，解决两端延迟叠加（×10）
4	86 路并行 tar cf - \| tar xf -	~0	过度并行，与 MDS 目录索引争抢现象一致

注意：阶段 1 与阶段 3 的输出形态不同（生成 tar 文件 vs 重建目录树），不是严格 apples-to-apples 对比。但作为优化过程的观察值，从 50 KB/s 到 318 MB/s 的量级差异仍然说明了方案选择的巨大影响。

参考文献
Weil S A, et al. Ceph: A Scalable, High-Performance Distributed File System. OSDI 2006. — CephFS 架构设计：数据(OSD)与元数据(MDS)分离，小文件瓶颈在 MDS 的理论来源。
Sevilla M A, et al. Mantle: A Programmable Metadata Load Balancer for the Ceph File System. SC 2015. — 提出可编程 CephFS 元数据负载均衡器，说明 metadata distribution policy、locality 和同步开销会显著影响性能。
Carns P H, Lang S, Ross R, et al. Small-File Access in Parallel File Systems. IPDPS 2009. — 说明并行文件系统上 small-file workload 对 metadata path 很敏感，批量化与元数据优化能显著提升 create/stat/remove 吞吐，本文关于小文件特性的背景依据。
Luu H, et al. A Multiplatform Study of I/O Behavior on Petascale Supercomputers. HPDC 2015. — 基于多平台 Darshan 日志展示 I/O 行为和吞吐在不同平台与 workload 下差异很大，提示参数选择应以实测为准。
Patil S V, Gibson G A. Scale and Concurrency of GIGA+: File System Directories with Millions of Files. FAST 2011. — 说明去除目录服务中的全局同步/串行化有助于提升高并发 small-file create 的扩展性，为高并发目录操作下的同步开销提供了相关机制背景。
Ren K, Zheng Q, Patil S, Gibson G A. IndexFS: Scaling File System Metadata Performance with Stateless Caching and Bulk Insertion. SC 2014. — 通过 bulk insertion、stateless caching 和按目录增量分区提升 metadata-intensive workload 吞吐，其批量化思想与本文对小文件搬运的优化方向一致（但并非对 tar 与 cp 的直接比较）。
