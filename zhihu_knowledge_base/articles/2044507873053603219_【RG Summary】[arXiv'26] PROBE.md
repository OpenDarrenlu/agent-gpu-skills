PROBE：通过实时预测预取，在 MoE 推理中协同平衡计算与通信

作者：Qianchao Zhu, Xucheng Ye, Yuliang Liu, Haodong Ouyang, Chengru Song

专家负载不均衡示意图

内容总结：这篇文章关注大规模 MoE 模型在线推理中的专家负载不均问题。MoE 通常在 Attention 层使用数据并行，在 MoE 层使用专家并行；由于 router 会按照 token 语义选择少数专家，某些专家可能在短时间内变成热专家，导致部分 GPU 承担远高于平均值的 token 负载。负载不均不仅会让快 GPU 等待慢 GPU，形成计算气泡，还会放大 All-to-All 通信中最大通信 rank 对整体时延的影响。文章使用 Imbalance Ratio（IR）刻画这种不均衡程度，IR 越接近 1 说明 GPU 间负载越均衡。

在线推理场景进一步放大了这个问题。Prefill 阶段会一次性处理大量 prompt token，如果请求语义集中，热点专家会突发出现；decode 阶段受到 continuous batching 和生成语义变化影响，专家热度会持续波动。已有方法 DeepSeek-EPLB 通过历史窗口跟踪热专家并复制冗余专家，可以改善长期负载分布，但它对短期语义漂移反应较慢，并且会引入额外迁移和显存开销。

层级预取直观示意

PROBE 的核心思想是 layer-level prefetching：在当前层执行期间预测下一层的专家访问分布，提前规划并预取冗余专家，把负载均衡开销隐藏在主推理流水线之外。系统包含四个主要模块：Lookahead Predictor 用 cloned router 加 residual MLP 预测下一层 routing；Balance-Optimal Planning 根据负载、专家大小、带宽和可隐藏时间窗口决定是否迁移专家；P2P Expert Transfer 使用 NVSHMEM 和双缓冲 replica region 进行 GPU-to-GPU 专家传输；Phase-Locked Co-Scheduling 将预测、规划、传输与 Dispatch、MoE compute、Combine 等阶段对齐，避免和关键路径通信争用带宽。

实验方面，PROBE 集成在 SGLang 中，并使用 DeepEP 作为通信后端。作者在 8 张 NVIDIA H200 上测试 Qwen3-MoE-235B 和 GPT-OSS-120B，并使用中文、代码和 Synthetic Repeat 数据集评估性能。结果显示，PROBE 在 prefill 阶段相比 SGLang 最高达到 1.32x 加速，在 decode 阶段相比 DeepSeek-EPLB 最高达到 1.26x 加速；在线蒸馏后 predictor 的 Top-K 准确率达到 87%-94%，Top-Half-K Hit-Rate 和 2xTop-K Recall 接近 100%；在延迟拆解实验中，IR 从 2.13 降低到 1.09，说明系统确实缓解了 GPU 间专家负载倾斜。

Q&A

Q：为什么最后 latency breakdown 里 Combine 时间有 710 微秒，但是 Dispatch 只有 170 微秒？

A：因为这里的 Combine 时间其实包含了同步等待时间。图中记录的只是其中一个 rank 的时间线，它在 Combine 阶段不只是做通信，还需要等待其他 rank 完成专家计算，所以这个阶段看起来会明显更长。

Q：为什么负载均衡后通信时间也会减小？

A：在负载不均的情况下，通信时间基本可以看成由 token 负载最大的 rank 决定。负载均衡以后，最大 rank 上需要处理和通信的 token 数减少了，因此 All-to-All 中最慢 rank 的通信时间也会下降，整体通信时间就会变短。

Q：图里 prefetch 时间大概只有 1.2 ms 左右，可以被计算很好地掩盖。但实验只用了 8 卡 NVLink，如果换成 16 卡、32 卡 EP，这个时间还能被掩盖吗？

A：这也是对 PROBE 的一个主要疑问。它可能在跨机或者更大规模 EP 场景下就不一定适用，因为跨节点带宽和延迟会让 prefetch 更难隐藏。不过这本质上也是一个通信和计算的相对比例问题：论文使用的是 H200 场景，计算本身也很快；如果某个硬件环境里通信相对于计算足够快，那么 prefetch 仍然可能被较好地掩盖。

Q：基于上一个问题，一般来说要想用计算掩盖 prefetch 通信时间，是不是需要把 batch size 打得很大？这样会不会违背 SLO？

A：从论文实验看，它确实使用了比较大的 batch size。因此，PROBE 更适合 batch size 较高、当前层计算窗口足够长的场景；否则计算时间不足以掩盖通信，prefetch 开销就可能暴露到关键路径上。SLO 是相对的，在低 batch 或严格时延要求下，这种方法确实可能一定程度上违背 SLO。

Q：这个蒸馏过程用了很多数据集。那实际服务某个领域的数据集时，系统怎么知道应该使用哪个数据集对应的权重？

A：这里并不是为每个数据集单独维护一套权重，然后运行时显式选择。Predictor 的权重是基于多数据集蒸馏出来的，当不同领域的数据进入时，它会根据上一层 hidden state 自身携带的语义信息来预测当前热点专家。也就是说，模型通过输入的 hidden state 自动适配当前 workload，而不是手动判断当前属于哪个数据集。
