1. 先建立整体心智模型

先把 low-latency 路径看成下面这条流水线：

输入 token x
  -> 可选 clean low-latency buffer
  -> low_latency_dispatch: 发 RDMA 请求，把结果写入 fixed-capacity packed buffer
  -> hook/event: 决定何时真正完成接收
  -> local expert compute
  -> low_latency_combine: 按原 token 回程归并
  -> 得到最终结果

3. low-latency 和 normal mode 的根本差别

如果你已经读过 V1 high-throughput，那 low-latency 最核心的变化其实就一句话：

normal mode 追求紧凑接收结果
low-latency mode 追求更低的 decode 时延

normal mode 里，CPU 会先等精确接收规模，再分配：

recv_x: [num_recv_tokens, hidden]

这样做的好处是输出紧凑，但代价是：

需要 CPU 同步等待接收规模元信息

low-latency 路径反过来选择：

直接返回固定容量 packed buffer
不再为了拿到精确接收规模而让 CPU 卡住
用 recv_count 告诉你其中有多少条目有效
重要发现
V1 low-latency 的核心不是“更快算出精确 recv size”，而是 不再依赖精确 recv size 才继续前进。
它把“紧凑输出”换成了：

fixed-capacity packed buffer
recv_count
hook / event

7. 为什么 low-latency 不再等精确 recv size

normal mode 里，CPU 要等 moe_recv_counter，因为它要创建：

recv_x: [num_recv_tokens, hidden]

low-latency 里，这件事被彻底绕开了。

因为它一开始就按固定上界分配：

[num_local_experts, num_ranks * num_max_dispatch_tokens_per_rank, hidden]

于是：

CPU 不需要等“精确收到了多少”
数据不再返回为紧凑 tensor
而是由 recv_count 和 layout_range 告诉你哪部分有效

这就是 decode 场景里“延迟优先”设计的核心。

重要发现
low-latency 不返回 normal mode 那种紧凑 recv_x: [num_recv_tokens, hidden]，正是为了绕开：

先等精确 recv size
再创建紧凑输出 tensor

这一步在 decode 场景里时延代价太高。

8. hook() 到底推迟了什么

native low_latency_dispatch 里最关键的一段在：

https://github.com/deepseek-ai/DeepEP/blob/d4f41e4e93602a15e95f55f6ee8df8f1aaa0e4bb/csrc/legacy/buffer.hpp#L1543
auto launcher = [=, this](int phases) {
    internode_ll::dispatch(..., phases);
};
launcher(return_recv_hook ? LEGACY_LOW_LATENCY_SEND_PHASE : (LEGACY_LOW_LATENCY_SEND_PHASE | LEGACY_LOW_LATENCY_RECV_PHASE));

if (return_recv_hook)
    recv_hook = [=]() { launcher(LEGACY_LOW_LATENCY_RECV_PHASE); };

这段代码把 low-latency dispatch 明确拆成两个 phase：

SEND_PHASE
RECV_PHASE

所以 hook() 推迟的不是“整个 dispatch”，而是：

先只 issue send/request
把真正 receive phase 留到你调用 hook() 时再执行

这就是文档里那句：

the actual tensor will not be received only if you call hook()
重要发现
hook() 的作用不是“再做一次通信”，而是 补上被延后的 receive phase。
重要发现
从代码实现上看，hook 本质上就是 native 返回的一个 std::function<void()>：

dispatch 里是 recv_hook = [=]() { launcher(LEGACY_LOW_LATENCY_RECV_PHASE); }
combine 里也是同样模式

所以它不是抽象回调，而是一个非常具体的“延后执行 RECV_PHASE 的函数句柄”。

9. 为什么文档说 overlap “without any SM occupation”

这句话要小心理解。

它不是说 low-latency 完全不需要 GPU kernel，也不是说 GPU 永远不做事。

更准确的意思是：

在 return_recv_hook=True 的模式下
先把 RDMA request issue 出去
后面的网络传输在后台进行
中间 overlap 窗口里，计算部分不需要额外占着 SM 做轮询式接收
直到你真正调用 hook()，才补上 receive phase

这就是为什么测试里会写：

large_gemm_with_hook(hook) if return_recv_hook else None

见：

https://github.com/deepseek-ai/DeepEP/blob/d4f41e4e93602a15e95f55f6ee8df8f1aaa0e4bb/tests/legacy/test_low_latency.py#L203

它表达的是：

先 issue 通信
中间做一大段 compute
最后调用 hook() 补 receive
10. 双缓冲为什么导致“一次最多只能 hold 两份结果”

native 代码里能看到：

LowLatencyLayout layout(...);
auto buffer = layout.buffers[low_latency_buffer_idx];
auto next_buffer = layout.buffers[low_latency_buffer_idx ^= 1];

也就是说 low-latency runtime 维护的是一个双缓冲：

当前 buffer
下一个 buffer

而返回的 packed tensor 直接复用了这些底层 buffer 里的内容或与其同生命周期的配套结构。

所以如果你 hold 了太多轮 low-latency 结果：

后续 kernel 可能会覆盖前面的 buffer

这就是 Python docstring 里这句 warning 的来源：

as there are only two buffers, and the returned tensors reuse the buffer, you cannot hold more than 2 low-latency kernels’ result tensors at a single moment.
重要发现
low-latency 不是只有 fixed-capacity buffer，还是 fixed-capacity + double buffer：

固定容量解决的是“不等精确 recv size”
双缓冲解决的是“上一轮/下一轮通信如何轮换推进”

也正因为只有两套 buffer，结果 tensor 的生命周期才会受限。

hook() 延后的是 receive phase，不是整个通信。
“0 SM overlap” 说的是 overlap 窗口里不需要额外占着 SM 做接收轮询。
双缓冲意味着同一时刻不能长期持有超过两份 low-latency 结果。
src_info + layout_range 是 low-latency handle 的核心。
zero_copy=True 跳过的是“拷进 combine RDMA 发送 buffer”的那一步。

如果你继续往下读源码，建议顺序是：

docs/legacy.md 里的 decoding 示例
deep_ep/buffers/legacy.py 的 low-latency API
tests/legacy/test_low_latency.py
csrc/legacy/buffer.hpp 的 low_latency_dispatch / low_latency_combine
再进入 csrc/kernels/legacy/internode_ll.cu

这样最容易先建立正确心智模型，再下钻内核细节。
