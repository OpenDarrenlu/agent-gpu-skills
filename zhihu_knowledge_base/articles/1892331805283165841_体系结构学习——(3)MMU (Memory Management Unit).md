​背景与作用

核心问题：为什么需要MMU?

MMU（Memory Management Unit，内存管理单元） 是计算机硬件中的核心组件，主要负责管理内存访问和地址转换。它是现代操作系统实现虚拟内存、内存保护和多任务运行的关键硬件支持。MMU，CPU，memory的关系如图所示。


MMU，CPU，memory的关系

在虚拟存储的概念出现之前，处理器需要直接使用物理地址访问存储。当程序大于物理内存时，需要程序员将程序划分为多个小于物理内存的片段存在外部存储，当运行时再把运行的片段放入物理内存，随着程序的变大，这会大大降低程序员的效率。


MMU的作用

虚拟地址到物理地址的转换：程序运行使用的是虚拟地址（逻辑地址），而非直接操作物理内存地址。MMU负责将虚拟地址转换为物理地址，使程序无需关心物理内存的实际分配。例如：一个程序可能认为自己拥有连续的4GB地址空间，但实际上物理内存可能分散或不连续。
内存保护与隔离：MMU通过权限位（读、写、执行）为每个内存页设置访问权限，防止进程越界访问其他程序或内核的内存空间。例如：用户程序无法直接修改操作系统内核的内存，避免系统崩溃或安全漏洞。
分页与内存分配优化：支持分页机制（Page Table），将内存划分为固定大小的页（如4KB），按需加载到物理内存或置换到磁盘（交换空间）。例如：物理内存不足时，MMU配合操作系统将不常用的页换出到磁盘，腾出空间供其他程序使用。
支持多任务与虚拟内存：每个进程拥有独立的虚拟地址空间，MMU确保不同进程的地址空间隔离，实现安全的多任务并行。例如：浏览器和游戏可以同时运行，彼此不会干扰对方的物理内存。
地址转换

核心问题：虚拟地址到物理地址的转化过程

目前大多数的处理器使用基于分页（page）的虚拟存储器方法。该方法以页为单位对虚拟空间进行划分，典型的页大小为4KB，相应的物理地址空间也进行了划分，称为frame（为便于表述，本文称其为帧），页和帧的大小必须相等以方便偏移对应。分页后的虚拟地址（Virtual Address，VA）由VPN（Virtual Page Number）和Page Offset组成。对于一个典型的4KB的页，它需要12bit的地址进行索引页内的地址内容，因此VA[11:0]就是页内偏移，VA剩余的部分就是VPN，用于索引具体的页号。同样，物理地址（Physical Address，PA）也被划分成了一个个的frame，所以PA也由PFN（Physical Frame Number）和frame offset组成。由于物理存储的帧和虚拟存储的页大小相同，因此PA[11:0]同样作为帧中的地址偏移，剩余部分索引帧号。显然，虚拟地址到物理地址的转换也就是VPN到PFN的转换。这里举一个例子说明地址转换的过程。

假设处理器的位宽为16比特，虚拟存储空间为64KB，包含16个VPN，而配置的内存大小为32KB包含8个PFN，显然虚拟空间无法和物理内存一一对应，需要进行地址转换才能进行访问。实际上，所谓转换就是一种映射关系，一个虚拟地址根据映射关系得到对应的物理地址。现在有一个大于32KB的程序，它无法直接全部放入内存中，因此剩余部分需要放在更下一级的存储中（比如硬盘）。只有当程序片段被需要时才从硬盘调入内存中使用。下图演示了地址转换的示例过程，其中一部分的虚拟地址已经映射到了物理空间，比如VPN0映射到了PFN2，VPN1映射到了VPN0。

转化过程

现在先说明如何进行对应。

假设在处理器中，需要在虚拟地址空间5中取数。此时地址转换会通过VA和PA的映射关系进行对应。在图中，虚拟空间的地址5属于VPN0，它映射到了PFN2，因此可以说VA[15:12]=0映射为PA[14:12]=2，而页内偏移不需要转换，因此PA[11:0]=VA[11:0]=5，所以虚拟地址5转换为8192+5=8197。物理存储不需要关心虚拟地址是什么，它只知道自己需要读出8197这个地址的数据。
假设在处理器中，需要在虚拟地址空间20500中取数。由于20500在VPN5中，而VPN5映射到PFN3，因此转换过后的物理地址就应该是映射的物理起始地址加上页内的偏移地址，即12288+（20500-20480）=12308。说简单一点，VPN到PFN的转换只是改变空间的起始地址，只需要将转换的起始地址加上偏移地址就得了物理地址。
假设虚拟空间地址32700进行load操作。32700在page8中，page8和PFN没有映射关系，也就是说page8的内容不在物理内存中，而是存在更大的硬盘内。当MMU发现这种映射关系不存在时，会产生一个page fault的异常发送给处理器，如何处理器转到页缺失的异常处理程序中进行处理。在处理页缺失异常时，异常处理程序会先寻找一个已有的映射关系进行解除（一般选择最老的映射关系），如果被解除映射关系的frame是被修改过的（也就是说和下一级存储的内容不一致，这种情况也称为“脏” dirty），那还需要将该frame的内容更新到下一级存储中，然后在下级的存储中找到需要的page8的内容放到解除关系的frame中，并建立起该frame和page8的映射关系。处理完成后，该load操作就可以重新执行了。
单级页表

核心问题：如何实现虚拟地址到物理地址的转化？

从上一小节的地址转化可以看出，虚拟地址到物理地址有着一种映射关系，处理器需要使用物理地址时，需要根据映射关系进行转化。在实现上，可以使用一张表记录这种映射关系，使用虚拟地址作为这张表的索引，就可以得到该表项记录的内容——物理地址，这张表即页表（page table）。

现代处理器需要能同时处理多个程序，比如我们在打开网页的同时还可以打开word码字。在物理存储中，不同的程序被放在不同的地址空间段。不同的程序也有各自的页表，因此，为了找到各自页表的在物理内存中的起始地址，处理器中还会包含一个寄存器，用于存放当前页表在物理内存中的位置，这个寄存器称为页表寄存器（page table register，PTR）。每次操作系统将一个程序调入物理内存中执行的时候，就会将PTR配置为对应页表在物理内存中的起始地址。当然，这也就要求页表在物理空间中是连续存放的。（连续存放意味着对存放大小较为敏感）




一级页表

虚拟地址通过页表转化为物理地址的过程如上图所示，由于地址的转换实际上是VPN到PFN的转换，因此该表的索引是VPN，而表项是PFN，页内偏移不需要转换，直接和PFN组成物理地址。在图中，PTR找到当前程序对应页表在物理内存中的起始地址，VPN作为页表偏移查询出PFN，PFN和页内偏移组成真正的物理地址。为了指示这种映射关系是否有有效，页表的表项还有1bit的valid的标志位，当该valid为1时，则表明该表项的映射关系有效，而valid为0时，则表明当前的VPN并没有建立映射关系，此时就会产生page fault异常。

不足之处：页表存放了虚拟地址到物理地址的映射关系，而页表一般同样存放在物理内存中。我们可以计算一下图所示的页表需要占用多大的物理内存。虚拟地址空间大小为4GB，页的大小是4KB，那就需要4GB/4KB个表项以存放映射关系。因为页表存放在物理内存中，所以假设其数据位宽和物理内存一致，为4字节，那么一个页表所需的存放空间为4GB/4KB*4B=4MB。如果处理器只运行一个程序，那这个存储空间是可以接受的，但是假如一个处理器同时处理上百个进程，并且每个进程都需要一个4MB的物理空间存放页表，其代价就变得不可接受。事实上，一个程序在运行时很难占用整个虚拟空间，这就使得其页表大部分内容是空的，并没有使用，如果把他们都存在物理内存中就会产生很大的浪费。

为了减少进程页表对存储空间的需求，最常用的方法是使用多级页表（Hierarchical Page Table），而本节介绍的即单级页表（Single Page Table）或线性页表（Linear Page Table）。

多级页表

核心问题：解耦单级页表

下图演示了多级页表的结构。它将上一小节中的一个页表转化为了需要查询两次的页表，并且每个页表变得更小。实际上，它将VPN分为了两个字段P1和P2，从而分别对应两级页表中的偏移。图中的PTR存放着第一级页表（Level 1 Page Table）在物理内存中的起始地址。在转化时，字段P1在第一级页表中进行查询，得到的内容是第二级页表的起始地址，字段P2再根据查询到的第二级页表起始地址查询到真实的物理地址。

二级页表

多级页表的优点

页表存放灵活：每一张页表都需要连续的物理空间进行存放。单级页表只有一张表格（4MB），因此操作系统需要在存储空间中找到连续的4MB，而多级页表将表格拆分成多个页表，如图中每个页表的大小仅4KB，可以将各个表放在不连续的空间，增大了物理内存的使用灵活性。
节省存放页表空间：大部分的程序并不需要使用全部4GB的虚拟空间，对于暂未使用的虚拟地址，操作系统可以不为其创建次级页表。编译器在编译程序时，可以为程序创建连续的虚拟地址空间，这样就不需要所有的页表表项。比如，图中，假设某一虚拟地址经过第一级页表查询，得到第二级页表的应该使用第一个，那操作系统只需要创建出第二级页表中的第一个页表即可将虚拟地址转换成物理地址。
易于扩展页表级数：我们在图中讨论的处理器位宽为32bit，当处理器位宽增加时，只需要为页表在加上几级，即可进行扩展。
页表功能扩展

核心问题：MMU的其他功能

在前几节介绍了MMU地址转换，以及页表记录映射关系。实际上，MMU还可以通过在页表中增加其它信息以实现更多的功能，比如权限管理等。




表项内容

图中显示了扩展后的页表内容，其中valid用于表示当前的虚拟地址是否存在和物理地址的映射关系，当valid为0时表示映射关系不存在，也就是物理内存中不存在当前虚拟地址对应得内容，需要发起page fault异常。在发起page fault异常后，操作系统需要找到一个不常用的映射关系，将从下一级寻找到的需要放入物理内容的内容替换其frame，替换时还需判断该frame的状态是否为脏（dirty），即是否被修改过，如果修改过则需要先将被替换的frame内容同步到下一级物理存储中。

对于页表选项，可以使用一个Use标志位用于表示其最近是否被使用过。MMU可以设置定期对其进行更新，当需要替换时，就可以根据Use的值判断哪个映射表项最近未被使用。

MMU除了地址映射，还能进行权限管理。在ARM中，页表中设置3bit的AP编码，不同的AP编码表示了当前页表不同的使用权限。比如在‘b011时表示Privileged模式和User模式均可以读写，而'b011时只有Privilege可以读写当前表项对应的物理地址。

虚拟地址转换成物理地址后需要访问Cache以获得数据，而在处理器中，一部分的存储映射（memory map）是不允许被缓存的。比如外设寄存器，因为它需要直接给寄存器赋值以控制外设的工作，如果该值被缓存则被保存在了D-cache而不是直接赋值。因此在虚拟地址转换时还需要注意该地址是否是被缓存的，如果是可以缓存，则使用转换后的物理地址访问Cache，否则直接访问物理存储空间或者外设寄存器。

TLB
直接使用页表存在的问题

由于页表存放在物理内存中，因此每次访问页表就需要访问一次物理内存。对于2级页表，下图表示了其访问物理内存的流程。首先PTR和虚拟字段的P1进行第一次物理内存访问，以获得第一级页表中对应的二级页表的起始地址，然后字段P2根据二级页表起始地址查找到对应的frame地址，这就需要两次访问物理内存才能得到物理地址。


2级页表访问物理内容

​我们知道，处理器访问物理内存的代价较大，因此在物理内存和处理器流水线之间需要缓存（Cache）进行缓冲，Cache利用了时间相关性（Temporal Locality）和空间相关性（Spatial Locality）原理，将最可能会被使用的一小部分内容从物理内存中复制单独存放，因为查询较小的空间往往会有更快的速度。

页表也需要缓存

既然查询页表是通过查找物理内存以获得信息，那也可以使用类似Cache的技术将最可能使用的页表信息单独存放，这就是TLB（Translation Lookaside Buffer），实际上，TLB就是PTE的Cache，只不过TLB只有时间相关性，即现在访问的页后续很可能在以后继续被访问，而对于空间相关性并没有表现出明显的规律。

在现代的处理器中，很多都采用两级TLB，第一级采用哈佛结构，分为指令TLB（I-TLB）和数据TLB（D-TLB），一般采用全相联模式。而第二级TLB是指令和数据共同，一般采用组相联模式，这和多级Cache的设计方法类似。


TLB内容

​图中展示了一个全相联方式的TLB的内容，它和页表的表项相比，多了VPN的选项，但是需要注意，这里的VPN不是用作地址进行索引，而是作为内容的一部分判断是否命中（hit），即根据内容进行匹配。

假设VPN hit并且Valid为1，，则直接使用TLB中包含的PFN，否则就需要在物理内存中查找，如果物理内存中也不存在该映射关系，就需要发出Page fault异常进行处理了。

由于TLB实际上属于Cache，因此当发生TLB缺失时的操作和Cache操作是异曲同工的，其设计也遵循Cache设计，因此这里不进行过多赘述。不过这里有一个词需要注意，Page Table Walk，它指的是当TLB发生缺失时要从页表中找到对应的映射关系，并将其写回TLB的过程。

TLB与物理内存的Cache

前面提到的TLB时页表的Cache，设计TLB是因为直接访问页表需要访问物理内存，效率不高，同样的，当虚拟地址转换为物理地址后，执行load或者store的过程也并非直接对物理内存进行操作，而是对物理内存的Cache进行操作。现在我们可以捋清楚虚拟地址到物理内存的过程了，如图所示，其中虚线表示物理Cache和物理内存的数据一致性维护过程。


现在我们发现，CPU在对物理Cache进行操作时，需要先经过TLB进行地址转换，这就加了CPU到物理Cache的时间。在处理器设计中，我们希望CPU能更快的拿到数据而不是陷入等待。那既然使用了虚拟地址最终拿到数据，那不如将Cache直接使用虚拟地址进行索引，如图。CPU直接使用虚拟地址访问虚拟Cache进行操作，只有虚拟Cache发生缺失时，再经过TLB转化为物理地址，然后再从物理内存中获得对应的数据。

使用虚拟Cache在流水实现上不会对处理器的时钟产生明显的负面影响，但是也会产生新的问题，这需要额外的硬件进行解决。直接使用虚拟Cache主要引入了同义与同名问题。

同义问题：多个不同的虚拟地址可能对应相同物理地址。这种情况会造成虚拟Cache的浪费，因为多个存放空间可能是相同的内容，另外就是假如其中某一个虚拟地址Cache内容进行了修改，就需要对其他有着相同物理地址的位置也进行修改，否则其他虚拟地址无法处理到最新的值。
同名问题：不同进程相同的虚拟地址对应不同的物理地址。当切换进程时，新的进程可能会从虚拟Cache得到上一进程的数据从而错误。——解决：为每个进程加上PID（Process ID）编号，或者称为ASID（Address Space IDentifier），把编号作为虚拟地址的一部分。
