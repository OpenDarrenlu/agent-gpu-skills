# [CuTeDSL防晕] 无B200卡 从0开始调优GEMM到1840TFLOPS（一）

---

我计划写一个系列文章，记录如何将一个 GEMM kernel 从基础版本，逐步调优到目前 B200 上的最优性能（1840 TFLOPS）。相关的代码已开源在：GitHub - deciding/cutez。用的modal直接安装modal就能跑，无需B200。


目前能参考的 CuTe 范例，大多要么是高度完善的工业级实现，要么是较为基础的 Demo，较少有系统性展示调优思路和中间形态的版本。本系列希望填补这一空白。

Disclaimer：1840 TFLOPS是根据cutlass tutorial默认的设置来的，它使用了randint。目前cutez可以在01_dense_gemm.py设置INIT_MODE="gaussian",此时性能为1400+TFLOPS。目前modal测试还没有找到锁频的方法，性能会有波动。


TL;DR：本期摘要


当前进展：本文实现了一个“麻雀虽小五脏俱全”的基础版本https://github.com/deciding/cutez/blob/main/blackwell/dense_gemm_1.py（这个版本性能约 340 TFLOPS，最终版本是1840TFLOPS）。初期的重点不在于极限性能，而在于建立清晰的编程模型。
核心问题：CuTeDSL 极高的自由度容易导致“形状眩晕”和难以排查的潜在 Bug。
优化策略：通过主动放弃部分自由度，建立严格的命名规范与形状切分范式，能够显著降低出错概率，并在多层切分后依然保持代码的高可读性。
关键结论：
命名规范 ([层级][操作/结果][数据源])：例如 tCgA 代表当前 Thread 为了执行 MMA C 乘法操作，从 Global Memory A 中切分出的 Tile。
API 的隐性输入输出约束：
local_tile ：输出是平铺的
partition_*：通常接受 Flat（平铺） 形状的输入。
tma_partition：不接受 Flat 形状，其输入必须是经过 group_modes 等操作后的层级化（嵌套）形状(tma_tile, *Rests)。
形状的逻辑层级 (Shape Hierarchy)：矩阵切分在物理与逻辑上遵循 Block (TMA) > MMA > EPI > Store 的递进关系，理解这一链条是推演最终 Tensor 形状的基础。
1. 痛点：CuTeDSL 开发中的认知负担

在编写 CuTeDSL 时，一个显著的痛点是：Tensor 的 Size 往往难以直观理解。即使通过打印输出，看到类似 ((64,1), 1, ((1,4), 1, 1)) 的形状嵌套，也会带来较大的认知负担。
这种现象的根源在于自由度过高。以 TMA、MMA 或 TMEM 的形状操作为例，由于 API 允许极高的灵活性，开发者往往可以通过反复调试拼凑出一个正确的结果（例如将上述形状改为 ((64,1), 1, 1, (1, 4, 1, 1)) 同样可以运行）。但代价是代码可读性的急剧下降。
此外，相较于 Tile-based DSL（如 Triton，开发者主要关注 CTA 层级的切分），CuTe 暴露了大量 SIMT 层面的内存与同步细节。例如在 2SM 的 MMA 架构下，TMA mbar 的 expect_tx 应如何动态调整？何时判断 is_leading_cta？Consumer 线程数如何分配？这些细节极易引发错误。
因此，编写 CuTe 是一项极具挑战的工作。市面上有许多介绍 CuTe 语法的指南，但如果不置于具体的算子语境中，很难产生深刻的理解。
本系列被命名为【防晕】系列，核心目的在于：在编写代码的初期，就明确 Tensor 的形状、API 的预期输入，以及 Producer 与 Consumer 之间的协作模式。 这样我们才能将精力集中于性能调优，而非陷入无意义的 Debug 中。


2. 规范一：建立结构化的命名范式

我的第一个建议是：主动限制 CuTeDSL 的部分自由度，形成一套固定的命名和 Shape 风格。 以下是我的编码规范（该规范与官方的部分优秀示例保持一致）：
当我们看到 tCgA 或 tAgA 时，它们分别代表什么？

后缀（数据源与内存层级）：gA 代表 Global Memory 中的矩阵 A，sA 代表 Shared Memory，rA 代表 Register。在 Blackwell 架构中，tA 代表 TMEM。
前缀（线程层级）：t 代表 Thread 级别，b 代表 Block (CTA) 级别。
中间字母（操作属性）：在 tAsA 中，首个 A 并非指代“矩阵 A”，而是代表一个 Op（操作）。这带有编译器中间表达（IR）的思想：一个标识符既代表 Value，也代表生成它的 Op。此处的 A 代表 tma_load A 操作。同理，tCgA 中的 C，代表的是 C = A x B 这个 MMA 乘法操作。

综合解析：
tCgA 的含义是：为了满足当前 Thread 中 MMA 乘法操作 (C) 的需求，从 Global Memory A (gA) 中切分出来的 Tile 视图。
(注：这并不意味着 GEMM 是直接从 Global Memory 读取 A 进行计算，而是表示 tCgA 描述了 MMA 所需的数据布局。随后，该需求会被传递给 tma_partition，进而生成 tAgA，用于实际的 TMA load 操作。)
其他操作的命名原则：
对于 tma_store、tmem_load、smem_store 等操作，建议放弃“以结果命名”的方式，转而直接描述数据流向。例如，用 tRSsC 代表从 Register (R) 到 SMem (S) 的 Store 操作,这个操作在本thread需要的sC上的部分就是tRSsC。
对于 Block 层面的切分，虽然 TMA 在逻辑上管理整个 Block，严格来说应使用 b 前缀，但鉴于当前社区已有大量使用 tAsA 的习惯，为降低阅读成本，建议统一使用 t 作为前缀。


对于不涉及partition的原始 Tensor：

mC：代表整个矩阵。
gC：代表当前 CTA 负责的全局内存切片。
sC：代表 Shared Memory 上的指针。

补充：还一个情况是cA这种，指的是coord A，一般用 cute.make_identity_tensor(A.shape)来创建，结果是一个 坐标矩阵。一般用来做masking。

(0, 0), (0, 1), (0, 2)....
(1, 0), (1, 1), (1, 2)...
...

和上述都不一样，c并不代表memory层级，而是一个功能性描述，针对A创建的coord。

3. 规范二：厘清形状变换 API 的输入输出约束
与 Triton 提供的 splat, split 等宏观视角操作不同，CuTe 需要在 Thread 和 Op 维度进行细粒度控制。以下是三种最基础的 Divide 操作：

tiled_divide : ((TileM, TileN), RestM, RestN, L, ...)
zipped_divide: ((TileM, TileN), (RestM, RestN, L, ...))
flat_divide : (TileM, TileN, RestM, RestN, L, ...)

这三者在字面意义上较易理解，但关键在于它们在实际 API 中的隐性约束。

补充：理论上来讲在cutlass cpp里

logical_divide : ((TileM, RestM), (TileN, RestN), L, ...)

但是在cutedsl我亲测和tiled_divide完全是一个效果。不知道是不是bug。


一、 local_tile：结合坐标的切片与展开
local_tile 内部调用了 zipped_divide，并在切分后根据 coord 获取 Rest 部分的数据，最后将 Tile 部分展开。因此，local_tile 的返回值始终是一个 Flat（平铺）格式的 Tensor。
示例推演：
mA.shape: (1024, 256), tiler: (128, 64), coord: (0, None)
演变过程：zipped_divide(128, 64), (8, 4) -> indexing(128, 64), 4 -> 展开128, 64, 4


常规的两种使用模式：

传入具体 coord：优势在于可以利用 proj 参数精准定位当前 CTA 的 Tile。
传入 (None, None, ...)作为coord：优势在于编写 Persistent Kernel 时，可提前定义好 gA 的布局，在循环体内部根据动态 coord 获取 Tile。

二、 thr_op.partition_*：线程级别的细粒度切分
Cutlass 的核心 Op 主要是 Copy 和 MMA（包含众多如 Ld16x64bOp, MmaF16BF16Op 的子类）。其标准的构建流程为：op_atom -> tiled_op -> thr_op。

举个例子：

tmem_atom = cute.make_copy_atom( 
 tcgen05.Ld32x32bOp(tcgen05.Repetition.x64),
 cutlass.Float32, 
)

其实就是对应了一个ptx
tcgen05.ld.sync.aligned.32x32b.x64.b32

tmem_tiled_copy = tcgen05.make_tmem_copy(tmem_atom, tensor)

就是根据tensor提供的layout信息创建一个专门为tensor适配的CopyOp

最后

tmem_thr_copy = tmem_tiled_copy.get_slice(tidx)

获取了当前thread的CopyOp，我们可以用它来切出当前thread下的tile

tTRtC = tmem_thr_copy.partition_S(tCtAcc_epi)


核心约束：
假设传入 partition_* 的形状是平铺的 (M, N, L)，其输出必然是：
(atom_tile, RestM, RestN, L)

这个atom_tile是各个op都不一样的，比如mma可能是(Atom_MMA_M, Atom_MMA_K), e.g. (128, 16)

而tmem load可能是((Atom_TMEM_V, Atom_TMEM_T), Rest_TMEM_V), e.g. ((64, 32), 1)

但是我们只要把细节忽略掉只看抽象的情况下，他是一个非常清晰的一个结果。和local_tile的区别是它的atom_tile是不展开的。


三、 tma_partition：具有严格输入格式的 API
一个常见的误区是将 tma_partition 与 partition_* 等同视之。实际上，两者的输入要求存在本质区别。
输入约束：tma_partition 通常不接受 Flat 形状的输入，它需要一个经过 tiled_divide 或者group_modes处理后的层级化形式，即 (tma_tile, RestM, RestN, L)。这解释了为什么在代码中，往往需要先用 cute.group_modes 对 local_tile 的 Flat 结果进行打包，再将其喂给 tma_partition。
输出结果：其输出结构依然是 (tma_tile, RestM, RestN, L)，区别在于内部的 tma_tile 被转换为了符合 TMA 硬件要求的 atom_tile 格式。结构的嵌套层级并未发生改变。

举例来说

a. local_tile的结果如果是

(128, 64, 16) -> (bM, bN, RestK)

b. 我们先group_modes成

((128, 64), 16) -> ((bM, bN), RestK) -> (tma_tile, RestK)

c. 再tma_partition

((64, 128), 1), 16) -> (tma_atom_tile, RestK)

如果我们把(128, 64) 理解成tma_tile，((64, 128), 1)其实只是符合tma atom tile格式的tma_tile而已。形状前后其实是没有变化的。

这里要注意很多代码在步骤b会用tCgA而不是gA来进行操作

((128, 16), 1, 4, 16) group_modes 成 (((128, 64), 1, 4), 16) -> ((mma_atom, *mma_rests), RestK)

但实际上(mma_atom, *mma_rests)依然是tma_tile，所以tma_partition得到的结果是一样的，大家可以试一试。


4. 终极验证：解析多层切分后的 Shape 层级
在经历了一系列复杂的 API 调用后，如何确保最终的 Shape 符合预期？这要求我们在逻辑上建立清晰的层级结构（Hierarchy）：
Block (TMA) > MMA > EPI > St


Block/TMA 层级：每个 Block 处理一个较大的 Tile，TMA 执行 CTA 级别的 Load 操作。
MMA 层级：Block Tile 会被进一步切分为多个 MMA 块来执行（例如 K 维度的上限为 16，若 BLOCK_K 为 64，则需要执行 4 次 MMA）。
Epilogue (EPI) 层级：为了缓解寄存器压力，MMA 的计算结果通常需要人为使用 flat_divide 切分为多个批次进行 Epilogue 操作。
Store (St) 层级：最后，在将数据写回时，需根据具体 Store Op 的硬件指令限制，进一步切分执行。

映射到最终 Tensor 的形状上，它会呈现出从内到外严格的嵌套形态：
(st_atom, *epi_rests, *mma_rests, *block_rests)
典型链路推演：
示例 1：矩阵 A 的数据流转 (设 M, N, K = 1024, 1024, 1024, bM, bN, bK = 128, 256, 64)

gA (No partition): (128, 64, 16) -> 对应 (*block)
tAgA (经 TMA 切分): (((64, 128), 1), 16) -> 对应 (tma_atom ((64, 128), 1), *block_rests)
tAsA (SMem 视角): ((8192, 1), 3) -> 对应 (tma_atom (8191,1), *buffer)(注：SMem 空间受限，通常以 Software Pipeline 的 buffer size 替代 block reduction size)
tCgA (MMA 视角的读取需求): ((128, 16), 1, 4, 16) -> 对应 (mma_atom (128, 16), *tma_rests (1, 4), *block_rests (16))
tCrA (Register 存储的 SMem 描述符): (1, 1, 4, 3) -> 对应 (mma_atom (1), *tma_rests (1, 4), *buffer (3))

注意 标注 no partition的意思是这一步没有经过partition的操作，而且接下来是需要传入一个partition操作的，所以形状会是全展开的。

需要注意tCrA虽然是register，但是存放的实际上是smem descriptor, 所以整个mma_atom就会是1的样子。

示例 2：矩阵 C 的 Epilogue 与写回 (设 bM, bN, bK = 256, 512)


gC (No partition): (256, 512) -> 对应 (*block)
tCgC (MMA 视角对 C 的切分): ((128, 256), 2, 2) -> 对应 (mma_atom (128, 256), *block_rests)
tEPIgC (EPI 切分视角): (128, 64, 1, 4, 2, 2) -> 对应 (*epi_tiler (128, 64), *mma_rests (1, 4), *block_rests)
tTRtC (Store 操作前): (((64, 32), 1), 1, 1, 1, 4, 2, 2) $\to$ 对应 (st_atom ((64, 32), 1), *epi_rests (1, 1), *mma_rests (1, 4), *block_rests)
tTRrC (Register 写回视图): ((64,1),1,1) -> 对应 (st_atom (64, 1), *epi_rests)
(注：在 tTRrC 阶段，为了在循环中复用寄存器，我们通常会将 *epi_rests, *mma_rests, *block_rests 合并为一个平铺的循环体（Flat Loop），因此寄存器视图的形状中不再包含这几层外部维度。) (注2：(64,32)代表的是一个warp，所以从128变成了32，其实形状没有变)

结语
通过将形状和变量名纳入这套严谨的规范体系中，我们能够在很大程度上消除底层开发的盲目性。当你阅读变量名或检查其 Shape 时，便能迅速映射出它所处的硬件操作层级。
掌握了这种“防晕”的模型，我们在后续引入 Software Pipelining、TMA Multicast 等高级优化策略时，才能确保逻辑依然清晰可控。下一篇，我们将在此基础上，正式开启向 1840 TFLOPS 性能迈进的调优过程。
