0. 前言

近些年来，随着深度学习模型的广泛应用，越来越多的模型需要被部署在不同的硬件上。这项工作需要大量工程师的努力，才能使得模型能够高效地部署在各种后端上（GPU，Edge Device，加速器）。对此TVM尝试并且在一定程度上较好地解决了这个问题。（图自https://tvm.apache.org）

TVM通过Unified IR将不同的前端和后端联系起来

但是随着硬件专业化的浪潮到来，问题变得更加棘手。现在越来越多的硬件不再简单采用传统的通用计算（CPU的标量计算和GPU的向量计算），转而采用专用的加速器，来加速特定的计算（通常加速单元针对矩阵乘法和卷积有特殊加速）。这些计算指令通常基于多维的张量计算，也给机器学习系统和编译器的优化带来了巨大的挑战。为了充分利用硬件特性，通常我们需要在张量计算程序(Tensor Program)中添加张量计算指令(Tensor Intrinsic)。我们把这种带有张量计算指令的程序称为张量化程序(Tensorized Program)。

硬件加速不断专业化，对张量计算提供加速
1. TensorIR简介

TensorIR的设计初衷就是为了解决Tensorized Program的编译优化问题。我们把编译优化问题分为三个方面：

怎么写Tensorized Program
怎么优化Tensorized Program
怎么把同一个Tensorized Program部署在不同的硬件后端上

对此，TensorIR也针对性地提供了解决方案（接下来分章节详细介绍）：

TVMScript：用基于Python AST的语法来TensorIR从而表达Tensorized Program。
交互式Schedule：通过Schedule Primitive实时优化Tensorized Program。
独立的Schedule Primitive：方便开发者针对不同硬件添加特殊的Primitive。
2. TensorIR 编程范式

针对深度学习领域的常见的编程范式目前有两种：

手写算子（cuDNN，cutlass等）
算子声明+Schedule（TVM Tensor Expression等）

前者需要强大的专业知识，并且工作量巨大；后者表达能力和优化空间有限，针对复杂算子或复杂张量指令支持有限。

TensorIR采取了全新的编程范式：手写算子+Schedule。这种方式结合了手写算子的表达能力，以及（Auto）Schedule带来的便利。用户可以根据算子的计算方式，以及后端指令的需要，先写出一个能够正确运行的结果，再借助Schedule的能力去提升性能。针对一些简单算子（如gemm），Tensor Expression已经能够很好地表达了，TensorIR也支持了从TE导入的能力。

如下图，TensorIR以TVM IRModule为载体，从TVMScript中导入，并且可以不断进行Schedule变换，以及Pass变换，最终将优化后的IRModule build成为一个能够在不同硬件上运行的Module。

3. TVMScript

TVMScript是一种基于Python语法的脚本语言（类似torchscript），用来表达TVM TensorIR。这是一种round-trippable的语法，这意味着我们不但可以从TensorIR打印成TVMScript，也可以随时将TVMScript Parse回IR本身。需要注意的是，TVMScript仅仅使用了Python的语法和AST，并没有通过Python Interpreter，因此所有在TVMScript外定义的变量均无法在TVMScript内部解析。

下面是一段用TVMScript写的针对向量计算 C = exp(A+1) 的代码：

# C = exp(A + 1)
from tvm.script import tir as T

@T.prim_func
def fuse_add_exp(a: T.handle, c: T.handle):
    # Define multi-dimensional buffer
    A = T.match_buffer(a, (64,))
    C = T.match_buffer(c, (64,))
    B = T.alloc_buffer((64,))
 
    # Loop nests
    for i in range(8):
        with T.block("B") as [vi]:
            # Block representing vectorized/tensorized 
            # computation Add 8 elements at a time
            vi = T.axis.S(8, i)
            T.reads(A[vi * 8: vi * 8 + 8])
            T.writes(B[vi * 8: vi * 8 + 8])
            for k in range(8):
                B[vi * 8 + k] = A[vi * 8 + k] + 1 

    for j in range(64):
        with T.block("C"):
            vi = T.axis.S(64, j)
            C[vi] = exp(B[vi])
4. 针对Tensorized Program的交互式优化

在Schedule层面，TensorIR相较于现有TVM TE Schedule有了两个主要的改进点：Tensorized Program支持和交互式Schedule

Imperative Scheduling（交互式Schedule）

TE Schedule是由Halide IR改进而来，其核心是基于Schedule Tree的静态声明模式，在初期通过声明一系列Primitives，并在最后生成代码的时候一起把目标IR输出。TensorIR采用的是实时动态的变换模式，每一个Primitive即时作用于IR本身，修改IR。

这类似于Tensorflow 1.x（静态）和PyTorch（动态）的区别，我们相信Imperative Scheduling会给开发者和研究者带来更好的易用性。

原生的Tensorized Program支持

Halide从设计本身就是基于标量计算设计的优化工具，虽然TVM在其基础上增加了和Tensorize的支持。但其用的是匹配替换的方式做的支持，使得Tensorize之后的代码就无法再做任何修改，大大增加了Tensorized Program的优化难度。TensorIR将张量计算作为First-class citizen，做到了原生支持。在Schedule的任何阶段，都可以对任何张量计算做变换和优化。

5. TensorIR 独立Primitive

TensorIR针对CPU和GPU提供了绝大多数常用的Primitive。通过这些Primitive能够生成充分利用硬件性能的程序（目前已经实现了与Cutlass齐平的性能）。当然，针对其他加速器，这些Primitive通常来说是不够的，需要额外添加Schedule Primitive也是常有的情况。

TensorIR提供了完全独立的Primitive实现方式，使其不依赖于任何独立数据结构，仅仅通过IRModule本身就能完成Primitive的实现（类似Pass）。下面是一个典型的Primitive的实现方式：

小结

TensorIR作为TVM社区全新的Low-level IR，采用了全新的编程范式。在易用性、可扩展性相较于现有TVM TE Schedule有了巨大的进步。并且添加了Tensorize Intrinsic的原生支持，针对硬件加速器和加速单元有了更好的支持。

目前TensorIR将作为experimental feature在TVM v0.8发布，感兴趣的同学可以去体验一下。目前我们也在继续为TensorIR开发meta-schedule(auto-scheduling和auto-tensorization算法)，更多更新请关注TVM社区和TVMCon 2021。本系列后续文章也将持续更新。
