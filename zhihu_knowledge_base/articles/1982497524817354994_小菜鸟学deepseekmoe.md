什么是MOE？

经典的transformer模型是核心是attention+ffn，这个称作tranformer block。这里ffn是线性层，结构越复杂模型的表达能力越强。dense模型时代，ffn通常是2层线性层，先做up projection（上投影）维度变成intermediate_size（llama通常是hiddendim的4倍），经过激活函数之后在做down projection（下投影) 恢复到hiddendim。ffn层权重矩阵通常是很大的矩阵，在inference阶段计算代价就很大。

deepseekmoe论文证明引入moe能增加参数量可以有效控制inference计算代价。moe带来变化就是集中的厚ffn层变成了分布式一组小ffn，每个ffn被称作expert。deepseek把expert分成2类：

shared expert：通用知识，全局只有1个
routed expert：256个，每个token在做ffn（mlp）只激活8个

moe带来价值：计算成本是模型参数的1/32。ffn或者mlp在模型权重占比最高，这部分计算成本节省对整体有直接价值。从deepseek v3/R1之后，moe成为llm标配，也把推理从单机时代推向了分布式推理时代。所以，菜鸟也需要了解一下这方面。

假定在做一个inference，attention之后要去做moe mlp，从上面sglang代码看出，选择topk部分forward中一步。

moe计算包括一下4个步骤:

select topk（done）：发生local
all-to-all dispatch：分布式
routed experts计算(mlp)：发生在local/remote（任意一个node）
all-to-all combine：分布式

其中，1和3都是单个node上事件；2和4是分布式事件

all-to-all dispatch

多对多分布式通讯范式：每个node定义好如下数据：

自己要发送那些token（index），发送给谁（dest rank，dest expert），我是谁（cur rank）
发送 (通过数据传输协议)
接受其他node发来做expert计算的token，计算（expert mlp），然后ack给发送端

这么看是一坨代码吧，例子代码并不是这样。它利用torch的分布式原语，通过对称式广播交换每个node对其他node信息。

写过很多procedure programming，有些api喜欢返回一个动态的buffer，调用时候需要先用NULL指针调用一遍得到bufsize，分配buffer，然后再用分配好的buffer调用一遍就可以了。torch.all_to_all_single调用方式逻辑（套路）跟上面动态buffer类似。

为什么3部分需要2个all_to_all_single，第1个all_to_all_single后我们知道buffer中每个部分边界，但是不清楚每段buffer数据结构。后面的all_to_all_single交换元数据，相当于告诉对方数据结构定义。

在这个case里，send_buf和recv_buf中保存的token数组，token是embedding过的，每个token都是hiddendim长度的。更正一下：最后一个all_to_all_single中交换的元数据只用来combine阶段把做完expert计算的token发回去用的。

# 1.3 token nums to recv from each rank
  dist.all_to_all_single(recv_counts_t, send_counts_t) # 同步调用，互相交换发送和接受token个数

  dist.all_to_all_single(
      recv_buf,
      send_buf,
      output_split_sizes=recv_counts_t.tolist(),
      input_split_sizes=send_counts_t.tolist(),

dispatch和combine含义

dispatch：交换数据，函数返回后每个node都得到它需要处理的token

combine：当前node把它要处理的token拿去做ffn（mlp）之后需要把数据发回个sender的rank继续做下一层的attention计算。为什么要这样？因为kvcache缓存之前k/v结果，kvcache是分层的，推理下一个token时候需要reuse之前结果。

combine做了dispatch类似事情，元数据处理相对简单。dispatch从sender方式处理每个token的元数据，在3个all-to-all-single之后获得receiver端meta数据，在combine阶段token发回的元数据就是dispatch返回的那个。

combine结果是把从不同node做完expert计算结果converge（乘以权重，topk时候得到的）到out_tokens。对，这才是attention结果做moe ffn得到的结果。

torch.all_to_all_single是pytorch提供的一个分布式通信原语，底层可以有不同实现方式。从代码上torch的all_to_all_single交换数据需要存储在一个连续buffer里，代码里繁琐的python中list of list整理代码。
