3.3 使用高级分支预测减少分支预测成本/Reducing Branch Costs With Advanced Branch Prediction

由于需要通过分支冒险和停滞来强制执行控制依赖关系，分支会对流水线性能造成影响。循环展开是减少分支冒险数量的一种方式；还可以通过预测分支行为来减少性能损失。

关联分支预测器

2位预测方案仅使用单个分支的最近行为来预测该分支的未来行为。如果不仅仅关注要预测的分支，而是还考虑其他分支的最近行为，可能可以提高预测准确性。考虑来自eqntott基准测试的一个代码片段，它是早期SPEC基准套件的一部分，展示了特别糟糕的分支预测行为。

使用其他分支的行为来进行预测的分支预测器被称为关联预测器或两级预测器。现有的关联预测器通过添加最近分支的行为信息来决定如何预测给定的分支。

例如，（1, 2）预测器根据最后一个分支的行为，从一对2位的分支预测器中进行选择，来预测一个特定的分支。在一般情况下，（m, n）预测器使用最后m个分支的行为来从2^m个分支预测器中选择，每个分支预测器都是针对单个分支的n位预测器。

这种类型的关联分支预测器可以提供比2位预测方案更高的预测率，仅需要一小部分额外的硬件。

简单的观察发现：最近m个分支的全局历史可以记录在一个m位移位寄存器中，其中每个位记录该分支是被执行还是未被执行。然后，分支预测buffer可以使用分支地址的低位与m位全局历史的串联来进行索引。

例如，在一个总共有64个项的（2, 2）buffer中，分支的低4位地址（word地址）和表示最近执行的两个分支行为的2位全局位组成6位索引，可用于索引64个计数器。将局部信息和全局信息组合起来，可以使用结果索引预测器表，并且可以像标准的2位预测器一样快速获取预测结果。

Tournament预测器：自适应地结合局部和全局预测器

关联分支预测器的主要动机来自于一个观察结果，即仅使用局部信息的标准2位预测器在某些重要分支上失败了，增加全局历史信息有助于改善这种情况。

Tournament预测器将这种见解提升到了一个新的水平，它使用多个预测器，通常是一个全局预测器和一个局部预测器，并通过选择器在它们之间进行选择，如图3.5所示。全局预测器使用最近的分支历史来索引预测器，而局部预测器则使用分支的地址作为索引。

Tournament预测器可以在中等大小（8K～32K位）时实现更高的准确性，并能有效地使用非常大数量的预测位

锦标赛预测器的优势在于其能够为特定的分支选择正确的预测器。

带标签的混合预测器

截至2017年，表现最佳的分支预测方案涉及将多个预测器组合起来，这些预测器跟踪预测是否与当前分支相关联。预测器的一个重要类别是基于一种名为PPM（通过部分匹配进行预测/Prediction by Partial Matching）的统计压缩算法。PPM就像分支预测算法一样，试图根据历史来预测未来的行为。这类分支预测器称为带标签的混合预测器，使用一系列的全局预测器，这些预测器使用不同长度的历史进行索引。

如图3.7所示，一个带标签的混合预测器包含五个预测表：P(0)、P(1)、... P(4)，其中P(i)通过PC的哈希值和最近i个分支的历史（保存在移位寄存器h中，与gshare相同）进行访问。

使用多个历史长度来索引单独的预测器。
在P(1)到P(4)的表中使用的标签可以很短，因为不需要100%的匹配（4～8位的小标签可以获得大部分优势）。
只有当标签与分支地址和全局分支历史的哈希值匹配时，才会使用来自P(1)、... P(4)的预测。
P(0...n)中的每个预测器都可以是标准的2位预测器。3位计数器（需要三次错误预测才能改变预测）比2位计数器好一些。
图3.7 一个由五个组件组成的带标签的混合预测器包含五个单独的预测表，这些表通过分支地址的哈希值和长度为0～4的最近分支历史段（在图中标记为“h”）进行索引。哈希可以像gshare中的那样简单，只是一个异或操作。每个预测器都是一个2位（或可能是3位）预测器。标签通常是4～8位。

给定分支的预测是具有最长分支历史且标签也匹配的预测器。P(0)总是匹配，因为不使用标签，如果P(1)到P(n)中的任何一个都不匹配，将成为默认。

标记混合预测器（有时被称为TAGE——Tagged GEometric预测器），这类预测器在内存使用适度（32~64 KB）的情况下表现优于gshare和Tournament预测器，此外，这类预测器似乎能够有效利用更大的预测缓存来提高预测精度。

3.4 使用动态调度克服数据冒险/Overcoming Data Hazards With Dynamic Scheduling

一个简单的静态调度流水线会抓取指令并发射它，除非在流水线中已经存在的指令与获取的指令之间存在无法通过bypassing或forwarding隐藏的数据依赖关系（forwarding减少了有效的流水线延迟，使得某些依赖关系不会导致冒险）。如果存在无法隐藏的数据依赖关系，则冒险检测单元将从产生依赖的指令开始暂停流水线，直到依赖关系被清除。

动态调度技术，通过重新排序指令执行来减少停滞，同时保持数据流和异常行为。

动态调度具有几个优点：

允许某个流水线编译的代码在不同的流水线上高效运行，消除不同微架构拥有多个二进制文件和重新编译的需要
能够处理一些在编译时未知的依赖关系
允许处理器容忍不可预测的延迟，例如在等待缓存未命中解决时执行其他代码

虽然动态调度的处理器不能改变数据流，但它会尝试在存在依赖关系时避免停滞。相比之下，编译器进行的静态流水线调度试图通过分离相关指令来最小化停顿，从而避免产生冒险。

动态调度：想法

在简单流水线技术中的一个主要的限制是使用顺序发射指令和执行：指令按照程序顺序发射，如果某条指令在流水线中停滞，那么后续的指令就无法继续执行。因此，如果流水线中两个相邻的指令之间存在依赖关系，就会引发冒险，并导致停顿。如果存在多个功能单元，这些单元可能会处于空闲状态。

如果指令j依赖正在流水线中执行的指令i，那么指令j之后的所有指令都必须暂停，直到指令i执行完成。例如以下代码：

在典型的5级流水线中，结构冒险和数据冒险都可以在指令解码（ID）阶段进行检查：当一条指令可以无冒险地执行时，在所有所有的数据冒险都已解决情况下从ID阶段发射。

为了让能够开始执行前面例子中的fsub.d指令，必须将指令发射过程分为两个部分：检查是否存在结构冒险、等待数据冒险消失。因此，仍然使用程序顺序发射指令，但希望指令在其数据操作数可用时就开始执行（流水线进行乱序执行，这意味着乱序完成）。

乱序执行引入了WAR（写后读）和WAW（写后写）冒险的可能性，而这在5五级整数流水线及其逻辑扩展到顺序浮点流水线中是不存在的。思考下面的RISC-V浮点代码序列：

PS：寄存器重命名（register renaming）在乱序执行中常用，可以解放寄存器的压力。

乱序完成也会给异常处理带来很大的难题。动态调度的处理器通过延迟相关联的异常通知直到处理器知道该指令需要在接下来被完成，来保留异常行为。

动态调度的处理器可能会产生不精确的异常。不精确的异常可能有以下两种情况：

流水线可能有已经完成的指令，但是在程序顺序上比导致异常的指令晚。
流水线可能有还没有完成的一些指令，并且在程序顺序上比导致异常的指令早。

不准确的异常也会导致在异常之后很难重启执行。

为了实现乱序执行，将简单的5级流水线中的ID（指令解码）阶段划分为两个阶段:

发射：指令解码，结构性冒险检查。
读操作：等待直到没有数据冒险，读取操作数。

IF（指令抓取）阶段在发射阶段之前，可以抓取到指令寄存器或指令等待队列中；然后指令从寄存器或队列中发射。EX（执行）阶段跟在读操作数阶段之后，根据操作数读不同可能需要花费多个周期。

在动态调度的流水线中，所有指令都按顺序发射。在第二阶段（读取操作数），可能会停滞或相互bypass，从而进入乱序执行。记分板（Scoreboarding）是一种在资源充足且没有数据依赖的情况下允许指令乱序执行的技术。

Tomasulo算法通过有效地动态重命名寄存器来处理反依赖和输出依赖。此外，Tomasulo算法可以扩展到处理猜测，通过预测分支的结果，在预测的目标地址执行指令，并在预测错误时采取纠正措施，来减少控制依赖的影响。

基于Tomasulo的方式实现动态调度

IBM 360/91浮点单元采用了一种复杂的方案来允许乱序执行。该方案由Robert Tomasulo发明，跟踪指令的操作数可用情况，以最大限度地减少RAW冒险，并引入寄存器重命名机制以最大限度地减少WAW和WAR冒险。

尽管近期处理器方案有许多变化，但都依赖于两个关键原则：动态地确定指令是否可执行，以及重命名寄存器以避免不必要冒险。

通过只在操作数可用时执行指令，可以避免RAW冒险（Scoreboarding方式）。由名称依赖关系引起的WAR和WAW冒险，可以通过寄存器重命名消除。通过重命名所有目标寄存器，包括较早期指令产生的待处理读或写操作的寄存器，来消除这些冒险，这样乱序写操作就不会影响任何依赖于较早操作数的值的指令。如果ISA中有足够的寄存器可用，编译器通常可以实现这种重命名。

冒险检测和执行控制是分布式的：每个功能单元的保留站保存的信息确定了在该单元何时可以开始执行指令。
结果是直接从保留站传递到功能单元，而不经过寄存器。

这种bypassing是通过一个共享结果总线来完成的，允许所有等待操作数加载的单元同时加载（称为共享数据总线，Common Data Bus，CDB）。在每个时钟周期发出多条指令并且拥有多个执行单元的流水线中，将需要多个结果总线。

指令从指令单元发送到指令队列，然后按照先进先出（FIFO）的顺序发出。保留站包括操作和实际操作数，以及用于检测和解决冲突的信息。加载缓冲区有三个功能：（1）保存有效地址的组成部分，直到计算完成；（2）跟踪等待内存加载的未完成加载；（3）保存已完成加载的结果，等待CDB的结果总线。类似地，存储缓冲区有三个功能：（1）保存有效地址的组成部分，直到计算完成；（2）保存等待数据值进行存储的未完成存储的目标内存地址；（3）保存地址和值，直到内存单元可用进行存储。来自FP单元或加载单元的所有结果都放在CDB上，CDB连接到FP寄存器、保留站和存储缓冲区。FP加法器执行加法和减法操作，而FP乘法器执行乘法和除法操作。

加载buffer和存储buffer暂存来自和访问内存的数据或地址在行为上几乎与保留站完全相同。浮点寄存器通过一组总线连接到功能单元，并通过单一总线连接到存储buffer。除了加载buffer外，所有其它功能单元和内存的结果都在通用数据总线上发送。所有保留站都有tag字段，通过流水线控制。

一条指令要经过的步骤，只有三个，每个步骤占用任意数量的时钟周期：

发射（Issue）：从指令队列的头部获取下一条指令（按照FIFO顺序，确保正确的数据流）。如果存在一个匹配的空闲保留站并且操作数值在当前寄存器中，发射指令和操作数到该保留站中；如果没有空闲的保留站，即存在结构性冒险，指令发射将停滞直到有保留站或缓冲区空闲；如果操作数不在寄存器中，跟踪产生操作数的功能单元，在这一步中对寄存器进行重命名，消除WAR和WAW冒险。（在动态调度的处理器中，这个阶段有时被称为分发（dispatch））
执行（Execute） ：如果一个或多个操作数不可用，在等待计算时监视共享数据总线。当一个操作数可用时，放到任意等待该操作数的保留站。当所有操作数都可用时，可以在相应的功能单元中执行操作。通过延迟指令的执行直到操作数可用，可以避免RAW冒险。（一些动态调度的处理器将这一步骤称为发射（Issue））
写结果（Write result） ：当结果可用时，将其发送到CDB上，然后写入寄存器和等待该结果的所有保留站（包括存储buffer）。存储指令在存储buffer中，直到存储数值和存储地址都可用，一旦内存单元空闲，结果就会立即被写入。

在Tomasulo中，结果值通过总线（CDB）进行广播，通过保留站对其进行监视。共享结果总线与从总线中取回结果的保留站的组合实现了在静态调度流水线中forwarding和bypassing。这个过程中，在source和result之间引入了一个时钟周期的延迟，因为结果的匹配和使用必须等到写结果阶段结束，而不是简单流水线中的执行阶段结束。因此，在动态调度的流水线中，产生指令和消耗指令之间的有效延迟至少比产生结果的功能单元的延迟长一个时钟周期。

在Tomasulo的方案中，标记指的是将产生结果的buffer或单元；当一条指令发射到保留站时，寄存器名称会被废弃。

每个保留站有7部分包含信息的字段：

Op：要在源操作数S1和S2上执行的操作。
Qj、Qk：将产生相应源操作数的保留站；值为零表示源操作数已经在Vj或Vk中可用，或者是不必要的。
Vj、Vk：源操作数的值。对于每个操作数，只有V或Q中的一个是有效的。对于加载指令，Vk用于保存offset。
A：用于保存加载或存储的内存地址计算信息。最初，存储指令的立即数字字段；在地址计算之后，存储有效地址。
Busy：表示该保留站及其相应的功能单元是否被占用。

register file中有1个包含信息的字段：

Qi ：包含应将其结果存储到该寄存器的操作的保留站序号。如果Qi的值为空（或为0），表示当前没有正在计算结果并将其存储到该寄存器的活动指令。
