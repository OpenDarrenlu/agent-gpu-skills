5D并行的通信原语
并行模式	通信产生原理	通信操作
DP（纯数据并行）	反向梯度更新时需要进行 all reduce 将梯度聚合. 优化：
1. gradient计算与AllReduce重合
2. gradient合并成更大的桶，gradient buketing。bucket_cap_mb=25MB，gradient_as_bucket_view=True，find_unused_parameters=True
3. 优先考虑DP，卡不够了再考虑gradient accumulation，以达到预期的global batch size	AllReduce Gradient
ZeRO1（优化器状态并行）	首先ReduceScatter更新本地gradient，再通过本地opt state更新本地weight，需要多执行一次 all gather 将weight聚合。
优化：
1. gradient计算与ReduceScatter重叠
2. AllGather的重叠需要考虑Prefetching	Bwd：ReduceScatter Gradient
Fwd：AllGather Weight
ZeRO2（梯度并行）	和上面一样，就是gradient每张卡只需要维护本地的即可	Bwd：ReduceScatter Gradient
Fwd：AllGather Weight
ZeRO3（权重并行）FSDP	前向需要先AllGather weight，推理完立即清除。反向的时候先AllGather weight，再计算gradient，ReduceScatter。不需要再AllGather了。可以用prefetching下一层来隐藏AllGather通信时间。	Bwd: AllGather Weight + ReduceScatter Gradient
Fwd: AllGather Weight
TP（张量并行）	-GEMM: 有row-linear，column-linear两种，分别需要Scatter+AllReduce，以及Broadcast+AllGather。
- MLP：col-linear + row-linear (Broadcast+AllReduce)
- Attention: WQ WK WV都是col-linear，或者说head-linear，每个worker一个head，WO是row-linear
- Bwd在算完之后不需要一个平均的AllReduce，因为大家数据是一样的，只是weight shard不一样。
- 通信计算重叠只能特殊处理：比如Bwd grad_tensor communication可以和grad_weight computation重叠。尽量使用节点内NVLink	Fwd
Split Row: Scatter Input + AllReduce Output
Split Col: Broadcast (Noop) Input + AllGather Output
MLP/Attention:Broadcast Input+ AllReduce Output

Bwd (反向而且共轭的操作）
Split Row: Noop Input + AllGather Output
Split Col：Noop Input (无需scatter) + AllReduce Output
SP(序列并行）	和TP同时使用，节省激活值的内存消耗。
- LayerNorm和Attention之间：AllGather所有sequence切片
- MLP和Dropout之间：ReduceScatter sequence替代AllReduce
- 通讯操作量是TP-only的两倍，但是AllReduce=ReduceScatter+AllGather, 所以其实是一样的。
- 通信计算重叠也是和TP一样不容易：Fwd partial input的矩阵乘和全部input的AllGather可以重叠	AllGather+ReduceScatter
Bwd也是一样。但是LayerNorm会额外需要一步AllReduce，这个在TP是不需要的。
CP（上下文并行）	在TP的基础上再把seq dim也切分。利用RingAttention以及Online Softmax来重叠KV的传输以及Attention Score的计算。利用ZigZag来均衡causal mask的情况。KV传输既可以是一次性的AllGather（内存占用大）也可以是一次一次的Ring上P2P（更多重叠）。
Bwd：KV向前传输，dK dV利用双工逆向传输。	Fwd：AllGather/P2P（P2P是RingAttention中逐轮交换KV。AllGather是开始的时候一次性全部交换完）
Bwd：AllGather/P2P，最后一轮AllReduce weights。
PP（层间并行）	解决大模型跨节点传输。传输次数比TP要少很多。前向和反向时需要各 rank 传递激活值，容易形成bubble。可以用micro-batch来形成pipeline。
优化：
- 所有的micro batch里的activation占用显存，可以从AFAB转换成1F1B来及时释放。
- interleave: 增加stage数目，代价是增加了通信
- Depth-First vs. Breadth-First: 先跑完一个batch还是先填流水线。深度优先主要可以解决显存不够的问题。
- ZeroBubble：把weight gradient和input gradient分开，input gradient比较紧急，weight gradient可以延后用来填充bubble
- DualPipe：DeepseekV3	Send
Recv （点对点通讯）
EP（专家并行）	FFN的时候需要根据gate分发（dispatch）到不同的GPU算完之后需要再聚合reduce回原本的GPU（combine）	AlltoAll Dispatch/Combine

FSDP

FSDP因为需要prefetch weights，为了掩盖通信时长，偏好更长的computation，比如更大的batch size和sequence length。

PP

偏好较大的grad_acc也就是micro batch size，以及interleaving stages。

FSDP和PP结合的话，weight的通信成本过高，还不如使用ZeRO-1或者2

TP/SP

通信代价大，扩展困难。实现依赖具体模型层结构

CP

MLP LayerNorm都可以独立处理CP切片，但是Attention不行，得用RingAttention

EP/MoE

在 MoE（混合专家模型）并行的 All-to-All 通信中，Pair-wise（两两交换) 以及 分层通信（Hierarchical Communication）是两种主流的算法实现，它们分别针对低延迟（LL）和高吞吐（HT）场景进行了优化。

维度	Pairwise 算法 (成对通信)	Hierarchical 算法 (分层通信)
核心解决痛点	传统 All-to-All 导致的网络多对一拥塞 (Incast) 与通信阻塞。	节点内 (NVLink) 与节点间 (RDMA) 的物理网络带宽极度不平衡。
基本工作原理	将全局全连接通信拆解，让多卡/多节点之间成对(Pair-by-Pair)地进行数据对冲交换，分批异步完成。	拓扑感知。先在节点内(Intra-node)聚合 Token，再通过跨节点(Inter-node)网卡发送，最后在目标节点内分发。
通信与计算重叠(Overlap) 策略	流水线重叠：将大块通信切分为细粒度成对数据流，隐藏在专家前向计算 (Expert Forward) 的背后。	带宽隔离：高带宽的 NVLink 与低带宽的 InfiniBand 互不干扰，通过在本地减少数据体积来缩短总体通信耗时。
对网络拓扑的依赖	较低。主要针对通信逻辑的串行/并行调度，在各种拓扑下都能减轻网卡突发过载。	极高。强依赖集群的物理硬件层级（如几台机器在一个机架内，哪些卡走 NVLink 等）。
代表性技术/框架	NVIDIA Megatron-LM TensorRT-LLM (One-Sided A2A)	Microsoft Tutel (Topology-Aware A2A) DeepSeek-V3 (Device-Limited 路由)
核心算法优势	避免了网卡瞬间被塞爆，极限提升了 GPU 的 SM 利用率，通信延迟被有效隐藏。	极大削减了昂贵且缓慢的跨机架/跨节点通信数据量，打破了百亿/千亿参数 MoE 的扩展性瓶颈。
主要工程挑战	需要精确的异步流水线调度（如利用硬件级 TMA 指令），代码实现复杂度高。	路由算法在分流时就需要考虑物理卡号 (Rank)，对动态路由的灵活性有一定限制。

DBT Double Binary Tree

https://developer.nvidia.com/blog/massively-scale-deep-learning-training-nccl-2-4/

单纯的树叶子节点很低效，因为他们只能向父节点传输，浪费了一半的带宽。

MPI和NCCL都是用了double binary tree，让同一个节点在一棵树上是叶子节点，在另一个颗上就是内部节点。这样每次传输就可以是双向了。
