Chapter 3 Instruction-Level Parallelism and Its Exploitation
3.1 指令级并行：概念与挑战/Instruction-Level Parallelism: Concepts and Challenges

大约自1985年开始，所有处理器都使用流水线技术来重叠指令的执行并提高性能。这种指令之间的潜在重叠被称为指令级并行（instruction-level parallelism，ILP）。

对于基本流水线不熟悉的可以看附录C（后面视情况补充）。

利用指令级并行性（ILP）主要有两种不同的方法：

依赖于硬件来动态地发现和利用并行性；
依赖于软件技术在编译时静态地寻找并行性。

程序和处理器的一些特性限制了可以在指令之间利用的并行性的数量，以及程序结构与硬件结构之间的关键映射关系。

流水线处理器的每条指令周期数（cycles per instruction，CPI）是基本CPI和所有stall和:

流水线CPI = 理想流水线CPI + 结构性停滞+ 数据冒险停滞 + 控制停滞

Pipeline CPI = Ideal pipeline CPI + Structural stalls + Data hazard stalls + Control stalls

理想流水线CPI是可以达到的最大性能的度量。通过减少右边的每一项，减少了整个流水线CPI，或者增加了每时钟指令数（instructions per clock ，IPC）。

什么是指令级并行？

在一个直接代码序列，除了入口和出口没有分支的基本块中，可用的并行度非常小。对于典型的RISC程序，平均动态分支频率通常在15% ~ 25%之间，意味着在一对分支之间可执行3 ~ 6条指令。因为这些指令可能相互依赖，我们可以利用的基本块内重叠的数量可能小于平均基本块大小。为了获得实质性的性能提升，我们必须跨多个基本块利用ILP。

提高ILP的最简单和最常见的方法是利用loop迭代之间的并行性。这种类型的并行通常被称为loop级并行。下面是一个简单示例，两个包含1000个元素数组的加法，并且是完全并行的:

for (i=0; i<=999; i=i+1) 
 x[i] = x[i] + y[i]; 

Loop的每个迭代都可能与其他迭代重叠，不过在每个循环迭代中，几乎或根本没有重叠的机会。

利用loop级并行性的另一种重要方法是在向量处理器和图形处理器（GPU）中同时使用SIMD。SIMD通过并行操作少量到中等数量的数据项(通常为2到8个)来利用数据级并行性。

数据依赖和冒险
冒险 hazard

判断一条指令对另一条指令的依赖程度，对于判断程序中存在多大程度并行性以及如何利用这些并行性至关重要。特别是，为了利用指令级并行性，必须确定哪些指令可以并行执行。

如果两条指令是并行的，它们可以在任意深度的流水线中同时执行，而不会造成任何停滞，前提是流水线有足够的资源(因此不存在结构性冒险)。
如果两条指令是依赖的，它们就不是并行的，必须按顺序执行，尽管它们可能会部分重叠。

如果两个指令是数据依赖的，必须按顺序执行。这种依赖关系意味着在两个指令之间会有一个或多个数据冒险。同时执行指令会导致具有流水线互锁（流水线深度大于指令之间的周期）的处理器检测到冒险并停滞，从而减少或消除重叠。

在一个没有使用互锁且依赖编译器调度的处理器中，编译器无法以完全重叠的方式调度相互依赖的指令，因为这样做会导致程序执行不正确。指令序列中的数据依赖反映了生成这些指令的源代码中的数据依赖。

依赖性是程序的一种属性。一个特定的依赖是否会导致实际的冒险被检测到，以及这个冒险是否真的会导致流水线停顿，这是流水线组织结构的属性。理解这个差异对于如何利用指令级并行性至关重要。

数据依赖表达了三件事:

发生冒险的可能性
计算结果的顺序
可能利用的并行度的上限

依赖可以通过两种不同方式克服：

保持依赖性但避免冒险
通过转换代码来消除依赖

在不改变依赖的情况下避免冒险的主要方法是调度代码，这种调度既可以通过编译器也可以通过硬件来完成。

数据可以在指令之间通过寄存器或内存位置流动。

当数据流通过寄存器时，检测依赖关系是很简单的，因为寄存器名在指令中是固定的，虽然当分支介入，正确性问题迫使编译器或硬件保守时更加复杂。
流经内存位置的依赖关系更难检测，因为两个地址可能表示相同的位置，但看起来不同。例如，100(x4)和20(x6)可能有相同的内存地址。此外，加载或存储的有效地址可能在每次指令执行时发生变化(因此20(x4)和20(x4)可能不同)，进一步使依赖关系的检测变得复杂。
名称依赖

当两条指令使用相同的寄存器或内存位置（称为名称），但与该名称关联的指令之间没有数据流时，就会发生名称依赖。

在程序顺序中，指令j与前面的指令i之间有两种类型的名称依赖：

当指令j写入指令i读取的寄存器或内存位置时，指令i和指令j之间就会发生反依赖。为了确保读取到的值正确，必须保持原来的顺序。
当指令i和指令j写在同一个寄存器或内存位置时，发生输出依赖。必须保持指令之间的顺序，以确保最终写入的值对应于指令j。

反依赖和输出依赖都是名称依赖，而不是真正的数据依赖，因为指令之间并没有传递值。由于名称依赖不是真正的依赖，因此如果更改指令中使用的名称（寄存器编号或内存位置），使指令不冲突，那么涉及名称依赖的指令就可以同时执行或重新排序。

对于寄存器操作，可以更容易地进行这种重命名，这被称为寄存器重命名（register renaming ）。寄存器重命名可以由编译器静态完成，也可以由硬件动态完成。在描述由分支引起的依赖关系之前，让我们先考察依赖关系与流水线数据风险之间的关系。

数据冒险

当指令之间存在名称或数据依赖，并且这些指令足够接近，导致执行过程中的重叠会改变对依赖操作的访问顺序时，就会存在冒险。

由于存在依赖关系，我们必须保持所谓的程序顺序，也就是说，如果按照原程序每次执行一条指令的顺序。软件和硬件技术的目标都是通过仅保留影响程序结果的程序顺序来利用并行性。检测和避免冒险可以确保保留必要的程序顺序。

数据冒险（Data hazard）可根据指令中读写访问的顺序分为三种类型。考虑两条指令i和j，其中i在程序顺序中先于j。可能的数据冒险有：

读后写（read after write，RAW）。指令j尝试在指令i写之前读初始数据，所以指令j错误地获得了未更新的值。这种冒险是最常见的类型，是真正的数据依赖。必须保留程序顺序以确保指令j受到指令i的值。
写后写（write after write，WAW）。指令j尝试在指令i写之前进行写操作。写操作会被以错误的顺序完成，导致在目标地址中最后的值是被指令i写入而不是指令j。这种冒险反应了输出依赖。WAW冒险只存在于在多个流水阶段写入数据或允许一条指令在之前指令停滞时继续执行的流水线中。
写后读（write after read，WAR）。指令j尝试在指令i读之前写值到目标地址中，所以指令i会错误地得到一个更新的值。这种冒险来自于反依赖(或名称依赖)。WAR风险在大多数静态发射流水线中不会发生，即使在更深的流水线或浮点流水线中也不会发生，因为所有的读取都是早期的（如ID阶段），所有的写入都是晚期的（如WB阶段）。WAR冒险发生在有些指令较早地写入结果，而其他指令较晚地读取初始值，或指令被重新排序时。
读后读（read after read，RAR）不是一种冒险。
控制依赖

控制依赖关系决定了指令i相对于分支指令的执行顺序，使得指令i按照正确的程序顺序执行，并且只在应该执行时执行。除了程序中的第一个基本块中的指令外，每个指令都依赖于某些分支的控制。通常，为了保持程序顺序，这些控制依赖关系必须保留。

控制依赖关系最简单的例子之一是if语句的“then”部分中的语句对分支的依赖。例如代码：

if P1 { S1; }
if P2 { S2; } 

S1依赖于P1，S2依赖于P2但不依赖于P1。

一般来说，控制依赖会有两个约束。

控制依赖于分支的指令不能移动到分支指令之前。例如，我们不能将if语句的then部分中的指令移动到if语句之前。
不依赖于分支控制的指令不能移动到分支指令之后。例如，我们不能将if语句之前的语句移动到then部分中。

当处理器严格保持程序顺序，控制依赖会被保留。

当处理器保持严格的程序顺序时，控制依赖关系会被保留。然而，我们可能会愿意执行本不应执行的指令，从而违反控制依赖关系，只要这样做不会影响程序的正确性。因此，控制依赖关系并不是必须保留的关键属性。相反，对程序正确性至关重要的两个属性是异常行为和数据流，通常通过保持数据和控制依赖关系来保持。

保留异常行为意味着指令执行顺序的任何改变都不能改变异常在程序中出现的方式。通常，这意味着指令执行的重新排序一定不会在程序中导致任何新的异常。

总结循环展开和调度

在实际操作中，必须以有条理的方式由编译器或硬件执行。为了获得最终的展开代码，我们需要做出以下决策和转换：

通过发现循环迭代（除了循环维护代码）是独立的，确定展开循环是有用的
使用不同寄存器以避免用相同寄存器进行不同计算时产生的不必要约束（如名称依赖）
消除额外的测试和分支指令，并调整循环终止和迭代代码
通过观察来自不同迭代的load和store是独立的，确定可以在循环展开中交替加载和存储
对代码进行调度，保留产生与原始代码相同结果所需的任何依赖关系

所有这些转换背后的关键要求是理解一条指令如何依赖于另一条指令，以及在给定依赖关系的情况下如何更改或重新排序这些指令。

三种不同的效应限制了循环展开带来的收益：

每次展开后分摊的开销减少。当我们将循环展开四次时，它会在指令之间产生足够的并行性，使得循环可以在没有停滞周期的情况下进行调度。如果将循环展开八次，则每个元素的开销将从1/2个周期减少到1/4个周期。
代码大小限制。对于较大的循环，代码大小的增加可能是一个问题，特别是如果导致指令缓存未命中率增加。另一个原因是过多的并行调度，导致寄存器分配压力增大。
编译器限制。复杂的转换导致编译器的复杂性显著增加。
