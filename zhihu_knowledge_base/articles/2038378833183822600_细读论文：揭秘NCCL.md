揭秘 NCCL：万卡集群下的 GPU 通信“黑盒”到底是如何运作的？

在大模型（LLM）训练和推理的今天，NVIDIA NCCL (Collective Communication Library) 已经成为了分布式计算的“事实标准”。然而，尽管它是开源的，其内部设计逻辑——如通信通道的编排、协议的选择、内存移动的处理——在很大程度上仍然是一个“黑盒”，缺乏详尽的文档支持。

最近，来自 ETH Zürich、NVIDIA 和 Broadcom 的研究者们发表了这篇重量级论文，首次系统性地拆解了 NCCL 的内部架构。本文将带你深度复盘这篇文章的核心干货。

1. 为什么 NCCL 需要被“揭秘”？

随着 AI 集群规模向万卡甚至十万卡演进，高效的 GPU 到 GPU 通信 是决定系统性能的关键。NCCL 专门针对 NVIDIA GPU 优化，利用了 NVLink、PCIe 和 InfiniBand (IB) 等底层硬件特性。

由于官方 API 文档虽全，但关于拓扑构建、算法选择、流水线管理等核心逻辑描述模糊，导致性能调优和瓶颈诊断极其困难。

B. 创建方式

1卡1进程: CPU 和 GPU 被组织成不同的 NUMA 域。将进程绑定（固定）到本地 NUMA 节点后，它就可以通过 PCIe 或 NVLink 直接与本地 GPU 进行通信，而无需跨越 QPI/UPI 这类 CPU 内部互连总线，从而显著降低延迟。但是对cpu之间的通信不利
1卡1线程：有利于进程内memory共享，可以绕过CPU进行P2P_DIRECT，减少通信时间。
多卡1线程：坏处是需要串行创建kernel，好处是cpu的overhead少，没有竞争，确定的执行顺序，简单好实现，适合做个prototype。

C.通信通道

NCCL 通过GPU、CPU 和网络接口卡 (NIC) 三个硬件组件的协同来编排通信, GPU负责reduction和gpu buffer移动数据，CPU负责launch gpu kernel以及协调工作，NIC负责发送包。为了避免单个流处理器 (SM) 成为处理大消息时的性能瓶颈（单SM过载以致于打不满NVLink），NCCL 将每个集合通信操作细分为多个通信通道 (Communication Channels)。

C1. 通道的核心机制与并行化

SM 级别并行：每个通道在 GPU 上作为一个独立的 CUDA Block 启动，运行在专属的 SM 上。NCCL 会将输入缓冲区划分成不相交的数据块，让各个通道并行处理，从而极大提升总吞吐量。
硬件负载均衡：在拥有多个 NIC 的 NVLink 平台上，多通道设计允许每个通道通过不同的网卡独立离开节点，从而平衡 NVLink、PCIe 和 InfiniBand 等互联链路的负载，减少空闲时间。

C2. 性能权衡与启发式调整

网络效率挑战：虽然多通道增加了 GPU 侧的并行度，但过激的通道化可能损害网络效率。
缓冲区利用率：如果每个通道分配的数据块过小（例如小于 NIC 传输采用的 512 KiB FIFO 缓冲区），会导致代理线程发送填充不满的缓冲区，从而降低 PCIe 和网络吞吐量。当多个QP（IB的queue pair）被用于Equal-Cost Multi-Path Routing (ECMP)负载平衡的时候尤为严重
动态调整：NCCL 内部会使用启发式算法，在处理较小消息时自动减少通道数量（nChannels），以在并行度与网络利用率之间取得平衡。

注：GPUDirect RDMA

路径原理：在跨节点通信中，GPUDirect RDMA 优化的核心在于绕过了 CPU 主机内存的中转（Staging）。它允许网卡 (NIC) 直接通过 PCIe 交换机 (PCIe Switch) 访问 GPU 显存,。
硬件要求：GDRDMA 的实现前提是网卡和 GPU 必须连接在同一个 PCIe 交换机下,。
作用：虽然数据不再经过 CPU 和系统内存（DRAM），但它物理上仍需通过显卡插槽和服务器主板上的 PCIe 链路传输到网卡，然后再进入网络

C3. 通道管理与拓扑构建

Communicator 层级协调：通道管理在通信器（Communicator）初始化时完成，每个 GPU 被分配一个唯一的 Rank。NCCL 会根据系统拓扑、架构默认值，预先建立通道。
运行时选择：在执行具体任务时，NCCL 会根据选择的策略，当前消息大小和可用带宽，动态决定使用多少个预先建立的通道，以及每个通道多少个threads。
逻辑拓扑分配：每个通道在communicator 初始化时都被分配了一个逻辑通信拓扑：
环形拓扑 (Ring)：每个 GPU 识别其直接前驱和后继，形成单向通信。
树形拓扑 (Tree)：每个 GPU 跟踪其父 Rank 和子 Rank。NCCL 特别采用了双二叉树 (Double Binary Tree) 结构来最大化带宽利用率。

C4. 点对点 (P2P) 操作的并行性

在使用 ncclGroupStart 和 ncclGroupEnd 进行的分组点对点操作中，NCCL 会尽可能将每个传输分配到独立的通道。
这种设计实现了任务级的并行性，允许物理上独立的多次发送和接收任务并发运行。
2. 三大核心通信协议：Simple, LL, LL128

NCCL 并不是用一套逻辑跑天下，它根据数据大小和网络延迟动态切换三种协议，用来平衡latency和bandwidth：

Simple 协议：主要针对高带宽需求。它在大块数据传输中表现最优，通常用于节点间通过 IB 传输梯度的场景。
LL (Low Latency) 协议：专为低延迟设计。它通过在数据块中嵌入标志位（flags）来避免昂贵的同步开销。
LL128 协议：它是 LL 协议的变体，利用 NVLink 的特性（如使用 128 字节的数据块）来进一步压缩延迟，适合节点内（Intra-node）的小数据同步。

论文详细对比了这些协议的设计权衡及其动态选择逻辑，帮助我们理解为什么同样的任务在不同规模下耗时差异巨大。

每个通道维护一个ncclIbSendComm数据结构
每个ncclIbSendComm有两个一组的QP bundle：forward QP （传数据用的，high bandwidth）， reverse QP（CTS用的，low latency）
当thread proxy中high-level的sendComm->isend()发起时， 他会在2个ncclIbSendComm handles里进行round robin分发数据。分发的数据可以更多，而且有效利用ECMP。而且GPU不感知
QP bundle的意义
forward QP和reverse QP一个是需要高bandwidth 的data traffic一个是需要low latency 的control traffic，两个如果混到一起，容易发生head of line blocking，就是重要而且可以很快完成的control signal会被data chunk阻塞
forward QP：sender发给receiver
多个RDMA_WRITE request和一个最终的RDMA_WRITE_WITH_IMM（包含transfer size）组成。
reverse QP：receiver发给sender
clear-to-send（CTS）信号，对面可以往这发数据了
RDMA_WRITE request，包含了remote buffer address, rkeys, 以及 tag information
Local Flush with Loop-back RDMA_READ
为了防止NIC以为远程GPU传来的数据已经读完了，但其实卡在了buffer或者PCIe Switch上，NCCL创建一个专门的QP自己连自己并在把所有数据写给本地GPU之后发起一个dummy RDMA_READ。
其实就是利用IB verbs layer的读写机制创建了一个廉价的fence。
RDMA 协议规定，在同一个传输队列（QP）中，后发起的读操作（RDMA_READ）必须等待之前发起的写操作（RDMA_WRITE）在物理层面完成后才能返回。
一致性保证：当你通过 Verbs API 发起那个“虚假”的读请求时，Verbs 驱动和硬件逻辑（Verbs Layer） 会强制清空 PCIe 管道中的在途写数据。只有当 Verbs 层确认之前的写入已经物理抵达 GPU 显存，它才会让当前的读操作完成




4. 集合通信算法：Ring 还是 Tree？

NCCL 目前主流使用两种核心算法：

Ring（环形算法）：
优点：带宽利用率高。
缺点：延迟随 GPU 数量线性增长（O(N)），在超大规模集群（数千卡以上）中性能会急剧下降。




Tree（树形算法）：
优点：延迟随 GPU 数量呈对数增长（O(\log N)）。
适用场景：在大规模、跨机架的 AllReduce 操作中，双二叉树（Double Binary Tree）算法能有效缓解延迟瓶颈。

A. 算法概括

CollNet主要就是利用network中支持SHARP的单元（比如IB Switch）来辅助计算比如AllReduce。节点内的链接有两种方式，都是仅支持Simple AllReduce：
CollNet Direct，支持all-to-all的通信
CollNet Chain， GPU是连成链子的，reduction up the chain，broadcast down the chain
NVLS利用NVSwitch，节点内利用第三代NVSwitch的NVLink SHARP辅助reduction计算。节点外的计算有两种方式：
NVLS，和CollNet一样利用IB Switch的 SHARP。除了AllReduce还支持ReduceScatter和AllGather。适用于拥有满配 IB 网络且交换机开启了 SHARP 功能的超大规模集群。
NVLS Tree，当网络侧不支持 IB SHARP，或需要利用树形拓扑平衡跨节点延迟时(O(N)->O(logN))，用双二叉树fan in fan out。

补充1：

在传统的 NCCL LL 协议实现中，CPU 是不可或缺的；但在最新的“设备启动通信”（Device-initiated Communication）架构下，LL 逻辑确实可以完全绕过 CPU，由 GPU 直接接管。

以下是根据来源对这一演进过程的详细拆解：

1. 传统 NCCL LL 协议：为什么必须经过 CPU？

在传统的 NCCL 架构中，即使是在 NVLink 环境下，LL 协议的设计也带有明显的“CPU 代理”痕迹：

Host 内存中转：传统 LL 协议强制将中间缓冲区放在 Host 内存（DRAM）中。
CPU 轮询职责：这是因为 CPU 代理线程（Proxy Thread）需要通过 PCIe 轮询 Flag，以检测数据是否准备好通过网卡（NIC）发送出去。由于 CPU 跨 PCIe 轮询 GPU 显存非常慢，所以才必须把数据和 Flag 放在 Host 内存里。
路径：GPU \rightarrow (PCIe) \rightarrow Host Mem (CPU 轮询) \rightarrow NIC \rightarrow 网络。
2. 现代演进：如何实现绕过 CPU？

随着 NVIDIA 推出 NCCL Device API（从 2.28 版本开始）以及 DeepSeek 等团队推动的 DeepEP 库，LL 的逻辑发生了质变，实现了真正的设备启动通信：

A. 节点内（Intra-node）：利用 LSA 和对称内存

在 NVLink 域内（如 DGX 机箱内的 8 卡或 NVL72 机柜），GPU 之间使用对称内存（Symmetric Memory）：

直接访问：通过 LSA (Load/Store Accessible) 模块，每一块 GPU 都能获得其他 GPU 内存的直接指针。
GPU 直接轮询：接收端 GPU 的 CUDA 内核可以直接在显存中轮询对端的 Flag，发送端 GPU 也可以直接执行 8 字节原子写到对端显存。这个过程完全不需要 CPU 代理线程参与，从而消除了跨 PCIe 的延迟和 CPU 调度的开销。
B. 节点间（Inter-node）：利用 GIN 技术

对于跨机通信，新的 GIN (GPU-Initiated Networking) 模块允许 GPU 直接与硬件对话：

GPU 直接发车：GPU 内核直接填充工作请求（WR）并写入网卡的 Doorbell 寄存器，直接启动 RDMA 操作。
同步逻辑下沉：Flag 的轮询和加法操作（如 ncclGin_SignalAdd）直接由 GPU 内核在显存中处理，不再需要返回 Host 内存让 CPU 去对单号。

所以新技术：

LSA (Load/Store Accessible)： 让 GPU 像访问本地显存一样，直接使用 CUDA 的 load/store 指令 读写 NVLink 域内其他 GPU 的内存。
GIN (GPU-Initiated Networking)： 提供设备端可调用的 RDMA 原语：
ncclGin_Put / ncclGin_Get：读写远程 GPU 内存。
ncclGin_SignalAdd：原子增加远程信号，用于同步。
3. 绕过 CPU 的核心价值

将 LL 协议从“CPU 编排”转变为“GPU 启动”，对 DeepSeek-V3/V4 等模型至关重要：

极致低延迟：消除了 GPU-CPU 间的通知开销。在 MoE 的解码（Decoding）阶段，由于 Batch Size 极小（1-128 token），每一微秒的节省都直接转化为推理吞吐量的提升。
计算与通信深度融合：允许在同一个 CUDA Kernel 内同时进行量化（Quantization）、数据搬运和同步，不需要频繁切换内核或返回 CPU。
释放 CPU 资源：在大规模集群中，原本繁忙的 CPU 代理线程被解放，系统可以更专注于调度任务。
总结

LL 协议已经不再局限于那个必须依赖 Host 内存的旧定义，而是进化成了由 GPU SM 直接驱动、利用 NVLink 和 GPU-Direct RDMA 实现的超低延迟通信流。

补充2：根据最新的来源（包含 NCCL 2.30.x 的发布说明及 NCCL EP 等扩展），NCCL 的算法与集合通信操作矩阵已经显著扩展。

NCCL 已经从传统的 5 种 核心操作演进为支持 8 种 标准 Host API 操作，有了新算法PAT，并引入了针对 Blackwell 架构优化的 CE 以及针对 MoE 架构的 EP 等新算法/执行模式。

关键更新与新增特性说明：

Host 侧 API 补齐 (2.28+)：
ncclAlltoAll、ncclGather 和 ncclScatter 已正式成为 Host 侧标准集合通信 API。
在之前的版本中，这些操作通常需要开发者通过 P2P 原语手动模拟。
CE (Copy Engine) 集合通信 (2.28+)：
专门用于 AlltoAll、Gather 和 Scatter，旨在减少 SM（流处理器）占用。
在同一个 NVLink 域（MNNVL）内，通过使用 CE 替代 SM 执行搬运，可释放 GPU 计算资源给应用程序。
从 2.29.2 开始，在 Blackwell 系统上对于大消息尺寸的 AllGather 也会透明地切换到 CE。
PAT (Parallel Aggregated Trees) (2.23+)：
专门为大规模集群设计的算法，主要服务于 AllGather 和 ReduceScatter 操作。
相比 Ring 算法，PAT 旨在解决超大规模节点下的延迟线性增长问题。在 2.26.2 中，PAT 算法通过异步计算与执行分离得到了显著加速。
Expert Parallelism (EP) 扩展 (2.30+)：
引入了专门针对 MoE（混合专家模型）架构的 ncclEpDispatch 和 ncclEpCombine 原语。
虽然这属于实验性的扩展库，但它利用了 NCCL Gin (GPU-Initiated Networking) 技术，实现了由 GPU SM 直接启动的 All-to-All 通信。
硬件与协议优化 (2.30+)：
TMA (Tensor Memory Accelerator)：在对称内存内核中增加了对 TMA 的支持，以进一步优化 NVLink 带宽利用率。
LL128 异构支持：在 2.30.3 中，开启了 Hopper 及更高版本 GPU 在异构场景下的 LL128 协议支持。
Blackwell 优化：针对 Blackwell 架构调整了最大 CTA 数量（从 32 降至 16），以平衡 SM 开销与性能，并默认启用了 PXN over C2C 以提升跨网卡通信性能。

总结： 2.31 时代的 NCCL 已经从“单纯的 Host 控制”全面转向“设备启动通信 (Device-initiated)”与“硬件负载卸载 (TMA/CE)”并重的阶段，极大地提升了 MoE 等新型 AI 工作负载的通信效率。
