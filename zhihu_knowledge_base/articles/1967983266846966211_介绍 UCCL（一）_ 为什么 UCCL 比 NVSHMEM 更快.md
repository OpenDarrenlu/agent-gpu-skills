1. UCCL 核心逻辑

UCCL 核心逻辑包含了两个痛点，其中之一是现存的 通信 解决方案，和 NVIDIA 设备耦合度非常高。我想表明一点是 软硬件协同 (synergy of hareware and software co-design) 不等于 软硬件耦合 (couple of hardware and software) [1]。

由于核心通信 IP 主要在 Intel、Synposis 和 BroadCom， 而目前唯一跨机通信方案最可靠的第三方平台之一，仍是博通。因此通过 AMD 平台 ， 以及 AMD 和 博通的关键合作是最有可能解决非NV计算通信问题（供应链议价安全）的关键路径。如果 AMD 和 博通 解决不了这个问题，其他厂商也很难解决。

通过解藕适用于小消息的通信软硬件，是 UCCL 解决第一个问题。

从 switch transformer 开始，到 deepSeek MLA-MoE 架构，再 到 GPT-OSS full-sink-attention-MoE， 以及最新 基于 attention recurrent format 3:1 混合线性attention - MOE， MoE 为代表的 MLP 模块成了最不可撼动的模组，带来了 由数十 KB 组成数据平面小消息模式， 而这个问题在推理 decoding 阶段尤为明显。

SGLang DP attention : noted that SGLang uses reduce scatter after (for TP ranks) MLA , then gating tokens (for TPxDP ranks) with reduced scatter to each MoE DP rank before MoE MLP modules

对于 DP-attention ，由于 相同的卡 ，TP size 变小，但是每个 TP rank 处理的 token 变少，在H100 上 针对计算密集算子 MLA，比如 prefill 3 节点，我们推荐 TP=24(3x8), DP=2（SGlang 独立 DP 设定）。这相当于 TP_attn=12 卡来做 tp attention，分别处理 1/2 tokens. 假定 batch_size=bs=concurrency, 序列=S，hidden=hdim，投这种通信 [bs /2 , S, 1/*head*/, hdim / TP_attn] 约 KB，机间延迟 约 数 微妙。

在SGlang中采用 Reduce Scatter 进行TP rank通信， 这样每个 GPU 的 token memroy footprint 更小，可以让模型有更高的本地 KV cache 预算，支持长的上下文。

1.1 MoE 小消息模式

Fused MoE 由每个 DP rank 上的 expert 执行 group matmul 组成 DP rank MLP。 首先进行 第一次 all-to-all 通信，将每个 DP rank 的 token , dispatch 到对应EP所在 的 DP rank (srcRank -> dstRank)。

这是一个不均匀的在线映射，因为无法预判每个 Expert 对应多少个 token，在我们最新开发的算法中只需增量映射即可，但可以肯定，通讯量是 [bs / TP, routed_tokens_dp_rank_i, hdim]， 计算通信量 < 1 KB。

每个 DP rank 通过 MoE Align & Sort [2] 和第一次 all-to-all 拿到per-rank数据后，进行MLP计算。首先 up_proj 和 down_proj 进行合并 （m13）进行更紧凑的 group gemm 运算；运算完后 进行 第二次 all-to-all 将每个 EP 计算合并(EP Reduce)到每个 DP rank。




1.2 UCCL 关键创新
CPU-GPU C2C communication via lock-free queue


之前普遍观点（DeepSeek） 认为 IBGDA 采用的技术方案更优于IBRC ，这是因为由于all-to-all网络的控制平面(control plane) 和 数据平面(data plane)，都在 放到GPU，事实上变成了 2-sided communication 机制 (芯片内置的DMA模块实际上允许 one-sided communication)，从而CPU端不再参与信息同步。

但是 在 我们初步基于 A卡rocShmem all-to-all 技术方案的支持尝试中 [4]，发现：

dispatch/combine kernels 采用了很多更适合 CPU 的处理 tensor meta 信息分发的逻辑。

UCCL 团队证实了这种猜测，他们将控制平面的代码还给了 CPU 但是 ，GPU 计算单元 SMs/CUs 继续专注在 数据平面 中 数据的拷贝 （bulk data transfering），而且 由于 DMA 单边性，可以更快释放 GPU 计算资源，增加 MLP的并行度。

我们通过测试 MI350 平台 RCCL，发现 小消息通信首次领先 NCCL：

UCCL 在 RCCL/NCCL 也得到了比较好的了比较好的测试数据，并首次超越 NIXL (NVIDIA nvshmem based solution)。

由于 控制平面的信息十分小，在 CPU-GPU C2C通讯中，并不会带来很多延迟，拆分传输逻辑，对更重的数据传输成了优化的关键。

3. 讨论：用户态线程通讯代码构建 control plane 通信是否可取？
GRPC Pub/Sub kernel space dependent communication demo in 2019 [7]

基于 RPC 消息通信 通常发生在 内核态，如GPRC pubsub solution demo 2019 [7]，在 自动驾驶 RTOS 场景，为了更稳定服务，基于抢占式调度被实时线程替代，内核态依赖程序被用户态线程程序代替。

采用 libverbs API 可以避免采用类似 GRPC 这样的内核态依赖程序，-- 因为他们并不适合 HPC/LLM 高性能推理，并使用 rdma-core 通过 verbs API 进行 GPU 间通讯。
