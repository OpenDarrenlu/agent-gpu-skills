最近在处理GNN的CPU性能优化问题，遇到几个问题挺有意思，这里开贴介绍一下相关的优化思路，也可以作为一个典型的"发现问题->解决问题"的范例。本篇是以 PyG (pytorch-geometric)为背景的，其他的GNN 框架这里不做讨论。这里主要介绍优化策略，不讨论GNN的原理用法等。

Profiling

一般来讲 performance optimization 的第一步都是profiling。这里所选取的对象是 PyG自带的examples 外加ogb 的benchmark，各种各样的模型我们跑了很多，经过总结后发现从性能角度讲，模型的行为可以分成两大类。核心区别是 EdgeIndex 的存储格式，这个 EdgeIndex是一个记录source node到dest nodes连接的 Tensor，逻辑上用 Adjacency Matrix来表达，如 Fig-1所示：

EdgeIndex的存储格式是COO (Coordinate format)

第一类情况对应经典的 PyG应用场景，EdgeIndex 是一个 [2, num_edges] 的2D Tensor，第一行记录 source node的 index，第二行记录 dest node的 index。这类情况对应的 hotspot一般是 aten::scatter_add 或者 aten::scatter_reduce，对应 Message Passing 中 information aggregation 过程，用到了哪个和具体选择的 reduce type 有关系；需要注意的是 scatter_add 或者 scatter_reduce 在 torch 里面其实是两个 operator，但最终是调用同一套 kernel。

scatter_reduce 对应的 backward 是 gather，后文只介绍 scatter_reduce 的优化策略，gather 的优化比较简单就不写了。

EdgeIndex的存储格式是CSR (Compressed Sparse Row)

第二类情况对应在 DataLoader中做了 to_sparse 的情况，这时候 EdgeIndex 会存成一个 Sparse CSR 格式的 Tensor。这个时候，hotspot 是 torch_sparse::spmm_reduce，同样根据 reduce type的不同可能是 spmm_sum, spmm_mean 等等。

理解 CSR 的格式非常重要，不然就很难弄明白为什么算spmm的时候 CSR比 COO快很多！

spmm_reduce 的 backward优化也比较简单，就不写了。后文重点介绍 spmm_reduce 的优化。

ScatterReduce 应用的特殊性

这个operator 本身的语义，官方的 doc 上面都有介绍，index 应该是和 src 同样大小的一个 tensor。比如在实际模型里面我们可以打印出所有输入输出的 shape和stride，例如：

### scatter_add: dim: -2
self.sizes(): [19717, 16]; strides(): [16, 1]
index.sizes(): [108365, 16]; strides(): [1, 0]
src.sizes(): [108365, 16]; strides(): [16, 1]
out.sizes(): [19717, 16]; strides(): [16, 1]

注意这里 index 的strides 是 [1, 0]，那么物理内存实际大小其实是108365，而不是108365x16。index 是一个 extended tensor，后续的优化都是在这个基础之上做文章的，实际上是从 src 中按行来做 reduce （如 Fig-2 所示）。这就给了我们操作空间，不然按照 scatter_reduce 原本的语义，这个 operator 优化的空间不大。

从这个地方我们也可以看到 strides 的重要性：strides 表征物理内存的存储方式，这个信息非常关键。

我们知道了 index 是一块 “名不副实”(logical shape != physical shape) 的tensor，怎么高效访问呢？以下两种方法都可以：

通过“指针”访问：前提是需要 check 物理上是否连续。
通过 TensorAccessor 访问：不需要check 物理上是都连续，本质上是通过 x0 * stride0 + x1 * stride1 + ... 来计算index。

注意这里不能调用 .contiguous() 然后再访问，不然会引入额外的 copy 操作。

原kernel性能低下的原因

首先 scatter_reduce 里面有个输入参数是 dim，在 GNN的场景中都是 0，这个 0 就是“万恶之源”：torch原有的kernel 会选择在 dim这个维度上串行执行，而在所有其他维度上做并行化。这种切分方式完全是从“逻辑”的维度角度出发的，没有考虑“物理”的维度。这么做的好处是，不会有 thread write conflicts，如 Fig-3 所示：

如果 dim刚好是 lastdim，这种处理方式是可以达到最佳性能的，其他情况都很差。比如在 GNN中，dim 刚好是0，那么就刚好是在outer dimension做串行，在inner dimension上做并行，性能会很差，原因在于：

每个thread拿到的数据是不连续的
没法做向量化
每个thread需要重复load index

在 Fig-3 示意图中 M代表 num_nodes，N代表 num_edges，K代表 num_features，在实际应用中 M和N可能都很大，比如在reddit上面 M=135k, N = 447k, K = 256。所以这个尺寸 index是没法 L2 cache 命中的。

ScatterReduce的优化

原有 kernel性能差的主要症结是在 inner dimension上做了并行化，为了提高性能我们需要在 outer dimension上做并行化，这样才能保证 contiguous memory access 并且好做向量化。但这个时候一个比较直观的问题是，会有写冲突：

图 Fig-4 所示，有可能不同的thread会去同时写同一个地址，导致写冲突。解决这类问题有很多种手段，比如：

Dual Parallel Reduction：借用一块 [T, MK]的 buffer，分两步完成 reduce: step 1) src -> buffer; step 2) buffer -> self。这种方法适用于 N >> M 的情况，这样借用buffer的开销会比较小，e.g. BatchNorm2d (channels last)。GNN不是这种情况，M动辄上百万。
Atomic：这种方法写起来非常简单，但有个问题是 atomic 本身在 x86 上只支持一个float的操作，而不是一整行，所以K=1的时候用这种方法比较合适，GNN也不是这种情况。
Sorting: 对 index 进行排序，好让拥有相同index的node都会被分到同一个thread上面，如 Fig-5 所示。

经过各种尝试，最终选择了最后一种方法。还是上面的 SAGE + reddit，经过优化之后 profilig的结果如下，可以看到 scatter_add 的时间从 37.8s 降到了6.5s，经过优化之后 scatter_add已经不是最主要的 hotspot了。

SpmmReduce 性能优化

原本这个operator是在torch-sparse这个包里面的，PyG作者意见是把优化放到torch里面，所以后续优化是直接做在 torch 里面的。前面提过这种情况对应 EdgeIndex 是 CSR 格式的，例如在 GCN+ogbn-products 里面，初始的inference profiling 结果如下：

可以看到几乎所有时间都花在了 spmm_sum上面。spmm_reduce 形式上和常规的 spmm非常相似，A 矩阵是 CSR，B和C都是dense的：

这里面A矩阵实际上就是 Adjacency Matrix，nnz (number of nonzeros) 就是num_edges。M对应 num_nodes，K对应 num_features。在ogbn-products这个dataset中：M= 2.4M, nnz = 126M, K =256。

原本torch-sparse中的实现主要有两个问题：

thread payload imbalance：原有kernel直接在M上做parallel，也就是每个thread分到同样多的nodes。但实际数据集中每个node所拥有的connections数量相差非常大，就会导致 thread之间非常不均衡。
没做向量化：这是非常典型的 FMA 操作，不过bottleneck 应该是 memory bandwidth：读的时候存在重用的可能性，不过因为 Matrix A sparse 的属性，这种可能性非常低，所以就是在刷 memory bandwidth。

优化的思路很明确：按照“总工作量”进行任务切分，从而保证thread payload balance。另外可以在nnz和K两个方向上做blocking，从而减少 write memory access。经过优化之后，profiling结果如下，可以看到 spmm_sum 的时间从 56s 降到了 11.8s。

反向能拿到的 speedup 比前向要更大，因为原有的 CPU 反向是完全没有优化的朴素版本。reduce type 选择 “max”的时候拿到的 speedup 也要大一些，因为做了一些额外的优化：比如 inference 省却 argmax 的输出，等等。
