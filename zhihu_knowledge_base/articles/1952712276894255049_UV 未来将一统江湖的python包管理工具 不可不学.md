小志哥聊UV

uv（https://docs.astral.sh/uv/ ）是一个先进、强大、优雅的python包管理工具，小志哥预测其将结束当前python包管理的春秋战国现状，一统江湖。小志哥在工业实际生产上经过了半年多的使用，简短地为大家总结下uv的特点，顺带做一下科普。




理念先进

传统上，尤其是高校实验室里，大家习惯用anaconda，一个环境、或者少数几个环境包打天下。什么包都往这个环境里装，能跑就跑，跑不起来就拉倒。uv，因为是一个崭新的工具，没有什么太大的历史负担，所以在设计之初，便采纳了长期python实践里总结出来的先进理念。和一套环境包打天下有着云泥之别。

可以这么说，学习uv，就是学习了这些先进理念，对于个人开发能力、技术架构理解能力的提升有着立竿见影的效果。

uv的先进的理念包括但不限于：

python环境依附于项目，而非项目依附于环境。

uv会为每个python项目分别创建一个独立的虚拟环境。不同的python项目，拥有不同的python虚拟环境，从根本上避免了不同python项目的可能的依赖冲突问题，是一种以python项目为中心的先进理念。
这强迫用户去思考，你当前要做一个什么项目，这个项目的依赖是什么，从而避免多个项目耦合在一起。




python工具和开发项目解耦。

在日常工作中，我们经常需要用到一些以python包形式存在的工具，比如格式化工具black、isort，或者自己写的一些小脚本。我们不需要在自己的项目里import他们，而是通过命令行调用、直接运行脚本等方式使用。
uv的理念是，为这些独立的工具提供单独的python环境，从而避免其依赖污染我们的目标开发项目。而且，提供了非常人性化、易用、且强大的命令行使用方式。




每个项目的依赖是明确的。

uv会要求每个python项目维护一份自己的pyproject.toml，并依据它创建一个虚拟环境，并且以一个uv.lock文件的形式，记下来所有依赖的实际生效版本。
在工业环境中，往往有些包需要从不同的源下载。比如，A包从pypi镜像源下载，B包从公司的私有仓库里下载，C包需要从第三方公司提供的仓库下载。uv在pyproject.toml里提供了灵活的定制方式，来实现这些需求。
由此，通过pyproject.toml和uv.lock两个文件，实现绝对的程序可复现性。




项目的依赖是可以灵活调整的，灵活试错的。

uv创建、调整python虚拟环境是如此的飞快，以至于当依赖出现问题时，我可以将整个虚拟环境删除，并且在10秒内重新创建。又因为uv.lock的存在，虚拟环境可以完全复现。所以用户对虚拟环境的试错，没有任何心理负担。
通过git记录，用户也可以从uv.lock里发现，依赖版本发生什么样的调整、变更，从而能更快地定位问题。




python项目分为library和application，前者的依赖的支持版本需要宽，后者的依赖的支持版本需要窄。

lib和app的最大区别在于，lib会被其他lib、app import，而app只被自己import。一般情况下，大多数人开发的python程序都算是application。uv在创建project时，显式地要求用户明确，要创建的是lib还是app，是一个先进的理念。
uv通过uv.lock，来锁定依赖版本，从而支持app的完全可复现性。但另外一方面，uv通过uv workspace来支持lib、app的联合使用，从而提供lib的多依赖版本的可测试性、可复现性。







速度飞快

天下武功，唯快不破。uv拥有所有python包管理工具里最快的速度，比pip快10-10000000000倍。在有缓存的情况下，一个虚拟环境的创建，可能快至1秒。

why uv so fast? 简单地抄一下这篇文章：https://medium.com/devdotcom/heres-why-uv-s-speed-and-simplicity-are-winning-over-devs-for-ai-and-cloud-projects-fa23a63ab8ef

rust实现

Rust 高效的内存管理与并发能力为 UV 奠定了快速运行的基础。而 pip 运行于 Python 之上，其性能可能受限于 Python 解释器，UV 则借助 Rust 的性能优势实现了快速执行。




全局缓存设计

UV 会维护解压缩后归档文件的全局缓存，并在安装时采用重连（reflink）或硬链接（hardlink）技术。这种方式优化了热操作（如重新安装软件包）的效率，同时节省磁盘空间，在管理多个环境时优势尤为明显。




零复制反序列化

UV 将元数据以与内存结构匹配的格式存储在磁盘中，通过指针类型转换（pointer casts）实现反序列化。这确保反序列化时间不会随数据大小增长而增加，效率极高。




优化的版本解析

UV 使用单个 64 位整数（U64）表示超过 90% 的软件包版本，支持栈分配与快速比较。相比传统方式，该优化将依赖解析速度提升了 3 至 4 倍。




高效的元数据获取

UV 通过范围请求（range requests）仅从软件包归档文件（wheel 格式）中获取中央目录与元数据，无需下载大型文件（例如大小可达数百兆字节的 PyTorch wheel 包）。







可能有人会问，装包快有什么用呢？一般情况下，我可能也就只装一次啊。

确实，在小项目里，区别不是很大。但是到了拥有复杂依赖、复数项目的工业界，装包快会成为一个非常重要的前置项，能够解锁更高级的开发范式： MonoRepo。

强大的Mono-Repo范式

如果你是创业公司的CTO，现在公司里有几百个工程师构成的开发团队，需要支持几个商业AI项目，而这些AI项目由几十万行、几百万行代码支撑，分散在几十个项目中。那么，你要怎么做好这些项目的协同，同时保证代码开发的敏捷性、复用性，以及运行环境的一致性、可复现性？

简单来说，业界有两种方式做这个事情：

multi-repo，简单易行，大多数公司采用的方式，尤其是python语言。

典型例子，你写了两个python项目，`pip install -e . `装在同样一个python环境里。
每个项目用一个git仓管理，不同项目之间通过相互import来复用代码。复用代码。




mono-repo，复杂成本高，少数公司采用，在C++/Java等编译型语言上使用较多。

典型的例子，谷歌公司的Mono仓，一个仓支撑了大多数公司产品；自动驾驶公司的车端部署C++代码。
所有项目集中在一个git仓里。不同项目之间，一般是分别开发、分别测试，最终通过制成品（.exe/.so/.wheel等）来在生产环境里按需使用。







以下是一个对比，这个对比是基于需要支持多个项目的前提，否则肯定是单仓胜出。




	multi-repo	mono-repo
多项目开发/协作效率	低效。单一个依赖冲突和重复造轮子，就是梦魇。	能够scale，和单项目差别不大。简化依赖与代码管理。跨项目修改，在一个MR里解决。便于统一标准。便于追溯和回滚。
对团队开发能力的要求	低，一个字就是莽。	高，学习曲线陡峭。win together or die together，没有谁可以滥竽充数。仓库体积膨胀、权限控制局限、构建性能挑战、专用管理工具，单仓带来了新的挑战。
对领导层的要求	低。下属开发团队可以自由发挥。领导只要检查最终交付结果，不用关心代码。	高。一个不懂代码的领导层，在单仓模式下是灾难性的，后续肯定会回退到貌合神离的多仓模式。
适合场景	适合简单项目、个人项目。适合复杂但开发排期宽松的项目。尤其适合开发人力过剩、领导层不懂代码的公司。起码不拖后腿。	适合复杂项目、工业项目适合复杂且时间紧的项目，但是要有优秀工程师构成的管理层、素质中上的开发团队。适合有浓厚工程文化的公司。

在uv之前，MonoRepo范式是很难在python项目里推广的，需要公司有强大的infra、CI/CD团队。现在的年轻团队、年轻公司，恭喜你们，强大的uv workspace能够解决这个问题，为你们带来了强大的、优雅的、同时还保留python敏捷开发特点的python MonoRepo开发范式！
