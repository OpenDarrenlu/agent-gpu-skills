背景

不管是 AI application developer 还是 AI infra Engineer，包括作者本身在内的大多数人在使用/学习 vllm 的时候都是直接上手，简单阅读过 Quick Guide 之后就开始深度代码细节当中了，开始陷入 vllm 百万行源码的深渊，开始试图去理解 Scheduler、Engine 等一系列组件的实现细节。而在学习的过程中，很多涉及到的预备知识时不时会跳出来成为我们学习过程中的拦路虎、绊脚石，逐渐蚕食我们起初的雄心壮志，浇灭我们“七天时间深入理解 vllm 源码”的伟大理想。

本文将从 vllm 官方文档入手，从一个较为高层的角度去尝试理解 vllm，包括以下内容:

vllm 到底是什么？
为什么要用 vllm？
vllm 整合了学术界/工业界中的那些技术？
什么是 vLLM？

vLLM 是一个快速且易用的用于 LLM 推理和服务的库[1]。

基于官方文档中给出的定位，我们可以提出两个问题：

问题 1：vLLM 是如何达到“快速”这个目的的？

众所周知，vLLM 的主要目标是对大语言模型的推理和服务部署进行加速，但是这个快速到底是如何衡量的？一般衡量模型推理速度的指标有响应时间（latency）、吞吐量（throughput）， 更底层用于衡量硬件的还有 FLOPS 等。

根据作者的理解，vLLM 中综合了多种不同的优化策略，从多个角度、多个层次去对 LLM 的推理进行加速，主要包括以下几个方面：

从系统层面看，vLLM 通过设计的调度策略，从到达的多个请求中评估和分析，“先来先服务，后来先抢占”，结合硬件资源与请求的实时执行情况去编排执行顺序，配合 PagedAttention 的分页显存管理算法，提升了系统级别的整体资源利用率。在硬件资源出现短缺时，为了达到整体的最大效率，某些请求的资源还会被抢占，从这一点看，vLLM 针对的是服务整体的吞吐量进行加速和优化的。
从算法层面看，vLLM 支持 Speculative Decoding，Chunked Prefill 等不同的算法，这些算法改进了推理过程本身，旨在减少推理延迟，提高解码效率。
从硬件层面看，vLLM 支持使用支持 CUDA 和 HIP 架构的硬件加速模型的执行，并且还支持 GPTQ/AWQ/INT4/INT8/FP8 等多种量化算法的 kernel 实现，并且实现了多种融合算子，如 FlashAttention, SiluAndMul 等，这些特性加速了模型在硬件上的执行效率，减少了内存瓶颈对整体执行效率的影响，因此能够减少单个请求执行的响应时间。

vLLM 正式从多个层级，多个角度实现了对 LLM 推理过程的加速，最终达到了文档中提到的「State-of-the-art serving throughput」的效果。

问题 2：vLLM 的易用性体现在哪些方面？

从对主流开源社区的兼容性角度看，vLLM 与 Hugging Face 模型进行无缝兼容，在服务方面又兼容了 OpenAI 相同格式的 API，这种兼容性能够让用户在使用时更快上手。
从对于学术界/工业界的 SOTA 算法支持角度看，vLLM 支持多种解码算法，如 parallel sampling, beam search 等，同时还支持 TP/PP/EP 等分布式推理算法，再到 Prefix caching 和 Multi-LoRA 算法。vLLM 对这些主流算法的支持，不仅仅丰富了其本身的特性，更能满足用户对于推理的多样化需求。
从对于底层硬件的兼容性角度来看，vLLM 不仅支持 CPU 和 Nvidia GPU 设备，还支持 AMDGPU，各种加速器等等，对于这些不同类型硬件的广泛兼容性，能够减少用户二次适配的成本，在大多数主流硬件设备上都能够做到开箱即用，且能够达到不错的性能体验。
为什么要使用 vLLM？

vLLM 最初的提出的目的只有一个：为了有效的管理显存空间的使用，通过大名鼎鼎的 PagedAttention 技术来将 LLM 推理过程中产生的 KVCache 进行统一分配和管理，从而达到减少内存碎片，提高内存利用率的目的。

随着 vLLM 在社区和工业界中热度的不断上升，开发者团队又向其中集成了许多学术界/工业界中较为受欢迎的算法，如 Chunked Prefill/Speculative Decoding 等，这些算法极大的丰富了 vLLM 的功能，提升了 vLLM 在特定场景下的性能和效率，但也使得 vLLM 的源码体积急速膨胀，对于像作者一样后接触 vLLM 且不了解这些算法的小白，在阅读源码的过程中感到非常摸不着头脑，经常打退堂鼓。

因此在本小节作者针对 vLLM 中整合的许多算法，其算法本身以及想要解决的问题做以概述，方便读者能够对这些算法有一个初步的印象。

vLLM中设计到的优化技巧
KVCache

在大语言模型中，Transformer 模型占据着整个模型中首屈一指的位置。现代 LLM 大多采用的是 Decoder Only[2]的架构，在推理过程中，tansformer 模型「基于用户提供的 prompt 和之前生成 token，每次生成一个 token」，上述过程将一直重复直到模型输出结束 EOS（End of State）token 为止。

假设用户输入了 m 个 tokens 作为 prompts，表示为

\bold{t}=[t_1, t_2,...,t_m]\in R^{m\times1}

经过 token embedding 层之后，每个 token 会被转换成 embedding vector, 产生 embedding 矩阵

\bold{x}=[\hat{x}_1, \hat{x}_2,...,\hat{x}_m]\in{N^{m\times{d}}}

对于一个 L 层的 Transformer model 中的第 l 层

q_l,k_l,v_l = W_q\bold{x_l}, W_k\bold{x_l}, W_v\bold{x_l}

\bold{x_{l+1}}=softmax({q_l}{k_l}^T)·v_l （暂时忽略 scale）

图1 transformer模型中attention模块的计算示意图

图 1 展示了一个简单的 attention 模块在 LLM 三次迭代计算过程中的计算示意图，其中第一行表示第一次迭代计算，生成第一个 token；第二行表示第二次迭代计算，生成第二个 token...

假设我们以 A_{11}表示第一次迭代产生的唯一一个 token 向量（图中第一行唯一一个绿色长条）， A_{21} 表示第二次迭代产生的两个 token 向量中的第一个（图中第二行产生的第一个绿色长条）,则有：

\begin{array}{l} expsum_1=e^{q_1{k_1}^T} \\A_{11}=softmax(q_1{k_1}^T)·v_1\\=(e^{q_1{k_1}^T}/expsum_1)·v_1 \end{array}

\begin{array}{l} expsum_2=e^{q_1{k_1}^T}+e^{q_1{k_2}^T} \\A_{21}=softmax(q_1{k_1}^T)·v_1\\=(e^{q_1{k_1}^T}/expsum_2+^{q_1{k_2}^T}/expsum_2)·v_1 \end{array}

而 LLM 本身是一种因果模型（Causal Model）[3]，当前产生的 token 不受之前产生 token 的影响，正因如此，在 Attention 的计算过程中还需要添加一层 mask，将 qk^T 矩阵中不符合因果关系的值全部 mask 为 -\infty ，这样在 softmax 的计算过程中后一个 token 就不会对之前的 token 的计算结果产生影响，从而有

\begin{array}{l} expsum_2=e^{q_1{k_1}^T}+e^{q_1{k_2}^T} \\=e^{q_1{k_1}^T}+e^{-\infty} \\=e^{q_1{k_1}^T} \\=expsum_1 \\A_{21}=softmax(q_1{k_1}^T)·v_1 \\=(e^{q_1{k_1}^T}/expsum_2+e^{q_1{k_2}^T}/expsum)·v_1 \\=(e^{q_1{k_1}^T}/expsum_2+e^{-\infty}/expsum_2)·v_1 \\=(e^{q_1{k_1}^T}/expsum_1)·v_1 \\=A_{11} \end{array}

同理可得 A_{l1}=...=A_{31}=A_{21}=A_{11}

这种人为设计的模型特性，使得在 LLM 推理过程中第 n 个 token 的计算结果与其后产生的 token 完全无关。

也就是说图 1 所示的计算过程中存在很多重复的计算，如果能避免这些重复计算，那么就能很大程度上节省宝贵的计算资源。

由此，KV Cache 的设计便应运而生。

图2 使用了KV Cache机制的Attention模块计算示意图

如图 2 所示，使用了 KV Cache 机制的 Attention 模块，每次进行计算之前，都会将本次计算出的 kv Tensor 缓存下来供后续迭代的计算过程使用。而每次的计算过程都只需要使用新生成的 q Tensor 和之前缓存的 kv Tensor，并且计算也从矩阵乘法（GEMM）变成了矩阵向量乘法（GEMV），从而大大降低了计算的复杂度。

KV Cache 机制的提出，通过空间换计算量的策略，大大降低了 Transformer 模型中 Attention 模块的冗余计算，从而大幅减少了整体计算量，尤其是在长序列场景下，减少了对缓存非常不友好的大尺寸矩阵乘法的使用频率。

Paged Attention

在 LLM 推理过程中，显存的使用可以大致分为三个部分：模型权重、KV Cache 和其他。

在文章「Efficient Memory Management for Large Language Model Serving with PagedAttention」[4]中，作者在 Nvidia A100 GPU 上（40GB DRAM）以一个 13B 参数的 LLM 为例进行了分析，数据显示大约 65% 的显存用于存放模型的权重，这部分显存在模型推理的过程中是保持不变的。大约有 30% 的显存用于存储模型的 KV Cache，如上文中提到的，这些 KVCache 表示 LLM 推理的上下文，用于从之前生成的 token 来生成序列中下一个新的 token。剩下的约 5% 的显存空间用于存放包括 activation 值在内的其他数据。

由于模型的权重在推理期间保持不变，而包括激活值在内的其他数据仅仅占用小一份显存空间，那么使用什么方式去管理 KVCache 就显得举足轻重了，如果不能有效的管理这些产生的 KVCache，会很大程度上限制 LLM 推理的最大 batchsize，从而限制整个系统能够达到的最大吞吐量。而在 vLLM 带着 pagedAttetion 算法出现之前，这些 kv cache 确实没有得到很好的管理。

KV Cache 和传统的神经网络中的 Tensor 有所不同，它具有以下特点：

随着新的 token 的生成，一个请求的 KV Cache 会随着的增长或者收缩。每生成一个 common token，通常会将其对应的 K Tensor 和 V Tensor 写入 KV Cache 中，用于下一个 token 生成的计算。因此随着时间的推移，KV Cache 会不断地增加和累积。当生成的 token 是 EOS 时，标志着已经为当前请求生成了完整的序列，意味着当前请求的结束，那么在返回最终结果的同时也会将该请求对应的 KVCache 进行清空和回收，释放空间给其他请求使用。另外，当使用了上下文滑动窗口算法时，如果当前生成的上下文长度超过了设定的阈值，那么最先生成的第一 token 的 KVCache 就会被释放。
一个请求的 KV Cache 的生命周期和长度无法提前预知。对于一个请求而言，其结束的标志是 EOS token 的生成，因此一个请求何时结束，结束时 KVCache 的长度具体是多少，在这个请求实际完成之前是不得而知的。

基于 KVCache 的这两个特点，为其有效管理带来了以下的挑战：

由于 KVCache 的长度和生命周期不可预知，因此之前的 System 做法是使用预分配机制，为每个请求直接静态分配支持的最大序列长度的显存空间，直到请求结束后再进行归还。这种做法往往会导致内部显存碎片的产生，考虑以下极端场景，System 每个请求都分配可以容纳 2000 token 单位 KVCache 的显存空间，而每个请求实际上只生成 10 token 就会结束，预分配的显存空间出现了极大的浪费，且由于对于显存使用量的极度估计偏差，导致了 batch size 降低了约 200 倍，大大降低了整个系统的吞吐量。这是由于 KVCache 的不确定性带来的显存管理的挑战。
退一万步讲，即使我们能够准确估计每个请求的 KVCache 的显存使用量，这种预分配的机制还是会存在问题，考虑一下场景：当我们为一个请求分配了 200 token 个单位的 KVCache 的显存空间时，在这个请求完成之前这 200 个单位的内存空间是被这个请求所独占的，在整个请求服务过程中大部分时刻很多的显存并没有被真正使用到，仅仅是被该请求所占据而已，其他的请求并不能够重复利用这部分的显存。除此之外，如果 KVCache 的用量的可以预估的，那么为每个请求预分配的显存空间也将不尽相同，这么这必然会带来外部显存碎片的问题。
如今的 LLM 服务会根据采样参数（sampling parameter），为每个请求生成多个输出结果，在这种场景下， 每个请求会由多个序列（Sequence）组成，这些序列会共享部分相同的 KVCache，在之前的 System 中，由于一个序列 KVCache 被存储在连续的显存空间中，所以序列之间复用这些相同 KVCache 是不太可能的，这样的话由于冗余的存储又会造成显存的浪费。

为了解决上述问题，pagedAttention 横空出世了。

图3 PagedAttention算法实现对离散KV Cache的寻址和计算4]

vLLM 借鉴了操作系统中动态内存管理的分页管理[5]的设计理念，将一个请求的 KV Cache 划分成多个 block，每个 block 存储固定数量的 token。在这种设计下，一个请求的多个 block 也不必按照连续的方式进行存储，而是通过一个类似于操作系统中的 pageTable 的数据结构进行记录和查询，在 PagedAttention 算法中称之为 blockTable。

图4 vLLM通过使用Block来离散存储单个请求的KV Cache[4]

通过这样的设计，vLLM 可以像传统的操作系统一样，以一种更为灵活的方式管理 KVCache 所占用的显存，我们可以将一个请求想象成一个进程，而该请求的 KV Cache 所占用的 Blocks 就对应着操作系统中的 Pages，每个 Block 中所存储的 tokens 就对应着 Page 中存储着的字节。当一个进程需要找到某个对应的 KVCache 时，只需要查询 BlockTable，找到相应的 Block Id 和 TokenId 即可。

正如操作系统的分页机制一样，vLLM 对于 KVCache 的分块管理机制通过将显存划分为大小相等的固定 Block，并且为每个请求实行运行时的按需分配策略，有效缓解了碎片的问题。

图5 vLLM通过逻辑映射的方式共享相同的KV Cache[4]

不单如此，vLLM 还借鉴了操作系统中的虚拟内存的概念，将 Block 分为 Logical Block 和 Physical Block 两个概念。操作系统的设计人员考虑到在多进程/多用户的场景下，多个用户可能会执行相同的程序，为了节省 RAM 的内存空间，操作系统会为每个进程都创建一个独立的 PageTable，他们中的某个 Virtual Page 会映射到 RAM 中的唯一一份 Physical Page，而对于数据部分则映射到各自独立的 Physical Page，从而实现了在避免冲突的情况下实现只读部分的共享。vLLM 中每个请求都用于各自的 BlockTable，通过将 Logical Block 映射到 Physical Block 实现 KVCache 的共享。

图6 vLLM的KV Cache的管理策略带来的显存使用效率提升[4]

从图 6 可以看出，经过使用 PagedAttention 算法，将 KV Cache 按照 Block 进行管理，并通过逻辑映射的方式实现重复 Block 的共享，vLLM 对于 KVCache 部分的显存效率确实得到了非常大幅度的飞跃，不管是内部碎片还是外部碎片的问题都得到了很大程度的缓解，整体内存的使用效率达到了惊人的 96.3%。

Chunked Prefill

LLM的推理过程分为两个核心阶段：P 和 D。P就是我们说的预填充Prefill，而D则表示的是解码Decode。

这两个过程在计算特性上完全相反：如果序列长度是N，Prefill要负责计算出前面N个token的KvCache, 每个token的Cache都需要和前面N-1个token进行Attention进行计算，因此Prefill本身是一个巨大的GEMM（通用矩阵乘法），这也导致了它是一个计算密集型（Compute Bound）的操作；而Docde阶段则只需要当前一个token与之前的N-1个token进行attention计算，生成下一个token并保存kvcahce即可，从计算角度而言是一个GEMV（通用矩阵向量乘法），是一个显存带宽密集型（Memory Bound）的操作。

由于vLLM的调度器是按照per token为间隔单位进行请求调度与资源分配的，而过去一个完整的Prefill与Decode由于处理单个token计算量差距太大，Prefill和Decode阶段是完全在两个独立的批次。想象这样一个场景：一个128K上下文的Prefill请求和一个32K的Decode请求被放在了同一个批次中进行处理，这就意味着他们需要使用同一个PagedAttention算子的不同线程进行计算，那么即便负责decode请求的线程已经早早的完成了计算，他也会被算子内部的sync机制拦住，需要等待128K Prefill请求完成后一起返回。

这对于Decode这种直接影响到用户体验，对于延迟敏感度极高的操作简直是噩梦。因此，早期的vLLM调度器选择将P和D分别放在不同的队列进行独立处理，这就是人们常说的PD分离——哦不好意思开玩笑的，这并不是>_<。

影响用户体验：将Prefill和Decode进行分别处理能够让处于Decode阶段的请求免受Prefill请求的打扰，以较低的延迟生产出下一个token，但也仅仅是下一个token了，别忘了我们提到过，vLLM使用一个以token为间隔单位进行调度的调度器，并且策略是Prefill优先，那么很有可能当你当前这个token生成完了之后，vLLM觉得it's time for handle the prefill batch，然后突然去调度执行一个prefill batch，你只能眼睁睁的等着这个巨慢无比的请求结束了Prefill，生成了first token之后，才能继续执行，这里也提现了Continous Batch的基本思想~。

对于上述情况，用户侧较为直观的感受就是生成到中间突然变卡了，那么很有可能就是生成到一般突然有请求”插入“了，对于最坏的情况，这种抢占式的调度还有可能造成decode请求的饿死，后果恶劣。

资源利用率低：除了给用户带来糟糕的体验之外，Prefill和Decode的独占式执行还会造成系统资源的空闲：无论当前执行的是哪种操作，计算资源和内存带宽资源总会有一个处于利用率不饱和的状态，这样是无法发挥出系统的最大效率的。

显存不友好：另外，超长的Prompt在执行Prefill时需要一次性分配大量的显存空间，用于存放Activation Tensor和 KV Cache，将显存占用量瞬时推高，极易造成OOM，并且对于消费级显卡和入门级玩家非常非常不友好。

Chunked Prefill是怎么做的？

Chunked Prefill的出现将为上述问题带来了曙光。

首先，CP将一个超长的等待prefill操作的prompt进行切分，同时将原先的prefill优先策略改为了Decode优先。每次调度器在调度时，由于对prefill请求进行了切分，所以不用再去考虑改调度prefill batch还是 decode batch了，而是将prefill request 和decode request混合在同一个batch中进行处理，先尽可能的向一个batch中添加decode req，然后利用剩余的可用空间来放置一些被切分之后的prompt chunk。这样一个kernel中的每个线程不管是处理P还是处理D，他们有着相差不大的计算时间，不需要再让D线程空置等待P线程了，这样的方式解决了早期不同类型的操作无法混合处理的问题。

也正是因为P和D请求能够放置在同一个batch中被同一个kernel进行处理，也就顺理成章的解决了单边资源利用率闲置的问题，同一个batch中的P和D对计算资源和带宽资源都能够进行很好的利用，充分发挥了系统效率。

另外，将一个超长的prompt进行切分之后，每个chunk在处理时需要用到的显存资源大大降低，更好的规避了OOM的风险，通过人为调整切分阈值，使得用户能够在消费级显卡上，在显存容量有限的情况下使用vLLM获得更好的用户体验和资源利用率。

当一个Prompt的所有chunk都完成了KV Cache的计算之后，他们的KvCache会被拼接成一个完整的KV Cache，以便后续Decode阶段进行使用，生成的KVCache和后续的计算结果与原来相比没有任何差异。
