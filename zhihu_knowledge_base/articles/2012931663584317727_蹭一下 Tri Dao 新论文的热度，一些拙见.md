1. Background

Speculative Speculative Decoding (SSD) 本质上是一种并行投机解码 (Parallel Speculative Decoding) 范式. 我们假设读者对 Speculative Decoding 已经很熟悉了，且对相关进展 (EAGLE, MTP) 有一些了解.

在并行投机解码中，draft model 和 target model 是并行推理的。为了更好的帮助读者理解，我们建议读者从标准投机解码的过程来理解，把 pre-verify 对应到标准 Spec 的 draft 过程；把 post-verify 对应到标准 Spec 的 verify 过程：

对于标准 Spec 的 draft 过程，target model 是停止的。而 pre-verify 策略就是让 target model 并行的基于输入 prefix 进行推理得到 1 个 token，用来验证 draft 生成的第一个 token。如果第一个 token 就错了，就不用单独的为这批 draft tokens 做 verify，省下一轮 verification 时间。
对于标准 Spec 的 verify 过程，draft model 是停止的。而 post-verify 策略就是让 draft model 基于上一轮生成的 draft tokens 并行的进行推理下一轮 draft tokens。如果上一轮 draft tokens 全对，就不用单独的再次进行一轮 draft 过程，省下一轮 drafting 时间。

那我们是不是可以合理的猜测：t_i^* 就在被拒绝 token x_i 对应的 logit distribution p_{i-1} 的 top-k 里？

很明显，这个猜测是有支撑 evidence 的。因此我们就找到了一种推测 t_i^* 的手段！因此，我们就可以对于上一轮的每个 draft token x_i , 假设 x_i 被拒绝，可能的修正 token 是 t_i^* , 然后让 draft model 基于 t_i^* 继续生成 draft tokens。如果 x_i 真的被拒绝了，并且修正 token 真的是 t_i^* ，那我们就可以把基于 t_i^* 生成的结果喂给 target model，仍然做 post-verify。在 SSD 原文中，他们把基于 t_i^* 生成的结果叫做 “cache”，命中上述情况称为 cache-hit. 如下图所示：

5. 并行投机采样的深入分析

另外，我注意到 SSD 其实仍然面临 PEARL 的一个问题：draft model 需要独占 GPU。对应的一个问题是，一台 8 卡服务器，draft model 占用至少 1 张卡，target model 最多只能使用 7 张卡。虽然 nano-PEARL 通过 padding 的手段允许 target model 使用 非 2 幂次的卡量，但仍然会引入冗余计算量，减弱整体加速效果。

我认为他们这是在答非所问。而这个问题真正的解决方案应该是 DT 分离。事实上，我认为投机采样的发展历程里面，DT 分离是必然的，只不过是长期趋势，短期很难实现。

6. 为什么要 DT 分离
draft model 的结构越来越复杂化，例如 eagle3 和 dflash。想在同一个系统和设备上同时管理 draft model 和target model 本身就越来越困难。
draft model 的参数量在scaling。像 eagle3 的参数量已经几乎是 eagle1 的两倍了 (特指 attn 部分)，dflash 更是使用了多层 draft model。MiMo 的 MTP 也是用了 3 层的结构。当 draft model 的显存占用显著提高，target model 的可用显存会被大幅压缩。另外，draft model 需要捕获的 cuda graph 等，包括为不同的 batch size 和解码深度捕获不同的 cuda graph，也会造成极大的资源压力。
draft model 的本职是猜测 target model 想要什么——就像 LLM 猜测人类想要什么。本质上来说，把 target model 当成顾客，draft model 就是服务者。因此，标准 LLM 系列可用的优化，基本都会在 draft model 上再走一遍：continuous batching / chunked prefill / TP / prefix cache 等等。让 draft model 单独占据 device，并暴露为一个完整的服务，会显著降低工程上集成各种优化技术的难度。
