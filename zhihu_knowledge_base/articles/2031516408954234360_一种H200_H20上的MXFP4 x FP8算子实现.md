DeepSeek V4的MoE层使用的是MXFP4格式。

目前主流推理框架中对于H卡上的MXFP4权重主要是使用Marlin算子进行W4A16推理，也就是说权重使用MXFP4，激活值使用BF16。然而Marlin算子在H卡上性能较差（不支持WGMMA），且不支持H卡上的FP8。导致H卡的性能在推理MXFP4权重时完全无法发挥出来。

我开发的通用高性能量化推理算子库Humming支持W4A16和W4A8的推理，其中W4A16的性能还不错，大Batch性能基本与cublas bf16xbf16持平，但W4A8的性能在大BS+小group下的性能就和W8A8（大group）有差距了。今天想到一种优化方法，可以大幅提升大BS下的性能，拿出来分享一下。

（相关优化已经合并进入vLLM主线，欢迎使用和反馈：https://github.com/vllm-project/vllm/pull/41083 ）

为了方便，从最简单W4A16情况讲起。

        a = get_next_a()
        b = dequant(get_next_qb())
        final_accum += mma(a, b * bs)




W4A8的实现

与W4A16类似，把FP4反量化到FP8的第一步也很简单：







但与W4A16不同的是，CUDA中并没有提供FP8的算术运算指令（指普通的加减乘除），FP8的计算必须通过MMA指令来完成一整块的矩阵乘法。Blackwell卡（B200/5090等）支持在MMA中使用scale，但H卡和4090上不支持，因此在进行MMA之前我们就没有机会乘以scales。

因此我们需要在MMA计算结果上再来乘以scales（MMA计算结果是FP32），如果使用torch演示的话大概相当于：

mma_shape_m = 16
mma_shape_n = 8
mma_shape_k = 32

final_accum = torch.zeros((mma_shape_m, mma_shape_n))
accum = torch.zeros((mma_shape_m, mma_shape_n))
while has_next_group():
    accum.zero_()

    bs = get_next_bs()
    for _ in range(group_size // mma_shape_k):
        a = get_next_a()
        b = dequant(get_next_qb())
        accum += mma(a, b)

    final_accum += accum * bs

这种办法也能得到准确的结果，但是：

需要双倍的寄存器来存储结果（一份存储最终结果，一份存储临时结果），但在大batchsize下的kernel中，寄存器本来就紧张，这样会限制尺寸。
乘以scales所需的计算量比W4A16那种方式大得多，并且groupsize越小，计算量就越大。而MXFP4的groupsize只有32，对应的计算量就会大得多。
W4A8的新版实现

MXFP系列的量化方法的特点是它使用的scales是E8M0类型的，当我们给一个数字乘以E8M0 scales时，其实基本相当于在它的指数位上做加减操作。因此，我们可以通过整数加减法来模拟出FP8E4M3乘以E8M0的行为，从而使用和W4A16类似的计算路径，节省了寄存器和计算量。但是有几个点需要解决

FP8E4M3的指数位只有4位，而E8M0有8位，理论上E8M0的动态范围非常大，根本没办法通过整数加减实现完美的模拟。
需要注意指数位全是0的情况，这种情况属于非规约数，使用的是额外的计算逻辑。
E4M3的动态范围问题

事实上，尽管E8M0的动态范围非常大，但实际权重中scales的动态范围很小，拿典型的gpt-oss和DeepSeek-V4举个例子，算一下动态范围。

import glob

import safetensors.torch
import torch
from tqdm import tqdm

channel_range_max = 0
tensor_range_max = 0
model_name = "DeepSeek-V4-Pro"
for filename in tqdm(glob.glob(f"{model_name}/**.safetensors")):
    data = safetensors.torch.load_file(filename)
    for key, tensor in data.items():
        if tensor.dtype != torch.float8_e8m0fnu and not key.endswith("_proj_scales"):
            continue

        tensor = tensor.view(torch.uint8)
        max_value = tensor.max(-1)[0].view(-1)
        min_value = tensor.min(-1)[0].view(-1)
        channel_range_max = max(channel_range_max, (max_value - min_value).max())

        if "gpt" in model_name:
            max_value = tensor.view(tensor.size(0), -1).max(1)[0]
            min_value = tensor.view(tensor.size(0), -1).min(1)[0]
        else:
            max_value = tensor.max().view(-1)
            min_value = tensor.min().view(-1)
        tensor_range_max = max(tensor_range_max, (max_value - min_value).max())


print(channel_range_max.item(), tensor_range_max.item())

而FP4反量化FP8之后指数位的16个数值中只有最小的4个是有效的，我们可以容忍scales的指数位小于12的动态范围。因此新的解决方案是：

按channel/tensor维度调整scales（视为uint8），统一减去对应维度的最小值。在反量化得到的FP8种通过整数加法应用这个scales。
把对应维度的最小值变成channelwise / tensorwise的scales，在epilogue阶段应用。

Humming的实现中，由于此前的开发并不支持groupwise + channelwise两种scale组合的形式，因此暂时使用的是groupwise + tensorwise组合的形式。gpt-oss的tensor维度动态范围最大值大于12，我使用截断的方式处理（把小于最大值-12的值截断），因此是部分有损的，但实际影响很小（相当于一个最大值是1的矩阵下，把小于0.0005的数值截断为0.0005），并且我们可以对权重部分也进行重新缩放来减小误差，实际只会影响小于0.00005的数值。

事实上，这个过程完全等价于把MXFP4反量化到BF16，再量化到FP8，自然是没有损失的。

非规约数问题

在FP8数值上应用scales的时候，我们不能直接在指数位上做加法，因为要考虑非规约数。这里我们只考虑FP4数值对应的情况：

对于指数位不全部为0的情况：直接在指数位上做加法
对于指数位全部位0，但底数位不是0的情况：在指数位上做加法，加数为第一种情况的数值减去1，同时把底数位改为0。
对于数值0：不做任何处理




要处理这些不同情况，我们当然不能在cuda中用if-else来处理，会对性能有极大影响。humming使用的是 prmt的PTX指令，这里把核心逻辑拿出来注释解读：

CUDA_INLINE uint2 dequant_single_for_mxfp4_fp8(const uint32_t qb, const uint32_t exp_offset) {
  // qb打包了8个FP4值，这8个值在同一个group里面（共享同一个scale）
  // 8个FP4值是按02134657的顺序打包的，反量化后会变回01234567的顺序
  // exp_offfset位根据上面描述得到的处理后的scale。

  // res: 存储反量化结果。需要2个uint32_t来存储8个FP8
  uint32_t res[2];
  // signs: 里面每个数做(x & 0x80808080)即为8个FP8的符号位
  uint32_t signs[2] = {qb << 4, qb};
  // others: 8个FP8的除了符号位以外的其他位
  uint32_t others[2] = {qb & 0x07070707, (qb >> 4) & 0x07070707};

  // 两个buffer共8个bytes，分别对应FP4低3位的8种可能数值
  // 对于0: 不做任何处理，所以取0
  // 对于大于1的数值（即指数位不为0）：左移3位（乘以8），与FP8指数位对齐
  // 对于1：乘以8再减(1 << 2 = 4)（用于消除底数位），注意需要特别处理exp_offset为0的情况
  uint32_t exp_offset_buffer1 = (exp_offset * 0x08080800) + (exp_offset ? -0x00000400 : 0);
  uint32_t exp_offset_buffer2 = exp_offset * 0x08080808;

  // prmt指令: 以FP4的低3位的所有可能取值为索引，取得之后要加到FP8上的数值
  uint32_t exp_offsets[2] = {
    __byte_perm(exp_offset_buffer1, exp_offset_buffer2, qb),
    __byte_perm(exp_offset_buffer1, exp_offset_buffer2, qb >> 16)
  };

  PRAGMA_UNROLL
  for (uint32_t i = 0; i < 2; i++) {
    // lop3_and_or(a, b, c)即(a & b) | c，可以通过单条lop3指令完成
    uint32_t val = lop3_and_or(signs[i], 0x80808080, others[i] << 2);
    // prmt指令：把02134657变回01234567的顺序
    val = val + __byte_perm(exp_offsets[0], exp_offsets[1], 0x6420 + 0x1111 * i);
    res[i] = val;
  }

  // 注意uint2是两个uint32_t打包，不是指2bit的uint
  return *reinterpret_cast<uint2*>(res);
}
