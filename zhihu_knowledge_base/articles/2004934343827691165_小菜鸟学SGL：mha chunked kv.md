背景

deepseek v3/r1系列模型优化时，long prefix场景有个特别优化叫做chunked kv或者chunked prefix

首先，我们可以先看一下long prefix表示什么含义

common prefix：全局上下文公共的token序列
chunked prefill：chunked context

如果是第一种情况，prefix很长意味着query中大部分token已经计算好了。若这是common case，prefill阶段需要计算的工作量就不大了，而且prefill可以组成很大的batch。但，这与目前prefill阶段特征不符合的。

若以上分析有一定的可理性，long prefix暂时可以当作是支持long context的chunked prefill场景

mha chunked kv

pr：https://github.com/sgl-project/sglang/pull/5113

从下图可以看出计算分为两个部分：

extend attention：q/k/v都是hidden_states变换计算出的（forward_batch.extend_seq_lens）
extend与prefix attention计算：q（extend），k/v (从latent cache读出后做了kv_b_proj变换）
DeepseekV2AttentionMLA::forward_normal_chunked_kv

在进行prefix的attention计算之前，forward_normal_chunked_kv调用forward_batch.prepare_chunked_prefix_cache_info统计每个chunk每个prefix token序列对应的元数据。

元数据组成
prefix_chunk_cu_seq_lens/prefix_chunk_max_seq_lens（fa的blocktable是变长）：backend计算
prefix_chunk_starts，prefix_chunk_seq_lens，prefix_chunk_cu_seq_lens：chunk中prefix token序列的latent cache在kvcache存储位置

此过程只执行一次，通过forward_batch.num_prefix_chunks判断，第一次调用被赋值。

prefix划分几个chunk呢

chunk capacity描述chunk中每个prefix token序列要处理token；bs中各个prefix token序列平分chunk capacity，即使某个prefix没有需要处理的token，这部分capacity也不会被其他prefix使用。

对于一个bs，需要的chunk个数由最长的prefix token序列决定。

class ForwardBatch:
    ...
    def get_max_chunk_capacity(self):
        # Maximum number of tokens in each chunk
        # TODO: Should be changed to a better value, maybe passed through server args
        return 128 * 1024
    ...

    # Called before each attention module if using chunked kv cache for prefill
    # Some of the codes are adapted from https://github.com/vllm-project/vllm/blob/main/vllm/v1/attention/backends/mla/common.py
    def prepare_chunked_prefix_cache_info(self, device: torch.device):

        from sglang.srt.mem_cache.memory_pool import MLATokenToKVPool

        assert isinstance(
            self.token_to_kv_pool, MLATokenToKVPool
        ), "Currently chunked prefix cache can only be used by Deepseek models"

        if self.prefix_chunk_len is not None:
            # Chunked kv cache info already prepared by prior modules
            return

        self.prefix_chunk_idx = -1

        # chunk_capacity is the maximum number of tokens in each chunk
        chunk_capacity = self.get_max_chunk_capacity()
        self.prefix_chunk_len = chunk_capacity // self.batch_size

        self.num_prefix_chunks = (
            max(self.extend_prefix_lens_cpu) + self.prefix_chunk_len - 1
        ) // self.prefix_chunk_len

        ...

        # Precompute the kv indices for each chunk
        self.prepare_chunked_kv_indices(device)

从req_to_token_pool.req_to_token获取kvcache offset

forward_batch.prepare_chunked_prefix_cache_info在执行最后调用prepare_chunked_kv_indices从model_runner.req_to_token_pool.req_to_token表中读取(req_id, token_index)开始长度为prefix_chunk_len的token序列在kvcache分配的位置。chunk_kv_indices记录当前chunk中prefix token序列在kvcache offset，有了这些信息便可直接从kvcache读取latent cache。以后的处理过程与forward_normal从latent_cache解压key/value类似。
