1. TC的基本使用

任何python函数或pytorch module都可以作为参数传递给TC，从而获得优化和加速。pytorch会用优化过的函数来替换掉原本的函数。

TC的使用方式有两种,一是显式调用，二是使用装饰器。

直接调用

@torch.compile
def opt_foo2(x, y):
    a = torch.sin(x)
    b = torch.cos(y)
    return a + b
装饰器（decorator）
t1 = torch.randn(10, 10)
t2 = torch.randn(10, 10)

@torch.compile
def opt_foo2(x, y):
    a = torch.sin(x)
    b = torch.cos(y)
    return a + b
print(opt_foo2(t1, t2))
使用TC对pytorch Module instance 进行优化。
def nested_function(x):
    return torch.sin(x)

@torch.compile
def outer_function(x, y):
    a = nested_function(x)
    b = torch.cos(y)
    return a + b

print(outer_function(t1, t2))

通过torch.compiler.disable可以在禁用对嵌套过程中某个函数的trace。下面的例子将禁用对函数complex_function的编译， 其所包含的函数调用也会被禁用。

def complex_conjugate(z):
    return torch.conj(z)

@torch.compiler.disable(recursive=True)
def complex_function(real, imag):
    # Assuming this function cause problems in the compilation
    z = torch.complex(real, imag)
    return complex_conjugate(z)

def outer_function():
    real = torch.tensor([2, 3], dtype=torch.float32)
    imag = torch.tensor([4, 5], dtype=torch.float32)
    z = complex_function(real, imag)
    return torch.abs(z)

# Try to compile the outer_function
try:
    opt_outer_function = torch.compile(outer_function)
    print(opt_outer_function())
except Exception as e:
    print("Compilation of outer_function failed:", e)

值得注意的是，torch.compiler.disable函数的参数recursive的取值会影响该函数的作用范围：

若recursive=True，则TC对该函数及该函数调用的函数完全不处理。
若recursive=True，则TC仅不处理当前作用函数，对其所调用仍会处理。

除了torch.compiler.disable，还有一个方法用于将某个无法被TC编译的函数/Module从图中排除的方式是使用torch._dynamo.disallow_in_graph，被排除的函数将会在图执行的过程中从途中break out， 并以eager mode的方式被执行。

与torch._dynamo.disallow_in_graph相对应的，torch._dynamo.allow_in_graph会将一个函数以和黑盒的形式置于生成的计算图中。凡是使用了此方法的函数将会省略一切的安全检查，包括是否会跳出图，对于闭包的处理等，在使用时务必需要小心。

上述两个API的作用范围都是global的，因此在切换不同的compiler backend时，之前disallow的operator需要通过API重新allow in graph。

3. TC的最佳实践

TC会递归地编译 target function 或 target Module中的每个函数调用，除过内置指令和函数，以及torch命名空间内的某些函数（http://torch.xxx）。

最佳实践：

尽量以一个函数/Module的highest level call作为entrypoint来递归的编译这个函数/Module，当其中某些函数调用出现错误或者问题时使用torch.compiler.disable来禁用其编译。

将某个函数或者Module集成至当前函数/Module时，提前尝试使用TC对其进行编译。By doing this, 能够及早发现可能存在的问题，避免后期挨个排查。

无法通过TC编译的函数或sub-module，将其禁用后再集成。

从leaf function和leaf submodule开始编译包含多个、多重函数调用和sub-module的复杂模型。

4. TC的技压群雄

相对于之前的TorchSript script, TorchScript tracing以及fx tracing. TC主要具有以下优势：

对原代码的改动最小。

更好地支持条件分支。

无需推断数据类型，从而无需在代码中注释数据类型。

更好地支持非pytorch函数

数据流	支持	不支持	不支持	支持
类型注释	需要	不需要	不需要	不需要
非pytorch函数	不支持	不支持	不支持	支持
代码改动	需要	不需要	不需要	不需要
5. TC的底层技术

TorchDynamo是支持TC实现的重要组件之一，负责just in time 将任意的python代码编译为FX graph，再进一步对所得的FX graph进行优化。TorchDynamo在运行时分析python字节码并检测PyTorch算子的调用。

TorchInductor作为TC的另一个重要组件，进一步将FX graph编译成优化后的kernel。但TI并不是TC的唯一后端，TorchDynamo允许用户使用不同的后端，包括自定义后端。下例通过自定义后端来检查TD产生的FX graph是如何处理data flow的。

from typing import List
def custom_backend(gm: torch.fx.GraphModule, example_inputs: List[torch.Tensor]):
    print("custom backend called with FX graph:")
    gm.graph.print_tabular()
    return gm.forward

# Reset since we are using a different backend.
torch._dynamo.reset()

def bar(a, b):
    x = a / (torch.abs(a) + 1)
    if b.sum() < 0:
        b = b * -1
    return x * b

opt_bar = torch.compile(bar, backend=custom_backend)
inp1 = torch.randn(10)
inp2 = torch.randn(10)
opt_bar(inp1, inp2)
opt_bar(inp1, -inp2)

6. TC的进阶参数

具体而言，TC会尝试编译特定范围内的所有 python frame（函数调用），并将编译后的结果缓存起来，如果之前编译的结果与后面的调用不符，那么这个frame会被编译多次，称之为"guard failure"。

需要注意的时，编译结果的缓存每个“代码对象”一份，而不是每个frame一份。

如何理解？代码对象即python代码编译而成的字节码，是操作执行和变量信息的集合，为静态概念。frame指的是python的执行帧，时python运行过程中所创建，包含上下文，调用栈信息，变量具体值等信息，为动态概念。

前文意为，TC为每个Python Code Object绑定一个编译结果的缓存，而非为每个由此Code Object所创建的python frame都绑定各自的编译缓存，同一个Code Object创建产生的frame共享同一个编译缓存。

def compile(
    model: _Optional[_Callable] = None,
    *,
    fullgraph: builtins.bool = False,
    dynamic: _Optional[builtins.bool] = None,
    backend: _Union[str, _Callable] = "inductor",
    mode: _Union[str, None] = None,
    options: _Optional[_Dict[str, _Union[str, builtins.int, builtins.bool]]] = None,
    disable: builtins.bool = False,
)

Args:

model [Callable]：欲优化之Module / func。

full_graph：是否强制将输入的model作为一张图来捕获，如果不可行，则报错。

dynamic：是否使用动态shape追踪，若开启可以尽可能避免因为shape的变化而导致的重新编译。

backend：指定捕捉到的FX graph会被送到何处处理，默认为"Inductor"，可使用torch._dynamo.list_backends()来查看所有可用选项。也可以自定义一个backend用于DIY对图的优化处理。

mode: 可指定为下面几种mode中的一种

"default": 默认模式，平衡了性能和开销。
"reduce-overhead"： 使用了CUDA graph，降低python层的开销，代价是增加了显存用量。
"max-autotune"：此模式利用基于Triton或模板的matmul，在GPU上运行时默认使用cuda graph。
"max-autotune-no-cudagraph"： 同上，区别是不使用cuda graph。

options：一个字典，用于将一些选项传给backend，详情可torch._inductor.list_options()来查看可接受的参数。

disable：如果设置为True，则 TC不对传入的模型进行任何编译优化操作，直接返回原始模型。通常用于测试目的。
