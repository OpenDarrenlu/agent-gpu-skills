最近外网看到一个分享还不错的，推荐可以扫一扫，油管搜Context Engineering for AI Agents with LangChain and Manus。下文为阅读笔记。

随着大语言模型（LLM）能力的飞跃，我们正站在技术范式转换的关键点。自2022年12月ChatGPT引发“提示工程”（Prompt Engineering）浪潮之后，大约在2023年5月，伴随“Agent之年”概念的兴起，一个新的核心技术领域——上下文工程（Context Engineering, CE）开始占据中心地位。

上下文工程的本质在于“精心策划和设计上下文窗口，使其恰好包含下一步所需信息” [Conversation History (CH)]。对于自主运行的AI Agent而言，上下文工程不仅是性能优化的手段，更是保证其功能可靠性的基石。

Agent时代的上下文挑战

为什么上下文工程在Agent领域变得如此关键？

AI Agent 通常需要不断调用工具（如文件操作、代码执行、搜索）并接收观察结果。以 Manus 和 Anthropic 的生产级 Agent 为例，执行一项典型任务可能需要数十次甚至数百次工具调用。每一次调用及其结果都会被添加到消息历史中，导致上下文快速爆炸。

这种上下文的无限增长直接导致了Agent性能的下降：

上下文中毒（Context Poisoning）： 模型产生幻觉，例如编造并试图重复使用一个不存在的工具或项目。
上下文干扰（Context Distraction）： 当上下文超过某一阈值（例如100k Token）时，模型会倾向于重复已有的动作，而不是采纳新的计划。
上下文混乱（Context Confusion）： 工具数量增多，特别是当工具功能相似时，模型表现变差。
上下文冲突（Context Clash）： 连续的工具调用结果相互矛盾，导致模型决策质量下降。
不要过早尝试模型微调

面对Agent性能下降，开发者常会陷入“微调或后训练”（Post-training）的诱惑。然而，这是初创公司需避开的首要陷阱：

产品迭代受限于模型迭代： 训练和评估周期漫长，产品创新速度会受限于模型迭代速度。
重复基础模型公司的劳动： 如果Agent需要支持复杂、可无限扩展的行动空间（例如多上下文推理，MCP），通过强化学习进行微调需要巨大的工程投入，这相当于在重复基础模型公司已完成的工作。

因此，上下文工程成为了应用层与模型层之间最清晰、最实用的界限。优化通用模型的使用效率，而非重构模型本身，是Agent开发者的核心策略。

上下文工程的五大核心策略

为了应对上下文爆炸的挑战，社区通过实践总结出了五大共通的策略：

卸载上下文 (Offload context)： 将工作内存或冗余信息转移到外部存储（如文件系统），例如用文件记录笔记、计划或保存Token密集型内容。
减少上下文 (Reduce context)： 通过总结或精简消息历史、工具输出，减少Token消耗。但需警惕信息丢失的风险。
检索上下文 (Retrieve context)： 使用混合检索方法和重排系统，按需将相关信息（包括工具描述）重新组装到提示中。
隔离上下文 (Isolate context)： 将任务分解，通过多Agent设置分割上下文，让每个子Agent拥有独立的、更小的上下文窗口。
缓存上下文 (Cache context)： 优化Token缓存。例如，缓存Agent指令和工具描述作为前缀（Prefix），只将可变上下文和最新观察结果添加到后缀（Suffix），这能显著降低推理成本（如 Claude-sonnet 缓存输入 Token 便宜 10 倍）。
五大上下文工程策略在不同Agent项目中的应用概览。注：Manus 倾向于 Offloading 避免信息损失
Manus的工程实践：深度解析

Manus 的实践为我们提供了应对上下文挑战的具体方法，尤其在上下文减少、隔离和工具卸载方面。

1. 上下文减少：可逆的压缩 vs. 不可逆的总结

Manus 将上下文减少分为两种核心操作，重点区分了它们的可逆性：

A. 压缩（Compaction）：可逆的精简

压缩用于处理Token密集型操作，它将实际内容卸载到文件系统或外部状态，只在消息历史中保留最小必要的信息（如文件路径或URL）。

示例：文件写入操作的压缩 在工具调用中，完整的 content 字段被移除，只保留 path。

压缩操作示例图

这种方法是可逆的。Agent 需要时，可以通过路径或 URL 重新检索完整内容。这对于依赖先前动作进行链式推理的Agent至关重要。

B. 总结（Summarization）：谨慎处理的不可逆操作

只有当多次压缩后上下文长度的增益微小时，才应引入总结。总结是传统的、不可逆的操作。

总结的触发阈值，被称为“腐烂前阈值”（Pre-rot threshold），通常设定在 128K 到 200K Tokens 之间，低于模型的硬性上下文限制。

总结操作示意图

技巧： 在进行总结时，为了确保输出稳定和高召回率，建议使用结构化输出（Schemas）来指导模型，而不是采用自由格式提示。Manus 始终保留最后几轮工具调用及其结果的完整细节，以确保模型平稳接续工作。

上下文长度与减少操作的决策图。随着上下文长度增长（Turns），Agent 优先使用 Compaction，当接近 Pre-rot threshold 时，才转而使用 Summarization 的决策过程。
2. 上下文隔离：借鉴并发模型

多Agent设置是隔离上下文的有效手段，但它也带来了沉重的同步开销和协调问题。Manus借鉴了编程语言中并发处理的智慧：

模式	描述	优势/应用
通过通信共享记忆 (By communicating)	主 Agent 向子 Agent 发送一个明确指令，子 Agent 仅在其指令的上下文内工作。	适用于任务短、指令明确，且主 Agent 只关注最终结果的场景。简单、高效。
通过共享记忆进行通信 (By sharing context)	子 Agent 可以查看完整的先前上下文（包括工具使用历史）。	适用于需要完整历史支持的复杂场景；代价是 Token 预填充昂贵，且无法复用 KV 缓存。

3. 上下文卸载：工具分层抽象

除了卸载工作内存到文件，Manus还通过分层行动空间来卸载工具本身，从而解决工具过多导致的“上下文混乱”问题：

级别	名称	描述	优势
Level 1	功能调用 (Function Calling)	数量固定的、原子性的核心功能（如搜索、文件读写）。	标准、Schema安全。
Level 2	沙盒实用程序 (Sandbox Utilities)	Agent 在完整的虚拟机沙盒中运行，可以调用预安装的命令行工具（CLI）。	无需修改模型的 Function Calling 空间即可扩展能力；适合处理大型输出（直接写入文件）。
Level 3	包与API (Packages & APIs)	Agent 编写 Python 脚本来调用预授权的第三方 API 或自定义包。	适用于数据密集或链式任务（如“获取城市 → 获取ID → 获取天气”），保持模型推理上下文的清洁。

关键设计： 所有三个层级都通过标准的功能调用（Function Calls）作为代理接口来访问。例如，Agent通过调用一个 shell 功能，来执行 Level 2 或 Level 3 的复杂操作，从而保持了接口的简洁性、缓存友好性及设计的正交性。

总结

上下文工程的五大策略并非孤立存在，而是相互协同的：

卸载和检索增强了减少操作的效率。
可靠的检索使得隔离更加安全可行。
隔离延缓了上下文的增长速度，从而降低了减少操作的频率。

此外，我们还要警惕上下文过度工程化。Manus 团队的经验表明，最大的性能提升往往来自于简化和移除不必要的技巧。上下文工程的目标是让模型的工作变得更简单，而不是更复杂。

“构建更少，理解更多。”
