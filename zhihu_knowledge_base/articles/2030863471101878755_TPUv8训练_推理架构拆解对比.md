谷歌于 2026 年 4 月 22 日 Google Cloud Next '26 主题演讲中正式揭晓第八代 TPU。神奇的是，v6 Trillium(延龄草)、v7 Ironwood(铁木)建立的植物命名传统在 v8 上被放弃。官方市场名回归纯数字加后缀。将训练与推理在硅片层面彻底分离,推出 TPU 8t(代号 Sunfish,由 Broadcom 协同设计)和 TPU 8i(代号 Zebrafish,由 MediaTek 联发科协同设计)。第八代 TPU 重回 v5e/v5p 式的训练/推理分叉战略，这意味着 Google 在经历了 Trillium (v6e) 和 Ironwood (v7) 两代"统一 SKU"之后，再次确认训练与推理工作负载的经济学与架构需求分歧正在扩大而非收敛。

1.1 推理和训练需求

训练的 batch 是同构的。所有样本同时进入同一个前向、反向、参数更新循环，走完全相同的计算路径，每一层共享同一组权重。整个训练步是一张静态计算图。框架通过数据并行把这个 batch 摊到几千张芯片上，每张芯片本地的 micro-batch 不大，但有效全局 batch 是几千乘以 micro-batch，量级到几百万 token。这个数字直接决定了训练的算术强度——每个权重字节读进芯片之后，都要被几百万 token 在前向、反向 dx、反向 dW 三个 pass 里反复触碰，复用次数往往到几千以上。结果就是训练步的算术强度稳稳停在几千 FLOP/byte 量级，远超任何现代加速器的 ridge point，瓶颈毫无悬念地落在算力上。

推理的 batch 是异构的。同一个 batch 里可能有三个用户在跑 prefill chunk、二十七个用户在跑 decode 单 token，每个请求的长度、生成进度都不同。继续往下走一个 iteration，batch 的形状可能完全变了——已经生成完的请求出 batch，新到达的请求入 batch。vLLM、Orca、SARATHI 这些系统的核心调度技术 continuous batching 就是为了应对这种动态形状。多 batch 同样能提升算术强度，但提升机制和上限和训练完全不是一回事。

具体说，把 B 个用户的 decode 打到一起时，模型权重在它们之间共享——权重读一次给所有用户用，相当于把 decode 那个可怜的 1 FLOP/byte 直接乘以 B。但 KV cache 不能共享，每个用户的 KV 必须独立读自己那一份。所以算术强度的提升曲线是：batch=1 时大约 1 FLOP/byte，batch=8 时到 7，batch=32 到 25，batch=64 到 45，batch=128 到 75，batch=256 到 110。看起来不错，但和 H100 的 ridge AI 295 始终隔着一段——关键在于 KV cache 那一项随 batch 线性增长，分母不会真正消失。这是和训练相比的结构性差异：训练的 batch 越大算术强度越高、没有上限；推理的 batch 把算术强度抬高之后会渐近收敛到一个有限值。

更现实的约束是 KV cache 容量直接把 batch 锁死。Llama-3 70B 在 GQA-8 下每 token 需要 320 KB 的 KV，batch 256 配 8K 序列就吃掉 671 GB——一颗 H100 80GB 装不下，TPU v5p 95GB 装不下，连 TPU 8i 的 288GB 也得三颗才够。换成长上下文场景更夸张，batch=8 配 128K 序列就要 335 GB。所以推理 batch 不是想开多大开多大——它被 HBM 容量减去模型权重之后的剩余空间锁死。训练几乎没有这个约束，因为 ZeRO 或 FSDP 把优化器状态摊到几千张芯片上，每张芯片剩下的 HBM 足够容纳激活。

推理还有一道训练完全不存在的硬约束：延迟 SLO。TPOT 必须控制在几十毫秒以内，不可能等 256 个用户凑齐再开 batch——最早到的那个用户已经超时了。所以云端推理只能靠 continuous batching 动态打包：iteration 级别重新组装 batch、不停地有请求进出、batch 的内容每秒变化几百次。

公开的 benchmark 数字可以印证这个 regime。TPU v5e-8 跑 Gemma 7B 的 JetStream 部署典型 batch 32 到 64，MBU 大约 60%；TPU v5p-256 上 PaLM 540B 的 decode-heavy 工作负载在每副本 16 到 32 batch 下 MFU 约 30%；Ironwood pod 跑 Gemini Pro 在 64 到 128 batch 下 MBU 能上 70%；H100 八卡 NVLink 跑 Llama-3 70B 64 到 256 batch 下 MBU 60% 到 80%。值得注意的是 v5e 单卡只有 16 GB HBM，但靠 multi-chip TP 把权重和 KV cache 摊到 v5e-8 切片合计 128 GB，仍能跑中型模型的高 batch 服务——Google 那个"v5e 比 v5p 便宜 3.5×"的推理性价比数字就是建立在这种多卡小芯片摊开的部署模式上。

把这些数字串起来会发现一个反直觉但重要的事实：云端推理多 batch 缓解 memory-bound，并不是真的把工作负载推到 compute-bound 那一侧，而是把自己稳稳停在 memory-bound 那一侧的高效率区——也就是 MBU 接近峰值的位置。batch 不到 100 的时候是纯 memory-bound regime，目标只是让 HBM 带宽利用率从单 batch=1 时的不到 5% 拉到 60% 到 80%，TPOT 由 HBM 带宽直接决定；batch 大到 300 以上理论上可以越过 ridge 进入 compute-bound，但实际上几乎没人这么跑，因为 KV cache 容量和 TPOT SLO 早就把 batch 锁死了。Groq 是个例外，它的纯 SRAM 架构让有效带宽极高、ridge AI 极低，所以它的 decode 几乎自然就在 compute 平台上，但代价是模型规模灵活性。

总结成一句话：训练和推理虽然都靠多 batch 来提升芯片利用率，但训练通过 DP 把有效 batch 推到几百万 token、把算术强度直接停在 compute 平台上；推理通过 continuous batching 把单请求的 1 FLOP/byte 拉到几十甚至上百 FLOP/byte、MBU 从不到 5% 跃升到 60% 到 80%，但从未真正翻越 ridge point，只是停在 memory-bound regime 的高效率区。两边都用 batch，但停在 roofline 上的位置不同——训练硬件因此优化算力，推理硬件因此优化 HBM 带宽与容量。

1.2 计算核心：MX 单元里的"块缩放乘法"

"训练用 BF16/FP32"已经是 2023 年之前的旧范式。NVIDIA H100 引入原生 FP8（前向用 E4M3、反向用 E5M2）、DeepSeek-V3 在 671B 参数 / 14.8T token 全程 FP8 预训练中证明损失曲线可与 BF16 baseline 几乎重合——这三件事确立了 2024 年起前沿规模预训练进入 FP8 主导、FP4 试验的时代。到 2026 年，训练精度已固化为分层混合：前向、反向的 GEMM 走 FP8 乃至 FP4（占总 FLOPs 九成以上），attention accumulator、LayerNorm、优化器状态则保留 BF16/FP32 兜底。

FP8 之所以分两套格式，是因为训练前向和反向对数值分布的需求不一样。E4M3（4 位指数、3 位尾数）尾数多一位、精度更高但动态范围窄，适合前向里分布相对集中的权重与激活；E5M2（5 位指数、2 位尾数）多一位指数、量级跨度大但精度低，匹配反向里梯度的长尾分布。这套分工从 H100 沿用到 Ironwood，是事实上的标准。FP4 在此基础上把位宽再砍半，这对于计算，访存，通信上都有巨大的好处。

两颗芯片均搭载 2 个 TensorCore 与原生 FP4 数据通路，但 8t 的把 block-scale 乘法原生融入 MXU 流水线。FP4 训练能成立的前提是 microscaling 格式——单个 4 比特数仅 16 个可表示值，必须配合 block 共享 scale factor 才能把有效动态范围拉回 FP16 量级。举例子看看其他的精度，OCP 的 MXFP4 用 32 元素 block 加 E8M0 共享 scale（只能表示 2 的整数次幂），NVIDIA 的 NVFP4 走另一条路：block 缩到 16 个元素、scale 改用 E4M3（真正意义上的 FP8）、再叠一层 per-tensor 的 FP32 二级 scale，靠双层缩放换更细的局部动态范围。两者并非"类似路径"——格式选择本身就是 scale 处理代价与精度损失之间的取舍。

但现在谷歌还没有公布具体的FP4的具体数值格式，可以从从计算单元大致反推。NVIDIA 的 Tensor Core 本质是类似MAC tree的结构——一组并行 FP4 乘法器把乘积送进归约树、在短链路 dot product 末端完成累加与裁剪——天然契合 NVFP4 的 16 元素 block：dot product 的并行宽度与 block 边界对齐，scale 在归约树末端做一次乘法即可，硬件代价很轻。脉动阵列的 dataflow 完全不同：一个输出元素来自整条 tile 边长（典型 128 到 256）的长规约，mantissa 沿数据流逐级累加，scale factor 必须以与 mantissa 同步的节奏注入正确的 PE。直接照搬 NVFP4 的 16 元素 block，每条 systolic 行内要切换 8 到 16 次 scale，broadcast 网络与 PE 寄存器代价都会抬高；MXFP4 的 32 元素 block 缓解一部分但仍不对齐。合理的工程动作是把 block 长度对齐到 tile 维度（一条边一次 scale），或者走两层 block——外层与 tile 边长同步以消除频繁的 scale 切换，内层保留更细的 microscaling 服务于数值精度。两种方案都需要自定义格式。

关键问题在于 scale factor 的处理位置直接决定流水线效率。Ironwood 这一代是原生 FP8 加上软件模拟的 FP4，FP4 路径只能走回退流程：VPU 先把 FP4 反量化到 BF16、MXU 做 GEMM、VPU 再在 epilogue 重新量化回 FP4——VPU 在每个 GEMM 首尾各阻塞一次，systolic array 利用率被严重摊薄。8t 把"反量化—GEMM—再量化"三段流水线一次性折叠进 MXU，让 block scale 的乘法与 mantissa 乘加在同一组单元里同时发生。至于在 MXU 哪一级吸收 scale（每个 PE 内部融合、epilogue 内联、还是独立的广播通道），和上面的格式选择是同一问题的两面——格式定了 scale 出现的频率，dataflow 决定了 scale 注入的位置，必须协同设计。

Amin Vahdat 在 keynote 上对此的原话是 "redefined performance capability by moving block scale multiplication directly inside the MX units… delivering nearly three times the compute performance per pod"。这 3 倍需要拆开看：8t 单芯 FP4 12.6 PFLOPS 对 Ironwood 单芯 FP8 4.6 PFLOPS 本身就是 2.7 倍，其中位宽折半贡献约 2 倍、N3 到 N2 的工艺红利贡献约 1.3 到 1.4 倍——也就是说这 3 倍的大部分来自精度位宽加工艺节点，原生量化消除 VPU 阻塞是叠加在这两者之上的额外增量，让持续吞吐比标称峰值更接近，而不是 3 倍的全部来源。与之配套，8t 加宽了 VPU，让 MXU 解放出的 VPU 周期用于把量化、softmax、LayerNorm、随机舍入等向量算子与 GEMM 在时间上充分叠加，消除 transformer block 边界的气泡。这与 NVIDIA Hopper 引入 TMA 解放 SM 的逻辑是同一个道理。

8i 同样具备原生 FP4 通路，且峰值 FP4 算力（10.1 PFLOPS/芯）仅比 8t（12.6 PFLOPS/芯）低 20%——但这 20% 的差距不是计算单元被裁剪的结果，而是硅面积预算重新分配的结果。同代两颗芯的 HBM 配置已经把这层取舍讲明白：8t 是 216 GB HBM3e、6.5 TB/s 带宽，8i 反而上到 288 GB、8.6 TB/s 带宽——推理芯比训练芯带宽更高。Google 把本应用于扩展 MXU 阵列或加宽 VPU 的硅片面积，转投向 CAE chiplet 与 384 MB 的片上 SRAM 阵列，这也是因为训练比推理更需求算力，推理比训练更需要数据复用和。

1.3 存储层级：完全相反的 byte/FLOP 配比

两颗芯片在存储子系统上的取舍方向相反。原因是训练和推理的 bytes/FLOP 需求差一个量级：训练前向反向反复打满 MXU，瓶颈在 FLOPS；推理 decode 一个 token 只生成几 KB，却要把整个 KV-cache 反复读出来，瓶颈在访存。

8t 把 HBM 降速。带宽从 Ironwood 的 7.37 TB/s 降到 6.5 TB/s（−12%），容量从 192 GB 增到 216 GB（+12.5%），片上 SRAM 持平在 128 MB。Google 故意把 HBM3e 降速以压低成本：FP4 把每参数字节数砍半，MXU 内融 scale 又让更多 working set 留在 SRAM，HBM 标称带宽不必再追。NVIDIA Rubin 同期上到 22 TB/s HBM4，Google 选择留在 HBM3e 降频。训练芯按千卡万卡规模采购，单芯省一刀都会被规模放大。

8i 反向加码。HBM 容量上到 288 GB（+50%）、带宽上到 8.6 TB/s（+16%）、片上 SRAM 上到 384 MB（Ironwood 的 3 倍）。Google 的官方说法是 "With 3× more on-chip SRAM over the previous generation, TPU 8i can host a larger KV Cache entirely on silicon, significantly reducing the idle time of the cores during long-context decoding."384 MB 足够让 70B 级稠密模型的 per-token KV slice 或 MoE 单个激活专家的 active slice 完全驻留 die 内，消除 decode 每个 token 一次 HBM 往返。Bytes/FLOP 从 8t 的 0.52 升到 8i 的 0.85。HBM 带宽也比 8t 高，因为 SRAM 只装 KV-cache 热区，冷的部分仍要从 HBM 调度，agent 场景的上下文切换会反复打这条路径。

1.4 互连：相同物理带宽，截然不同的 topology

Google 正在进行功能解耦。SparseCore 这种处理稀疏数据搬运的单元在训练端（8t）依然不可或缺，不仅可以处理稀疏参数的梯度更新，也可以承担了一部分数据清洗和特征转换的任务。但在推理端（8i），SparseCore在TPU架构中占用明显的芯片面积。因此，8i使用Collectives Acceleration Engine替代sparse core，为了给 CAE和更大的 SRAM 腾出面积Die Area。

这本质上是把‘访存优化’的资源换成了‘通信优化’的资源。Cloud TPU 产品页将其称为 SC-CAE（SparseCore-Collectives Acceleration Engine）， 暗示 CAE 在微架构 DNA 上沿用了 SparseCore 的 dataflow 引擎，但功能彻底转向集合通信而非 embedding。这样的话，auto-regressive decode 的每步 reduction 不再阻塞 TensorCore 的 MXU，而是 commit 到 CAE 的 reduction tree FIFO 中流水线化。这一变化也针对MoE推理进行了优化。MoE的瓶颈不在于局部访存，而在于专家路由产生的大量全对全通信。（具体可以看硅谷101对那个TPU架构师的采访）

两颗芯片的 ICI scale-up 带宽都是 19.2 Tb/s 双向，是 Ironwood 的 2 倍。这一档基线决策训练和推理都需要：MoE 路由把 all-to-all 通信量翻倍，无论训练时同步专家梯度还是推理时跨芯片转发 token，scale-up 带宽不上就会成为瓶颈。

但同样的 19.2 Tb/s 在两颗芯上挂在不同拓扑下。8t 沿用 Ironwood 的 3D torus，每芯 6 条 ICI 链路连最近邻，superpod 规模做到 9,600 芯。3D torus 的 bisection bandwidth 随节点数缓慢增长，路径长度在大规模下也长，但训练并不在意单跳延迟——pre-training 是同步式 mini-batch 迭代，每步 all-reduce 把梯度规约一遍即可，长路径可以用流水线和异步 prefetch 覆盖掉。3D torus 在每芯链路数和总线总长之间取的平衡，对大规模同步训练是甜区。

8i 完全换了一套拓扑。Google 称之为 Boardfly，是一个分层胖树式结构（推测），最大网络直径据称比 Ironwood 砍掉一半以上。推理对延迟的敏感度和训练完全不同：针对moe结构，agent 场景下每生成一个 token 都可能触发 expert 路由、跨芯片 KV 调度、甚至 multi-agent 之间的 collective，每一跳都直接计入 user-facing latency。3D torus 上跨 pod 一头到另一头要十几跳，这种路径长度在训练里能被 batch 摊掉，在 decode 路径上则直接体现为响应时间。Boardfly 缩短最大网络直径就是直接换 tail latency。8i 还配套引入 CAE chiplet 把 all-reduce/all-gather 这些频繁集合通信操作从 TensorCore 卸载下来，即引入CAE。综合以上因素，根据 Google 披露，单次 collective 的片上延迟降到原来的五分之一——这一项同样只对推理高频通信场景有意义。

8t 之外还有两个数据通路特性，是训练专属的：

TPUDirect RDMA 让 HBM 与 NIC 直接互传，绕过 host CPU 与 DRAM。训练在 data parallel 维度上要持续做跨节点 gradient all-reduce，host 这一跳是固定开销，砍掉之后通信延迟和 CPU 占用同步下降。

TPUDirect Storage 让 HBM 直连 Managed Lustre 存储集群（10 TB/s 聚合带宽、80 PB 容量），数据从存储到 HBM 不再经 CPU 中转，比 Ironwood 时代的存储访问快约 10 倍。前沿模型的预训练数据集已经到百 PB 级别，多模态训练还要混入图像、视频、音频，data loader 跑不动会直接拖慢整个训练循环；这条路径就是为了让 MXU 在 ingest 阶段也不会被存储饿死。

这两条都不出现在 8i 上。推理负载里数据是模型权重加 KV-cache，权重一次性加载、KV-cache 在 die 内驻留，既不需要持续从存储 ingest 大数据集，也不需要跨节点同步梯度datapath简单很多。




1.5 其他

两颗芯片均报为 TSMC N2 级工艺，但这一标注未经官方确认，主要来自 The Next Platform 的推断。N2 的实际量产化时间窗口尚有争议——多家媒体的判断是 N2 真正进入大规模产能要到 2027 年下半年。最合理的读法是分两步走：2026 年下半年 GA 的 8t/8i 用的可能是 N3P 早期版本，2027 年的 refresh 才换到 N2，对应 Anthropic 3.5 GW 部署的目标节点。Meta MTIA 据称是首颗在 N2 量产的 AI 芯片，这条时间线暗示 TPU v8 在 N2 上爬坡的实际节奏可能比 Google 官方 "later this year" 的措辞所暗示的更晚。这种工艺-节奏拆分在前几代 TPU 上有先例：先用成熟节点的早期版本铺量、再用真正的下一代节点做容量翻番，对超大规模采购方而言是降低风险的常规做法。

封装层面两颗芯走出明显的 chiplet 化分工。8t 是双 compute die 加一颗 IO die、再加若干辅助 chiplet 的多 die 封装，compute 面积优先，所有功能尽量塞进 compute die 以保持训练负载的紧密耦合。8i 的关键差异是单 compute die 加一颗独立的 CAE chiplet die——把负责 collective 通信加速的 CAE 从 compute die 上独立出来做成单独 chiplet。这是一步同时收良率和复用收益的工程动作：CAE 作为延迟敏感模块需要单独优化时序，独立成 die 之后不再被 compute die 的数字逻辑约束；同时 CAE 这一模块在未来推理芯（v8e、第三家代工方设计的 inference TPU）上有复用空间，独立 chiplet 化也方便在不同工艺节点上分别迭代。

两颗芯片首次抛弃 x86 主机，全部改用 Google 自研的 Axion ARM CPU（基于 Neoverse 内核，V2 还是 N3 Google 未明确）。这一切换本身既有成本动机也有定制化动机：x86 主机的授权成本、功耗预算、与 Google 自研 IO 模块的耦合密度都不如自家 Arm 设计可控。

8i 在主机侧再走一步，把每台服务器的 CPU 主机数翻倍，并启用 NUMA 隔离——每颗 TPU 的数据预处理流量绑定到本地 NUMA 域，避免 Ironwood 时代 x86 主机上常见的跨插槽 DRAM 访问抖动。Google 对此的官方说法是 "eliminate the host bottleneck caused by data preparation latency"。这条优化对推理负载尤其重要：agentic 场景下的请求是高并发、低延迟、且每个请求的数据预处理（tokenization、prompt 拼装、采样参数解析）都是短小但路径敏感的工作，跨插槽 DRAM 抖动会直接落到 user-facing latency 上。训练侧的 8t 不需要这一层——前向反向迭代是 batch 级长任务，host 这一跳的抖动可以被 batch 摊掉，单台主机服务多颗 TPU 也不会成为瓶颈。

至此 8t/8i 的分工已经从计算单元（FP4 native + scale 内融）、存储子系统（HBM 降速 vs SRAM 三倍）、互联拓扑（3D torus vs Boardfly）一路贯穿到了封装与主机侧（多 die 紧耦合 vs CAE chiplet 独立 + 双主机 NUMA 隔离）。同代两颗芯共用大量 IP 模块，但每一层硅片到系统的取舍方向都按训练-推理的负载差异分别展开，这正是把同一代设计预算切成两套 SKU 的工程意义所在。
