0. 什么是并行？为什么要并行？

并行即同时运行，从算法的层面来看，就是空间换时间的经典体现。对于LLM推理来说，仍然存在许多的空间 (算力) 冗余，因此一个很自然的想法就是利用并行来实现加速。最理想的情况就是，在任一时刻，我们所有的计算资源都在执行计算任务，即完全榨干了所有的算力；同时所有的读写/通信操作都被overlap，此时整个推理过程就达到了不可加速的状态。

2. Data Parallel (DP)、Distributed Data Parallel (DDP)、Fully Shared Data Parallel (FSDP)
NOTE: 这三种并行技术主要应用于训练阶段，核心思想是数据层面的并行和切分

首先简单介绍Data Parallel，它的核心思想是，将模型复制到多个 GPU 上，然后将一个批次的数据切分并发送到每个 GPU 进行计算。计算完成后，将结果和梯度汇总到主 GPU 上进行处理。如果只考虑推理阶段的应用，DP等价于复制多份模型，并行处理数据。

如果从训练的角度看，DP的主要瓶颈在于计算完梯度之后的通信以及主GPU的负载不平衡（同时要负责计算和梯度汇总与回传）。

为了解决这两个问题，我们可以从工程上进行优化得到Distributed Data Parallel (DDP)，核心思想是 All Reduce技术：在反向传播计算梯度的同时，DDP 会在后台启动一个名为 AllReduce 的通信操作。所有进程（GPU）会同时相互通信，将各自的梯度进行求和与平均。这个过程是去中心化的，没有主次之分。Pytorch官方给的图如下，可能有一些不太好理解，但只需要知道DDP是对DP在通信上的优化即可。

图源 https://docs.pytorch.org/docs/stable/notes/ddp.html

以上两种data parallel已经实现了在数据维度上的并行，但它们有着一个共同的要求：需要将模型复制到多个GPU上，每个 GPU 仍需加载完整的模型参数、梯度和优化器状态，这使得单个 GPU 的显存成为训练巨大模型的瓶颈。而我们之前的分析已经看到，矩阵的切片是不影响最终运算结果的，因此一个改进的方法，即FSDP，就是把模型参数、梯度和优化器状态全部进行分片，每个 GPU 只负责存储和更新其中的一小部分。对流程进行简单描述：

初始化：在训练开始前，将完整的模型参数、梯度、优化器状态沿着参数维度进行切分。每个 GPU（进程）只拥有完整状态的一个分片
前向传播：当需要计算某一层时，所有 GPU 会通过 All-Gather 操作，临时从其他 GPU 那里收集计算该层所需要的所有参数分片，从而在自己的 GPU 上动态地重构出完整的层。完成该层的前向计算后，立即丢弃刚刚收集来的非本地参数分片，释放显存。
反向传播：过程和前向传播类似
参数更新：每个 GPU 使用本地存储的梯度分片和优化器状态分片，来更新它所负责的那一小部分模型参数

3. Model Parallel (MP)、Pipeline Parallel (PP)

当模型参数很大，单张GPU无法装载下模型的参数，应该怎么办？

一个很自然的想法就是把模型分发到多个GPU上，然后顺序执行. 如果是训练阶段，还需要顺序执行梯度回传，这会显著降低GPU的利用率，导致训练速度非常缓慢。

图源 https://medium.com/byte-sized-ai/pipeline-parallelism-explained-in-2-mins-6bdf1ab29053

那么我们从system的角度来考虑，多个device的运行，是否有并行的空间呢？本科学操作系统时就有pipeline的概念，我们把LLM的运行过程看成一个pipeline，则这个pipeline上存在非常多的bubble。通过合理的overlap 这些bubble，我们就可以得到pipeline parallel。思想上很简单，通过把输入的batch进行切分为不同的mini-batch，得到多条pipeline；再把多条pipeline进行overlap以减少bubble，从而提高GPU的利用率。

图源 https://arxiv.org/pdf/1811.06965

感谢评论区指正：在推理过程中，不需要传播梯度，因此上图中的右半部分是没有的；因此PP对于推理的帮助比较有限。
