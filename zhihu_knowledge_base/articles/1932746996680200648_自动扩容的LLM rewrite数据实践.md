用 LLM 处理数据+filter 已经是日常了，总结下来有这些需求：

高吞吐。计算并行 or 数据并行，进程要启动很多个——占满 GPU 的一切——尽可能快地处理完数据
简单。
好调试，毕竟 process 逻辑总要改
逻辑无关的代码要少，才好维护
业务可靠。
LLM generation 不总是符合格式要求；
openai API 有个超时时间，client 给的压力太大的话，请求会断连
高效。处理完就尽快退出来，别影响其他人用卡

因为要调试 prompt，肯定不能在 client 直接加载 LLM、试 prompt、再加载，

要分成 CS 模式（Client-Server 通信）。

0x01 早期糙法：手动

服务端，开 tmux 手动 or sleep 启动 8 个vllm-instance 。因为 vllm 当前版本有缓存冲突，不间隔地同时开多个会崩溃。

客户端。

如果代码写多进程，就要切分原始数据、要 reduce 处理好的数据；多进程也不方便调试 process 逻辑（额外安装 remote-pdb）

那便只能是 async，可 client 流量控制怎么办呢，总不能几千个请求一起发？

写个修饰器 limit_async_func_call ，控制最大 running object 个数
def limit_async_func_call(max_size: int, waitting_time: float = 0.1):
..
        @wraps(func)
        async def wait_func(*args, **kwargs):
            nonlocal __current_size
            while __current_size >= max_size:
                await asyncio.sleep(waitting_time)
            __current_size += 1
            result = await func(*args, **kwargs)
            __current_size -= 1
            return result
..

@limit_async_func_call(32)

负载均衡方面，从所有的 base_url list，随机挑一个发请求
api_base_list = [f"http://localhost:{port}/v1" for port in range(8000, 8008)]

勉强满足了需求，还是不满意：

client 的代码还是太多了。在不同服务器上，就为了 LLM rewrite 点数据，要拷贝装饰器代码，get_event_loop代码；处理 async 代码要动脑子
vllm serve 启动退出都太狼狈
8 个进程，要么脚本 sleep 等半天，再检查是否都起来了；要么命令行敲半天
深夜代码跑完了，server 无法及时退出，占着显存、GPU 利用率为 0
0x02 目前糙法：基于 ray

ray 对 server 侧已经做了很好的支持，像这样就能解决所有问题：

min_replicas 为 0，刚启动不占显存；会随着服务压力扩展/收缩实例。
内部把实例都代理成 8000 端口，不需要 base_url_list
内部处理 vllm 崩溃，自动重启
llm_config = LLMConfig(
    model_loading_config={
        "model_id": 'ray',
        "model_source": '/data/share/Qwen'
    },
    deployment_config={
        "autoscaling_config": {
            "min_replicas": 0,
            "max_replicas": 8,
        }
    },
    engine_kwargs={
        "max_model_len": 32768,  # Or increase KV cache size.
        "tensor_parallel_size": 1,
    },
)

# Deploy.
app = build_openai_app({"llm_configs": [llm_config]})
serve.run(app, blocking=True)

client 侧，还是得基于 ray 自己写，用 ray.data.map 比较方便控制并行度：

    ds = ray.data.read_json(input_paths)

    # Apply the transformation to our dataset
    transformed_ds = ds.map(
        do_transform,
        concurrency=256,  # 并行度
    )
    transformed_ds.take_all()
其中 do_transform 是自己的处理逻辑，不用管 sync/non-sync 的事
当 concurrency=1时，可以很方便地 pdb 调试（不为 1 时日志就乱一些）
