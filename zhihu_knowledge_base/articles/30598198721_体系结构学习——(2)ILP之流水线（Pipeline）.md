计算机的性能指标是程序所花的时间，它可以分为三个部分：程序的指令数，每条指令的周期，每个周期所花时间。第一项和源代码，编译器技术以及指令集（ISA）相关，复杂或者专用的ISA可以使用更少的指令完成同样的程序；第二项CPI，则和ISA以及处理器的微架构相关；第三项是周期时间，它和微架构以及所采用的工艺相关。因此，想提升计算的性能，可以是经验丰富的程序员使用巧妙快速的算法实现，也可以是软件编译器合理的将高级语言转化为很少的指令，或者是处理器采用的指令集，实现的微架构和采用的制造工艺具有优势。

1、流水线概述

对于接触过体系结构相关或者处理器相关的工作者来说，流水线是基本的也是必须了解的技术，至少需要知道它是如何提升我们的处理器性能的。首先需要明确的一点是，单周期流水线的IPC接近但不可能大于1，它提升性能的方法是增加了处理器的工作频率（铁律第三项）。流水线通过划分指令执行阶段（如取指、译码、执行、访存、写回）提高了吞吐量（单位时间内完成的指令数），但 IPC 关注的是每个时钟周期完成的指令数。即使流水线深度增加，在理想情况下，单发射流水线每个周期提交一条指令。

1.1 流水线简单介绍

流水线是指在一条长路径的组合逻辑中插入寄存器，使得每一级寄存器间的组合逻辑可以同时处理不同的数据，这对于多数据流时可以使得组合逻辑的利用率增大。如同下图的小老鼠图，如果小老鼠只有一只，那流水线其实并没有帮助，但是小老鼠有一群时，就提高了这群小老鼠的通过效率。

流水线就像小老鼠通过管道，上方是一只一只通过，下方是紧密通过。当小老鼠紧密的通过时，就相当于使用了流水线。每一只小老鼠可以想象成每一级流水级中的数据。

所以，流水线的有效利用于多数据流的地方，比如总线的数据交互，处理器的指令处理，多数据的计算等。

1.2 流水线带来的性能提升

流水线的引入可以带来频率的提升，从而使得吞吐量提升，那么是否流水线级数越多，就能更多的提升性能呢？这里需要明确的是，流水线并没有缩短单条指令的执行时间，而在插入寄存器之后，寄存器反而会带来更多的延时。带来的延时主要体现在两个方面：寄存器延时和时钟偏差。数据经过流水线寄存器有传播延迟，此外还需要满足寄存器的建立时间，也就是发出触发操作的时钟信号之前，寄存器输入必须保持稳定的时间。而时钟偏差是时钟到任意两个寄存器时刻之间的最大延迟，它也决定了时钟周期的下限。我们可以使用一个简单的建模公式讨论流水线插入后的性能影响。假设原本的数据通路延迟为D，如果我们插入n级流水线，每一级流水线带来的延迟开销为s，那么在流水线插入之后的时钟周期为(D+ns)/n，其时钟频率就是1/(s+D/n)。如果我们的n极大，那么时钟频率最大也就是趋近于1/s，性能并不会根据流水级的插入而无限提升。此外，当寄存器插入过多导致两级寄存器之间的延迟过小时，后端实现可能需要考虑保持时间带来的问题。

当我们仅考虑流水级对频率的影响时，我们发现如果拍脑袋的增大流水级，只是性能趋近一个上限，似乎没有带来什么坏处。其实不然，插入寄存器意味着带来了面积的开销，也意味着带来了更多的功耗。因此无脑盲目增加流水级数对于需要考虑PPA权衡的设计师来说是不行的。如果把面积考虑进去，我们就需要在性能和面积之间找一个流水线级数的权衡，以更好的应对我们的需求（成本优先或者性能优先）。

1.3 处理器流水线与依赖

流水线并非处理器的专项，但是在处理器中实现流水线面临着比一般计算单元实现流水线更多的问题，其主要原因在于处理器的指令之间往往存在着依赖关系。依赖的说法很多，比如数据依赖（data dependence），名称依赖（name dependence）以及控制依赖（control dependence）。又或者说结构冒险（structural hazard），数据冒险（data hazard）以及控制冒险（control hazard）。如果要细说的话，依赖理解为一种属性，冒险理解为一种实现时的冲突冒险（个人理解），但是实际表达的含义并没有太大区别。本文将数据依赖总结如下三类：

结构冒险：当多条指令需要相同的资源时，就会发生结构冒险，比如两条指令想写同一个Mem。这种冒险是资源不够带来的，因此只要增加资源就可以解决，比如一个Mem实现两个写口。

数据冒险：根据指令中读访问和写访问的顺序，可以将数据冒险分为3类：

RAW（read after write，写后读）—— 前面的指令将结果写入某源位置，而后面的指令需要使用该源位置的值。按照指令的实际顺序，后指令应该使用前指令计算出的新值，但是在流水线中，后指令从源位置读取数据的时候，前指令计算的值可能还没写回到寄存器，使得后面的指令拿到了错误的旧值。这一冒险和真数据依赖对应，可以使用Forwarding/Bypassing解决，即不等写回就直接拉线将计算结果拿到ALU输入进行计算。下图中，DSUB指令，AND指令，OR指令都使用了bypass拿到DADD还未写回的计算结果。
WAW（write after write，写后写）—— 两条指令写同一位置时，按照指令顺序应该先写前面的，再写后面的指令，但是由于有的流水线允许多个流水级进行写操作，或者允许前一指令stall时后一指令继续执行，就有可能发生这种WAW冒险。该冒险与输出依赖对应。
WAR（write after read，读后写）—— 这种冒险与RAW相反，后续指令在前面指令读取目标位置之前写入该位置，这样前面指令错误的取到新值。这一冒险对应于反依赖（或名称依赖）。实际上，经典的单发射五级流水线不会发生这种冒险，因为读寄存器总是在写寄存器之前。但是，在乱序超标量处理器中就需要警惕这种情况。

控制冒险：分支指令导致后续指令的执行路径不确定。在分支指令时，由于分支跳转的方向需要通过计算才能确定。此时有两种最基本的解决方法，一是暂停向流水线塞入指令，得到跳转方向确认后再进行后续的正确指令；二是直接塞入下一条指令，等跳转方向确认后，如果不跳转则塞入的指令正确，跳转则清空已经塞入的指令，重新取正确的指令执行。前者会导致硬件总是处于空闲状态，后者则会导致错误时存在惩罚代价。现在的高性能处理器一般都会采用分支预测的方法，并且分支预测已经能达到百分之九十几的准确率。将分支判断尽量提前，也可以使得惩罚代价变小。

依赖的解决有两种大的思考方向，一种是编译器生成指令编码时解决，另一种则是在硬件实现上解决。在将代码编译成指令时，编译器可以尽量的避免存在结构冒险和数据冒险的代码，比如将指令的存放数据分配到不同的寄存器。而对于控制冒险（循环跳转代码），编译器也可以使用调度和循环展开的方式尽量去解除这种依赖关系，当然代价就是需要有足够的存放资源。调度则是通过调整指令的顺序，在两条依赖指令间插入不相关指令，当两条依赖指令间存在足够的流水级间隔时，就不存在不同流水级的错误了。

2 异常与中断

处理器在执行的过程中，除了分支类型的指令之外，其他很多情况也可以打断程序的执行，这些情况统称为异常（exception）。不同指令集中异常包含的内容会有不同，可以概况为以下几种：

处理器外部时间发生的异常，这种异常也称为中断（interrupt）。由于发生在处理器外部，中断本质上和处理器执行的指令没有必然的联系，处理器在执行的任意阶段都可能收到中断，因此也称为异步的异常。
虚拟地址（virtual address）到物理地址（physical address）的转换引起的异常。
指令自身引起的错误，比如未定义的指令，用户态下的非法指令，整数运算时的溢出等。
指令自身产生的异常，比如MIPS指令集中的SYSCALL和Trap指令，前者是无条件产生异常，后者是有条件产生异常。普通的程序可以通过这种类型的指令来调用操作系统的某些任务，一个处理器要支持操作系统，这种类型的指令必须要有。
异常的处理过程：

对于一个特定的处理器来说，所有类型异常的处理是一样的，从处理器外部来看，产生异常之前的所有指令都已经完成，而产生异常的指令及其之后的指令不允许完成。处理器跳转到对应的异常处理程序（exception handler）的入口地址，开始执行这个异常程序，当其执行完成后会返回到刚才发生中断的位置，重新将这条指令取到流水线中，就像这个异常没有发生过一样，这种方式也称为精确的异常（precise exception）。

在处理异常后，大部分情况下是将发生异常的的pc值重新取出执行，但是在某些情况下不能这样做。比如发生异常的是SYSCALL和Trap指令，加入返回PC仍然将其取出执行，程序就会发生死循环，这种情况应该取其他PC值进行计算。

为了能在异常返回时恢复异常发生时的状态，还需要将发生异常的指令PC值（或者下一PC值，根据异常类型而定）保存到专用的位置中。MIPS使用EPC寄存器保存异常发生时的PC值；而对于CISC通常使用堆栈保存这个PC值。

异常发生时还需要考虑通用寄存器的处理，因为在异常处理程序中可能会更改通用寄存器的内容。所以在异常处理器开始时需要对涉及到的寄存器进行保存。不管是RISC还是CISC都将通用寄存器的内容保存在堆栈中。

流水线中的异常：

在流水线处理器中，由于各个阶段同时进行处理，因此各个阶段可能会同时发生异常。以经典五级流水线为例，各个流水级可能出现的异常如下如下：

取值（Fetch）—— 取指令发生I-TLB缺失或者Page Fault，或者取指令的地址不存在，又或者受保护的区域取指令；
译码（Decode）—— 遇到未定义的指令；
执行（Execute）—— 算数运算发生溢出或者除0运算；
访存（Memory）—— 访问存储器发生D-TLB缺失或者Page Fault，或者访问地址不存在，或者非对齐访问。

由于流水线的存在，各个阶段可能同时发生异常，但是对于异常的处理需要遵循程序的原始顺序，因此处理器会在提交点（commit point）位置才对异常进行处理。比如在译码阶段指令i发生了异常，而在下一周期时指令i到达执行阶段，但是指令i-1在访存阶段发生了异常。虽然指令i的异常更早发生，但是处理器会先处理指令i-1发生的异常。在处理指令i-1的异常时，指令i-1后续的指令会被抹掉。

异常和中断的概念看似简单，无非就是发生异常时让异常前的指令执行完，保存现场，查看是何种异常然后处理相应程序，然后再恢复现场，但实际上不太容易理解如何操作，因此我让deep seek帮助写了一个简单的处理器处理外部中断的例子。中断向量表也可以是一个软件可配的存储空间。

// 模块划分：
// 1. 处理器核心 (CPU_CORE)
// 2. 中断控制器 (INTR_CONTROLLER)
// 3. 中断向量表 (INTR_VECTOR_TABLE)
// 4. 中断处理程序 (INTR_HANDLER)

//---------------------------
// 模块1：中断控制器
//---------------------------
module intr_controller(
    input clk,
    input rst,
    input [3:0] ext_intr,     // 外部中断输入
    output reg [3:0] intr_id, // 当前中断ID
    output intr_req,          // 中断请求信号
    input intr_ack            // 中断应答信号
);
    reg [3:0] pending_intr;
    
    // 中断优先级仲裁
    always @(posedge clk or posedge rst) begin
        if(rst) begin
            pending_intr <= 4'b0;
            intr_id <= 4'h0;
        end else begin
            // 锁存中断请求，pending_intr表示当前需要存在的中断
            pending_intr <= pending_intr | ext_intr; 
            
            // 简单固定优先级仲裁
            if(intr_ack) begin
                // 前导零检测
                casex(pending_intr)
                    4'b1xxx: intr_id <= 4'h3;
                    4'b01xx: intr_id <= 4'h2;
                    4'b001x: intr_id <= 4'h1;
                    4'b0001: intr_id <= 4'h0;
                    default: intr_id <= 4'h0;
                endcase
                // 清除处理的中断
                pending_intr <= pending_intr & ~(1 << intr_id);
            end
        end
    end

    assign intr_req = |pending_intr;
endmodule

//---------------------------
// 模块2：处理器核心
//---------------------------
module cpu_core(
    input clk,
    input rst,
    input intr_req,
    output intr_ack,
    input [3:0] intr_id,
    input [31:0] vector_addr,
    output [31:0] pc
);
    // 处理器状态定义
    typedef enum {
        EXECUTE,
        SAVE_CONTEXT,
        FETCH_VECTOR,
        HANDLE_INTR,
        RESTORE_CONTEXT
    } cpu_state_t;
    
    reg [31:0] pc_reg;
    reg [31:0] status_reg;
    reg [31:0] stack[0:7];
    reg [2:0] sp;  // 堆栈指针
    cpu_state_t state;

    assign pc = pc_reg;
    assign intr_ack = (state == SAVE_CONTEXT);

    always @(posedge clk or posedge rst) begin
        if(rst) begin
            pc_reg <= 32'h0;
            status_reg <= 32'h0;
            sp <= 3'b0;
            state <= EXECUTE;
        end else begin
            case(state)
                EXECUTE: begin
                    // 正常执行指令
                    pc_reg <= pc_reg + 4;
                    if(intr_req) begin
                        state <= SAVE_CONTEXT;
                    end
                end
                
                SAVE_CONTEXT: begin
                    // 保存现场：PC和状态寄存器，都存放在栈
                    stack[sp] <= pc_reg;
                    stack[sp+1] <= status_reg;
                    sp <= sp + 2;
                    state <= FETCH_VECTOR;
                end
                
                FETCH_VECTOR: begin
                    // 从中断向量表获取处理地址
                    pc_reg <= vector_addr;
                    state <= HANDLE_INTR;
                end
                
                HANDLE_INTR: begin
                    // 执行中断处理程序
                    if(/* 中断处理完成信号 */) begin
                        state <= RESTORE_CONTEXT;
                    end
                end
                
                RESTORE_CONTEXT: begin
                    // 恢复现场
                    sp <= sp - 2;
                    pc_reg <= stack[sp-2];
                    status_reg <= stack[sp-1];
                    state <= EXECUTE;
                end
            endcase
        end
    end
endmodule

//---------------------------
// 模块3：中断向量表
//---------------------------
module intr_vector_table(
    input [3:0] intr_id,
    output reg [31:0] vector_addr
);
    always @(*) begin
        case(intr_id)
            4'h0: vector_addr = 32'h0000_1000; // 中断0处理地址
            4'h1: vector_addr = 32'h0000_2000; // 中断1处理地址
            4'h2: vector_addr = 32'h0000_3000; // 中断2处理地址
            4'h3: vector_addr = 32'h0000_4000; // 中断3处理地址
            default: vector_addr = 32'h0000_0000;
        endcase
    end
endmodule

//---------------------------
// 顶层模块
//---------------------------
module soc_top(
    input clk,
    input rst,
    input [3:0] ext_intr
);
    wire intr_req, intr_ack;
    wire [3:0] intr_id;
    wire [31:0] vector_addr;
    wire [31:0] pc;
    
    intr_controller u_intr_ctrl(
        .clk(clk),
        .rst(rst),
        .ext_intr(ext_intr),
        .intr_id(intr_id),
        .intr_req(intr_req),
        .intr_ack(intr_ack)
    );
    
    cpu_core u_cpu(
        .clk(clk),
        .rst(rst),
        .intr_req(intr_req),
        .intr_ack(intr_ack),
        .intr_id(intr_id),
        .vector_addr(vector_addr),
        .pc(pc)
    );
    
    intr_vector_table u_vector_table(
        .intr_id(intr_id),
        .vector_addr(vector_addr)
    );
endmodule

3 整数浮点解耦流水线

在教材中的经典五级流水线针对的是整数的ALU，只需要一级执行就可以完成。但是当增加浮点数的处理时，一个时钟周期不足以完成计算。这样做只能使用更低的时钟频率或者更大的浮点计算单元去满足，或者两者都需要。但在实际情况中，浮点流水线可以允许更长的操作延迟。

上图显示了如何综合整数计算和浮点计算，它包含了主要整数计算单元（一级流水），乘法计算单元（7级流水），浮点数加法（4级流水）以及除法（没有完全实现流水化，需要24个周期完成）。在ID阶段后，允许指令分别发往不同的流水，假如在没有数据依赖的情况下，允许他们在不同的周期写回寄存器。

比如上面的四条指令，浮点乘法，浮点加法，整数加法和存数，虽然乘法是最先发射的指令，但是他是最后写回的，而整数加法则是最先写回的。由于它们的写回是在不同的时间，所以也不会发生冲突。不过，在存在依赖的指令发生时，就需要注意以下问题：

由于除法单元未完全流水化，所以可能发生结构冒险。需要对这些冒险进行检测，还需要停顿指令的发射；
因为指令的允许时间不同，所以一个周期写回寄存器次数可能大于1；
由于写回顺序不同，可能存在WAW冒险；
指令的完成顺序可能不同于其发射顺序，从而导致复杂的异常问题；
由于流水线的加深，因RAW冒险引起的停顿将更加频繁；

第1，2，3，5从原理上是数据冲突带来的，解决方案采用更复杂的检测，旁路和停顿等。这里主要解释一下4带来的问题，假如执行下面三条指令：

fdiv    f0, f2, f4
fadd.d  f10,f10,f8
fsub.d  f12,f12,f14


这三条指令没有相关性，但是由于除法需要的周期很多，所以会出现后面的加法和减法先完成的情况。假如加法完成除法还没完成时，减法发生了浮点算术异常，就会发生我们应该尽力避免的不精确异常。由于加法修改了寄存器的数，所以想恢复到除法之前的状态变得困难。

在CS152课程中，老师还讲解了一个解耦例子，做个记录。考虑如下的浮点数操作代码：

for (i=0; i<N; i++) {
    A[i] = B[i] + C[i];
 }

其中A，B和C是浮点数数组，而i显然是整数，将其编译成指令如下

loop: fld   f0, 0(x2)  // x2 points to B
      fld   f1, 0(x3)  // x3 points to C
      addi  x3, x3, 8
      addi  x2, x2, 8
      fadd  f2, f0, f1 
      addi  x1, x1, 8
      fsd   f2, -8(x1) // x1 points to A
      bne   x1, x4, loop 


其中x2，x3分别作为B和C的索引，x1作为A的索引。实际上，在将循环代码转化为汇编的时候采用了调度的技巧。因为浮点数的计算周期更长，使用汇编将x2,x3的计算放在了fld指令的后面，同时，将x1的更新放在了fadd和fsd之间，使用fsd存放数据时需要减8。这样的指令排序可以使得流水线中的指令更加紧凑。当然还可以考虑另一种解耦方式（decouple），将浮点数指令放在一起，整数指令放在一起，并且将浮点数和整数分为两条流水，整数指令计算循环的地址，我们可以称其为access，而浮点数指令进行计算，称为execute，如下：

loop: fld   f0, 0(x2)  // x2 points to B
      fld   f1, 0(x3)  // x3 points to C
      fadd  f2, f0, f1 
      fsd   f2, 0(x1) // x1 points to A
      addi  x3, x3, 8
      addi  x2, x2, 8      
      addi  x1, x1, 8      
      bne   x1, x4, loop 


在流水中将整数处理和浮点处理分开，整数流水线负责将指令操作分解到操作队列里，并将浮点计算的地址预先放在存在的地址队列中，将浮点操作源数据放在数据队列中。浮点操作流水级对数据进行计算后，将数据放入浮点数数据队列，根据对应的地址进行写入寄存器。这样浮点和操作和整数的操作就可以分开处理。
