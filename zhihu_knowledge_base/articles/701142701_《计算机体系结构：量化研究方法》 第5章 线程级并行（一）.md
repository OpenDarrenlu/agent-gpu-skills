5.1 介绍/Introduction

线程级并行（TLP）意味着存在多个PC（program counter），主要通过MIMD（多指令多数据）架构来开发利用。

本章节中，多处理器（multiprocessors）定义为由紧密耦合的处理器组成的计算机，其中处理器的协调和使用通常由单个操作系统控制，并通过共享地址空间共享内存。

这种系统通过2个不同的软件模型来开发线程级并行（TLP）。

并行处理：执行一个紧密耦合的线程组，协同完成一个任务。
请求级（request-level）并行：执行多个相对独立的进程，这些进程可能来自一个或多个用户。

请求级并行可能应用在多个处理器中的运行的单个程序，例如数据库响应查询，或者多个独立运行的应用程序，通常称为多程序（multiprogramming）。

多处理器架构：问题和方法

为了充分利用一个有n个处理器的MIMD多处理器，通常需要至少n个线程或进程来执行；目前大多数多核芯片采用多线程，线程数量要高个2-4倍。单个进程内的独立线程通常由程序员指定或由操作系统创建（来自多个独立请求）。在另一种极端情况下，一个线程可能包含几十次迭代组成的循环，通过并行编译器利用循环中的数据并行性进行生成。

分配到一个线程的计算量，称为粒度（grain size）。粒度大小在考虑如何高效利用线程级并行性时很重要，但与指令级并行的重要定性区别在，线程级并行是由软件系统或程序员在高层次上确定的，并且线程包含可能并行执行的数百到数百万条指令。

线程也可以用于利用数据级并行，尽管开销通常高于SIMD处理器或GPU。这种开销意味着粒度必须足够大，以有效地利用并行。例如，虽然向量处理器或GPU能够高效地在短向量上进行并行操作，但当并行度分配到许多线程时，得到的粒度可能非常小，导致开销使得在MIMD中并行度利用的代价变得很高。

现有的共享内存的多处理器可以分为两类，取决于处理器数量，进而影响内存组成和互连策略。

（1）对称（共享内存）多处理器（SMP）或集中式共享内存多处理器。这类处理器的核心数量较少，通常为32个或更少；对于这种处理器数量较少的多处理器，所有处理器都可以平等访问一个共享的集中式内存，因此称为对称；在多核芯片中，内存通常在核心之间以集中式的方式共享；大多数现有的多核芯片都是SMP。

SMP架构有时也被称为统一内存访问（uniform memory access，UMA）多处理器，因为即使内存被分为多个bank，但是所有处理器从内存中访问数据的延迟是均匀的，如图5.1所示。

图5.1 基于多核芯片的集中式共享内存多处理器的基本结构。多个处理器-缓存子系统共享同一块物理内存，通常在多核上有一级共享缓存，每个核心有一级或多级私有缓存。




（2）分布式共享内存（DSM）多处理器。采用分布式物理内存，如图5.2所示。为了支持更多的处理器数量，内存必须分布在处理器之间；否则，内存系统将无法满足大量处理器的带宽需求，从而导致访问延迟过长。在节点之间分配内存增加带宽，并且降低本地内存的延迟。

DSM多处理器也被称为非统一内存访问（nonuniform memory access，NUMA），因为访问时间取决于内存中数据的位置。

图5.2 2017年分布式内存多处理器的基本架构，通常包含带有内存、可能带I/O和互连网络接口的多核多处理器芯片。

总的来说

处理器数量少的时候，SMP方式的更合适；处理器数量多的时候，DSM的方式更加合适。不过随着处理器性能的迅速提升和对内存带宽需求的增加，分布式内存倾向的多处理器的规模在变小。
DSM的主要缺点是在处理器之间通信数据变得稍微更加复杂，需要在软件方面付出更多来有效利用分布式内存提供的内存带宽增益。
无论是SMP还是DSM架构，线程之间的通信都是通过共享的地址空间进行，这意味着如果有正确的访问权限，任何处理器都可以对任何内存地址进行访问。
并行处理的挑战

多处理器的应用范围很广，从基本没有通信的独立任务涵盖到需要线程通信才能完成任务的并行程序。要克服这些障碍的全面的方法，涉及算法选择和算法实现方式、底层编程语言和系统、操作系统和支持的功能，以及体系结构和硬件实现。在许多情况下，其中一个是关键瓶颈，但当扩展到更大规模的处理器数量（接近100个或更多）时，通常需要关注软件和硬件的所有方面。

障碍1：与程序中受限的并行度有关。可用并行度的受限导致在任何并行处理器中实现加速都会变得困难，可以通过Amdahl定律公式进行评估。

障碍2：与通信成本相对较高有关。并行处理器中远程访问的延迟较大，不同核心之间的数据通信可能需要35-50个时钟周期，而在不同芯片上的核心之间的通信可能需要100个时钟周期甚至更多。

假设在有32个处理器的多处理器上运行一个应用程序，远程内存访问需要100 ns的延迟。假设所有本地内存访问都是命中的，处理器在远程请求的期间处于停滞，工作时钟速率为4 GHz。 如果基本CPI（假设本地缓存全命中）为0.5，远程内存访问占0.2%。

在实际中，性能分析要复杂得多，因为非通信访问的一部分会在本地层次结构中未命中，并且远程访问时间没有一个固定的值，具体取决于通信机制、互连网络类型和多处理器的规模等。

并行度受限和远程通信延迟高是在使用多处理器时面临的两个最大的性能挑战。 （1）应用程序并行度限制的问题主要需要通过软件来解决，采用新的算法提供更好的并行性能，通过软件系统最大化使用所有处理器的执行时间。 （2）减少远程延迟的影响可以通过架构和程序员两个方面来解决。例如，可以通过硬件机制（如缓存共享数据）或软件机制（如重组数据，产生更多本地访问）来减少远程访问的频率。
