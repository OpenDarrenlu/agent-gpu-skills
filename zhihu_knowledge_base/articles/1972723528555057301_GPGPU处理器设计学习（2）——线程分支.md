前言

本节核心问题：什么是线程分支，以及在GPGPU中线程分支带来的问题

GPGPU采用SIMT执行模型，在逻辑上以线程为单位调度，而物理上以warp为单位在SIMD单元上执行。在无分支情况时，每个warp中的线程在SIMT Core内锁步执行相同的指令流，而warp中的多个Thread对应到SIMT Core中的多个ALU Line同步执行。但是当如if……else……的分支语句存在时，不同的线程需要走不同的分支路径，但由于SIMT Core只能执行Warp的一个指令流，所以GPGPU会执行掩码机制：（1）先执行一个分支路径，屏蔽不走该路径的线程；（2）再执行另一个分支路径，屏蔽已执行过的线程；（3）最终合并执行后续指令。

下面以一个经典的嵌套分支说明这一过程，它的CUDA代码和编译的PTX如下所示。这里假设其所调用的线程为4个。

do{
    t1 = tid * N;            // A
    t2 = t1 + i;
    t3 = data1[t2]; 
    t4 = 0;
    if( t3 != t4 ) {
        t5 = data2[t2];     // B
        if( t5 != t4 ){
            x += 1;         // C
        } else {
            y += 2;         // D
        }
    } else {
        z += 3;            // F
    }
    i++;                   // G
}while(i<N);


PTX:

A:  mul.lo.u32         t1, tid, N;
    add.u32            t2, t1, i;
    ld.glocal.u32      t3, [t2];
    mov.u32            t4, 0;
    setp.eq.u32        p1,t3,t4;
@p1 bra                F;
B:  ld.global.u32      t5, [t2];
    setp.eq.u32        p2, t5, t4;  
@p2 bra                D; 
C:  add.u32            x, x, 1; 
    bra                E; 
D:  add.u32            y, y, 2; 
E:  bra                G; 
F:  add.u32            z, z, 3; 
G:  add.u32            i, i, 1; 
    setp.le.u32        p3, i, N;
@p3 bra                A; 

显然，对于编译器，它无法知道数据情况也就无法知道哪些路径会被执行，所以它需要将整个C的嵌套代码编译出，并且表现为串行的指令流。在执行时，硬件再根据实际的判断条件，对于B,C,D等分支路径，屏蔽掉不符合条件的线程，而让符合条件的线程执行，也就是说，硬件需要将不同分支的线程串行执行。相比于无分支情况时可以同时执行所有的线程，出现分支的分时复用无疑极大降低了硬件执行的效率。比如，对于两个分支，且分支的路径长度等长，则硬件执行的效率会降低到50%。

这里对比一下CPU与GPGPU处理分支的差异。在CPU上，为了维持深流水线的高效运转，采用了分支预测和推测执行技术。其性能损失主要来自于预测错误，这会导致流水线被清空，之前为错误路径所做的工作被废弃。而在GPGPU（采用SIMT架构）上，当Warp内的线程出现分支分化（即走向不同的分支路径）时，GPU必须将这些路径串行执行——先执行一条路径，同时禁用不走该路径的线程，然后再执行另一条路径。因此，GPGPU的分支代价来源于分支分化导致的线程资源利用率下降和执行周期增加。

线程分支的处理：SIMT Stack

前面讨论的是什么是分支以及其为什么会带来性能损失，本节先不考虑性能问题，而考虑如何实现功能，那么，我们如何实现这种分支控制？这里涉及两个问题：

在一个warp里的线程执行不同的线程时，如何控制部分线程执行而其它指令不执行？
如何管理和切换执行分支？

对于问题1，GPGPU引入了类似于向量处理器中谓词寄存器（predicate register）的概念来支持线程分支。每个线程都配备有若干谓词寄存器，用于存储条件判断的结果。在硬件上，一个warp的谓词寄存器集合被用来控制warp中每个线程（对应一个SIMD通道）的执行。当执行一条谓词化的指令时，只有谓词寄存器值为1的线程对应的通道会执行该指令并写回结果；谓词寄存器值为0的通道则不会执行（或不会写回结果）。这样，通过谓词寄存器，可以实现对分支路径的屏蔽和串行化执行。注意：在GPU的上下文中，谓词寄存器通常是在编译时由编译器生成代码来设置的，例如在PTX汇编中，我们可以看到使用谓词寄存器的指令，比如@p1 bra，以及条件执行指令@p1 add ...

对于编译器而言，按照CFG中的谓词掩码，4个线程都会执行基本块A（Basic Block A）。然后有3个线程进入基本块B，对应代码的第7行，而有一个线程进入基本块G，对应代码第16行。在代码块B中遇到了嵌套的分支（第8行），又是又产生了两个新的分支C和D，他们在代码块E出汇合，然后所有线程在G处汇合。编译器根据CFG生成了对应的PTX，那么，现在关系的问题回到问题2：在仅支持每周期执行单条指令的SIMD数据通路上，GPU硬件是如何实现线程束内各线程沿不同代码路径执行？

在GPU的SIMT架构中，当Warp内的线程发生分支分化时，需要串行化地执行不同的分支路径。一种关键的硬件机制是SIMT Stack。它用于管理分支的嵌套和线程的重新汇合。其工作原理是：当Warp遇到一个分支指令时，它将当前线程的汇合点程序计数器（Reconvergence PC） 和对应的执行掩码压入栈中。最先遇到的最外层分支被压入栈底，后续遇到的嵌套分支被依次压入栈顶。在执行时，硬件根据栈顶元素所记录的信息（PC和掩码）来引导Warp中活跃的线程组执行相应的代码路径。当一个路径执行完毕时，弹出栈顶元素，硬件根据新的栈顶信息恢复另一组线程的执行上下文，从而实现线程在汇合点的重新组织，并串行地执行完所有分化的路径。

在遇到一个分支时，有几个因素决定了后续的操作：

该分支的下一条指令是什么？ 这对应了下一条分支的PC值（Next PC）。
该分支的“终点”在哪里? 实际上，“终点”即分支的汇聚点，它也指向了一条PC，称为RPC（Reconvergence PC）。\textbf{RPC的值由最早的重汇聚点指令PC确定，因此称为直接后续重汇聚点（Immediate Post-DOMinate reconvergence point， IPDOM）}。它告诉栈，到这里这条分支结束，可以被弹出栈了。
哪些线程执行在该分支下活跃？ 这可以使用掩码表示。

这里推测一下硬件如何实现这种栈：在代码的第6行是跳转指令@p1 bra F，它根据p1的值确定了是否跳转，而p1的值在第5行计算，因此可以知道每个线程是否活跃。如果跳转指令出压栈时p1的结果仍在后续流水线中没有写回，可以使用bypass方式读取，如果没有计算出结果，GPGPU不会等待计算结果，而是插入其它warp的指令执行，当分支结果计算出来后再继续执行原warp的分支。若所有线程在某一分支都不活跃，则可以从栈中直接将该分支弹出（或者不压入栈直接舍弃），执行下一分支。

这里分析一个问题：为什么在条件指令和跳转指令之间执行其它warp的指令？跳转指令和条件指令是有依赖关系的，在CPU中，往往采用分支预测，在条件判断还没出结果时继续流水预测的分支方向，如果预测错误则冲刷掉。但是在GPGPU中，它的流水线上寄存器很多，冲刷这些寄存器会导致较大的功耗和资源浪费。另外，由于一个SIMT Core可以分配到多个warp，相当于有多个线程可以选择，所以在两条依赖指令之间插入其它warp的无关指令既不会影响这种依赖，也塞满了硬件计算的带宽。实际上，GPGPU正是使用这种多warp之间的调度来避免依赖或者stall的硬件空闲。

Stack存在的问题

Stack的特征是，需要根据栈顶的条目信息执行，当栈顶指令需要依赖于栈底的指令时，就会发生卡死。比如下面的代码段，假设它的A，B的执行需要依赖于X，Y的执行，而else分支被压到栈底if分支在栈顶时就会发生死锁。

    if (threadIdx.x<4){
        A;
        B;
    }else{

Stackless的SIMT架构

根据上一节知道，SIMT stack的缺陷在于，其通过LIFO顺序严格固定了分支执行顺序，这个顺序由代码嵌套结构静态决定且不可改变，因此可能会导致特定编程模式（如warp内自旋锁）与SIMT执行模型冲突。那如果希望解决这种缺陷，我们希望有更灵活的执行顺序管理方式。

同样的，一个更灵活的执行顺序管理方式应该满足什么条件？首先，该方式应该具备分支的执行管理功能，即能知道分支的起点（NPC）和终点（RPC）以及哪些线程会活跃（active mask）；其次，我们希望在可能发生死锁的情况下（线程之间存在依赖的情况下），能灵活的决定分支执行的顺序。

更新的GPGPU（NVIDIA Volta（专利：Execution of divergent threads using a convergence barrier））将这种工作主要交给了编译器。编译器为了管理分支，会在分支开始位置插入"ADD Label”指令，在汇聚点插入“WAIT Label”指令，其中的Label用于指明ADD和WAIT的对应关系。而存在依赖关系时，编译器会插入“YIELD”指令，让死锁情况下的部分线程暂时让步，其他线程先继续执行以解开死锁。例如在前面的例子中，编译器会在while循环后设置“WAIT”作为分支屏障，以管理分支。当其中一个线程拿到原子锁后，YIELD指令可以使其他线程先进行让步，使得拿到原子锁得线程可以执行到C以释放，然后其他线程再继续执行。下图展示了基于栈的线程执行流和基于“分支屏障”的线程执行流，基于分支屏障的线程可以通过"yield"指令暂时让步，使得线程执行更加灵活。

为了协助完成这些调度方式，硬件需要维护基本得线程状态，其字段如下图所示。屏障参与掩码（Barrier Participation Mask）用于跟踪给定warp内哪些线程参与给定的收敛屏障。一个给定的warp可能有多个屏障参与掩码。在常见情况下，由给定屏障参与掩码跟踪的线程将相互等待，以到达程序中发散分支后的一个共同点，从而一起重收敛。为了支持这一点，屏障状态字段（Barrier State）用于跟踪哪些线程已到达给定的收敛屏障。线程状态（Thread State）跟踪warp中的每个线程是准备执行、在收敛屏障处阻塞（如果是，是哪个屏障），还是已让出（对应yield）。线程rPC字段跟踪每个非活动线程要执行的下一条指令的地址。线程活动字段则指示warp中的相应线程是否处于活动状态。

更高效的线程分支执行方案

在解决了分支管理之后，回到最开始提到的性能问题：如何提升分支情况下GPGPU的硬件性能？前面介绍得分支解决方案是将分支串行执行，且由于每一个分支只有部分线程活跃，会导致部分硬件lane处于空闲状态，这实际上按照相同warp相同pc进行分组的执行方式。

同warp同pc进行分组造成GPGPU性能损失的本质在于，在一个SIMT Core中能同时执行的线程不够，一个能执行32个线程的SM在一个32线程的warp中寻找相同pc的线程。那如何是在两个warp，三个warp中找呢？那就更容易使用相同PC的线程填满一个Core执行，因此提升硬件性能的另一种方式是按照不同warp相同pc线程进行重组和合并。

另外，可以考虑将SIMT Core的指令流扩宽，让两个分支的PC都去控制不同的硬件Lane，类似于MIMD的架构，这里分支之间就可以并行执行了，这种方案是按照相同warp不同PC进行重组和合并的。

对于更碎片化的情况，比如只有if条件但没有else，或者if分支和else分支不平衡，那么即使采用同warp不同PC的组合也会存在较大的性能损失，那么另一种重组方式则是按照不同warp不同PC线程进行重组和合并，这像是更进一步的MIMD方式。

需要注意的是，采用不同重组方式必然会导致硬件设计更为复杂，通过硬件设计的复杂度换取性能提升的性价比需要在实现时好好评估。
