0x00 总纲

作为工业背景的 RL 新手，跑 open-r1 是想增强模型精度指标、或者指标没提升但用户体验更好，再或者扩展 agent 的可行性。目的是 <thought> 这坨标记的复现方法，对过去的工作能起到啥帮助。

之前（5min理解open-r1代码结构）介绍 open-r1 由几个独立脚本构成，这篇也从几方面分享一下。

千算万算，算不到第一个头疼的是 evaluation。

跑了 open-r1 自带的 evaluation.py 不太稳定：运行到末尾可能就卡死了；issue 也有说测试结果对不齐的； lighteval 这个框架封装太多，实在提不起改的欲望。oc 依赖 3.10、可靠，但用 vllm 比较慢的样子。

0x02 运行训练

如果用我的 repo，训练命令在 run_3B.py 可以找到：

ACCELERATE_LOG_LEVEL=info accelerate launch --config_file recipes/accelerate_configs/zero2.yaml     --num_processes=5 src/open_r1/grpo.py     --config recipes/Qwen2.5-3B-Instruct/grpo/config_simple_rl.yaml

做了几处修改：

调整默认 reward funcs、改了些训练参数
官方 prompt 写得不对，发了 PR 还没 merge https://github.com/huggingface/open-r1/pull/325
模型改 instruct

为啥这么改呢？

2.1 base v.s instruct

经常炼 LLM / 刷 arXiv 的都知道，如果非要分阶段训练，重要任务要先训。从 base 开始，模型上限会更高。base 也带来新问题：

即便 llama3/qwen 有退火，但指令不遵循
system prompt 对 format_reward 得分影响很大 （所以检查了 prompt 书写，发了个 fix PR）
原本 RL 收敛就慢，用 base 肯定会更慢

我猜这也是为啥 r1 论文需要几千条 sft 冷启动；我猜只是要学 format 的话，LoRA 同样可以冷启动。

回到最初的目标：优化精度。

检查过程肯定比<think> 这个标记本身重要（配合 HuixiangDou2 的实验结论，LLM 检查过程比检查 [过程，结果] 更高效）
所以 format_reward 可以舍，但得换成检查过程的 reward 方法

为了方便 LLM 出过程、system prompt 生效，选 instruct。 当然 base+格式学习是更好的起点。

2.2 reward_funcs

反复试了几轮后，发现关键在一组 reward_funcs 实现上。它有点像空域 fft 到频域的味道：

你先讲清楚预期结果符合什么规范。再把这个中文标准，翻译成若干个 function，拟合你说的中文。

我不在意<think>标签本身，只希望模型输出充满逻辑、尽可能长、但又不要啰嗦。
对于 response 正确和长度组合，存在一套偏序。例如优先既长又对、其次是短且对、再次是长且错，最后是短且错。

有了这个描述，再看 open-r1 实现的 rewards，就明白了啥要啥不要：

format_reward 不要
len_reward 实现太简单了，不要
accuracy_reward 检查结果，必然要
reasoning_steps_reward+get_repetition_penalty_reward 检查过程且不能重复，留着
get_cosine_scaled_reward 符合偏序诉求，留着

当然组合不固定，可以探索更多 reward 实现。这大概率..是个能水文章的点？

2.3 模型尺寸、context_length、epoch 和 lr
以下都是主观感受，没有卡来验证
对比	sft	grpo
不同尺寸	1.5 / 4 / 7 / 14 不同尺寸，效果逐渐 scaling	更容易崩。 7B 明显比 1.5B 稳定一大截，但又没啥 gpu 跑太多次 7B。只能用 3B 折中
epoch	不管啥尺寸，训练都容易收敛	感觉有个“渡劫”的过程，1 个 epoch 渡不完。也和 lr 有关
context_length	就是 YaRN 那堆	- 官方默认 1024/2048，但你统计一下训练数据长度，得 16k 才能覆盖多数
- 实测，直接上 16k，精度比 4k 要高
lr	按 qwen 技术报告来	官方的 2e-5 和 3e-6 好。如果按 qwen 技术报告换算，最后会得到一个正常的、普通的 chat 模型
