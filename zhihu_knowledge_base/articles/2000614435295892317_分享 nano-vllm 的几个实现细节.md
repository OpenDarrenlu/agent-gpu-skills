细节一：torch.compile貌似会映射到 cutlass 实现

印象里的torch.compile很好拿捏，“随随便便”就能写出更快的版本。

所以当看到这个 rms_norm实现的时候，就好奇为啥这么写。多花0.5天的工作量不能更快么？

    @torch.compile
    def rms_forward(
        self,
        x: torch.Tensor,
    ) -> torch.Tensor:
        orig_dtype = x.dtype
        x = x.float()
        var = x.pow(2).mean(dim=-1, keepdim=True)
        x.mul_(torch.rsqrt(var + self.eps))
        x = x.to(orig_dtype).mul_(self.weight)
        return x

细节二：silu_and_mul pass 优化

我一开始以为 nano-vllm 的 MLP 会原样仿照 Qwen3.py 的实现，也就是：

self.down_proj(F.silu(self.gate_proj(x)) * self.up_proj(x))

可以把它画成图：

可以直观看到 x 需要 bcast，乘以不同的linear.weight。

为了提高性能，nano-vllm 把 2 个 linear 权重 concat 到一起；同时 fuse 了 silu 和 dot，新的 pipline 变成这样。类似的技巧同样出现在 affine qkv。

细节三：RoPE 算子 Singleton

RoPE，那个利用“偶数维空间自带复结构”特殊性质，给 qk 增加相对位置编码的。

为了加速计算，nano-vllm 准备了一些 base 对应的 cos/sin 的 cache

        cos = freqs.cos()
        sin = freqs.sin()
        cache = torch.cat((cos, sin), dim=-1).unsqueeze_(1)
        self.register_buffer("cos_sin_cache", cache, persistent=False)

这个函数是无损的，那就可以巧妙地做成单例，避免 qwen3 的多个 decoder 重复申请 cache。节约显存。
