前言

本章核心问题：软件Kernal如何分配到GPGPU的SM执行？

正如之前提过，GPGPU的编程模型按照Grid，Block，Thread的划分，并且在之前提到了Block会按照不同的GPGPU架构拆分成Warp在硬件SM上执行。而GPGPU的高性能秘诀在于，一个SM通过对多个Warp进行交叉执行，掩盖访存带来的性能损失，从而使得计算单元处于繁忙状态提高计算带宽。因此，我们需要理解的是：

Warp可执行的条件是什么？当SM存在多个就绪Warp时，又如何选择执行的Warp？
SM中的Warp是如何从Grid得到的？这些Warp按照什么方式分配给SM？

为了能更好的理解这一流程，本文还参考了开源项目的设计作为参考。最后，本文还会讨论该流程不同的分配调度方式会给GPGPU的性能带来什么影响。

Warp的调度

在一个SM中，根据架构的不同，其能承载的最大Warp数也不相同，这些Warp可能来自不同的Block甚至不同的Grid。在SM中，需要一个Warp Schedule单元负责Warp的调度执行，可以在任何已经就绪的 Warp 中挑选一个并采用锁步（lockstep）方式执行。锁步执行使得所有的执行单元都执行同一条指令，从而简化控制逻辑，把硬件更多的留给执行单元。另外，可能有多个warp并行执行的情况，比如在Ampere架构里有四个warp schedule，那它就可以同时发射4个warp的指令进行执行。

Warp选择执行示意图。图中所示为两个Warp Schedule从多个Warp Pc中选择就绪Warp后并行执行

由于Warp Scheduler在多个就绪Warp中选择一个进行后续执行，所以实际的指令流会出现不同Block的不同Warp交替执行的情况，某个Schedule执行的执行顺序可能是：block 0的warp 0 → block 1的warp 0 → block 0的warp 1 → ...。

Warp的Stall

这里首先讨论一个问题：什么样的Warp可以认为是准备就绪的？在处理器中，就绪的指令一般包含三个条件：下一条指令以及取到，指令的所有相关性都已解决，需要的执行单元可用。对于GPGPU架构而言，Warp的就绪条件也类似，主要还是涉及资源，存储访存和依赖关系：

资源方面：比如如果指令运行的功能单元正忙，Warp的发射需要停顿；
访存方面：比如Cache的Miss，或者访存满载导致的load/store无法执行时，需要停顿；
依赖关系：比如Warp在等待同步指令，如CUDA中的_syncthreads()要求线程块中的所有线程到达后继续执行下一指令；以及执行时存在输入的依赖关系待解决，需要停顿；
Warp调度策略

这里讨论的问题是，如何从就绪的Warp中进行选择？基本的调度方式有两种，一是公平性最高的轮询（Round-Robin，RR）调度策略，它在调度是对处于就绪状态的线程束赋予相同的优先级，并按照轮询方式依次选择就绪的Warp；另一种则是贪心（Greedy-Then-Oldest，GTO）调度策略，它运行一个Warp一直执行到不能继续执行为止，然后选择其它最老的就绪Warp继续执行。两种策略对于性能而言各有其它优势和劣势，后续会对其进行进一步讨论。

Ventus：在Ventus中，Warp Schedule负责了两处的调度：

选择发给icache的warp，通常采用贪婪的策略，但当icache发生miss，或ibuffer已满时会将该warp的pc回退并切换到下一个warp发射。
选择发给执行单元的warp，通常采用轮询的策略，选择出当前指令缓冲有效、记分板未显示冲突、执行单元空闲的指令。切换warp仅需要一个周期。
看起来是更精细化的调度：warp schedule有两个输出：一个是发给icache的warp，采用贪心策略；另一个是确定issue执行的warp，采用轮询策略，在可发射的warp中轮询。

取值和发射的调度机制解耦，避免了空闲的浪费。

GPGPU-Sim：GPGPU-sim同样将调度分了两级，对于具备资格的 warp 按循环轮询顺序被调度访问 I-Cache。一旦被选中，系统会向 I-Cache 发送读请求，地址指向当前调度 warp 的下一条指令。默认情况下会连续获取两条指令。无论命中或未命中，循环轮询获取调度器都会继续处理下一个 warp。命中时，所获取的指令将被送往解码阶段；未命中时，I-Cache 将生成请求。当收到未命中响应时，相应块会被填入指令缓存，且该 warp 需要重新访问 I-Cache。在未命中请求待处理期间，该 I-Cache 不会访问指令缓存。

对于Issue阶段：第二轮循环仲裁器 (round robin) 从 I-Buffer 中选择一个 warp，将其指令发射至流水线的后续阶段。该循环仲裁器与用于调度指令缓存访问的循环仲裁器相互独立。Issue调度器可配置为每周期从同一 warp 发射多条指令。当前被检查 warp 中的每条有效指令（即已解码未发射）若满足以下条件即具备发射资格：（1）其所属 warp 未在 barrier等待;（2）其 I-Buffer 条目中包含有效指令（有效位已置位）;（3）通过记分牌检查;（4）指令流水线的操作数访问阶段未处于停滞状态。

Block的分配和调度

在GPGPU编程时，程序员需要通过<<<grid_dim, block_dim>>>语法显式指定Grid和Block的大小，所以Grid到Block的划分在软件端完成。一般而言，在编译完成后，CPU发送以Block为基本单位给GPU发送任务，而GPGPU中首先有一个线程块调度模块（如下图的线程块调度器）负责Block的调度。它会将接收到的Block拆分成Warp，并根据Block的大小将这些Warp分配给执行的硬件SM。需要注意，同一个Block的Warp只能在同一个SM上运行，但是同一SM可以容纳来自不同Block甚至不同Grid的若干个Warp。

GPGPU硬件层级

在GPGPU的SIMT流程执行下，线程块会经历分配和调度过程。分配指的是哪些线程块会被安排到哪些可编程处理器执行，而调度指的是已分配的线程块按照什么顺序执行。

线程块的分配

线程块的分配通常使用上述提到线程块调度器实现。在NV的架构中，该模块称为千兆线程引擎（giga Thread Engine），开源项目Ventus也实现了Block的调度模块，称为CTA Scheduler，而开源项目Vortex中并没有该模块，而是使用软件直接分配Warp。

线程块的分配通常使用轮询（Round-Robin，RR）策略。线程块调度器会按照轮询方式，根据Block所需要的资源（比如Warp数，寄存器和共享存储等）将其拆分成Warp并分配到剩余资源足够的SM上，同一Block的Warp会分配到同一SM中。Block所需的资源由内核函数告知，比如下表，在Ventus中，线程块的Warp数，Warp的线程数和共享存储的基地址都由软件配置在CSR寄存器中。

Ventus提供的CTA任务配置寄存器

当SM剩余的空闲资源不足以继续分配线程块时，线程块分配器会根据需求等待SM的空闲情况，当SM之前的任务完成并释放资源后再为其分配线程块。在《通用图形处理器设计》一书中给出了这样的例子以说明线程块调度的情况：假设一个GPGPU中有3个SM，记为SM0，SM1和SM2，每个SM运行最多同时执行2个线程块（记为TB），一个内核函数声明了12个线程块TB0~TB11，那么按照轮询策略，分配情况可能如下图所示：

最初时，线程块调度器按照轮询给SM0~SM2依次分配TB0~TB5，由于TB1先结束，资源得以释放，所以为其分配了TB6，同样的，后续结束的顺序依次为TB0，TB2……，所以依次填补TB7，TB8……。可以看到，初始轮的线程块分配顺序有规律，但是第二轮开始的线程块分配则按照执行进度安排。

线程块调度

同一个线程块是拆分成Warp后分配给同一个SM的，所以Block的调度是以Warp级的调度体现的，区别在于，Block的调度执行考虑Block间可能带来的影响，而Warp的调度更多考虑Warp内和Warp间的影响。

讨论：性能影响—资源和存储考虑

本节简单讨论不同策略的Warp调度和Block调度对性能的影响。

线程束调度优化

在GPGPU架构中，数据访存延时仍然是影响性能的主要因素，因此发掘数据的局部性是提高性能的重要手段。由于SIMT计算的并行性，拆分成线程的数据往往具有两种数据局部性：Warp内部局部性（intra-warp locality）和Warp间局部性（inter-warp locality）。对于GTO策略，尽可能连续的访问一个Warp的指令，是有利于Warp内局部性的，而对于轮询策略，尽可能访问不同Warp的相同指令，则利于Warp间的局部性。

考虑下面的一个例子说明两种调度可能带来的不利情况。假设对于16个Warp：W0，W1，W2……，W15。他们每个线程束都有16条指令，前面的15条为计算指令，而第16条为访存指令。如果采用轮询方式，则会某一时间段内因为所有Warp都在执行访存指令而导致流水线因为没有就绪Warp停顿。如果采用GTO则可以避免这里的额多个Warp同时访存的情况，但是由于对一个Warp一直取指，如果cache很小，则又会发生cache重用不足甚至抖动现象。因此，采用哪种调度方式，或者如何对调度进行改进，需要结合实际的架构考虑。总的而言，在Warp调度时，我们希望使用Warp切换能掩盖长延时操作，也希望cache的数据能有较高的利用率。

线程块的调度优化

线程块的调度优化和线程束的调度优化类似，需要考虑对存储系统的访存情况。初次之外，线程块的调度还有一个现象：向SM多分配的线程块越来越多时，整个性能可能会出现“先上升、再平缓、后下降”的趋势，其主要原因来自于资源的竞争和缓存的抖动等。因此再线程块的分配调度时，还需要考虑SM所分配的线程块既可以满足利用并行性掩盖长延时操作，也要考虑多个Block的资源竞争带来负面影响。

写在最后

文本主要梳理了kernel函数如何拆分成Block以及Warp映射到SM的执行过程，以及Block的调度和Warp的调度策略，参考了几个开源项目的实现方式，讨论了不同调度的性能影响。但是由于认知和精力有限，没有对各种调度优化方案进行深究，因此这一部分需要参考更新的架构设计或者论文。

实际上，不论时Block的分配调度还是Warp的调度，其本质功能都是很简单的，即判断就绪条件和选择就绪任务。但是不同的调度策略却能有更深层次的影响，并且这种影响是很难感知的，往往需要运行大量的benchmark才能确定最优的配比，因此从这一块儿的学习也能认识到架构调试性能的重要性。
