例如让 LLM 给一段文本打分，基本要求：

首先概念清晰。打分是打什么分？指代消歧、句子顺滑还是主成分析，对应经典 NLP 哪个任务？
然后背景充分无歧义。打分得先明确啥叫“好”。主谓宾都有算好，还是有信息量才算好？
再次指令明确。你要输出啥格式，json还是直接str；满分 10 分还是 100 分？

更高级地，要考虑模型特性：

如果做打分这种偏理解的任务，稍加一些示例更佳。有文章表示 LM few-shot 和训练等效[1]；但如果做生成类任务，还是别加具体示例[2]
尽量避免 prompt 包含无关内容。计算上会影响 decoder 里面的 attend
不要一次性要求 LLM 完成多个任务，one function one job，否则会失焦[3]

想理解 prompt，要读一些关于 LLM 可解释性的论文（如 physics of LMs）。但只是想写好 prompt，这里有个模板：

定义角色-阐述任务-输出要求-格式要求-举例-注意事项

那怎么让 LLM 具备这种分步求解的规划能力呢？我知道的是 post-training 训出来的，也就是要构造这种训练数据（sft 或 RL）。2025 的 LLM 都具备这种 step by step 执行的能力。




function call 和 agent

现实世界里，各个山头已经挤满了人。遗留的上古数据、工具非常多。

单个模型不够用，纯文本也难以适应很多场景——例如生物的 DNA 序列、医疗的 CT 图。那 LLM 配合 coding 是一种解决思路，毕竟编程是图灵完备的，理论上足以完成所有任务。剩下的就是把现世的所有东西，做成 LLM 工具和血包。

“狭义的” agent，就是 LLM + function call。但只是工具调用的话.. bert也能干，显然 LLM 能力不止于此！

现实往往比 “周杰伦年龄比我年龄大几岁” 复杂得多：WebSearchTool 输入 周杰伦，唉查无此人！ 数据库只有 JayChou，运行终止。

“广义的” agent 是期望 LLM 能自己解决问题。任务规划、执行工具不必说：

中间结果怎么组织。存表，存无结构的图，还是别的啥？
出现幻觉如何消除。反思、循环自证还是引入外部同义词库？
如果智能体有多个，信息怎么有效传递？




mcp 和 claude skills：有一种需要，是大厂觉得你需要

前面提到，“把现世的所有东西，做成 LLM 工具和血包”，世上那么多 api 接口，自然需要一个 onnx ！

2024 年这些工具的 onnx 叫做 mcp，实质是一套极琐碎的前后端接口。把大象关冰箱里得几百行代码。

这些只是商业公司定义的工具调用方式..不妨想想 onnx 和 torch，想想 ofo 小黄车和摩拜，鹿死谁手还未可知。
