一、并行计算的基本规则与 LLM 中并行计算的层次
1.1 并行化计算：Amdahl 定律与 Gustafson 定律

在计算与程序设计中，并行化是提高效率的重要方法，那么什么样的程序可以并行化，并行化能够提高的极限在哪里呢？本节将讨论这些问题，首先我们可以将程序分为3种类型：

顺序程序。该类程序的每个部分都依赖于以前执行的任务，不能分解为独立的任务。如遍历链表。
并行程序。该类可以很容易地分解为独立的任务，并独立执行。例如，判断是个数字是否为质数。
混合程序。该类程序由顺序和并行部分共同组成。在现实世界中的大多数程序都是这种。对于这类程序，从并行编程中获得多少效率，可以通过使用 Amdahl 定律计算理论加速来确定。

让我们用 s 表示程序的顺序部分所需时间，用 p 表示并行部分所需时间，用 N 表示处理器的数量，则

1 各处理器完成任务所需的时间为 T_1 = s+p
N 各处理器完成任务所需的时间为 T_N = s+p/N

现在我们进行归一化，即将1 个处理器完成任务所需的时间看作单位 1，此时 s 表示程序的顺序部分所需时间占比，p 表示并行部分所需时间占比，则有 1 = s+p ，则 T_N = 1-p +p/N ，则加速比为 S(peedup) = \frac{1}{1-p+p/N}\\ 此即 Amdahl 定律的方程表示。

稍加分析一下以上定律，即可以绘制加速比 S 与 p, N 之间的关系曲线如下，可以得到如下规律：

增加处理器在一定程度上提高了性能，但达到一定数量后，就会边际效益递减；
并行计算的占比对加速比的影响非常巨大，并决定了加速比的上限

这两点对于算法开发和性能优化至少有如下启发：

在算法设计时需要非常关注并尽可能提高模型中可并行的部分，如 RNN 和 Transformer 的区别；
使用多个处理器的并行计算的优势仅在程序高度可并行化时才有益；

Amdahl 定律告诉了我们关于并行计算的非常重要的信息，但是其有一个关键假设，即：要解决的问题是静态的。通过增加设备的目的是更快地完成计算，而在实际场景中，随着计算设备的增加，期望解决的问题的规模也会增加。这就要提到 Gustafson 定律。

在 Gustafson 定律中，常数不是问题的大小，而是我们想要等待结果的时间。Gustafson 观察到，问题的平行部分与问题大小成比例，而顺序部分几乎不成比例。即如果将 N 各处理器应用于具有比例 s 的顺序任务，则缩放任务所需的时间与以前相同。而如果应用于具有比例 p 的并行任务，工作速度是并行部分的 1 个处理器的 N 倍。那么，则有加速比为 \begin{aligned} \text { S(peedup) } & =\mathrm{s}+\mathrm{p} \times \mathrm{N} \\ & =\mathrm{s}+(1-\mathrm{s}) \times \mathrm{N} \\ & =\mathrm{N}+(1-\mathrm{N}) \mathrm{s} \end{aligned}\\Amdahl 定律是悲观的，而 Gustafson 定律是乐观的。从 Amdahl 定律的角度来看并行性可能会令人沮丧。Gustafson 证明，当我们增加平行部分所做的工作量时，顺序部分施加的瓶颈变得不那么恶性。

事实上，当应用于现代并行系统时，这两个定律可能并不准确，因为通信成本对速度有很大影响。随着处理器数量的增加，因为通信成本最终会变得如此之高，以至于它们抵消了增加工作负载大小所获得的任何好处。

1.2 LLM 中并行计算的几个层次

随着 LLM 的兴起，预训练阶段的资源消耗巨大，动辄使用数千颗GPU，花费数月时间。为了提高计算效率，节省计算时间，便出现 3D 并行，乃至 4D 并行等方法，并行训练在当前已成为一门“显学”。另一方面，在推理场景中，由于 token by token 的生成方式，decode 阶段计算效率非常低，通过并行计算来提高计算效率，缩短对话延时，已经变成一个迫切的需求。

本节将从宏观的角度，梳理不同层次的并行计算方法及其典型方法，不同层次间是相互正交的，即可以组合使用。当然不同层次的方法适用范围也不尽相同，在之后的章节中，都会进行详细讨论，期望通过这一过程，对模型、对算法、对计算、对硬件都有更进一步的理解和认识。

二、通用型并行计算方法

最通用的并行计算方法主要在一大一小两个极端上，即指令并行和数据并行。

指令并行，即我们最常理解的并行计算，最典型的就是 CUDA 编程模型。其通用的原因是系统提供商以及众多开发者支持了广泛的算子，以至于封装成更上层的框架，用户可以无感知的利用上这些并行计算；
数据并行，即最基础最常用的模型并行训练方法，其特点就是将一个模型看做一个整体放到一个 device 上完成计算，并在反向的时候进行同步计算均值。其特点就是可以模型看做一个整体，而无需关注其中实现的细节（前提则是模型可以在单个 device 上进行计算），这样只需要增加 device 就可以完成并行计算。

下面将更加详细介绍此二者。

2.1 CUDA 并行与 CUDA 内核

本节主要讨论 CUDA 并行计算的核心思想及设计理念，关于 CUDA 编程的具体操作细节可以参考笔者之前的专栏，不是本节讨论的重点

现代 GPU 由数千个 CUDA core内核的小型处理单元组成。这些 core 并行协同工作，使 GPU 对于可以划分为更小的独立操作的任务非常有效。CUDA 本质上对 GPU 的巨大计算能力提供了一个编程模型和一组工具，允许开发人员编写可在 GPU 上执行的代码。其工作原理：

并行处理：CUDA 能够将计算任务划分为更小的部分，这些部分可以在不同的GPU内核上同时执行。与在 CPU 上按顺序运行相同任务相比，这大大加快了计算速度。
Threads 和 Blocks ：在 CUDA 中，计算被划分为 Threads 并组织成 Blocks。 Threads 是单独的计算单元，而 Blocks 则将 Threads 组合在一起。一个 Blocks 内可以同时工作大量 Threads，多个 Blocks 可以并行运行。
SIMD 架构：GPU 使用单指令、多数据 （SIMD） 架构。这意味着可以同时在多个数据元素上执行一条指令。比较典型的就是一个 Wrap (通常 32 个Threads) 在同一时间执行一条指令。

事实上以上思想也并非 CUDA 的原创，例如久远的古希腊步兵方阵即遵循类似的运行原理（单个步兵可看做一个 core，一次执行一条指令；同一排组成 wrap, 同时完成同一条指令；一个方阵构成一个 block, 多个方阵构成一个 grid）。

GPU 结构与步兵方阵

理解了以上的原理后，接下来关键的便是其内存模型与编程模型。

GPU 具有不同类型的内存，具有不同的速度和大小。CUDA 提供了用于管理不同内存空间中数据的机制，简单来说主要是：

全局内存：在主机和设备之间共享。速度较慢，但容量较大。
共享内存：在块中的线程之间共享。速度更快，但容量有限。
本地内存：每线程的专用内存。
常量和纹理内存：针对特定访问模式进行了优化。

CUDA 编程涉及编写主机代码（在 CPU 上运行）和设备代码（在 GPU 上执行）。主机代码管理 CPU 和 GPU 之间的数据传输，而设备代码在 GPU 上执行实际计算：

主机代码：在 CPU 上执行并管理 GPU 资源。
设备代码：在 GPU 内核上并行运行。
内核启动：主机代码启动设备代码内核以在 GPU 上运行。
内核函数 ：在 CUDA 中定义要在 GPU 上执行的计算。这些函数由数千个线程并行执行。每个线程都有唯一 ID，这允许它根据其数据执行适当的计算。
2.2 数据并行及其优化

数据并行 (DP) 是深度学习中最基础也最常用的并行方式。其基本原理如下：假设一个 batch 有 n 个样本，一共有 k 个 GPU，第 j 个 GPU 分到 m_j 个样本，考虑等分情况，则 m_j = \frac{n}{k} ，如果考虑总损失函数 loss 对参数 w 求导，则有

\begin{aligned} \frac{\partial \text{ Loss}}{\partial w} &= \frac{\partial \Big[ \frac{1}{n} \sum_{i=1}^{n} f(x_i, y_i) \Big] }{\partial w} \\ &= \frac{1}{n} \sum_{i=1}^{n} \frac{\partial f(x_i, y_i) }{\partial w} \\ &= \frac{m_1}{n} \frac{\partial \Big[ \frac{1}{m_1} \sum_{i=1}^{m_1} f(x_i, y_i) \Big] }{\partial w} + \frac{m_2}{n} \frac{\partial \Big[ \frac{1}{m_2} \sum_{i=m_1 + 1}^{m_1 + m_2} f(x_i, y_i) \Big] }{\partial w} + \cdots + \frac{m_k}{n} \frac{\partial \Big[ \frac{1}{m_k} \sum_{i=m_{k-1} + 1}^{m_{k-1} + m_k} f(x_i, y_i) \Big] }{\partial w} \\ &= \frac{m_1}{n} \frac{\partial l_1}{\partial w} + \frac{m_2}{n} \frac{\partial l_2}{\partial w} + \cdots + \frac{m_k}{n} \frac{\partial l_k}{\partial w} \\& =\frac{1}{k}\left[\frac{\partial l_{1}}{\partial w}+\frac{\partial l_{2}}{\partial w}+\cdots+\frac{\partial l_{k}}{\partial w}\right] \end{aligned}

从上面的计算公式中可以看出，所有卡上总 batch 的平均梯度，和单卡上 mini-batch 的平均梯度汇总之后在平均的结果是一样的。而最终的平均梯度汇总中的 + 号，即 allreduce 通信过程。

事实上，FSDP 也存在一些问题，主要体现在两点：

FSDP 为层动态分配内存，有时需要比实际需要更多的内存
在反向传递过程中，会出现“让路效应”的现象。即在reduce_scatter操作之前，有很多准备计算（通信下的小操作）。小型计算与主计算流并行运行，严重减慢了通信速度。这导致通信之间存在很大的间隙，因此，计算流中也会出现相同的间隙。如下图所示：

针对以上问题 YaFSDP 提出了如下优化方案：

增加 buffers。分配两个缓存用于存储中间权重和梯度。每个奇数层使用第一缓存，每个偶数层使用第二缓存。这样，不同层的权重就存储在同一个内存中。如果各层具有相同的结构，缓存将始终相同！重要的是要确保当需要第 X 层时，缓冲区具有第 X 层的权重。所有参数都将存储在缓冲区中相应的内存块中

有了缓存之后，需要建立通信以实现以下效果：

直到该层的权重收集到其缓冲区中后，该层上的前向/后向传递才会开始。
在某一层的前向/反向传递完成之前，不会在该层的缓存中收集其他层。
直到使用相同梯度缓存的前一层上的 reduce_scatter 操作完成后，该层上的反向传递才会开始。
直到相应层的反向传播完成后，缓存中的 reduce_scatter 操作才会开始。

为了达到以上效果，还需要以下优化：

使用 CUDA Streams 来促进并发计算和通信。为了促进流之间的通信，您可以使用“event”原语（Torch 中的event = torch.cuda.Event()）。我们可以将一个事件放入流（event.record(stream)）中，然后它会像微内核一样附加到流的末尾。我们可以在另一个流（event.wait(another_stream)）中等待该事件，然后该流将暂停，直到第一个流到达该事件。

只需要两个流来实现这一点：计算流和通信流。可以通过以下方式设置执行以确保满足条件 1 和 2。如下图所示，粗线标记event.record()，虚线标记event.wait()。直到该层上的all_gather操作完成（条件 1）后，第三层上的前向传递才会开始。同样，直到使用相同缓存的第一层的前向传递完成（条件 2）后，第三层的all_gather操作才会开始。由于该方案中没有循环，因此不可能出现死锁。

    def get_forward_pre_hook(self):
        def forward_pre_hook(module, input):
            self.all_gather(self.comm_stream)
            self.weight_reusable_buffer.start_processing(self)
            if self.is_hpz_layer and not self.hpz_inited:
                self.hpz_buffer.copy_(
                    self.weight_reusable_buffer.get().narrow(
                        0, self.hpz_shard_size * self.hpz_rank, self.hpz_shard_size
                    )
                )
                self.hpz_inited = True

        return forward_pre_hook

    def get_forward_hook(self):

            self.weight_reusable_buffer.end_processing(self)

三、特异型的并行计算方法

之所以将其他方法称之为“特异型”，是因为这类方法都需要考虑模型本身的结构特征，无法简单作为一个黑盒看待。以相对宏观的模型并行为例，普遍的残差连接的设计使得流水线并行(PP) 更加细化的切分变得非常困难，尽管可以以层为单位进行切分，也仍然需要考虑 embedding 层和 LM 层；而张量并行(TP) 则更加复杂，仅以 MLP 为例，线性层是否有 bias，线性层的个数都会对切分方法及通信方式产生很大影响。

3.1 模型并行与序列并行

模型并行包括两个维度，即：流水线并行(PP) 和张量并行(TP)。其原理及细节笔者在之前的文章中都有详细介绍，在此不与赘述，感兴趣可以阅读原文

本节关注的重点在针对长文本的序列并行(Sequence Parallelism, SP 或者 Context Parallelism, CP)。在当前 SP 又分为两个技术流派，即 DeepSpeed-Ulysses 和 Ring-Attention，而这两种路线的底层原理恰好又分别对应着 TP 和 PP，下面来让我们一探究竟。

DeepSpeed Ulysess：沿序列长度切分 Q、K、V 并分发到不同 GPU 上，在计算全局信息时通过 allgather 获取全局信息，通过 reduce-scatter 按照多头分发， 由于这两个通信原语的通信量只与数据量有关，因此随着 GPU 数量增加并不会增加通信量。其思想类似于 TP，在该方法出现之前，Megatron-LM SP, ColossalAI SP 也采用了类似的方案，该方法受限于 Attention 头数。
Ring-Attention：同样沿序列长度切分切分Q、K、V，但是其计算并不是在空间上并行的，而是在时间上并行，核心是通信和计算重叠。其思想类似于 PP，在此之前最经典的算法是 FlashAttention。

即然 PP 和 TP 可以结合使用，以上两种方法也可以结合使用，这就是 UNIFIED SEQUENCE PARALLELISM 。更多的分析还需要做更多实验，待续。
