最近刚投完稿子，我想写一篇轻一点的技术梳理，回头看看最近几款 LLM decode 加速器到底是怎么设计的。过去读这些工作时，很多时候是带着某个具体问题去查；但这次我更想换个角度，从硬件架构出发，把它们放在同一张图里比较一下：它们各自瞄准了什么瓶颈，为什么会长成现在这个样子，以及这些设计有没有一些更本质的共性。一边分析它们的设计逻辑，一边也帮助自己把这个方向重新梳理清楚，和大家探讨一下我的理解有没有问题。

首先先看llm decode是怎么样的memory-bound，粗略推导一下，为什么传统面向TPU/GPU的可能不适合。

在分析 LLM decode 的硬件瓶颈时，我们先暂时排除 attention，因为其计算量相对较小（当然长文本，尤其是agent场合除外，这个后面写文章分析）只关注其余占主导地位的线性算子，例如 Q/K/V/O projection、FFN 中的 up/down/gate projection，以及 MoE expert 内部的线性层。对这类算子，可以统一抽象为一次矩阵乘法：输入矩阵 A 的形状是 M×K，权重矩阵 B 的形状是 K×N，输出矩阵 C 的形状是 M×N。

因此，这类算子的计算密度，也就是 arithmetic intensity，大约可以写成：

2MNK / (MK + KN + MN)

由于在 decode 中通常满足 M 远小于 N 和 K，因此数据搬运量往往主要由权重项 KN 主导。于是上式可以进一步近似为：

2MNK / KN = 2M

也就是说，这类算子的计算密度大约就是 2M 次运算每元素。M一般是batch数目或者attention参数这些，

最近几款NVIDIA H200（dense BF16）：约 206 FLOP/Byte 。NVIDIA GB200 / Blackwell（dense BF16，按 2-GPU superchip）：约 313 FLOP/Byte。Google TPU v6e / Trillium（BF16）：约 574 FLOP/Byte Google TPU7x / Ironwood（BF16）：约 320 FLOP/Byte。也就是说可能batch几百才能到他们的拐点，还有各种非线性算子和通信算子（虽然可能一般都是流水线掩盖掉了，但调度上确实也麻烦）。从这个角度，做一个decode加速器还是挺有必要的。 从供给路径看，现有 decode 加速器本质上都在回答同一个问题：如何把每个 token 需要的权重和状态更高效地送到计算单元。沿着这条主线看，主流方案大体收敛为两类：一类是尽可能靠 SRAM ，做大或者做近，用对应的数据流设计计算硬件。另一类则类似的思路，靠 DRAM。

不过，上面这种“SRAM 路线 / DRAM 路线”的二分法，只适合作为一个粗略的观察视角，并不是严格的硬件分类。例如，Groq 的 LPU 常被理解为偏向大规模片上供给和确定性执行的一类方案，但最近也有争议扩展片上内存是3d-sram还是3d dram。以上很多也不是decode加速器的异构，比如然后wafer的celebras的话，他也是计算prefill和decode。它的 WSE-3 一方面拥有极大的片上存储和带宽，官方公开资料强调其 44GB on-chip SRAM 和极强的片上数据供给。天天和麦老师组里的交流，正好写个技术文章来分析总结一下一下。SambaNova 的 RDU 片上 SRAM、HBM，以及更大容量的 DDR 同时参与数据供给。
