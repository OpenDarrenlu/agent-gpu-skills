现在这个小小的kernel library 六个月前终于开源了，那我们来稍微详细说说。

这个kernel 主要是用来加速 EmbeddingBag (https://docs.pytorch.org/docs/stable/generated/torch.nn.EmbeddingBag.html#embeddingbag) 这个 op。

直观上的解释是，想象我们的网站上有一百万个电影要给用户做推荐，每个电影用一个长度128 (embedding_dim)的向量(embedding)表示。那我们有个embedding table，大小 1M * 128。

每个用户我们知道他最喜欢的64部电影。那我们可以把这100部电影的embedding 拿出来，做一个平均，得出这个用户的特征向量。

那接着假设我们的batchsize 是1024，即在1M * 128 的embedding table 里抓取 1024 * 64个embedding。

这很显然是一个memory intensive 的操作。那写kernel的时候那些常用的trick 就可以放上来了。CuEmbed 对这些trick做了些抽象和包装，这里推荐先读一下 leetCuda 里的朴素kernel。

Vectorized load.
Loop unrolling.
还有一个小trick 是可以先把indices load 到shmem 里省掉了重复读取，然后再去取embedding。这个在LeetCuda 的kernel里没有。

都是些很常规的操作。2021到2022年的时候ldgsts刚提出，还有TMA可以稍微考虑下但是很难用。所以写的时候就用了这些。

CuEmbed里的 kernel 长这样，封装主要考虑了一些用法上的问题：

lookup indices 可能是稀疏矩阵的COO, CSR 表示，所以就写个indexer 和 addresser
output embedding 有可能是加权平均，或者其他的表示方法，所以有combiner 和 IsWeighted
loop unrolling 这个比较重要，后面讲。
template <class IndexLoaderT,
          class AddresserT,
          class CombinerT,
          bool IsWeighted,
          int UnrollFactor>
__device__ __forceinline__ void EmbeddingLookupImpl(
    const typename AddresserT::InputType* __restrict__ params,
    const int embed_width,
    const int batch_size,
    const typename IndexLoaderT::IndexType* __restrict__ indices,
    const typename IndexLoaderT::OffsetType* __restrict__ offsets,
    const int num_hots,
    const typename IndexLoaderT::WeightType* __restrict__ weights,
    typename AddresserT::OutputType* __restrict__ ret) {
  const int sample_id = blockIdx.x * blockDim.y + threadIdx.y;
  IndexLoaderT index_loader(
      batch_size, sample_id, indices, weights, offsets, num_hots);

  if (sample_id >= batch_size) {
    return;
  }
  CombinerT combiner;

  AddresserT addresser(params, ret, sample_id, num_hots, embed_width);

  int64_t embed_row_offset = threadIdx.x;
  int64_t output_row_offset = threadIdx.x;

#pragma unroll UnrollFactor
  for (int i = 0; i < index_loader.GetHotness(); ++i) {
    auto index = index_loader.GetLookUpIndex(i);
    if constexpr (IsWeighted) {
      auto weight = index_loader.GetWeight(i);
      combiner.Gather(addresser.GetEmbeddingAddress(index) + embed_row_offset,
                      weight);
    } else {
      combiner.Gather(addresser.GetEmbeddingAddress(index) + embed_row_offset);
    }
    combiner.OutputForConcatIfNeeded(addresser.GetConcatOutputAddress(i) +
                                     output_row_offset);
  }
  combiner.OutputForReductionIfNeeded(addresser.GetOutputAddress() +
                                      output_row_offset);
}




下面开始被老板/同事吊打。

这很明显是个memory intensive 的op。那我们的希望是能够打满DRAM bandwidth (？)

我们可以写一写benchmark code 来测这个kernel。
能想到的是需要定embedding table 大小，需要定lookup indices的个数，look up indices 随机平均分布生成 (?), 然后测achieved DRAM BW。
于是我们很轻易的在这个benchmark 上打满了DRAM BW。我们很开心。
于是拿来实际的embedding table usecase 去测，发现同样的kernel用他们的数据，同样大小的embedding table 和用户batchsize + num_hotness，DRAM BW util 只能徘徊在50%-70%。
那为什么我们的benchmark 打满了，真的数据就达不到呢？
查了下用户数据和我们benchmark 的 cache hit。发现用户数据的l1 / l2 cache hit 远高于我们随机平均分布生成的 cache hit。
这个解释很直观。1024个用户最喜欢的64个电影，很可能有重复。
于是我们在benchmark 的时候需要考虑到这样的分布。
那这个分布的名字叫 power law。我们在造数据的时候要造这样的数据，并且要有用power law 去拟合用户数据分布的能力。
