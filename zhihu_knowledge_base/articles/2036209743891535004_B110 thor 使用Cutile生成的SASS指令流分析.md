# B110 thor 使用Cutile生成的SASS指令流分析

**作者**: 你好努力求职中

**原文链接**: https://zhuanlan.zhihu.com/p/2036209743891535004

---

为了深入理解Blackwell架构的MatMul计算过程，我使用工具追踪了Cutile生成的SASS指令流，结果清晰地展现了指令是如何被warp specialization所编排的。

在前面的文章中介绍了一个自己开发的 SASS 指令阅读器（SASS 指令阅读器，以及对于 MatMul SASS 的初步阅读），并尝试着初步对 Cutlass 生成的 MatMul SASS 指令进行分析（代码开源在 HTTPS://GitHub.com/zhao008/cubin_cfg_html?tab=readme-ov-file）。在上面的思路下，为了能够进一步对 MatMul SASS 进行分析，又为 SASS 指令阅读器增加了一个指令流分析的功能。目前程序依然选择和 Thor 上 cuTile MatMul 实践与 Cache 异常现象分析这篇文章相同的 MatMul 程序。

对于 B110 / Thor 这一类 Blackwell GPU，MatMul 相关的 SASS 指令可以粗略理解为围绕 global memory、shared memory、Tensor Memory（TMEM）和 Tensor Core 构成的一条数据流：UTMALDG.2D 负责通过 TMA 机制把 global memory 中的二维矩阵 tile 按 tensor descriptor 异步搬运到 shared memory，它不是普通的线性 LDG，而是面向矩阵 tile 的 bulk tensor load，能够处理 stride、边界、swizzle 等复杂访问模式；UTMASTG.2D 则是反方向的 TMA store，用于把 shared memory 中整理好的二维输出 tile 异步写回 global memory；LDTM.* 和 STTM.* 则是 Blackwell 新引入的 Tensor Memory 相关指令，前者表示从 TMEM 中读取矩阵 fragment 到普通寄存器，常用于 epilogue 阶段把 Tensor Core 累加结果取出来做 scale、bias、activation 或类型转换，后者表示把寄存器中的矩阵 fragment 写回 TMEM，可用于初始化 accumulator 或构造后续 Tensor Core 操作需要的中间数据；最核心的计算指令是 UTCHMMA，它可以看作 Thor / Blackwell 上的 Uniform Matrix Multiply and Accumulate，对应第五代 Tensor Core 的矩阵乘累加操作，大致执行 D = A × B + D，并且其重要特征是 accumulator 主要保存在 TMEM 中，而不是像早期 HMMA 路径那样大量占用普通寄存器。

在分析 MatMul kernel 的指令流时，目前只重点关注五类和 Blackwell / Thor MatMul 强相关的 SASS 指令：LDTM.*、STTM.*、UTMALDG.2D、UTMASTG.2D 和 UTCHMMA。这些指令分别覆盖了 Tensor Memory 读写、TMA 数据搬运以及真正的 Tensor Core 矩阵乘计算，因此可以把它们看成 MatMul 执行过程中的几个关键观测点。基于这些观测点，工具会在函数 CFG 中追踪那些从 entry 可达、并且最终能够通向 exit 的控制流片段。这里的目标不是寻找一条全局意义上的最短路径，而是想还原这些关键指令所在的“执行走廊”：也就是说，它更关心某条指令前后有哪些控制流结构与之相关，这些结构是否真的处在一条可执行的 MatMul 路径上。下面这几幅图是路径分析的结果：

经过数据流分析发现指令位于三个数据流上面，也就是说对于 MatMul 的具体执行 warp 被至少特化成三个不同的指令流，不同的指令流负责 MatMul 的不同部分。也就是说 cutile 会尝试使用 warp specializatio 的方式排布 SASS 指令，优化 MatMul 的计算效率。也就是说 warp specializatio 已经成为一种必须优化手段。（这是不是也从侧面验证了硅仙人说过 cuda 和 GPGPU 并不一定是一种高效手段，欢迎各位不同意见的大佬们谈谈自己的看法）
