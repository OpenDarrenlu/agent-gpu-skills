年末在疫情的笼罩下，人工智能领域发布了不少新进展。 最火的当然是 AIGC(ChatGPT, Diffusion Models)，不过 PyTorch 2.0 也赚足了眼球。借此机会来聊聊 PyTorch 2.0 里面大家比较关注但是还没有足够重视的部分 TorchInductor: a PyTorch-native Compiler with Define-by-Run IR and Symbolic Shapes。

1.1 TorchDynamo Graph Capture

TorchDynamo 在这个问答"如何看待PyTorch 2.0？"里面已经得到了很多讨论和介绍。简要说就是直接 hook Python bytecode 的解释器并捕获和改写图（如下图)。

TorchDynamo Behavior

显然这是一种嵌入在 Python 解释器里面的 JIT 优化器；这是区别于其他方式的关键点(torch.jit.script 和 torch.jit.trace 是 AOT 导出到非 Python 环境; LazyTensor 则 JIT 运行在逻辑抽象的 Torch Lazy Device 后端中)。

此处略过许多Graph Capture机制的优劣对比，如果感兴趣可以查看 The nuances of PyTorch Graph Capture 中的讨论总结。

Python bytecode 层级的 trace，编译优化和 fallback，使得 TorchDynamo 能够兼顾开发调试的灵活性和 JIT 导出优化的效率。

1.2 Define-By-Run And Symbolic Shapes

这点我认为是 PyTorch Native Compiler 架构最最核心的理念!!!

Inductor 编译优化过程

与 G 家和先前其他的 AI 编译器生态相比，PyTorch Native Compiler 编译当前走了一条迥然不同的编译优化过程。Inductor 的编译优化过程，从最左边的 fx.GraphModule，中间进行 fx.Interpreter 解释运行 decompose 和 lower，再经过 Scheduler 的调度优化和fusion，到最右边的 Triton CodeGen，全程均发生在 Python 环境中。

这也许有 Python Native 口号的因素在里面。但是，在我看来更本质的原因是 Python 的动态语言编程特性给 Inductor 实现一系列特性提供了极大的便利。这些特性包括(不仅限于):

借助 python 层级的 __torch_dispatch__ 实现丰富的自定义功能(What (and Why) is __torch_dispatch__?)。具体例子有 FakeTensor 和 ProxyTensor。
借助 python 层级的 fx.Interpreter 完成对 fx.GraphModule 的解析和预先运行；Inductor 中 torch ops 的 decompose 和 lower 便是基于此；同时一些常量折叠和化简也在此过程中完结。
借助 python 层级的 duck typing 和 monkey patch 完成结构和类型方面的变化。
贯穿始终的 ShapeEnv 和 SymPy，结合 python 的 callback 函数，解释器对 FakeTensor 和 Symblic Shape 部分meta信息的预先执行，将极大地提升了 Inductor 对 alias mutation，动态 Shape/Broadcast 和 Layout 的支持能力。
Define-By-Run + Symbolic Shape 示例

更多的技术细节就不展开了。这里我想 Highlight 的一点是，Python 的动态语言特性，将给予了 Inductor 通过 Define-By-Run 便利地 JIT 化简 PyTorch 非 SSA 特性，非 Tensor 计算和动态 Shape/Layout 的能力。

1.3 Triton Fusion & Template CodeGen

AI 编译器发展的到今天，对访存密集部分进行 Fusion CodeGen，对高效的密集计算 Pattern 和优化策略进行模板化和组合，几乎已经成为了标配。

Inductor 后端中对编译的策略目前相对聚焦和简化，包括对 Reduction/Pointwise 一类算子的 Kernel Fusion，Tiling 以及 CodeGen；对部分卷积和矩阵乘法类型的进行模板化；对 PyTorch 及外部算子库的 external call 支持。（盲猜 PyTorch 社区对当前模型结构收敛趋势是拥有态度的，即使将来发生变化也不慌，毕竟迭代效率这么快）

GPU 上对 Triton 编程语言的选择可以说 PyTorch 社区将 Python Native 理念发挥到了极致。相比于 CUDA, Triton 的编程抽象更加偏向于深度学习，提供了更简洁的抽象(当然也以舍弃了一些更高阶编程能力为代价)。这种选择将会大幅降低用户编写和改进算子的负担，同时提供不逊于 cuda 算子的性能。

2 浅谈 AI 编译技术发展态势

又到了我喜欢的环节，哈哈哈~~~

AI 框架前端收敛的趋势越来越明显，但是变化仍然无处不在。随着人工智能技术的进一步发展，更多应用场景的发现，模型计算对算力的更大需求，AI 框架仍然会继续演化。但是站在这个节点上面，不得不说 PyTorch 前端又扳下了一局。我已经开始幻想，2023年3月份，PyTorch 用户将会纷纷更新自己的版本，默认打开 torch.compile，如果事实如此，那么这个趋势将是不可逆的。也意味着，用户将习惯打开 AI 编译，AI 编译将不再会是普通用户陌生的词汇。

兼容 PyTorch 已然是许多 AI 框架明智的选择，但却不会是 TensorFlow/JAX 的未来！那么，对于 AI 加速硬件的厂商该怎么选择？厂商必定要兼容支持主流的 AI 框架。事实摆在面前，想想 ONNX 在硬件厂商们之间的生命力(不好意思吐槽下)。

3 脑暴一下对未来的憧憬

我这回看好 PyTorch 2.0，但并不意味着否定其他技术路径。客观实事求是一点来说，我觉得 Inductor 为支持动态性提供了很好的思路，就是 Define-by-RUN + Cache + Compile 模式，在此基础上系统要不断持续优化:

JIT Graph Capture 的效率，Define-By-Run 的 Overhead
提高 Compiler 优化 Cache 命中和 multiple dispatch 的效率
Compile 更关注与 Define-By-Run 之后取得的 IR 表示，在这层 IR 表示上将是 PSC(Pure-and-Statically composed) 的，更利于编译优化的
硬件厂商针对深度学习应用场景，提供类似于 triton 层级和粒度的编程抽象，同时对 DSA 加速指令进行合适的抽象封装，指令流水和数据排布可以在底层编译器中更多地自动化

还有一点期待，期待 TensorFlow/JAX 和 MLIR 社区能够适应这个潮流，推陈出新，同时对抗下 PyTorch 2.0 进一步 AI 编程碎片化的趋势。
