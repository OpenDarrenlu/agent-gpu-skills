前言

MoE模型无论是对于算法同学还是对于我们infra土木老哥都已经不是什么新鲜事了，不过感谢年初DeepSeek一波大开源又带来了超大量小型专家的新范式，既然又有了饭吃也就毅然决然从年初开始投入到大规模MoE推理中来，奈何手速确实还是不够快以及写起来遇到的细节中的魔鬼过于多，一直到上个月才算阶段性把项目做出来顺便挂了paper。感谢机器之心的报道让一些老师和专家关注到了这篇工作，也产生了一些疑问，这里就还是来介绍一下工作的具体内容和从一开始想做到最后做完的心路历程，顺便解释一些大家比较关注的问题。

省流版

TLDR: EaaS作为一个大规模MoE推理系统，主要的好处在于与sglang和vllm等和DeepEP结合的主流框架相比几乎不损失效率的前提下: (1)支持以1张GPU为单元的超细粒度扩缩容，更容易在动态流量的serving场景下节约成本。(2)在部分GPU出错离线的情况下也能在恢复期间维持较高的吞吐，不影响实例内的其他计算。(3)更容易支持动态的专家负载均衡算法，在调节过程中更不易影响到其他节点的效率。 实现上EaaS的通信库采用了IBGDA通信bypass了CPU的参与，进一步提高了效率，并进行了一系列算子优化。

TLDR DLC: 与现有工作的区别：(1) EaaS更多关注的是在线serving的成本与鲁棒性问题而非相对于DeepEP的性能提升，因为理论上EaaS的通信量与DeepEP是对等的，我们主要是在相同推理效率下解决了实际落地场景下的弹性扩缩容与容错的灵活性。(2)形式上，我们将专家设计成了无状态的服务，除了正常单模型推理外也可以支持不同进度的attn用户访问，潜在的也可以支持同一部分attn主干在不同场景下访问不同的专家服务以获取最佳效果。(3) 与既往工作相比，EaaS首次提出了无需CPU参与，非对称的高效机间异步P2P通信库（也是这几个月一直在折磨我们的最大的东西之一），采用ibgda实现。通信库在无需cpu参与及访问host memory的情况下，与现有工作相比（StepMesh等）降低了通信延迟，同时也支持了attn和专家端分别的整图cuda graph capture, 节约了很多通信kernel launch的开销。更详细的也欢迎阅读paper的Section 5.5 以及intro中的相关部分。

那么接下来我们就可以正式进入Paper的内容了，没读过的小伙伴也可以先简单阅读一下原文。

Motivation

大家做专家并行也不是一天两天了，DeepSeek在过年时候开源了V3和R1之后（求求了能不能别放假之前一两天开源了）又在后面的开源周公开了DeepEP解决方案，尤其是在公布了当时320卡的best practice之后其实其中一个直接想法就是如果我部署了一个这么大规模的推理实例有的时候跑不满怎么办（和老板也有过一番激烈讨论）。同时当时一直到五月份Sglang支持了DeepEP之前都没有一个能够很好的支持DeepSeek 671B大MoE推理的框架，所以觉得还是有必要做一套适配这个架构的同时能够支持高效且弹性的MoE Serving的框架，毕竟当时除了传统EP以外甚至很多情况下都还在用TP来切expert，确实有很大的效率提升空间。在当时我们的目标就是能达到和DeepEP同等的效率下能够灵活scale，做大规模容错，同时也能想办法支持一下动态负载均衡。后面为了实现这个想法试了两套方案，第一套需要走CPU的感觉效率不行，结果在重新开始写最后这一套ibgda的方案的时候踩了太多的坑，中间就陆续经历了megascale infer, SGlang支持DeepEP, Step3带火了一波AFD，体感上就是一边写一边baseline疯狂上涨（流下了写代码太慢的泪水）。

碎碎念过了，用正式一点来讲的话，我们观察到主流推理框架（如 SGLang、vLLM ，以及五月过后二者与DeepEP 的组合）大多依赖单一通信组（例如NCCL的group）内的大规模 All-to-All 集体通信来实现专家并行，这种设计确实在性能上可以达到很高的效率，但是在实际落地中会有三个比较大的问题：

扩缩容粒度较为粗糙，例如之前提到过的DeepEP best practice为 320 张 GPU一个单元，一旦用户请求量有波动，系统无法立即on the fly地按需增减资源，不利于实际应用中的推理成本控制。除此以外，在调整实际使用的GPU数的时候很多系统也需要GPU数能被hidden dim整除以满足TP等并行的条件，使得能够选择的GPU单位数进一步受到限制。
就像上面提到的，大规模单一EP通信组中只要任一 GPU 或网络节点发生异常，整个 All-to-All 操作即告失败，通常需要整体重启才能恢复，而在大规模集群中，GPU 每周发生掉线、ECC 错误、链路抖动等现象十分常见，使得这种架构在真实服务环境很容易集体崩盘。
专家激活分布具有高度动态性和任务相关性，不同输入场景下“热门专家”会频繁变化，而静态切分策略无法实时感知这种负载倾斜，最终导致某些 GPU 长期过载、某些 GPU 长期闲置，整体吞吐被最“堵”的节点拖垮。负载均衡的工作确实已经有很多了，负载均衡的算法本身也不在本文的讨论重点内，我们主要想设计一种能够更方便的做在线负载均衡模式切换的框架。
一个4096卡集群按周统计的掉卡次数以及原因

对现有推理系统的分析可以发现，前述的扩缩容粗粒度、容错脆弱性和负载倾斜问题，虽然表面上属于“调度”或“通信机制”的缺陷，但其根源实则在于架构层面对不同计算部分的错误建模。当前主流系统将Attention与专家捆绑在同一个进程组内，忽略了两者对于状态依赖的差别。Attn层需要维护 KV Cache 来对应每个用户会话，具有强状态依赖性；而 Expert 前向计算则是标准的FFN，输入输出完全可重复，无需记忆上下文，本质上是无状态的纯函数。所以，传统系统实际上将本应灵活的无状态算子强行绑定在状态驱动的执行框架中，结果便是整体被迫继承了注意力部分的最坏属性：无法独立扩缩容、无法冗余复制、无法容忍部分节点失败。

EaaS设计思路

EaaS 选择从根本上打破这种耦合，将 Expert 从原有计算图中剥离，不再将其当作模型层的一部分，而是将其重新抽象为可独立托管的推理服务（Expert-as-a-Service）。同时，Attn层也被我们拆分成为单独的单元而不再使用TP，作为一个个单独的客户端，分别储存各自独立的KV cache。当Router确定某个 token 需要访问某个 Expert 时，不再通过 All-to-All 将数据广播到通信组内部，而是通过异步 P2P 通信将请求发送至任一拥有该 Expert 副本的服务器，并在计算完成后获取结果。这种服务化架构不仅不再依赖固定规模的通信进程组，也不再要求系统启动时分配固定的专家拓扑。任一 Expert 实例都可以随时上线、下线或复制，并通过一个中心监控器注册到全局可见的专家池中。

如何处理GPU掉线

这样做的好处也很直观，我们这样做之后所有的GPU，无论是attn客户端还是expert服务器，都完全是独立的存在了，你过你的我过我的，如果你这个服务器掉了那我直接查地址本再去找一个活着的有我要的专家的服务器就是了。只要保证还有至少一台服务器上有这个专家，就可以随时通过切换路由完成专家计算，整个大的服务就不会崩。

除了容错以外，做弹性调度和支持动态负载均衡也就很简单了,只需要实时根据当前的用户流量以及负载增加或减少有包含特定专家的服务器就好了。

想法还是很直观简单的，至少我开始的时候也觉得这玩意挺简单的，结果实现一写就是半年。

实现

要实现上面说的这一套服务，是需要写一个异步P2P的通信库的，同时客户端和服务端的数量和操作都不对等，所以这个通信最好还是可以支持非对称内存访问的。同时我们要用分开的节点来处理attn和expert，那么这一套通信最好是能支持机间rdma的。而站在今年年初，并不存在这样一套通信库能满足以上所有的条件，DeepEP本身也有点难学，融入到现有框架里也还有时间，遂决定自己写一套。技术选型有两个，第一个是用传统的rdma接口，但是需要CPU和host memory的介入，另一个则是在DeepSeek论文里提到过的IBGDA(InfiniBand GPUDirect Async), 可以bypass CPU的介入。

需要CPU介入的RDMA通信(https://developer.nvidia.com/zh-cn/blog/improving-network-performance-of-hpc-systems-using-nvidia-magnum-io-nvshmem-and-gpudirect-async/)
绕过了CPU的IBGDA (https://developer.nvidia.com/zh-cn/blog/improving-network-performance-of-hpc-systems-using-nvidia-magnum-io-nvshmem-and-gpudirect-async/)

后者显然是效率更高的，但是问题在于虽然已经不是这两年刚提出的技术了，IBGDA在社区里能查阅的资料和代码并不是很多，而且默认使用的也是对称显存，要用到EaaS这一套思路上就要自己改更底层的代码，开发代价还是相当大的。于是想要快速验证disaggregation有没有用的我们毅然决然的投向了前者，其中更重要的一个原因是年初的时候读到了 @陈乐群 大佬的一整个系列libfabric使用指南 驾驭3200Gbps网络，对于本身并非做网络出身的我真的是久旱逢甘霖，再次无比感谢乐群大佬的经验分享。

初期试错版：Libfabric

第一版选定了libfabric作为通信库之后，就可以开始设计具体的实现方式了。首先我们所要的是一个尽量stateless的专家服务器，所以能够确定的第一点是所有的通信请求都应该由客户端发起，那么正好rdma有read和write两个当方面操作，我们这里就让每一个attn客户端对应server上的一块注册好的显存，当当前decode的tokens经过router之后，我们先将token scatter到num token*k块对应的本地显存上准备进行rdma写。这里参考了驾驭3200Gbps网络(6): GPUDirect RDMA WRITE，利用了带立即数的rdma write来触发写的完成事件以便让server侧了解写操作已经完成。但是每次client侧向server写入的数据都不一定是同一个长度，而是会随着Batch size和router的结果动态变化, server侧是并不知道这些信息的。于是在每次请求之前，我们都会追加一个request header，其中包括了magic校验码，随机种子，token对应的expert id，layer编号，以及最重要的，这一波request的具体shape。header的大小是固定的，那么server每次拿到完成队列中的信息后就可以通过读对应位置buffer上写好的内容来获知后面跟了多大的tensor。同时为了减少通信量，tensor的fp8量化是在客户端完成的，所以实际上每个request实际是由一个固定大小header，(b, 1, h)的fp8 tensor，（b, 1, h）的router score，以及(b, 1, h//128)的scale构成的。

到了这一步，我们的第一步：把tensor写到专家服务器，就已经完成了。下一步就是expert拿到tensor进行计算了。这里我们用的是小小魔改了的deepgemm，并采用了continuous batching的模式，一次性将当前所有写过来的tensor pop出来打成一个大batch计算来提升效率，这里我们也做了一些小的优化，例如在group gemm的过程中识别并跳过没有被激活的expert等等。计算结束之后server会将计算结果写回原来buffer的对应位置，但是客户端该怎么知道server已经算完了呢？为此我们额外设计了一个标志位，用0-3四个数字分别表示空，客户已写完，服务器已计算完，以及客户端离线。当server计算结束并写回buffer后，这个标志位就会被置为2，而在计算期间客户端其实一直在poll这个标志位直到读到2为止。这之后client就会再发起一个rdma read，将结果读回去。




EaaS服务器的流程设计

这便是我们初版的完整的一套流程，初次之外还有一些其他的小细节，比如大家都知道decode阶段是必须开cuda graph来降低大量细碎kernel launch带来的时间开销的，但是libfabric的通信操作都是需要cpu参与进来的，这再cuda graph capture中是不被允许的。所以在这一版实现中，我们把cuda graph拆成了两节，在通信这里断开，引入了一些额外的时间开销但是至少保证了其他部分不受影响。

在我满怀希望的爬出bug堆终于把这一套东西在DeepSeek V3上跑起来并且对上精度之后，就迎来了绝望的事实：现在这一版实现不够快。这不仅仅是cuda graph断开的问题，更是当时有很多操作都是串行没有并行化，以及kernel效率低下一系列导致的连锁反应。此时四五月的会议的ddl已经近在眼前，更何况此时SGLang社区支持了DeepEP导致baseline又变高了，我们也没有充足的时间把性能在这一套上做到能打的程度了，于是干脆放弃了这一个版本。反正时间变多了，那么干脆还是抛弃包装好了的libfabric，从最底下开始写一套基于ibgda的，真正的CPU-free的通信库吧。

放弃幻想，面对现实。

最终版本：基于IBGDA的CPU-free非对称异步P2P通信

没有了现成包装好的high level库，就只能自己基于nvshmem重写一遍这些API，包括但不限于建联和读写等等，而nvshmem不支持非对称内存，同时又包含了很多不必要的额外信息和代码，所以进行了一番魔改。

首先，在建联阶段，客户端和服务器分别初始化各自的P2P通信模块，这会创建InfiniBand设备、分配GPU内存、初始化传输层，并创建所有需要的队列对（QP）包括DCI和RC类型；接着，双方分配GPU内存，然后调用将内存注册到InfiniBand硬件以获得访问键值；随后，双方分别调用get_local_conn_handle()和get_local_mem_handle()获取各自的连接句柄和内存句柄，这些句柄包含了QP号、本地ID、端口信息等关键连接参数；接下来是关键的句柄交换阶段，双方交换连接句柄和内存句柄，设置对端的内存句柄，然后传入对端的连接句柄来建立实际的网络连接；在传输层，系统会执行完整的QP状态转换（从RST到INIT到RTR再到RTS），配置每个队列对的属性，建立可靠的InfiniBand连接；最后，系统会将完整的连接状态信息同步到GPU内存中，包括QP指针、内存键值、完成队列等信息，这样GPU内核就可以直接访问这些硬件资源进行零拷贝的高性能通信了。

利用IBGDA之前的建联流程

这样建联之后，GPU本身就已经包含了进行RDMA读写的所有信息而不需要再通过CPU和host memory了，除此之外的通信流程也大致与上面libfabric相似，不过在write的过程中我们也统一改为了用state来判断client是否已经写完成，同时用了原子操作来保持一致性。而NV的blog中也说到，使用IBGDA是要占用一些SM的。我们第一版尝试了用一个warp对应一个server的发送方法，但是发现这样并没有办法解决不同server的负载不均问题，于是我们改为了根据数据发送量来切分warp负载的办法，而非单纯的根据发送的server来做映射。除此以外我们又做了一些其他细节kernel的并行优化，在这里就不多展开了。

改为IBGDA后，另一个很直观的好处是现在又可以进行整图CUDA Graph capture了，通信也不再需要额外的kernel launch时间，又省下来一大笔开销。在这里顺便提一下自己在写double batch的两个独立stream的cuda graph capture之后因为没有经验试了好多次，最终得到的一个较为稳定的实现是类似于：

# 以简单的拆分成attn和expert两个操作为例

with torch.cuda.stream(stream_a):
    op_attn_a(0)

torch.cuda.current_stream().wait_stream(stream_a)

for i in range(total_layers * 2 - 1):
    if i % 2 == 0:
        stream_a.wait_stream(torch.cuda.current_stream())
        stream_b.wait_stream(torch.cuda.current_stream())
        with torch.cuda.stream(stream_a):
            op_expert_a(i // 2)
        with torch.cuda.stream(stream_b):
            op_attn_b(i // 2)
        torch.cuda.current_stream().wait_stream(stream_a)
        torch.cuda.current_stream().wait_stream(stream_b)
    if i % 2 == 1:
        stream_a.wait_stream(torch.cuda.current_stream())
        stream_b.wait_stream(torch.cuda.current_stream())
        with torch.cuda.stream(stream_a):
            op_attn_a(i // 2 + 1)
        with torch.cuda.stream(stream_b):
            op_expert_b(i // 2)
        torch.cuda.current_stream().wait_stream(stream_a)
        torch.cuda.current_stream().wait_stream(stream_b)

with torch.cuda.stream(stream_b):
    op_expert_b(total_layers - 1)

虽然这个实现可能不太是最佳实践，但是确实是一个比较好理解且稳定的实现方法，也欢迎评论区大佬给出一些别的更好地实现模式。

利用double batch进行了overlap

在我们琢磨IBGDA的途中，阶跃也是放出了Step3，其中的AFD也是吸引了大家的关注，所以我们在实验中也包含了EaaS和StepMesh的通信库从torch端发出到收回的end2end延迟测试（不包含计算），可以看出利用ibgda的CPU-free特性还是可以在通信上节省出相当一大部分时间的。
